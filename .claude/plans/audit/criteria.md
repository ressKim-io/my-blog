# 판정 기준

## 2축 분류 (Level × Type)

블로그는 2축으로 봅니다

### Level 축 (학습 성장 트랙)

| Level | 특징 | slug 힌트 |
|-------|------|-----------|
| **L1. 기본** | 개념·기초 튜토리얼 | istio-intro, k8s-pod-flow, istio-part* |
| **L2. 심화** | 특정 영역 깊이 탐구 | istio-traffic, istio-security, istio-observability, istio-ambient, eks-troubleshooting |
| **L3. 실무통합** | go-ti 4개월 프로젝트 (2026-01~04) | goti-, challenge1-game-server, queue-poc-loadtest 등 |

> 블로그 전체 서사가 "기본 → 심화 → 실무통합" 성장 궤적을 보여주는 것 자체가 중요

### Type 축 (글 유형)

| 유형 | 특징 | 판정 포인트 |
|------|------|------------|
| **A. 단순 트러블슈팅** | 에러 → 원인 → 해결 | 완결성만 있으면 OK (서사구조 강요 X) |
| **B. 의사결정 서사** | ADR, 비교·선택, 아키텍처 결정 | 3요소 필수: 대안/시도/선택이유(+컨텍스트) |
| **C. 학습/정리** | 개념 설명, 튜토리얼 | 독창성·깊이 평가 (L1과 겹침) |
| **D. 기록/로그** | 메타 회고, 작업 후기 | 인사이트 유무로 가치 판정 |

### Level × Type 조합 해석

- L1-C: 기본 개념 글. 독창성보다 **명확성**이 중요
- L2-A/B: 심화 트러블슈팅/결정. B면 서사 3요소 필수
- L3-A: go-ti 실전 트러블슈팅. 이 블로그의 핵심 자산
- L3-B: go-ti ADR·선택 글. **가장 중요한 영역**, 프로젝트 맥락 why 필수
- L3-D: go-ti 메타 회고. 반복되면 공해 가능성, 인사이트 고유성 검증

## 1. 중복 의심 신호

- 제목 80%+ 유사 (예: `istio-part1`, `istio-intro-part1` 같은 두 시리즈 겹침)
- 동일 시리즈 내 파트끼리 내용 겹침 (Phase 2에서 샘플 크로스체크 필요)
- 같은 문제(에러코드/키워드)를 다룬 독립 글이 2개 이상
- 태그 70%+ 겹치면서 카테고리 같음

## 2. 저가치 의심 신호

- **너무 짧음**: 본문 1500자 미만 (excerpt 제외)
- **코드 비중 과도**: 코드 블록이 본문의 60%+ (설명 부족)
- **단순 설정 기록**: "이렇게 설정했다" 수준, 왜·대안·실패가 없음
- **제목 대비 얕음**: 제목은 거창한데 본문이 일반론

## 3. 의사결정 서사 점검 (B 유형만)

3요소 체크 — 하나라도 결여면 리라이트 후보

- [ ] **대안 제시** — 선택 외에 다른 옵션 A/B/C 를 검토했는가
- [ ] **실패/배제 이유** — 배제한 대안의 단점을 구체적으로 설명했는가
- [ ] **선택 이유 + 프로젝트 맥락** — 단순히 "이게 좋아서"가 아니라 "우리 프로젝트는 X 특성이라 Y 선택" 서사

### 3요소 추출 방법

- 키워드 스캔: "대안", "비교", "선택", "A vs B", "왜", "이유", "그래서"
- 섹션 헤더 스캔: 🤔 원인, ✅ 해결 같은 이모지 헤더
- 부재 판정: 키워드 0~1개 + 의사결정성 글 → 리라이트 후보

## 4. "프로젝트 맥락 why" 점검

예시 기준 (사용자가 명시한 이상적인 서사)
- 외부 현실(AWS 리전 장애, 대전 화재) → 우리 요구(무중단·대용량) → 그래서 GCP/Cloudflare 도입

점검 포인트
- [ ] 외부 트리거/제약이 언급되는가 (장애 사례, 규제, 비용, 인력)
- [ ] 우리 프로젝트의 고유 특성이 언급되는가 (go-ti 규모, 목표 트래픽, 팀 크기)
- [ ] 그 특성이 선택에 어떻게 연결되는가 (~이기 때문에 ~선택)

## 5. 날짜 재분배 기준

- 현재 `date:` 기준으로 월별 분포 확인
- 3~4월 집중 글 중 2월로 옮겨도 시리즈 연속성·내용 모순이 없는 글 추림
- 우선 이동 후보: 독립 글, 시리즈 시작점, 주제 연결성 약한 글
- 제외: 시리즈 중간 파트(연속성), 다른 글이 "전편" 명시적으로 참조

### go-ti(L3) 타임라인 자연스러움

go-ti 실무통합 프로젝트는 **2026-01~04 4개월** 진행된 것으로, 2월 작업 로그가 3월에 발표됐을 뿐 내부적으로 2월에 해당하는 작업이 존재합니다. 따라서 goti- 글을 2월로 재분배해도 프로젝트 타임라인상 자연스럽습니다

- 2월 이동 시 프로젝트 단계(초기 인프라/중기 트러블슈팅/후기 결정) 감안
- 가능하면 초기 설정·PoC성 글 → 2월 상순, 후기 결정·회고 → 4월 잔존

## 6. 유형 메타 태그 정책 (사용자 확정)

**원칙**: 글 1개 = 주제 1개 유지 (묶음 병합 안 함). 대신 **유형 메타 태그를 부여하여 트러블슈팅과 정보글을 명확히 분리**

### 유형 태그 (4종)

| 태그 | 대상 유형 | 예상 편수 |
|------|----------|----------|
| `troubleshooting` | A유형 (에러 → 원인 → 해결) | ~113편 |
| `adr` | B유형 (의사결정 서사, 대안 비교, 선택) | ~62편 |
| `concept` | C유형 (개념·학습·튜토리얼) | ~8편 |
| `retrospective` | D유형 (메타 회고, 작업 후기) | ~13편 |

### 부여 규칙

- 모든 글에 유형 태그 **1개**만 부여
- 기존 주제 태그는 **그대로 유지** (예: `istio`, `go-ti`, `kafka`)
- 유형 태그는 `tags` 배열 **마지막**에 추가 (기존 `tags[0] == "go-ti"` 컨벤션 보존)
- 태그 명칭은 영문 소문자 (기존 태그 스타일과 일치)

### 효과

- `/blog?tag=adr` → 의사결정 서사 글만 모아보기 가능
- `/blog?tag=troubleshooting` → 단순 트러블슈팅 모아보기
- go-ti 필터(`tags[0]`) 기존 동작 유지
- (선택) 블로그 상단에 `[전체] [ADR] [트러블슈팅] [개념] [회고]` 필터 버튼 신설 가능 — 별도 UI 작업

### 부여 방식

1. `scripts/classify-types.js`로 자동 추정 (키워드 기반)
2. 추정 결과를 `type-assignments.md`에 목록화
3. 사용자 검수 후 일괄 적용

## 7. 핵심 정책 합의 (사용자 확정)

- **기본 플랜 + 태그 분리** 방향으로 확정
- 글 병합은 중복 명백한 경우(🔴 3건)만 진행, 나머지는 "글 1개 = 주제 1개" 원칙 유지
- 서사 글(B유형) 돋보이게 하는 것이 품질 상승 방향
- 양 감소는 부차적 목표 (중복 제거로 한정)
