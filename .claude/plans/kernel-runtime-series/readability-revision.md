# kernel-runtime-tradeoffs 시리즈 가독성 개정

> **세션 재개용 문서**. 2~6부 깊이 개정을 이어갈 때는 이 파일부터 통독한 뒤
> §3단계 추적 표에서 첫 미착수 부부터 진행합니다. 1부 파일럿(2026-07-22)은 완료·커밋됨.

## Context

팀원이 runtime 카테고리의 "커널과 런타임으로 톺아보는 Rust·Go·Java" 시리즈(당시 27편,
`src/content/essays/runtime/`)를 읽고 이해가 안 된다는 피드백을 줬습니다. 구체적으로:

- 배경지식 부족 — 전제 개념 설명 없이 바로 시작
- 논리 비약 — 결론은 있는데 그 결론에 이르는 추론 과정 생략
- 문장이 너무 압축적 — 한 문장에 정보가 과밀
- 예시·비유 부족 — 추상적 설명만 있고 구체적 그림이 안 그려짐

사용자 확인: 27편 전체가 대상이며, 초심자도 따라올 수 있게 "풀어쓰기"를 원함.
필요하면 편을 나눠도 되고, 부(部)별로도 훌륭한 독립 지식 단위가 되길 원함.

이 시리즈는 실측·소스 확정 밀도가 핵심 자산입니다(`plan.md`, `part6-fact-audit.md`).
가독성을 높이되 이 신뢰성 있는 사실 기반은 그대로 보존해야 합니다. 또한 편들은 전역
순번("N편")으로 서로를 참조하는 구조라 — 시리즈 하나를 쪼개면 그 뒤 모든 편의 "N편"
표기가 밀립니다. 사용자는 "정말 과밀한 편만 실제로 분할하고, 전체 리넘버링은
감수하겠다"고 확정했습니다.

## 재작성 방법론 — 스타일 가드레일 안에서 "풀어쓰기" (2~6부에도 동일 적용)

CLAUDE.md의 절대 규칙(격식체·짧은 문장·마침표 생략·이모지 금지·직역 은유 금지)은
그대로 유지합니다. "풀어쓰기"는 문장을 길게 늘이는 게 아니라 **문단 수를 늘리고,
각 문단이 담는 개념 수를 줄이는 것**입니다.

1. **배경 프라이밍 문단 추가** — 새 개념 뭉치를 던지기 전에, 왜 이 문제가 중요한지
   실감 나는 상황 1~2문장을 먼저 놓습니다
2. **단계적 논증(계단식 서술)** — "결론 한 문장 + 근거 한 문장"으로 끝내지 않고
   중간 추론 단계를 명시적 문장으로 하나씩 놓습니다
3. **개념당 문단 분리** — 한 문단·한 불릿에 여러 개념이 몰린 곳을 쪼개 각 개념이
   숨 쉴 공간을 줍니다
4. **자연스러운 한글 비유** — 영어 은유 직역은 여전히 금지(기존 렉시콘 유지).
   새로 넣는 비유도 이 원칙을 따라야 하며, 정의를 대체하지 않고 정의 앞뒤에서
   직관을 보조하는 역할만 합니다
5. **절 전환부 브릿지 문장** — 지금까지 무엇을 봤고 왜 다음 절로 넘어가는지
   1문장을 절 경계마다 명시적으로 추가합니다

## 분할 판단 기준

한 편에 독립적으로 완결된 개념 단위가 2개 이상 있고 둘 다 현재 압축돼 있으면 분할을
검토합니다. 분할은 전역 번호 밀림(아래 §전역 번호 절차)이라는 비용을 동반하므로
"정말 과밀한 편만" 신중히 적용 — 기본은 in-place 개정입니다.

## 전역 번호(N편) 밀림 처리 절차 (분할이 있는 부에 재사용)

부1 편성: 1편(syscall, 변경 없음) → 2편(이론부, 기존 슬러그) → **3편(언어 구현부,
신규)** → 4편(runtime-weight). 2부~6부의 기존 4~27편은 전부 **+1** → 5~28편으로
완료했습니다.

- **series.order는 부-로컬**이라 분할이 일어난 부만 order가 바뀝니다(2~6부는 불변)
- **date 필드는 소급 변경하지 않습니다** — 이미 발행된 날짜를 건드리지 않고, 신규
  글은 실제 작업일로 발행합니다. `date = 06-25+N` 공식은 1부 분할 시점부터 깨짐
  (날짜 순번보다 발행 이력 보존이 우선이라는 원칙을 이후 부에도 유지)
- **전역 번호 치환 절차**: `grep -oE '[0-9]+편' *.md`로 전수 조사 → `perl -i -pe
  's/(?<!부 )(\d+)편/(조건) ? ($1+1)."편" : $1."편"/ge'`로 일괄 치환(단일 정규식
  패스라 순서 무관, "N부 M편" 같은 부-로컬 라벨은 `(?<!부 )` lookbehind로 보호)
- **분할 경계에 걸친 애매한 번호(예: 2편)는 자동 처리 대상에서 제외**하고 내용을
  읽어 수동 판정. 1부에서는 "2편" 인용 14곳 중 8곳이 실제로는 언어 구현부(신규
  3편) 내용을 가리키고 있어 개별 확인 후 링크·번호를 `mn-scheduler-go-java-rust`로
  재배선. 나머지는 이론부(2편) 유지가 맞아 그대로 둠. 1곳은 이론+구현을 동시에
  가리켜 "2편과 3편"으로 병기
- **다음/이전 편 링크는 slug 기반**이라 번호와 무관 — 텍스트 속 숫자만 손보면 됩니다
