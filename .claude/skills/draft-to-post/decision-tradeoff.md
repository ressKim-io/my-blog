# 🧭 선택지 비교 섹션 가이드

블로그 글에 **"왜 이 방법을 택했는가"**를 드러내기 위한 섹션 스캐폴딩입니다. 트러블슈팅/ADR 모두에 조건부로 적용합니다.

## 왜 필요한가

단순 트러블슈팅(`문제 → 원인 → 해결`)은 "무엇을 고쳤는지"만 보여주고 **의사결정 과정**을 빠뜨리는 경우가 많습니다. 실제로는 해결 경로가 여러 개 있었고, 어떤 제약·기준으로 하나를 골랐는지가 글의 가장 가치 있는 부분인 경우가 있습니다. 이 가이드는 그 맥락을 놓치지 않기 위한 장치입니다.

## 언제 넣는가 (판단 신호)

원본 draft를 읽고 **아래 중 하나라도 해당**하면 `🧭 선택지 비교` 섹션을 넣습니다.

### 강한 신호 (넣는다)

- 원본에 `Option A/B/C`, `안1/안2`, `방안 1/2/3` 같은 대안 나열이 있음
- "~~를 고려했으나 ~~때문에 ~~로 결정"같은 비교 문장이 있음
- 도구/아키텍처/기술 스택 선정이 본문 주제 (ADR 성격)
- 같은 문제를 풀 수 있는 해결법이 **여러 개**였다고 원본에서 말함
- 롤아웃 순서/단계 설계가 본문 주제 (D0~D7 같은 단계별 고민)

### 약한 신호 (넣을 수도 있음 — 원본이 충분하면)

- "처음엔 ~~로 시도했지만 실패해서 ~~로 바꿨다"같은 시행착오
- "다른 팀은 ~~를 쓰지만 우리는 ~~"같은 맥락 비교
- 해결 과정에서 트레이드오프를 명시적으로 저울질한 기록

약한 신호는 원본에 **정보가 충분할 때만** 넣습니다. 1줄 언급만 있으면 억지로 섹션을 만들지 않습니다.

## 언제 넣지 않는가 (중요)

- 단일 원인 → 단일 해결: typo 수정, env 누락, 버전 불일치, 오타
- 원본에 대안 언급이 **전혀 없는** 글
- 해결책이 공식 문서대로 따르기만 하면 되는 경우
- 원본이 후처리 cleanup 기록(`삭제 완료`, `정리 완료` 등)

**절대 규칙: 원본에 없는 대안을 만들어내지 않습니다.** "만약 이런 선택지도 있었을 것이다"는 추측은 블로그 글에 넣지 않습니다. 이는 사실 훼손이며, 원본 draft의 진실성을 해칩니다.

## 섹션 구조

### 위치

- **트러블슈팅**: `🔥 문제 → 🤔 원인 → 🧭 선택지 비교 → ✅ 결정 + 이유 → 📚 배운 점`
- **ADR**: 기존 `배경 → 선택지 비교 → 결정 → 근거` 그대로. `🧭` 이모지는 선택지 비교 섹션에만 붙임(생략 가능, ADR은 일관성 유지 우선).
- **혼합형**: 트러블슈팅이 주제지만 해결책 선정에 의사결정이 있었다면, `✅ 해결` 앞에 `🧭 선택지 비교`를 끼워 넣음.

### 내부 구조 (3단)

```markdown
## 🧭 선택지 비교

### 고려한 옵션

| 옵션 | 핵심 아이디어 | 장점 | 한계 |
|------|---------------|------|------|
| A. <이름> | 1줄 요약 | ... | ... |
| B. <이름> | 1줄 요약 | ... | ... |
| C. <이름> | 1줄 요약 | ... | ... |

### 기각 이유

- **A 탈락**: <제약 조건> 때문에 불가. 예: "Cloud SQL은 pglogical extension 미지원"
- **B 탈락**: <제약 조건> 때문에 부적합. 예: "Patroni/etcd는 1주 시연 대비 과투자"

### 결정 기준과 최종 선택

**C를 채택했습니다.**

결정 기준은 다음 우선순위입니다.
1. <1순위 기준>: 예) "1주 시연 범위에서 끝낼 수 있는가"
2. <2순위 기준>: 예) "롤백 가능한가"
3. <3순위 기준>: 예) "운영 복잡도가 낮은가"

C는 ...한 점에서 이 기준을 가장 잘 만족합니다.
```

옵션이 2개일 때는 표 대신 문단으로 풀어써도 괜찮습니다. 옵션이 4개 이상이면 표 + 기각 이유를 짧게.

## 결정 기준 예시

어떤 기준을 "우선순위 1"로 쓸지 판단할 때 참고하는 공통 축:

- **시간 제약**: 런칭/시연 데드라인, 남은 스프린트
- **비용**: 인프라 비용, 라이선스, 운영 인건비
- **운영 복잡도**: HA 구성 필요 여부, 팀 스킬셋, 모니터링 포함 여부
- **안정성**: 장애 블라스트 반경, 롤백 용이성, 검증된 기술인지
- **확장성**: 향후 규모 증가 대응, 벤더 록인 회피
- **재작업 비용**: 잘못 골랐을 때 되돌리는 비용

원본에서 어떤 축이 강조됐는지 먼저 찾아내고, 그것을 1순위로 드러냅니다. 원본에 없는 기준은 쓰지 않습니다.

## 좋은 예시

### 예시 1: pglogical 채택 ADR (S3-02)

원본에 "Cloud SQL은 pglogical 미지원", "AlloyDB는 GCP 전용", "Patroni+etcd는 1주 시연 대비 과투자" 3개 옵션이 명시됐습니다. 이런 경우:

```markdown
## 🧭 선택지 비교

| 옵션 | 구성 | 결과 |
|------|------|------|
| A. Cloud SQL 페어링 | GCP Cloud SQL ↔ AWS RDS | pglogical extension 미설치 → 탈락 |
| B. AlloyDB | AlloyDB on GCP + AWS RDS subscriber | AlloyDB는 GCP 전용 + 비용 과투자 → 탈락 |
| C. GCE VM + pglogical | GCE VM 위 Postgres 16 + pglogical 2.x | 범위에 맞음 → 채택 |

### 결정 기준
1순위는 **1주 시연 범위 내 구현**이었습니다. HA/백업은 Phase B에서 별도 처리하기로 하고, Phase A는 **기본 기능 검증**에 집중했습니다.
```

### 예시 2: 롤백 설계가 선정 이유인 경우

```markdown
## 🧭 선택지 비교

두 방식을 저울질했습니다.

**옵션 A — Blue/Green**: 신규 버전과 구버전을 동시에 운영한 뒤 트래픽을 한번에 전환. 장점은 즉시 롤백이지만, 리소스가 2배 필요.

**옵션 B — Rolling 롤백 훅**: 기존 Deployment를 점진적으로 교체하되, 실패 시 Helm rollback을 자동 트리거. 리소스 1배, 다만 롤백에 수 분 소요.

**결정 기준**은 **"실패 시 1분 내 롤백 가능한가"**였습니다. 과거 incident에서 롤백 지연으로 blast radius가 확산된 경험이 있었기 때문입니다. 옵션 A를 채택했습니다. 리소스 2배는 CI/CD 파이프라인 시간 한정이라 허용 가능한 비용입니다.
```

## 나쁜 예시 (이렇게 하면 안 됨)

### 나쁜 예 1: 원본에 없는 대안 창작

원본에 "Redis로 바꿨다"만 적혀 있는데 블로그 글에서:
```markdown
### 고려한 옵션
- A. Memcached — ...
- B. Hazelcast — ...
- C. Redis — 채택 ✓
```

→ **금지**. Memcached/Hazelcast 비교는 원본에 없는 창작입니다. 사실 훼손.

### 나쁜 예 2: 기준 없이 "이게 좋았습니다"

```markdown
### 결정
Redis를 선택했습니다. Redis가 가장 좋기 때문입니다.
```

→ 기준이 없음. 독자가 "왜?"를 알 수 없음. 원본에서 **어떤 축**으로 저울질했는지 꼭 드러냅니다(시간/비용/운영복잡도 등).

### 나쁜 예 3: 섹션을 억지로 끼운 트러블슈팅

단순 env 누락 버그인데:
```markdown
## 🧭 선택지 비교
- A. env 다시 설정
- B. env 기본값 하드코딩
→ A 채택
```

→ **금지**. 대안이랄 것이 아닌 유일한 해결법을 선택지처럼 분식하면 글이 길고 어색해집니다. 이런 글은 기존 4섹션 구조(`🔥🤔✅📚`)만 사용.

## blog-writer agent에서의 사용 흐름

1. 원본 draft를 Read하고 **"대안 신호"** 검색 (위 "강한 신호" 목록 기준)
2. 신호가 있으면 `🧭 선택지 비교` 섹션을 넣을 계획을 잡고 이 파일 전체를 Read
3. 원본에서 옵션·기각 이유·결정 기준을 **추출만** 함. 만들어내지 않음
4. 섹션 구조대로 작성
5. 완료 보고에 `🧭 선택지 비교 섹션 포함 여부`를 명시

신호가 약하거나 없으면 이 파일을 Read하지 않고 기존 4섹션 구조로 진행합니다.
