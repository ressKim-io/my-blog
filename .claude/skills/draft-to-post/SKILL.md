# Draft → Blog Post 변환

## 입력/출력

- **입력**: `drafts/<name>.md` (원본 로그/ADR/학습 정리 — 반말체·내부 표현·ASCII 그림 등 블로그 양식 미준수 상태)
- **출력**: `src/content/{track}/{category}/<slug>.md`
- **자산**: `public/diagrams/{slug}-{n}.svg`, `public/images/{file}.png` — **평탄 구조 유지** (본문의 `/diagrams/...` 경로 변경 X)

## 0단계: 시리즈 확인 (가장 먼저)

- 진행 중인 시리즈는 시리즈별 plan 문서가 SSOT: `.claude/plans/kernel-runtime-series/plan.md` (커널·런타임 시리즈, 편별 slug·order·상태 표 포함)
- **plan의 확정 규칙이 이 스킬보다 우선**. 반드시 통독 후 시작하고, 편 완료 시 상태 표를 갱신한다
- 어느 시리즈에도 속하지 않으면 단독 글 (`series` 필드 생략)
- 계획에 없는 시리즈성 draft를 만나면 **사용자에게 확인** (추측 금지)

> **레거시 주의**: goti 실전 기록(197편)과 rust-cs-layer(7편)는 변환이 **완료**된 과거 프로젝트다. 당시 관례(이모지 섹션 헤더, `go-ti` 첫 태그, `goti-` slug, 마침표 혼용)는 기존 발행 글에만 남아 있으며 **소급 수정하지 않는다**. 신규 변환은 전부 아래 단일 규칙을 따른다.

## 트랙·카테고리 결정 (1순위 SSOT)

**디렉토리 위치가 트랙·카테고리의 단일 진실 원천**입니다. frontmatter `type` 필드는 표시 메타로만 쓰이며 트랙 격리에 영향 없음.

**트랙 결정** — 본문 구조로 판단:
- `essays/` — 개념 정리 / ADR / 회고 (다듬은 글). 심화 해설 시리즈는 항상 essays
- `logs/` — 트러블슈팅 / 작업 노트 ("증상 → 원인 → 해결")
- **한 시리즈는 한 트랙으로 통일** (혼합 시 시리즈 navigation이 갈라짐)

**카테고리 9종**: `kubernetes` · `istio` · `challenge` · `monitoring` · `argocd` · `cicd` · `network` · `rust` · `runtime`

frontmatter `category`와 디렉토리 위치가 일치하도록 작성합니다. 카테고리 신설 시 표시명 맵 4곳(`src/app/page.tsx`, `src/app/essays/page.tsx`, `src/components/PostCard.tsx`, `src/components/HomeCategoryFeed.tsx`)과 CLAUDE.md 갱신 필요.

## 네이밍

- 접두사 없는 영문 주제 slug (예: `syscall-mode-switch-cost.md`). 시리즈 plan에 slug 제안이 있으면 그걸 따름
- 원본의 날짜 접두어(`2026-03-22-`)는 제거 — 파일명이 URL slug
- slug는 전역 유일 (트랙·카테고리가 달라도 URL 충돌)

## Front Matter

```yaml
---
title: "명확한 한 줄 제목 — 부제로 포인트 강조"
excerpt: "무엇을 다루는지 1~3문장 (검색·카드 노출용)"
category: runtime  # 9종 중 택1, 디렉토리와 일치
tags:
  - <주인공 언어/기술>   # 그 글의 중심 기술부터 앞에
  - <기술명2>
series:
  name: "<series-slug>"  # 단독 글이면 series 필드 전체 생략
  order: 1
date: "YYYY-MM-DD"     # 발행일
---
```

- `title` 60자 이내. 질문형 후킹 관례 권장 (예: "왜 X일까 — Y")

## 문체 (단일 규칙 — 2026-07-10 사용자 확정)

- **격식체 100%** (~합니다 / ~입니다 / ~했습니다). 원본이 반말·해요체여도 전부 변환. 리듬용 "~죠"와 의문형 "~까요?"는 가끔 허용
- **이모지 섹션 헤더 금지**. 헤더는 텍스트만, 질문형/서술형 + 필요시 "— 부제". 장식 이모지도 금지
- **마침표 최대한 생략 + 짧은 단락으로 호흡**:
  - 한 단락 = 한 생각. 1~2문장으로 짧게 끊고 **엔터(단락 분리)로 흐름을 만든다** — 마침표 대신 단락이 경계 역할
  - 단락·불릿·표 셀 끝 마침표 생략. `?` `!`는 유지, URL/버전/소수점/약어는 예외
  - 부득이 한 단락에 문장이 여러 개면 중간 경계에만 마침표를 찍고, 가급적 단락을 나눠 그 상황 자체를 피한다
- 한 문장 50자 이내 권장
- 모든 코드블록에 언어 명시 (`bash`/`yaml`/`text` 등)
- **번역투 금지**: 문장 중간의 대시(—) 삽입구·절 연결 남발, "~라는 점" 반복, 영어 구조 직역 금지. 자연 연결어(그래서, 이때, 반대로, 결국)로 끊어 쓰기. 소리 내 읽어 어색하면 실패
  - 대시 허용 예외: 제목 부제("제목 — 부제"), 불릿의 "**굵은 라벨** — 설명" 형식

### 직역 은유 렉시콘 (2026-07-11 신설 — `scripts/lint-post.mjs`의 LEXICON과 함께 갱신)

빈도·성능을 온도·신체동작으로 옮기는 영어 은유 직역 금지. **draft 원문에 있어도 따라 쓰지
않는다** — 보존 대상은 수치·사실·기전이지 표현이 아님 (2026-07-11 사용자 피드백: "한글에선
존재하지 않는 말투". 실제로 draft 원문의 "뜨거운 소수만"이 그대로 유입된 사고가 원인):

| 금지 (직역) | 대체 |
|---|---|
| 뜨거운 메서드/함수/루프, ~가 뜨겁다 (hot) | 자주 실행되는 · 자주 불리는 · 핫(hot) — 첫 등장에 풀이 병기 |
| 시동/성능이 기어간다 (crawl) | 크게 느려진다 · 한참 걸린다 |
| 스레드/큐가 마비된다 (paralyze) | 감당하지 못한다 · 큐가 밀린다 |
| CPU/코어를 훔쳐 간다 (steal) | 가져가 쓴다 · 잠식한다 |
| 프로파일을 컴파일에 먹인다/되먹인다 (feed) | 반영한다 · 입력으로 준다 |
| 연산이/컴파일이 비싸다 (expensive) | 비용이 크다 · 시간이 많이 든다 |

허용(자연스러운 한국어 용법): 청구서·대가·값을 치르다·안전망·계단·병목·복선을 갚다,
"서비스 마비"(대규모 장애 서술), work stealing 풀이의 "훔쳐 오다", 엿보다 뜻의 "훔쳐보다"

- `~세요`(놓아 보세요·눈여겨보세요)도 해요체 — "~겠습니다", "눈여겨볼 곳은 ~입니다"로
- 문단을 완성할 때마다 소리 내 읽고, **존재하지 않는 단어(모델 신조어·오타)가 없는지 명사 위주로 재확인**

## 깊이 = 자립성 (심화 글 필수)

- **축약 금지**: 원본의 수치·자료구조·기전을 그대로 보존. 밀도 ≠ 축약, 깊되 풀어서
- **용어 첫 등장 시 정의부터** (예: "MSR은 OS가 CPU 동작을 설정하는 특수 레지스터 모음"). 그 문단만 읽어도 검색 없이 이해되게
- 원본의 학습 목표 목록·"이 문서의 성격" 류 메타 블록은 서두 산문에 녹이고 섹션으로 남기지 않음
- 원본의 고정 구조(8단 등)는 글 흐름에 맞게 재편하되, 시리즈 척추 섹션(예: 3언어 수렴 비교)과 심화 질문 섹션은 유지
- 버전 민감 사실은 시리즈 plan의 기준 버전 표와 대조, 필요 시 웹 검색으로 확인

## 섹션 구조

**트러블슈팅(logs)** — 기본 4섹션 (이모지 없이):

```markdown
## 한 줄 요약
> 무엇을 발견했고 어떻게 해결했는지 1~2문장

## 문제: <증상 한 줄>
## 원인: <진짜 원인 한 줄>
## 해결: <어떻게 풀었는지>
## 배운 점
```

**ADR(essays)**: `## 배경 → ## 선택지 → ## 결정 → ## 근거`

**시리즈 심화 글(essays)**: 고정 템플릿 없음. 서사 흐름에 맞게 구성하되
- 글 시작: 시리즈 소개 블록쿼트 ("시리즈 X의 N편" + 이전 편 연결 + 이번 편 후킹)
- 글 끝: "핵심 요약" bullet + "[다음 편 예고]"

## 선택지 비교 섹션 (조건부)

원본에 다음 **신호** 있을 때만 추가:
- "Option A/B/C", "안1/안2", "~를 고려했으나"
- 도구·아키텍처·기술 스택 선정이 본문 주제

신호 없으면 추가 금지 — **원본에 없는 대안을 만들어내는 건 사실 훼손**입니다. 상세 규칙은 `./decision-tradeoff.md`.

## 표·다이어그램 설명 분량

표/다이어그램 뒤에는 **반드시 상세 설명**. 단순 요약 금지.

| 유형 | 분량 |
|---|---|
| 핵심 개념 비교표 | 3-4 문단 |
| 아키텍처 다이어그램 | 4-6 문단 + 역할 목록 |
| 트래픽 흐름 다이어그램 | 단계별 번호 목록 + 1-2 문단 |
| 기능 지원 현황표 | 지원/미지원 분류 + 2-3 문단 |
| 요약 정리표 | 2-3 문단 |

**설명 7원칙**: ① 문단 분리 ② 구체적 수치·시나리오 ③ 단계별 번호 ④ "왜"를 설명 ⑤ 도입부+마무리 ⑥ 목록 구조 ⑦ 지원·미지원 분류

추상적 설명 대신 구체값. "업그레이드 문제가 있다"가 아니라 "1000개 Pod에서 Rolling Restart 시 수 시간 걸린다".

## 이미지·다이어그램

- 사이즈 힌트(alt 텍스트): `|short`(600px) / 기본(800px) / `|tall`(1100px) / `|xtall`(1600px) / `|auto`
- SVG 직접 작성이 기본: `public/diagrams/{slug}-{n}.svg`, 스타일은 `.claude/plans/diagram-conversion/best-practices.md` 준수. **라벨은 영문**, 상세 설명은 본문 한글로
- ASCII 박스/다단계 흐름 금지 — CLAUDE.md "ASCII 박스/다이어그램 정책" 참고. 평탄화 우선순위: 표 형태 → markdown 표 / 공간·구조·흐름 → SVG / 1~2줄 흐름 → 인라인
- **능동 시각화 판단 (사용자 확정 규칙)**: 원본에 그림이 없어도 섹션마다 필요성을 판단해 SVG를 신규 작성한다. 기준 — 그림 = 공간·구조·대응·상태 전이·시간 축 개념 / 표 = 수치·나열 비교 / 산문 = 서사·인과. 변환 마지막에 **전 섹션 재감사**를 한 번 더 돈다
- SVG가 대신하게 된 표/ASCII는 삭제(중복 금지). SVG 번호는 문서 등장 순서와 일치

## 변환 절차

1. `drafts/<name>.md` 정독
2. 시리즈 확인 + 해당 plan 문서 통독 (0단계)
3. 트랙·카테고리 결정
4. 같은 시리즈의 기존 글 1편 참조 (톤·구조)
5. front matter 작성 + 섹션 구성 + 문체 규칙 적용 (격식체·마침표 최소화·짧은 단락·이모지 금지)
6. 다이어그램: ASCII 평탄화 + 능동 시각화 판단 + SVG 작성 → 전 섹션 재감사
7. 코드블록 언어 명시 + 이미지 힌트 + 표·다이어그램 뒤 분량 가이드 적용
8. `src/content/{track}/{category}/<slug>.md`에 Write
9. **`npm run lint:post -- <파일>` 실행 — error 0이어야 발행** (직역투 렉시콘·해요체·마침표·
   박스문자·펜스 lang·front matter·이미지 실존·SVG 폰트를 0토큰으로 검사. blog-reviewer로
   같은 항목을 재검사하지 않는다 — 리뷰어는 부(배치) 단위 1회, 판단 항목만)
10. 시리즈 plan의 상태 표 갱신. drafts 원본은 손대지 않음 (참조용 보존)

## Gotchas

- **디렉토리 = SSOT, frontmatter `type` 명시 불필요**: 트랙 격리는 디렉토리만 본다
- **시리즈 분리 위험**: 한 시리즈는 한 트랙으로 통일
- **자산 경로 변경 금지**: 본문의 `![](/diagrams/foo-1.svg)`·`/images/...`는 평탄 그대로
- **slug 충돌**: 파일명(slug)은 전역 유일
- **drafts 원본 손대지 않음**: 변환 후에도 보존
- **마침표 습관 주의**: 산문 훈련된 문장은 마침표가 붙기 쉽다. 탈고 때 단락 끝 마침표를 일괄 점검하고, 긴 문단은 단락 분리로 해소

## 안 되는 것

- 원본에 없는 사실(성능 수치·실무 사례·추가 맥락) 지어내기
- 실명·민감 고유명사 노출 (이니셜·역할명으로 익명화)
- 이모지 (헤더·장식 모두)
- 긴 서론·결론
