## 🔍 순수 대기열 성능 비교

### API 레이턴시

1000 VU와 3000 VU에서 모든 대기열 API 레이턴시를 비교했습니다.

**1000 VU:**

| API (p95) | 수연 | 준상 | 성전 | 최고 |
|-----------|------|------|------|------|
| queue_status | 2.18s | 361ms | **279ms** | 성전 |
| queue_seat_enter | - | 452ms | **295ms** | 성전 |
| http_req 전체 | 2.18s | 390ms | **282ms** | 성전 |

**3000 VU:**

| API (p95) | 수연 | 준상 | 성전 | 최고 |
|-----------|------|------|------|------|
| queue_status | 2.13s | 3.10s | **500ms** | 성전 |
| queue_seat_enter | 2.29s | 2.99s | **562ms** | 성전 |
| http_req 전체 | 2.13s | 3.10s | **501ms** | 성전 |

성전이 **모든 API 레이턴시에서 압도적 1위**예요.
3000 VU에서 status p95가 500ms — 준상(3.1s)의 1/6, 수연(2.13s)의 1/4 수준입니다.

ZSET의 O(log N) 연산과 이벤트 기반 승격이 서버 부하를 최소화하는 구조가 레이턴시에 직접적으로 반영된 결과예요.

### 대기열 통과율

| VU | 수연 | 준상 | 성전 |
|----|------|------|------|
| 300 | - | 99.26% | **100%** |
| 1000 | 81.56% | **100%** | **100%** |
| 3000 | 57.90% | **100%** | **100%** |

준상과 성전은 **전 구간 100% 통과**를 유지합니다.
수연은 3000 VU에서 57.90%로 하락해요. 빠른 슬롯 회전이 동시 좌석 경합을 유발하면서 예매 실패 → 재진입 비율이 높아지는 구조적 특성입니다.

### 대기 시간: 중간값 vs 꼬리

대기 시간은 p50(중간값)과 p95(꼬리)를 분리해서 봐야 해요.

**1000 VU:**

| 지표 | 수연 | 준상 | 성전 |
|------|------|------|------|
| queue_wait p50 | 56.3s | 66.4s | **11.7s** |
| queue_wait p95 | **1m17s** | 2m16s | 2m54s |

**3000 VU:**

| 지표 | 수연 | 준상 | 성전 |
|------|------|------|------|
| queue_wait p50 | 2m19s | 53s | **23s** |
| queue_wait p95 | 2m54s | **2m31s** | 4m48s |

흥미로운 패턴이 보여요.

**p50(대부분 유저)은 성전이 압도적**입니다. 3000 VU에서도 23초.
하지만 **p95(꼬리)는 성전이 가장 길어요.** 3000 VU에서 4분 48초.

이것은 이벤트 기반 승격의 특성이에요. 초반에 빠르게 통과하는 유저가 많지만, 대기열이 길어지면 이벤트 발생 간격이 불규칙해지면서 뒤쪽 유저의 대기가 길어집니다.

### 슬롯 회전 속도 (처리량)

| VU | 수연 | 준상 | 성전 |
|----|------|------|------|
| 300 iter | - | **4,476** | 457 |
| 1000 iter | **5,495** | 3,713 | 417 |
| 3000 iter | **6,574** | 763 | 290 |

처리량은 수연이 압도적이에요.
3000 VU에서 수연(6,574)은 준상(763)의 **8.6배**, 성전(290)의 **22.7배** 처리합니다.

## 🔍 서버 부하 비교

성능만큼 중요한 것이 **서버가 얼마나 부담을 받는가**예요.

### Redis 연산 추정

| 구현체 | polling 간격 | heartbeat | VU당 Redis ops/sec | 3000 VU 합계 |
|--------|-------------|-----------|-------------------|-------------|
| 수연 | 2초 | 없음 | 0.5 | ~1,500 ops/s |
| 준상 | 1초 | 없음 | 1.0 | ~3,000 ops/s |
| 성전 | 5초 | 10초 | 0.3 | ~900 ops/s |

성전은 polling 5초 + heartbeat 10초로 **VU당 Redis 부하가 가장 낮아요**.
준상은 1초 polling으로 3000 VU에서 Redis에 3,000 ops/s가 들어갑니다.

### 총 HTTP 요청

| 구현체 | 3000 VU 총 요청 | recv | sent |
|--------|-----------------|------|------|
| 수연 | 445,086 | 453MB | 32MB |
| 준상 | 329,262 | 204MB | 22MB |
| 성전 | **309,449** | **172MB** | 25MB |

성전이 서버 요청 수와 네트워크 트래픽 모두 **가장 적습니다**.
수연은 빠른 회전으로 총 요청이 가장 많아요.

### 스케일링 내성

VU가 10배 증가할 때 레이턴시가 얼마나 올라가는지를 보면:

| 구현체 | status p95 증가율 (300→3000 VU) | 특성 |
|--------|-------------------------------|------|
| 성전 | 281ms → 500ms (**1.8x**) | 거의 선형 |
| 수연 | - → 2.13s (기준 없음) | pod 한계에서 안정 |
| 준상 | 404ms → 3.10s (**7.7x**) | 비선형 포화 |

성전은 VU가 10배 늘어도 레이턴시가 1.8배만 증가해요.
준상은 7.7배 — 3000 VU에서 서버가 포화 상태에 진입합니다.

## 📊 승격 메커니즘 비교

세 구현체의 내부 동작을 나란히 놓으면:

| 항목 | 수연 | 준상 | 성전 |
|------|------|------|------|
| **자료구조** | Hash + Lua script | Hash (O(1) 카운터) | ZSET + HASH |
| **승격 트리거** | status 호출 시 동적 계산 | Scheduler 1초 주기 | 이벤트 기반 (leave/expire) |
| **Heartbeat** | 없음 | 없음 | 있음 (30s TTL, 10s 갱신) |
| **만료 처리** | Scheduler 30초 주기 | Redis TTL 자동 | Key Expiration Event |
| **토큰** | JWE (AES-256-GCM) | JWT | Custom AES (@RequestParam) |
| **분산 락** | Lua script (원자적) | 없음 | Redisson |
| **Redis 키 수/VU** | ~2개 | ~2개 | ~5개 |

수연의 핵심은 **"status 호출 = 승격 판단"**이에요.
별도 Scheduler 없이 매 polling에서 publishedRank를 동적으로 계산해서 즉시 승격합니다.
이게 슬롯 회전이 빠른 이유예요.

성전은 **이벤트 기반**이라 서버 부하는 적지만 승격 타이밍이 이벤트 발생에 의존합니다.
leave나 expire가 없으면 승격이 일어나지 않아서 슬롯 회전이 느려요.

## 📊 종합 순위

9개 카테고리로 최종 정리했습니다.

| 카테고리 | 1위 | 2위 | 3위 |
|----------|-----|-----|-----|
| API 레이턴시 | **성전** | 준상 | 수연 |
| 대기열 통과율 | **준상/성전** (100%) | - | 수연 |
| 대기 시간 (중간값) | **성전** | 준상 | 수연 |
| 대기 시간 (꼬리) | **수연** | 준상 | 성전 |
| 슬롯 회전 (처리량) | **수연** | 준상 | 성전 |
| 서버 부하 (Redis) | **성전** | 수연 | 준상 |
| 스케일링 내성 | **성전** | 수연 | 준상 |
| 안정성 (에러율) | **성전** (0%) | 준상 | 수연 |
| 코드 안정성 (버그) | **성전** (0건) | 준상 (2건) | 수연 (3건) |

성전이 **9개 중 6개에서 1위**입니다.
순수 대기열 성능만 놓고 보면 성전이 가장 뛰어나요.

하지만 처리량 — 실제로 몇 명이 예매를 완료하느냐 — 에서는 성전이 최하위입니다.
3000 VU에서 290 iterations vs 수연 6,574.

이 트레이드오프를 깨는 변수가 등장합니다.

## 🏆 최종 선정: 수연 + CDN 캐싱

### EKS Prod 3000 VU 테스트

Kind 클러스터에서의 비교를 마친 후, **EKS Prod 환경**에서 최종 검증을 진행했습니다.

**ALB 직접 (CDN 미적용):**

| 항목 | 수연 | 준상 | 성전 |
|------|------|------|------|
| queue_pass_rate | 87.44% | **100%** | **100%** |
| queue_timeouts | 157건 | **0건** | **0건** |
| queue_status avg | 926ms | 2.66s | **158ms** |
| queue_status p95 | 1.98s | 7.48s | **719ms** |
| 처리량 (req/s) | 655 | 500 | 602 |
| 1K→3K status 증가율 | 16x | 24x | **4x** |

EKS Prod에서도 Kind와 같은 패턴이 확인됩니다.
성전이 레이턴시와 안정성에서 최고, 준상이 3000 VU에서 포화(status avg 2.66s).

### CDN 캐싱의 결정적 차별점

수연 구현체에 Cloudflare CDN 캐싱을 적용한 결과:

| 항목 | ALB 직접 | CDN 캐싱 | 변화 |
|------|---------|---------|------|
| **queue_status avg** | 926ms | **220ms** | **4.2x 개선** |
| **queue_status p95** | 1.98s | **334ms** | **5.9x 개선** |
| **처리량 (req/s)** | 655 | **1,044** | **1.6x 증가** |
| **처리량 (iterations/5min)** | 907 | **1,889** | **2.1x 증가** |
| **http_req_failed** | 1.26% | **0.08%** | **16x 감소** |

이 숫자들이 말하는 바를 생각해봅시다.

CDN 캐싱을 적용하면 status API의 응답 시간이 926ms에서 **220ms**로 떨어집니다.
이 220ms는 성전의 ALB 직접 결과(158ms)에 근접하는 수치예요.

하지만 핵심 차이가 있어요. 성전의 158ms는 **origin 서버가 직접 응답**한 겁니다.
수연의 220ms는 **Cloudflare 엣지가 캐시에서 응답**한 거예요.

VU가 3000에서 10,000, 100,000으로 올라가면 어떻게 될까요?

```
{/* TODO: Draw.io로 교체 */}

                    성전 (ALB 직접)
┌──────────┐     ┌──────────────┐     ┌───────┐
│ 3000 VU  │────▶│  ALB         │────▶│ Redis │
│ polling  │     │  → Pod       │     │ ZSET  │
│ 5초 간격 │     │  → Redis 조회│     │ 900   │
└──────────┘     └──────────────┘     │ ops/s │
                                      └───────┘
  VU ↑ → Redis 부하 ↑ (선형 증가)


                    수연 (CDN 캐싱)
┌──────────┐     ┌──────────────┐     ┌──────────────┐     ┌───────┐
│ 3000 VU  │────▶│  Cloudflare  │──×──│  ALB         │     │ Redis │
│ polling  │     │  Edge        │     │  → Pod       │     │       │
│ 2초 간격 │     │  캐시 HIT    │     │  (요청 소수) │     │ 부하  │
└──────────┘     │  max-age=1   │     └──────────────┘     │ 일정  │
                 └──────────────┘                           └───────┘
  VU ↑ → CDN 캐시 히트율 ↑ → origin 부하 거의 일정
```

**polling 요청을 엣지에서 흡수**하면 origin 부하는 VU 수와 거의 무관하게 일정합니다.
CDN `max-age=1`이라서 대기열 상태는 최대 1초 지연되지만, 사용자 체감에는 미미한 수준이에요.

### 선택 근거

1. **CDN 캐싱이 결정적 차별점**: polling 요청을 엣지에서 흡수 → status 926ms → 220ms
2. **확장성**: VU 증가 시 CDN 캐시 히트율이 높아져 origin 부하는 거의 일정
3. **처리량 최고**: CDN 경유 1,044 req/s, 1,889 iterations/5min
4. **에러율 최저**: http_req_failed 0.08% (ALB 직접 대비 16배 개선)
5. **timeout 이슈는 해결 가능**: Cloudflare Free plan rate limit으로 signup이 실패한 것 — signup을 ALB 직접으로 분리하면 해결

### 트레이드오프

솔직히 순수 성능만 보면 성전이 더 안정적이에요. Timeout 0건, 에러 0건.

하지만 성전은 CDN 캐싱이 불가능한 구조예요.
5초 polling + heartbeat 방식이라 응답이 요청 시점에 종속되어 캐시 히트율이 낮습니다.
VU가 늘어나면 결국 origin Redis가 병목이 됩니다.

수연은 CDN을 통해 **origin 부하를 VU 수와 분리**할 수 있어요.
3000 VU든 30,000 VU든 엣지가 흡수하면 origin은 비슷한 부하만 받습니다.

이것이 최종 선정의 결정적 이유다.

## 📚 핵심 포인트

**첫째, 순수 성능만으로는 최선을 판단할 수 없다.**
성전이 9개 카테고리 중 6개에서 1위였지만, CDN이라는 변수 하나가 판도를 뒤집었습니다.
아키텍처 결정에서는 "현재 성능"보다 **"확장 가능한 구조"**가 더 중요해요.

**둘째, polling 요청을 엣지에서 흡수하면 origin 부하는 VU와 무관하게 거의 일정하다.**
대기열 상태 `max-age=1`은 사용자 체감에 미미한 지연이지만, 서버 부하 측면에서는 엄청난 차이를 만듭니다.

**셋째, CDN 캐싱 가능 여부는 설계 단계에서 결정된다.**
준상(Redis 직접 조회)과 성전(heartbeat)은 구조적으로 CDN 캐싱이 어렵습니다.
수연의 JWE 토큰 방식은 상태를 토큰에 담아 stateless하게 만들기 때문에 CDN 캐싱이 자연스럽게 가능했어요.
