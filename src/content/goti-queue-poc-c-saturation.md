## 한 줄 요약

> CDN POC 대기열(POC C) 구현체를 K6로 300 / 1000 / 3000 VU 단계별 부하를 걸었습니다. iterations/s가 12~13 TPS에서 평탄해지면서 단일 pod 처리 한계가 드러났고, queue_enter 응답 지연이 분산락 경합으로 인해 9배 이상 급등하는 것을 확인했습니다.

---

## 테스트 목적

CDN POC 대기열 구현체의 처리 한계(saturation point)를 정량적으로 파악하는 것이 목적입니다.

단일 pod 환경에서 동시 사용자 수 증가에 따른 성능 변화를 측정하고, 대기열 순환(leave → 승격) 메커니즘이 고부하 상황에서도 정상 동작하는지 검증합니다. 또한 스케일아웃 필요 시점을 판단할 수 있는 기준 데이터를 확보합니다.

SLO 기준은 다음과 같이 정의했습니다.

- `http_req_failed` < 1%
- `http_req_duration` p(95) < 2,000ms
- `queue_pass_rate` > 80%
- `goti_ticket_success_rate` > 80%

## 테스트 환경

### 인프라 구성

| 구성요소 | 스펙 | 비고 |
|----------|------|------|
| Kind 클러스터 | 5노드 (1 CP + 4 worker) | Ubuntu 32GB RAM, 12 CPU |
| goti-queue (POC C) | 1 pod (replica=1) | CPU 100m-500m, Memory 512Mi-1Gi |
| goti-ticketing | 6 pods | 좌석 선택/결제 처리 |
| Redis | 호스트 PC, DB 0 | 분산락 + 대기열 상태 저장 |
| maxCapacity | 100 | 동시 처리 가능 인원 |
| 네트워크 경로 | Mac → Cloudflare CDN → Istio Gateway → Queue Service | 외부 네트워크 지연 포함 |

단일 pod 구성으로 의도적으로 제한했습니다. 스케일아웃 이전에 pod 하나의 처리 한계를 먼저 파악해야, 이후 pod 수를 늘렸을 때 선형 증가하는지 또는 다른 병목이 나타나는지 비교할 수 있기 때문입니다.

maxCapacity=100은 동시에 "대기열을 통과한" 사용자 수의 상한입니다. 이 값이 대기 시간의 길이를 결정하지만, 처리 능력 자체와는 다른 지표입니다. 이 구분은 뒤의 "성능 지표 해석" 절에서 자세히 다룹니다.

### 애플리케이션 설정

- Java OTel Agent 2.25.0
- Istio sidecar injection 활성화
- Cloudflare CDN/Proxy 경유 (Mac 로컬 → 외부 네트워크)

### 테스트 도구

- K6 (Mac 로컬 실행)
- 시나리오: 대기열 진입 → 폴링 대기 → 좌석 선택 → 결제 E2E 흐름

## 테스트 시나리오

| # | 시나리오 | 설명 | VU | 비고 |
|---|----------|------|----|------|
| 1 | Saturation 300 VU | 안정 동작 확인 | 300 | 전체 threshold 통과 |
| 2 | Saturation 1000 VU | 처리 한계 탐색 | 1000 | threshold 실패 |
| 3 | Saturation 3000 VU | 한계 확인 | 3000 | threshold 실패, 포화 확정 |

3단계로 나눈 이유는 saturation point를 찾기 위해서입니다. 300 VU에서 통과 → 1000 VU에서 실패 → 3000 VU에서도 실패가 관측되면, 단일 pod의 처리 한계는 300~1000 VU 사이 어딘가에 있음을 의미합니다.

### 시나리오별 요청 흐름

```
1. POST /queue/enter          — 대기열 진입
2. GET  /queue/status (poll)  — 대기 상태 폴링 (통과까지 반복)
3. POST /seat/select          — 좌석 선택
4. POST /payment              — 결제
```

하나의 VU가 위 4단계를 완주하면 iteration 1건으로 카운트됩니다. 따라서 `iterations/s`는 pod이 초당 몇 명의 E2E 티켓팅을 완료시키는지를 의미하는 핵심 지표입니다.

## 테스트 결과

### 종합 요약

| 지표 | 300 VU | 1000 VU | 3000 VU | SLO 기준 |
|------|--------|---------|---------|----------|
| exit code | 0 | 99 | 99 | 0 |
| queue_pass_rate | 89.79% | 84.3% | 82.8% | > 80% |
| goti_ticket_success_rate | 89.77% | 84.0% | 82.5% | > 80% |
| http_req_failed | 0.64% | 0.45% | 0.2% | < 1% |
| http_req_duration p(95) | 402ms | 1,808ms | 1,209ms | < 2,000ms |
| http_req_duration p(99) | 556ms | 미측정 | 미측정 | - |
| iterations | 4,764 | 4,477 | 4,577 | - |
| iterations/s | **13.4/s** | **11.9/s** | **12.1/s** | - |
| http_reqs | 75,931 | 158,908 | 418,453 | - |
| http_reqs/s | 214/s | 421/s | 1,109/s | - |

표에서 가장 주목해야 할 줄은 `iterations/s`입니다. VU를 300에서 1000으로 3.3배 늘렸지만 처리량은 13.4에서 11.9로 오히려 감소했습니다. VU를 3000까지 10배 늘려도 12.1에 머물렀습니다.

이것이 전형적인 saturation 패턴입니다. 병목 지점 이전이라면 VU 증가에 비례해 iterations/s가 증가해야 합니다. 평탄해진다는 것은 pod이 더 이상 일을 빠르게 처리할 수 없다는 신호입니다.

반면 `http_reqs/s`는 214 → 421 → 1,109로 계속 증가했습니다. 이는 대기 시간이 길어지면서 폴링 요청 수가 누적되기 때문입니다. 폴링이 실제 티켓팅 처리량을 의미하지는 않으므로, 겉보기 처리량과 실제 처리량을 혼동하지 않아야 합니다.

### 대기열 메트릭 상세

| 지표 | 300 VU | 1000 VU | 3000 VU | 비고 |
|------|--------|---------|---------|------|
| queue_enter_ms p(95) | 532ms | 4,847ms | 2,754ms | 분산락 경합 지표 |
| queue_enter_ms avg | 미측정 | 1,448ms | 1,136ms | |
| queue_seat_enter_ms p(95) | 511ms | 미측정 | 1,320ms | |
| queue_wait_duration avg | 미측정 | 67.4s | 158.2s | maxCapacity 설계에 의한 대기 (성능 지표 아님) |
| queue_wait_duration p(95) | 19.0s | 88.3s | 237.7s | |
| queue_poll_count avg | 5.75 | 미측정 | 54 | |
| queue_immediate_pass_rate | 2.05% | 미측정 | 1.3% | |
| queue_e2e_duration avg | 19.2s | 미측정 | 162.7s | 대기 시간 포함 전체 E2E |

`queue_enter_ms` p(95) 지표가 300 VU에서 532ms였는데 1000 VU에서 4,847ms로 약 9.1배 급등했습니다. 이 지표가 분산락 경합의 직접적 증거입니다.

반면 `queue_wait_duration`은 VU가 늘면 거의 비례해서 증가합니다. 3000 VU에서는 평균 158.2초가 나왔습니다. 이 수치는 무서워 보이지만 "병목"이 아닙니다. maxCapacity=100으로 고정된 상태에서 3000명이 줄을 서면, 2900명이 100명씩 회전하며 입장하므로 대기 시간이 길어지는 것은 설계 의도대로의 동작입니다.

### 처리량 비교 (핵심)

| 지표 | 300 VU | 1000 VU | 3000 VU | 분석 |
|------|--------|---------|---------|------|
| iterations/s | **13.4/s** | **11.9/s** | **12.1/s** | **완전 포화 — VU 10배 올려도 처리량 동일** |
| http_reqs/s | 214/s | 421/s | 1,109/s | 폴링 횟수 증가(대기 시간 증가)에 의한 req 수 증가 |

이 표가 테스트의 결론에 가까운 데이터입니다. VU를 10배 올려도 iterations/s가 12~13 수준에서 평탄하다는 것은 단일 pod이 초당 12~13건의 E2E 티켓팅을 완료시키는 것이 한계라는 의미입니다.

### 에러 분석

| 에러 유형 | 300 VU | 1000 VU | 원인 |
|-----------|--------|---------|------|
| http_req_failed | 0.64% | 0.45% | 타임아웃/5xx (정상 범위) |
| queue 미통과 (leave) | 10.21% | 15.7% | maxCapacity 제한에 의한 정상 탈락 |
| 티켓 실패 | 10.23% | 16.0% | 대기열 미통과에 비례 |

에러율 자체는 1000 VU에서도 0.45%로 SLO(1% 이하) 이내입니다. 1000 VU에서 threshold가 실패한 이유는 5xx 비율 때문이 아니라 `http_req_duration` p(95)가 1,808ms로 2,000ms SLO에 근접했고, `queue_enter_ms`가 급등했기 때문으로 판단합니다.

## 성능 지표 해석 가이드

대기열 부하테스트에서는 "무엇을 보고 판단하는가"가 일반적인 API 부하테스트와 다릅니다. 대기 시간이 길다고 무조건 나쁜 것이 아니기 때문입니다.

| 관점 | 메트릭 | 의미 | 이 테스트 결과 |
|------|--------|------|---------------|
| **처리 능력** | `iterations/s` | pod이 초당 몇 건의 E2E를 완료하는가 | 12~13 TPS (pod 1개 한계) |
| **응답 속도** | `queue_enter p95` | 진입 API 95th percentile 응답시간 | 300VU=532ms, 3000VU=2.7s |
| **안정성** | `http_req_failed` | 과부하에서 서버가 5xx를 뱉는가 | 0.2~0.6% (안정) |
| **대기열 효율** | `queue_pass_rate` | 줄 선 사람이 타임아웃 없이 입장하는가 | 82~90% |

### `queue_wait_duration`은 성능 지표가 아닙니다

대기 시간은 `maxCapacity`(동시 수용 인원)와 `VU`(접속자 수)의 비율에 의해 결정됩니다. 동일한 서버에서도 이 두 값을 어떻게 조합하느냐에 따라 대기 시간이 완전히 달라집니다.

- `maxCapacity=100`, `VU=3000`: 대기 158초는 정상입니다. 2900명이 100명씩 순환하므로 필연적으로 대기가 발생합니다.
- `maxCapacity=5000`, `VU=3000`: 대기가 거의 없습니다. 전원이 즉시 입장하기 때문입니다.

따라서 대기 시간이 늘어난다고 해서 pod 성능이 나쁘다고 판단하면 안 됩니다. 이 지표는 비즈니스 요구(동시 수용 인원)와 부하(접속자 수)의 함수일 뿐입니다.

### 핵심 판단 기준

`iterations/s`가 VU를 올려도 증가하지 않으면 pod 처리 한계에 도달한 것입니다. 이때가 스케일아웃이 필요한 시점입니다. 이번 테스트에서는 300 VU에서 이미 한계에 근접했고, 1000 VU와 3000 VU에서는 명확히 평탄해졌습니다.

## 병목 지점 분석

### 단일 pod 처리 한계 도달

`iterations/s`가 300 VU(13.4/s)와 1000 VU(11.9/s), 3000 VU(12.1/s)에서 거의 동일합니다. 전형적인 saturation 징후입니다.

VU를 3.3배 늘려도 처리량이 증가하지 않고 오히려 11.2% 감소했습니다. 이는 경합 증가로 인한 약간의 역효과로 해석됩니다. 병목은 queue pod 하나의 처리 능력, 구체적으로는 CPU 500m 상한에 있을 가능성이 높습니다.

### 분산락 경합

`queue_enter_ms` p(95) 지표가 532ms에서 4,847ms로 9.1배 급등했습니다. Redis 분산락 획득 대기가 동시 요청 증가에 비례하여 증가한 결과입니다.

단일 pod 내에서 분산락으로 직렬화된 구간이 throughput ceiling을 결정합니다. 분산락 없이 처리될 수 있는 부분은 병렬화되지만, 락을 획득해야만 진행되는 구간은 CPU 경합과 별개로 순차 처리됩니다.

### 대기 시간 선형 증가

`queue_wait_duration` p(95) 지표가 19.0초에서 88.3초로 4.6배 증가했습니다. maxCapacity=100이 고정된 상태에서 VU가 증가하면 대기열 길이가 늘어나고, 그만큼 폴링 횟수와 대기 시간이 비례 증가합니다.

이는 대기열 순환 메커니즘이 정상 동작함을 의미합니다. 병목이 아니라 설계 의도입니다.

### 네트워크 경로 오버헤드

테스트는 Mac → Cloudflare → Kind PC 경로를 거쳤습니다. 이 외부 네트워크 왕복 지연이 `http_req_duration`에 포함됩니다. 순수 서버 성능을 측정하려면 Kind PC 로컬에서 K6를 실행해 네트워크 지연을 제거한 테스트를 추가로 진행해야 합니다.

## 개선 방안

| # | 개선 항목 | 예상 효과 | 우선순위 |
|---|----------|----------|----------|
| 1 | queue pod replica 2~3개 스케일아웃 | iterations/s 선형 증가, enter 지연 분산 | P0 |
| 2 | Kind PC 로컬 K6 실행 (네트워크 지연 제거) | 순수 서버 성능 기준선 확보 | P1 |
| 3 | POC A, POC B 동일 조건 비교 테스트 | A/B/C 구현체 성능 비교 데이터 | P1 |
| 4 | maxCapacity 튜닝 (100 → 150/200) | 대기열 회전율 향상, 대기 시간 감소 | P2 |
| 5 | 분산락 최적화 (TTL/재시도 간격 조정) | enter 지연 감소 | P2 |

P0 항목인 스케일아웃이 가장 먼저입니다. 단일 pod 한계가 명확히 측정됐으므로 replica를 늘려서 선형 증가하는지 확인해야 합니다. 만약 선형 증가하지 않는다면 Redis 분산락이나 DB가 다음 병목입니다.

P1 항목 중 네트워크 지연 제거는 "순수 서버 성능"의 기준선을 잡기 위해 필요합니다. 현재 수치에는 Cloudflare 왕복이 섞여 있어 서버 자체의 응답 특성을 분리하기 어렵습니다.

POC A/POC B 동일 조건 비교는 3개 구현체의 성능 특성을 객관적으로 비교하기 위한 다음 단계입니다. 이 테스트는 다음 편 글에서 다룹니다.

## 기준선

최초 saturation 테스트이므로 비교 대상이 없습니다. 이 결과를 향후 비교를 위한 기준선(baseline)으로 사용합니다.

| 기준선 (300 VU, 단일 pod) | 값 |
|---------------------------|-----|
| iterations/s | 13.4/s |
| http_req_duration p(95) | 402ms |
| queue_enter_ms p(95) | 532ms |
| queue_pass_rate | 89.79% |

향후 스케일아웃 테스트나 분산락 최적화 테스트에서 이 기준선을 대조해 개선 효과를 측정합니다.

## 📚 배운 점

### Saturation point는 iterations/s의 평탄화로 판별합니다

`http_reqs/s`는 VU가 늘어나면 계속 증가합니다. 폴링 요청까지 포함되기 때문입니다. 대기열 시스템에서는 겉보기 요청 수와 실제 티켓팅 처리량을 구분해야 합니다.

`iterations/s`가 평탄해지는 지점이 진짜 처리 한계입니다. 이번 테스트에서는 300 VU에서 이미 한계에 근접했다는 사실을 1000 VU와 3000 VU에서의 평탄화로 확인했습니다.

### 대기 시간과 처리 능력은 다른 지표입니다

`queue_wait_duration`이 158초라고 해서 시스템이 느린 것이 아닙니다. 이 값은 `maxCapacity`와 `VU`의 비율로 결정되는 설계 변수입니다. 성능 지표로 오인하면 잘못된 최적화에 시간을 씁니다.

성능 지표는 `iterations/s`(처리 능력), `queue_enter_ms`(응답 속도), `http_req_failed`(안정성)입니다. 대기 시간은 비즈니스 정책의 결과값입니다.

### 분산락 경합은 p(95) 급등으로 드러납니다

`queue_enter_ms` p(95)가 532ms에서 4,847ms로 9.1배 급등한 것이 분산락 경합의 가장 명확한 신호입니다. CPU 사용률이 100%가 아니어도 락 대기가 직렬화 구간을 만들어 응답 지연을 폭증시킵니다.

이런 패턴이 보이면 락 TTL, 재시도 간격, 락 획득 경합 횟수 메트릭을 함께 봐야 합니다. 단순히 pod 수를 늘리는 것만으로는 해결되지 않을 수 있습니다.

### 네트워크 경로는 기준선 수치에 포함됩니다

Mac → Cloudflare → Kind PC 경로로 측정한 수치는 순수 서버 성능이 아닙니다. 외부 네트워크 왕복이 섞여 있습니다.

프로덕션 SLO를 정의할 때는 사용자 관점의 end-to-end 지연이 맞지만, 병목 분석을 위해서는 네트워크를 제거한 로컬 테스트도 함께 진행해야 합니다. 기준선 데이터는 테스트 환경의 전제를 명시해 두어야 이후 비교가 유효합니다.
