## 런타임이 얇다와 기계어가 좋다는 다른 말입니다

1부에서 Rust를 제로코스트 극단에 놓았습니다. 런타임 층이 사실상 비어 있다는 뜻이었죠

그런데 런타임이 얇다는 것과 실행되는 기계어가 좋다는 것은 별개입니다. 런타임을 다 걷어내도 컴파일러가 형편없는 기계어를 뽑으면 프로그램은 느립니다

그래서 2부의 관심사는 하나입니다. **누가, 언제, 어떤 정보로 기계어를 만드는가**

Rust의 답은 순수한 AOT(Ahead-Of-Time), 즉 실행 전 사전 컴파일입니다. `cargo build --release`가 끝나는 순간 CPU가 실행할 모든 기계어는 이미 실행 파일 안에 확정돼 있습니다

실행 중에 새 명령이 생기는 일은 없습니다. 이 불변성이 이번 편 전체의 주제이고, 6편에서 볼 JVM 코드 캐시의 "실행마다 달라지는 기계어"와 정확히 대척점에 섭니다

## rustc의 파이프라인 — 컴파일러는 사실 둘입니다

소스 코드가 기계어가 되기까지 네 개의 중간 표현을 지납니다. IR(Intermediate Representation, 중간 표현)이란 컴파일러가 내부적으로 코드를 담아 두는 자료구조를 말합니다

![The rustc pipeline — safety decided in MIR, performance decided in LLVM](/diagrams/rust-aot-zero-cost-codegen-1.svg)

그림을 위에서부터 따라가겠습니다

**HIR(High-level IR)** — 문법에 가까운 고수준 표현입니다. 파싱과 타입 추론, 트레이트 해석이 여기서 끝납니다

여기서 MIR로 내려가며 디슈가링(desugaring)이 일어납니다. 문법 설탕을 벗겨 단순한 형태로 되돌리는 작업인데, `for`는 `loop`으로, `?` 연산자는 `match`로, 클로저는 구조체와 호출 메서드로 바뀝니다

**MIR(Mid-level IR)** — rustc 고유의 표현이자 이 파이프라인의 심장입니다. 코드를 기본 블록(basic block)의 제어 흐름 그래프로 표현하는데, 기본 블록이란 분기 없이 위에서 아래로 쭉 실행되는 명령 묶음입니다

MIR에서 세 가지 일이 벌어집니다

- **borrow check** — 소유권과 참조 수명이 안전한지 판정하는 검사입니다. MIR 위에서 도는 데이터플로 분석(NLL, Non-Lexical Lifetimes)으로 수행되고, 통과하면 기계어에 흔적을 하나도 남기지 않습니다
- **drop 자동 삽입** — 각 값이 소멸해야 할 지점을 찾아 해제 코드를 기본 블록의 종결자(terminator)로 박아 넣습니다. 3편에서 본 RAII, 즉 자원의 수명을 변수의 수명에 묶는 기법의 실체가 이 삽입입니다
- **MIR 최적화** — 상수 전파, MIR 수준 인라이닝, 도달 불가 코드 제거. 목적은 성능이 아니라 LLVM에 넘길 IR의 부피를 미리 줄이는 것입니다

이 셋 가운데 borrow check가 제로코스트의 정체입니다. 검사가 컴파일 시점에 끝나므로 런타임에는 "이 참조는 유효하다", "drop은 여기서 일어난다"는 컴파일 타임 계약만 남습니다

**LLVM IR** — LLVM 컴파일러 기반이 쓰는 언어 중립적 중간 표현입니다. Rust든 C++든 Swift든 자기 언어를 여기로 번역해 넘기면, 그다음은 LLVM이 책임집니다

여기서 단형화 인스턴스가 폭발적으로 늘어납니다(다음 절에서 다룹니다). 그리고 진짜 성능의 대부분이 여기서 결정됩니다

`-O3` 최적화 파이프라인이 인라이닝, 루프 언롤링, 벡터화, 전역 값 번호 매기기(GVN, 같은 값을 두 번 계산하지 않게 묶는 최적화), 그리디 레지스터 할당, 명령 스케줄링을 차례로 수행합니다

벡터화는 두 갈래로 돕니다. 루프를 통째로 SIMD 명령으로 바꾸는 루프 벡터라이저와, 인접한 스칼라 연산들을 한 벡터 명령으로 묶는 SLP(Superword-Level Parallelism)입니다

**타깃 기계어** — x86-64 또는 AArch64 명령이 실행 파일의 `.text` 섹션에 박힙니다

여기서 핵심은 컴파일러가 둘이고 역할이 갈린다는 점입니다

rustc 프런트엔드는 MIR까지, 즉 Rust 고유의 안전성과 의미를 결정합니다. borrow checker는 여기 살고 있죠

LLVM 백엔드는 LLVM IR부터, 즉 성능을 좌우하는 고전 최적화 전부를 맡습니다. 그래서 Rust의 런타임 속도는 상당 부분 LLVM의 품질에 위임되어 있습니다

"rustc가 느리다"는 말도 대개 이 구조에서 나옵니다. rustc가 느린 게 아니라 LLVM에 넘긴 IR이 방대한 것입니다

확인은 간단합니다

```bash
cargo build --release
objdump -d target/release/myapp   # .text에 박힌 확정 기계어
```

몇 번을 돌려도 이 바이트는 변하지 않습니다

## 단형화 — 추상화가 기계어에서 증발합니다

단형화(monomorphization)는 제네릭 함수를 실제로 쓰인 구체 타입마다 별도 함수로 복제 컴파일하는 기법입니다

`fn max<T: Ord>(a: T, b: T) -> T`에서 `T = i32` 인스턴스를 뽑아 최적화하면 이렇게 나옵니다

```rust
pub fn max_i32(a: i32, b: i32) -> i32 { if a > b { a } else { b } }
```

x86-64에서는 System V ABI에 따라 첫 인자가 `edi`, 둘째가 `esi`, 반환값이 `eax`입니다

```asm
max_i32:
    cmp   edi, esi
    mov   eax, esi
    cmovg eax, edi   ; 분기 없는 조건 이동
    ret
```

AArch64에서는 AAPCS64 규약에 따라 `w0`, `w1`을 쓰고 반환도 `w0`입니다

```asm
max_i32:
    cmp   w0, w1
    csel  w0, w0, w1, gt   // 조건 선택 한 줄
    ret
```

제네릭도, `Ord` 트레이트도, `if/else`도 사라졌습니다. 남은 것은 분기조차 없는 조건 이동 명령 한 줄뿐입니다

`cmov`(x86)와 `csel`(ARM)은 조건에 따라 값을 고르되 실제 분기를 만들지 않는 명령입니다. 분기 예측이 틀릴 여지 자체가 없죠

어떤 구현을 부를지 실행 중에 표에서 찾는 과정도, 런타임 디스패치도 없습니다. 이것이 제로코스트 추상화의 물리적 실체입니다

`T = String`이면 어떻게 될까요? 문자열 비교를 부르는 별도 함수가 또 하나 생성됩니다

그런데 대가가 있습니다. 그리고 둘 다 정량적입니다

**코드 팽창** — 같은 로직이 타입마다 복제되니 `.text`가 커집니다. 커진 코드는 명령어 캐시(i-cache)와 명령어 TLB(iTLB)를 압박하죠

i-cache는 CPU가 실행할 명령을 담아 두는 전용 캐시이고, iTLB는 명령 주소를 물리 주소로 옮기는 변환 테이블 캐시입니다. 코드가 이들 용량을 넘기면 미스가 나고, 미스는 곧 지연입니다

제로코스트 추상화가 캐시 미스로 되돌아오는 역설입니다. `Vec<T>`, `HashMap<K, V>`, 이터레이터 어댑터 체인이 수십 개 인스턴스로 불어나면 바이너리가 수 MB씩 커집니다

**컴파일 시간 폭발** — LLVM이 삼켜야 할 IR 부피는 대략 (단형화 인스턴스 수 × 함수 크기)에 비례합니다

`serde`나 `diesel`처럼 제네릭이 깊게 중첩된 크레이트의 빌드가 느린 근본 원인이 이것입니다

## 크레이트 경계를 넘는 인라이닝의 물리

인라이닝은 함수 호출을 없애고 본문을 호출 지점에 펼쳐 넣는 최적화입니다. 최적화의 어머니라 불릴 만큼 중요한데, 다른 최적화의 전제 조건이 되기 때문입니다

그런데 컴파일러가 남의 크레이트에 있는 함수를 인라인하려면 그 함수의 본문을 볼 수 있어야 합니다. 문제는 라이브러리가 이미 기계어로 컴파일되어 배포된다는 점입니다

![What a crate ships in its rlib — opaque object code vs serialized MIR](/diagrams/rust-aot-zero-cost-codegen-2.svg)

크레이트 A를 컴파일하면 `libA.rlib`가 나옵니다. rlib는 Rust 라이브러리의 배포 형식으로, 컴파일 산출물과 메타데이터를 함께 담은 아카이브입니다

그림처럼 그 안에는 두 종류가 공존합니다

왼쪽은 이미 기계어가 된 함수의 객체 코드입니다. 다른 크레이트에서 보면 속을 들여다볼 수 없는 불투명한 덩어리죠

오른쪽은 MIR로 직렬화된 함수입니다. 제네릭 함수와 `#[inline]`이 붙은 함수, 그리고 컴파일러 휴리스틱이 충분히 작다고 판단한 함수가 여기 들어갑니다

이 차이가 곧바로 결과를 가릅니다. 링크 시점 최적화(LTO)를 끈 기본 설정에서, 일반적인 비제네릭·비`#[inline]` 함수 호출은 크레이트 경계에서 `call` 명령으로 남습니다

본문을 볼 수 없으니 인라인이 불가능합니다

반면 제네릭 함수와 `#[inline]` 함수는 경계와 무관하게 호출부에 인라인됩니다. 크레이트 B가 그 MIR을 읽어 자기 쪽에서 단형화하고 코드를 생성하기 때문입니다

그러니까 "경계를 넘는다"는 건 비유가 아닙니다. rlib에 함수의 MIR이 실제로 직렬화되어 들어가고, 하위 크레이트가 그것을 다시 컴파일하는 물리적 동작입니다

라이브러리 저자가 뜨거운 함수에 `#[inline]`을 붙이는 것도 관용이 아니라 스위치입니다. 경계 너머 인라인을 허가한다는 선언이죠

경계를 더 광범위하게 허물고 싶다면 LTO로 갑니다. 곧 다룹니다

## dyn Trait — 비용을 숨기지 않습니다

단형화는 코드 팽창을 부릅니다. 반대로 코드 팽창을 막으려면 하나의 함수가 여러 타입을 처리해야 하고, 그러려면 런타임에 어떤 구현을 부를지 결정해야 합니다

Rust에서 이 선택이 `dyn Trait`입니다. 그리고 그 비용을 감추지 않습니다

![Fat pointer and vtable — the physical price of dynamic dispatch](/diagrams/rust-aot-zero-cost-codegen-3.svg)

`&dyn Draw`는 일반 참조와 크기부터 다릅니다. 데이터 포인터와 vtable 포인터를 나란히 담은 2워드짜리 fat pointer입니다

vtable(virtual method table)은 그 타입이 트레이트의 각 메서드를 어떻게 구현하는지 적어 둔 함수 포인터 표입니다. 타입마다 하나씩, 읽기 전용 데이터 섹션(`.rodata`)에 컴파일 타임에 정적으로 만들어집니다

그림의 vtable 레이아웃을 보면 앞의 세 슬롯이 고정입니다. 소멸자(`drop_in_place`) 포인터, 크기, 정렬이 먼저 오고 그다음부터 트레이트 메서드 포인터가 이어집니다

`x.method0()` 호출은 AArch64에서 두 명령이 됩니다. vtable에서 슬롯을 로드하고(`ldr x8, [x_vtable, #24]`), 그 주소로 간접 점프하는 것(`blr x8`)이죠

여기서 두 가지 비용이 발생합니다

첫째, 간접 호출은 분기 대상 버퍼(BTB)가 예측에 실패하면 파이프라인을 비우게 만듭니다. 목적지 주소를 레지스터에서 읽어야 알 수 있으니 예측이 어렵습니다

둘째, 호출 대상을 컴파일 타임에 모르므로 인라인이 원천 차단됩니다. 인라인이 막히면 그 뒤의 최적화도 줄줄이 막히죠

Rust가 제로코스트라는 말은 **정적 디스패치를 골랐을 때** 참입니다

`dyn`을 쓰면 그 비용이 명시적으로 드러납니다. 비용을 숨기지 않는다는 철학의 실물이죠

이 vtable 간접 호출은 다음 편에서 볼 Go의 `itab`, 그다음 편의 Java megamorphic 호출과 같은 계열의 비용입니다. 셋을 나란히 놓으면 다형성의 물리 가격표가 보입니다

## codegen-units와 LTO — 최적화 지평과 빌드 시간

LLVM 최적화는 자기가 볼 수 있는 범위 안에서만 일합니다. 이 범위를 최적화 지평이라 부르겠습니다

지평이 넓으면 좋은 코드가 나오지만 빌드가 느려집니다. 이 교환을 조절하는 노브가 둘입니다

### codegen-units — 크레이트 안의 분할

rustc는 크레이트의 단형화 아이템들을 N개의 코드 생성 단위(CGU, codegen unit)로 쪼개 LLVM을 병렬로 돌립니다

![Codegen units split the crate — parallel build vs a single optimization horizon](/diagrams/rust-aot-zero-cost-codegen-4.svg)

기본값은 릴리스 빌드가 16, 디버그 빌드가 256입니다. 최대 최적화는 1이죠

그림 왼쪽처럼 CGU가 많으면 LLVM 인스턴스를 여럿 띄워 병렬로 돌릴 수 있어 빌드가 빠릅니다. 대신 CGU 경계를 넘는 인라이닝이 막혀서 최적화 기회를 잃습니다

오른쪽 `codegen-units = 1`은 정반대입니다. 크레이트 전체가 하나의 지평에 들어오지만 병렬성이 0이라 빌드가 느려집니다

디버그 빌드가 256인 이유도 여기서 나옵니다. 최적화 품질은 어차피 필요 없고, 빌드 속도만 필요하기 때문입니다

### LTO — 크레이트 경계를 링크 시점에 허물기

CGU가 크레이트 안의 지평이라면, LTO(Link-Time Optimization)는 크레이트 사이의 지평을 넓히는 장치입니다. 링크할 때 IR을 다시 꺼내 최적화하는 것이죠

방식이 둘인데, 성격이 꽤 다릅니다

![ThinLTO's summary index vs fat LTO's single merged module](/diagrams/rust-aot-zero-cost-codegen-5.svg)

**ThinLTO**는 세 단계로 움직입니다

1. 각 모듈이 함수 요약(summary)을 방출합니다. 본문 IR이 아니라 콜 그래프, 함수 크기, 심볼 연결 정보 같은 메타데이터입니다
2. thin-link 단계에서 이 요약 인덱스를 훑어 "어느 함수를 어디로 가져올지"를 결정합니다
3. 백엔드를 병렬로 돌리되, 가져오기로 결정된 함수만 끌어와 인라인합니다

**fat LTO**(`lto = true`)는 단순 무식합니다. 전체 크레이트의 IR을 하나의 거대한 모듈로 합쳐 통째로 재최적화합니다

최고 성능이 나오지만 빌드가 가장 길고 메모리를 가장 많이 씁니다. 대형 프로젝트에서는 링크 단계가 병목이 되죠

ThinLTO의 결정적 차이가 바로 요약 인덱스입니다. 전부 합치지 않고 요약만 보고 필요한 것을 고르기 때문에, 전역 최적화 이득의 대부분을 병렬성 손실을 최소화한 채 얻습니다

실무에서 `lto = "thin"`을 먼저 시도하라는 권고가 여기서 나옵니다

## AOT를 제대로 쓰는 법

### 릴리스 프로파일

각 노브가 어떤 기전으로 작동하는지 주석으로 달았습니다

```toml
[profile.release]
opt-level = 3          # LLVM -O3: 루프·SLP 벡터화, 공격적 인라인
lto = "thin"           # 크레이트 경계 인라인 (요약 인덱스 기반). "fat"은 최대 성능·최장 빌드
codegen-units = 1      # 단일 최적화 지평. 병렬 빌드를 희생하고 인라인 기회를 최대화
panic = "abort"        # 언와인딩 테이블(.eh_frame) 제거 → .text 감소
strip = "symbols"      # 심볼 제거 → 바이너리 축소
```

`panic = "abort"`를 설명하겠습니다. 기본 동작인 언와인딩(unwinding)은 패닉이 나면 호출 스택을 거슬러 오르며 각 프레임의 값을 정리하는데, 이를 위해 `.eh_frame` 섹션에 정리 정보를 실어야 합니다

`abort`로 바꾸면 패닉 시 곧바로 프로세스를 죽이므로 그 테이블과 정리 코드가 통째로 사라집니다. 대신 패닉을 잡아서 복구하는 경로도 함께 사라지죠

`strip = "symbols"`는 바이너리를 줄이지만 관측성을 해칩니다. 디버그 심볼은 `split-debuginfo`로 별도 아티팩트에 분리 보관하는 편이 안전합니다

### target-cpu — 성능이 아니라 배포 전략입니다

`target-cpu`는 어떤 CPU 명령 집합까지 써도 되는지를 컴파일러에 알려 주는 플래그입니다

x86-64는 마이크로아키텍처 레벨이 누적적으로 정의돼 있습니다

| 레벨 | 추가되는 것 | 대표 세대 |
| --- | --- | --- |
| `x86-64-v1` | 베이스라인 (SSE2) | 기본값 — 어디서나 동작, SIMD 최소 |
| `x86-64-v2` | SSE4.2 | Nehalem 이후 |
| `x86-64-v3` | AVX2 | Haswell 이후 |
| `x86-64-v4` | AVX-512 | Skylake-X 등 |

AArch64의 기본값은 `armv8-a`입니다

```bash
RUSTFLAGS="-C target-cpu=x86-64-v3"     # AVX2가 보장되는 최신 x86 풀 대상
RUSTFLAGS="-C target-cpu=znver4"        # AMD Genoa (AVX-512)
RUSTFLAGS="-C target-cpu=neoverse-v2"   # AWS Graviton4 (SVE2)
```

기본값이 이토록 보수적인 이유가 있습니다. 빌드하는 기계와 배포되는 기계가 다를 수 있기 때문입니다

`target-cpu=native`로 AVX-512를 켜서 빌드한 바이너리를 그 확장이 없는 CPU에서 돌리면 어떻게 될까요?

CPU가 미지원 옵코드를 만나 `#UD`(invalid opcode) 예외를 올리고, 커널이 그 예외를 받아 프로세스에 SIGILL을 보내 죽입니다. 성능이 좀 떨어지는 게 아니라 그냥 죽습니다

그래서 `target-cpu`는 성능 노브가 아니라 배포 타깃을 얼마나 특정할 수 있느냐의 문제입니다

### 할당자에 대한 정확한 사실

여기서 흔한 오해를 하나 정정하고 갑니다

`Vec`, `Box`, `String`은 `brk`나 `mmap` 같은 메모리 시스템 콜을 직접 부르지 않습니다

이들은 `#[global_allocator]`에 위임하고, 그 기본값은 시스템 `malloc`입니다. `brk`와 `mmap`을 관리하는 것은 그 아래의 malloc이죠

`#[global_allocator]`를 바꾸면 jemalloc이나 mimalloc으로 교체할 수 있습니다. 이 계층은 시리즈 후반 메모리 편에서 커널까지 내려가며 해부합니다

### 벡터화, 그리고 ARM이 준 선물

벡터화(SIMD)는 한 명령으로 여러 데이터를 동시에 처리하는 기법입니다. LLVM의 자동 벡터화(루프 벡터라이저와 SLP)에 맡기거나, `std::simd`로 명시할 수 있습니다

문제는 x86의 벡터 폭이 코드에 박힌다는 점입니다. AVX2(256비트)로 짠 코드와 AVX-512(512비트) 코드는 서로 다른 기계어입니다

AOT로 하나만 고르면 다른 CPU에서 폭을 못 씁니다. 그래서 `#[target_feature]`와 `is_x86_feature_detected!`를 조합해 여러 버전을 바이너리에 넣고 런타임에 고르는 함수 다중 버전화(function multiversioning)를 해야 합니다

ARM의 SVE(Scalable Vector Extension)와 그 후속인 SVE2는 여기서 다른 답을 냅니다

VLA(Vector Length Agnostic, 벡터 길이 불가지론) 설계라 벡터 폭을 코드에 하드코딩하지 않습니다

폭을 `vscale`이라는 심볼로 남겨 두고, 실제 값은 실행되는 하드웨어가 정합니다

![One VLA binary adapts to different SVE hardware widths without a rebuild](/diagrams/rust-aot-zero-cost-codegen-6.svg)

AWS Graviton 두 세대를 비교하면 이 성질이 선명해집니다

| | Graviton3 (Neoverse V1) | Graviton4 (Neoverse V2) |
| --- | --- | --- |
| 확장 | SVE | SVE2 |
| 벡터 파이프 | 2개 × 256비트 | 4개 × 128비트 |
| 벡터 연산당 폭 | 256비트 | 128비트 |
| 총 처리 폭 | 512비트/사이클 | 512비트/사이클 |

같은 총 폭에 도달하는 방식이 정반대입니다. Graviton3는 넓은 파이프 두 개로, Graviton4는 좁은 파이프 네 개로 갑니다

x86이었다면 두 바이너리를 따로 빌드해야 할 상황입니다. 그런데 VLA 코드는 폭이 코드에 박혀 있지 않으니 같은 바이너리가 재컴파일 없이 양쪽에 적응합니다

이 성질이 AOT의 최대 약점, 즉 빌드 시점에 타깃을 못 정하면 손해를 본다는 문제를 아키텍처 차원에서 완화합니다

물론 대가도 있습니다. 벤치마크를 돌릴 때 코드 개선분과 하드웨어 폭 차이를 분리하기가 까다로워지죠

### PGO와 BOLT — AOT도 실행 프로파일을 먹습니다

PGO(Profile-Guided Optimization)는 프로그램을 한 번 실행해 어떤 분기가 자주 참인지, 어떤 함수가 뜨거운지 측정한 뒤, 그 프로파일을 다시 컴파일에 먹이는 기법입니다

```bash
cargo pgo instrument              # 계측 빌드
./app < 대표트래픽                 # 프로파일(.profraw) 수집
cargo pgo optimize                # 프로파일로 재빌드 (인라인·코드 배치·레지스터 할당 재조정)
cargo pgo bolt build --with-pgo   # 링크 후 BOLT로 기본 블록 재배치
```

BOLT는 링크가 끝난 바이너리를 다시 열어 뜨거운 기본 블록을 물리적으로 한데 모으는 도구입니다. 자주 함께 실행되는 코드를 메모리상 가까이 배치하면 i-cache와 iTLB 미스가 줄어듭니다

실측치가 있습니다. PGO 단독으로 대체로 런타임 10% 이상 개선되고, rustc 자신을 PGO로 빌드한 사례에서는 최대 15% 가까이 나왔습니다

BOLT는 그 위에 추가 이득을 얹습니다

여기서 이번 편의 가장 중요한 통찰이 나옵니다

**PGO는 JIT의 전유물이 아닙니다.** Rust도, Go도(1.21부터 정식), GraalVM native-image도 실행 프로파일을 빌드 타임에 먹습니다

### 빠른 개발 루프

`cranelift` 백엔드는 LLVM보다 훨씬 빠르게 코드를 생성하는 대신 최적화를 거의 하지 않습니다. 디버그 빌드의 반복 주기를 줄이는 용도로 씁니다

최적화는 릴리스에서 LLVM에 맡기면 됩니다

## 반영 시점이라는 축 — 2부의 척추

PGO가 JIT의 전유물이 아니라면, AOT와 JIT의 진짜 차이는 무엇일까요?

프로파일을 먹느냐가 아닙니다. **언제, 몇 번 반영하느냐**입니다

![Profile feedback timing — AOT applies it once at build, JIT applies it continuously](/diagrams/rust-aot-zero-cost-codegen-7.svg)

위쪽 AOT는 빌드 시점에 프로파일을 한 번 반영합니다. 그 순간 기계어가 확정되고, 배포된 다음에는 트래픽이 어떻게 변하든 코드는 그대로입니다

아래쪽 JIT는 실행 내내 프로파일을 수집하며 기계어를 다시 짭니다. 어떤 분기가 실제로 뜨거운지, 어떤 가상 호출이 사실상 한 종류 타입만 받는지를 실행 중에 배워 반영합니다

이 축이 2부 전체의 척추입니다. AOT PGO는 빌드 시 한 번 찍은 스냅샷이고, JIT는 연속적인 적응입니다

그래서 AOT의 약점은 정적이라 재최적화를 못 한다는 것이지, 프로파일을 못 본다는 것이 아닙니다. 이 구분을 흐리면 6편의 반전을 이해할 수 없습니다

## 세 언어 수렴 비교 — 기계어의 예측 가능성

| 항목 | Rust (4편) | Go (5편) | Java (6편) |
| --- | --- | --- | --- |
| 기계어 생성 시점 | 빌드 타임 (AOT) | 빌드 타임 (AOT) | 런타임 (JIT), 계층형 |
| 백엔드 | LLVM (깊은 최적화) | 자체 SSA (얕고 빠름) | C1·C2 (프로파일 기반) |
| 추상화 소거 | 단형화 → `cmov`·`csel` | 제한적 단형화, 인터페이스는 `itab` 간접 호출 | 런타임 역가상화 + 인라인 |
| 경계 인라인 | `#[inline]`·제네릭은 MIR 직렬화, LTO로 확장 | 패키지 경계에서 보수적 | 인라인 여부 자체가 런타임 결정 |
| 아키텍처 특화 | `target-cpu`로 빌드 시 결정 | `GOARCH`로 빌드 시 고정 | 실행 CPU를 감지해 그 아키텍처로 |
| 프로파일 반영 | PGO·BOLT (빌드 1회) | PGO (빌드 1회, 1.21+) | 실행 내내 연속 |
| 실행 예측성 | 최고 (불변 `.text`) | 높음 | 낮음 (웜업·역최적화 스파이크) |
| 꼬리 지연 | p99 ≈ p50 | 안정적 | 불안정 (웜업·역최적화·세이프포인트) |

표의 첫 행만 보면 Rust와 Go가 같은 편처럼 보입니다. 둘 다 AOT니까요

하지만 둘째 행부터 철학이 갈립니다. Rust는 LLVM에 기대어 느리고 깊게 갑니다

코드 팽창과 긴 컴파일 시간을 감수하고 최선의 기계어를 얻는 쪽이죠

Go는 정반대로 빠르고 얕게 갑니다. 자체 백엔드로 공격적 최적화를 일부러 포기했는데, 그 이유와 천장은 다음 편의 주제입니다

Java는 축이 아예 다릅니다. 빌드 타임에는 아키텍처 불가지론적인 바이트코드만 만들고 진짜 기계어 생성을 런타임 JIT에 미룹니다

그래서 표에서 유일하게 Java만 실행 중인 CPU를 실제로 보고 그 아키텍처의 기계어를 뽑는 사치를 부립니다. AVX-512인지 SVE2인지를 실행 시점에 알고 그에 맞춰 코드를 냅니다

Rust는 이것을 빌드 시점의 `target-cpu`로 미리 결정해야 하죠. 같은 정보를 언제 얻느냐의 차이입니다

## 트레이드오프 소결

Rust의 거래를 한 문장으로 요약하면 이렇습니다

> 컴파일러에게 최대한의 정보(타입·소유권·수명)를 주고 빌드 타임 비용(느린 컴파일·코드 팽창)을 선불로 치른 뒤, 런타임에는 얇고 예측 가능한 기계어를 받는다

**강점의 물리** — `.text`가 불변이라는 사실에서 세 가지가 따라 나옵니다

- 분기 예측기가 언제나 같은 코드 패턴을 학습합니다
- JIT 재컴파일로 인한 실행 중단, 즉 코드 캐시 갱신이 없습니다
- GC 정지도 없습니다

결과는 p99가 p50에 바짝 붙는 것입니다. 3편에서 말한 제로코스트의 성능적 실체가 이 결정론적 꼬리 지연입니다

**약점의 물리** — 정적이라 실행 실측에 기반한 재최적화를 스스로 하지 못합니다

어떤 분기가 실제로 뜨거운지, 어떤 `dyn` 호출이 사실상 단일 타입만 받는지를 런타임에 배워 반영할 수 없습니다. PGO와 BOLT는 부분적 보완이지만 빌드 시 한 번 찍은 스냅샷일 뿐, 상황 변화에 적응하지 않습니다

결국 Rust는 예측 가능성(꼬리 지연)을 적응성(정점 처리량의 상한)과 맞바꿉니다

이 맞바꿈이 왜 롱런 서버에서 뒤집힐 수 있는지는 6편과 7편에서 회수합니다

## DevOps 관점에서 더 파고들 질문

1. **멀티 마이크로아키텍처 노드풀** — `target-cpu=neoverse-v2`로 빌드한 바이너리를 Graviton3 노드에 스케줄하면 미지원 SVE2 옵코드에서 SIGILL로 죽습니다. 노드풀을 아키텍처별로 분리할까요(`nodeSelector`·taint), 런타임 CPU 디스패치로 단일 이미지를 유지할까요? 이미지 수, 바이너리 크기, 운영 복잡도의 교환은 어떻게 저울질할까요?
2. **CI 빌드 시간 폭발** — `codegen-units = 1`에 `lto = "fat"`을 얹으면 릴리스 빌드가 수 배로 늘고 링크 메모리도 커집니다. `sccache`(공유 캐시), `cargo-chef`(레이어 캐시), thin LTO로 어디까지 상쇄될까요? 성능 최대 릴리스와 빠른 PR 빌드를 프로파일로 어떻게 분리할까요?
3. **프로덕션 PGO 파이프라인** — 대표성 있는 `.profraw`를 얻으려면 실트래픽 분포를 캡처해야 합니다. 카나리에 계측 빌드를 소수 비율로 굴려 프로파일을 모으는 파이프라인은 어떻게 설계할까요? 트래픽 패턴이 바뀌어 프로파일이 낡으면 오히려 느려지는 위험은 어떻게 감시할까요?
4. **벡터화 검증** — 뜨거운 루프가 실제로 벡터화됐는지 `perf annotate`로 어셈블리에서 AVX2·SVE 명령의 존재를 확인하고, `perf stat`의 IPC와 부동소수 연산 카운터로 SIMD 활용률을 재는 절차는 무엇일까요? "벡터화된 줄 알았는데 스칼라"를 `criterion` 벤치로 CI에서 조기에 잡으려면요?
5. **정적 바이너리의 관측성** — `strip`한 릴리스 바이너리에 eBPF `uprobe`나 `perf`가 심볼을 붙이게 하려면 어떻게 해야 할까요? `split-debuginfo`로 디버그 심볼을 별도 아티팩트로 관리하고 프로덕션엔 스트립 바이너리를 배포하는 심볼 서버 워크플로는 어떻게 구성할까요?
6. **SVE 벤치마크 정규화** — VLA 바이너리를 Graviton3(2×256비트)와 Graviton4(4×128비트)에서 벤치할 때 벡터 폭과 파이프 수 차이를 어떻게 정규화해 코드 개선과 하드웨어 차이를 분리할까요? `vscale`별 루프라인 분석은 어떻게 설계할까요?

## 핵심 요약

- rustc는 컴파일러 둘의 합입니다. 프런트엔드가 MIR까지 내려오며 안전성을 결정하고(borrow check는 여기서 끝나 런타임 흔적이 0), LLVM 백엔드가 LLVM IR부터 성능 최적화 전부를 맡습니다
- 단형화는 제네릭을 타입마다 복제해 `cmov`·`csel` 한 줄로 녹입니다. 대가는 코드 팽창(i-cache·iTLB 압력)과 컴파일 시간 폭발이며, 둘 다 정량적입니다
- 크레이트 경계 인라인은 비유가 아닙니다. 제네릭과 `#[inline]` 함수는 MIR로 rlib에 직렬화되어 하위 크레이트가 재컴파일하고, 나머지는 경계에서 `call`로 남습니다
- `dyn Trait`는 2워드 fat pointer와 `.rodata`의 vtable로 구현됩니다. 간접 분기 예측 실패와 인라인 차단이라는 비용이 명시적으로 드러납니다
- codegen-units는 크레이트 안의 최적화 지평을, LTO는 크레이트 사이의 지평을 조절합니다. ThinLTO는 함수 요약 인덱스로 가져올 함수만 고르기 때문에 병렬성을 크게 잃지 않고 전역 최적화 이득의 대부분을 얻습니다
- `target-cpu`는 성능 노브가 아니라 배포 전략입니다. 미지원 옵코드는 `#UD` 예외를 거쳐 SIGILL로 프로세스를 죽입니다
- ARM SVE의 VLA는 벡터 폭을 코드에 박지 않습니다. 그래서 같은 바이너리가 Graviton3(2×256비트)와 Graviton4(4×128비트)에 재컴파일 없이 적응합니다
- PGO는 JIT의 전유물이 아닙니다. Rust(10% 이상, rustc 자체 최대 15%)도 Go(1.21+)도 GraalVM도 프로파일을 먹으며, 차이는 반영 시점입니다. AOT는 빌드 시 1회, JIT는 실행 내내
