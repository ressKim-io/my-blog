## 같은 설계도, 다른 건물

2편에서 세운 좌표계 위에서 Go는 선점형이면서 투명한 위치에 서 있고, Java 가상 스레드는 준선점형이면서 투명한 영역에 자리합니다. 반면 Rust의 tokio 런타임은 협력형이면서 함수 색칠이 요구되는 위치를 택했습니다

이 위치 선정이 곧 각 언어가 추구하는 철학입니다. 왜 이 지점에 자리 잡았는지, 실제 구현을 하나씩 열어 확인해 보겠습니다

## Go의 답 — GMP 스케줄러

Go는 정교한 M:N 스케줄러를 언어 런타임 내부에 기본 탑재했습니다. 개발자는 복잡한 비동기 문법 대신 `go f()` 단 한 줄만으로 고루틴을 생성하며, 기존과 동일한 방식으로 동기화·블로킹 코드를 작성합니다

이 강력한 동시성을 실현하는 중심 엔진이 바로 GMP 스케줄러입니다. 이름 그대로 세 가지 구성 요소의 머리글자를 딴 이름인데, 각각을 하나씩 짚어 보면 왜 이런 조합이 필요했는지가 드러납니다

- **G(Goroutine, 고루틴)** — 사용자 애플리케이션의 경량 실행 단위입니다. 최초 2KB의 작은 스택으로 시작하며, 메모리가 더 필요해지면 런타임이 더 큰 스택을 할당해 통째로 복사하는 방식으로 성장합니다. 2편의 병원 비유에서는 "환자"에 해당합니다
- **M(Machine, OS 스레드)** — 운영체제가 인식하는 실제 커널 스레드입니다. 물리 코어에서 기계어를 실행하는 실체이며, 비유에서는 "의사"에 해당합니다
- **P(Processor, 논리 프로세서)** — 고루틴(`G`)을 OS 스레드(`M`)에 태워 실행할 수 있는 논리적 권리이자, 로컬 실행 대기열(Local Run Queue)을 소유하는 객체입니다. 생성되는 개수는 `GOMAXPROCS` 환경 변수(기본값은 호스트의 사용 가능 코어 수)로 결정됩니다. 비유에서는 "진료실 배정권"에 가깝습니다

`G`와 `M`만 있다면 왜 부족할까요? 만약 고루틴이 OS 스레드에 곧바로 매달린다면, 특정 스레드가 시스템 콜로 블로킹되는 순간 그 스레드에 딸린 다른 고루틴들도 함께 멈춰 버립니다. `P`라는 중간 단계를 하나 더 두면, 블로킹된 `M`에서 `P`만 쏙 빼내 다른 `M`에 옮겨 붙일 수 있습니다. 이 유연함이 GMP 세 글자가 필요한 이유입니다

스케줄러가 지키는 핵심 규칙은 하나입니다. 고루틴(`G`)이 물리 CPU에서 실행되려면 논리 프로세서(`P`)를 획득해야 하며, 이 `P`가 OS 스레드(`M`)와 결합되어 있어야 합니다. 따라서 어떤 순간에 실제로 동시에 연산을 수행하는 고루틴의 최대 개수는 `P`의 개수, 즉 `GOMAXPROCS` 설정값과 정확히 일치합니다

![Go GMP — per-P run queues, work stealing, global queue and the netpoller](/diagrams/mn-scheduler-go-java-rust-1.svg)

그림에서 눈여겨볼 부분은 각 논리 프로세서 `P`가 독립적인 로컬 실행 대기열을 갖고 있다는 점입니다. 대부분의 스케줄링이 전역 락(Lock) 없이 각자의 로컬 대기열 안에서 빠르게 처리되므로 연산 오버헤드가 극히 낮습니다

로컬 대기열에 미처 담지 못한 고루틴은 전역 실행 대기열(Global Run Queue)로 이동하며, 각 `P`가 주기적으로 이를 확인해 분배받습니다. 네트워크 I/O를 기다리는 고루틴들은 별도의 네트워크 폴러(Netpoller) 장치에 잠들어 대기합니다

이 구조가 실제로 어떻게 움직이는지, 네 가지 핵심 메커니즘으로 나눠 순서대로 확인하겠습니다

### 메커니즘 1 — 작업 훔치기(Work Stealing)

특정 `P`의 로컬 대기열이 비어 할 일이 없어지면, 그 `P`는 다른 `P`를 무작위로 선택해 대기열에 있는 고루틴의 절반을 훔쳐 옵니다. 간호사 한 명의 대기줄이 텅 비었다면, 옆 간호사의 대기줄에서 절반을 넘겨받아 오는 것과 같습니다

이 분산적이고 무작위적인 훔치기 메커니즘 덕분에 특정 코어만 놀고 다른 코어만 과부하되는 상황이 방지되며, 다중 코어 전체에 부하가 고르게 분산됩니다

### 메커니즘 2 — 시스템 콜 핸드오프(Syscall Handoff)

1편에서 "런타임이 커널 관문을 감싼다"고 표현했던 개념의 실제 작동 방식이 바로 이것입니다. 고루틴이 파일 읽기 등 커널 대기가 필요한 블로킹 시스템 콜을 호출하면, 런타임의 `entersyscall` 훅이 즉시 개입합니다

이때 런타임은 곧 커널에 묶이게 될 OS 스레드(`M`)로부터 논리 프로세서(`P`)를 분리하고, 대기 중인 다른 물리 OS 스레드에 붙여 줍니다. 의사 한 명이 검사실에 들어가 당분간 나오지 못하게 되면, 그 의사가 쥐고 있던 "진료실 배정권"만 빼내 대기 중인 다른 의사에게 넘기는 것입니다

![Syscall handoff — the P detaches from a blocked M and keeps scheduling on another M](/diagrams/mn-scheduler-go-java-rust-2.svg)

그림 왼쪽처럼 고루틴 `G1`이 시스템 콜로 커널 모드에 진입하면 `M0` 스레드도 커널 대기열에 묶입니다. 하지만 오른쪽을 보면 `P`가 묶여 버린 `M0`을 미련 없이 버리고 새로운 `M`으로 옮겨 가면서, 대기열에 있던 `G2`와 `G3`가 중단 없이 물리 코어에서 실행을 이어 갑니다

시스템 콜 처리가 끝나고 돌아온 `G1`은 런타임의 `exitsyscall` 훅에서 다시 비어 있는 `P`를 획득해야만 실행을 재개할 수 있습니다. 스레드 하나가 블로킹되어도 전체 런타임이 멈추지 않고 매끄럽게 돌아가게 만드는 핵심 설계입니다

### 메커니즘 3 — 네트워크 폴러(Netpoller)

소켓 통신을 포함한 네트워크 I/O는 훨씬 더 효율적인 샛길을 이용합니다. 리눅스의 `epoll`(macOS의 `kqueue`, Windows의 `IOCP`)은 수천 개의 소켓 이벤트를 커널 내부에서 감시하다가 준비된 소켓만 알려 주는 비동기 통지 메커니즘입니다

Go 런타임은 네트워크 소켓을 전부 논블로킹(Non-blocking) 모드로 생성한 뒤 이를 네트워크 폴러에 등록해 감시합니다. `conn.Read()`로 소켓에서 데이터가 오길 기다리는 고루틴은 무거운 OS 스레드를 멈추지 않고, 즉시 네트워크 폴러 대기 상태로 재워집니다(Park)

수백만 개의 소켓 커넥션이 대부분 잠든 상태로 단 몇 개의 OS 스레드만 효율적으로 공유하는 구조이며, 이것이 Go가 I/O 중심 네트워크 서버에서 탁월한 성능을 내는 원동력입니다

### 메커니즘 4 — 비동기 선점(Asynchronous Preemption, Go 1.14+)

초기 Go 런타임은 오직 함수 호출 경계에서만 스케줄러가 개입할 수 있었습니다. 그래서 `for {}` 같은 무한 루프나 계산 중심 루프가 돌면 고루틴이 CPU를 독점해 다른 고루틴들이 굶주리는 현상이 발생했습니다

Go 1.14부터는 이 문제를 해결하기 위해 런타임이 직접 감시자를 두었습니다. 10ms 이상 CPU를 독점한 `M` 스레드에 유닉스 `SIGURG` 시그널을 발송하고, 시그널 핸들러가 해당 고루틴의 실행 레지스터를 강제로 힙에 저장한 뒤 로컬 대기열로 강제로 내립니다

이 메커니즘 덕분에 Go의 M:N 스케줄러는 완벽한 선점형(Preemptive) 런타임으로 완성되었습니다. 2편의 좌표계에서 Go가 가장 윗줄(선점형)에 위치하는 이유가 바로 여기 있습니다

### GOMAXPROCS와 컨테이너 환경의 주의점

```go
// 고루틴 10만 개 — 1:1 OS 스레드였다면 스택만 수십 GB를 소모합니다
for i := 0; i < 100_000; i++ {
    go func(id int) {
        resp, _ := http.Get(url) // 블로킹 함수처럼 보이지만
        _ = resp                 // 내부적으로 netpoller에 Park되면서 OS 스레드(M) 반환
    }(i)
}
```

이토록 훌륭한 설계에도 오랜 시간 운영 엔지니어를 괴롭힌 함정이 하나 있었습니다. Go 1.24 버전까지 런타임은 쿠버네티스 파드(Pod)에 설정된 컨테이너 CPU 제한(Limit)을 인식하지 못하고, 호스트 노드의 전체 물리 코어 수를 그대로 읽었습니다

예를 들어 64코어 물리 노드에서 `resources.limits.cpu: "2"`로 2코어만 쓰도록 제한한 파드를 띄우면, Go 런타임은 논리 프로세서 `P`를 64개나 만들어 버립니다. 64개의 `P`가 동시에 연산을 시도하다가 cgroup(Linux Control Group, 컨테이너 리소스 격리 메커니즘)에 할당된 CFS 쿼터를 단숨에 소진해 버리면, 남은 스케줄링 주기 동안 파드 전체가 멈춰 서는 CPU 스로틀링(Throttling) 벌칙을 받게 됩니다

이것이 컨테이너 환경에서 간헐적으로 응답 지연이 수백 ms까지 치솟는 꼬리 지연(Tail Latency, 상위 99% 구간의 응답 지연) 문제의 대표적인 원인이었습니다

Go 1.25 버전부터는 런타임 기본값으로 이 문제를 마침내 해결했습니다. 커널의 cgroup CPU 제한 설정(`cpu.cfs_quota_us` 및 cgroup v2의 `cpu.max`)을 런타임이 직접 읽어 `GOMAXPROCS`를 파드 리밋에 맞춰 자동으로 설정하며, 약 30초마다 설정 변화를 감지해 동적으로 재조정합니다

이 자동 조절 기능을 끄고 싶다면 환경 변수 `GODEBUG=containermaxprocs=0`을 지정하면 됩니다. 단 런타임은 CPU 제한(`limit`)만 참고하며 최소 보장량(`request`)은 무시한다는 점은 유의해야 합니다

> **DevOps 함의** — Go 1.24 이하 버전을 쿠버네티스 클러스터에 배포한다면 Uber의 `automaxprocs` 라이브러리를 임포트해 `GOMAXPROCS`를 수동 보정하는 것이 필수 정석이었습니다. Go 1.25부터는 언어 런타임이 알아서 처리해 줍니다. "여유로운 2코어 파드가 왜 간헐적으로 스로틀링에 걸릴까?"라는 의문의 정답이 바로 여기에 있습니다

## Java의 답 — 플랫폼 스레드에서 가상 스레드로

Java는 이제 두 가지 스레드 모델을 동시에 지원합니다. 오랜 시간 Java 생태계를 지배해 온 "JVM 스레드는 OS 스레드와 무조건 1:1"이라는 상식이 깨진 전환점입니다

기존의 전통적인 플랫폼 스레드(Platform Thread)는 여전히 OS 스레드와 일대일로 매핑됩니다. `new Thread()`나 기존의 `ThreadPoolExecutor`가 만드는 스레드이며, 스택 하나당 수백 KB에서 1MB를 점유하는 무거운 물리적 비용도 2편에서 확인한 그대로입니다

따라서 플랫폼 스레드 기반의 thread-per-request(요청당 1스레드) 모델로 구성된 전통적인 Java 서버는 수백에서 수천 개의 동시 요청이 물리적 한계였습니다. 이 한계를 극복하기 위해 WebFlux 같은 리액티브(Reactive) 스택이 광범위하게 도입되었으나, 콜백 중첩과 비동기 연산자 체인으로 코드 읽기와 디버깅이 난해해지는 비용을 치러야 했습니다

가상 스레드(Virtual Thread, JDK 21 정식 출시)는 완전히 다른 길을 제시합니다. 무거운 OS 스레드 스택을 사용하는 대신, 실행 중단 상태를 힙(Heap) 메모리에 가볍게 저장하는 초경량 유저 레벨 스레드입니다

여기서 중심이 되는 개념이 바로 컨티뉴에이션(Continuation, 실행을 멈춘 지점의 호출 스택과 레지스터 상태를 담은 힙 객체)입니다. 특정 시점까지 실행된 함수의 호출 스택을 그대로 복사해 JVM 힙 메모리에 일반 객체처럼 저장해 둔 것이라고 이해하면 직관적입니다

![Virtual thread lifecycle — mount on a carrier, unmount to the heap on blocking I/O, remount later](/diagrams/mn-scheduler-go-java-rust-3.svg)

그림에 나타난 생애주기의 순환이 가상 스레드 작동 방식의 전부입니다. 가상 스레드가 실제로 물리 CPU에서 실행되려면, 먼저 운송반 역할을 하는 캐리어 스레드(Carrier Thread, 내부적으로 코어 수만큼 생성된 `ForkJoinPool`의 플랫폼 스레드)에 올라타야(Mount) 합니다

실행 도중 네트워크 소켓 대기나 데이터베이스 조회 같은 블로킹 I/O를 만나면, JVM이 사전에 계측해 둔 지점에서 컨티뉴에이션 객체로 스택 상태를 힙에 떠 놓고 캐리어 스레드에서 즉시 내립니다(Unmount). 해방된 캐리어 스레드는 대기열에서 기다리는 다른 가상 스레드를 태워 쉼 없이 연산을 이어 갑니다

I/O 응답이 도착해 연산을 재개할 때가 되면, 가상 스레드는 비어 있는 임의의 캐리어 스레드에 다시 올라타(Remount) 중단되었던 지점부터 실행을 계속합니다. 초기 메모리 풋프린트가 단 수백 바이트에서 수 KB 수준에 불과하므로, 단일 JVM 내에서 수백만 개의 스레드를 자유롭게 생성할 수 있습니다

```java
// 직관적인 thread-per-request 스타일 코드가 초경량으로 되살아납니다
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            var body = client.send(req, ofString()); // 블로킹 호출 → 컨티뉴에이션 힙 저장 후 캐리어에서 Unmount
            return body.body();
        });
    }
}
```

개발자 입장에서는 WebFlux처럼 복잡한 리액티브 코드를 짤 필요 없이, 익숙하고 동기적인 블로킹 코드를 그대로 작성합니다. Go가 추구했던 "투명한 동시성 처리" 철학과 정확히 수렴하는 결실입니다

### 피닝(Pinning) 현상과 JEP 491의 해결

가상 스레드 도입 초기 가장 치명적인 아킬레스건은 피닝(Pinning, 고정) 문제였습니다. 가상 스레드가 블로킹 I/O를 만났음에도 정상적으로 내리지(Unmount) 못하고, 물리 캐리어 스레드를 꽉 쥐어 잡고 있는 현상입니다

캐리어 스레드가 고정되어 묶여 버리는 만큼 M:N 스케줄링의 이점이 사라지며, 극단적인 경우 대기 중인 다른 가상 스레드들이 굶주리다 서버 데드락으로 이어집니다

JDK 21에서 JDK 23까지 이 피닝을 유발했던 가장 큰 주범은 바로 Java 기본 동기화 키워드인 `synchronized` 블록이었습니다. `synchronized`는 객체마다 부여되는 모니터 락(Monitor Lock, JVM 내부 동기화 잠금)으로 임계 구역을 보호하는데, 당시 JVM은 이 락의 소유권을 가상 스레드가 아닌 하부 물리 캐리어 스레드 기준으로 추적했습니다

이 때문에 `synchronized` 블록 내부에서 데이터베이스 호출 등의 블로킹이 발생하면 JVM 안전성 문제로 Unmount가 차단되었습니다. 수많은 오픈소스 레거시 라이브러리가 `synchronized`로 보호되고 있었기 때문에, 가상 스레드 전환 시 기대했던 성능 이점이 무력화되곤 했습니다

JDK 24에 적용된 JEP 491(Java Enhancement Proposal 491, 모니터 락 아키텍처 개편안)이 마침내 이 오래된 난제를 완전히 풀어냈습니다. 모니터 락 소유권과 내부 상태 추적을 가상 스레드 단위로 완벽하게 분리하도록 JVM 커널을 재작성해, 이제 `synchronized` 블록 내부에서 블로킹이 일어나도 정상적으로 Unmount되어 캐리어를 해방합니다

현재 장기 지원(LTS) 버전인 JDK 25에는 이 개선 사항이 기본 탑재되어 있습니다. 다만 JNI(Java Native Interface) 네이티브 C 메서드 호출 도중의 블로킹이나 클래스 초기화 시점(`static` 블록)의 블로킹 같은 예외적인 상황에서의 피닝은 여전히 주의해야 합니다

> **DevOps 함의** — 가상 스레드를 프로덕션에 도입할 때는 `-Djdk.tracePinnedThreads` JVM 옵션을 켜거나 JFR(JDK Flight Recorder, JVM 내장 성능 분석기)의 `jdk.VirtualThreadPinned` 이벤트를 모니터링해 숨은 피닝 지점을 찾아내야 합니다. 사용하는 Java 버전이 JDK 24 미만이라면 `synchronized` 락이 심각한 병목 지점이 되므로, 라이브러리의 락을 `java.util.concurrent.ReentrantLock`으로 치환하거나 상위 JDK로의 업그레이드를 신중히 판단해야 합니다

### 컨테이너 리소스 인식은 Java가 훨씬 앞섰습니다

재미있는 역사적 사실이 하나 있습니다. JVM은 이미 JDK 10(2018년 정식 배포)부터 `UseContainerSupport` 플래그를 기본 활성화해, 리눅스 cgroup의 CPU 및 물리 메모리 제한을 스스로 정확히 파악해 왔습니다

`Runtime.availableProcessors()` 메서드가 파드의 CPU 제한 설정을 반영해 값을 반환하므로, 가상 스레드의 기반이 되는 `ForkJoinPool` 캐리어 스레드 개수와 가비지 컬렉터(GC) 스레드 수가 컨테이너 한계에 자동으로 맞춰집니다. Go 런타임이 2025년(1.25 버전)에야 기본 탑재한 컨테이너 자동 인식을, Java는 7년이나 일찍 갖추고 있었던 셈입니다

물론 JVM은 이 물리 메모리 인식 결과를 힙(`-Xmx`) 크기 자동 산정과 엮으면서 또 다른 형태의 컨테이너 OOMKilled 문제를 유발하기도 하는데, 이 흥미로운 트레이드오프는 5부에서 심도 있게 다룹니다

## Rust의 답 — 1:1 유지, 비동기는 라이브러리 선택

Rust는 다른 두 언어와 달리, 무거운 M:N 스케줄러를 언어 내장 런타임에서 의도적으로 완전히 제거했습니다

1.0 버전 정식 출시 이전인 2014년까지는 Rust도 M:N 기반의 그린 스레드(`libgreen`)를 실험했습니다. 그러나 모든 컴파일된 바이너리에 무거운 유저 스케줄러와 메모리 관리 코드를 강제하는 것이, Rust의 핵심 철학인 제로코스트 추상화(Zero-Cost Abstraction, 쓰지 않는 기능에는 어떤 런타임 비용도 지불하지 않는다)와 정면으로 배치되어 과감히 삭제했습니다. 이 역사적 결정의 전모는 4편에서 낱낱이 해부합니다

그 결과 현재 Rust의 동시성 모델은 명확히 두 개의 층으로 분리되어 동작합니다

**`std::thread`** — 커널의 OS 스레드와 정확히 1:1로 일치하는 표준 스레드입니다. Java의 전통적인 플랫폼 스레드와 같은 무게(기본 2MB 스택)를 지니며, CPU 자원을 집약적으로 소모하는 병렬 수치 연산 작업에는 이 1:1 모델이 가장 이상적인 정답입니다(`rayon` 같은 고성능 데이터 병렬 라이브러리도 이 스레드 모델 위에서 작동합니다)

**`async`/`.await` 키워드** — 비동기 함수를 작성하면 언어 런타임이 개입하는 것이 아니라, 컴파일러가 해당 함수를 상태 기계(State Machine, 실행 중단점마다의 변수 상태를 열거형 변수로 관리하는 기계어 변환 체계)로 재구성합니다. 이렇게 변환된 `Future`(비동기 연산의 최종 완료 상태를 표현하는 핵심 트레이트) 객체는 힙 메모리 할당이나 무거운 런타임을 절대 강제하지 않습니다

하지만 Rust 표준 라이브러리는 이 `Future`를 실제로 구동하고 배차해 줄 실행기(Executor)를 제공하지 않습니다. 개발자가 프로젝트 특성에 맞게 `tokio`, `async-std`, `smol` 같은 외부 비동기 런타임 라이브러리를 직접 선택해 조립해야 합니다

### tokio — 라이브러리로 구현한 M:N, 단 협력형 런타임

```rust
// tokio: 비동기 스케줄링은 협력형이므로 오직 .await 지점에서만 자발적으로 양보합니다
#[tokio::main]
async fn main() {
    let mut tasks = Vec::new();
    for _ in 0..100_000 {
        tasks.push(tokio::spawn(async {
            let resp = reqwest::get(URL).await.unwrap(); // .await 지점에서 스케줄러에게 양보
            resp.text().await
        }));
    }
}
```

Rust 생태계의 표준처럼 자리 잡은 `tokio`의 멀티스레드 런타임은 물리 코어 수만큼 생성한 워커 OS 스레드 풀 위에서 수많은 비동기 태스크를 작업 훔치기(Work Stealing) 방식으로 스케줄링합니다. 네트워크 및 파일 I/O는 `mio`라는 저수준 라이브러리를 거쳐 `epoll` 이벤트 루프로 처리합니다

이 기본 구조만 놓고 보면 Go의 GMP 스케줄러와 매우 닮아 있습니다. 하지만 실제 운영 측면에서는 두 가지 결정적인 차이점이 존재합니다

**함정 1: 함수 색칠 문제(Function Coloring)** — 비동기 함수(`async fn`)는 다른 비동기 함수(`.await`) 내부에서만 자연스럽게 호출됩니다. 이로 인해 코드베이스 전체가 동기(Sync) 세계와 비동기(Async) 세계로 명확히 단절되는 구조적 제약이 발생합니다

**함정 2: 협력형 스케줄링의 대가** — `tokio` 런타임에는 Go의 `SIGURG` 시그널 강제 선점이나 Java 가상 스레드의 자동 Unmount 같은 구제 장치가 없습니다. 어떤 태스크가 `.await` 지점 없이 CPU 연산을 오래 독점하거나, 일반 블로킹 함수(`std::thread::sleep` 등)를 호출하면 해당 워커 스레드 전체가 즉시 멈춰 섭니다

이 함정이 실제로 어떤 순서로 문제를 일으키는지 짚어 보겠습니다. 워커 스레드 하나에는 여러 비동기 태스크가 번갈아 올라탑니다. 그중 하나가 `.await` 없이 블로킹 함수를 호출하면, 그 워커 스레드는 협력형 스케줄러의 규칙대로 제어권을 돌려받지 못한 채 그대로 멈춥니다. 같은 워커에 배정되어 있던 다른 태스크들도 순번이 돌아올 방법이 없으므로 함께 굶주립니다. Go라면 `SIGURG` 시그널이, Java라면 자동 Unmount가 강제로 끼어들어 이 상황을 풀어 줬겠지만, tokio에는 그런 강제 개입 장치가 없습니다

```rust
// 비동기 태스크 내부에서 일반 블로킹 함수를 호출하면 워커 스레드 전체가 동결됩니다
tokio::spawn(async {
    std::thread::sleep(Duration::from_secs(1)); // .await가 아님 → 해당 워커 스레드의 스케줄러 정지
    // 무거운 연산이나 블로킹 호출이 필수라면 반드시 tokio::task::spawn_blocking으로 격리해야 합니다
});
```

런타임이 커널 블로킹이나 무거운 연산을 알아서 흡수해 주지 않으므로, 개발자가 직접 `tokio::task::spawn_blocking` 같은 격리 스레드 풀로 작업을 떼어 내야 합니다. 편한 자동화를 포기한 대신 모든 시스템 제어권을 개발자에게 명시적으로 위임한 설계입니다

> **컨테이너 각주** — `tokio`가 기본적으로 생성할 워커 스레드 수는 하부의 `num_cpus` 크레이트(Crate, Rust 패키지)가 결정합니다. 이 크레이트와 Rust 표준 라이브러리의 `available_parallelism` API는 꾸준히 개선되어 현재는 리눅스 cgroup의 CPU 할당 쿼터를 정확히 읽어 옵니다. 따라서 최신 버전에서는 Go 1.24 이전과 같은 "호스트 전체 코어 수 오인" 문제가 기본적으로 해소되었습니다. 다만 어떤 계열의 라이브러리나 프레임워크가 코어 수를 산정하는지 프로젝트마다 다를 수 있으므로, 컨테이너 배포 시에는 환경 변수 `TOKIO_WORKER_THREADS`를 명시해 워커 스레드 수를 확정 짓는 것이 안전합니다

## 세 언어 수렴 비교

| 항목 | Rust (std) | Rust (tokio) | Go | Java (플랫폼) | Java (가상) |
| --- | --- | --- | --- | --- | --- |
| 모델 | 1:1 | M:N (협력형) | M:N (선점형) | 1:1 | M:N (준선점형) |
| 스케줄 주체 | 커널 스케줄러 | tokio 런타임 | Go 런타임 | 커널 스케줄러 | JVM (ForkJoinPool) |
| 태스크 무게 | 스택 약 2MB | 상태 기계(`Future`) | 스택 2KB (동적 성장) | 스택 약 0.5~1MB | 힙 컨티뉴에이션 |
| 선점 방식 | OS 타임슬라이스 | 없음 (`.await` 양보) | `SIGURG` 시그널 선점 | OS 타임슬라이스 | 블로킹·마운트 지점 |
| 코드 스타일 | 동기 블로킹 | 비동기 색칠 분리 | 투명한 동기 작성 | 동기 블로킹 | 투명한 동기 작성 |
| 블로킹 흡수 | 스레드 메모리 낭비 | `spawn_blocking` 수동 | `P` 핸드오프 자동 | 스레드 메모리 낭비 | 캐리어 Unmount 자동 |
| 컨테이너 인식 | cgroup 쿼터 반영 | `num_cpus` 쿼터 파싱 | Go 1.25부터 기본 | JDK 10부터 기본 | JDK 10부터 기본 |

표에서 Java 가상 스레드를 "준선점형"으로 표기한 명확한 이유가 있습니다. 가상 스레드는 물리 코어의 실행 시간(Time Slice)을 기준으로 강제 선점되는 것이 아니라, 오직 I/O 블로킹이나 마운트 지점에서만 양보합니다. 그래서 타이트 루프를 강제로 중단하는 Go의 완전 선점형과는 구별됩니다

이 비교표를 가로질러 관찰할 수 있는 거대한 수렴 현상이 이번 편의 핵심입니다. Go와 Java 가상 스레드는 출발점과 역사가 다름에도, 결국 **"개발자에게는 투명한 기존 블로킹 코드를 제공하고, 하부 런타임이 I/O 대기 비용을 알아서 흡수한다"**는 동일한 궁극의 결론에 도달했습니다

오직 Rust만이 이 친절한 자동화를 거절했습니다. 제로코스트 추상화와 극도의 성능 예측 가능성을 수호하기 위해, 비동기 함수 색칠과 수동 블로킹 격리라는 무거운 책임을 명시적으로 개발자에게 남겼습니다

결국 1편에서 내렸던 결론과 동일한 철학적 축 위로 귀결됩니다. 같은 기술적 난제를 두고 런타임이 모든 것을 책임질 것인가(Go, Java), 아니면 통제권을 개발자에게 온전히 넘길 것인가(Rust)의 선택입니다

## 트레이드오프 소결

1:1 모델은 구조가 단순하고 실행 경로가 명확하며, CPU 자원을 집약적으로 소모하는 고성능 병렬 연산 작업에 가장 훌륭한 선택지입니다. 하지만 대규모 동시 접속과 I/O 대기가 폭주하는 서버 환경에서는 2편에서 확인한 메모리 폭발과 문맥 교환 과부하로 무너집니다

M:N 모델은 수십만 개의 I/O 대기 태스크를 극히 적은 물리적 비용으로 매끄럽게 흡수합니다. 대신 런타임 내부의 복잡도가 크게 증가하며, 피닝(Pinning), 워커 스레드 블로킹, `GOMAXPROCS` 설정 오류 등 이전에는 없었던 새로운 형태의 장애 양식을 만들어 냅니다

그리고 어느 모델을 선택하든 서버가 가진 물리 코어 수라는 절대적 연산 천장은 변하지 않습니다. M:N 스케줄링은 데이터가 오기를 '기다리는' 수많은 태스크를 값싸게 다룰 뿐, 실제로 CPU를 써서 '계산하는' 태스크의 속도를 물리적으로 더 빠르거나 늘려 주지는 않습니다

## DevOps 관점에서 더 파고들 질문

1. **스로틀링 진단** — 컨테이너의 `/sys/fs/cgroup/cpu.stat` 파일에서 `nr_throttled`(스로틀링 횟수)와 `throttled_time`(누적 지연 시간)을 확인하면 CFS 스로틀링 여부를 즉시 포착할 수 있습니다. Go 1.24 이하 버전이나 워커 스레드 수가 과도하게 설정된 서비스가 CPU 제한에 걸려 간헐적인 꼬리 지연을 만들고 있지는 않나요?
2. **가상 스레드 피닝 탐지** — JVM 옵션 `-Djdk.tracePinnedThreads`나 JFR의 `jdk.VirtualThreadPinned` 이벤트를 활성화해 운영 중인 서비스의 숨은 피닝 지점을 식별할 수 있습니다. 대상 시스템이 JDK 24 미만이라면 플랫폼 업그레이드 비용과 레거시 `synchronized`를 `ReentrantLock`으로 치환하는 리팩토링 비용 중 어느 쪽이 더 합리적일까요?
3. **가상 스레드 vs 리액티브 스택** — Java 가상 스레드(Project Loom) 출시 이후 WebFlux 같은 리액티브 프레임워크의 존재 이유가 상당 부분 희석되었습니다. 그럼에도 백프레셔(Backpressure, 데이터 흐름 제어)나 정교한 데이터 스트리밍 조합이 핵심인 시스템에서는 여전히 리액티브 모델이 유리할까요?
4. **tokio 블로킹 격리 장애** — 비동기 핸들러 내부에서 무거운 암호화 해시 연산이나 파일 I/O를 `spawn_blocking` 없이 직접 수행하면, 왜 서버 전체 처리량이 순식간에 반토막 날까요? 비동기 서버에서 별도 블로킹 스레드 풀 크기는 얼마로 잡아야 병목이 생기지 않을까요?
5. **최소 보장량(`request`)과 상한 제한(`limit`)의 불일치** — Go 1.25 런타임은 cgroup의 CPU 제한(`limit`)만 참고하며, 쿠버네티스의 최소 보장량(`request`)은 완전히 무시합니다. `requests.cpu: "1"`과 `limits.cpu: "4"`로 설정된 파드에서 `GOMAXPROCS`는 4로 설정되는데, 이 고무줄 같은 간극이 노드의 빈패킹(5부에서 다룰 자원 밀집 배치) 전략과 충돌할 때 어떤 부작용이 발생할까요?

## 핵심 요약

- Go의 GMP 스케줄러는 고루틴(G)·OS 스레드(M)·논리 프로세서(P) 세 요소로 나뉘며, 작업 훔치기·시스템 콜 핸드오프·네트워크 폴러·`SIGURG` 비동기 선점 네 가지 메커니즘이 맞물려 완전한 선점형 M:N을 구현합니다
- Java 가상 스레드는 힙에 저장된 컨티뉴에이션을 캐리어 스레드에 Mount/Unmount하는 방식으로 동작합니다. JDK 21~23의 `synchronized` 피닝 문제는 JDK 24의 JEP 491로 해결됐습니다
- Rust는 제로코스트 추상화 철학을 지키기 위해 언어 내장 M:N 런타임을 1.0 이전에 과감히 포기했습니다. 외부 라이브러리인 `tokio`가 그 역할을 훌륭히 대신하지만, 협력형 스케줄링 구조이므로 블로킹 작업을 격리하는 것은 전적으로 개발자의 의무입니다
- 컨테이너 CPU 제한 인식 기능은 Java(JDK 10)가 Go(1.25)보다 무려 7년 앞서 탑재했습니다. 컨테이너 CPU 리밋 설정과 언어 런타임의 논리 스레드(`GOMAXPROCS` 및 워커 수)가 어긋나면 CFS 스로틀링이 발생해 치명적인 꼬리 지연을 일으킵니다
- Go와 Java 가상 스레드는 "개발자에게 투명한 동기 블로킹 코드 제공과 런타임의 자동 I/O 흡수"라는 궁극의 해답으로 수렴했고, Rust는 극도의 제어력과 명시성을 선택했습니다. 2편의 좌표계로 보면 왼쪽 위(Go·Java)와 오른쪽 아래(Rust)로 갈라선 셈입니다
