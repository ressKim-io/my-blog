> **시리즈 "커널과 런타임으로 톺아보는 Rust · Go · Java"의 1편**
> 언어 성능 이야기는 벤치마크 숫자로 시작해서 결국 취향 싸움으로 끝나는 경우가 많습니다. 그래서 이 시리즈는 순서를 뒤집어 보려고 합니다. 숫자 대신, 세 언어가 똑같이 딛고 서 있는 바닥부터 봅니다. 커널에서 출발해 스레드 모델과 런타임, 메모리 할당과 GC를 지나 클라우드 비용까지, 아래에서 위로 하나씩 쌓아 올릴 예정입니다.
> 첫 편의 주제는 이 모든 이야기의 씨앗이 되는 시스템 콜(System Call)입니다. 왜 시스템 콜은 일반 함수 호출보다 10배 넘게 비쌀까요? 그리고 Rust, Go, Java는 이 비용을 각자 어떻게 감당하고 있을까요?

## 성능 이야기를 왜 커널에서 시작할까

파일을 읽고, 소켓에 쓰고, 메모리를 매핑하는 일. 어떤 언어로 짜든 서버가 하는 일의 본질은 결국 이것들입니다. 그리고 이 작업들은 전부 커널의 손을 빌려야만 할 수 있습니다. 다시 말해, 어떤 언어를 쓰든 피해 갈 수 없는 공통 비용의 맨 아래층에 시스템 콜이 있습니다.

그런데 프로파일링 결과(flame graph)를 열어 보면 `read`나 `write` 같은 시스템 콜이 애플리케이션 로직보다 굵은 핫스팟으로 잡히는 경우가 많습니다. 커널 안에서 하는 일 자체는 몇 줄 되지도 않는데 왜 이렇게 비쌀까요? 이번 편은 이 질문 하나를 붙잡고 바닥까지 내려갑니다. CPU가 왜 두 개의 실행 모드를 강제하는지 확인하고, `syscall` 명령어 한 줄이 실제로 무엇을 하는지 뜯어본 다음, 그 비용을 Rust, Go, Java가 각자 어떻게 감당하는지 순서로 다시 올라오겠습니다.

## CPU는 왜 두 개의 모드를 강제할까 — 보호 링과 예외 레벨

애플리케이션이 디스크 컨트롤러에 직접 명령을 내리거나, 아무 물리 메모리나 읽거나, 메모리 변환 테이블을 마음대로 바꿀 수 있다면 어떻게 될까요? 프로세스 격리도, 컨테이너 격리도, OS라는 개념 자체가 성립하지 않습니다. 그래서 모든 현대 CPU는 "누가 무엇을 할 수 있는가"를 하드웨어 회로 수준에서 강제합니다. 이걸 특권 분리라고 부릅니다.

이름은 아키텍처마다 다릅니다. x86-64는 보호 링(Protection Ring), ARM(AArch64)은 예외 레벨(Exception Level, EL)이라고 부릅니다. 클라우드에서 ARM 서버(AWS Graviton, Ampere Altra, Apple Silicon)가 주류로 올라선 지금은 두 모델을 나란히 봐 두는 편이 좋습니다.

![x86 protection rings vs ARM exception levels — two privilege ladders side by side](/diagrams/syscall-mode-switch-cost-1.svg)

그림의 두 사다리를 비교해 보겠습니다. 왼쪽 x86은 링이 4개(Ring 0~3) 있지만 실제로는 두 칸만 씁니다. 애플리케이션은 가장 권한이 낮은 Ring 3에서 돌고, 커널은 가장 권한이 높은 Ring 0에서 돌면서 하드웨어를 독점합니다. 중간의 Ring 1, 2는 사실상 버려진 칸입니다. CPU가 지금 어느 링에서 실행 중인지를 나타내는 값을 CPL(Current Privilege Level)이라고 부르는데, 앞으로 자주 나올 용어라 기억해 두시면 좋습니다.

오른쪽 ARM은 네 칸을 전부 씁니다. 앱은 EL0, 커널은 EL1, 하이퍼바이저는 EL2, 펌웨어와 시큐어 모니터(TrustZone)는 EL3에 삽니다. 여기서 두 설계의 철학 차이가 드러납니다. x86은 가상화를 링 사다리와 무관한 별도 모드(VMX root/non-root)로 나중에 덧붙인 반면, ARM은 하이퍼바이저와 시큐어 월드를 권한 사다리 안에 처음부터 설계해 넣었습니다. 지금은 사소해 보이는 이 차이가, 뒤에서 볼 KPTI(멜트다운 완화)가 arm64에서 더 싸게 먹히는 배경이 됩니다.

그럼 앱이 규칙을 어기고 특권 명령을 몰래 실행하면 어떻게 될까요? x86에서는 `HLT`(CPU 정지)나 `MOV to CR3`(메모리 변환 테이블 교체), 포트 I/O(`IN`/`OUT`) 같은 명령을 Ring 3에서 실행하는 순간 General Protection Fault(#GP)라는 하드웨어 예외가 터지고, 커널이 그 프로세스를 강제 종료합니다. ARM도 마찬가지입니다. 시스템 레지스터를 건드리는 `MSR` 같은 특권 명령이 EL0에서 실행되면 동기 예외(Synchronous Exception)가 발생해 실행이 EL1의 커널로 넘어갑니다.

여기서 중요한 건, 이 모드 구분이 소프트웨어끼리의 약속이 아니라 CPU 회로가 매 명령마다 검사하는 하드웨어 상태라는 사실입니다. 우회할 방법이 없습니다. 그래서 파일 읽기, 소켓 전송, 메모리 매핑처럼 특권이 필요한 작업은 반드시 커널에게 대신 해 달라고 요청해야 하고, 그 공식 요청 통로가 바로 시스템 콜입니다.

> **핵심**: 시스템 콜은 "조금 무거운 함수 호출"이 아닙니다. 함수 호출은 같은 권한 안에서 실행 위치만 옮기지만, 시스템 콜은 CPU를 다루는 권한 자체가 앱에서 커널로 넘어갔다가 되돌아옵니다. 하드웨어가 강제하는 이 경계(Ring 3→0, EL0→1)를 왕복하는 비용이, 이 글에서 다룰 모든 비용의 근원입니다.

## 관문의 실체 — syscall과 svc 명령어

과거의 리눅스는 `int 0x80`이라는 소프트웨어 인터럽트로 커널에 진입했습니다. 이 경로가 느려서, 현대 x86-64는 시스템 콜 전용 명령인 `syscall`과 짝이 되는 복귀 명령 `sysret`을 씁니다.

이 명령이 하는 일을 이해하려면 레지스터 몇 개의 역할부터 알아야 합니다. 잠깐 짚고 가겠습니다.

- **RIP** — "다음에 실행할 명령의 주소"를 담는 레지스터입니다. 이 값을 바꾸면 프로그램의 실행 위치가 그 주소로 점프합니다
- **RFLAGS** — 연산 결과나 인터럽트 허용 여부 같은 CPU 상태 플래그를 모아 둔 레지스터입니다
- **MSR**(Model Specific Register) — 일반 프로그램이 아니라 OS가 CPU의 동작 방식을 설정할 때 쓰는 특수 레지스터 모음입니다. "커널로 들어오는 문이 어디 있는지" 같은 값을 부팅 때 한 번 등록해 두는 곳이라고 생각하면 됩니다

이제 `syscall` 명령이 실제로 하는 일을 보겠습니다. 아래 네 가지 레지스터 이동이 명령 하나로 일어납니다.

![What the syscall instruction does — four register moves in one instruction](/diagrams/syscall-mode-switch-cost-2.svg)

1. 현재 `RIP`, 그러니까 복귀할 주소를 `rcx`에 저장합니다. 나중에 `sysret`이 이 값으로 되돌아옵니다
2. 현재 `RFLAGS`를 `r11`에 저장하고, `IA32_FMASK` MSR에 지정된 플래그를 끕니다. 인터럽트 플래그 정리 같은 일이 여기서 일어납니다
3. `IA32_LSTAR` MSR에 커널이 부팅 때 등록해 둔 커널 진입점 주소를 `RIP`에 로드합니다. 이 순간 실행 흐름이 커널 코드로 점프합니다
4. `IA32_STAR` MSR 값으로 코드/스택 세그먼트(`CS`/`SS`)를 커널용으로 교체합니다. CPL이 3에서 0으로 바뀌는, 특권 전환의 실체가 바로 이 단계입니다

정리하면 `syscall`은 "복귀 주소와 상태를 백업하고, 미리 등록된 커널 입구로 점프하면서, 특권 레벨을 올리는" 명령입니다. 관문의 실체는 이게 전부입니다.

그럼 커널에게 "무슨 시스템 콜을, 어떤 인자로" 요청하는지는 어떻게 전달할까요? 레지스터에 실어서 보냅니다. 리눅스 x86-64의 약속은 이렇습니다.

| 레지스터 | 역할 |
| --- | --- |
| `rax` | 시스템 콜 번호 (read=0, write=1, mmap=9 …) |
| `rdi` `rsi` `rdx` | 1~3번 인자 |
| `r10` | 4번 인자 |
| `r8` `r9` | 5~6번 인자 |
| `rax` (복귀 시) | 반환값, 음수면 `-errno` |

눈썰미가 있으시다면 이상한 점을 하나 발견하셨을 겁니다. 일반 함수 호출 규약(System V ABI)에서 4번 인자는 `rcx`인데, 시스템 콜에서는 왜 `r10`일까요? 위 그림의 1번 화살표가 답입니다. `syscall` 명령 자체가 복귀 주소를 저장하느라 `rcx`를 덮어써 버리기 때문에, 4번 인자가 갈 곳을 잃고 `r10`으로 밀려난 것입니다.

그리고 주의 깊게 봐야 할 사실이 하나 더 있습니다. 그림 하단에 적어 둔 것처럼, `syscall`은 스택 포인터(`rsp`)를 바꿔 주지 않습니다. 커널에 진입한 순간에도 `rsp`는 여전히 유저 스택을 가리키고 있습니다. 스택 교체와 메모리 변환 테이블 교체는 커널 소프트웨어가 손수 해야 하는 일이고, 그 "손수 하는 일들"이 다음 절에서 볼 진입 경로의 계단입니다.

ARM은 같은 개념을 다른 명령으로 구현합니다.

| 항목 | x86-64 | ARM AArch64 |
| --- | --- | --- |
| 진입 명령 | `syscall` | `svc #0` (Supervisor Call) |
| 콜 번호 | `rax` | `x8` |
| 인자 | `rdi` `rsi` `rdx` `r10` `r8` `r9` | `x0` ~ `x5` |
| 반환값 | `rax` | `x0` |
| 복귀 주소 | `rcx` (syscall이 저장) | `ELR_EL1` |
| 상태 저장 | `r11` (RFLAGS) | `SPSR_EL1` |
| 진입점 | `IA32_LSTAR` MSR | 예외 벡터 테이블 (`VBAR_EL1`) |

표에서 드러나는 철학 차이가 재미있습니다. x86의 `syscall`이 MSR에 등록된 단일 고속 진입점으로 곧장 점프하는 전용 명령이라면, ARM의 `svc`는 예외 벡터 테이블(`VBAR_EL1`)의 정해진 위치로 트랩되는 동기 예외입니다. 앞 절에서 앱이 특권 명령을 몰래 실행했을 때와 같은 예외 메커니즘을 재사용하는 셈입니다. x86은 "시스템 콜 전용 지름길"을 만들었고 ARM은 "기존 예외 체계에 편입"시켰지만, 권한을 한 칸 올려 커널로 들어간다는 본질과 비용 구조는 같습니다.

## 커널 진입은 공짜가 아니다 — 진입 경로의 계단

`syscall`로 커널에 점프한 직후, 커널은 실제 작업(`do_syscall_64`)을 부르기 전에 준비 작업을 순서대로 밟아야 합니다. 리눅스 커널의 `entry_64.S`라는 어셈블리 파일이 하는 일인데, 이 계단의 한 칸 한 칸이 전부 CPU 사이클을 태웁니다.

![syscall entry and return path — swapgs, KPTI CR3 swap, register save, speculation mitigations](/diagrams/syscall-mode-switch-cost-3.svg)

계단을 하나씩 풀어 보겠습니다.

1. **swapgs** — GS라는 레지스터의 베이스 주소를 교체합니다. 유저 공간에서 GS는 스레드마다 갖는 전용 저장소(TLS, Thread Local Storage)를 가리키는데, 커널은 같은 레지스터를 CPU 코어마다 갖는 전용 변수(per-CPU 변수, 현재 실행 중인 태스크 포인터 등)에 접근하는 용도로 씁니다. 하나의 레지스터를 두 세계가 다른 용도로 쓰다 보니 경계를 넘을 때마다 갈아 끼우는 것입니다
2. **KPTI 페이지 테이블 스왑** — 페이지 테이블은 프로그램이 보는 가상 주소를 실제 물리 메모리 주소로 변환하는 지도이고, CR3는 그 지도의 루트 주소를 담는 레지스터입니다. 멜트다운(Meltdown) 취약점 완화를 위해 리눅스는 유저용/커널용 지도를 분리해 두었고, 그래서 진입과 복귀 때마다 CR3를 갈아 끼웁니다. 시스템 콜을 특히 비싸게 만든 주범이라 아래에서 따로 해부합니다
3. **스택 교체** — 유저 `rsp`를 저장하고 커널 스택으로 `rsp`를 바꿉니다. 앞 절에서 봤듯 `syscall` 명령이 해 주지 않아서 커널이 직접 하는 일입니다
4. **레지스터 저장과 살균** — 커널 코드가 레지스터를 덮어쓰기 때문에 유저 값들을 커널 스택에 백업합니다. 그리고 커널의 정보가 유저에게 새어 나가지 않도록 일부 레지스터를 0으로 지웁니다
5. **투기적 실행 완화** — Spectre 계열 공격을 막기 위한 조치들입니다. 분기 예측기 상태를 제한하거나(IBRS), 리턴 주소 예측 버퍼를 비우거나(RSB untrain), 분기 히스토리를 지웁니다(BHB clear). 프로그램의 논리적 결과는 그대로지만, 캐시를 통한 정보 유출을 막는 대신 성능을 냅니다. 뒤에서 볼 간접 비용의 원천입니다

이 다섯 계단을 다 내려가야 비로소 `do_syscall_64`가 호출되고, 진짜 `read`/`write`가 실행됩니다. 그리고 복귀할 때 같은 계단을 역순으로 다시 밟습니다. 그림에서 빨간 박스(KPTI 스왑, 투기 실행 완화)가 진입과 복귀 양쪽에 다 있다는 점에 주목해 주세요. 이 보안 세금은 편도 요금이 아니라 왕복 요금입니다.

## 비용의 해부 — 직접 비용과 간접 비용

"시스템 콜이 느리다"는 말은 사실 두 층위를 섞어 부르는 표현입니다. 이 둘을 분리해야 최적화 판단이 섭니다.

### 직접 비용 — 고정 오버헤드

진입·복귀 경로에서 항상 실행되는 명령들의 비용입니다. 시스템 콜 종류와 무관하게 대체로 일정합니다.

크기를 가늠해 볼 좋은 방법이 있습니다. 리눅스에는 vDSO라는 장치가 있는데, `clock_gettime`처럼 자주 불리는 읽기 전용 시스템 콜의 코드를 커널이 유저 공간에 미리 매핑해 둔 것입니다. 덕분에 커널에 진입하지 않고 일반 함수처럼 호출됩니다. 같은 `clock_gettime`을 커널 경유와 vDSO 경유로 나눠 재면, 순수한 "커널 왕복 비용"이 드러납니다. 2.3GHz 노트북에서의 측정 예시입니다.

| 방식 | 호출당 사이클 |
| --- | --- |
| 직접 syscall (커널 진입) | 약 1,428 cycles |
| vDSO (유저 공간 함수처럼 호출) | 약 157 cycles |
| **차이** | **약 1,270 cycles (약 9배)** |

같은 일을 하는 코드인데 커널을 넘느냐 마느냐로 9배가 갈립니다. 절대 시간으로 보면 모드 전환의 바닥값은 대략 수백 나노초 수준입니다. 빠른 하드웨어에서는 100ns 아래로도 내려가지만, 보안 완화가 켜진 서버에서는 더 커집니다. "시스템 콜은 일반 함수 호출의 10배 이상"이라는 통념이 여기서 나옵니다.

한 가지 정직하게 짚어 두자면, 위 수치는 루프 안에서 반복 측정한 값이라 아래에서 볼 간접 비용이 섞여 다소 부풀려져 있을 수 있습니다. 그래서 간접 비용을 따로 봐야 합니다.

### 간접 비용 — CPU의 가속 상태가 무너진다

진짜 무서운 쪽은 이쪽입니다. 배경 지식 두 가지만 깔고 가겠습니다. 현대 CPU는 명령 하나를 20~30단계로 잘게 쪼개 컨베이어 벨트처럼 겹쳐 실행합니다(파이프라인). 그리고 if문 같은 분기를 만나면 결과를 기다리지 않고 "아마 이쪽일 것"이라고 찍고 미리 달립니다(분기 예측과 투기적 실행). 이 두 장치 덕분에 CPU는 정격 속도보다 훨씬 빠르게 코드를 처리합니다.

문제는 커널 진입이 이 가속 장치들을 무너뜨린다는 점입니다. 시스템 콜이 끝나고 유저 코드로 돌아온 뒤에도 한동안 코드가 느리게 도는 이유입니다.

| 교란 대상 | 무슨 일이 일어나나 | 체감 효과 |
| --- | --- | --- |
| 명령 파이프라인 (20~30 스테이지) | syscall은 앞선 명령이 모두 끝나야 통과하는 직렬화 지점이라, 진입 전에 파이프라인을 비움(drain) | 복귀 후 빈 파이프라인이 다시 찰 때까지 처리량 하락 |
| BHB (분기 히스토리 버퍼) | 보안 완화 조치로 지워짐 | 복귀 후 분기 예측 실패 증가 |
| RSB (리턴 스택 버퍼) | untrain됨 | 함수 리턴 예측이 실패해 느린 예측기로 폴백 |
| IBRS | 진입/복귀 시 켜고 끄는 명령 추가 | 직접 비용 가중 |

이 과정을 시간 축에 펼치면 이번 절의 핵심 통찰이 보입니다.

![Indirect cost timeline — after returning from a syscall, user code runs below its warmed-up speed until predictors retrain](/diagrams/syscall-mode-switch-cost-4.svg)

그림의 곡선은 유저 코드의 체감 실행 속도입니다. 시스템 콜을 만나면 파이프라인을 비우고 커널로 들어가느라 속도가 뚝 떨어지는데, 진짜 문제는 그다음입니다. 커널에서 돌아온 직후의 CPU는 분기 예측기가 지워지고 파이프라인이 빈, 말하자면 식어 버린 상태입니다. 그래서 유저 코드는 한동안 원래 속도(점선)에 못 미치는 채로 돌면서 예측기를 다시 훈련시켜야 합니다. 그림의 붉은 구간이 그 재예열 요금입니다.

정리하면, 시스템 콜의 진짜 비용은 커널 안에서 실행되는 몇 개의 명령이 아닙니다. CPU가 애써 쌓아 둔 가속 상태를 무너뜨리고, 복귀한 유저 코드가 그걸 다시 데우는 데 드는 시간입니다. flame graph에서 시스템 콜이 실제 하는 일에 비해 유독 굵은 핫스팟으로 잡히는 이유가 바로 이것입니다.

> 단, 최신 CPU는 enhanced IBRS 같은 하드웨어 완화를 내장해서 BHB 클리어나 RSB untrain이 아무 일도 안 하는 명령(no-op)이 되기도 합니다. 즉 이 비용은 CPU 세대와 커널 설정에 따라 크게 달라집니다. "우리 서버에서는 얼마인가"는 측정해 봐야 아는 값입니다.

## KPTI — 시스템 콜마다 걷어가는 보안 세금

2018년 공개된 멜트다운(Meltdown) 취약점은 유저 프로세스가 투기적 실행을 악용해 커널 메모리를 훔쳐볼 수 있게 했습니다. 리눅스의 대응이 KPTI(Kernel Page Table Isolation)입니다. 유저용 페이지 테이블에서 커널 매핑을 아예 걷어내서, 유저 모드에서는 커널 메모리가 지도에 존재하지도 않게 만드는 물리적 차단입니다.

![KPTI before and after — kernel mappings removed from the user page table, CR3 swapped on every crossing](/diagrams/syscall-mode-switch-cost-5.svg)

그림 왼쪽이 KPTI 이전의 세계입니다. 프로세스는 페이지 테이블을 하나만 갖고, 그 지도 안에 유저 매핑과 커널 매핑이 함께 삽니다. 커널 영역은 권한 비트로만 가려져 있었는데, 멜트다운은 투기적 실행을 이용해 바로 이 가림막을 우회했습니다. 지도에 존재하기는 하니까, 훔쳐볼 틈이 생긴 것입니다.

오른쪽이 KPTI가 켜진 지금의 기본값입니다. 페이지 테이블을 유저용과 커널용 둘로 쪼갭니다. 유저용 지도에는 커널로 들어가는 데 필요한 최소한의 코드(trampoline)만 남기고 커널 매핑을 전부 걷어냅니다. 이제 유저 모드에서는 커널 메모리가 지도에 아예 없으니, 투기적 실행으로도 볼 수 없습니다. 대신 시스템 콜이 드나들 때마다 CR3를 두 지도 사이에서 갈아 끼워야 합니다.

문제는 이 스왑의 대가입니다. CPU는 주소 변환 결과를 TLB라는 전용 캐시에 담아 두는데, CR3를 갈아 끼우는 행위는 이 캐시를 무효화시킬 수 있습니다. PCID/INVPCID라는 하드웨어 기능 덕분에 전면 무효화는 피하지만, 그래도 비용이 남습니다. 규모 감을 잡아 보면 CPU당 초당 5만 syscall 수준에서 약 2%, 시스템 콜과 인터럽트가 집약된 워크로드에서는 1~10%까지 CPU를 더 태운다는 보고가 있습니다. 세금이라는 비유가 정확한 것이, 시스템 콜 호출 빈도에 정확히 비례해서 청구됩니다.

> **ARM 각주 — arm64의 KPTI가 더 싼 이유**: AArch64는 유저와 커널의 주소 변환을 애초부터 TTBR0_EL1(유저), TTBR1_EL1(커널)이라는 두 레지스터로 분리해 둡니다. 그래서 KPTI를 적용해도 x86의 CR3 전면 스왑보다 구조적으로 가볍습니다. 게다가 멜트다운에 취약한 ARM 코어는 일부(Cortex-A75 등)뿐이라, arm64에서 KPTI는 취약 코어에 한해 선택적으로 켜집니다. Graviton 같은 ARM 서버의 시스템 콜 세금이 x86 대비 상대적으로 낮은 배경 중 하나입니다. 다만 Spectre 계열 분기 예측 완화는 ARM도 동일하게 부담합니다.

> **DevOps 함의**: "이 서비스가 왜 CPU를 더 먹지?"의 답이 애플리케이션 코드가 아니라 시스템 콜 빈도 × 커널 완화 세금일 수 있습니다. `perf stat`으로 컨텍스트 스위치와 syscall 카운트를 봐야 비로소 보이는 비용입니다.

## 회피 전략 — 결국 "커널을 덜 넘는 것"

비용의 근원이 모드 전환(crossing) 그 자체라면, 답은 하나로 수렴합니다. 넘는 횟수를 줄이는 것. 모든 고성능 I/O 설계의 공통 목표입니다.

![Four patterns for reducing kernel crossings — batching, vDSO, io_uring, eBPF](/diagrams/syscall-mode-switch-cost-6.svg)

그냥 두면 작업 하나마다 관문을 한 번씩 넘습니다(그림 맨 왼쪽). 여기서 벗어나는 길은 셋입니다. 여러 작업을 모아 한 번에 넘거나(배칭), 커널 코드를 유저 공간으로 내려보내 넘을 일 자체를 없애거나(vDSO), 반대로 내 로직을 커널 안으로 올려보내는 것입니다(eBPF). 구체적인 도구로 펼치면 이렇습니다.

| 전략 | 원리 | 대표 사례 |
| --- | --- | --- |
| vDSO | 자주 쓰는 읽기 전용 syscall을 유저 공간에 매핑해 커널 진입 없이 호출 | `clock_gettime`, `gettimeofday` |
| 버퍼드 I/O | raw `read`/`write` 대신 유저 버퍼에 모아서 한 번에 | stdio, `BufReader` |
| scatter/gather | 여러 버퍼를 syscall 한 번으로 처리 | `readv` / `writev` |
| io_uring | 유저와 커널이 공유하는 큐에 I/O 요청을 쌓아 두면 커널이 배치로 처리 | 고성능 네트워크/디스크 I/O |
| eBPF | 로직 일부를 커널 안으로 올려 crossing 자체를 제거 | 필터링, 관측, 정책 |

표에서 io_uring은 배칭을 극한까지 밀어붙인 경우라 한마디 보태겠습니다. 유저와 커널이 메모리를 공유하는 큐 두 개(제출 큐, 완료 큐)를 만들어 두고, 앱은 요청을 큐에 적기만 하고 커널은 결과를 큐에 채워 넣습니다. 요청 여러 개가 crossing 한 번에 실려 가고, 폴링 모드까지 동원하면 crossing이 0에 수렴합니다.

이 표를 외울 필요는 없지만, 표의 존재 이유는 기억해 두시면 좋습니다. 뒤 편에서 만날 Go의 netpoller(epoll), Java NIO(epoll), Rust tokio(epoll/io_uring)가 전부 "이벤트 하나마다 커널을 넘지 않겠다"는 같은 동기에서 나온 설계이기 때문입니다.

## 세 언어는 이 관문을 어떻게 대하는가

이제 커널에서 언어의 세계로 올라올 차례입니다. 이 절이 이 시리즈의 척추입니다. 같은 시스템 콜 관문을 세 언어가 어떻게 다르게 대하는지 보고 나면, 이후 모든 성능 서사가 이 차이에서 자라난다는 걸 알게 됩니다. 앱 코드가 `read()`를 부르는 순간을 따라가 보겠습니다.

![How Rust, Go and Java each cross the kernel gate when calling read()](/diagrams/syscall-mode-switch-cost-7.svg)

**Rust는 커널과 직거래합니다.** `std::fs::read`는 libc의 `read()` 래퍼를 거쳐 곧장 syscall로 갑니다. 런타임의 개입이 없습니다. 블로킹 syscall이면 그 OS 스레드가 커널 안에서 그냥 멈춰 기다립니다. 얇고 예측 가능하다는 것이 최대 장점이고, 블록된 스레드가 아무 일도 못 하고 논다는 것이 대가입니다. 그 대가를 채우려고 나온 것이 tokio 같은 비동기 런타임인데, 2편에서 다룹니다.

**Go는 런타임이 관문을 감쌉니다.** Go는 리눅스에서 libc조차 거치지 않고 직접 syscall을 발행합니다. 대신 그 전후를 `runtime.entersyscall`과 `runtime.exitsyscall`이 감쌉니다. 고루틴이 블로킹 syscall에 들어가면 런타임이 P(논리 프로세서)를 그 OS 스레드에서 떼어내 다른 고루틴에게 넘겨 버립니다(handoff). 네트워크 I/O는 아예 netpoller(epoll)로 논블로킹 전환해서, 수만 개의 고루틴이 소수의 OS 스레드를 나눠 쓰게 만듭니다. 런타임이 관문을 가로채 유저 레벨에서 스케줄링한다는 이 발상이 GMP 모델의 씨앗이고, 2편과 3편의 주인공입니다.

**Java는 관문 위에 추상화를 쌓아 왔습니다.** Java의 I/O는 JVM 네이티브 코드와 libc를 거쳐 syscall에 도달합니다(`System.nanoTime` 같은 것은 vDSO를 탑니다). 전통적인 blocking I/O는 플랫폼 스레드(OS 스레드와 1:1)를 통째로 붙잡는 방식이었고, 그 한계를 NIO(Selector, 내부는 epoll)로 우회했고, 이제는 가상 스레드(Virtual Thread, Project Loom)가 블로킹 지점에서 캐리어 스레드를 놓아주는(unmount) 단계까지 왔습니다. 무거운 런타임이 관문 위에 여러 세대의 추상화를 겹겹이 쌓아 온 역사입니다.

세 줄로 요약하면 이렇습니다.

- **Rust**: 커널과 직거래하는 얇은 층. 비용이 투명하고 예측 가능
- **Go**: 런타임이 syscall을 가로채 유저 레벨 스케줄링으로 블로킹을 흡수
- **Java**: 런타임이 관문 위에 여러 세대의 추상화(blocking → NIO → Loom)를 쌓음

## 트레이드오프 정리

| 관점 | Rust | Go | Java |
| --- | --- | --- | --- |
| 커널 접근 층 | 얇음 (libc 직거래) | 런타임이 감쌈 | 런타임 + 추상화 겹층 |
| 블로킹 syscall 시 | OS 스레드가 블록 | P handoff로 흡수 | Loom이 unmount (최신) |
| 비용 투명성 | 높음 (예측 가능) | 중간 (런타임 개입) | 낮음 (추상화 뒤에 숨음) |
| 최적화 여지 | 수동·명시적 | 런타임이 자동 처리 | JIT + 런타임이 자동 |
| 대가 | 블록된 스레드의 낭비 | 스케줄러 복잡도 | 런타임 무게, 웜업 |

표를 관통하는 결론은 하나입니다. syscall이라는 물리적 비용은 세 언어 모두 똑같이 냅니다. 커널 관문 앞에서는 어떤 언어도 특별 대우를 받지 않습니다. 차이는 "그 비용을 누가, 어느 층에서 관리하느냐"입니다. Rust는 개발자에게, Go는 런타임 스케줄러에게, Java는 무거운 런타임과 JIT에게 맡깁니다.

그리고 이 관리 주체의 차이가 다음 편들에서 볼 체급 차이(런타임의 무게), 웜업 차이(JIT), 클라우드 비용 차이(메모리 베이스라인)로 그대로 이어집니다. 시리즈가 끝날 때까지 이 문장을 계속 확인하게 되실 겁니다.

## DevOps 관점에서 더 파고들 질문

지식을 실무 계측과 운영 감각으로 연결해 보고 싶다면, 이런 질문들을 파 볼 만합니다.

1. **계측** — `strace -c ./app`과 `perf stat -e raw_syscalls:sys_enter ./app`으로 syscall 횟수를 세면 애플리케이션 로직 대비 "커널 왕복"의 비중이 보입니다. 내 서비스의 syscall/sec는 KPTI 세금(1~10%)을 걱정할 수준일까요?
2. **컨테이너 보안 vs 비용** — 컨테이너 런타임은 seccomp-bpf로 허용 syscall을 필터링하는데, 이 필터는 모든 syscall에 BPF 검사 비용을 추가합니다. 공격 표면 축소와 오버헤드 사이의 트레이드오프를 어디서 잡아야 할까요?
3. **완화 스위치** — 신뢰된 워크로드만 도는 노드라면 `mitigations=off`로 부팅해 syscall 비용을 크게 줄일 수 있습니다. 어떤 노드(예: 격리된 배치 처리 노드)에서 사이드채널 위험을 감수할 수 있을까요?
4. **vDSO와 가상화** — 클라우드 VM에서 vDSO의 `clock_gettime`이 항상 유저 공간에서 처리될까요? `clocksource`가 `tsc`가 아니면 vDSO의 이점이 사라질 수 있습니다.
5. **Go 스케줄러와 syscall 폭주** — Go 서비스에서 블로킹 syscall이 급증하면 왜 P 분리와 OS 스레드 증가로 이어져 스케줄러 지연과 컨텍스트 스위치가 늘어날까요? (2편의 GMP에서 답을 찾게 됩니다)
6. **io_uring 도입 판단** — 초당 수십만 I/O를 하는 프록시라면 io_uring이 crossing을 줄여 CPU를 얼마나 되찾아 줄까요? 반대로 복잡도와 커널 버전 의존성이라는 비용은 얼마일까요?

## 핵심 요약

- 시스템 콜은 실행 위치만 옮기는 함수 호출과 다릅니다. CPU를 다루는 권한이 앱에서 커널로 넘어갔다 되돌아오는 왕복이고, 그 경계(Ring 3→0, EL0→1)는 하드웨어가 강제합니다.
- `syscall` 명령의 실체는 네 가지 레지스터 이동입니다. 복귀 주소와 상태를 백업하고, 부팅 때 등록된 커널 입구로 점프하면서 특권 레벨을 올립니다.
- 진입·복귀 경로의 계단(swapgs → KPTI CR3 스왑 → 스택 교체 → 레지스터 저장 → 투기 실행 완화)이 직접 비용을 만들고, 파이프라인과 분기 예측기가 리셋되는 간접 비용이 복귀 후에도 이어집니다.
- KPTI(멜트다운 완화)는 시스템 콜마다 걷어가는 세금으로, syscall 집약 워크로드에서 1~10%의 CPU를 가져갑니다. arm64는 TTBR0/1 분리 구조 덕에 상대적으로 쌉니다.
- 고성능 I/O 설계(vDSO, io_uring, eBPF, epoll)는 전부 crossing 횟수 줄이기라는 하나의 목표로 수렴합니다.
- Rust는 커널과 직거래하고, Go는 런타임이 관문을 감싸 유저 레벨 스케줄링으로 흡수하며, Java는 관문 위에 여러 세대의 추상화를 쌓았습니다. 비용은 같고, 관리 주체가 다릅니다.
