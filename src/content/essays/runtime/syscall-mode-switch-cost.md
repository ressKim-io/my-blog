> **시리즈 "커널과 런타임으로 깊이 파헤치는 Rust · Go · Java"의 1편**
> 프로그래밍 언어의 성능을 이야기할 때 단순한 벤치마크 숫자 비교나 취향 다툼으로 끝나는 경우가 많습니다. 그래서 이 시리즈는 순서를 바꿔 접근합니다. 숫자를 나열하는 대신, 세 언어가 똑같이 딛고 서 있는 하부 구조부터 단계별로 살펴봅니다. 운영체제의 핵심인 커널에서 출발해 스레드 모델과 런타임, 메모리 할당과 가비지 컬렉션(GC)을 지나 클라우드 인프라 비용까지 아래에서 위로 차근차근 쌓아 올립니다
> 첫 번째 편의 주제는 이 모든 성능 이야기의 출발점이 되는 시스템 콜(System Call, 운영체제 기능 요청)입니다. 왜 시스템 콜은 일반 함수 호출보다 10배 넘게 비쌀까요? 그리고 Rust, Go, Java는 이 무거운 비용을 각자 어떤 방식으로 감당하고 있을까요?

## 성능 이야기를 왜 커널에서 시작해야 할까

우리가 만드는 서버 애플리케이션의 핵심 역할은 결국 파일을 읽고, 네트워크 소켓을 통해 데이터를 보내고, 메모리를 할당받아 쓰는 일입니다. 어떤 프로그래밍 언어를 쓰든 상관없이, 이러한 핵심 작업들은 모두 운영체제의 중심인 커널(Kernel)에게 부탁해야만 실행할 수 있습니다. 다시 말해 어떤 언어를 선택하든 절대 피해 갈 수 없는 공통 비용의 가장 아래층에 바로 시스템 콜이 자리하고 있습니다

실제로 서버 성능 분석기(Flame Graph)를 열어 보면 `read`나 `write` 같은 시스템 콜이 우리가 작성한 비즈니스 로직보다 훨씬 넓은 면적을 차지하는 경우가 많습니다. 커널 내부에서 수행되는 실제 작업 코드는 몇 줄 되지 않는데, 왜 이렇게 무거운 성능 하락을 일으킬까요

이번 편에서는 이 의문을 풀기 위해 컴퓨터 하드웨어의 바닥까지 내려갑니다. CPU가 왜 안전망을 위해 두 개의 실행 권한 모드를 분리했는지 살펴보고, 시스템 콜 전용 명령어(`syscall`)의 동작 구조를 뜯어본 뒤, 이 비용을 Rust, Go, Java가 어떻게 각자의 철학으로 다루는지 확인하겠습니다

## CPU는 왜 두 개의 실행 모드를 강제할까 — 보호 링과 권한 분리

회사 건물을 예로 들어 보겠습니다. 일반 직원 카드로는 사무실 문만 열리고, 전산실이나 배전반처럼 잘못 건드리면 건물 전체가 멈추는 구역은 관리자 카드로만 열립니다. 이 등급 구분이 없다면 누구나 배전반 스위치를 내릴 수 있고, 건물의 안전은 유지될 수 없습니다

CPU도 같은 문제를 안고 있습니다. 만약 일반 애플리케이션이 디스크나 네트워크 하드웨어를 마음대로 조작하거나, 물리 메모리의 아무 영역이나 훔쳐보고, 주소 변환표를 고칠 수 있다면 프로세스 간의 격리는 물론 컨테이너나 운영체제(OS) 자체의 안전성이 전혀 유지될 수 없습니다. 그래서 모든 현대 CPU는 "현재 실행 중인 코드가 하드웨어를 다룰 권한이 있는가"를 회로 수준에서 철저하게 감시하고 통제합니다. 이를 '권한 분리(Privilege Separation)'라고 부릅니다

이 권한 분리 계층을 부르는 이름은 CPU 설계 방식(아키텍처)마다 다릅니다. x86-64 아키텍처는 '보호 링(Protection Ring)'이라 부르고, ARM(AArch64) 아키텍처는 '예외 레벨(Exception Level, EL)'이라고 부릅니다. 최근 클라우드 환경에서 ARM 기반 서버(AWS Graviton, Apple Silicon 등)의 비중이 커진 만큼 두 모델의 차이를 나란히 비교해 볼 필요가 있습니다

![x86 protection rings vs ARM exception levels — two privilege ladders side by side](/diagrams/syscall-mode-switch-cost-1.svg)

위 그림의 두 권한 사다리를 비교해 보겠습니다. 왼쪽 x86 구조는 총 4개의 링(Ring 0~3)을 만들어 두었지만 실제로는 양극단의 두 칸만 사용합니다. 일반 애플리케이션은 가장 권한이 낮은 `Ring 3`에서 실행되고, 운영체제의 핵심인 커널은 가장 권한이 높은 `Ring 0`에서 실행되며 하드웨어를 독점합니다. 중간의 Ring 1과 Ring 2는 사실상 사용되지 않습니다. 현재 CPU가 어느 권한 등급에서 실행 중인지를 나타내는 지표를 '현재 권한 등급(CPL, Current Privilege Level)'이라 부르며, 앞으로 자주 등장할 핵심 용어입니다

오른쪽 ARM 구조는 네 칸을 목적에 맞게 모두 활용합니다. 일반 애플리케이션은 `EL0`에, 커널은 `EL1`에, 가상화 하이퍼바이저는 `EL2`에, 하드웨어 보안 제어기(TrustZone)는 가장 높은 `EL3`에 위치합니다. 여기서 두 설계의 철학 차이가 드러납니다. x86은 가상화 기능을 권한 사다리와 별개의 모드(VMX root/non-root)로 나중에 덧붙인 반면, ARM은 하이퍼바이저와 보안 구역을 처음부터 권한 계층 안에 체계적으로 설계했습니다. 이러한 구조적 차이는 이후 설명할 멜트다운 보안 방어(KPTI) 비용이 ARM 서버에서 더 저렴하게 처리되는 배경이 됩니다

그렇다면 일반 애플리케이션(`Ring 3`나 `EL0`)이 규칙을 어기고 특권 명령어를 몰래 실행하면 어떻게 될까요. x86에서는 CPU 정지(`HLT`)나 주소록 교체(`MOV to CR3`) 같은 특권 명령을 부르는 순간 즉시 하드웨어 보호 오류(General Protection Fault, `#GP`)가 발생하며, 커널이 해당 프로세스를 강제로 종료시킵니다. ARM 역시 동일하게 동작합니다. 시스템 설정을 건드리는 특권 명령이 `EL0`에서 실행되면 '동기 예외(Synchronous Exception)'가 터지면서 제어권이 즉시 `EL1`의 커널로 넘어가 프로그램을 중단시킵니다

여기서 기억해야 할 핵심은 이 권한 구분이 소프트웨어 사이의 구두 약속이 아니라, CPU 회로가 명령어 매 줄을 실행할 때마다 검사하는 절대적인 하드웨어 장벽이라는 점입니다. 회사 카드 리더가 등급을 매번 자동으로 확인하는 것과 같습니다. 따라서 파일 읽기나 네트워크 전송처럼 하드웨어 접근이 꼭 필요한 작업은 반드시 커널에게 대신 해 달라고 공식 요청을 보내야 합니다. 이 유일하고 공식적인 요청 통로가 바로 '시스템 콜(System Call)'입니다

> **핵심**: 시스템 콜은 단순한 '조금 무거운 함수 호출'이 아닙니다. 일반 함수 호출은 같은 권한 등급 안에서 코드 실행 위치만 이동하지만, 시스템 콜은 CPU를 다루는 권한 모드 자체를 애플리케이션(`Ring 3` / `EL0`)에서 커널(`Ring 0` / `EL1`)로 올렸다가 다시 내려오는 위험하고 복잡한 왕복 과정입니다. 하드웨어가 강제하는 이 권한 장벽을 넘나드는 왕복 비용이 바로 우리가 겪는 모든 성능 손실의 근본 원인입니다

## 관문의 실체 — syscall과 svc 명령어

과거의 리눅스는 인터럽트 번호(`int 0x80`)를 호출하는 방식으로 커널 문을 두드렸습니다. 하지만 이 방식은 처리 과정이 너무 느려, 현대의 x86-64 CPU는 시스템 콜 진입 전용 명령어인 `syscall`과 복귀 전용 명령어인 `sysret`을 별도로 도입했습니다

이 명령어가 작동하는 원리를 이해하려면 우선 CPU 내부의 임시 저장소인 레지스터 몇 가지의 역할을 알아야 합니다

- **RIP (명령어 포인터)** — CPU가 "다음에 실행할 명령어의 메모리 주소"를 담아두는 곳입니다. 이 값을 바꾸면 프로그램의 실행 위치가 해당 주소로 점프합니다
- **RFLAGS (상태 플래그)** — 연산 결과나 인터럽트 허용 여부 등 현재 CPU의 상태 정보들을 모아 둔 공간입니다
- **MSR (모델 특수 레지스터)** — 일반 애플리케이션이 아닌 운영체제가 CPU의 세부 동작 방식을 설정할 때 쓰는 관리용 레지스터입니다. "커널로 들어가는 공식 입구 주소가 어디인가" 같은 핵심 설정값을 부팅 시 한 번 저장해 두는 장소입니다

이제 `syscall` 명령어를 실행할 때 CPU 회로 내부에서 어떤 동작이 동시에 일어나는지 살펴보겠습니다

![What the syscall instruction does — four register moves in one instruction](/diagrams/syscall-mode-switch-cost-2.svg)

1. 현재 실행 중이던 애플리케이션 주소(`RIP`), 즉 나중에 돌아올 복귀 주소를 임시 보관 레지스터(`rcx`)에 자동으로 백업합니다
2. 현재의 상태 플래그(`RFLAGS`)를 또 다른 레지스터(`r11`)에 백업하고, 관리자 설정(`IA32_FMASK MSR`)에 따라 인터럽트 차단 등 필요한 플래그를 정리합니다
3. 부팅 때 커널이 미리 등록해 둔 공식 입구 주소(`IA32_LSTAR MSR`)를 `RIP`에 덮어씁니다. 이 순간 실행 위치가 커널 영역으로 즉시 점프합니다
4. 코드 세그먼트 설정(`IA32_STAR MSR`)을 커널용으로 교체합니다. 이때 CPL(현재 권한 등급) 숫자도 3에서 0으로 바뀌며 특권 모드 전환이 완성됩니다

정리하자면 `syscall`은 "복귀 주소와 상태를 안전하게 백업하고, 커널 공식 입구로 점프하면서, 권한 레벨을 최고 등급으로 올리는" 전용 명령어입니다. 커널 관문 진입의 물리적 실체는 바로 이 네 가지 레지스터 교체입니다

그렇다면 커널에게 "어떤 기능(예: 파일 읽기)을, 어떤 설정값으로" 실행해 달라고 알리려면 어떻게 해야 할까요. 호출 인자들을 미리 약속된 레지스터에 싣고 출발하면 됩니다. 리눅스 x86-64 환경의 약속표는 다음과 같습니다

| 레지스터 | 역할 |
| --- | --- |
| `rax` | 요청할 시스템 콜 번호 (read=0, write=1, mmap=9 등) |
| `rdi` `rsi` `rdx` | 1~3번 매개변수 |
| `r10` | 4번 매개변수 |
| `r8` `r9` | 5~6번 매개변수 |
| `rax` (복귀 후) | 반환값 (에러 발생 시 음수 형태의 에러 번호) |

표를 자세히 살펴보면 한 가지 특이한 부분을 발견할 수 있습니다. 일반적인 C 함수 호출 규칙에서는 4번 매개변수를 `rcx` 레지스터에 넣는데, 시스템 콜에서는 왜 `r10`을 사용할까요. 앞서 설명한 1번 동작 때문입니다. `syscall` 명령어 자체가 복귀 주소를 기억하기 위해 `rcx` 레지스터를 자동으로 덮어써 버리므로, 4번 매개변수를 안전하게 지키기 위해 `r10`으로 자리를 옮긴 것입니다

또한 놓치기 쉬운 아주 중요한 하드웨어 동작이 있습니다. `syscall` 명령어는 스택 메모리 주소를 가리키는 스택 포인터(`rsp`)를 자동으로 바꿔 주지 않습니다. 커널 모드로 진입한 직후에도 `rsp`는 여전히 권한이 낮은 애플리케이션 스택을 가리키고 있습니다. 따라서 스택 주소를 안전한 커널 스택으로 교체하고 보호하는 작업은 온전히 커널 소프트웨어가 손수 처리해야 하며, 이 작업들이 바로 다음 절에서 살펴볼 진입 준비 과정입니다

ARM 아키텍처에서는 동일한 개념을 다른 명령어로 구현합니다

| 구분 | x86-64 아키텍처 | ARM AArch64 아키텍처 |
| --- | --- | --- |
| 진입 명령어 | `syscall` | `svc #0` (Supervisor Call) |
| 시스템 콜 번호 | `rax` | `x8` |
| 매개변수 전달 | `rdi` `rsi` `rdx` `r10` `r8` `r9` | `x0` ~ `x5` |
| 반환값 수신 | `rax` | `x0` |
| 복귀 주소 저장 | `rcx` 레지스터 | `ELR_EL1` 레지스터 |
| 상태 플래그 저장 | `r11` 레지스터 | `SPSR_EL1` 레지스터 |
| 커널 진입점 | `IA32_LSTAR MSR` 설정값 | 예외 벡터 테이블 (`VBAR_EL1`) |

두 아키텍처의 철학 차이가 흥미롭습니다. x86은 커널로 바로 점프할 수 있는 고속 전용 통로(`syscall`)를 별도 설계한 반면, ARM은 앞서 살펴본 예외 처리 시스템(`VBAR_EL1`)의 규칙을 재활용하여 시스템 콜(`svc`)도 동기 예외의 한 종류로 세련되게 통합했습니다. 진입 방식의 모양은 다르지만, 권한을 올려 커널 영역으로 건너가는 본질적 비용 구조는 정확히 일치합니다

## 커널 진입의 숨은 대가 — 진입 경로의 계단

`syscall` 명령어로 커널 영역에 들어왔다고 해서 곧바로 우리가 원했던 `read`나 `write`가 실행되는 것은 아닙니다. 리눅스 커널은 실제 작업 함수(`do_syscall_64`)를 호출하기 전, 안전과 보안을 위해 엄격한 준비 절차(`entry_64.S` 코드)를 밟습니다. 이 준비 과정의 매 단계가 모두 연산 자원을 소모합니다

![syscall entry and return path — swapgs, KPTI CR3 swap, register save, speculation mitigations](/diagrams/syscall-mode-switch-cost-3.svg)

커널 진입 경로의 5단계 준비 과정을 하나씩 풀어보겠습니다

1. **GS 레지스터 용도 전환 (swapgs)** — CPU 레지스터 중 `GS`라는 공간은 두 세계에서 쓰임새가 완전히 다릅니다. 일반 애플리케이션은 스레드별 개별 저장소(TLS, Thread Local Storage)를 가리키기 위해 `GS`를 쓰고, 커널은 CPU 코어별 고유 변수(현재 실행 중인 태스크 정보 등)를 찾기 위해 `GS`를 씁니다. 하나의 공간을 두 세계가 다르게 쓰다 보니, 모드가 바뀔 때마다 주소값을 서로 맞바꾸는 `swapgs` 작업을 반드시 거쳐야 합니다
2. **주소록 루트 교체 (KPTI 페이지 테이블 스왑)** — 가상 메모리 주소를 물리 주소로 바꿔주는 변환표(페이지 테이블)의 최상위 위치는 `CR3` 레지스터가 쥐고 있습니다. 멜트다운 보안 취약점을 막기 위해 현대 리눅스는 애플리케이션용 주소록과 커널용 주소록을 완전히 분리했습니다. 그 결과 커널로 들어가고 나올 때마다 `CR3` 값을 교체해야 하는 부담이 추가되었습니다
3. **스택 교체 및 백업** — 앞서 보았듯 `syscall`이 스택을 바꿔주지 않으므로, 커널은 애플리케이션의 스택 주소(`rsp`)를 안전한 곳에 백업하고 커널 전용 스택으로 주소를 변경합니다
4. **레지스터 저장 및 민감 정보 청소** — 커널 로직이 기존 레지스터 값들을 덮어쓰기 전에 애플리케이션의 데이터들을 커널 스택에 안전하게 보관합니다. 반대로 복귀할 때는 커널 내부의 민감한 메모리 정보가 외부로 유출되지 않도록 레지스터를 깨끗하게 지웁니다
5. **투기적 실행 방어 조치** — CPU가 분기를 예측하여 미리 연산하는 '투기적 실행'을 악용한 해킹(Spectre 취약점)을 막기 위한 보안 조치들입니다. 분기 예측기의 기능을 제한하거나(`IBRS`), 리턴 주소 예측 버퍼를 청소하고(`RSB untrain`), 분기 기록을 초기화(`BHB clear`)합니다. 정보 유출을 원천 봉쇄하는 대신 CPU의 가속 능력을 크게 떨어뜨립니다

이 다섯 계단을 모두 안전하게 내려가야 비로소 실제 요청한 작업이 수행됩니다. 작업이 끝나면 이 계단들을 역순으로 밟아 다시 애플리케이션으로 돌아옵니다. 위 그림의 붉은색 상자(KPTI 주소록 교체, 투기적 실행 방어)는 진입과 복귀 양쪽에서 모두 실행됩니다. 즉 이 무거운 보안 세금은 편도가 아닌 왕복 요금으로 매번 청구됩니다

## 비용의 해부 — 직접 비용과 간접 비용

우리가 흔히 "시스템 콜이 무겁다"고 표현할 때는 두 가지 서로 다른 성격의 비용이 섞여 있습니다. 이 둘을 구분해야 어디서 성능을 개선할 수 있을지 정확히 판단할 수 있습니다

### 직접 비용 — 관문을 넘는 고정 오버헤드

진입과 복귀 경로에서 준비 계단을 밟느라 소모되는 고정된 시간입니다. 어떤 시스템 콜을 호출하든 기본적으로 발생하는 공통 통행료입니다

이 고정 비용을 깔끔하게 증명해 주는 메커니즘이 리눅스의 '가상 시스템 콜(vDSO, Virtual Dynamic Shared Object)'입니다. `clock_gettime`처럼 현재 시간 조회용으로 아주 자주 호출되는 읽기 전용 작업은, 커널이 애플리케이션 메모리 영역에 코드를 미리 공유해 둡니다. 덕분에 커널 모드로 넘어가지 않고 일반 함수 부르듯 실행할 수 있습니다. 2.3GHz 노트북에서 같은 시간 조회 기능을 커널 진입 방식과 vDSO 방식으로 비교한 결과는 다음과 같습니다

| 실행 방식 | 호출 1회당 소모 사이클 |
| --- | --- |
| 커널 진입 시스템 콜 (일반 syscall) | 약 1,428 사이클 |
| 유저 공간 가상 호출 (vDSO 사용) | 약 157 사이클 |
| **순수 모드 전환 왕복 비용** | **약 1,270 사이클 (약 9배 차이)** |

완전히 동일한 연산을 수행하는데도, 커널 장벽을 왕복하느냐 여부에 따라 속도가 9배나 벌어집니다. 절대적인 시간으로 계산하면 모드 전환의 기본 비용은 수백 나노초(ns) 수준입니다. 최신 CPU에서는 수십 나노초까지 줄어들기도 하지만, 보안 방어 옵션이 켜진 클라우드 서버 환경에서는 이 통행료가 훨씬 커집니다. 우리가 "시스템 콜은 일반 함수 호출보다 10배 이상 비싸다"고 부르는 근거가 바로 여기에 있습니다

다만 위 실험 데이터는 루프를 돌며 연속으로 재는 과정에서 아래에 설명할 간접 비용 일부가 상쇄된 수치입니다. 실제 운영 서버에서 겪는 진짜 아픔은 다음 절의 간접 비용에서 찾아옵니다

### 간접 비용 — CPU의 가속 엔진이 식어버린다

시스템 콜이 일으키는 더 치명적인 손실은 CPU 내부의 가속 엔진을 리셋시킨다는 점입니다. 현대 CPU는 명령어 하나를 20~30개 단계로 잘게 나누어 공장의 컨베이어 벨트처럼 겹쳐서 고속 처리합니다(파이프라인). 또한 조건문을 만나면 결정을 기다리지 않고 "과거 통계상 이쪽 갈림길일 것"이라 예측하여 미리 계산을 진행합니다(분기 예측과 투기적 실행). 이 가속 기술들 덕분에 CPU는 표시된 클럭 속도보다 훨씬 더 많은 명령어를 처리합니다

문제는 커널 모드 진입이 이러한 가속 엔진의 흐름을 완전히 끊어버린다는 사실입니다. 시스템 콜 작업이 다 끝나고 우리 코드로 돌아온 이후에도 한동안 서버 처리 속도가 둔해지는 이유가 바로 여기에 있습니다

| 파괴되는 가속 장치 | 커널 왕복 중 발생하는 일 | 우리 코드 복귀 직후 겪는 체감 피해 |
| --- | --- | --- |
| 명령어 파이프라인 (20~30단계) | 시스템 콜은 이전 명령이 다 끝나야 통과하는 직렬화 지점이라 컨베이어 벨트를 비움 | 복귀 후 빈 파이프라인 벨트가 다시 가득 찰 때까지 연산 처리량 급감 |
| 분기 히스토리 버퍼 (`BHB`) | 보안 차단 조치로 분기 예측 기록을 강제 삭제 | 복귀 후 우리 코드의 if문 갈림길에서 예측 실패율 폭증 |
| 리턴 주소 예측 버퍼 (`RSB`) | 함수 복귀 주소 예측 기록을 초기화 | 함수에서 리턴할 때 예측이 빗나며 느린 기본 검색으로 후퇴 |
| 간접 분기 예측 방어 (`IBRS`) | 진입과 복귀 때마다 방어 회로를 켜고 끄는 제어 명령 실행 | 명령어 실행 자체의 고정 비용 추가 소모 |

이 과정을 시간 흐름에 따라 시각화하면 우리가 왜 시스템 콜을 주의해야 하는지 명확히 이해할 수 있습니다

![Indirect cost timeline — after returning from a syscall, user code runs below its warmed-up speed until predictors retrain](/diagrams/syscall-mode-switch-cost-4.svg)

위 그래프의 곡선은 우리 애플리케이션 코드가 겪는 실제 체감 속도입니다. 시스템 콜을 만나는 순간 컨베이어 벨트가 비워지며 속도가 바닥으로 떨어집니다. 하지만 진짜 문제는 복귀 직후입니다. 커널에서 막 돌아온 CPU는 분기 예측 데이터가 지워지고 파이프라인이 텅 빈, 말 그대로 엔진이 차갑게 식어버린 상태입니다. 따라서 우리 코드는 한동안 본래 속도(점선)를 내지 못한 채 예측기를 다시 학습시키는 '예열 요금(그래프의 붉은 영역)'을 지불해야 합니다

정리하자면 시스템 콜의 진짜 비용은 커널 안에서 실행되는 몇 줄의 I/O 로직이 아닙니다. CPU가 열심히 달구어 놓은 가속 엔진을 일순간에 얼어붙게 만들고, 복귀한 애플리케이션 코드가 그 엔진을 다시 데우는 데 버려지는 숨은 시간입니다

> 단, 최신 CPU들은 이러한 분기 예측 차단 비용을 줄이기 위해 하드웨어 내장 방어 회로(`enhanced IBRS` 등)를 탑재하고 있습니다. 이 경우 소프트웨어가 강제로 버퍼를 청소하는 명령어들이 빈 동작(`no-op`)으로 처리되어 손실이 줄어듭니다. 즉 간접 비용의 크기는 운영 중인 서버 CPU 세대와 커널 패치 상태에 따라 크게 달라집니다

## KPTI — 멜트다운 방어를 위해 시스템 콜마다 지불하는 세금

2018년에 세상에 알려진 멜트다운(Meltdown) 취약점은 CPU의 투기적 실행 결함을 악용하여 일반 프로세스가 커널 메모리의 비밀 키나 암호화 데이터를 훔쳐볼 수 있게 한 치명적 보안 사고였습니다. 이를 막기 위해 리눅스는 '커널 페이지 테이블 격리(KPTI, Kernel Page Table Isolation)'를 도입했습니다. 일반 애플리케이션이 실행될 때는 커널 영역의 주소표 자체를 아예 제거하여 물리적으로 훔쳐볼 틈을 없애버린 강력한 조치입니다

![KPTI before and after — kernel mappings removed from the user page table, CR3 swapped on every crossing](/diagrams/syscall-mode-switch-cost-5.svg)

그림 왼쪽은 KPTI 도입 이전의 주소록 구조입니다. 프로세스는 단 하나의 통합 주소 변환표를 가졌고, 그 안에 일반 영역과 커널 영역이 함께 들어 있었습니다. 커널 구역은 권한 비트로만 접근을 가려두었는데, 멜트다운 공격은 투기적 실행의 허점을 이용해 이 권한 가림막을 뚫고 데이터를 읽어냈습니다

그림 오른쪽은 KPTI가 적용된 현재의 표준 구조입니다. 주소 변환표를 애플리케이션용과 커널용으로 아예 두 권으로 나눴습니다. 애플리케이션용 주소록에는 커널 입구로 진입하기 위한 아주 작은 발판(`trampoline`) 코드만 남기고 나머지 커널 주소는 전부 지웠습니다. 이제 일반 모드에서는 커널 데이터가 주소록에 존재하지 않으므로 해킹이 불가능합니다. 하지만 그 대가로 시스템 콜을 드나들 때마다 주소록의 위치 레지스터(`CR3`)를 매번 번갈아 끼워야 하는 큰 짐이 생겼습니다

이 주소록 교체 작업(`CR3` 스왑)이 미치는 영향은 가볍지 않습니다. CPU는 주소 변환 속도를 높이기 위해 전용 캐시(`TLB`)를 사용하는데, 주소록을 바꾸면 이 캐시 정보가 얽힐 위험이 있습니다. 다행히 현대 CPU는 주소록마다 고유 태그(`PCID`)를 붙여 캐시가 통째로 지워지는 현상은 막아주지만, 주소록 루트 교체 자체의 연산 부담은 남습니다. 일반적으로 초당 5만 회의 시스템 콜이 발생하는 서버에서는 약 2%, I/O 요청이 집중되는 무거운 서버에서는 최대 10%의 CPU 자원이 이 보안 세금으로 사라집니다

> **ARM 아키텍처 각주 — ARM 서버의 KPTI 세금이 저렴한 이유**: ARM AArch64 아키텍처는 처음 설계할 때부터 일반 주소록 레지스터(`TTBR0_EL1`)와 커널 주소록 레지스터(`TTBR1_EL1`)를 하드웨어적으로 아예 분리해 두었습니다. 따라서 KPTI 격리를 켜더라도 x86처럼 주소록 레지스터 하나를 통째로 교체하는 과중한 부담이 없습니다. 게다가 멜트다운에 취약한 ARM 코어 설계는 일부 모델에 불과하여, 안전한 ARM 서버에서는 KPTI가 켜지지 않거나 최소한으로만 동작합니다. AWS Graviton 같은 ARM 서버가 시스템 콜이 잦은 환경에서 x86 대비 높은 효율을 보이는 중요한 기술적 이유입니다

> **운영 실무 팁**: "우리 서버 코드는 단순한데 왜 이렇게 CPU 실사용률(`sys` CPU)이 높을까?"라는 고민이 생긴다면, 애플리케이션 로직이 아니라 과도한 시스템 콜 호출 카운트와 KPTI 보안 세금의 곱셈이 원인일 수 있습니다. 리눅스 계측 도구인 `perf stat`으로 컨텍스트 스위칭 횟수와 시스템 콜 발생량을 확인해 보면 실마리를 찾을 수 있습니다

## 커널 관문 회피 전략 — 결국 "선 넘는 횟수 줄이기"

성능 손실의 근본 원인이 권한 경계를 넘나드는 모드 전환 왕복 그 자체라면, 해결책은 명확한 하나로 귀결됩니다. 바로 '장벽을 넘는 횟수 자체를 최소화하는 것'입니다. 오늘날 사용되는 모든 고성능 입출력(I/O) 라이브러리와 프레임워크는 바로 이 목표를 향해 진화해 왔습니다

![Four patterns for reducing kernel crossings — batching, vDSO, io_uring, eBPF](/diagrams/syscall-mode-switch-cost-6.svg)

가장 단순하게 구현하면 데이터 묶음 하나를 처리할 때마다 커널 장벽을 한 번씩 왕복하게 됩니다(그림 맨 왼쪽). 이 비효율에서 벗어나는 길은 크게 세 가지 방향입니다. 여러 요청을 모아서 한 번에 건너거나(배치 처리), 자주 쓰는 커널 코드를 애플리케이션 영역으로 내려보내 왕복을 없애거나(`vDSO`), 반대로 우리 비즈니스 로직의 일부를 커널 안으로 올려보내 장벽 내부에서 처리(`eBPF`)하는 것입니다

| 회피 전략 | 작동 원리 | 실무 활용 사례 |
| --- | --- | --- |
| 가상 시스템 콜 (`vDSO`) | 읽기 전용으로 자주 호출되는 커널 로직을 유저 메모리에 매핑하여 모드 전환 없이 즉시 실행 | 시간 조회(`clock_gettime`), 시스템 프로파일링 |
| 버퍼 입출력 (Buffered I/O) | 바이트 단위의 잦은 호출 대신 메모리 버퍼에 데이터를 모아 한 번에 통째로 전송 | 표준 C 라이브러리의 `stdio`, Rust의 `BufReader` |
| 다중 영역 전송 (Scatter/Gather) | 떨어진 여러 메모리 공간의 데이터를 묶어 단 1회의 시스템 콜로 동시 처리 | 분산 버퍼 읽기/쓰기(`readv` / `writev`) |
| 비동기 I/O 링 (`io_uring`) | 일반 영역과 커널 영역이 공유하는 원형 큐에 작업만 쌓아두고 커널이 모아서 일괄 처리 | 고성능 대용량 데이터베이스, 최신 웹 서버 프록시 |
| 커널 내 안전 실행 (`eBPF`) | 모래상자(Sandbox) 검증을 거친 우리 로직을 커널 안으로 직접 적재하여 데이터 가공을 내부에서 완결 | 고성능 패킷 필터링, 실시간 서버 관측(Observability) |

특히 `io_uring`은 배치 처리의 효율을 극단까지 끌어올린 혁신적인 메커니즘입니다. 애플리케이션과 커널이 함께 바라보는 우체통 역할을 하는 원형 큐 2개(작업 제출용 큐, 결과 수신용 큐)를 메모리에 만들어 둡니다. 애플리케이션은 편지봉투(작업 요청)를 큐에 밀어 넣기만 하고, 커널은 편지를 처리한 뒤 결과 봉투를 완료 큐에 쌓습니다. 수백 개의 I/O 요청이 단 한 번의 커널 진입으로 처리되며, 폴링(상시 감시) 모드까지 켜면 커널 모드 전환 횟수가 `0`에 수렴하는 극적인 속도 향상을 얻습니다

이 기술들의 세부 명칭을 다 외울 필요는 없지만, 이러한 회피 전략들이 존재하는 근본 이유는 꼭 기억해야 합니다. 이후 글에서 자세히 살펴볼 Go 언어의 네트워크 스케줄러(`netpoller`), Java의 고성능 논블로킹 I/O(`NIO epoll`), Rust의 비동기 런타임(`tokio`)이 모두 "네트워크 이벤트 하나 때문에 무겁게 커널 문을 드나들지 않겠다"는 똑같은 고민에서 탄생했기 때문입니다

## Rust · Go · Java는 이 관문을 어떻게 다루는가

이제 커널 밑바닥에서 프로그래밍 언어들의 층위로 올라올 차례입니다. 이 비교는 이번 전체 시리즈를 지탱하는 중심축입니다. 물리적인 하드웨어 장벽 앞에서는 어떤 언어도 특혜를 받지 못합니다. 하지만 애플리케이션 코드가 데이터를 읽기 위해 `read()`를 호출하는 순간, 세 언어가 커널 관문을 대하는 방식은 완전히 갈라집니다

![How Rust, Go and Java each cross the kernel gate when calling read()](/diagrams/syscall-mode-switch-cost-7.svg)

**Rust는 커널과 중간 계층 없이 직거래합니다**

Rust 표준 라이브러리의 `std::fs::read`는 C 표준 라이브러리(`libc`)의 함수를 호출하고, 이는 곧바로 `syscall` 명령어로 이어져 커널에 도달합니다. 언어가 숨겨둔 별도의 런타임 간섭이 전혀 없습니다. 만약 호출한 작업이 대기해야 하는 블로킹 작업이라면, 해당 운영체제 스레드는 커널 안에서 잠든 채 얌전히 대기합니다

구조가 얇아 실행 비용이 완전히 투명하고 예측 가능하다는 것이 최고의 장점이지만, 대기 상태에 빠진 스레드가 다른 일을 하지 못한 채 메모리와 자원을 묶어둔다는 것이 대가입니다. 이 대가를 해결하기 위해 Rust 생태계는 개발자가 명시적으로 비동기 런타임(`tokio` 등)을 조립해 쓰는 방식을 선택했습니다

**Go는 거대한 런타임이 커널 관문을 완벽히 감쌉니다**

Go 언어는 리눅스 환경에서 C 표준 라이브러리(`libc`)조차 거치지 않고, 자체 작성한 어셈블리로 `syscall` 명령어를 직접 쏩니다. 그리고 이 모든 호출 전후를 Go 런타임의 관리 함수(`runtime.entersyscall`, `runtime.exitsyscall`)가 꼼꼼하게 감싸고 있습니다

만약 어떤 고루틴(Go의 경량 스레드)이 커널 안에서 대기해야 하는 상황을 만나면, Go 런타임은 즉시 해당 OS 스레드에서 실행 권한(`P`, 논리 프로세서)을 떼어내어 대기 중인 다른 고루틴에게 넘겨버립니다(핸드오프). 특히 네트워크 I/O 작업은 내부적으로 커널 이벤트 감시기(`netpoller epoll`)를 이용해 논블로킹 방식으로 자동 변환하므로, 수십만 개의 고루틴이 단 몇 개의 OS 스레드 위에서 멈춤 없이 쾌적하게 굴러갑니다. 커널 관문 진입을 런타임이 가로채서 자체 스케줄링으로 완충한다는 이 아이디어가 바로 Go `GMP` 모델의 핵심이며, 2편에서 그 실체를 자세히 뜯어봅니다

**Java는 관문 위에 여러 세대의 추상화 계층을 쌓아 왔습니다**

Java 애플리케이션의 I/O 요청은 JVM 내부의 네이티브 코드를 거쳐 C 표준 라이브러리를 통해 시스템 콜에 도달합니다(물론 `System.nanoTime()` 같은 시간 조회는 고속 `vDSO`를 탑니다)

초기 Java의 블로킹 I/O는 OS 스레드 하나를 통째로 묶어두는 무거운 방식이었습니다. 대용량 트래픽에서 이 한계에 부딪히자 다중 채널 감시 기능인 `NIO(Selector)`를 도입해 효율을 높였고, 최근 21 버전 이후에는 가상 스레드(`Virtual Thread, Project Loom`)를 탑재했습니다. 이제 Java의 가상 스레드가 I/O 대기를 만나면, JVM 런타임이 알아서 짐을 내려놓고(`unmount`) 바탕이 되는 실제 OS 스레드가 다른 가상 스레드를 실행하도록 돕습니다. 거대한 JVM 런타임이 커널 관문 위에 30년간 진화해 온 추상화 계층들을 겹겹이 쌓아 올린 역사입니다

세 언어의 접근 방식을 요약하면 다음과 같습니다

- **Rust**: 커널과 1:1로 직거래하는 얇은 계층. 비용 구조가 완벽히 투명하고 예측 가능함
- **Go**: 런타임이 모든 시스템 콜을 감시하고 가로채, 경량 스레드 스케줄링으로 대기 비용을 흡수함
- **Java**: 런타임 위에 전통적 블로킹부터 NIO, 최신 가상 스레드까지 여러 세대의 추상화를 두껍게 적층함

## 트레이드오프와 철학 비교

| 비교 관점 | Rust | Go | Java |
| --- | --- | --- | --- |
| 커널 접근 계층 | 얇음 (`libc` 직거래) | 런타임이 감싸고 통제함 | JVM 런타임 + 추상화 겹층 |
| 블로킹 대기 발생 시 | OS 스레드 자체가 대기 상태 진입 | 논리 프로세서 분리(`P handoff`)로 완충 | 가상 스레드 분리(`unmount`)로 완충 |
| 비용의 투명성 | 매우 높음 (작성한 대로 동작) | 중간 (런타임의 개입 존재) | 낮음 (추상화 계층 뒤에 숨음) |
| 성능 최적화 주체 | 개발자가 명시적으로 제어 | Go 런타임이 자동으로 관리 | JVM 런타임과 `JIT` 컴파일러가 관리 |
| 지불해야 하는 대가 | 대기 스레드 자원 관리 복잡도 | 스케줄러 자체의 CPU 오버헤드 | 무거운 런타임 메모리와 예열(`Warm-up`) 시간 |

이 표가 이야기하는 결론은 분명합니다. 시스템 콜(`syscall`)이라는 하드웨어 장벽을 넘는 물리적 비용은 세 언어 모두 1원도 빠짐없이 똑같이 지불합니다. 커널 관문 앞에서는 어떤 프로그래밍 언어도 특별 할인을 받지 못합니다

결정적인 차이는 **"그 무거운 왕복 비용과 대기 시간을 누구의 손에 맡겨 관리하느냐"**에 있습니다. Rust는 그 제어권을 개발자에게 투명하게 넘기고, Go는 런타임의 스케줄러에게 일임하며, Java는 정교하게 적층된 JVM과 실시간 컴파일러(`JIT`)에게 맡깁니다

그리고 이 관리 주체의 선택이 바로 다음 편들에서 살펴볼 언어별 메모리 체급 차이(런타임 무게), 부팅 직후의 속도 차이(예열 문제), 그리고 클라우드 인프라 청구서의 차이로 직결됩니다

## 실무 운영 관점에서 짚어볼 질문들

여기서 익힌 커널 밑바닥의 지식을 실제 서버 장애나 성능 최적화 실무와 연결해 보고 싶다면, 다음과 같은 질문들을 함께 파고들어 볼 수 있습니다

1. **내 서비스의 실제 계측** — 리눅스 명령어 `strace -c ./app`이나 `perf stat -e raw_syscalls:sys_enter ./app`을 실행해 보면, 우리 애플리케이션이 로직 연산보다 커널 왕복에 얼마나 많은 시간을 쓰는지 직접 눈으로 확인할 수 있습니다. 현재 운영 중인 서버의 호출 빈도가 KPTI 보안 세금을 걱정해야 할 수준인지 점검해 보셨나요
2. **컨테이너 보안과 오버헤드 균형** — 도커나 쿠버네티스의 컨테이너 보안 도구(`seccomp-bpf`)는 악성 시스템 콜을 막기 위해 모든 커널 진입점마다 필터링 검사를 끼워 넣습니다. 보안을 위해 추가되는 이 필터링 오버헤드와 서버 성능 사이의 타협점을 어디로 잡아야 할까요
3. **분리된 워크로드에서의 보안 옵션 조정** — 만약 외부 인터넷과 단절된 내부 데이터 분석 배치 노드라면, 부팅 설정(`mitigations=off`)으로 투기적 실행 방어 조치를 끄고 시스템 콜 속도를 획기적으로 높일 수 있습니다. 우리 인프라 중 어떤 서버가 이러한 타협을 안전하게 선택할 수 있을까요
4. **가상화 환경에서의 `vDSO` 동작 여부** — 클라우드 가상 머신(VM) 환경에서는 시간 동기화 설정(`clocksource`)이 하드웨어 직접 클럭(`tsc`)으로 잡혀 있어야 `vDSO`가 유저 영역에서 고속으로 동작합니다. 설정이 꼬여 매초 수천 번의 로그 타임스탬프 조회가 무겁게 커널을 드나들고 있지는 않은지 확인해 보셨나요
5. **Go 스케줄러와 과도한 시스템 콜** — Go로 만든 API 서버에서 외부 I/O 대기가 급증할 때, 왜 Go 런타임이 OS 스레드를 계속 새로 만들며 컨텍스트 스위칭이 폭증하는 현상이 일어날까요? (2편에서 다룰 `GMP` 스케줄러 구조에서 그 명쾌한 해답을 찾을 수 있습니다)

## 핵심 요약

- **권한 모드의 물리적 분리** — 시스템 콜은 단순한 함수 호출이 아닙니다. CPU 하드웨어가 강제하는 보호 장벽(`Ring 3` → `Ring 0` / `EL0` → `EL1`)을 넘어 권한을 올렸다 내리는 무거운 왕복 과정입니다
- **`syscall` 명령어의 동작** — 단 한 줄의 `syscall` 실행 뒤에는 복귀 주소 백업, 커널 공식 입구 점프, 권한 레벨 상향이라는 하드웨어 레지스터 교체가 동시에 일어납니다
- **직접 비용과 간접 비용** — 주소록 교체(`CR3`), 스택 교체, 레지스터 백업 같은 진입 준비 단계가 직접 비용을 만들고, 파이프라인 컨베이어 벨트가 비워지고 분기 예측 데이터가 초기화되는 가속 엔진 파괴가 복귀 후에도 이어지는 큰 간접 비용을 낳습니다
- **멜트다운 방어 세금 (`KPTI`)** — 보안 취약점을 막기 위해 주소 변환표를 완전히 분리하면서 시스템 콜 호출 빈도에 비례해 CPU 자원의 1~10%가 통행 세금으로 지불됩니다. ARM 아키텍처는 하드웨어 구조 덕에 이 세금이 상대적으로 가볍습니다
- **언어별 커널 관문 대처법** — Rust는 중간 개입 없이 얇게 직거래하고, Go는 런타임이 관문을 감싸 경량 스레드로 완충하며, Java는 관문 위에 여러 세대의 추상화 계층(`Virtual Thread` 등)을 적층했습니다. 비용은 모두 같고 관리 주체만 다릅니다
