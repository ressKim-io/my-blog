## JVM은 무겁고 Rust는 가볍다, 그런데 정확히 무엇이?

"JVM은 무겁고, Go는 가볍고, Rust는 훨씬 더 가볍다"

개발 현장에서 자주 듣는 말이지만 막상 그 속을 따져 보면 모호합니다. 바이너리 파일 크기가 크다는 뜻일까요, 실행 중에 메모리를 많이 차지한다는 뜻일까요, 아니면 프로그램 시동이 느리다는 뜻일까요?

이 질문들을 하나로 뭉뚱그려 버리면 "가장 무겁다는 JVM이 정점 처리량에서는 가장 빠르다"와 같은 역설적인 문장 앞에서 방향을 잃게 됩니다

그래서 이번 편에서는 먼저 런타임이라는 단어의 정확한 범위를 한정하고, "무겁다"는 개념을 다섯 가지 독립된 축으로 분해합니다. 그다음 세 언어가 각자 어떤 무게를 왜 기꺼이 짊어졌는지 낱낱이 파헤쳐 보겠습니다

## 런타임이란 정확히 무엇인가

현장에서 런타임이라는 단어는 두 가지 의미로 혼용되곤 합니다. 혼란을 피하기 위해 구분부터 명확히 하겠습니다

- **런타임 라이브러리(Runtime Library)** — `printf`, 문자열 처리, 컬렉션 자료구조처럼 개발자가 직접 호출해서 사용하는 표준 라이브러리 함수들입니다. 이는 모든 프로그래밍 언어에 존재합니다
- **언어 런타임(Language Runtime)** — 개발자가 명시적으로 호출하지 않아도 프로그램 실행과 동시에, 그리고 애플리케이션 코드 밑바닥에서 자동으로 도는 관리형 엔진 코드입니다

이 글에서 다루는 핵심 주제는 바로 후자인 언어 런타임입니다. 이 계층 안에 어떤 요소들이 배치될 수 있는지 살펴보겠습니다

![What can live in the language runtime layer between app code and the kernel](/diagrams/runtime-weight-zero-cost-vs-fat-1.svg)

그림 가운데에 위치한 층이 바로 언어 런타임입니다. 이 안에 들어갈 수 있는 핵심 모듈 후보들을 하나씩 풀면 다음과 같습니다

- **메모리 관리 엔진** — GC(Garbage Collector, 가비지 컬렉터)와 동적 메모리 할당자입니다. GC는 더 이상 참조되지 않는 힙 메모리를 실행 도중에 주기적으로 추적하여 회수하는 자동 청소 장치입니다
- **스케줄러 계층** — 2편에서 다룬 Go의 GMP나 Java 가상 스레드처럼, 값싼 유저 레벨 태스크를 소수의 물리 OS 스레드에 효율적으로 배차해 주는 제어 층입니다
- **JIT 컴파일러** — 프로그램이 실행되는 도중에 자주 호출되는 핫스팟 바이트코드를 실시간으로 분석해 최적화된 네이티브 기계어로 새로 굽어 내는 동적 컴파일러입니다
- **리플렉션 및 런타임 타입 정보(RTTI)** — 프로그램이 도는 중에도 객체의 타입 구조와 메서드 정보를 조회하고 동적으로 조작할 수 있게 돕는 메타데이터 관리 시스템입니다
- **예외 언와인딩(Unwinding) 및 패닉 처리기** — 치명적인 오류나 예외가 발생했을 때 호출 스택을 역순으로 거슬러 올라가며 할당된 자원과 메모리를 안전하게 정리하는 복구 절차입니다
- **동적 디스패치(Dynamic Dispatch)** — 다형성 호출을 처리할 때 어떤 구현체의 함수를 호출할지 실행 도중에 vtable(가상 함수 테이블, 동적 디스패치용 함수 포인터 목록)을 조회하여 결정하는 기법입니다
- **클래스 로더 및 바이트코드 인터프리터** — 컴파일된 중간 코드(`.class`)를 동적으로 읽어 들여 한 줄씩 해석하고 구동하는 실행 엔진입니다

핵심 질문은 단 하나로 귀결됩니다. **"이 중간 계층에 무엇을, 그리고 얼마나 채워 넣었는가?"**

Rust는 이 계층을 거의 완전히 비워 냈습니다. Go는 실속 있는 엔진들만 골라 단단하게 채웠으며, JVM은 가상 머신이라는 하나의 거대한 운영체제 급 플랫폼을 통째로 얹었습니다

## "무겁다"의 다섯 가지 의미

언어별 물리적 무게를 달아보기 전에 저울의 눈금을 다섯 개로 세분화해야 합니다

여기서 베이스라인 RSS(Resident Set Size, 상주 물리 메모리 크기)는 애플리케이션이 외부 요청을 전혀 받지 않고 유휴 상태로 대기할 때 프로세스가 실제로 점유하는 물리 메모리 양을 의미합니다. 정상상태 오버헤드는 애플리케이션이 연산을 수행하는 동안 런타임이 백그라운드에서 유지 유지비로 계속 소모하는 CPU 및 메모리 자원이며, 정점 처리량은 예열이 완전히 끝난 후 낼 수 있는 최대 초당 처리 성능(Peak Throughput)입니다

| 축 | Rust | Go | JVM (전통 JIT) | JVM (native-image) |
| --- | --- | --- | --- | --- |
| 바이너리·이미지 크기 | 아주 작음 | 중간 | 큼 (JRE 필요) | 작음 |
| 베이스라인 RSS | 아주 작음 | 작음 | 큼 | 작음 |
| 시동 (Cold Start) | ms | ms | 초 단위 (웜업 필요) | ms |
| 정상상태 오버헤드 | 거의 0 | GC + 스케줄러 | GC + JIT 컴파일러 스레드 | GC |
| 정점 처리량 (Peak) | 높음 | 중상 | 최고 (JIT 적응 최적화) | 중 (-10~25%) |

표를 세로 방향으로 읽어 내려가면 우리가 가진 첫인상이 그대로 증명됩니다. Rust 열은 거의 모든 축에서 지표가 매우 작고 가볍지만, JVM 전통 열은 거의 모든 축에서 무겁고 큽니다

그러나 맨 마지막 행인 정점 처리량에 도달하면 이 순서가 완전히 극적으로 뒤집힙니다. 위쪽 네 가지 축에서 가장 무겁다는 비판을 받았던 전통 JVM이 정점 처리량 항목에서는 당당히 가장 높은 성능을 기록합니다

무거운 초기 비용을 지불하고 실행 중에 코드를 깎아 내는 **런타임 적응성(Adaptability)**을 샀기 때문이며, 이 반전 매커니즘이 바로 다음 2부의 핵심 주제입니다

`native-image` 열의 지표 변화도 눈여겨볼 만합니다. 바이너리 크기와 상주 메모리, 시동 시간은 Go 수준으로 가벼워졌지만, 그 대가로 정점 처리량이 기존 JIT 대비 10~25%가량 낮아졌습니다

동일한 JVM 생태계 내부에서도 초기 무게와 장기 처리량을 서로 맞바꾸는 명확한 트레이드오프가 작동하고 있습니다

이 역학 관계를 하나의 스펙트럼 화살표로 펼치면 다음과 같은 통찰을 얻게 됩니다

![The weight spectrum flips when you read it as peak throughput](/diagrams/runtime-weight-zero-cost-vs-fat-2.svg)

위쪽 화살표는 초기 런타임의 무게입니다. 오른쪽으로 이동할수록 바이너리가 커지고 베이스라인 물리 메모리 점유가 높으며 시동 시간이 오래 걸립니다

아래쪽 화살표는 웜업 이후 장기 운용 시 달성하는 정점 처리량입니다. 화살표의 방향이 위쪽과 정확히 일치합니다

즉 무게 순서의 가장 오른쪽 끝, 가장 무거운 전통 JVM이 수 시간에서 수일간 쉬지 않고 달리는 대규모 엔터프라이즈 서버 환경에서는 오히려 가장 뛰어난 연산 효율을 발휘할 수 있습니다

**"무게는 죄가 아니라 전략적인 교환 결과물이다"**라는 이번 편의 결론이 이 도표 한 장에 요약되어 있습니다

## 세 언어는 이 층을 어떻게 채웠나

본격적인 세부 해부에 앞서 세 언어의 전체 아키텍처 지도를 먼저 펼쳐보겠습니다. 각 언어가 런타임 층을 구성한 두께와 배치가 근본적으로 다릅니다

![How thick each language's runtime layer is and where it lives](/diagrams/runtime-weight-zero-cost-vs-fat-3.svg)

그림 왼쪽의 Rust부터 확인해 보겠습니다. 애플리케이션 코드가 곧 하드웨어가 실행할 수 있는 네이티브 기계어이며, 커널과의 사이에는 극히 얇은 초기 설정 코드만이 존재합니다

사실상 언어 런타임 층이 완전히 비어 있는 제로코스트 구조입니다

가운데에 위치한 Go는 하나의 정적 컴파일 바이너리 안에 애플리케이션 코드와 언어 런타임이 함께 밀봉되어 들어갑니다. 유저 스케줄러와 GC, 메모리 할당자, 네트워크 폴러가 모두 그 단일 바이너리 내부에 패키징됩니다

따라서 중간 계층은 꽤 두껍지만, 바이너리 외부 환경에 어떤 가상 머신이나 별도 의존성 파일도 요구하지 않습니다

오른쪽의 JVM은 아키텍처의 패러다임 자체가 다릅니다. 애플리케이션 코드는 CPU 기계어가 아니라 가상 머신용 바이트코드(`.class`) 형태로 배포되며, 이를 읽고 실행해 줄 별도의 무거운 가상 머신 프로세스가 사전에 구동되어야 합니다

세 언어 중 유일하게 런타임 프로세스가 애플리케이션 본체보다 훨씬 더 큰 부피를 차지하는 구조입니다

이 세 가지 도면이 곧 세 언어의 철학적 뿌리입니다. 이제 각 언어가 왜 이러한 구조를 택했는지 깊이 뜯어보겠습니다

## Rust — 런타임을 거의 비웠다

### 제로코스트 추상화가 런타임을 금지한다

Rust는 C++ 시대부터 내려온 엄격한 시스템 프로그래밍 원칙을 그대로 이어받았습니다

> "내가 직접 사용하지 않는 기능에 대해서는 어떠한 실행 시간 비용도 지불하지 않는다. 그리고 내가 사용하는 고수준 추상화 기능은 숙련된 개발자가 저수준에서 손으로 직접 작성한 코드보다 더 빠르거나 최소한 동등해야 한다"

제네릭, 이터레이터 체이닝, 비동기(`async`) 같은 고수준 문법을 쓰더라도 컴파일 결과물은 런타임 오버헤드가 정확히 '0'인 기계어로 번역되어야 한다는 뜻입니다

이 원칙은 Rust가 언어 내부에 GC 같은 관리형 엔진을 두는 것을 엄격히 금지합니다. 만약 언어 런타임에 GC가 기본 탑재된다면 GC가 전혀 필요 없는 단순 수치 연산이나 임베디드 모듈조차 메모리 추적과 스레드 중지(STW)라는 부당한 비용을 강제로 지불해야 하기 때문입니다

### 그렇다면 Rust 런타임에는 무엇이 남았나

Rust 프로그램의 `main()` 함수가 실행되기 직전, `lang_start`라는 매우 얇은 진입점 코드가 잠깐 실행됩니다. 이 코드가 수행하는 역할은 스택 오버플로를 방지하기 위한 가드 페이지(Guard Page) 설정, 패닉 발생 시 동작할 훅 준비, 명령행 인자 파싱 및 TLS(Thread Local Storage, 스레드 로컬 저장소) 초기화가 전부입니다

그리고 그 작업이 끝나는 순간 런타임은 완벽하게 퇴장합니다. GC 없음, M:N 유저 스케줄러 없음, JIT 동적 컴파일러 없음이 물리적 현실입니다

그렇다면 GC 없이 무수한 힙 객체의 동적 메모리를 누가 언제 안전하게 해제할까요? Rust는 이 메모리 수명 관리의 짐을 프로그램 '실행 시점(Runtime)'에서 컴파일러가 작동하는 '개발 시점(Compile-time)'으로 완전히 이전했습니다

컴파일러 내장의 소유권(Ownership) 및 빌림 검사기(Borrow Checker)가 소스 코드를 분석하며 "이 힙 메모리는 어느 스코프가 끝날 때 해제하는 것이 100% 안전한가"를 수학적으로 증명합니다. 그리고 판정이 끝난 정확한 소멸 시점에 메모리 반납 기계어(`drop()`)를 알아서 끼워 넣습니다

자원의 할당과 소멸 수명을 변수의 스코프에 엄격하게 묶는 RAII(Resource Acquisition Is Initialization, 자원 획득 즉시 초기화 및 스코프 기반 수명 관리) 기법의 완성형입니다

메모리 해제 시점을 실행 도중에 동적으로 추적할 필요가 없으니 무거운 GC 스레드가 힙을 스캔할 이유가 사라집니다. 이 수명 관리의 구체적인 동작 원리는 시리즈 후반부 메모리 편에서 저수준 `malloc`/`free` 레벨까지 깊이 파고듭니다

추상화 기능들 역시 똑같은 방식으로 런타임 비용을 소거합니다. 예를 들어 제네릭 `Vec<T>` 자료구조는 실제로 코드에서 사용되는 구체 타입(`Vec<i32>`, `Vec<String>` 등)마다 최적화된 전용 기계어 코드로 각각 복제되어 구체화됩니다(단형성화, Monomorphization)

실행 중에 어떤 함수를 부를지 vtable을 찾아보는 동적 디스패치(Dynamic Dispatch)는 오직 개발자가 `dyn Trait` 키워드로 명시적 다형성을 요청했을 때만 제한적으로 사용됩니다

결과적으로 모든 추상화 계층이 컴파일 타임에 인라인되어 기계어 단에서는 흔적도 없이 사라집니다. 이것이 바로 제로코스트 추상화가 실현되는 메커니즘입니다

한 발 더 나아가면 표준 라이브러리마저 걷어 내는 `#![no_std]` 속성이 존재합니다. 힙 메모리 할당자나 운영체제 커널 API에 의존하는 모든 표준 모듈을 떼어 내고 베어메탈(Bare-metal) 환경에서 구동하는 모드입니다

임베디드 하드웨어 펌웨어, OS 부트로더, eBPF(Extended Berkeley Packet Filter, 커널 내부 안전 실행 샌드박스) 프로그램, 리눅스 커널 모듈(Rust for Linux)까지 Rust가 빠르게 침투할 수 있는 배경이 바로 여기에 있습니다. 런타임이 없으므로 어떤 척박한 시스템 바닥에도 이식할 수 있습니다

### 그린 스레드를 과감히 버린 역사 — 2편의 매듭

1.0 버전 정식 출시를 앞둔 2014년 당시의 Rust는 놀랍게도 두 가지 스레드 모델을 모두 내장하고 있었습니다. M:N 유저 스케줄러 기반의 그린 스레드(`libgreen`)와 1:1 OS 스레드 기반의 네이티브 스레드(`libnative`)입니다

그러나 이 이중 체제는 두 가지 치명적인 딜레마를 낳았습니다

첫째, 그린 스레드가 돌기 위해서는 모든 Rust 바이너리에 스택 분할 스케줄러와 컨텍스트 스위치 런타임 엔진을 강제로 탑재해야 했습니다. 이는 M:N 스레드가 전혀 필요 없는 프로그램까지 런타임 무게를 짊어지게 하므로 제로코스트 추상화 원칙에 정면으로 위배되었습니다

둘째, C 언어 라이브러리와 상호 작용(FFI, Foreign Function Interface)할 때마다 그린 스레드의 스택과 C 언어의 네이티브 스택 간에 무거운 스택 전환 및 호환 오버헤드가 지속적으로 발생했습니다. 이는 C 언어 라이브러리와 운영체제 커널 생태계를 완벽히 대체하거나 공존하겠다는 Rust의 핵심 목표를 가로막았습니다

치열한 논쟁 끝에 내려진 역사적 결론이 바로 RFC 230(Request for Comments 230, 공식 기술 제안서 230번, 2014년)입니다. 그린 스레드(`libgreen`)를 언어와 표준 라이브러리에서 완전히 제거하고 1:1 네이티브 스레드만을 유일한 표준으로 확립했습니다

그리고 대규모 I/O 처리를 위한 M:N 동시성이 필요한 경우 언어 런타임 대신 `tokio` 같은 외부 비동기 런타임 라이브러리를 개발자가 직접 선택해 유저 영역에서 해결하도록 위임했습니다

2편에서 확인했던 "Rust 표준(`std::thread`)은 1:1이고, 비동기(`async`)는 라이브러리 선택"이라는 명확한 역할 분담이 바로 이 2014년의 거대한 결단에서 비롯되었습니다. 여기서 2편에서 남겨 두었던 역사적 실타래가 완벽히 풀리게 됩니다

### 가벼움을 얻기 위해 치른 대가

세상에 공짜 점심은 없듯 Rust의 극단적인 가벼움도 거대한 비용을 동반합니다. Rust는 물리적 실행 시점(Runtime)의 자원 소모를 0으로 만든 대신, 그 모든 하중을 개발 시점(Compile-time)의 인지적 복잡도와 컴파일 시간으로 전가했습니다

소유권 및 빌림 검사기와의 끝없는 인내심 싸움, 거대한 단형성화로 인해 길어지는 빌드 시간, 비동기 함수와 동기 함수의 생태계 단절(함수 색칠 문제), 명확한 메모리 수명(`<'a>`)을 손으로 직접 설계해야 하는 중압감이 개발자가 매일 치러야 할 청구서입니다. 이 개발 인건비의 무게가 클라우드 인프라 비용 절감과 어떻게 교환되는지는 시리즈 5부 클라우드 경제학 편에서 종합적으로 저울질합니다

## Go — 무겁지만 완벽히 자족적이다

Go는 Rust와는 정반대의 현실적인 설계를 선택했습니다. 현대 클라우드 서버에 꼭 필요한 핵심 런타임 엔진들을 알차게 챙겨 단일 실행 바이너리 안에 튼튼하게 박아 넣었습니다

Go 바이너리 하나를 빌드하면 그 내부에 자동 탑재되는 구성 요소들입니다

- **GMP 스케줄러** — 2편에서 심층 분석한 수백만 고루틴 배차 및 작업 훔치기 엔진입니다
- **GC 엔진** — 애플리케이션 연산과 동시에 도는 동시성(Concurrent) 마크-스윕 수집기입니다. Go 1.26부터는 메모리 스캔 효율을 극대화한 Green Tea GC가 기본 탑재됩니다. 이 수집기의 동작 비밀은 4부 GC 편에서 상세히 딥다이브합니다
- **동적 메모리 할당자** — 구글의 TCMalloc(Thread-Caching Malloc) 구조를 계승하여 논리 프로세서(`P`)마다 독립된 로컬 캐시(`mcache`, `mcentral`)를 두는 고속 할당자입니다
- **고루틴 연속 스택 관리기** — 초기 2KB에서 시작해 필요할 때마다 더 큰 메모리로 통째로 복사하며 성장하는 연속 스택 제어 모듈입니다
- **네트워크 폴러(Netpoller)** — `epoll`/`kqueue`를 통해 소켓의 비동기 이벤트를 감시하고 대기 중인 고루틴을 깨우는 장치입니다 (2편)
- **sysmon(시스템 모니터)** — 백그라운드에서 별도 OS 스레드로 돌며 10ms 이상 독점된 고루틴에 선점 시그널(`SIGURG`)을 보내고 GC 주기를 통제하는 감시자입니다

이 든든한 엔진들을 탑재한 만큼 물리적인 기본 무게는 분명히 존재합니다. 아무 로직이 없는 `hello world` 정적 바이너리를 빌드해도 크기가 약 1.5~2MB에 달하는데, 이는 언어 런타임 엔진 전체가 통째로 포함되기 때문입니다

또한 정상상태 오버헤드도 발생합니다. GC 주기적 스캔 작업과 `sysmon` 감시 스레드가 애플리케이션 로직과 병행하여 CPU와 메모리를 꾸준히 소모합니다

하지만 그 대가로 Go는 완벽한 자족성(Self-contained)을 선물합니다. 외부 동적 라이브러리(`libc` 등)나 바이트코드 인터프리터가 전혀 필요 없기 때문에 도커(Docker) 컨테이너 이미지를 다음처럼 경이로운 수준으로 압축할 수 있습니다

```dockerfile
# 런타임이 바이너리 내부에 완전히 자급자족으로 들어 있으므로 무거운 베이스 이미지가 필요 없습니다
FROM scratch
COPY app /app
ENTRYPOINT ["/app"]   # 최종 컨테이너 이미지 크기 ≈ 바이너리 파일 크기 단일 결합 (단 수 MB)
```

게다가 정적 AOT(Ahead-Of-Time) 컴파일 방식이라 JVM 같은 예열(Warm-up) 시간이 전혀 없습니다. 프로세스 구동 후 단 수 ms 만에 시동을 마치고 즉시 100% 정점 처리량을 뿜어냅니다

Rust만큼 극단적으로 가볍진 않지만 현대 인프라에서 감당하기에 충분히 가벼우면서도, 완벽한 배포 자족성과 ms 단위의 고속 시동, 그리고 GC가 주는 메모리 안전성의 편안함까지 모두 챙겼습니다. 이 절묘한 실용적 균형점이 도커와 쿠버네티스 같은 클라우드 네이티브 핵심 인프라 생태계가 Go를 선택하게 만든 절대적 동력입니다

## JVM — 가장 무겁고 가장 영리하다

JVM은 언어 런타임 계층을 두껍게 채운 정도가 아니라, 아예 독립적인 가상 머신(Virtual Machine) 운영 시스템 하나를 통째로 밑바닥에 깔았습니다

Java 컴파일러가 만들어 내는 `.class` 파일은 CPU가 이해하는 네이티브 기계어가 아니라 오직 JVM만이 이해할 수 있는 중간 플랫폼 바이트코드입니다. 이를 실제 물리 하드웨어에서 구동하려면 클래스 로더, 바이트코드 인터프리터, JIT 컴파일러, 고도의 GC 엔진을 모두 품은 거대한 JVM 프로세스가 구동되어야 합니다

이 거대한 JVM 프로세스의 실제 물리 메모리 지도를 파헤쳐 보겠습니다. 왜 JVM이 무겁다고 평가받는지 그 실체가 여기에 담겨 있습니다

![JVM process memory — the Java heap is only one tenant, the rest lives off-heap](/diagrams/runtime-weight-zero-cost-vs-fat-4.svg)

우리가 JVM 옵션으로 자주 다루는 Java 힙(Heap) 영역은 이 전체 메모리 지도에서 단지 세입자 중 하나일 뿐입니다. 각 구역이 맡은 물리적 역할은 다음과 같습니다

- **Java 힙(Heap)** — 우리가 코드로 생성한 Java 객체들이 살아가고 GC가 청소하는 중심 구역입니다. `-Xmx` 옵션으로 상한선을 통제합니다
- **메타스페이스(Metaspace)** — 클래스 로더가 읽어 들인 클래스의 메타데이터, 메서드 구조, 바이트코드가 저장되는 네이티브 메모리 영역입니다
- **JIT 코드 캐시(Code Cache)** — JIT 컴파일러(`C1`, `C2`)가 실시간으로 분석하여 번역해 낸 고성능 네이티브 기계어가 보관되는 전용 메모리 캐시 구역입니다. `C1`은 빠른 속도로 가벼운 최적화를 수행하고, `C2`는 시간을 들여 최고 난도의 정점 최적화를 수행하는 단계별 컴파일 엔진입니다
- **스레드 스택(Thread Stacks)** — 생존해 있는 물리 스레드마다 `-Xss` 설정값만큼 가상 메모리에 개별 할당되는 호출 스택 영역입니다
- **다이렉트 메모리(Direct Memory)** — Java NIO나 Netty 프레임워크가 커널 소켓 통신 오버헤드를 줄이기 위해 힙 바깥(Off-heap)에 직접 할당하는 고속 I/O 네이티브 버퍼(`DirectByteBuffer`)입니다
- **GC 및 컴파일러 스레드** — 힙 청소와 JIT 컴파일을 백그라운드에서 병행하기 위해 JVM이 자체적으로 띄워 구동하는 런타임 제어 스레드군입니다

가장 위험한 함정은 Java 힙(`-Xmx`) 바깥에 존재하는 이 모든 네이티브 구역들이 **`-Xmx` 설정값 상한의 통제를 전혀 받지 않는다**는 사실입니다. Java 힙 사용량이 0이더라도 이 하부 런타임 구역들이 기본적으로 수백 MB의 물리 메모리를 선점합니다

특히 NIO 서버에서 쓰이는 `DirectByteBuffer`는 운영 환경의 치명적인 복병입니다

힙 메모리 바깥에 할당되므로 일반적인 JVM 힙 모니터링 그래프에는 징후가 나타나지 않지만, 프로세스의 상주 물리 메모리(RSS)는 조용히 부풀어 오르다가 결국 컨테이너 메모리 한계를 뚫어 버립니다. 게다가 `-XX:MaxDirectMemorySize` 옵션으로 상한을 지정하지 않으면 기본 허용 한도가 대략 `-Xmx` 힙 최대치와 동일하게 잡히므로, 힙을 3GB로 설정한 파드가 다이렉트 메모리를 추가로 최대 3GB까지 더 소모할 수 있습니다

**"JVM은 힙 메모리만 계산해서 파드를 띄우면 100% OOMKilled를 맞이한다"**는 DevOps 격언의 명확한 기술적 근거입니다. 컨테이너 리밋과 JVM 네이티브 메모리 간의 사투는 5부 클라우드 편에서 심층적으로 분해합니다

시동 시간에도 무거운 청구서가 붙습니다. JVM은 구동 초기에는 바이트코드를 인터프리터로 한 줄씩 해석하며 천천히 실행하다가, 반복 호출되어 달아오른 핫스팟 메서드를 감지하면 `C1` 컴파일러로 1차 최적화하고, 더욱 빈번히 도는 핵심 루프는 `C2` 컴파일러로 다시 구워 내는 복잡한 단계를 거칩니다

따라서 서버가 100% 제 성능을 발휘하는 정점 처리량에 도달하기까지 수십 초에서 수분간의 예열(Warm-up) 곡선을 견뎌야 합니다. 이 예열 기간의 동적 컴파일 매커니즘이 바로 2부에서 깊이 다룰 주제입니다

하지만 JVM은 이 묵직한 초기 무게와 예열 시간을 지불한 대가로 엄청난 가치를 사들였습니다. 바로 프로덕션 실시간 트래픽의 실행 프로파일을 관찰하며 코드를 끊임없이 재구성하는 **적응성(Adaptability, JIT 재최적화)**, 25년간 축적된 방대한 엔터프라이즈 생태계, 그리고 세계 최고 수준의 대용량 GC 엔진들입니다

초기 무게와 장기 적응성의 극적인 교환. JVM은 이 거래를 소프트웨어 엔지니어링 역사상 가장 극단적인 수준까지 밀어붙인 위대한 런타임 플랫폼입니다

### JVM이 자신의 무거움을 공격한다 — Leyden과 GraalVM

클라우드 네이티브와 서버리스(Serverless) 시대가 도래하면서 JVM 진영 역시 자신들의 아킬레스건인 무거운 초기 무게와 예열 지연을 극복하기 위해 정면 도전을 시작했습니다. 현재 두 갈래의 혁신적인 개조 작업이 빠르게 진행되고 있습니다

**프로젝트 레이든(Project Leyden) — JVM 플랫폼을 유지하면서 초기 시동 작업을 사전에 마친다**

프로젝트 레이든의 핵심 개념은 훈련 실행(Training Run) 파이프라인입니다. 애플리케이션을 배포하기 전에 실전과 유사한 대표 트래픽을 주입해 훈련 실행을 거친 뒤, 그 과정에서 분석된 클래스 로딩 경로와 JIT 프로파일 데이터를 AOT 캐시 파일로 영구 저장해 둡니다. 그리고 실제 프로덕션 구동 시에는 이 캐시를 통째로 읽어 들여 무거운 초기 시동 단계를 생략합니다

- **JEP 483 (JDK 24)** — 클래스 로딩 및 링킹 분석 결과를 훈련 실행 시 AOT 캐시로 구워 내어 초기 부하를 덜어 냅니다
- **JEP 514·515 (JDK 25)** — 훈련 실행 워크플로를 대폭 자동화하고, JIT 컴파일러의 메서드 프로파일까지 캐시 데이터로 저장하여 예열 곡선을 획기적으로 앞당깁니다

이 개선의 효과는 실증 데이터로 입증되었습니다. 대표적인 벤치마크인 Spring PetClinic 애플리케이션의 시동 시간이 JDK 23 기준 4.486초에서 JDK 24의 AOT 캐시 적용 시 2.604초로 **약 42% 단축**되었습니다

JVM의 동적 적응성과 생태계를 전혀 포기하지 않고도 이뤄 낸 훌륭한 타협점입니다

**GraalVM 네이티브 이미지(native-image) — 가상 머신을 완전히 버리고 AOT 정적 네이티브 컴파일**

`native-image` 기술은 닫힌 세계 가정(Closed-world Assumption, 실행 시 필요한 모든 클래스와 도달 가능 코드가 빌드 타임에 100% 확정된다는 전제)을 기반으로 구동합니다. 정적 분석 엔진이 애플리케이션 진입점부터 도달 가능한 모든 코드를 추적하여 JVM 바이트코드 전체를 네이티브 기계어 실행 파일로 굽어 냅니다

이 결단의 결실은 극적입니다. 수 초에 달하던 시동 시간이 50~200ms 수준으로 **10~50배 단축**되며, 400MB급을 맴돌던 베이스라인 메모리 점유가 100MB급으로 **약 4분의 1로 격감**합니다

하지만 이 극적인 다이어트의 청구서는 런타임이 아닌 빌드 타임과 운영 유연성으로 이동합니다

- **빌드 타임 및 자원 폭증** — 애플리케이션 내의 어떤 클래스와 객체에 도달할 수 있는지 전역 경로를 파고드는 포인터 가리킴 정적 분석(Points-to Static Analysis)과 전역 AOT 컴파일이 맞물리면서 빌드 시간이 수 분에서 10분 이상까지 지연됩니다. 빌드 도중 시스템 물리 메모리의 최대 85%(최대 30GiB)를 폭식하며 자원이 부족하면 GC 오버헤드로 빌드가 마비되기도 합니다. 런타임은 ms로 줄었지만 빌드 타임은 분 단위로 늘어난 셈입니다
- **닫힌 세계 가정의 제약** — 리플렉션(Reflection), 동적 프록시, JNI 호출, 직렬화 같은 동적 기능들은 정적 분석기가 코드를 추적할 수 없으므로 사전에 도달 가능성 메타데이터 설정(JSON 힌트)을 명시해야만 바이너리에 포함됩니다. "일반 JVM에서는 잘 돌던 코드가 네이티브 이미지에서는 `ClassNotFoundException`으로 죽는" 난해한 디버깅이 발생합니다. Spring AOT나 Quarkus, Micronaut 프레임워크가 이 힌트를 자동 생성해 주지만 동적 프로그래밍의 자유도는 제한됩니다
- **GC 엔진 선택지의 한계** — 무료로 제공되는 Community Edition의 기본 GC는 풋프린트 절감에 특화된 직렬(Serial) GC입니다. 대용량 힙을 부드럽게 다루는 G1 GC는 리눅스 AMD64/AArch64 환경으로 한정되며, 초저지연 ZGC는 아예 지원되지 않습니다. 수십 GB 이상의 대용량 힙 서버에서는 오히려 치명적인 약점이 됩니다
- **정점 처리량의 하락** — 실행 중 프로파일링을 통한 JIT 동적 재최적화를 포기한 대가로, 장기 구동 시 정점 처리량이 기존 JIT 대비 약 10~25% 낮게 측정됩니다. 프로덕션 실행 프로파일을 빌드 타임에 다시 주입하는 PGO(Profile-Guided Optimization) 기법을 적용하면 격차의 30~50%를 회복할 수 있지만 빌드 파이프라인은 더욱 무거워집니다

| 비교 항목 | 프로젝트 레이든 (AOT 캐시) | GraalVM 네이티브 이미지 (`native-image`) |
| --- | --- | --- |
| JVM 가상 머신 | 플랫폼 유지 | 완전히 걷어 냄 (네이티브 실행 파일 단독 배포) |
| 시동 시간 단축 | 약 42% 단축 (PetClinic 벤치마크 기준) | 50~200ms (기존 대비 10~50배 단축) |
| 새로 지불하는 비용 | CI/CD 파이프라인 내 훈련 실행 단계 추가 | 수 분간의 고비용 빌드 + 닫힌 세계 제약 관리 |
| 정점 처리량 | 기존 JIT 동적 적응 성능 그대로 유지 | 기존 JIT 대비 -10~25% (PGO 적용 시 일부 회복) |
| 가장 어울리는 환경 | 기존 엔터프라이즈 JVM 백엔드 워크로드 전반 | 서버리스(Lambda), CLI 도구, 초경량 마이크로서비스 |

비교표가 보여 주듯 두 접근법 모두 완벽한 은탄환은 아닙니다. 레이든은 파이프라인 구성의 번거로움을, 네이티브 이미지는 빌드 시간과 동적 유연성의 상실을 대가로 요구합니다

특히 `native-image`는 컨테이너가 잦은 주기로 생성되고 소멸하며 힙 크기가 작은 서버리스 및 CLI 워크로드에서는 완벽한 순이득이지만, 대규모 힙을 들고 수주 간 쉬지 않고 연산하는 무거운 백엔드 서버에서는 JIT의 정점 처리량과 고성능 GC를 포기하는 손해일 수 있습니다

JVM은 더 이상 자신의 무거움을 무조건적인 약점으로 방치하지 않습니다. 그 무거움을 목적에 따라 선택하고 제어할 수 있는 **전략적 옵션**으로 탈바꿈시키고 있습니다

## 정면 비교 — 런타임에 박았는가, 라이브러리로 뺐는가

수백만 개의 비동기 I/O 태스크를 단 몇 개의 물리 OS 스레드로 가볍게 다룬다는 동일한 과제를 두고, Go와 Rust는 각자의 런타임 철학에 따라 정확히 대척점에 섰습니다. 이번 편의 핵심 통찰이 바로 이 구조적 대조에 있습니다

![Go's integrated runtime vs Rust's user-land assembly with the Waker seam](/diagrams/runtime-weight-zero-cost-vs-fat-5.svg)

그림 왼쪽의 Go 구조를 살펴봅니다. 고루틴은 독립적인 스택을 보유한 스택풀(Stackful) 코루틴으로 동작하며, 그 하부에 수백만 태스크를 다루는 GMP 스케줄러와 네트워크 폴러가 언어 공식 런타임 내부에 단단하게 내장되어 있습니다

개발자는 시스템 부품을 고민하고 조립할 필요가 전혀 없습니다. 비동기 호출을 위해 오직 `go f()` 단 한 줄만 작성하면 끝이며, 효율적인 M:N 배차와 논블로킹 I/O 감시가 런타임에 의해 자동으로 결합됩니다

하지만 그 편리함의 값으로 모든 Go 바이너리는 이 내장 엔진들의 물리적 무게와 관리 오버헤드를 변함없이 지불해야 합니다. 또한 이 견고한 일체형 런타임 아키텍처를 외부에서 임의로 교체할 수 없습니다

만약 프로젝트 특성상 작업 훔치기 방식이 맞지 않거나, 리눅스 커널의 고속 `epoll` 대신 최신 `io_uring`(리눅스 고성능 비동기 I/O 링 버퍼 인터페이스)을 네트워크 엔진으로 직접 꽂아 쓰고 싶어도 Go 내장 런타임을 갈아 끼울 방법은 존재하지 않습니다

그림 오른쪽의 Rust는 언어 공식 런타임 계층을 깨끗하게 비워 두고, 비동기 처리에 필요한 모든 부품을 유저랜드(User-land) 라이브러리로 분리하여 조립하는 방식을 택했습니다. 비동기 함수는 컴파일러에 의해 상태 기계(`Future`)로 번역될 뿐이며, 이를 물리 스레드에서 구동할 실행기(Executor, `tokio`)와 입출력 이벤트를 감시할 리액터(Reactor, `mio`)는 외부 크레이트에서 가져와 결합합니다

이 독립된 두 모듈 사이를 이어 주는 유일하고 표준화된 이음새가 바로 Waker(웨이커, 비동기 I/O 준비 완료 시 멈춰 있던 태스크를 실행기의 대기열에 다시 깨워 넣는 알람 인터페이스)입니다. 리액터는 I/O 이벤트가 도착하면 이 Waker를 호출하여 해당 태스크를 실행기에 다시 등록합니다

이 명확하고 얇은 인터페이스 이음새 덕분에 Rust는 프로젝트 요구사항에 맞춰 비동기 런타임 엔진 전체를 통째로 교체할 수 있습니다. 대표적인 사례가 바로 고성능 비동기 프레임워크인 `glommio`입니다

`glommio`는 `tokio`의 작업 훔치기 방식을 과감히 버리고 thread-per-core(코어당 독립 스레드 및 로컬 대기열 고정) 모델과 리눅스 `io_uring` 인터페이스를 결합했습니다. 스레드 간 작업 공유와 락 동기화 오버헤드를 원천 소거하고 물리 CPU 캐시 지역성(Cache Locality)을 극한까지 끌어올린 맞춤형 아키텍처입니다

동일한 프로그래밍 언어를 사용하면서도 애플리케이션의 성격에 따라 하부 런타임 아키텍처 자체를 완전히 갈아 끼울 수 있는 자유. 이것이 Go에는 없고 Rust에만 존재하는 강력한 무기입니다

한 줄로 요약하면 다음과 같습니다. **Go는 가장 범용적인 정답 하나를 훌륭하게 만들어 바이너리에 박아 넣었고, Rust는 런타임 계층을 비워 둠으로써 무한한 선택의 자유를 열었습니다**

이 차이가 바로 무거운 관리형 런타임(Fat Runtime)과 제로코스트 추상화 철학이 보여 주는 가장 순수한 대조입니다. 스택풀과 스택리스, 선점형과 협력형 스케줄링이라는 세부 기술적 차이는 2편에서 이미 충분히 다루었습니다

## 세 언어 수렴 비교

| 비교 항목 | Rust (std) | Go | JVM (전통 JIT) | JVM (`native-image`) |
| --- | --- | --- | --- | --- |
| 런타임 위치 | 거의 없음 (얇은 초기화) | 단일 바이너리에 내장 | 독립된 별도 VM 프로세스 | 네이티브 바이너리에 내장 |
| 메모리 관리 | 컴파일 타임 소유권/RAII | 동시성 런타임 GC | 고성능 런타임 GC | 직렬/G1 런타임 GC |
| 스케줄러 | 없음 (1:1 OS 스레드) | 내장 GMP 유저 스케줄러 | OS 스레드 + 가상 스레드 | OS 스레드 + 가상 스레드 |
| 코드 생성 방식 | 정적 AOT 컴파일 | 정적 AOT 컴파일 | 동적 JIT 적응형 컴파일 | 정적 AOT 컴파일 |
| 기본 바이너리 크기 | 수백 KB | 약 1.5~2MB | 작음 (별도 JRE 필수) | 중간 (수십 MB) |
| 베이스라인 RSS | 최소 (수백 KB급) | 작음 (수 MB급) | 큼 (수백 MB급) | 작음 (수십 MB급) |
| 시동 시간 (Cold Start) | ms 단위 | ms 단위 | 초 단위 (예열 필수) | ms 단위 |
| 정점 처리량 (Peak) | 매우 높음 및 극도의 예측성 | 중상 수준 | 세계 최고 (JIT 적응 최적화) | 중 (-10~25% 하락) |
| 무거움으로 구매한 가치 | 없음 (초기 자원을 지불하지 않음) | 완벽한 자족성 + 개발 생산성 | 실시간 적응성 + 생태계 파워 | 초고속 시동 + 풋프린트 절감 |

마지막 행이 이 전체 비교표의 핵심 결론입니다. 프로그래밍 언어의 물리적 무게는 언제나 그 대가로 가치 있는 무언가를 사들이기 위한 **구매 대금**이었습니다

Rust는 실행 시점에 아무런 초기 자원도 사지 않는 제로코스트를 선택했습니다. 그 대신 지불해야 할 모든 복잡도와 인건비 정산서를 물리적 실행 시점이 아닌 컴파일 타임의 개발자에게로 넘겼습니다

Go와 JVM은 동일하게 가비지 컬렉션을 사용하는 관리형 언어지만 짊어진 무게로 구매한 가치가 다릅니다. Go는 완벽한 배포 자족성과 고속 시동을 통한 운영 편의를 샀고, JVM은 가상 머신이라는 무거운 플랫폼을 지불하여 장기 구동 시의 실시간 적응성과 압도적인 정점 처리량을 샀습니다

`native-image`는 그 사이에 새로운 선택지를 열었습니다. 방대한 Java 생태계를 유지하면서 물리적 무게 축을 Go와 유사한 가벼운 영역으로 이동시킨 대신, JVM이 애써 샀던 실시간 JIT 적응성을 다시 시장에 반납했습니다

## 트레이드오프 소결 — 무게는 죄가 아니라 전략적 교환

- **Rust** — 언어 런타임을 거의 완벽히 비워 냄으로써 물리적 가벼움과 극도의 성능 예측 가능성, 그리고 어떤 저수준 환경에도 이식할 수 있는 자유를 얻었으며, 그 대금은 개발 시점의 인지적 복잡도와 컴파일 시간으로 치릅니다
- **Go** — 클라우드에 필수적인 실속 런타임 엔진들을 단일 바이너리에 박아 넣어 완벽한 배포 자족성과 생산성, ms 단위 고속 시동을 얻었으며, 그 대금으로 Rust보다 무거운 기본 풋프린트와 GC 오버헤드를 지불합니다. 실용성의 정점이자 균형점입니다
- **JVM** — 가상 머신 시스템 전체를 하부에 얹어 실시간 JIT 적응성, 장기 구동 시 세계 최고 수준의 정점 처리량, 25년의 생태계를 얻었으며, 그 대금으로 가장 무거운 물리 메모리 점유와 예열 지연을 냅니다. 현재는 프로젝트 레이든과 GraalVM을 통해 그 무거움마저 개발자가 선택할 수 있는 전략 옵션으로 전환하고 있습니다

무거움이나 가벼움 그 자체는 선이나 악이 아닙니다. 엔지니어가 시스템을 설계할 때 스스로에게 던져야 할 진정한 질문은 단 하나입니다. **"우리의 비즈니스 워크로드에서, 이 언어의 런타임 무게를 지불하고 사들인 가치가 진정으로 유용한가?"**

다음 2부에서는 그 대표적인 구매품을 집중 검증합니다. 실행 중에 기계어를 실시간으로 다시 구워 내는 JIT 컴파일러의 동적 적응성이 과연 정적 AOT 컴파일의 벽을 넘어설 수 있는지, 그 기술적 실체를 파헤칩니다

## DevOps 관점에서 더 파고들 질문

1. **컨테이너 이미지 경량화 전략** — Go와 Rust는 `FROM scratch` 베이스 이미지를 통해 최종 배포 크기를 수 MB 단위로 깎을 수 있지만, 전통 JVM은 무거운 JRE 베이스 이미지가 필수적입니다. `jlink` 도구로 모듈형 맞춤 JRE를 구성하는 것과 GraalVM `native-image`로 전면 전환하는 것 사이에서 배포 이미지 크기, 빌드 파이프라인 복잡도, 프로덕션 디버깅 난이도의 트레이드오프는 어떻게 평가해야 할까요?
2. **콜드 스타트와 서버리스(Serverless) 아키텍처** — AWS Lambda나 KNative처럼 트래픽에 따라 파드가 빈번히 소멸하고 다시 뜨는 환경에서 초 단위에 달하는 전통 JVM의 예열 곡선은 치명적인 약점입니다. 프로젝트 레이든의 AOT 캐시를 적용한 42% 시동 단축만으로 서버리스 SLA를 충족할 수 있을까요, 아니면 50~200ms 구동이 가능한 `native-image`로의 전환이 불가피할까요?
3. **베이스라인 RSS와 노드 빈패킹(Bin-packing) 밀도** — 물리 메모리 200MB 제한이 걸린 파드에 Spring Boot 애플리케이션(기본 힙 + JVM 오버헤드 350~550MB 소모)을 배포하지 못하는 결정적 이유가 Java 힙이 아니라 메타스페이스, JIT 코드 캐시, 스레드 스택, 다이렉트 메모리 때문이라면 쿠버네티스의 `request`와 `limit` 설정을 어떻게 분리해야 노드에 파드를 고밀도로 배치할 수 있을까요?
4. **`native-image` 도입 시 숨은 복병들** — 닫힌 세계 가정으로 인해 리플렉션이나 동적 프록시 의존성이 높은 프레임워크는 힌트 설정을 놓칠 경우 런타임 오류로 멈춥니다. 복잡한 힌트 유지보수 비용, JIT 대비 10~25% 낮아지는 정점 처리량, 그리고 초고속 시동이라는 이점을 종합표에 올렸을 때 정확히 어떤 성격의 마이크로서비스에서 순이득이 발생할까요?
5. **프로젝트 레이든 훈련 실행의 CI/CD 비용** — 레이든의 AOT 캐시 생성 단계는 실전과 100% 일치하는 대표 트래픽을 주입하는 훈련 실행(Training Run) 파이프라인을 전제합니다. 이 훈련 단계를 CI/CD 파이프라인에 구축하고 유지하는 운영 비용과, 훈련 트래픽이 실제 프로덕션 트래픽 패턴과 어긋났을 때 발생하는 최적화 왜곡 리스크는 어떻게 통제해야 할까요?
6. **멀티아키텍처(ARM/Graviton) 크로스 빌드 파이프라인** — Go와 Rust는 환경 변수(`GOARCH=arm64` 등) 설정만으로 x86 환경에서도 ARM 기반 AWS Graviton용 바이너리를 손쉽게 크로스 컴파일합니다. 반면 JVM 바이트코드는 아키텍처 중립적이지만, `native-image`는 반드시 타깃 아키텍처 환경과 동일한 머신에서 직접 빌드해야 합니다. ARM 서버 전환을 앞둔 팀에게 세 언어의 빌드 파이프라인 복잡도는 어떻게 달라질까요?

## 핵심 요약

- 언어 런타임은 애플리케이션 코드와 OS 커널 사이에서 자동으로 구동되는 관리 코드 엔진 계층입니다. Rust는 이 계층을 거의 비워 냈고, Go는 실속 있는 핵심 모듈을 단일 바이너리에 박았으며, JVM은 가상 머신 플랫폼을 통째로 얹었습니다
- "무겁다"는 평가는 반드시 다섯 가지 독립된 축(바이너리 크기, 베이스라인 RSS, 시동 시간, 정상상태 오버헤드, 정점 처리량)으로 분해하여 측정해야 합니다. 초기 무게의 순서는 장기 구동 후의 정점 처리량 항목에서 완전히 역전됩니다
- Rust는 2014년 RFC 230 결단을 통해 그린 스레드(`libgreen`)를 영구 제거하며 제로코스트 추상화 철학을 사수했습니다. 컴파일러의 소유권 검사가 GC를, 단형성화가 vtable을 대체하며 모든 정산은 컴파일 타임의 개발 인건비로 지불됩니다
- JVM 프로세스 내부에서 Java 힙은 단지 하나의 세입자일 뿐입니다. 메타스페이스, JIT 코드 캐시, 스레드 스택, 다이렉트 메모리가 `-Xmx` 설정값 바깥에서 프로세스 RSS를 크게 부풀리므로, 힙 메모리만 계산해 컨테이너 리밋을 잡으면 100% OOMKilled에 직면합니다
- 프로젝트 레이든은 JVM 플랫폼을 유지한 채 AOT 훈련 실행 캐시로 시동 시간을 약 42% 단축시켰고, GraalVM `native-image`는 가상 머신을 버려 ms 단위 시동을 얻은 대신 분 단위의 빌드 타임과 닫힌 세계 가정 제약을 짊어집니다
- Go는 M:N 스케줄러를 내장 런타임에 고정해 박았고, Rust는 Waker 표준 인터페이스를 통해 비동기 엔진을 외부 라이브러리로 조립하도록 열어 두었습니다. 런타임 무게는 죄가 아니라 전략적 교환이며, 우리가 짚어야 할 핵심은 "이 무게를 지불하고 사들인 가치가 우리 시스템에 참으로 필요한가"입니다
