## JVM은 무겁고 Rust는 가볍다, 그런데 뭐가?

"JVM은 무겁고, Go는 가볍고, Rust는 더 가볍다"

자주 듣는 말이지만 막상 따져 보면 모호합니다. 바이너리가 큰 걸까요, 메모리를 많이 먹는 걸까요, 시동이 느린 걸까요?

이걸 뭉뚱그리면 이상한 문장 앞에서 길을 잃습니다. 예를 들면 "가장 무거운 JVM이 정점 처리량에선 가장 빠르다" 같은 문장이죠

그래서 이번 편은 먼저 런타임이라는 말의 정확한 의미를 잡고, "무겁다"를 다섯 개의 축으로 분해합니다. 그다음 세 언어가 각자 어떤 무게를 왜 짊어졌는지 하나씩 뜯어봅니다

## 런타임이란 정확히 무엇인가

용어가 두 가지 뜻으로 혼용됩니다. 구분부터 하겠습니다

- **런타임 라이브러리** — `printf`, 문자열, 컬렉션처럼 개발자가 직접 부르는 표준 라이브러리 함수들. 모든 언어에 있습니다
- **언어 런타임** — 개발자가 부르지 않아도 프로그램과 함께, 프로그램 아래에서 자동으로 도는 관리 코드

이 글의 주제는 후자입니다. 그 층에 무엇이 들어갈 수 있는지부터 보겠습니다

![What can live in the language runtime layer between app code and the kernel](/diagrams/runtime-weight-zero-cost-vs-fat-1.svg)

그림의 가운데 층이 언어 런타임입니다. 들어갈 수 있는 후보를 하나씩 풀면 이렇습니다

- **메모리 관리** — GC(가비지 컬렉터)와 할당자. GC는 더 이상 쓰이지 않는 메모리를 실행 중에 찾아 회수하는 장치입니다
- **스케줄러** — 2편에서 본 GMP나 가상 스레드처럼, 값싼 유저 태스크를 소수의 OS 스레드에 배차하는 층
- **JIT 컴파일러** — 프로그램이 도는 동안 기계어를 새로 만들어 내는 컴파일러
- **리플렉션·런타임 타입 정보** — 실행 중에 타입의 구조를 조회하고 조작하는 기능
- **예외 언와인딩·패닉 처리** — 예외가 던져지면 호출 스택을 거슬러 오르며 정리하는 절차
- **동적 디스패치** — 어떤 구현을 부를지 실행 중에 vtable(함수 포인터 표)에서 찾아 결정하는 기법
- **클래스 로더·바이트코드 인터프리터** — 중간 코드를 읽어 들여 실행하는 장치

핵심 질문은 하나입니다. 이 중간 층에 무엇을, 얼마나 넣었는가

Rust는 거의 비웠습니다. Go는 실속 있게 채웠고, JVM은 가상 머신 하나를 통째로 넣었습니다

## "무겁다"의 다섯 가지 의미

무게를 재기 전에 저울부터 다섯 개로 나눠야 합니다

여기서 베이스라인 RSS(Resident Set Size)는 앱이 아무 일도 하지 않을 때 프로세스가 실제 점유한 물리 메모리를 말합니다. 정상상태 오버헤드는 앱이 일하는 동안 런타임이 백그라운드에서 계속 쓰는 CPU와 메모리, 정점 처리량은 웜업이 끝난 뒤 낼 수 있는 최대 처리량입니다

| 축 | Rust | Go | JVM (전통 JIT) | JVM (native-image) |
| --- | --- | --- | --- | --- |
| 바이너리·이미지 크기 | 아주 작음 | 중간 | 큼 (JRE 필요) | 작음 |
| 베이스라인 RSS | 아주 작음 | 작음 | 큼 | 작음 |
| 시동 (cold start) | ms | ms | 초 단위 (웜업) | ms |
| 정상상태 오버헤드 | 거의 0 | GC + 스케줄러 | GC + JIT 스레드 | GC |
| 정점 처리량 (peak) | 높음 | 중상 | 최고 (JIT) | 중 (-10~25%) |

표를 세로로 읽으면 첫인상 그대로입니다. Rust 열은 거의 모든 축에서 작고, JVM 전통 열은 거의 모든 축에서 큽니다

그런데 마지막 행에서 순서가 뒤집힙니다. 위 네 축에서 가장 무거웠던 JVM이 정점 처리량에서는 가장 앞섭니다

무게를 지불하고 런타임 적응성을 샀기 때문인데, 이 반전이 다음 2부의 큰 줄기입니다

native-image 열도 눈여겨볼 만합니다. 바이너리·RSS·시동은 Go 수준으로 줄었지만 정점 처리량이 JIT보다 10~25% 낮습니다

같은 JVM 생태계 안에서도 무게와 처리량을 맞바꾸는 교환이 보이죠

같은 내용을 스펙트럼 한 장으로 놓으면 이렇습니다

![The weight spectrum flips when you read it as peak throughput](/diagrams/runtime-weight-zero-cost-vs-fat-2.svg)

위쪽 화살표는 무게입니다. 오른쪽으로 갈수록 바이너리가 크고, 베이스라인 메모리가 높고, 시동이 느립니다

아래쪽 화살표는 웜업 이후의 정점 처리량입니다. 방향이 같습니다

즉 무게 순서의 오른쪽 끝, 가장 무거운 전통 JVM이 오래 달리는 서버에서는 유리해질 수 있습니다

무게는 죄가 아니라 교환이라는 이번 편의 결론이 이 그림에 이미 들어 있습니다

## 세 언어는 이 층을 어떻게 채웠나

본격 해부에 앞서 지도를 먼저 펼치겠습니다. 세 언어가 런타임 층을 채운 방식은 두께도 위치도 다릅니다

![How thick each language's runtime layer is and where it lives](/diagrams/runtime-weight-zero-cost-vs-fat-3.svg)

왼쪽 Rust부터 봅니다. 앱 코드가 곧 네이티브 기계어이고, 커널과의 사이에는 아주 얇은 시작 코드만 끼어 있습니다

런타임 층이 사실상 비어 있는 구조입니다

가운데 Go는 하나의 정적 바이너리 안에 앱 코드와 런타임이 함께 들어갑니다. 스케줄러, GC, 할당자, netpoller가 전부 그 안에 있죠

층은 두껍지만 바이너리 바깥에 아무것도 필요하지 않습니다

오른쪽 JVM은 구조 자체가 다릅니다. 앱은 기계어가 아니라 바이트코드(`.class`)로 배포되고, 그걸 실행해 줄 별도의 가상 머신 프로세스가 필요합니다

런타임이 앱보다 큰 유일한 구조입니다

이 세 그림이 곧 세 철학입니다. 이제 왜 이렇게 갈렸는지 하나씩 내려가 보겠습니다

## Rust — 런타임을 거의 비웠다

### 제로코스트가 런타임을 금지한다

Rust는 C++에서 온 원칙을 이어받았습니다

> 쓰지 않는 것에는 비용을 내지 않는다. 쓰는 것은 손으로 짜도 그보다 더 잘 짤 수 없다

제네릭, 이터레이터, async 같은 추상화가 런타임 비용 0으로 컴파일돼야 한다는 뜻입니다

이 원칙이 Rust에 관리형 런타임을 두지 못하게 강제합니다. GC를 넣는 순간 GC를 안 쓰는 프로그램도 그 비용을 지기 때문입니다

### 그래서 Rust 런타임에는 무엇이 남았나

`main`이 실행되기 전에 `lang_start`라는 얇은 진입 코드가 돕니다. 하는 일은 스택 오버플로 가드 페이지 설정, 패닉 핸들러 준비, 인자와 TLS(스레드 로컬 저장소) 초기화 정도입니다

그리고 그게 거의 전부입니다. GC 없음, M:N 스케줄러 없음, JIT 없음

그럼 GC 없이 메모리는 누가 관리할까요? Rust는 이 일을 실행 시점에서 컴파일 시점으로 옮겼습니다

소유권(ownership)과 빌림(borrow) 검사기가 컴파일 때 "이 값은 언제 해제해도 안전한가"를 판정합니다. 그 지점에 해제 코드(`drop`)가 자동 삽입되죠

자원의 수명을 변수의 수명에 묶는 RAII 기법입니다

해제 시점을 실행 중에 추적할 필요가 없으니 런타임 GC가 필요 없습니다. 이 라이프사이클은 시리즈 후반 메모리 편에서 malloc/free 수준까지 내려가 봅니다

추상화 쪽도 같은 방식입니다. 제네릭 `Vec<T>`는 실제 쓰인 타입마다 전용 코드로 복제 컴파일됩니다(monomorphization)

실행 중에 vtable을 찾는 동적 디스패치는 `dyn Trait`를 명시할 때만 씁니다

그래서 추상화가 인라인되어 기계어에서 사라집니다. 이게 제로코스트의 실체입니다

한 발 더 나가면 `#![no_std]`가 있습니다. 힙 할당과 OS에 의존하는 표준 라이브러리까지 걷어내고 베어메탈에서 도는 모드입니다

임베디드 펌웨어, 부트로더, eBPF, 커널 모듈, 그리고 리눅스 커널(Rust for Linux)까지 Rust가 들어가는 이유가 여기 있습니다. 런타임이 없으니 어디에든 심을 수 있습니다

### 그린 스레드를 버린 이야기 — 2편의 매듭

1.0 이전인 2014년의 Rust는 두 스레드 모델을 다 갖고 있었습니다. libgreen(M:N 그린 스레드)과 libnative(1:1)입니다

문제가 둘 있었습니다

첫째, 그린 스레드 스케줄러는 모든 Rust 프로그램에 무거운 런타임을 강제했습니다. 그린 스레드를 안 쓰는 프로그램까지 비용을 내야 하니 제로코스트 위반입니다

둘째, C 함수를 부를 때마다(FFI) 스택 전환 비용이 붙었습니다. C 라이브러리와 OS 안에 Rust를 심겠다는 목표와 정면충돌했죠

결론은 RFC 230(2014)입니다. 그린 스레드를 언어에서 제거하고 1:1 네이티브 스레드를 기본으로 삼았습니다

M:N이 필요하면 tokio 같은 라이브러리에서 각자 고르게 했습니다

2편에서 본 "Rust std는 1:1, async는 라이브러리" 구도가 바로 이 결정의 귀결입니다. 여기서 2편의 실이 매듭지어집니다

### 가벼움의 대가

가벼움은 공짜가 아닙니다. Rust는 무게를 실행 시점이 아니라 개발 시점의 복잡도로 지불합니다

borrow checker와의 씨름, 긴 컴파일 시간, async 함수 색칠, 수명을 손으로 설계하는 부담이 그 청구서입니다. 이 인건비가 클라우드 비용과 어떻게 저울질되는지는 시리즈 후반에서 다룹니다

## Go — Fat이지만 자족적이다

Go는 Rust와 반대로 선택했습니다. 실속 있는 런타임을 골라 담아 바이너리에 정적으로 박아 넣었습니다

바이너리 하나에 들어가는 것들입니다

- **GMP 스케줄러** — 2편에서 해부한 M:N 배차 장치
- **GC** — 앱과 동시에 도는 동시(concurrent) 마크-스윕 수집기. Go 1.26부터는 마킹 방식을 개선한 Green Tea GC가 기본입니다. 시리즈 후반 GC 편에서 딥다이브합니다
- **할당자** — TCMalloc 계열의 P별 캐시(mcache·mcentral) 구조. 역시 메모리 편에서 다룹니다
- **고루틴 스택 관리** — 2KB에서 시작해 통째로 복사하며 자라는 연속 스택
- **netpoller** — epoll로 네트워크 준비 이벤트를 받는 장치 (2편)
- **sysmon** — 오래 도는 고루틴 선점과 GC 트리거를 맡는 백그라운드 감시 스레드

무게는 분명 있습니다. `hello world` 정적 바이너리가 대략 1.5~2MB인데, 런타임을 통째로 담기 때문입니다

정상상태 오버헤드도 있습니다. GC 작업과 sysmon이 앱과 동시에 계속 돕니다

대신 자족적(self-contained)입니다. 외부 라이브러리도 인터프리터도 필요 없어서 컨테이너 이미지를 이렇게 만들 수 있습니다

```dockerfile
# 런타임이 바이너리 안에 있으니 베이스 이미지가 필요 없다
FROM scratch
COPY app /app
ENTRYPOINT ["/app"]   # 최종 이미지 크기 ≈ 바이너리 크기 (수 MB)
```

AOT 컴파일이라 웜업도 없습니다. ms 단위로 시동해서 바로 정상 성능을 냅니다

Rust만큼 가볍진 않지만 충분히 가볍고, 자족적이고, 시동이 빠르고, GC의 편안함까지 있습니다. 이 균형점이 Docker와 Kubernetes 같은 클라우드 네이티브 인프라가 Go로 쓰인 배경인데, 그 이야기는 시리즈 후반에서 이어집니다

## JVM — 가장 무겁고 가장 영리하다

JVM은 런타임 층을 채운 정도가 아니라 가상 머신 하나를 통째로 얹었습니다

`.class` 파일은 네이티브 기계어가 아니라 JVM용 중간 코드(바이트코드)입니다. 실행하려면 클래스 로더, 인터프리터, JIT 컴파일러, GC를 갖춘 JVM 프로세스가 필요합니다

그 프로세스의 메모리 지도를 보겠습니다. 무게의 정체가 여기 있습니다

![JVM process memory — the Java heap is only one tenant, the rest lives off-heap](/diagrams/runtime-weight-zero-cost-vs-fat-4.svg)

Java 힙은 이 지도에서 세입자 하나일 뿐입니다. 각 구역의 역할은 이렇습니다

- **Java Heap** — 내 객체들이 사는 곳. `-Xmx`로 상한을 정합니다
- **Metaspace** — 로딩된 클래스의 구조 정보(메타데이터) 저장소
- **JIT Code Cache** — JIT 컴파일러(C1·C2)가 만들어 낸 기계어 저장소. C1은 빠르고 가벼운 최적화, C2는 느리지만 공격적인 최적화를 맡는 단계별 컴파일러입니다
- **Thread Stacks** — 스레드마다 `-Xss`만큼 잡는 호출 스택
- **Direct Memory** — NIO·Netty가 쓰는 힙 밖 버퍼(DirectByteBuffer)
- **GC·컴파일러 스레드** — 백그라운드에서 도는 런타임 작업 스레드들

문제는 힙 아래 구역 전부가 `-Xmx` 바깥이라는 점입니다. 내 힙이 0이어도 이들이 먼저 자리를 차지합니다

특히 DirectByteBuffer는 이중 함정입니다

힙 밖이라 힙 모니터링에는 안 잡히고, RSS만 조용히 부풀다 컨테이너 리밋에 걸립니다. 게다가 `-XX:MaxDirectMemorySize`를 지정하지 않으면 기본 상한이 대략 `-Xmx`와 같아서, 힙을 3GB로 잡으면 다이렉트 메모리가 최대 3GB를 더 쓸 수 있습니다

"힙만 계산하면 K8s에서 OOMKilled"라는 격언의 근원입니다. 컨테이너 리밋과의 싸움은 시리즈 후반 클라우드 편에서 본격적으로 다룹니다

시동에도 청구서가 있습니다. JVM은 처음엔 바이트코드를 인터프리터로 한 줄씩 실행하고, 자주 도는 코드를 C1으로, 더 뜨거워지면 C2로 다시 컴파일합니다

그래서 정점 성능까지 시간이 걸립니다. 이 웜업 곡선이 바로 2부의 주제입니다

그런데 이 무게가 사는 것이 있습니다. 실행 중 프로파일을 보고 기계어를 다시 짜는 적응성(JIT 재최적화), 20년 쌓인 생태계, 세계 최고 수준의 GC들입니다

무게와 적응성의 교환. JVM은 이 교환을 가장 극단까지 밀어붙인 런타임입니다

### JVM이 자기 무게를 공격한다 — Leyden과 GraalVM

JVM 진영도 이 무게를 그냥 두고 보지 않습니다. 약점인 시동과 웜업을 정면으로 줄이는 두 흐름이 진행 중입니다

**Project Leyden — JVM은 유지하고, 시동 작업을 미리 해 둔다**

Leyden의 핵심은 훈련 실행(training run)입니다. 배포 전에 대표 워크로드로 앱을 한 번 돌려 보고, 그때 한 일을 캐시해 프로덕션 시동에서 반복하지 않는 방식입니다

- **JEP 483 (JDK 24)** — 클래스 로딩·링킹 결과를 훈련 실행 때 AOT 캐시로 저장합니다
- **JEP 514·515 (JDK 25)** — 워크플로를 단순화하고, AOT 메서드 프로파일까지 캐시해 JIT 웜업을 앞당깁니다

효과는 수치로 나와 있습니다. Spring PetClinic 시동이 4.486초(JDK 23)에서 2.604초(JDK 24, AOT 캐시)로 약 42% 줄었습니다

JVM을 버리지 않고 얻은 개선입니다

**GraalVM native-image — JVM을 버리고 통째로 AOT**

native-image는 closed-world 가정 위에서 동작합니다. 실행에 필요한 코드가 빌드 시점에 전부 보인다고 가정하고, 정적 분석으로 도달 가능한 코드만 모아 네이티브 실행 파일을 만듭니다

얻는 것은 극적입니다. 시동이 3~5초에서 50~200ms로 줄고(10~50배), 메모리는 400MB급에서 100MB급으로 약 4분의 1이 됩니다

대신 청구서가 실행이 아니라 빌드로 갑니다

- **빌드타임 폭발** — 어떤 코드와 객체에 도달할 수 있는지 전 프로그램을 추적하는 points-to 정적 분석, 거기에 전체 AOT 컴파일까지 겹치기 때문에 빌드가 분 단위(수 분에서 10분 이상)로 걸리고 RAM을 폭식합니다. 기본으로 시스템 메모리의 85%(최대 30GiB)까지 쓰고, 메모리가 부족하면 빌드 도중 GC로 시간의 절반 이상을 날리기도 합니다. 런타임은 ms, 빌드타임은 분인 셈이죠
- **closed-world 제약** — 리플렉션, 동적 프록시, JNI, 직렬화는 도달 가능성 메타데이터(힌트)를 명시해야 바이너리에 포함됩니다. "JVM에선 되는데 네이티브에선 ClassNotFound" 류의 디버깅이 생깁니다. Spring AOT·Quarkus·Micronaut가 힌트를 자동 생성해 완화하지만 제약 자체는 남습니다
- **GC 선택지 축소** — 기본이 작은 힙용 Serial GC입니다. G1은 Community Edition에 없고 Linux AMD64·AArch64 한정이며, ZGC는 미지원입니다. 대용량 힙·저지연 워크로드에는 오히려 불리합니다
- **정점 처리량 하락** — 런타임 적응을 포기한 값으로 JIT 대비 10~25% 낮습니다. PGO(실행 프로파일을 빌드에 되먹이는 최적화)를 쓰면 격차의 30~50%를 회복합니다

| | Leyden AOT 캐시 | GraalVM native-image |
| --- | --- | --- |
| JVM | 유지 | 버림 (네이티브 실행 파일) |
| 시동 개선 | 약 42% 단축 (PetClinic 기준) | 50~200ms (10~50배) |
| 새로 무는 비용 | 훈련 실행 파이프라인 | 분 단위 빌드 + closed-world 제약 |
| 정점 처리량 | JIT 그대로 | JIT 대비 -10~25% (PGO로 일부 회복) |
| 어울리는 곳 | 기존 JVM 워크로드 전반 | 서버리스·CLI·작은 힙 서비스 |

표가 보여 주듯 어느 쪽도 은탄환이 아닙니다. Leyden은 훈련 실행이라는 파이프라인 부담을, native-image는 빌드타임과 유연성이라는 비용을 대신 뭅니다

특히 native-image는 자주 죽고 켜지며 힙이 작은 워크로드(서버리스, CLI)에선 순이득이지만, 대용량 힙으로 오래 달리는 서버에선 GC 선택지와 정점 처리량 때문에 오히려 손해일 수 있습니다

JVM은 무거움을 없애려는 게 아닙니다. 무거움을 선택지로 바꾸려는 중입니다

## 정면 비교 — 런타임에 박았나, 라이브러리로 뺐나

수백만 개의 I/O 태스크를 소수의 스레드로 굴린다는 같은 문제를, Go와 Rust는 런타임 철학 그대로 반대로 풀었습니다. 이번 편의 정수가 이 비교에 있습니다

![Go's integrated runtime vs Rust's user-land assembly with the Waker seam](/diagrams/runtime-weight-zero-cost-vs-fat-5.svg)

왼쪽 Go부터 봅니다. 고루틴은 자기 스택을 가진 스택풀(stackful) 코루틴이고, 그 아래 GMP 스케줄러와 netpoller가 언어 런타임에 내장되어 있습니다

개발자는 조립할 게 없습니다. `go f()` 한 줄이면 끝이고, 스케줄러와 I/O 감시가 알아서 붙습니다

대신 모든 Go 바이너리가 이 무게를 집니다. 그리고 이 아키텍처를 바꿀 수 없습니다

work stealing이 싫어도, epoll 대신 io_uring을 쓰고 싶어도 런타임을 갈아 끼울 방법이 없습니다

오른쪽 Rust는 언어 런타임을 비워 두고 유저랜드에서 조립합니다. async 태스크는 컴파일러가 만든 상태 기계(Future)이고, 이를 굴리는 Executor(tokio)와 I/O 이벤트를 받는 Reactor(mio)를 라이브러리로 가져와 결합합니다

둘 사이의 유일한 이음새가 Waker입니다. I/O가 준비되면 Reactor가 Waker를 통해 해당 태스크를 Executor에 다시 깨워 넣는 표준 인터페이스죠

이 이음새 덕에 런타임을 통째로 교체할 수 있습니다. 대표 사례가 glommio입니다

work stealing 대신 thread-per-core와 io_uring을 택해, 코어 간 공유와 동기화를 없애고 캐시 지역성을 극대화한 아키텍처입니다

같은 언어에서 런타임 철학 자체를 갈아 끼우는 자유. Go에는 없는 것입니다

한 줄로 정리하면 이렇습니다. Go는 정답을 하나 정해서 박았고, Rust는 런타임을 비워 선택지를 열었습니다

이것이 Fat Runtime과 Zero-Cost 철학의 가장 순수한 표현입니다. 스택풀과 스택리스, 선점과 협력이라는 세부 차이는 [2편](/essays/thread-models-kernel-vs-user)에서 이미 다뤘습니다

## 세 언어 수렴 비교

| 항목 | Rust (std) | Go | JVM (전통) | JVM (native-image) |
| --- | --- | --- | --- | --- |
| 런타임 위치 | 거의 없음 | 바이너리에 내장 | 별도 VM 프로세스 | 네이티브에 내장 |
| 메모리 관리 | 컴파일타임 소유권 | 런타임 GC | 런타임 GC | 런타임 GC |
| 스케줄러 | 없음 (1:1) | GMP 내장 | OS + 가상 스레드 | OS + 가상 스레드 |
| 코드 생성 | AOT | AOT | JIT (적응형) | AOT |
| hello 바이너리 | 수백 KB | 약 1.5~2MB | 작음 (JRE 별도) | 중간 |
| 베이스라인 RSS | 최소 | 작음 | 큼 | 작음 |
| 시동 | ms | ms | 초 (웜업) | ms |
| 정점 처리량 | 높음·예측 가능 | 중상 | 최고 (JIT) | 중 (-10~25%) |
| 무게로 산 것 | 없음 (안 샀다) | 자족성 + 생산성 | 적응성 + 생태계 | 시동 + 풋프린트 |

마지막 행이 이 표의 요약입니다. 무게는 언제나 무언가의 구매 대금이었습니다

Rust는 아무것도 사지 않는 쪽을 택했습니다. 그 대신 지불도 실행 시점엔 없고, 전부 개발 시점으로 넘어갑니다

Go와 JVM은 같은 GC 언어지만 산 물건이 다릅니다. Go는 자족성과 빠른 시동이라는 운영 편의를, JVM은 적응성과 정점 처리량이라는 실행 성능을 샀습니다

native-image는 그 중간 지대를 새로 열었습니다. JVM 생태계를 유지하면서 무게 축만 Go 쪽으로 옮긴 대신, JVM이 산 적응성을 도로 반납했습니다

## 트레이드오프 소결 — 무게는 죄가 아니라 교환

- **Rust** — 런타임을 비워 가벼움, 예측 가능성, 어디에든 심을 수 있는 이식성을 얻고, 값은 개발 복잡도로 냅니다
- **Go** — 실속 런타임을 박아 자족성, 생산성, 빠른 시동을 얻고, Rust보다 무겁고 GC가 있다는 값을 냅니다. 균형점입니다
- **JVM** — 가상 머신을 통째로 얹어 런타임 적응성, 최고 정점 처리량, 생태계를 얻고, 최대 무게와 웜업을 냅니다. 지금은 Leyden과 GraalVM으로 그 무게마저 선택지로 되돌리는 중입니다

무게 자체는 좋은 것도 나쁜 것도 아닙니다. 물어야 할 것은 "이 무게로 무엇을 샀는가"입니다

다음 2부는 그 대표 구매품을 검증합니다. 실행 중에 기계어를 다시 짜는 JIT의 적응성이 정말 정적 컴파일을 이길 수 있는지입니다

## DevOps 관점에서 더 파고들 질문

1. **컨테이너 이미지 다이어트** — Go·Rust는 `FROM scratch`로 수 MB인데 JVM은 JRE 베이스가 필요합니다. jlink로 커스텀 런타임을 깎는 것과 native-image로 가는 것, 이미지 크기·빌드 복잡도·디버깅 난이도의 교환은 어떻게 저울질할까요?
2. **콜드 스타트와 서버리스** — Lambda나 Knative처럼 자주 죽고 켜지는 환경에서 초 단위 JVM 웜업은 치명적입니다. Leyden AOT 캐시(42% 단축)로 충분할까요, native-image(50~200ms)까지 가야 할까요?
3. **베이스라인 RSS와 빈패킹** — 200MB 리밋 파드에 Spring Boot(JVM 350~550MB)를 못 올리는 이유가 힙이 아니라 Metaspace·코드 캐시·스레드 스택이라면, request와 limit은 어떻게 잡아야 노드에 촘촘히 담길까요?
4. **native-image의 함정** — closed-world 가정 때문에 리플렉션·동적 프록시가 깨질 수 있습니다. 힌트 설정 비용, 처리량 -10~25%, 시동 이득을 놓고 어떤 워크로드에서 순이득일까요?
5. **Leyden 훈련 실행의 CI/CD 부담** — AOT 캐시는 대표 워크로드로 훈련 실행을 전제합니다. 이 단계를 파이프라인에 넣는 비용과, 캐시가 실제 트래픽과 어긋날 때의 리스크는 어떻게 관리할까요?
6. **멀티아치(ARM) 배포** — Go·Rust는 arm64 크로스 컴파일이 쉬워 Graviton용 멀티아치 이미지를 쉽게 만듭니다. JVM 바이트코드는 아키텍처 중립이지만 native-image는 타깃별로 따로 빌드해야 하죠. Graviton 전환 시 세 언어의 파이프라인 차이는 무엇일까요?

## 핵심 요약

- 언어 런타임은 앱과 커널 사이에서 자동으로 도는 관리 코드 층입니다. Rust는 거의 비웠고, Go는 실속 있게 채워 바이너리에 박았고, JVM은 가상 머신을 통째로 얹었습니다
- "무겁다"는 다섯 축(바이너리 크기, 베이스라인 RSS, 시동, 정상상태 오버헤드, 정점 처리량)으로 분해해야 합니다. 무게 순서는 정점 처리량에서 뒤집힙니다
- Rust는 RFC 230으로 그린 스레드를 제거하며 제로코스트를 지켰습니다. 소유권 검사가 GC를, monomorphization이 vtable을 대신하고, 값은 개발 복잡도로 냅니다
- JVM 프로세스에서 Java 힙은 세입자 하나일 뿐입니다. Metaspace, 코드 캐시, 스레드 스택, Direct Memory가 `-Xmx` 밖에서 RSS를 부풀립니다. 힙만 계산하면 K8s에서 OOMKilled를 만납니다
- Leyden은 JVM을 유지한 채 AOT 캐시로 시동을 약 42% 줄였고, native-image는 JVM을 버리고 ms 시동을 얻는 대신 분 단위 빌드타임과 closed-world 제약을 뭅니다
- Go는 스케줄러를 런타임에 박았고 Rust는 Waker 이음새로 라이브러리에 뺐습니다. 무게는 죄가 아니라 교환이며, 물을 것은 "이 무게로 무엇을 샀는가"입니다
