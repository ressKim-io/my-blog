> 이번 편에서는 시간 축을 앞당겨, 파드가 **막 기동되는 시점(cold start)** 의 특성을 확인합니다
> 결론을 먼저 요약하면 다음과 같습니다. **Java 프로세스는 초기 구동 시 성능이 저하되었다가 점진적으로 최적화되며, Go와 Rust는 구동 직후부터 최적화된 성능을 발휘합니다.** 이 웜업 지연은 오토스케일링과 서버리스 환경의 비용에 직접적인 영향을 미칩니다

## 웜업이란 무엇인가

핵심 용어를 먼저 정의합니다

**cold start(콜드 스타트, 인스턴스가 없는 유휴 상태에서 새 프로세스를 기동하여 첫 요청을 처리하는 초기화 과정)**: 실행 중인 인스턴스가 없는 상태에서 **새 프로세스를 기동하여** 첫 요청을 처리하는 과정입니다. 프로세스 시작, 초기화, 웜업에 소요되는 시간이 모두 첫 요청의 지연 시간에 포함됩니다

**warm-up(웜업, 프로세스가 기동된 이후에도 JIT 컴파일 등이 완료되지 않아 최대 처리 속도에 도달하기 전까지 일시적으로 성능이 저하되는 구간)**: 프로세스가 기동된 이후에도 **최대 성능에 도달하기 전까지** 성능이 저하되는 구간입니다. JVM 환경에서는 주로 JIT 컴파일이 완료되지 않았기 때문에 발생합니다

[7편](/essays/java-jit-c2-runtime-optimization)에서 HotSpot JVM의 **tiered compilation(계층형 컴파일, 바이트코드를 처음에는 빠른 인터프리터나 C1 컴파일러로 실행하다가 호출 빈도가 높은 핫 메서드만 선택적으로 고도화된 C2 컴파일러로 승급시키는 JVM 최적화 구조)** 구조를 다뤘습니다. 초기에는 바이트코드를 **인터프리터**로 실행하고, 호출 빈도가 높은 메서드만 **C1**(빠른 컴파일, 기본 최적화)을 거쳐 **C2**(느린 컴파일, 고도화된 최적화)로 승급시킵니다

이번 편에서는 **이 성능 향상 단계가 실제로 어느 정도의 격차를 나타내는지** 측정합니다

## 웜업 성능 향상 실측

단일 메서드를 반복 호출하며, 누적 호출 횟수에 따른 1회당 평균 실행 시간을 측정합니다

```java
static long work(int[] a, int seed) {
    long acc = seed;
    for (int i = 0; i < a.length; i++) {
        acc += (a[i] ^ acc) * 31L;
        acc ^= acc >>> 7;
    }
    return acc;
}
```

제한된 컨테이너 환경에서 측정한 결과입니다(JDK 25.0.3, 2코어)

```text
   200회    5578 ns  ████████████████████████████████████████████████████
   400회    1555 ns  ██████████████████████████████████
   600회    1633 ns  ████████████████████████████████████
   800회    1506 ns  █████████████████████████████████
  1000회    1186 ns  ██████████████████████████
  1200회     866 ns  ███████████████████
  1400회     830 ns  ██████████████████
  ...
 11000회     875 ns  ███████████████████
```

**최초 실행 속도는 5578ns였으며, 컴파일 정착 이후에는 830ns를 기록했습니다. 약 6.7배의 차이입니다**

![The warmup staircase, measured](/diagrams/cold-start-warmup-aot-remedies-1.svg)

그림의 가로축이 누적 호출 수, 세로축이 호출 1회당 성능입니다

**왼쪽의 초기 평탄 구간**은 인터프리터 모드입니다. 바이트코드를 한 줄씩 해석하므로 속도가 가장 느립니다

**중앙의 계단 구간**은 C1 모드입니다. 기계어로 컴파일되었으나 프로파일링 계측 코드가 포함되어 있어 최대 성능에는 미치지 못합니다

**오른쪽의 평탄 구간**은 C2 컴파일이 완료되어 정착된 상태입니다

그림 상단을 가로지르는 **수평선**은 Go와 Rust의 성능 궤적입니다. **AOT(Ahead-Of-Time, 애플리케이션 실행 전에 빌드 단계에서 사전에 코드를 네이티브 기계어로 컴파일해 두는 정적 컴파일 방식)** 로 사전 컴파일되므로 **실행 시점(t=0)부터 이 수준**을 유지하며, 웜업 단계가 존재하지 않습니다

### 컴파일 티어 전환 시점 분석

`-XX:+PrintCompilation` 플래그를 통해 해당 메서드의 승급 로그를 확인했습니다

```text
 9    8       3       Warmup::work (42 bytes)
 9    9 %     4       Warmup::work @ 6 (42 bytes)
19   90       4       Warmup::work (42 bytes)
20    8       3       Warmup::work (42 bytes)   made not entrant: not used
```

이 네 줄의 로그를 통해 JVM의 동적 최적화 과정을 확인할 수 있습니다

**첫 번째 줄:** 구동 9ms 시점에 **티어 3**(C1 + 프로파일링)으로 컴파일되었습니다

**두 번째 줄:** 동일한 9ms 시점에 **티어 4**(C2)로 진입했습니다. `%` 기호는 **OSR(On-Stack Replacement, 메서드가 종료되기를 기다리지 않고 실행 도중(루프 순회 중)에 스택 프레임을 유지한 채로 컴파일된 기계어 코드로 즉시 교체하는 런타임 최적화 기법)** 을 의미합니다. 메서드가 종료되기 전 **실행 도중에** 컴파일된 코드로 교체되었다는 뜻입니다. `@ 6`은 교체가 일어난 바이트코드 위치입니다

**세 번째 줄:** 19ms 시점에 정규 티어 4 코드가 완성되었습니다

**네 번째 줄:** 20ms 시점에 기존 티어 3 코드가 **`made not entrant`** 상태로 폐기되었습니다. [8편](/essays/jit-deopt-safepoint-tail-latency)에서 다룬 **deoptimization(역최적화, JIT 컴파일러가 가정한 최적화 전제가 깨졌거나 상위 컴파일 코드가 확보되었을 때 기존 컴파일 코드를 폐기하고 하위 컴파일러나 인터프리터 모드로 되돌리는 안전장치)** 로, 상위 최적화 코드가 확보됨에 따라 하위 티어 코드로의 진입을 차단하는 조치입니다

### JIT 승급 임계값에 대한 통념 검증

현업에서는 흔히 "메서드가 5,000번 이상 호출되어야 C2로 승급한다"고 알려져 있으며, JDK 25의 기본 설정값도 이와 일치합니다

| 플래그 | 값 |
|---|---|
| `Tier3InvocationThreshold` | 200 |
| `Tier4InvocationThreshold` | **5000** |
| `Tier4CompileThreshold` | 15000 |

그러나 실측 결과 **정착 지점은 1,200~1,400회 부근**에서 발생했습니다. 5,000회 임계치와는 상당한 격차가 있습니다

그 이유는 [7편](/essays/java-jit-c2-runtime-optimization)에서 확인했듯, 승급 조건이 호출 횟수 **단일 지표**로만 결정되지 않기 때문입니다

```text
i > Tier4InvocationThreshold                                     // 5000
  || (i > Tier4MinInvocationThreshold && i + b > Tier4CompileThreshold)  // 600, 15000
```

조건식 후반부의 `b`는 **backedge counter(백엣지 카운터, 반복문의 루프 끝에서 다시 시작점으로 되돌아가는 횟수를 계수하여 핫 루프를 조기에 감지하는 성능 지표)** 로, 반복문의 순회 횟수를 의미합니다

테스트에 사용된 `work()` 메서드는 내부에 512회의 루프를 포함합니다. 호출 횟수(`i`)가 600번만 누적되어도 `i + b` 합계가 15,000을 신속히 초과하므로, 5,000회를 대기하지 않고 **OSR을 통해 조기 승급**이 이루어졌습니다

**"5,000회 호출" 기준은 내부에 반복문이 없는 단순 메서드에만 적용되는 수치입니다.** 실제 비즈니스 로직은 대부분 반복문을 포함하므로 승급은 이보다 훨씬 이른 시점에 이루어집니다

## 컨테이너 환경이 웜업에 미치는 영향

컨테이너 환경의 자원 제한이 웜업에 개입하는 과정을 확인합니다

초기 가설은 `limits.cpu`를 강하게 제한할수록 승급에 필요한 호출 횟수가 늘어날 것이라는 점이었습니다. CPU를 4코어에서 0.5코어까지 낮춰 가며 확인했으나, **승급에 필요한 호출 횟수 자체는 변하지 않았습니다.** 임계값이 **호출 횟수** 기반이며 시간 기반이 아니기 때문입니다

다만 다른 파라미터에는 영향을 미쳤습니다

```bash
docker run --rm --cpus=2 eclipse-temurin:25-jdk java -XX:+PrintFlagsFinal -version
```

| `--cpus` | `CICompilerCount` | `ParallelGCThreads` |
|---|---|---|
| 4 | 3 | 4 |
| 2 | **2** | 2 |
| 1 | **2** | 0 |

**JIT 컴파일러 스레드 수가 컨테이너 CPU 한도에 비례하여 결정됩니다**

[20편](/essays/container-memory-accounting-cgroup)에서 다룬 `UseContainerSupport`가 `cpu.max`를 읽어 실효 코어 수를 계산하고, 이 값이 `CICompilerCount`로 연쇄 적용된 결과입니다

이는 다음을 의미합니다. **컴파일러 스레드가 줄어들면 컴파일 큐 처리 속도가 저하됩니다.** 승급 조건을 만족한 메서드도 대기 상태에 머물게 되며, 호출 횟수를 충족하고도 **최적화 코드 발급이 지연됩니다**

더 중요한 점은, 이 컴파일러 스레드가 **애플리케이션과 동일한 CPU 쿼터를 공유한다**는 사실입니다. 0.5코어로 제한된 파드에서 JIT 컴파일이 수행되는 동안, 그만큼의 CPU 자원이 비즈니스 로직 처리에서 제외됩니다

**파드 크기가 작을수록 웜업에 수반되는 비용이 증가합니다.** 밀도를 높이기 위해 파드를 소규모로 분할할 때, 이 비용이 반복적으로 발생함을 유의해야 합니다

## 초기 구동 구간의 메모리 프로파일 실측

현업의 통념 중 하나는 "구동 초기 시점에는 클래스 로딩·코드 캐시·arena가 일시적으로 팽창하여 RSS가 정상 상태 대비 **높게 스파이크를 형성한다**"는 것입니다

이를 검증하기 위해 대량의 클래스를 로드하고 2.5초간 연산을 수행한 뒤 **작업을 중지하고 유휴 상태로 전환**하는 프로그램을 컨테이너에서 200ms 주기로 관측했습니다

```text
   200ms     4 MiB  #
   400ms    94 MiB  #######################
   600ms   157 MiB  #######################################
   800ms   215 MiB  ########################################
  1000ms   282 MiB  ########################################
  1200ms   283 MiB  ########################################
  ...
  2600ms   283 MiB  ########################################   ← 여기서 작업 중지
  ...
  5400ms   283 MiB  ########################################
```

**스파이크 현상은 관측되지 않았습니다**

RSS는 약 1초 만에 4MiB에서 282MiB까지 상승합니다. **작업을 완전히 중단한 이후에도 메모리는 반환되지 않았습니다.** 3초 이상의 추가 유휴 시간에도 283MiB를 그대로 유지했습니다

![Startup memory climbs and never comes back down](/diagrams/cold-start-warmup-aot-remedies-2.svg)

그림의 **붉은 실선**이 실제 측정 궤적입니다. 급격히 상승한 뒤 평탄한 **plateau(고원, 초기 부하로 급격히 증가한 메모리 사용량이 연산 종료 후에도 하강하지 않고 평평하게 유지되는 고정 상주 상태)** 형태를 유지합니다

**점선**은 기존 통념이 예측했던 형태입니다. 스파이크 후 하락하는 궤적이나, 실제 측정 결과는 이와 달랐습니다

이유는 [20편](/essays/container-memory-accounting-cgroup)에서 확인한 커널의 메모리 회계 규칙과 일치합니다. **커널은 실제로 접촉된 페이지만을 계량하며,** JVM은 한 번 커밋한 힙 영역을 특별한 조건이 없는 한 OS로 반환하지 않습니다

따라서 초기 구간에 도달한 **최대 메모리(peak)가 곧 프로세스의 영구적인 상주 비용**으로 굳어집니다

운영 원칙은 다음과 같습니다. **`limits.memory`를 "평균적인 정상 상태 사용량"으로 설정하면 OOM 위험에 노출됩니다.** 이는 일시적 스파이크 때문이 아니라, **초기에 한 번 상승한 RSS가 하락하지 않기 때문입니다**

## 웜업 지연의 인프라 비용 환산

콜드 스타트와 웜업이 유발하는 성능 저하는 **지연 시간**과 **메모리** 두 측면에서 클라우드 과금 모델에 직접 영향을 미칩니다

### 오토스케일링(HPA) 환경의 효율성 저하

**HPA(Horizontal Pod Autoscaler, CPU나 메모리 부하 지표 증가를 감지하여 파드 인스턴스 개수를 수평적으로 자동 확장하는 쿠버네티스 컨트롤러)** 는 부하 증가를 감지하여 **사후적으로** 파드를 확장합니다

Java 프로세스가 최적화된 성능에 도달하기까지 시간이 소요되므로, 오토스케일러가 용량을 추가하더라도 **신규 파드는 한동안 기대 성능의 일부만 발휘합니다**

![New pods arrive at a fraction of their capacity](/diagrams/cold-start-warmup-aot-remedies-3.svg)

그림 상단의 **파란 선**은 유입되는 트래픽 부하를, 하단의 **초록색 계단형 영역**은 HPA가 추가한 파드 인스턴스를 나타냅니다

문제는 **붉은색 영역**입니다. 파드는 배치되었으나 아직 인터프리터 모드로 동작하여 실효 처리 용량이 부하를 따라가지 못하는 구간입니다

이를 회피하기 위해 실무에서는 여분 파드를 사전에 웜업해 두거나 트래픽 임계치 도달 이전에 조기 확장을 수행합니다. **두 방식 모두 유휴 자원을 점유하며, 유휴 자원은 곧 비용으로 이어집니다**

**웜업 시간이 길수록 확보해야 할 안전 여유가 커지고, 이는 곧 리소스 낭비로 이어집니다.** [21편](/essays/gc-headroom-bin-packing-density)에서 다룬 밀도 논리가 시간 축으로 확장된 것입니다

또한 `readinessProbe` 통과 기준을 애플리케이션의 단순 기동이 아닌 웜업 완료 시점으로 설정하지 않으면, **아직 인터프리터 모드로 동작하는 파드에 실제 트래픽이 라우팅됩니다.** 이는 배포 직후 사용자에게 지연 시간 스파이크로 나타납니다

### 서버리스(scale-to-zero) 환경의 제약

요청이 없을 때 인스턴스를 0으로 회수하고 호출 시 새로 기동하는 **scale-to-zero(스케일 투 제로, 요청이 없을 때는 유휴 서버 리소스 비용을 완전히 제로화하기 위해 컨테이너 인스턴스 수를 0개로 회수하는 서버리스 아키텍처 특성)** 서버리스 환경에서는 이 문제가 더욱 직접적으로 나타납니다

이 아키텍처에서는 **콜드 스타트 지연이 곧 사용자가 체감하는 응답 지연**과 일치합니다. 상대적으로 무겁고 웜업 기간이 긴 Java가 역사적으로 scale-to-zero 서버리스 채택에 불리했던 근본 원인입니다

## 웜업 오버헤드 해소를 위한 세 가지 기법

이러한 한계를 극복하기 위해 생태계는 크게 세 가지 접근 방식을 발전시켰습니다. **사전 캐시, 빌드 시점 네이티브 컴파일(AOT), 프로세스 상태 스냅샷** 방식입니다

### 기법 1: Leyden 프로젝트의 AOT 캐시(JDK 25 적용)

Project Leyden은 기존 JVM 아키텍처를 유지하면서, **훈련 실행(training run)** 을 통해 확보한 프로파일 데이터를 캐싱하여 다음 기동 시점의 웜업 속도를 단축시킵니다

이 기능은 **JDK 25 릴리스에 부분 통합**되어 실제 테스트가 가능합니다. 활성화 플래그의 존재 여부를 먼저 확인했습니다

```text
ccstr AOTCache          =  {product} {default}
ccstr AOTCacheOutput    =  {product} {default}
ccstr AOTMode           =  {product} {default}
```

이 아키텍처는 다음 JEP를 통해 구현됩니다. **JEP 483**(JDK 24)은 클래스 로딩, 파싱, 링킹 결과를 사전에 캐시합니다. **JEP 514**(JDK 25)는 단일 명령어로 캐시 생성을 단순화합니다. **JEP 515**(JDK 25)는 **자주 호출되는 핫 메서드의 프로파일 데이터**까지 캐시에 포함시켜, JIT 컴파일러가 기동 직후 최적화를 즉시 수행하도록 합니다

동일 컨테이너 환경에서 효과를 실측했습니다

```bash
# 훈련 실행 한 번으로 캐시 생성 (JEP 514)
java -XX:AOTCacheOutput=app.aot -cp app.jar App

# 그다음부터는 캐시를 물려 준다
java -XX:AOTCache=app.aot -cp app.jar App
```

| | 기동 시간 (중앙값 12회) |
|---|---|
| 기준선 | **42 ms** |
| AOT 캐시 | **35 ms** |

**초기 기동 시간이 약 17% 단축**되었으며, 캐시 파일의 용량은 11.8MB로 확인되었습니다

어떤 요인이 개선되었는지 클래스 로딩 로그를 통해 확인했습니다

```text
--- 캐시 없이 ---
    931  shared objects file
     14  java.base
      7  java.util.stream.Collectors
    총 958개
--- AOT 캐시 사용 ---
   1024  shared objects file
    총 1024개
```

이 결과에는 짚어야 할 부분이 있습니다. **기준선(baseline) 역시 이미 자체적인 최적화 혜택을 받고 있었습니다**

캐시 미적용 상태에서도 전체 958개 클래스 중 931개가 `shared objects file`에서 로드되었습니다. 이는 최신 JDK가 기본 적용하는 **CDS(Class Data Sharing, 다수의 JVM 프로세스가 공통적으로 사용하는 자바 기본 클래스들의 메타데이터를 메모리 매핑 아카이브 파일로 공유하여 클래스 로딩 시간과 메모리 사용량을 대폭 절감하는 JVM 최적화 기능)** 아카이브 덕분입니다. 즉 측정된 17%는 **이미 최적화가 적용된 기준선 위에 추가로 달성된 수치**입니다

테스트에 사용된 애플리케이션은 1,000여 개의 클래스만 포함된 경량 모듈입니다. **Spring과 같은 대형 프레임워크는 수만 개의 클래스 로딩·링킹 오버헤드가 수반되므로** AOT 캐시의 실효성이 더욱 커집니다. 따라서 이번 측정치인 17%를 실제 엔터프라이즈 환경에 그대로 대입해서는 안 됩니다

**수반되는 기회비용:** 대표성을 지닌 워크로드로 훈련 실행을 사전 수행해야 하며, 11.8MB 규모의 캐시 바이너리를 빌드·배포 파이프라인에서 관리해야 하는 운영 복잡도가 발생합니다

다만 **동적 클래스 로딩 등 JVM 고유의 런타임 동적성과 C2 컴파일러의 최대 처리량은 그대로 유지**됩니다

### 기법 2: GraalVM Native Image — 빌드 시점 정적 컴파일

애플리케이션 빌드 단계에서 **도달 가능한 모든 코드 경로를 정적 분석하여 네이티브 실행 파일로 컴파일**하는 기법입니다. 운영 환경에서는 JVM, JIT 컴파일러, 클래스 로딩 과정이 개입하지 않습니다

이 접근법이 제공하는 성능 이점은 상당합니다. 아래 수치는 자체 실측이 아닌 **여러 산업계 사례에서 보고된 평균 지표**입니다

- **초기 구동 시간:** JVM 모드 3~5초 → 네이티브 적용 시 **50~200ms**(최적화 시 10~100ms)
- **메모리 점유율:** 평균 250~400MB → **50~100MB** 수준
- **웜업 구간:** 존재하지 않음. 기동 즉시 컴파일된 기계어로 최대 성능 발휘

앞서 확인한 웜업 계단(5578ns → 830ns) 현상이 **원천적으로 제거**된다는 의미입니다

**수반되는 기회비용도 상당합니다**

첫째, **closed-world assumption(닫힌 세계 가정, 빌드 시점에 애플리케이션의 모든 도달 가능한 코드 경로와 바이트코드가 완결적으로 식별될 수 있으며 런타임에 동적으로 알 수 없는 코드가 추가되거나 로딩되지 않는다는 정적 컴파일 전제 조건)** 이 요구됩니다. 빌드 시점 정적 분석에서 식별되지 않은 코드는 최종 바이너리에 포함되지 않습니다. **리플렉션, 동적 프록시, JNI, 런타임 클래스 로딩**을 사용하는 경우 관련 메타데이터를 명시적으로 선언해야 하며, 누락 시 런타임 장애로 이어집니다

둘째, **최대 처리량의 하락**입니다. 런타임에서 수집되는 실제 프로파일 없이 빌드 시점 분석에만 의존하므로, 장기 실행 서비스에서는 JIT 대비 처리량이 낮아집니다. 흔히 보고되는 수치는 **10~25% 감소**입니다

다만 최근 기술 동향을 반영하면 이 통념은 갱신이 필요합니다

**PGO(Profile-Guided Optimization, 애플리케이션을 대표적인 워크로드로 사전 실행하여 수집한 분기 예측 및 메서드 호출 패턴 프로파일을 빌드 시점에 반영하여 정적 AOT 컴파일의 성능 한계를 극복하는 프로파일 기반 최적화 기법)** 를 적용하면 이 격차는 크게 완화됩니다. Oracle이 공개한 벤치마크에 따르면, PGO를 결합한 Native Image가 **초당 13,075 요청**을 처리해 동일 애플리케이션을 C2 JIT으로 구동한 **12,488 요청**을 오히려 **상회**했습니다

다만 이는 **벤더가 통제한 벤치마크**이므로 워크로드에 따라 결과가 달라질 수 있음을 감안해야 합니다

그럼에도 **"AOT 방식은 무조건 피크 성능이 낮다"는 도식은 더 이상 유효하지 않으며**, [9편](/essays/java-jit-inversion-conditions)에서 정립한 "장기 실행 환경에서는 JIT이 유리하다"는 명제 역시 PGO의 도입으로 그 경계가 점차 모호해지고 있습니다

### 기법 3: CRaC 및 SnapStart — 웜업 완료 상태의 스냅샷 복원

**CRaC(Coordinated Restore at Checkpoint, 웜업이 완료된 JVM 프로세스의 전체 메모리와 상태를 체크포인트 파일로 동결 저장해 두었다가 신규 구동 시 스냅샷으로부터 즉시 복원하여 초기 예열 시간을 제로화하는 메커니즘)** 는 **이미 웜업이 완료되어 실행 중인 JVM의 메모리 상태를 체크포인트(스냅샷)로 저장**하고, 이후 신규 프로세스 기동 시 이 스냅샷을 **복원**하는 방식입니다

이 기술의 기반은 리눅스 커널의 **CRIU(Checkpoint/Restore In Userspace, 리눅스 사용자 공간에서 실행 중인 프로세스의 메모리 매핑, 레지스터, 파일 디스크립터 상태를 통째로 파일에 저장하고 다른 시점에 정확히 복원하는 리눅스 커널 기술)** 기능입니다. 실행 중인 프로세스의 메모리 매핑 및 파일 디스크립터 상태를 통째로 파일에 저장하고 재현하는 기법입니다

복원된 프로세스는 **이미 JIT 컴파일이 완료되고 힙도 최적화된** 상태로 구동되므로, 기동 직후 최대 성능에 근접합니다

이 기술을 관리형 클라우드 Java 런타임에 적용한 대표 사례가 **AWS Lambda SnapStart**입니다. AWS는 함수 초기화 시점의 마이크로VM 스냅샷을 캡처하여 캐시합니다

공식 문서에 따르면 **"최대 10배 단축된 초기 구동 속도"** 를 제공하며, 초 단위였던 콜드 스타트를 1초 미만으로 단축합니다

**수반되는 기회비용**은 스냅샷 아키텍처의 특성에서 발생합니다. 스냅샷은 **외부 상태**를 그대로 동결합니다. 열려 있는 커넥션, 파일 핸들, 난수 생성기 시드, 시스템 시각, 그리고 **보안 시크릿**까지 포함됩니다

따라서 `beforeCheckpoint()`·`afterRestore()` 콜백 훅을 통해 커넥션을 재생성하고, **난수 시드와 시크릿 토큰을 복원 시점에 안전하게 갱신**해야 합니다. 스냅샷에 시크릿이 그대로 남는 것은 명백한 보안 위험입니다

### 최적화 기법 종합 비교

![Three remedies, three things given up](/diagrams/cold-start-warmup-aot-remedies-4.svg)

| 기법 | 핵심 원리 | 웜업 지연 | 최대 처리량 | 런타임 동적성 | 주된 기회비용 |
|---|---|---|---|---|---|
| **Leyden AOT 캐시** | 훈련 실행 프로파일의 디스크 캐싱 | 단축 | **유지** | **유지** | 훈련 실행 및 캐시 파이프라인 관리 |
| **Native Image** | 빌드 타임 정적 네이티브 컴파일 | **없음** | 10~25%↓ (PGO로 만회 가능) | **제약** | 리플렉션·JNI 사전 구성 복잡도 |
| **CRaC / SnapStart** | 웜업 완료 프로세스의 메모리 스냅샷 | **없음** | 유지 | 유지 | 상태 동기화 복잡도·스냅샷 보안 관리 |

인프라 운영 측면의 회피 기법도 존재합니다. 서버리스의 프로비저닝된 동시성, 쿠버네티스의 웜업 기준 readiness gate, 예측 기반 스케일링 등입니다

다만 이는 웜업을 **제거**하는 것이 아니라 잉여 리소스로 지연을 **은닉**하는 접근입니다

## 언어별 초기 구동 및 최적화 특성 비교

![Born warm, or warmed up](/diagrams/cold-start-warmup-aot-remedies-5.svg)

| 언어 | 초기 기동 성능 | 웜업 지연 | 사유 |
|---|---|---|---|
| **Rust** | 최고 성능(peak) 즉시 도달 | 없음 | 완전한 AOT 네이티브 컴파일과 최소화된 런타임 추상화. **구동 즉시 최적화 성능 확보** |
| **Go** | 최고 성능에 근접하여 시작 | 없음 | AOT 네이티브 컴파일. GC 및 고루틴 스케줄러 초기화만 ms 단위로 소요 |
| **Java** (기본) | **최대 성능 대비 6.7배 느림** | 있음 | 인터프리터로 시작 후 JIT 승급 과정을 거침. 다만 장시간 실행 시 **최고 수준의 성능(peak)** 확보 가능 |
| **Java + 최적화 기법** | 기법에 따라 향상 | 없음~단축 | 관리 복잡도 상승이라는 대가로 Go·Rust의 초기 구동 이점을 부분적으로 확보 |

표를 세로로 보면 각 언어의 설계 철학이 드러납니다

**Go와 Rust는 웜업이 없다는 이점을 언어 설계상 확보한 대신, 런타임 프로파일을 기반으로 코드를 재최적화하는 동적 적응 능력은 갖고 있지 않습니다**

Java는 반대로 초기 구동 시 성능 저하를 감수하는 대신, 장시간 실행되면 실제 트래픽 프로파일이 반영된 C2 코드에 도달합니다. [9편](/essays/java-jit-inversion-conditions)에서 다룬 장기 실행 환경에서의 성능 역전입니다

세 가지 기법은 이 두 극단 사이에서 **성능 우선순위를 조절하는 장치**입니다. Native Image는 Go·Rust 쪽(빠른 시작, 낮은 피크)으로 치우친 접근이며, Leyden과 CRaC는 빠른 시작을 확보하면서도 JIT 피크 성능을 유지하려는 절충안입니다

따라서 선택을 결정하는 핵심 질문은 다음과 같습니다

**콜드 스타트가 잦은가, 오래 도는가**

scale-to-zero 서버리스나 이벤트 함수처럼 **콜드 스타트가 빈번한 경우** Native Image와 SnapStart가 유리합니다. 장시간 연속 처리되는 고TPS 서비스라면 기본 JVM(Leyden 병행)의 JIT 피크 성능이 유리합니다

## 트레이드오프 소결

한 문장으로 정리하면 이렇습니다

**웜업 지연은 "런타임에서 코드를 동적으로 재최적화하는 기능"을 확보하기 위한 기회비용입니다**

JIT 컴파일러는 이 기능을 통해 장기 실행 시의 최고 성능을 확보하며, 그 대가로 **초기 구동 구간의 성능 저하**를 감수합니다

Go와 Rust는 이 기능을 설계 단계에서 배제함으로써 웜업 오버헤드를 구조적으로 면제받았습니다

Java의 세 가지 최적화 기법은 **이 기능의 어느 부분을 얼마나 양보할지 조절하는 장치**입니다. Native Image는 동적성과 피크 성능을, CRaC는 상태 관리 복잡도를, Leyden은 빌드 파이프라인 관리 부담을 대가로 지불합니다

**어떠한 최적화 기법도 대가 없이 웜업을 완전히 제거하지는 못합니다.** 5부 전반을 관통하는 원칙입니다

## 더 파고들 질문

1. **Readiness Probe와 웜업 완료 동기화** — `readinessProbe`를 프로세스 기동이 아닌 웜업 완료 시점을 기준으로 설정하려면 어떤 신호를 사용해야 할까요(예: 예열 요청 N회 처리 성공, 컴파일 큐 소진 여부 등)? 이 설정이 누락되면 배포 직후 사용자에게 어떤 지연 패턴이 관측될까요?
2. **소형 파드의 웜업 비용 정량화** — `CICompilerCount`가 컨테이너 CPU 한도에 종속됨을 확인했습니다. `limits.cpu`를 0.5로 제한한 환경에서 JIT 컴파일러 스레드가 애플리케이션과 CPU 쿼터를 경합하는 비용을 어떻게 계측해야 할까요? 이 비용을 파드 분할 여부를 결정하는 근거로 어떻게 반영해야 할까요?
3. **최적화 기법의 적합성 판별** — 하루 수천 회 콜드 스타트가 발생하는 이벤트 함수와, 수일간 연속 처리되는 고TPS 게이트웨이가 있습니다. 각각에 Native Image·CRaC·Leyden 중 무엇을 적용해야 하며, 그 근거를 "콜드 스타트 빈도 × 피크 처리량 민감도"로 어떻게 설명할 수 있을까요?
4. **PGO 도입에 따른 JIT/AOT 성능 역전의 한계** — Native Image에 PGO를 결합하면 피크 성능 격차가 크게 줄어들며, 벤더 벤치마크에서는 JIT을 상회하기도 합니다. 그렇다면 9편에서 정립한 "장기 실행 환경에서는 JIT이 우월하다"는 명제는 어디까지 유효할까요? 실제 워크로드에서 이를 어떻게 검증해야 할까요?
5. **CRaC 스냅샷의 상태 관리** — 체크포인트에 동결된 DB 커넥션·난수 시드·시크릿 토큰을 복원 후 어떻게 안전하게 재생성해야 할까요? 스냅샷에 시크릿이 그대로 남는 위험을 배포 파이프라인 단에서 어떻게 차단해야 할까요?
6. **초기 RSS 팽창 기반의 리소스 한도 산정** — RSS가 초기에 상승한 뒤 하락하지 않는 특성을 확인했습니다. `limits.memory`를 평균 사용량으로 설정하면 어떤 위험이 있으며, 피크 기준으로 설정하면 노드 밀도가 얼마나 저하될까요? 부하 테스트에서 이 초기 피크를 정확히 측정하는 방법은 무엇일까요?

## 핵심 요약

- **웜업 성능 향상을 실측했습니다.** 동일 메서드가 인터프리터 모드에서 **5578ns**, 정착 후 **830ns**를 기록해 **약 6.7배의 성능 격차**가 확인되었습니다
- `PrintCompilation` 로그를 통해 **티어3 → 티어4 OSR(실행 도중 교체) → 티어4 → 기존 코드 `made not entrant`(역최적화)** 순서를 확인할 수 있습니다
- **"C2는 5,000번 호출되어야 승급한다"는 통념은 반복문이 없는 메서드에만 해당합니다.** 승급 조건에 **백엣지 카운터**가 함께 반영되므로, 반복문을 포함한 메서드는 훨씬 이른 시점에 승급합니다. 실측에서는 1,200~1,400회에서 정착이 확인되었습니다
- **컨테이너 CPU 한도가 `CICompilerCount`를 결정합니다**(4코어→3, 2코어→2). 호출 횟수 임계값 자체는 변하지 않지만 **컴파일 큐 처리가 지연되며**, 컴파일러 스레드가 애플리케이션과 **동일한 CPU 쿼터를 공유합니다.** 파드 크기가 작을수록 웜업 비용이 증가합니다
- **초기 구동 구간의 메모리 팽창은 스파이크 형태가 아닌 것으로 확인되었습니다.** RSS는 1초 만에 282MiB까지 **상승한 뒤 작업을 중단해도 하락하지 않습니다.** **최대치가 곧 상주 비용**으로 굳어지므로, `limits.memory`를 평균 사용량으로 설정해서는 안 됩니다
- **Leyden의 AOT 캐시는 JDK 25에 이미 부분 통합되어 있습니다.** 훈련 실행 한 번으로 기동 시간이 **42ms → 35ms(약 17%)** 로 단축되었으며, 캐시 파일 용량은 11.8MB였습니다. 다만 **기준선도 이미 CDS 아카이브를 통해 931개 클래스를 공유 파일에서 로드하고 있었고**, 테스트 애플리케이션이 1,000여 개 클래스 규모의 경량 모듈이므로 **이 17%를 대형 프레임워크에 그대로 대입해서는 안 됩니다**
- **Native Image의 최대 처리량 저하(10~25%)는 PGO를 통해 상당 부분 보완됩니다.** 벤더 벤치마크에서는 JIT을 상회하는 경우도 확인됩니다(13,075 vs 12,488 req/s). 벤더 측정치이므로 그대로 신뢰하기는 어렵지만, **"AOT 방식은 무조건 피크 성능이 낮다"는 도식은 더 이상 유효하지 않습니다**
- **CRaC·SnapStart는 웜업이 완료된 프로세스의 상태를 스냅샷으로 저장합니다.** AWS 공식 문서 기준 최대 10배 빠른 구동 속도를 제공합니다. 대가는 **외부 상태와 시크릿이 스냅샷에 그대로 남는다는 점**이며, 콜백 훅을 통해 반드시 재생성해야 합니다
- **웜업은 런타임의 동적 재최적화 능력에 대한 기회비용입니다.** 세 가지 기법은 이 능력의 어느 부분을 양보할지 조절하는 장치일 뿐, 대가 없이 웜업을 제거하지는 못합니다
