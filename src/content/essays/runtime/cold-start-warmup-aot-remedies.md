> **시리즈 "커널과 런타임으로 톺아보는 Rust · Go · Java"의 21편**
> [18](/essays/container-memory-accounting-cgroup)·[19](/essays/gc-headroom-bin-packing-density)·[20편](/essays/offheap-oomkilled-nmt)은 전부 **정상 상태**의 메모리 이야기였습니다
> 이번 편은 시간 축을 앞으로 감아, 파드가 **막 뜨는 순간**을 봅니다. 이 구간은 물리가 다릅니다
> 한 문장으로 앞당겨 말하면 이렇습니다. **Java 프로세스는 차게 태어나 데워집니다. Go와 Rust는 태어날 때 이미 데워져 있습니다.** 그 데워지는 시간이 오토스케일링과 서버리스의 경제성을 직접 때립니다

## 웜업이란 무엇인가

용어부터 세웁니다

**콜드 스타트(cold start)** 는 실행 중인 인스턴스가 없어 **새 프로세스를 처음부터 띄워** 첫 요청을 처리하는 상황입니다. 프로세스 시작 + 초기화 + 웜업이 전부 첫 요청의 지연에 얹힙니다

**웜업(warm-up)** 은 프로세스가 살아난 뒤에도 **한동안 피크 성능이 안 나오는** 구간입니다. JVM에서 이건 주로 JIT 컴파일이 아직 안 끝나서입니다

[6편](/essays/java-jit-c2-runtime-optimization)에서 티어드 컴파일을 뜯었습니다. HotSpot은 처음엔 바이트코드를 **인터프리터**로 실행하고, 자주 불리는 메서드만 **C1**(빠른 컴파일·보통 코드)을 거쳐 **C2**(느린 컴파일·최고 코드)로 승급시킨다고요

그때는 기전을 봤습니다. 이번엔 **그 계단이 실제로 얼마나 가파른지**를 잽니다

## 계단을 직접 재 봤습니다

메서드 하나를 반복해서 부르며, 배치마다 호출 1회당 평균 시간을 찍습니다

```java
static long work(int[] a, int seed) {
    long acc = seed;
    for (int i = 0; i < a.length; i++) {
        acc += (a[i] ^ acc) * 31L;
        acc ^= acc >>> 7;
    }
    return acc;
}
```

컨테이너 안에서 돌린 결과입니다 (JDK 25.0.3, 2코어)

```text
   200회    5578 ns  ████████████████████████████████████████████████████
   400회    1555 ns  ██████████████████████████████████
   600회    1633 ns  ████████████████████████████████████
   800회    1506 ns  █████████████████████████████████
  1000회    1186 ns  ██████████████████████████
  1200회     866 ns  ███████████████████
  1400회     830 ns  ██████████████████
  ...
 11000회     875 ns  ███████████████████
```

**첫 배치가 5578ns, 정착이 830ns입니다. 6.7배**

![The warmup staircase, measured](/diagrams/cold-start-warmup-aot-remedies-1.svg)

그림의 가로축이 누적 호출 수, 세로축이 호출 1회당 성능입니다

**왼쪽 맨 아래 평평한 구간**이 인터프리터입니다. 바이트코드를 한 줄씩 해석하니 가장 느립니다

**가운데 계단**이 C1입니다. 기계어로 컴파일됐지만 프로파일을 수집하느라 계측 코드가 붙어 있어 최고 속도는 아닙니다

**오른쪽 평평한 구간**이 C2 정착입니다

그리고 그림 위를 가로지르는 **직선**이 Go와 Rust입니다. AOT 네이티브 기계어라 **t=0부터 저 높이**입니다. 계단이 없습니다

### 티어가 실제로 언제 바뀌나

`-XX:+PrintCompilation`으로 그 메서드의 승급 이력을 뽑았습니다

```text
 9    8       3       Warmup::work (42 bytes)
 9    9 %     4       Warmup::work @ 6 (42 bytes)
19   90       4       Warmup::work (42 bytes)
20    8       3       Warmup::work (42 bytes)   made not entrant: not used
```

네 줄에 이 편의 절반이 들어 있습니다

**첫 줄** — 9ms 시점에 **티어 3**(C1 + 프로파일링)으로 컴파일됐습니다

**둘째 줄** — 같은 9ms에 **티어 4**(C2)로 갔습니다. `%` 표시가 붙어 있는데, 이건 **OSR(On-Stack Replacement)** 입니다. 메서드가 **실행되는 도중에** 그 자리에서 컴파일된 코드로 갈아탔다는 뜻이죠. `@ 6`은 갈아탄 바이트코드 위치입니다

**셋째 줄** — 19ms에 일반 티어 4 코드가 완성됐습니다

**넷째 줄** — 20ms에 아까 만든 티어 3 코드가 **`made not entrant`**, 즉 폐기됐습니다. 6편에서 본 **역최적화**입니다. 더 좋은 코드가 생겼으니 옛 코드로는 더 이상 들어가지 못하게 막습니다

### 여기서 흔한 요약이 깨집니다

"C2는 5000번 불려야 승급한다"는 말이 널리 퍼져 있습니다. 실제로 JDK 25의 기본값이 그렇습니다

| 플래그 | 값 |
|---|---|
| `Tier3InvocationThreshold` | 200 |
| `Tier4InvocationThreshold` | **5000** |
| `Tier4CompileThreshold` | 15000 |

그런데 위 측정에서 **정착이 1200~1400회쯤**에 일어났습니다. 5000회 근처가 아닙니다

[6편](/essays/java-jit-c2-runtime-optimization)에서 이미 답을 봤습니다. 승급 조건이 호출 횟수 **단독**이 아니기 때문입니다

```text
i > Tier4InvocationThreshold                                     // 5000
  || (i > Tier4MinInvocationThreshold && i + b > Tier4CompileThreshold)  // 600, 15000
```

뒤쪽 절반의 `b`가 **백엣지 카운터**, 즉 루프를 몇 바퀴 돌았는지입니다

제 `work()`는 안에서 512바퀴짜리 루프를 돕니다. 호출이 600번만 돼도 `i + b`가 15000을 훌쩍 넘죠. 그래서 5000회를 기다리지 않고 **OSR로 먼저 승급**했습니다

**"5000번"은 루프가 없는 메서드에만 맞는 이야기입니다.** 실제 서버 코드는 대부분 루프를 품고 있으니 훨씬 일찍 승급합니다

## 컨테이너가 웜업에 개입하는 방식

여기서 이번 편의 K8s 고리가 나옵니다

처음엔 `limits.cpu`를 조이면 웜업이 길어질 거라 예상하고 CPU를 4 → 0.5로 줄여 가며 재 봤습니다. 그런데 **"몇 번 불려야 승급하는가"는 전혀 안 변했습니다.** 당연합니다. 임계값이 **호출 횟수** 기반이지 시간 기반이 아니니까요

대신 다른 곳이 변합니다

```bash
docker run --rm --cpus=2 eclipse-temurin:25-jdk java -XX:+PrintFlagsFinal -version
```

| `--cpus` | `CICompilerCount` | `ParallelGCThreads` |
|---|---|---|
| 4 | 3 | 4 |
| 2 | **2** | 2 |
| 1 | **2** | 0 |

**JIT 컴파일러 스레드 수가 CPU 한도에서 파생됩니다**

[18편](/essays/container-memory-accounting-cgroup)에서 본 `UseContainerSupport`가 `cpu.max`를 읽어 실효 코어 수를 정하고, 그게 `CICompilerCount`까지 연쇄적으로 내려온 결과입니다

의미는 이렇습니다. **컴파일러 스레드가 적으면 컴파일 큐가 늦게 빠집니다.** 승급 조건을 만족한 메서드가 줄을 서서 기다리죠. 호출 횟수는 똑같이 채웠는데 **기계어가 늦게 나옵니다**

그리고 그 컴파일러 스레드는 **애플리케이션과 같은 CPU 쿼터를 나눠 씁니다.** 0.5코어 파드에서 JIT이 컴파일을 돌리는 동안, 그 CPU는 비즈니스 로직 몫에서 빠져나간 겁니다

**작은 파드일수록 웜업이 비싸집니다.** 밀도를 높이려고 파드를 잘게 쪼갠 대가가 여기서 또 한 번 청구됩니다

## 시작 구간의 메모리 — 통념이 빗나갔습니다

원본 정리에는 "시작 시점의 RSS가 정상 상태보다 **높이 튄다**"고 적혀 있었습니다. 클래스 로딩·코드 캐시·arena가 한꺼번에 부풀어 스파이크를 만든다는 이야기였죠

그래서 재 봤습니다. 클래스를 잔뜩 로드하고 2.5초간 일한 뒤 **일을 멈추고 대기**하는 프로그램을, 컨테이너 안에서 200ms 간격으로 관찰했습니다

```text
   200ms     4 MiB  #
   400ms    94 MiB  #######################
   600ms   157 MiB  #######################################
   800ms   215 MiB  ########################################
  1000ms   282 MiB  ########################################
  1200ms   283 MiB  ########################################
  ...
  2600ms   283 MiB  ########################################   ← 여기서 일을 멈춤
  ...
  5400ms   283 MiB  ########################################
```

**스파이크가 없습니다**

RSS는 1초 만에 4MiB에서 282MiB까지 올라갑니다. 그리고 **일을 멈춰도 내려오지 않습니다.** 3초를 더 놀아도 283MiB 그대로입니다

![Startup memory climbs and never comes back down](/diagrams/cold-start-warmup-aot-remedies-2.svg)

그림의 **붉은 곡선**이 실측 궤적입니다. 가파르게 올라간 뒤 **평평한 고원**이 됩니다

**점선**이 원본 정리가 예상했던 모양입니다. 튀었다가 내려오는 스파이크죠. 그런 건 없었습니다

이유는 [18편](/essays/container-memory-accounting-cgroup)에서 세운 규칙에 있습니다. **커널은 접촉된 페이지를 셉니다.** 그리고 JVM은 한 번 커밋한 힙을 웬만해선 OS에 돌려주지 않습니다

그래서 **피크가 곧 상주 비용**이 됩니다

실무 결론은 원본 정리와 같지만 근거가 다릅니다. **`limits.memory`를 "평상시 사용량"으로 잡으면 안 됩니다.** 스파이크가 잠깐 튀어서가 아니라, **한 번 오른 RSS가 안 내려오기 때문**입니다

## 웜업 세금이 왜 돈인가

두 얼굴입니다. **지연**과 **메모리**. 각각 클라우드의 두 과금 모델을 때립니다

### 오토스케일링에서

HPA는 부하가 오르면 파드를 **사후에** 늘립니다

그런데 새 Java 파드가 제 성능을 내는 데 시간이 걸리면, 오토스케일러가 용량을 추가해도 **그 용량이 한동안 반값짜리**입니다

![New pods arrive at a fraction of their capacity](/diagrams/cold-start-warmup-aot-remedies-3.svg)

그림 위쪽 **파란 선**이 들어오는 부하이고, 아래 **초록 계단**이 HPA가 추가한 파드 수입니다

문제는 **붉은 영역**입니다. 파드는 떴는데 아직 인터프리터로 굴러 실효 용량이 못 따라오는 구간이죠

그래서 실무에서는 여분 파드를 미리 데워 두거나 트래픽보다 앞서 스케일 아웃합니다. **둘 다 노는 자원이고, 노는 자원은 곧 청구서**입니다

**웜업이 길수록 안전 여분이 커지고, 여분이 곧 비용**입니다. [19편](/essays/gc-headroom-bin-packing-density)의 밀도 논리가 시간 축으로 확장된 것입니다

또 하나. `readinessProbe`를 웜업 완료 기준으로 잡지 않으면, **아직 인터프리터로 도는 파드에 트래픽이 라우팅됩니다.** 배포 직후 지연 스파이크가 사용자에게 새어 나가죠. 준비성 게이트를 "떴다"가 아니라 **"데워졌다"** 에 맞춰야 합니다

### 서버리스에서

여기선 더 직접적입니다. 요청이 없으면 인스턴스를 0으로 내리고, 오면 새로 띄웁니다

**콜드 스타트 지연이 곧 사용자 지연**입니다. 그래서 Java는 역사적으로 scale-to-zero 서버리스에 불리했습니다

## 웜업을 없애는 세 처방

접근이 셋으로 갈립니다. **하나는 캐시로 데우고, 하나는 아예 네이티브로 굽고, 하나는 데워진 상태를 스냅샷으로 뜹니다**

### 처방 1: Leyden의 AOT 캐시 — 직접 돌려 봤습니다

Project Leyden은 JVM을 버리지 않습니다. 대신 **훈련 실행(training run)** 에서 얻은 것을 캐시해 다음 실행의 시작을 당깁니다

**그리고 이건 JDK 25에 이미 들어와 있습니다.** 플래그가 실재하는지부터 확인했습니다

```text
ccstr AOTCache          =  {product} {default}
ccstr AOTCacheOutput    =  {product} {default}
ccstr AOTMode           =  {product} {default}
```

세 JEP가 이걸 만듭니다. **JEP 483**(AOT 클래스 로딩·링킹, JDK 24)이 클래스를 읽고 파싱·링크한 결과를 캐시하고, **JEP 514**(JDK 25)가 캐시 생성을 명령 한 줄로 줄였고, **JEP 515**(JDK 25)가 **핫 메서드 프로파일**까지 캐시에 담아 JIT이 프로파일을 처음부터 갖고 시작하게 합니다

컨테이너에서 직접 재 봤습니다

```bash
# 훈련 실행 한 번으로 캐시 생성 (JEP 514)
java -XX:AOTCacheOutput=app.aot -cp app.jar App

# 그다음부터는 캐시를 물려 준다
java -XX:AOTCache=app.aot -cp app.jar App
```

| | 기동 시간 (중앙값 12회) |
|---|---|
| 기준선 | **42 ms** |
| AOT 캐시 | **35 ms** |

**약 17% 단축**됐습니다. 캐시 크기는 11.8MB입니다

무엇이 줄었는지 클래스 로딩 로그로 확인했습니다

```text
--- 캐시 없이 ---
    931  shared objects file
     14  java.base
      7  java.util.stream.Collectors
    총 958개
--- AOT 캐시 사용 ---
   1024  shared objects file
    총 1024개
```

정직하게 짚을 게 있습니다. **기준선도 이미 이득을 보고 있었습니다**

958개 중 931개가 이미 `shared objects file`에서 로드됐죠. JDK가 기본으로 싣는 **CDS 아카이브**입니다. 즉 제가 잰 17%는 **이미 최적화된 기준선 위에 더 얹은 것**입니다

그리고 제 테스트 앱은 클래스 천 개짜리 장난감입니다. **스프링 같은 실제 프레임워크는 클래스가 수만 개**라 로딩·링킹 비중이 훨씬 커지고, 그만큼 이득도 커집니다. **제 17%를 실제 앱에 그대로 대입하면 안 됩니다**

**대가**는 이렇습니다. 대표 워크로드로 훈련 실행을 해야 하고, 캐시를 빌드·배포·버전 관리해야 합니다. 11.8MB짜리 산출물이 하나 더 생기는 거죠

대신 **JVM의 동적성과 피크 처리량은 그대로**입니다. 닫힌 세계로 굽는 게 아니니까요

### 처방 2: GraalVM native-image — 빌드 시점에 굽습니다

빌드 시점에 **도달 가능한 코드 전부를 정적 분석해 네이티브 실행 파일**로 컴파일합니다. 런타임에 JVM도, JIT도, 클래스 로딩도 없습니다

보고되는 효과는 극적입니다. 아래는 제가 잰 값이 아니라 **여러 사례에서 보고되는 수치**입니다

- **시작**: JVM 모드 3~5초 → 네이티브 **50~200ms** (전형적으로 10~100ms)
- **메모리**: 250~400MB → **50~100MB** 수준
- **웜업**: 없음. 처음부터 네이티브 코드입니다

앞서 잰 웜업 계단(5578ns → 830ns)이 **아예 사라진다**는 뜻입니다

**대가가 큽니다**

**닫힌 세계 가정(closed-world)** 이 첫째입니다. 빌드 시점에 안 보이는 코드는 바이너리에 없습니다. **리플렉션·동적 프록시·JNI·런타임 클래스 로딩**은 빌드 메타데이터로 명시해야 하고, 안 하면 런타임에 깨집니다

**피크 처리량 하락**이 둘째입니다. 런타임 프로파일 없이 빌드타임 분석만으로 최적화하니, 오래 도는 서비스에서 JIT보다 처리량이 낮습니다. 흔히 보고되는 폭이 **정상 상태 10~25%↓** 입니다

그런데 여기서 원본 정리를 한 번 갱신해야 합니다

**PGO(프로파일 기반 최적화)를 붙이면 이 격차가 크게 줄어듭니다.** 오라클이 공개한 벤치마크에서는 PGO를 붙인 native-image가 **초당 13,075 요청**으로, 같은 앱을 C2 JIT으로 돌린 **12,488 요청**을 오히려 **넘어섰습니다**

단서를 답니다. **벤더 자신의 벤치마크**이고 워크로드에 따라 결과가 크게 다릅니다. 그대로 믿을 숫자는 아닙니다

다만 **"AOT는 무조건 피크가 낮다"** 는 도식은 이제 안전하지 않습니다. [7편](/essays/java-jit-inversion-conditions)에서 세운 "롱런에서 JIT이 역전한다"는 명제도, PGO가 런타임 프로파일의 일부를 빌드타임으로 당겨 오면서 **경계가 흐려지고 있습니다**

### 처방 3: CRaC와 SnapStart — 데워진 프로세스를 얼립니다

**CRaC(Coordinated Restore at Checkpoint)** 는 **이미 웜업된 실행 중 JVM을 체크포인트(스냅샷)로 뜨고**, 다음엔 그 스냅샷에서 **복원**해 시작합니다

바탕 기술은 **CRIU(Checkpoint/Restore In Userspace)** 입니다. 실행 중인 프로세스의 메모리와 파일 디스크립터 상태를 통째로 파일에 떠 두었다가 되살리는 리눅스 기능이죠

복원된 프로세스는 **이미 JIT 컴파일되고 힙도 잡힌** 상태라 시작하자마자 피크에 가깝습니다

**AWS Lambda SnapStart**가 이 접근을 매니지드 Java 런타임에 구현한 것입니다. AWS는 함수 초기화 시점의 마이크로VM 스냅샷을 떠서 캐시해 둡니다

효과는 AWS 공식 표현으로 **"최대 10배 빠른 시작"** 입니다. 초 단위이던 콜드 스타트가 1초 아래로 내려갑니다

**대가**는 스냅샷의 성질에서 나옵니다. 스냅샷은 **외부 상태**를 그대로 얼립니다. 열린 커넥션, 파일 핸들, 난수 시드, 시각, 그리고 **시크릿**까지요

그래서 `beforeCheckpoint()`·`afterRestore()` 훅으로 커넥션을 닫았다 다시 열고, **난수와 시크릿을 복원 후 재생성**해야 합니다. 스냅샷에 시크릿이 박제되는 건 명백한 보안 문제입니다

### 세 처방 비교

![Three remedies, three things given up](/diagrams/cold-start-warmup-aot-remedies-4.svg)

| 처방 | 무엇을 하나 | 웜업 | 피크 처리량 | 동적성 | 주된 대가 |
|---|---|---|---|---|---|
| **Leyden AOT 캐시** | 훈련 실행을 캐시 | 단축 | **유지** | **유지** | 훈련 실행·캐시 관리 |
| **native-image** | 빌드타임 네이티브 | **없음** | 10~25%↓ (PGO로 만회 가능) | **제약** | 리플렉션 설정·빌드 복잡도 |
| **CRaC / SnapStart** | 웜업된 프로세스를 스냅샷 | **없음** | 유지 | 유지 | 외부 상태 훅·스냅샷 보안 |

운영 쪽 회피책도 있습니다. 서버리스의 프로비저닝된 인스턴스, K8s의 웜업 기준 readiness gate, 예측 스케일링

다만 이들은 웜업을 **없애는** 게 아니라 **비용으로 가리는** 쪽입니다

## 세 언어 수렴 — 태생 웜업과 데워짐

![Born warm, or warmed up](/diagrams/cold-start-warmup-aot-remedies-5.svg)

| | 시작 직후 성능 | 웜업 | 왜 |
|---|---|---|---|
| **Rust** | 피크 | 없음 | AOT 네이티브 + 런타임 초기화 최소. **태어날 때 데워져 있음** |
| **Go** | 피크에 근접 | 없음 | AOT 네이티브. 런타임 초기화(GC·스케줄러)만 ms 단위 |
| **Java** (기본) | **6.7배 느림** | 있음 | 인터프리터로 시작해 JIT 승급. 대신 **롱런 피크는 최고** |
| **Java + 처방** | 처방에 따라 | 없음~단축 | Go·Rust의 성질을 **대가를 치르고 수입** |

세로로 읽으면 이렇습니다

**Go와 Rust는 웜업이 없다는 이점을 태생으로 갖는 대신, 런타임 프로파일로 코드를 재최적화하는 능력이 없습니다**

Java는 반대로 차게 시작하는 대가를 치르는 대신, 오래 돌면 실행 실측을 녹여 넣은 C2 코드에 도달합니다. [7편](/essays/java-jit-inversion-conditions)의 롱런 역전이죠

세 처방은 이 둘 사이에서 **위치를 옮기는 장치**입니다. native-image는 Go·Rust 쪽(빠른 시작·낮은 피크)으로, Leyden과 CRaC는 "빠른 시작을 얻되 JIT 피크는 지키는" 절충으로

그래서 선택을 가르는 질문은 하나입니다

**콜드 스타트가 잦은가, 오래 도는가**

scale-to-zero 서버리스나 이벤트 함수처럼 **자주 차게 시작**하면 native-image와 SnapStart가 유리합니다. 며칠씩 도는 고TPS 서비스면 기본 JVM(+Leyden)의 JIT 피크가 유리하고요

## 트레이드오프 소결

한 문장으로 정리하면 이렇습니다

**웜업은 "런타임에 코드를 재최적화하는 능력"의 대가입니다**

JIT은 그 능력으로 롱런 피크를 삽니다. 그리고 그 대가를 **시작 구간의 느림**으로 냅니다

Go와 Rust는 그 능력을 애초에 포기해 웜업을 면제받았습니다

Java의 세 처방은 **능력의 어느 부분을 얼마나 포기할지 정하는 다이얼**입니다. native-image는 동적성과 피크를 내주고, CRaC는 상태 관리 복잡도를 내주고, Leyden은 빌드 파이프라인의 수고를 냅니다

**어느 쪽도 공짜로 웜업을 없애지 못했습니다.** 5부 내내 본 그 법칙입니다

## 더 파고들 질문

1. **readiness를 '데워짐'에 맞추기** — `readinessProbe`를 프로세스 기동이 아니라 웜업 완료 기준으로 잡으려면 무엇을 신호로 쓰겠습니까(예열 요청 N회 후 통과, 컴파일 큐 소진 등)? 이걸 안 하면 배포 직후 어떤 지연 패턴이 사용자에게 보입니까?

2. **작은 파드의 웜업 세금** — 이번 편에서 `CICompilerCount`가 CPU 한도에서 파생되는 걸 봤습니다. `limits.cpu`를 0.5로 조인 파드에서 JIT 컴파일러 스레드가 애플리케이션과 쿼터를 나눠 쓰는 비용을 어떻게 계측하겠습니까? 파드를 잘게 쪼개는 결정에 이 비용을 어떻게 반영하겠습니까?

3. **처방 선택** — 하루 수천 번 차게 시작하는 이벤트 함수와, 3일씩 도는 고TPS 게이트웨이가 있습니다. 각각 native-image / CRaC / Leyden 중 무엇을 택하고, 그 근거를 "콜드 스타트 빈도 × 피크 처리량 민감도"로 설명해 보십시오

4. **PGO가 바꾼 셈법** — native-image에 PGO를 붙이면 피크 격차가 크게 줄고, 벤더 벤치에서는 JIT을 넘기도 합니다. 그렇다면 7편의 "롱런에서 JIT이 역전한다"는 명제는 어디까지 유효합니까? 여러분의 워크로드에서 이걸 어떻게 검증하겠습니까?

5. **CRaC 스냅샷의 상태 관리** — 체크포인트에 얼린 DB 커넥션·난수 시드·시크릿을 복원 후 어떻게 안전하게 갱신하겠습니까? 스냅샷에 시크릿이 박제되는 위험을 배포 파이프라인에서 어떻게 차단합니까?

6. **한도를 피크로 잡기** — 이번 편에서 RSS가 올라간 뒤 안 내려오는 걸 봤습니다. `limits.memory`를 평상시 값으로 잡으면 왜 위험하고, 피크로 잡으면 밀도가 얼마나 나빠집니까? 부하 테스트에서 이 피크를 어떻게 재겠습니까?

## 핵심 요약

- **웜업 계단을 직접 쟀습니다.** 같은 메서드가 인터프리터에서 **5578ns**, 정착 후 **830ns**입니다. **6.7배**입니다
- `PrintCompilation`으로 승급을 관찰하면 **티어3 → 티어4 OSR(실행 도중 갈아탐) → 티어4 → 옛 코드 `made not entrant`(역최적화)** 순서가 그대로 보입니다
- **"C2는 5000번 불려야 승급한다"는 요약은 루프 없는 메서드에만 맞습니다.** 승급 조건에 **백엣지 카운터**가 함께 들어가서, 루프를 품은 메서드는 훨씬 일찍 승급합니다. 제 측정에서는 1200~1400회에 정착했습니다
- **컨테이너 CPU 한도가 `CICompilerCount`를 정합니다**(4코어→3, 2코어→2). 호출 횟수 임계값은 안 변하지만 **컴파일 큐가 늦게 빠지고**, 그 컴파일러 스레드가 애플리케이션과 **같은 쿼터를 나눠 씁니다.** 작은 파드일수록 웜업이 비쌉니다
- **시작 구간 메모리에 대한 통념이 빗나갔습니다.** "튀었다가 내려오는 스파이크"는 관찰되지 않았습니다. RSS는 1초 만에 282MiB까지 **올라간 뒤 일을 멈춰도 내려오지 않습니다.** **피크가 곧 상주 비용**이므로, `limits.memory`를 평상시 값으로 잡으면 안 됩니다
- **Leyden의 AOT 캐시는 JDK 25에 이미 들어와 있습니다.** 훈련 실행 한 번으로 기동이 **42ms → 35ms(약 17%)** 로 줄었고 캐시는 11.8MB였습니다. 다만 **기준선도 이미 기본 CDS 아카이브로 931개 클래스를 공유 파일에서 읽고 있었고**, 제 앱은 클래스 천 개짜리 장난감이라 **이 17%를 실제 프레임워크에 그대로 대입하면 안 됩니다**
- **native-image의 피크 열세(10~25%↓)는 PGO로 크게 만회됩니다.** 벤더 벤치에서는 JIT을 넘기도 합니다(13,075 vs 12,488 req/s). 벤더 수치라 그대로 믿을 건 아니지만, **"AOT는 무조건 피크가 낮다"는 도식은 더 이상 안전하지 않습니다**
- **CRaC·SnapStart는 웜업된 프로세스를 얼립니다.** AWS 공식 표현으로 최대 10배 빠른 시작. 대가는 **외부 상태와 시크릿이 스냅샷에 박제**되는 것이고, 훅으로 반드시 재생성해야 합니다
- **웜업은 런타임 재최적화 능력의 대가입니다.** 세 처방은 그 능력의 어느 부분을 포기할지 정하는 다이얼일 뿐, 공짜로 없애지 못합니다
