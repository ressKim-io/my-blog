> **시리즈 "커널과 런타임으로 톺아보는 Rust · Go · Java"의 22편 — 5부를 닫습니다**
> [17편](/essays/gc-cost-conservation-no-silver-bullet)에서 4부를 닫으며 "GC의 청구서가 클러스터로 새어 나가는 지점"을 예고했고, 5부의 네 편이 그 청구서를 하나씩 받아 왔습니다
> 이제 조각을 한자리에 모읍니다
> 한 문장으로 앞당겨 말하면 이렇습니다. **비용은 사라지지 않습니다. 누가 회계를 맡느냐에 따라, 그게 클라우드에서 어느 청구서로 날아올지가 달라질 뿐입니다**

## 네 편이 각각 무엇을 세웠나

**[18편](/essays/container-memory-accounting-cgroup)은 회계 규칙**을 세웠습니다

커널은 힙을 모르고 페이지를 셉니다. `anon`뿐 아니라 페이지 캐시·슬랩·소켓 버퍼까지 전부 같은 장부에 올립니다. 그리고 한도를 넘으면 늦추는 게 아니라 **죽입니다**

**[19편](/essays/gc-headroom-bin-packing-density)은 밀도**를 세웠습니다

GC가 남겨 둔 여유가 파드 상자의 크기가 되고, 상자 크기가 노드 밀도를, 밀도가 청구서를 정합니다

**[20편](/essays/offheap-oomkilled-nmt)은 안전마진**을 세웠습니다

런타임은 힙만 관리하고 로그에 남기는데 커널은 전부 합산합니다. 그 사각지대가 exit 137입니다

**[21편](/essays/cold-start-warmup-aot-remedies)은 시간 축**을 세웠습니다

Java는 차게 태어나 데워집니다. 그 데워지는 시간이 오토스케일링과 서버리스의 경제성을 때립니다

![The four axes all hang from one choice](/diagrams/cloud-economics-decision-matrix-1.svg)

그림 맨 위의 꼭짓점이 [17편](/essays/gc-cost-conservation-no-silver-bullet)에서 세운 **회계의 위치**입니다. 메모리 안전을 사람이 책임지는가, 컴파일러가 책임지는가, 런타임이 책임지는가

그 아래로 네 갈래가 뻗습니다. **회계·밀도·안전마진·시작 구간**이죠

**하나의 선택이 네 개의 청구서를 동시에 결정합니다**

네 축을 관통하는 문장은 이렇습니다

**런타임에 회계를 위임할수록 개발은 편해집니다. 대신 런타임이 세지 않는 곳이 사각지대가 되고, 런타임이 요구하는 여유가 곧 밀도와 비용이 됩니다**

Rust는 회계를 사람과 컴파일러에 두어 사각지대가 좁은 대신, **개발 비용을 앞당겨** 냈습니다

Java는 런타임에 크게 위임해 편한 대신, **힙 밖과 웜업**이라는 청구서를 받습니다

Go는 그 중간에서 손잡이 몇 개(`GOMEMLIMIT`·`GOGC`)로 위치를 조절합니다

## 5부가 직접 잰 숫자들

이 시리즈의 4부가 그랬듯, 5부도 수치를 인용하지 않고 **컨테이너 안에서 직접 쟀습니다**. 커널 6.8·cgroup v2·JDK 25.0.3·Go 1.26.5

한자리에 모읍니다

| 무엇을 | 잰 값 | 어디서 |
|---|---|---|
| `limits.memory: 512Mi`가 되는 것 | `memory.max = 536870912` | 18편 |
| 유휴 컨테이너의 커널 몫 대 익명 메모리 | `kernel` 1.08MiB vs `anon` 120KiB (**9배**) | 18편 |
| 512MiB 컨테이너의 JVM 힙 | **128MiB** (에르고노믹) | 18편 |
| **250~512MiB 컨테이너의 힙** | **전부 128MiB 고정** (바닥 규칙) | 18편 |
| 300MiB 파드의 힙 비율 | 25%가 아니라 **42.6%** | 18편 |
| Go 라이브 300MiB의 다음 GC 목표 | **562MiB** (한도 512MiB **초과**) | 18편 |
| `GOMEMLIMIT` 없이 → 있게 | exit 137 → 생존. **GC 7회 → 435회, CPU 0% → 6%** | 18편 |
| 유휴 프로세스가 커널에 청구하는 값 | Rust **2.3** · Go **4.8** · Java G1 **41.2** · ZGC **77.9** MiB | 19편 |
| 힙을 8배 키웠을 때 힙 밖 고정비 | 52 → 54MiB (**거의 안 변함**) | 19편 |
| ZGC를 세 가지로 재면 | reserved **9.35GiB** · committed **42.4MiB** · 실제 청구 **77.9MiB** | 19편 |
| ZGC 힙이 잡히는 기둥 | `anon`이 아니라 **`shmem`** (`memfd:java_heap`) | 19편 |
| 캡이 한도 **아래**일 때 죽는 법 | `OutOfMemoryError` · 스택 트레이스 · **exit 1** | 20편 |
| 캡이 한도 **위**일 때 죽는 법 | GC 로그 **0건** · 힙 사용률 **1%** · **exit 137** | 20편 |
| `MALLOC_ARENA_MAX=2`의 효과 | 기본 3MiB vs **96~113MiB 잔류** (통념과 반대) | 20편 |
| `GOMEMLIMIT=700MiB`인데 cgo가 960MiB | Go 힙 **0 MiB** · GC **0회** · **exit 137** | 20편 |
| 웜업 계단 | 인터프리터 **5578ns** → 정착 **830ns** (**6.7배**) | 21편 |
| 컨테이너 CPU가 정하는 것 | `CICompilerCount` 4코어→3 · 2코어→2 | 21편 |
| 시작 후 RSS | 282MiB까지 올라가서 **안 내려옴** | 21편 |
| Leyden AOT 캐시 | 기동 **42ms → 35ms** (약 17%), 캐시 11.8MB | 21편 |

이 표에서 제가 가장 값지게 여기는 건 **틀렸던 것들**입니다

원본 정리는 "컨테이너가 256MiB를 넘으면 힙이 25%"라고 했는데, 촘촘히 재 보니 **250~512MiB 구간이 통째로 128MiB에 고정**돼 있었습니다. 원본의 실측 표가 그 구간을 샘플링에서 건너뛴 탓이었죠

원본은 "ZGC가 힙의 3%를 고정 오버헤드로 쓴다"고 했는데 **출처를 찾지 못했습니다.** 대신 세 지표로 재 보니 순위가 전부 달랐고, ZGC의 힙이 `shmem`에 잡힌다는 걸 발견했습니다

원본은 "`MALLOC_ARENA_MAX=2`가 종종 가장 효과적인 처방"이라 했는데, 제 워크로드에서는 **그걸 켠 쪽이 오히려 100MiB를 안 돌려줬습니다**

원본은 "시작 시 RSS가 스파이크로 튄다"고 했는데, **스파이크는 없었습니다.** 올라가서 안 내려올 뿐이었죠

**재 보지 않았으면 전부 그대로 옮겨 적었을 것들입니다**

## 비용 보존 법칙의 클라우드 확장

[12편](/essays/allocation-convergence-why-gc)과 [17편](/essays/gc-cost-conservation-no-silver-bullet)에서 세운 비용 보존 표를 클라우드 축까지 늘립니다

각 칸은 **"그 언어와 GC가 이 비용을 어디로 옮겨 냈는가"** 입니다

| 축 | Rust | Go | Java G1 | Java ZGC | Java native-image |
|---|---|---|---|---|---|
| **회계 위치** | 사람·컴파일러 | 런타임(비이동) | 런타임(이동) | 런타임(동시 이동) | 컴파일러(빌드타임) |
| **유휴 고정비** | **2.3 MiB** | **4.8 MiB** | 41.2 MiB | **77.9 MiB** | 작음 |
| **메모리 헤드룸** | 없음 | 라이브 ~2배 | 힙 예비 10% + RSet | 압축 참조 없음 + 동시 여유 | 작음 |
| **힙 밖 위험** | 낮음 | 중간 (**cgo**) | 높음 | 높음 | 낮음 |
| **콜드 스타트** | 없음 | 없음 | **6.7배 느리게 시작** | 있음 | 없음 |
| **피크 처리량** | 최상 | 좋음 | **최상(롱런)** | 높음(배리어 세금) | 10~25%↓ (PGO로 만회) |
| **지연 꼬리** | 예측 가능 | Mark Assist로 튐 | 가끔 Full GC | **sub-ms** | 낮음 |
| **컨테이너 자동 인식** | 앱·라이브러리 | **CPU만** | CPU·메모리 | CPU·메모리 | 해당 없음 |
| **주된 청구서** | 개발 시간 | GC CPU 또는 헤드룸 | 힙 밖 여백·웜업 | 메모리·처리량 | 피크·동적성 |

**표를 가로로 읽으면 어느 행에도 승자가 없고, 세로로 읽으면 어느 열에도 공짜 칸이 없습니다**

![Cost is conserved — every approach only moves it](/diagrams/cloud-economics-decision-matrix-2.svg)

그림의 왼쪽이 각 접근이 **사는 것**, 오른쪽이 **치르는 것**입니다

화살표 넷이 **서로 다른 청구서에 도착합니다.** 어느 화살표도 허공으로 사라지지 않죠

Rust는 밀도·시작·안전마진을 다 가져갑니다. 대신 **개발 시간을 앞당겨** 냈습니다

ZGC는 지연을 극한으로 낮춥니다. 대신 **밀도와 처리량**을 냈습니다

native-image는 시작을 삽니다. 대신 **피크와 동적성**을 냈습니다

Go는 손잡이가 있습니다. 그런데 18편에서 봤듯 **그 손잡이를 안 돌리면 죽습니다**

**비용은 이동할 뿐 사라지지 않습니다**

## 그래서 무엇을 고르나

"무엇이 제일 좋냐"는 질문은 틀렸습니다. 옳은 질문은 **"이 클러스터가 무엇을 최적화하는가"** 입니다

![Pick the corner your workload actually needs](/diagrams/cloud-economics-decision-matrix-3.svg)

그림의 축이 5부가 세운 네 개의 지배적 비용입니다. **콜드 스타트·피크 처리량·지연·밀도**

각 워크로드는 이 중 **하나가 지배적**입니다. 그 하나를 정하면 선택이 좁혀집니다

| 워크로드 | 지배적 비용 | 우선 고려 | 근거 |
|---|---|---|---|
| scale-to-zero 서버리스·이벤트 함수 | 콜드 스타트 | Rust · Go · Java(native-image / SnapStart) | 21편 — 자주 차게 시작. 기본 JVM은 불리 |
| 며칠씩 도는 고TPS API | 피크 처리량 | Java(JIT) · Go · Rust | 21편·[7편](/essays/java-jit-inversion-conditions) — 롱런에서 JIT 피크가 유리 |
| sub-ms 지연 SLO (결제·광고입찰) | 지연 꼬리 | Java ZGC · Rust | [16편](/essays/generational-zgc-colored-pointer-stall) — 지연을 위해 밀도·처리량 지불 |
| 메모리 밀도·비용 극한 (멀티테넌시) | 노드 비용 | Rust · Go(`GOMEMLIMIT`) | 19편 — 유휴 고정비가 파드마다 곱해짐 |
| 사이드카·에이전트 (수천 개가 뜸) | 파드당 고정비 | **Rust · Go** | 19편 — Java는 파드당 41MiB, Go는 4.8MiB |
| 빠른 개발·넓은 생태계 | 인력·시간 | Java · Go | 회계를 런타임에 위임 = 개발 편의 |
| 운영 단순성 | 오설정 사고 | Java(양쪽 자동) | 18편 — `GOMEMLIMIT` 수동이 가장 흔한 함정 |

여기에 4부와 5부를 관통하는 리듬이 있습니다

**콜드 스타트가 잦으면 AOT로 기울고, 오래 돌수록 JIT으로 기웁니다**

**지연이 절박하면 밀도와 처리량을 내주고, 밀도가 절박하면 지연과 개발 시간을 내줍니다**

어느 쪽도 전부를 가질 수 없습니다

## "은탄환은 없다"의 클라우드 판

[17편](/essays/gc-cost-conservation-no-silver-bullet)이 "GC는 은탄환이 아니다"를 GC 삼각형으로 증명했다면, 5부는 그 명제가 **클라우드에서도 참**임을 보였습니다

관리형 런타임(GC·JIT)은 개발 편의를 줍니다. 그 편의는 **힙 밖 사각지대·GC 헤드룸·웜업 세금**이라는 형태로 컨테이너 청구서에 되돌아옵니다

반대로 그 청구서를 지우려는 처방들도 대가를 옮길 뿐입니다. 수동 튜닝은 **오설정 사고**로, native-image는 **피크와 동적성**으로, Rust는 **개발 비용**으로

그래서 실무의 성숙도는 **"어떤 언어가 최고냐"** 를 아는 데 있지 않습니다

**"우리 워크로드의 지배적 비용이 무엇이고, 그 비용을 감당 가능한 다른 비용으로 옮겼는가"** 를 아는 데 있습니다

5부는 그 판단에 필요한 계기를 손에 쥐여 주는 것으로 제 몫을 다합니다. cgroup의 `memory.stat`과 `memory.events`, JVM의 NMT, Go의 `gctrace`, JIT의 티어드 임계값, 그리고 `requests`와 `limits`

## 대시보드를 런타임별로 다르게 보십시오

5부의 실측에서 자연히 따라 나오는 귀결 하나를 덧붙입니다

![Watch a different gauge for each runtime](/diagrams/cloud-economics-decision-matrix-4.svg)

그림의 네 칸이 런타임별로 **실제로 봐야 할 계기**입니다. 같은 대시보드가 런타임마다 **다른 사고를 가립니다**

**Go 파드**는 `GOMEMLIMIT` 설정 여부와 **GC CPU 비중**을 같은 화면에서 봐야 합니다. 18편에서 봤듯 그 손잡이를 조이면 메모리는 잡히지만 **CPU 6%가 GC로 넘어갑니다.** 한쪽만 보면 맞바꿈이 안 보입니다

**G1 파드**는 힙 사용률이 아니라 **`memory.current`와 NMT의 격차**를 봐야 합니다. 20편의 그 죽음은 힙 계기판에 전혀 안 나타납니다

**ZGC 파드**는 `memory.stat`의 **`anon`이 아니라 `shmem`** 을 봐야 합니다. 19편에서 확인했듯 ZGC의 힙은 거기 있습니다. `anon`만 보면 힙이 사라진 것처럼 보입니다

**웜업이 있는 파드**는 readiness를 "떴다"가 아니라 **"데워졌다"** 에 맞춰야 합니다. 21편의 6.7배가 그대로 사용자에게 갑니다

## 더 파고들 질문

1. **지배적 비용 식별** — 담당 서비스의 지배적 비용(콜드 스타트·피크·지연·밀도·인력) 하나를 고른다면 무엇이며, 그걸 어떤 지표로 증명하겠습니까? 그 선택이 언어·GC·튜닝을 어떻게 좁힙니까?

2. **비용 이동의 가시화** — GC 헤드룸을 줄여 밀도를 높이면 GC CPU가 오릅니다. 이 맞교환을 한 대시보드에서 동시에 보이게 하려면 무엇을 그리겠습니까?

3. **혼합 클러스터의 원칙** — 한 클러스터에 서버리스형·롱런형·지연민감형 워크로드가 섞여 있습니다. 노드 풀·스케줄러 프로파일·런타임 선택을 어떤 원칙으로 나누겠습니까?

4. **경제성 회귀 감시** — 배포로 상자 크기·웜업 시간·힙 밖 사용이 조용히 늘어나는 걸 CI에서 잡으려면 어떤 게이트를 두겠습니까? NMT diff, 콜드 스타트 벤치, 밀도 지표 중 무엇을 먼저 넣겠습니까?

5. **재검토 주기** — 런타임 기본값은 버전마다 바뀝니다. 5부만 해도 Go 1.25의 `GOMAXPROCS` 자동화, JDK 25의 Leyden, native-image의 PGO가 셈법을 바꿨습니다. 언어·GC 선택을 언제 무엇을 근거로 재검토하겠습니까? 벤더 벤치를 어디까지 믿겠습니까?

6. **조직 표준의 강제** — Java는 컨테이너 인식이 자동이고 Go는 메모리가 수동입니다. `GOMEMLIMIT`·`MaxRAMPercentage`·캡의 총합을 어떻게 강제하겠습니까? 어드미션 웹훅·베이스 이미지·헬름 차트 중 무엇이 실효적입니까?

## 핵심 요약

- **5부의 네 축은 하나의 선택에서 갈라집니다.** 회계를 사람·컴파일러·런타임 중 누구에게 맡기느냐가 **회계·밀도·안전마진·시작 구간**의 네 청구서를 동시에 결정합니다
- **런타임에 위임할수록 개발은 편하지만**, 런타임이 세지 않는 곳(힙 밖·cgo)이 사각지대가 되고 런타임이 요구하는 여유가 곧 밀도와 비용이 됩니다
- **5부는 수치를 인용하지 않고 컨테이너 안에서 직접 쟀습니다.** 그리고 그 과정에서 원본 정리의 **네 가지 통념이 실측에서 무너졌습니다** — 힙 산정의 256MiB 경계, ZGC의 고정 3%, `MALLOC_ARENA_MAX=2`의 만능성, 시작 구간의 메모리 스파이크
- **비용 보존 표에 공짜 칸이 없습니다.** Rust는 밀도·시작·안전을 다 가져가되 개발 시간을 앞당겨 냈고, ZGC는 지연을 사려고 밀도와 처리량을 냈고, native-image는 시작을 사려고 피크와 동적성을 냈습니다
- **옳은 질문은 "무엇이 최고냐"가 아니라 "이 클러스터가 무엇을 최적화하는가"** 입니다. 지배적 비용 하나를 정하면 선택이 좁혀집니다
- **대시보드도 런타임별로 달라야 합니다.** Go는 GC CPU와 함께, G1은 `memory.current`와 NMT의 격차로, ZGC는 `anon`이 아니라 `shmem`으로, 웜업이 있는 파드는 readiness를 "데워졌다"에 맞춰서
