> [17편](/essays/gc-cost-conservation-no-silver-bullet)에서 4부를 마무리하며 이 비용이 클러스터 단위로 이어지는 지점을 예고했고, 5부의 네 편이 이를 하나씩 확인해 왔습니다
> 이번 편에서는 그 결과를 한자리에 모아 종합합니다
> 핵심 결론은 다음과 같습니다. **비용은 소멸하지 않습니다. 메모리 관리 주체(회계의 위치)를 어디에 두느냐에 따라, 비용이 청구되는 형태가 전환될 뿐입니다**

## 5부 각 편의 핵심 결론 종합

**[18편](/essays/container-memory-accounting-cgroup)은 메모리 회계 규칙을 정립했습니다.**
커널은 런타임의 힙 구조를 인지하지 않고 물리 페이지 단위로 자원을 계량합니다. `anon` 영역뿐 아니라 페이지 캐시·커널 슬랩·소켓 버퍼까지 모두 단일 한도에 합산하며, 한도 초과 시 스로틀링이 아닌 즉각적인 강제 종료(`SIGKILL`)를 수행합니다

**[19편](/essays/gc-headroom-bin-packing-density)은 인프라 밀도의 기준을 확인했습니다.**
GC 동작을 위해 확보한 메모리 여유(headroom)가 스케줄링 단계에서 파드의 요구 크기(`requests`)를 결정하고, 이는 노드의 집적 밀도 저하와 인프라 비용 상승으로 이어집니다

**[20편](/essays/offheap-oomkilled-nmt)은 안전 마진의 사각지대를 분석했습니다.**
런타임은 힙 영역 위주로 메모리를 관리하고 로그에 기록하는 반면, 커널은 힙 외부 영역까지 모두 합산하여 통제합니다. 이 가시성 불일치 구간에서 GC 로그에 징후가 없는 `exit 137` 장애가 발생합니다

**[21편](/essays/cold-start-warmup-aot-remedies)은 시간 축에 따른 웜업 지연을 측정했습니다.**
Java 프로세스는 기동 후 JIT 컴파일러의 동적 최적화가 완료되기 전까지 성능 저하를 겪습니다. 이 웜업 비용은 오토스케일링과 서버리스 환경에서 초기 리소스 투입 대비 실효 처리량 감소라는 경제적 손실로 이어집니다

![The four axes all hang from one choice](/diagrams/cloud-economics-decision-matrix-1.svg)

위 다이어그램 최상단의 꼭짓점은 [17편](/essays/gc-cost-conservation-no-silver-bullet)에서 규명한 **메모리 회계의 위치**를 나타냅니다. 메모리 안전에 대한 통제 권한을 개발자, 컴파일러, 런타임 중 어디에 둘 것인가에 대한 아키텍처 결정입니다

그 아래로 네 갈래가 뻗습니다. **회계·밀도·안전 마진·시작 구간**입니다

**하나의 선택이 네 가지 비용을 동시에 결정합니다**

네 축을 관통하는 원리는 다음과 같습니다

**언어 런타임에 메모리 회계를 위임할수록 개발 생산성은 향상되나, 런타임이 계측하지 못하는 사각지대가 발생하며 GC 여유 공간 요구량이 늘어 인프라 비용(밀도 하락)으로 전가됩니다**

- Rust는 메모리 관리 책임을 개발자와 컴파일러에 두어 사각지대와 런타임 비용을 최소화한 대신, **개발 초기 학습 곡선과 컴파일 시간**을 지불합니다
- Java는 런타임에 메모리 제어를 전면 위임해 개발 편의를 얻은 대신, **힙 외부 메모리 누수 위험과 웜업 오버헤드**를 감당합니다
- Go는 두 방식의 중간 지점에서 `GOMEMLIMIT`·`GOGC` 같은 제어 변수를 통해 성능과 안전성의 균형을 수동으로 조율합니다

## 5부 실측 데이터 종합

본 시리즈의 데이터는 외부 벤치마크 인용 없이 컨테이너 환경(커널 6.8, cgroup v2, JDK 25.0.3, Go 1.26.5)에서 직접 측정한 결과입니다

| 측정 지표 | 실측 결과값 | 분석 편 |
|---|---|---|
| `limits.memory: 512Mi`의 cgroup 반영 수치 | `memory.max = 536870912` | 18편 |
| 유휴 컨테이너의 커널 점유율 vs 익명 메모리 | `kernel` 1.08MiB vs `anon` 120KiB (커널 점유 9배 상회) | 18편 |
| 512MiB 한도 컨테이너의 JVM 힙 동적 할당량 | **128MiB** (Ergonomics 기반) | 18편 |
| **250~512MiB 컨테이너의 힙 크기 변화** | **전체 구간 128MiB로 고정** (JVM 최소 보정 Floor 규칙 적용) | 18편 |
| 300MiB 파드 한도 대비 할당 힙 비중 | 이론상 25%가 아닌 **실측 42.6%** 점유 | 18편 |
| Go 라이브 집합 300MiB 상태의 다음 GC 목표치 | **562MiB** (컨테이너 물리 한도 512MiB **초과** 상태) | 18편 |
| `GOMEMLIMIT` 미설정 대비 명시적 설정 변화 | OOM `exit 137` 발생 → 프로세스 생존 (단, **GC 연산 7회 → 435회 증가, CPU 점유 0% → 6% 상승**) | 18편 |
| 유휴 프로세스 단위 커널 청구(`memory.current`) | Rust **2.3** / Go **4.8** / Java G1 **41.2** / Java ZGC **77.9** (단위: MiB) | 19편 |
| 힙 최대 상한을 8배 확장 시 힙 외부 고정 비용 | 52 MiB → 54 MiB (**힙 크기에 비례하지 않고 고정됨**) | 19편 |
| ZGC 유휴 상태 3분할 지표 비교 | 가상 예약 **9.35GiB** / JVM 커밋 **42.4MiB** / 커널 실청구 **77.9MiB** | 19편 |
| ZGC 힙 메모리 매핑 항목 | `anon` 영역이 아닌 **`shmem` (`memfd:java_heap`)** 에 병합 | 19편 |
| 힙 외부 상한 옵션이 물리 한도 **미만**일 시 | `OutOfMemoryError` 발생, 스택 트레이스 기록, **정상 예외 코드 exit 1** | 20편 |
| 힙 외부 상한 옵션이 물리 한도 **초과**일 시 | GC 동작 **0건**, 힙 사용률 **1%** 상태에서 강제 종료 **exit 137** | 20편 |
| `MALLOC_ARENA_MAX=2` 적용 전후 메모리 반환율 | 기본 상태 3MiB 잔류 vs **적용 후 96~113MiB 잔류** (기존 통념과 상반됨) | 20편 |
| Go `GOMEMLIMIT=700MiB` 상태에서 cgo 모듈 960MiB 점유 시 | Go 힙 **0 MiB**, GC 연산 **0회** 상태에서 강제 종료 **exit 137** | 20편 |
| Java 웜업 구간 실행 성능 개선폭 | 초기 인터프리터 **5578ns** → C2 정착 이후 **830ns** (**6.7배 향상**) | 21편 |
| 컨테이너 CPU 제한이 유발하는 JIT 스레드 변동 | `CICompilerCount` 변동 (4코어 → 3개 스레드, 2코어 → 2개 스레드) | 21편 |
| 초기 구동 이후 RSS 궤적 | 282MiB까지 상승 후 **고원(Plateau) 형태로 유지, 하락하지 않음** | 21편 |
| JDK 25 Leyden AOT 캐시 적용 실효성 | 초기 기동 시간 **42ms → 35ms** (약 17% 단축), 캐시 용량 11.8MB 산출 | 21편 |

이 표에서 가장 값진 항목은 **기존 통념과 다르게 나타난 부분들**입니다

- 소형 파드에서도 JVM 힙 비율이 25%로 유지된다는 통념과 달리, 250~512MiB 구간에서는 힙 최소 기준선(floor) 규칙에 의해 점유율이 40~50% 수준으로 나타났습니다. 이는 기존에 참고한 자료의 실측 구간이 이 범위를 건너뛰었기 때문으로 추정됩니다
- ZGC가 힙의 약 3%를 고정 오버헤드로 사용한다는 수치는 출처를 확인할 수 없어 본문에 반영하지 않았습니다. 대신 reserved·committed·`memory.current` 세 지표를 비교한 결과 순위가 모두 달랐고, ZGC의 힙이 `shmem`에 매핑된다는 사실을 확인했습니다
- `MALLOC_ARENA_MAX=2` 설정이 흔히 효과적인 처방으로 알려져 있으나, 테스트한 워크로드에서는 오히려 이 설정을 적용한 쪽이 약 100MiB의 메모리를 반환하지 못했습니다
- 구동 초기 RSS가 스파이크 형태로 나타난다는 통념과 달리, 스파이크 없이 상승한 뒤 하락하지 않는 궤적을 보였습니다. 한 번 상승한 RSS는 프로세스의 영구적인 상주 비용으로 굳어집니다

실측을 거치지 않았다면 이러한 차이를 그대로 놓쳤을 항목들입니다

## 비용 보존 법칙의 클라우드 인프라 확장

[12편](/essays/allocation-convergence-why-gc)과 [17편](/essays/gc-cost-conservation-no-silver-bullet)에서 정립한 비용 보존 법칙을 클라우드 아키텍처로 확장합니다

각 항목은 해당 언어와 GC 방식이 **어떤 형태로 비용을 이전했는지**를 나타냅니다

| 평가 축 | Rust | Go | Java G1 | Java ZGC | Java Native-Image |
|---|---|---|---|---|---|
| **메모리 회계 위치** | 개발자 및 컴파일러 | 부분적 런타임 | 동적 런타임 (이동형) | 동적 런타임 (동시 이동형) | 컴파일러 (빌드 타임 AOT) |
| **유휴 상태 고정 비용** | **2.3 MiB** | **4.8 MiB** | 41.2 MiB | **77.9 MiB** | 미미함 |
| **GC 메모리 여유 공간** | 발생하지 않음 | 라이브 객체의 약 2배수 | 힙 10% 예약 + RSet 통제 | 객체 포인터(8B 고정) + 동시 수집용 여백 | 미미함 |
| **힙 외부 누수 사각지대** | 낮음 | 중간 (**cgo 연동 시**) | 높음 | 높음 | 낮음 |
| **콜드 스타트 성능 지연** | 없음 | 없음 | **약 6.7배의 초기 지연** | 존재함 | 없음 |
| **최대 스루풋(Peak)** | 최상 | 우수 | **최상 (장기 구동 시)** | 우수 (Barrier 연산 페널티) | 10~25% 감소 (PGO 적용 시 만회) |
| **지연 시간(Latency) 꼬리** | 예측 가능 수준 통제 | Mark Assist 시 간헐적 튐 | 간헐적 Full GC 위험 노출 | **1ms 이하 초저지연 보장** | 안정적 유지 |
| **컨테이너 한도 동적 인식** | API 지원(앱 통제) | **CPU에 한정 지원** | CPU 및 메모리 통합 지원 | CPU 및 메모리 통합 지원 | 해당 없음 |
| **최종 부과되는 인프라 비용** | 높은 개발 리소스 | 높은 GC 연산 및 여백 | 힙 외부 메모리 점유 및 웜업 지연 | 인프라 밀도 저하 및 스루풋 감소 | 스루풋 하락 및 런타임 동적성 제거 |

**표를 가로로 보면 어느 행에도 절대적 우위를 지닌 항목이 없고, 세로로 보면 어느 열에도 대가 없이 얻어지는 항목이 없습니다**

![Cost is conserved — every approach only moves it](/diagrams/cloud-economics-decision-matrix-2.svg)

위 다이어그램은 각 접근법이 **확보하는 지표(좌측)** 와 그 대가로 **지불하는 항목(우측)** 의 관계를 나타냅니다. 비용은 소멸하지 않고 다른 영역으로 이동합니다

- Rust는 밀도·시작 성능·안전성을 모두 확보하지만, 그 대가로 **개발 기간과 컴파일 복잡도**를 지불합니다
- Java ZGC는 응답 지연을 최소화하지만, 그 대가로 **집적 밀도와 처리량**을 지불합니다
- Native Image는 초기 구동 지연을 제거하지만, **최대 처리 성능과 리플렉션 등의 동적 유연성**을 지불합니다
- Go는 `GOMEMLIMIT` 같은 튜닝 수단을 제공해 트레이드오프 지점을 선택할 수 있게 하지만, 명시적으로 설정하지 않으면 커널 OOM으로 이어질 수 있습니다

**비용은 이동할 뿐 소멸하지 않습니다**

## 워크로드별 최적 런타임 아키텍처 선택 가이드

특정 기술 스택의 절대적 우위를 논하는 것은 적절하지 않습니다. 올바른 접근은 **"해당 클러스터가 최우선으로 확보해야 하는 핵심 지표가 무엇인가"** 를 정의하는 것입니다

![Pick the corner your workload actually needs](/diagrams/cloud-economics-decision-matrix-3.svg)

그림의 네 축이 5부에서 확인한 지배적 비용입니다. **콜드 스타트·최대 처리량·지연·밀도**입니다

각 워크로드는 이 중 **하나가 지배적**입니다. 이를 정하면 아키텍처 선택의 범위가 논리적으로 좁혀집니다

| 워크로드 특성 | 최적화 대상 지배적 비용 | 권장 런타임 및 기술 스택 | 근거 |
|---|---|---|---|
| Scale-to-zero 서버리스 및 이벤트 기반 함수 | 콜드 스타트 지연 방어 | Rust, Go, Java(Native-image / SnapStart) | [21편](/essays/cold-start-warmup-aot-remedies) — 반복적인 콜드 스타트에는 기본 JVM이 불리함 |
| 수일간 연속 구동되는 고부하 트래픽 API | 최대 처리량 확보 | Java(JIT), Go, Rust | [21편](/essays/cold-start-warmup-aot-remedies)·[7편](/essays/java-jit-inversion-conditions) — 장기 실행 시 JIT 피크가 유리함 |
| 1ms 이하 지연 통제(SLO) (금융 결제, 실시간 입찰) | 응답 지연 최소화 | Java ZGC, Rust | [16편](/essays/generational-zgc-colored-pointer-stall) — 지연 확보를 위해 밀도·처리량을 지불함 |
| 멀티테넌시 기반 인프라 비용 절감 | 노드 집적 밀도 향상 | Rust, Go (`GOMEMLIMIT` 조율 필수) | [19편](/essays/gc-headroom-bin-packing-density) — 유휴 고정비가 파드 수만큼 누적됨 |
| 사이드카 및 에이전트 인프라 | 파드당 고정비 최소화 | **Rust, Go** | [19편](/essays/gc-headroom-bin-packing-density) — Java G1은 파드당 41MiB, Go는 4.8MiB를 점유함 |
| 비즈니스 로직 변경이 빈번하고 생태계 활용이 중요한 경우 | 개발 리드타임 단축 | Java, Go | 메모리 회계를 런타임에 위임하여 비즈니스 로직에 개발 역량을 집중함 |
| 운영 복잡도 최소화 | 오설정에 의한 장애 방지 | Java (환경 자동 인식) | [18편](/essays/container-memory-accounting-cgroup) — `GOMEMLIMIT` 수동 설정 누락이 가장 흔한 함정임 |

여기에 4부와 5부를 관통하는 흐름이 있습니다

**콜드 스타트가 잦을수록 AOT 방식이 유리하고, 장기간 연속 실행될수록 JIT 방식이 유리합니다**

**지연이 최우선이면 밀도와 처리량을 양보하게 되고, 밀도가 최우선이면 지연과 개발 시간을 양보하게 됩니다**

어느 한쪽도 모든 것을 동시에 확보할 수는 없습니다

## 결론

소프트웨어 공학의 "은탄환은 없다"는 명제는 클라우드 네이티브 환경에서도 동일하게 적용됩니다. [17편](/essays/gc-cost-conservation-no-silver-bullet)이 이를 GC 삼각형으로 증명했다면, 5부는 같은 명제가 **클라우드 비용 구조에서도 성립함**을 보였습니다

GC와 JIT으로 대표되는 관리형 런타임은 개발 편의성을 제공하지만, 그 대가로 힙 외부 메모리 통제 불가, 과도한 GC 여유 공간 요구, 웜업 지연에 따른 리소스 낭비라는 인프라 비용을 청구합니다. 수동 메모리 제어(운영 오류 위험), AOT 컴파일(처리량·동적성 저하), Rust 도입(개발 비용 증가) 등 이를 해소하려는 대안 역시 비용을 다른 형태로 이전할 뿐입니다

따라서 아키텍처 의사결정의 핵심은 특정 언어를 맹목적으로 채택하는 데 있지 않습니다. "운영 중인 워크로드의 지배적 비용이 무엇이며, 그 비용을 감당 가능한 다른 비용으로 옮겼는가"를 지속적으로 검증하는 데 있습니다

5부는 이 판단에 필요한 계기를 제공하는 것으로 역할을 다합니다. cgroup의 `memory.stat`과 `memory.events`, JVM의 NMT, Go의 `gctrace`, JIT의 티어드 임계값, 그리고 `requests`와 `limits`가 그 계기입니다

## 런타임 환경에 특화된 개별 모니터링 대시보드 구축

모니터링 시스템은 런타임 생태계에 따라 관측 지표의 가중치를 다르게 설계해야 합니다. 단일 대시보드 템플릿은 언어별 고유한 병목을 가릴 위험이 있습니다

![Watch a different gauge for each runtime](/diagrams/cloud-economics-decision-matrix-4.svg)

그림의 네 항목이 런타임별로 실제로 확인해야 할 지표입니다. 동일한 대시보드가 런타임에 따라 서로 다른 장애를 가릴 수 있습니다

**Go 파드:** `GOMEMLIMIT` 설정 여부와 **GC가 유발하는 CPU 점유율**을 함께 확인해야 합니다. 메모리 한도를 조일 경우 상주 메모리는 억제되나 CPU의 최대 6%가 GC 연산으로 전가되기 때문입니다

**G1 파드:** 힙 사용률뿐 아니라 커널의 실제 청구량인 **`memory.current`와 NMT 커밋 지표 간의 격차**를 추적해야 합니다. 힙 외부 OOM은 힙 지표에 나타나지 않습니다

**ZGC 파드:** `memory.stat`의 **`anon`이 아닌 `shmem` 지표**를 확인해야 합니다. ZGC의 힙은 `anon`이 아닌 `shmem`에 매핑되므로, `anon`만 보면 힙 사용량이 누락된 것처럼 보입니다

**웜업이 있는 파드:** `readinessProbe` 통과 기준을 단순 기동이 아닌 **웜업 완료 시점**에 맞춰야 합니다. 그렇지 않으면 초기 지연이 그대로 사용자 경험으로 이어집니다

## 심화 분석 주제

1. **지배적 최적화 목표 식별** — 담당 서비스의 지배적 비용(콜드 스타트·최대 처리량·지연·밀도·개발 리드타임) 중 하나를 선정한다면 무엇이며, 이를 어떤 지표(SLI)로 증명하겠습니까? 그 선택이 언어·GC·튜닝 방식을 어떻게 좁힙니까?

2. **트레이드오프 지표의 상관관계 시각화** — GC 여유 공간을 줄여 밀도를 높이면 GC로 인한 CPU 점유율이 상승합니다. 이 상관관계를 하나의 대시보드에서 함께 확인하려면 어떤 PromQL 지표를 구성하겠습니까?

3. **혼합 워크로드 인프라 분리 설계** — 단일 클러스터에 서버리스형·장기 실행형·초저지연형 워크로드가 혼재할 경우, 노드 풀 분리와 스케줄러 프로파일을 어떤 기준으로 나누겠습니까?

4. **인프라 경제성 훼손 현상 통제** — 배포가 누적됨에 따라 웜업 시간과 힙 외부 메모리 점유율이 점진적으로 상승하는 현상을 CI/CD 파이프라인 단계에서 감지하려면 어떤 NMT 증감 분석 및 부하 테스트 체계를 도입하겠습니까?

5. **런타임 업데이트에 따른 아키텍처 재검토** — Go 1.25의 `GOMAXPROCS` 자동화, JDK 25의 Leyden, Native Image의 PGO 고도화처럼 런타임 기본값은 버전마다 바뀝니다. 언어·GC 선택을 재검토하는 시점과 근거를 어떻게 정하겠으며, 벤더 벤치마크는 어디까지 신뢰하겠습니까?

6. **조직 표준의 강제** — Java는 컨테이너 인식이 자동인 반면 Go는 메모리 설정이 수동입니다. `GOMEMLIMIT`·`MaxRAMPercentage`·상한 총합을 조직 차원에서 어떻게 강제하겠습니까? 어드미션 웹훅, 베이스 이미지, Helm 차트 중 무엇이 가장 실효적입니까?

## 핵심 요약

- **5부의 네 축은 하나의 선택에서 파생됩니다.** 메모리 회계를 개발자·컴파일러·런타임 중 누구에게 맡기느냐가 **회계·밀도·안전 마진·시작 구간**이라는 네 가지 비용을 동시에 결정합니다
- **런타임에 위임할수록 개발은 편해지지만**, 런타임이 계측하지 못하는 영역(힙 외부·cgo)이 사각지대가 되고 런타임이 요구하는 여유 공간이 곧 밀도와 비용으로 이어집니다
- **5부는 외부 수치를 인용하지 않고 컨테이너 환경에서 직접 측정했습니다.** 그 과정에서 기존에 통용되던 **네 가지 통념이 실측과 다르게 나타났습니다** — 힙 산정의 256MiB 경계, ZGC의 고정 3% 오버헤드, `MALLOC_ARENA_MAX=2`의 만능성, 구동 초기 메모리의 스파이크 형태
- **비용 보존 표에는 대가 없이 얻어지는 항목이 없습니다.** Rust는 밀도·시작 성능·안전성을 확보하는 대신 개발 시간을 지불했고, ZGC는 지연을 확보하는 대신 밀도와 처리량을, Native Image는 빠른 시작을 확보하는 대신 피크 성능과 동적성을 지불했습니다
- **올바른 질문은 "무엇이 최선인가"가 아니라 "이 클러스터가 무엇을 최적화하는가"입니다.** 지배적 비용 하나를 정하면 아키텍처 선택이 좁혀집니다
- **대시보드도 런타임별로 달라야 합니다.** Go는 GC CPU와 함께, G1은 `memory.current`와 NMT의 격차로, ZGC는 `anon`이 아닌 `shmem`으로, 웜업이 있는 파드는 readiness를 웜업 완료 시점에 맞춰 확인해야 합니다

`GOMEMLIMIT`이 수동 설정이라 GC가 동작하기 전에 프로세스가 종료되고([18편](/essays/container-memory-accounting-cgroup)), `GOGC=100`이 라이브 객체의 2배를 요구하며([19편](/essays/gc-headroom-bin-packing-density)), cgo를 사용하는 순간 계측 사각지대가 발생하고([20편](/essays/offheap-oomkilled-nmt)), Mark Assist가 지연 시간의 꼬리를 늘립니다([14편](/essays/go-gc-mark-assist-tail-latency))
