> **시리즈 "커널과 런타임으로 깊이 파헤치는 Rust · Go · Java"의 7편**
> [5편](/essays/rust-aot-zero-cost-codegen)의 Rust와 [6편](/essays/go-aot-fast-build-tradeoff)의 Go는 정반대 방향으로 튜닝된 AOT였는데도 같은 벽에 부딪혔습니다. 빌드하는 순간 찍은 스냅샷은 배포 이후의 세상을 모릅니다
> Java는 정반대 도박을 겁니다. 빌드 시점에는 기계어를 아예 만들지 않고, 실행 중에 프로파일을 보면서 기계어를 계속 다시 만듭니다
> 그 대가로 가장 느리게 시작합니다. 대신 충분히 오래 달리면 정적 컴파일이 원리적으로 놓치는 최적화를 잡아냅니다
> [4편](/essays/runtime-weight-zero-cost-vs-fat)에 심어 둔 복선, "가장 무거운 JVM이 왜 롱런 정점 처리량에서 앞서는가"의 답을 이번 편과 다음 편에 걸쳐 확인합니다

## 빌드가 끝나도 기계어가 없습니다

5편과 6편에서 rustc와 Go 컴파일러는 빌드가 끝나는 순간 기계어를 확정했습니다. `objdump`로 몇 번을 열어 봐도 언제나 같은 바이트가 나왔습니다

`javac`는 이 습관부터 깨고 시작합니다. `javac`가 만들어 내는 것은 기계어가 아닙니다. **bytecode(바이트코드, JVM이라는 가상 머신에서 실행하기 위해 정의된 중간 표현 명령어 형식)** 입니다

바이트코드는 특정 CPU의 명령이 아니라, JVM이라는 가상의 기계를 위해 정의된 중간 명령입니다. `.class` 파일에 담깁니다

그래서 x86이냐 ARM이냐가 이 시점에 **결정되지 않습니다**

진짜 기계어는 실행 중에, HotSpot JVM 안에서 태어납니다

![Machine code is born at run time, not at build time](/diagrams/java-jit-c2-runtime-optimization-1.svg)

그림을 위에서부터 따라가겠습니다

`javac`를 거쳐 나온 `.class`는 architecture-agnostic(아키텍처 중립적인, 특정 CPU 하드웨어 구조에 종속되지 않는) 중간 산물입니다. 이 상태로는 어떤 CPU도 직접 실행하지 못합니다

아래 큰 상자가 실행 중인 JVM입니다. 여기서 세 가지 일이 동시에 벌어집니다

**interpreter(인터프리터, 바이트코드를 기계어로 미리 컴파일하지 않고 한 줄씩 즉시 해석하여 실행하는 엔진)** 가 바이트코드를 한 줄씩 해석해 실행합니다. 느립니다

동시에 인터프리터는 자기가 본 것을 **MDO(MethodData Object, 메서드 실행 중에 수집된 타입 프로파일과 분기 빈도 등 런타임 관찰 기록을 저장하는 메타데이터 객체)** 에 적어 둡니다. 그림 가운데의 보라색 상자입니다

**JIT 컴파일러**가 그 기록을 읽고 자주 실행되는 메서드, 이른바 핫(hot) 메서드를 골라 기계어로 컴파일해 **code cache(코드 캐시, JIT 컴파일러가 바이트코드를 번역하여 생성한 네이티브 기계어 코드를 저장하는 실행 가능 메모리 영역)** 에 올립니다. 이후 그 메서드는 기계어로 실행됩니다

여기서 두 가지만 기억해 두면 이번 편이 훨씬 쉽게 읽힙니다

**code cache(코드 캐시, JIT 컴파일러가 바이트코드를 번역하여 생성한 네이티브 기계어 코드를 저장하는 실행 가능 메모리 영역)** 는 JIT이 만든 기계어가 쌓이는 메모리 영역입니다. `mmap`으로 잡힌 RX(Read-Execute, 읽기와 실행 권한만 부여되고 쓰기 권한이 차단된 메모리 보호 상태) 영역이고, **실행 중에 채워집니다**

그래서 같은 프로그램이라도 실행할 때마다 코드 캐시의 내용이 달라질 수 있습니다. 어떤 입력이 들어왔고 어떤 분기가 뜨거웠는지에 따라서입니다

5편에서 Rust 바이너리를 `objdump`로 열어 보면 언제나 같은 바이트가 나온다고 했습니다. 그 `.text` 불변성과 정확히 대척점에 서는 성질입니다

이 가변성이 예측 불가능성(웜업·스파이크)의 원인이면서, 동시에 적응력(peak throughput)의 원천이기도 합니다. 장점과 단점이 같은 곳에서 나옵니다

**MDO(MethodData Object, 메서드 실행 중에 수집된 타입 프로파일과 분기 빈도 등 런타임 관찰 기록을 저장하는 메타데이터 객체)** 는 각 메서드에 붙는 관찰 노트입니다. 인터프리터와 C1이 코드를 돌리면서 여기에 기록을 남기고, 나중에 C2가 이 노트를 읽고 최적화를 결정합니다

정적 컴파일러가 갖지 못한 정보, 곧 실행 실측이 실제로 쌓이는 곳이 MDO입니다

이번 편에서 다룰 최적화는 거의 다 여기서 출발합니다

## 컴파일러가 둘이고, 임계값이 있습니다

HotSpot은 JIT 컴파일러를 **둘** 씁니다

**client compiler(클라이언트 컴파일러, 빠른 빌드 속도를 위해 최적화를 얕게 수행하고 프로파일링 계측을 삽입하는 JIT 컴파일러)** 인 C1은 빠르게 컴파일하는 대신 최적화가 얕습니다. 그 대신 만들어 내는 코드에 프로파일 계측을 심습니다

**server compiler(서버 컴파일러, 복잡한 데이터 흐름 분석을 통해 공격적이고 깊은 최적화 기계어를 생성하는 JIT 컴파일러)** 인 C2는 느리게 컴파일하는 대신 최적화가 깊습니다

이 둘을 다섯 개의 레벨로 엮은 것이 **tiered compilation(티어드 컴파일, C1의 빠른 시작 속도와 C2의 높은 실행 처리량을 단계별로 결합한 다층 컴파일 전략)** 입니다

![The five tiers of HotSpot tiered compilation](/diagrams/java-jit-c2-runtime-optimization-2.svg)

그림은 아래에서 위로 읽습니다. 아래가 느리고 위가 빠릅니다

| 레벨 | 실행 주체 | 프로파일 수집 | 쓰이는 상황 |
| --- | --- | --- | --- |
| L0 | 인터프리터 | MDO에 수집 | 모든 메서드의 출발점 |
| L1 | C1 | 없음 | 더 관찰할 게 없는 자명한 메서드 |
| L2 | C1 | 카운터만 | C2 큐가 밀릴 때의 우회로 |
| L3 | C1 | 완전 수집 | C2에게 줄 데이터를 쌓는 구간 |
| L4 | C2 | 없음(소비만) | 핫 코드의 종착지 |

오른쪽 보라색 화살표가 표준 경로입니다. **L0 → L3 → L4**

L1과 L2는 점선으로 둔 곁길입니다. L1은 게터처럼 프로파일링해 보아야 얻을 게 없는 메서드가 가고, L2는 C2 컴파일 큐가 밀릴 때 임시로 거쳐 가는 길입니다

그렇다면 임계값이 정확히 무엇을 세는 것일까요

각 메서드에는 두 개의 카운터가 붙습니다. **invocation counter(호출 카운터, 메서드가 호출된 누적 횟수)** 와 **back-edge counter(백엣지 카운터, 메서드 내부 루프가 반복된 누적 횟수)** 입니다

승격 판정은 이 둘을 함께 봅니다. 호출 횟수를 `i`, 백엣지 횟수를 `b`라 하면 L3로 올라가는 조건은 다음과 같습니다

```text
i > Tier3InvocationThreshold                                      // 200
  || (i > Tier3MinInvocationThreshold && i + b > Tier3CompileThreshold)   // 100, 2000
```

L4로 올라가는 조건도 같은 모양이고 숫자만 커집니다

```text
i > Tier4InvocationThreshold                                      // 5000
  || (i > Tier4MinInvocationThreshold && i + b > Tier4CompileThreshold)   // 600, 15000
```

그림에 적힌 `i > 200`과 `i > 5000`은 이 조건의 앞쪽 절반입니다. 흔히 "200번 호출되면 L3, 5000번이면 L4"라고 요약되는 그 숫자입니다

뒤쪽 절반이 중요합니다. 호출은 100번뿐이어도 그 안의 루프가 2000바퀴 돌았다면 `i + b`가 임계를 넘어 L3로 승격됩니다

즉 **적게 불리지만 오래 도는 메서드**도 놓치지 않겠다는 설계입니다

이 값들은 전부 조정 가능합니다(`-XX:Tier3InvocationThreshold=` 같은 플래그). 다만 컴파일 큐 길이와 컴파일러 스레드 수에 따라 런타임이 임계값을 동적으로 스케일하므로, 실측 없이 손대면 대개 손해입니다

그런데 왜 굳이 C1을 거칠까요

C2 컴파일은 시간이 많이 듭니다. 시작하자마자 모든 메서드를 C2로 보내면 컴파일 큐가 감당하지 못해 시동이 크게 늦어집니다

그래서 C1으로 일단 빠르게 기계어를 내주면서 **동시에 MDO에 프로파일을 모으고**, 정말 자주 불리는 소수만 C2로 올립니다

C1은 성능을 위한 컴파일러이면서 동시에 **C2를 위한 정보 수집기**입니다

## MDO는 무엇을 담을까

MDO를 조금 더 풀어 보면 C2가 왜 정적 컴파일러보다 유리한지가 드러납니다

MDO에는 바이트코드 지점별로 대략 다음 내용들이 기록됩니다

| 기록 항목 | 무엇을 위해 |
| --- | --- |
| 호출 카운터 · 백엣지 카운터 | 얼마나 자주 실행되는가 — 티어 승격 판정 |
| 각 호출 지점의 타입 프로파일 | 이 자리에 실제로 온 리시버 타입들 (역가상화) |
| 각 분기의 taken · not-taken 빈도 | 분기 코드 배치, 죽은 분기 판정 |
| null 관찰 여부, 배열 길이 분포 | 널 검사·경계 검사 제거 판단 |

타입 프로파일이 특히 중요합니다. 예를 들어 어떤 줄의 `x.process()` 호출이 실행 중 99.4%는 `Cat`, 0.6%는 `Dog`를 받았다는 사실이 여기에 남습니다

정적 컴파일러는 이 정보를 **빌드 시점에 알 수 없습니다**

트래픽이 실제로 어떤 타입을, 어떤 분기를, 얼마나 자주 때리는지는 프로그램을 돌려 보아야 압니다

C2는 그것을 MDO라는 형태로 손에 쥐고 컴파일합니다. 앞으로 볼 최적화는 전부 "MDO를 읽었기에 가능한" 것들입니다

## 웜업 — 처리량이 계단으로 오릅니다

이 구조의 청구서가 웜업입니다

![Throughput climbs in steps during JVM warmup](/diagrams/java-jit-c2-runtime-optimization-3.svg)

그림의 가로축은 시간, 세로축은 처리량입니다

초록색 직선이 Rust와 Go입니다. `t=0`부터 평평하게 높습니다. 빌드가 끝난 순간 최선의 기계어가 이미 확정되어 있으니 오를 계단이 없습니다

보라색 계단이 Java입니다. 인터프리터(L0)에서 낮게 시작해, C1(L3)에서 한 번 오르고, C2(L4)에서 다시 오릅니다

주목할 곳은 오른쪽 끝입니다. **보라색 최종 고원이 초록색 직선보다 위에 있습니다**

이 편과 다음 편이 함께 이 차이를 설명합니다. 왜 저 위로 올라갈 수 있는지는 이번 편이, 그리고 저기에 도달하기까지 무엇을 치르는지는 다음 편이 맡습니다

붉게 칠한 왼쪽 구간이 웜업입니다. 비용이 세 겹입니다

첫째, 초기에 인터프리터와 L3로 도는 구간이 구조적으로 느립니다

둘째, JIT 컴파일 스레드가 애플리케이션과 **CPU를 나눠 씁니다**. 계산에 써야 할 코어 일부를 컴파일이 가져갑니다

셋째, 코드 캐시가 채워지는 동안 성능이 계단식으로만 오릅니다. 매끄럽게 수렴하지 않습니다

그림에서 붉은 구간에 걸린 요청은 tail latency(테일 레이턴시, 상위 99% 등 극단적 지연 시간에 해당하는 꼬리 응답 시간)가 나쁩니다. 콜드 상태에서 처리되기 때문입니다

그래서 서버리스 함수나 CLI 도구처럼 **짧게 사는 프로세스는 이 계단을 다 오르기 전에 죽습니다.** 비용만 내고 이득은 받지 못합니다

## C2의 다섯 가지 런타임 최적화

이제 고원의 정체를 살펴봅니다. C2가 MDO를 읽고 무엇을 하는지입니다

### 하나, 공격적 인라이닝

**인라이닝**은 함수 호출을 그 함수 본문으로 통째로 갈아 끼우는 최적화입니다. 6편에서 보았듯 진짜 이득은 `CALL` 하나를 없애는 것이 아니라, 호출이라는 벽이 사라져 그 너머로 다른 최적화가 흘러 들어가는 데 있습니다

Go는 이 판단을 정적 비용 점수와 예산 80으로 했습니다. Rust는 LLVM의 정적 휴리스틱으로 판단했습니다

C2는 **실측 빈도**로 결정합니다. MDO가 "이 호출은 빈번하다"고 알려 주니 확신을 갖고 깊게 인라인합니다

빌드 시점의 추정과 실행 중의 계측은 정보의 질이 다릅니다

### 둘, 역가상화와 인라인 캐시 상태기계

Java에서 메서드는 기본이 virtual(가상 메서드, 실행 시점의 실제 객체 타입에 따라 동적으로 메서드 구현이 결정되는 성질)입니다. 서브클래스가 언제든 오버라이드할 수 있습니다

그래서 원칙적으로는 매 호출이 간접 호출이어야 합니다. 실행 시점에 리시버의 실제 타입을 보고 목적지를 정해야 하기 때문입니다

이것을 걷어내는 장치가 **inline cache(인라인 캐시, 가상 메서드 호출 지점에서 과거에 관찰된 객체 타입과 메서드 주소를 메모리에 저장하여 동적 디스패치 속도를 가속하는 최적화 기법)** 입니다. 호출 지점마다 "지난번에 어떤 타입을 불렀는지"를 그 자리에 캐싱해 두는 기법인데, 상태가 진화합니다

동네 국숫집을 떠올려 보겠습니다. 늘 같은 손님만 온다면 사장님은 얼굴만 보고도 무엇을 시킬지 압니다. 손님이 둘로 늘어도 두 얼굴만 기억해 두면 여전히 바로 압니다. 그런데 손님이 계속 늘어 누가 올지 예측할 수 없어지면, 사장님은 결국 매번 무엇을 시킬지 새로 물어봐야 합니다. 인라인 캐시가 정확히 이 세 단계를 거칩니다

![The inline cache state machine](/diagrams/java-jit-c2-runtime-optimization-4.svg)

그림을 왼쪽에서 오른쪽으로 읽으면 성능이 무너져 내리는 순서가 됩니다

**monomorphic(단형, 호출 지점에서 단 하나의 구체 클래스 타입만 관찰된 상태)** 은 그 호출 지점이 지금까지 딱 한 타입만 보았다는 뜻입니다. 그러면 그 타입의 메서드로 직접 호출하도록 캐싱하고, 나아가 인라인해 버립니다. 가장 빠릅니다

**bimorphic(2형, 호출 지점에서 정확히 두 개의 구체 클래스 타입이 관찰된 상태)** 은 두 타입이 관찰된 상태입니다. C2는 둘 다 인라인하고 앞에 **타입 가드**를 심습니다. "리시버가 `Cat`이면 이쪽, `Dog`면 저쪽, 둘 다 아니면 이탈"이라는 분기입니다

여기서 정확히 해 둘 것이 있습니다. 인터프리터와 C1이 쓰는 인라인 캐시 자체는 단형과 다형 두 상태만 갖습니다

두 타입까지 가드 인라인을 시도하는 것은 C2가 MDO의 타입 프로파일을 읽고 내리는 판단입니다

그 프로파일이 호출 지점당 기억하는 타입 개수의 기본값이 2입니다(`-XX:TypeProfileWidth=2`)

`bimorphic`이 2에서 끊기는 이유가 바로 여기에 있습니다

**megamorphic(다형, 호출 지점에서 세 개 이상의 다양한 구체 클래스 타입이 관찰되어 캐싱 한계를 넘은 상태)** 은 타입이 계속 늘어난 상태입니다. C2는 포기하고 vtable 조회로 폴백합니다

**vtable(Virtual Method Table, 클래스에 정의된 가상 메서드들의 실제 함수 메모리 주소 포인터를 담고 있는 동적 디스패치 테이블)** 은 클래스마다 하나씩 두는 함수 포인터 배열입니다. 객체 헤더의 클래스 포인터를 따라가 테이블에서 메서드 주소를 읽고, 그 주소로 점프합니다

인터페이스 호출은 **itable(Interface Method Table, 클래스가 구현한 인터페이스 메서드들의 함수 포인터를 별도로 관리하는 동적 디스패치 테이블)** 을 씁니다. 클래스가 구현한 인터페이스별로 따로 두는 같은 성격의 표인데, 원하는 인터페이스를 먼저 찾아야 해서 vtable보다 한 단계 더 듭니다

이 지점은 인라인이 불가능하고, 목적지가 계속 바뀌니 CPU의 BTB(Branch Target Buffer, 분기 대상 주소 버퍼)도 흔들립니다. 예측이 빗나가면 파이프라인이 비워지고 수십 사이클이 지연됩니다

여기가 5편과 6편에서 예고한 **다형성 가격표 3부작**이 완결되는 자리입니다

Rust의 `dyn Trait`는 fat pointer와 vtable로 값을 치릅니다

Go의 인터페이스는 `itab.Fun` 테이블을 거친 `CALL CX`로 치릅니다

Java의 megamorphic 호출은 vtable 조회로 치릅니다. 셋 다 같은 계열의 비용입니다

차이는 가격표의 존재가 아니라 **누가 언제 그것을 깎아 주느냐**입니다

Rust는 단형화로 빌드 때 아예 없애거나, `dyn`을 쓰면 끝까지 냅니다. Go는 PGO 프로파일이 있으면 빌드 때 한 번 깎습니다

Java는 실행하는 내내 깎습니다. MDO가 "이 자리는 99% `Cat`"이라고 말하면, C2는 다형적으로 **선언된** 호출을 실측에 근거해 단형 직접 호출로 바꾸고 인라인까지 해 버립니다

정적 언어에서는 상상하기 어려운 최적화입니다. 그리고 그 대가가 다음 편에 나올 역최적화입니다

### 셋, 자동 벡터화 — SuperWord

6편에서 Go에는 자동 벡터화 패스가 없다고 했고, Java의 C2에는 **SuperWord(기본 스칼라 루프를 언롤링하고 데이터 의존성을 분석하여 동형 연산들을 하나의 SIMD 벡터 명령으로 병합하는 C2의 자동 벡터화 엔진)** 가 있다고만 언급하고 넘어갔습니다. 이제 그 상세를 살펴봅니다

**SIMD(Single Instruction Multiple Data, 하나의 명령으로 여러 데이터를 동시에 처리하는 병렬 연산 기법)** 는 한 명령으로 여러 데이터를 동시에 처리하는 하드웨어 기능입니다. `float` 8개를 한 번의 AVX 명령으로 더하는 식입니다

**자동 벡터화**는 사람이 손대지 않은 평범한 스칼라 루프를 컴파일러가 알아서 SIMD 명령으로 바꿔 주는 최적화 패스입니다

C2에서 그 일을 하는 패스가 SuperWord이고, 기본으로 켜져 있습니다(`-XX:+UseSuperWord`)

![SuperWord packs a scalar loop into one SIMD instruction](/diagrams/java-jit-c2-runtime-optimization-5.svg)

동작 원리는 그림 왼쪽에서 오른쪽으로 흐릅니다

먼저 루프를 몇 배로 **언롤**합니다. 한 바퀴에 한 번 하던 연산을 여덟 번 나란히 펼쳐 놓는 것입니다

그다음 그 여덟 개 사이의 **데이터 의존성을 분석**합니다. 앞의 결과를 뒤가 써야 한다면 묶을 수 없습니다

서로 독립이고 모양이 같은(isomorphic) 연산들만 골라 하나의 SIMD 명령으로 묶습니다

`a[i] + b[i]`를 여덟 번 반복하는 스칼라 루프가, `a[i..i+8] + b[i..i+8]`을 한 번에 처리하는 AVX 명령 하나가 됩니다. 그림 오른쪽의 여덟 개 레인이 바로 그것입니다

Go 컴파일러에는 이 패스가 없습니다. 그래서 수치 루프에서 Java는 Go보다 구조적으로 유리할 수 있습니다. 웜업만 넘기면 성립합니다

그리고 여기서 5편이 예고한 사치가 하나 더 드러납니다

C2는 **실행 중인 CPU를 실제로 보고** 코드를 생성합니다. AVX-512를 쓸 수 있는지, ARM이라면 SVE2가 있는지를 실행 시점에 알고 그에 맞는 명령을 선택합니다

Rust는 이것을 빌드 시점의 `target-cpu`로 미리 결정해야 합니다. Go는 `GOARCH`로 결정합니다

같은 정보를 언제 얻느냐의 차이입니다. 그리고 Java만 가장 늦게, 가장 정확하게 얻습니다

배포 대상이 x86과 ARM에 섞여 있고 세대도 제각각인 클러스터에서, 이것은 생각보다 큰 장점입니다. 하나의 `.jar`가 노드마다 다른 최적화 기계어로 실행됩니다

### 넷, 이스케이프 분석과 스칼라 치환

**이스케이프 분석**은 객체를 가리키는 포인터가 그 함수 바깥으로 새 나가는지를 판정하는 분석입니다. 6편에서 Go의 분석 방식을 보았습니다

Go는 이 판정으로 변수를 스택에 둘지 힙에 둘지 정합니다. Java는 한 발 더 나갑니다

탈출하지 않는 객체를 **아예 할당하지 않고, 그 객체의 필드들을 레지스터나 스택 슬롯으로 분해**해 버립니다. 이것이 **scalar replacement(스칼라 치환, 힙에 할당될 객체를 쪼개어 개별 필드들을 CPU 레지스터나 스택 슬롯에 직접 배치함으로써 객체 할당 자체를 소거하는 최적화)** 입니다

객체가 스택으로 내려가는 것이 아니라 통째로 증발합니다

![Scalar replacement makes a non-escaping object disappear](/diagrams/java-jit-c2-runtime-optimization-6.svg)

그림 왼쪽이 최적화 전입니다. 지역변수 `p`는 힙에 있는 `Point` 객체를 가리키는 참조이고, 그 객체는 헤더와 두 개의 필드로 이루어져 있습니다

눈여겨볼 곳은 붉게 칠한 **헤더**입니다. 마크워드와 클래스 포인터로 12~16바이트를 차지하는데, 정작 `x + y` 계산에는 한 바이트도 쓰이지 않습니다

오른쪽이 최적화 후입니다. C2가 "이 객체는 함수 밖으로 새 나가지 않는다"를 증명하면, 객체를 필드 단위로 쪼개 레지스터에 얹습니다

할당이 사라지고, 헤더가 사라지고, 객체라는 것 자체가 사라집니다. 남는 것은 레지스터 두 개에 담긴 정수 두 개뿐입니다

```java
// before — 힙 할당 1회
Point p = new Point(x, y);
int s = p.x + p.y;

// after — 할당 0회, Point라는 객체 자체가 사라짐
int s = x + y;
```

Go의 이스케이프 분석이 "힙 대신 스택"까지 갔다면, C2는 "메모리 자체를 건너뛰고 레지스터"까지 도달합니다

덤으로 **lock elision(락 생략, 스택에 고립되어 다른 스레드가 접근할 수 없다고 증명된 객체에 동기화 잠금이 걸렸을 때 해당 불필요한 잠금 오버헤드를 정적으로 제거하는 최적화)** 이 따라옵니다. 탈출하지 않는 객체는 다른 스레드가 볼 수 없으니, 거기에 걸린 `synchronized`는 아무도 경합하지 않는 잠금입니다. C2는 그 잠금을 제거합니다

### 다섯, 트랩 배치

C2는 MDO의 분기 빈도를 보고 차가운 분기를 핫 경로에서 들어냅니다

들어낸 자리에는 **uncommon trap(실행 확률이 극히 희박한 분기 경로에 도달할 경우, 즉시 최적화된 기계어 실행을 중단하고 인터프리터 상태로 되돌아가도록 지시하는 런타임 이탈 표식)** 을 심습니다. "이 경로로 실행이 흘러오면 최적화를 포기하고 인터프리터로 이탈하라"는 표식입니다

그러면 핫 경로의 명령이 촘촘해지고(코드 밀도), 분기 예측기가 볼 분기 수가 줄어듭니다

한 번도 실행되지 않은 분기는 아예 컴파일하지 않는 것이 C2의 기본 태도입니다

### 곁가지 — OSR, 달리는 중에 엔진 갈아 끼우기

메서드 호출 카운터로는 잡히지 않는 경우가 있습니다

`main` 안에 거대한 `for` 루프가 하나 있고 그 안에서 수억 번의 계산이 도는 코드를 생각해 보겠습니다. 호출은 딱 한 번입니다

호출 카운터가 늘지 않으니 티어 승격이 되지 않습니다. 프로그램이 끝날 때까지 인터프리터로 도는 셈입니다

그래서 백엣지 카운터가 있습니다. 앞의 승격 조건에서 `i + b`의 `b`가 이 루프 반복을 잡아냅니다

문제는 그다음입니다. 루프가 이미 실행 중인데 컴파일된 새 코드로 어떻게 갈아탈 수 있을까요

**OSR(On-Stack Replacement, 루프가 실행 중인 상태에서 현재 스택 프레임을 최적화된 컴파일러 프레임으로 동적 교체하여 즉시 기계어 실행으로 전환하는 기법)** 이 그 역할을 수행합니다. 실행 중인 스택 프레임을 컴파일된 코드의 프레임으로 교체하고, 루프의 한가운데로 진입시킵니다

말 그대로 달리는 자동차의 엔진을 바꿔 끼우는 최적화입니다. 다음 편의 역최적화와 정확히 반대 방향으로 같은 기계를 사용합니다

## 그런데 이 최적화는 공짜가 아닙니다

지금까지 본 다섯 가지 최적화는 전부 같은 성격을 공유합니다. MDO가 알려 준 실측을 믿고 낙관적으로 베팅한다는 점입니다

"이 호출은 항상 `Cat`이다", "이 객체는 함수 밖으로 안 나간다", "이 분기는 절대 안 탄다"

베팅이 맞아떨어지는 동안은 정적 컴파일러가 원리적으로 못 가는 자리까지 갑니다. 그런데 트래픽이 바뀌어 베팅이 틀리면 어떻게 될까요

이미 그 기계어를 실행 중이던 스레드를 어떻게 되돌리는가가 남은 문제입니다. 그리고 그 되돌리는 과정 자체가 스레드를 멈춰 세웁니다

다음 편은 이 청구서, 즉 역최적화와 세이프포인트부터 시작해 그 웜업 비용을 실무에서 다스리는 법까지 이어갑니다

## 실무 운영 관점에서 짚어볼 질문

1. **웜업과 배포 안전성** — 롤링 배포 중 갓 뜬 콜드 JVM에 트래픽이 쏠리면 웜업 구간의 나쁜 tail이 사용자에게 그대로 노출됩니다. 쿠버네티스 `readinessProbe`를 "웜업 완료"에 맞추려면 무엇을 신호로 쓸까요? 요청 N건 처리를 셀지, `PrintCompilation` 로그로 C2 수렴을 확인할지. 배포 전에 트래픽을 미러링해 코드 캐시와 MDO를 데우는 파이프라인은 어떻게 설계할까요?
2. **HPA 왜곡** — JIT 컴파일 스레드는 웜업 동안 CPU를 추가로 씁니다. CPU 사용률 기반 HorizontalPodAutoscaler가 이 일시적 스파이크를 부하 증가로 오인해 불필요한 스케일아웃을 하지 않게 하려면 어떻게 해야 할까요? 워밍업 구간의 CPU를 어떻게 배제하거나 평활할까요?

## 핵심 요약

- `javac`는 기계어를 만들지 않습니다. 아키텍처를 모르는 바이트코드까지만 만들고, 진짜 기계어는 실행 중 HotSpot 안에서 태어나 코드 캐시에 쌓입니다. 같은 프로그램도 실행마다 코드 캐시의 내용이 달라집니다. 5편에서 본 Rust `.text`의 불변성과 정확히 대척점입니다
- MDO(MethodData Object)는 각 메서드에 붙는 관찰 노트입니다. 호출·백엣지 카운터, 호출 지점별 타입 프로파일, 분기 빈도가 여기 쌓이고, C2의 모든 최적화가 이 노트를 읽고 결정됩니다
- 티어드 컴파일은 L0(인터프리터)부터 L4(C2)까지 다섯 레벨입니다. 표준 경로는 L0 → L3 → L4이고, 승격 조건은 호출 횟수 단독(200, 5000)이 아니라 호출과 백엣지의 합도 함께 봅니다. 적게 불리지만 오래 도는 메서드를 놓치지 않기 위해서입니다
- C1은 성능을 위한 컴파일러이면서 C2를 위한 정보 수집기입니다. 그래서 굳이 C1을 거칩니다
- 웜업 비용은 세 겹입니다. 초기 실행이 느리고, JIT 스레드가 CPU를 나눠 쓰고, 처리량이 계단식으로만 오릅니다. 짧게 사는 프로세스는 계단을 다 오르기 전에 죽습니다
- 인라인 캐시는 monomorphic에서 bimorphic을 거쳐 megamorphic으로 무너집니다. 단형이면 직접 호출로 인라인하고, 다형이 되면 vtable 조회로 폴백해 인라인이 막힙니다. Rust의 vtable, Go의 `itab`, Java의 megamorphic은 같은 계열의 비용이고, 차이는 누가 언제 깎아 주느냐입니다
- C2의 SuperWord가 자동 벡터화를 합니다. 루프를 언롤하고 데이터 의존성을 분석해 독립적이고 동형인 연산들을 하나의 SIMD 명령으로 묶습니다. Go에는 이 패스가 없습니다
- Java만 실행 중인 CPU를 실제로 보고 그 아키텍처의 기계어를 뽑습니다. AVX-512인지 SVE2인지를 실행 시점에 압니다. Rust의 `target-cpu`와 Go의 `GOARCH`는 빌드 시점의 결정입니다
- 스칼라 치환은 탈출하지 않는 객체를 스택에 내리는 게 아니라 통째로 증발시킵니다. 필드가 레지스터와 스택 슬롯으로 분해되고, 덤으로 경합 없는 `synchronized`가 지워집니다
- 이 다섯 가지는 전부 MDO의 실측을 믿는 낙관적 베팅입니다. 베팅이 틀렸을 때 무슨 일이 벌어지는지는 다음 편의 몫입니다
