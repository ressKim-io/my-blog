## 커널의 강제 종료와 런타임의 예외는 다른 현상입니다

"메모리 부족으로 종료되었다"는 표현에는 **서로 다른 두 가지 현상**이 섞여 있습니다. 이를 구분하는 것이 원인 분석의 출발점입니다

![Two deaths: one leaves a stack trace, one leaves nothing](/diagrams/offheap-oomkilled-nmt-1.svg)

그림 왼쪽은 **런타임 수준의 종료**를 나타냅니다

힙(`-Xmx`)이 포화 상태에 이르러 GC를 수행해도 공간을 확보하지 못하면, JVM은 자체적으로 `java.lang.OutOfMemoryError`를 **발생시킵니다**

이는 애플리케이션 레벨의 예외이므로 **stack trace(스택 트레이스, 예외나 오류 발생 시 스레드가 호출한 함수들의 역추적 경로를 단계별로 보여주는 진단 로그 기록)** 가 기록되며, 직전 GC 로그에는 연속적인 Full GC 수행 내역이 남습니다. **로그를 통해 추적이 가능합니다**

그림 오른쪽은 **커널 수준의 강제 종료**를 나타냅니다

프로세스 전체(힙과 힙 외부 전부)의 물리 페이지 합이 cgroup `memory.max`를 초과하면, [20편](/essays/container-memory-accounting-cgroup)에서 확인했듯 OOM killer가 **`SIGKILL`** 을 전송합니다

**SIGKILL은 애플리케이션에서 가로채거나 무시할 수 없습니다.** 셧다운 훅, `OutOfMemoryError`, 스택 트레이스 어느 것도 남지 않습니다. 종료 코드 `137`만 남고 **GC 로그는 마지막까지 정상 상태를 유지합니다**

"GC 로그는 정상인데 파드가 종료된다"는 상황은 정확히 두 번째 경우에 해당합니다. 힙은 정상 범위이므로 GC는 제 역할을 다하고 있고(그래서 로그가 깨끗함), 실제로 한도를 초과시킨 원인은 **힙 외부**에 있습니다

## 동일한 프로그램, 플래그 하나에 따른 결과 차이

이 현상을 직접 확인하기 위해 두 가지 종료 방식을 나란히 재현합니다

힙은 거의 사용하지 않고 **힙 외부만** 지속적으로 증가시키는 프로그램입니다. 다이렉트 버퍼를 8MiB씩 반복적으로 할당합니다

```java
List<ByteBuffer> keep = new ArrayList<>();
for (int i = 1; i <= 400; i++) {
    // 힙 밖 다이렉트 버퍼 8MiB. 힙에는 참조 하나(수십 바이트)만 남는다
    ByteBuffer b = ByteBuffer.allocateDirect(8 << 20);
    b.put(0, (byte) 1);
    keep.add(b);
    ...
}
```

**`ByteBuffer.allocateDirect()`** 는 자바 힙이 아닌 네이티브 메모리를 할당받습니다. Netty·gRPC·카프카 클라이언트가 I/O 버퍼로 사용하는 방식입니다. 힙에는 해당 버퍼를 가리키는 참조 하나만 남습니다

컨테이너 메모리 1GiB, 힙 상한 512MiB로 실행합니다

### (A) 기본 설정 — 런타임에 의한 자체 통제

```bash
docker run --memory=1g offheap java -Xmx512m -Xms512m -Xlog:gc Leak
```

```text
다이렉트  480 MiB 할당  |  힙 사용   6 / 494 MiB ( 1%)
[1.405s][info][gc] GC(0) Pause Full (System.gc()) 6M->1M(494M) 5.702ms
Exception in thread "main" java.lang.OutOfMemoryError:
    Cannot reserve 8388608 bytes of direct buffer memory
    (allocated: 511705088, limit: 518979584)
	at java.base/java.nio.Bits.reserveMemory(Bits.java:178)
	at java.base/java.nio.DirectByteBuffer.<init>(DirectByteBuffer.java:108)
	at Leak.main(Leak.java:10)

ExitCode=1  OOMKilled=false
```

JVM이 임계치 도달을 감지하고 **자체적으로 예외를 발생시켰습니다**

`MaxDirectMemorySize`의 기본값은 `0`이며, 이는 "무제한"이 아니라 **"상한을 `-Xmx`에 맞춘다"** 는 의미입니다. 로그의 `limit: 518979584`가 이 값에 해당합니다. 다이렉트 버퍼 할당량이 힙 크기에 도달하자 `OutOfMemoryError`가 발생했습니다

`Pause Full (System.gc())` 로그도 주목할 부분입니다. JVM은 예외를 발생시키기 직전 **의도적으로 Full GC를 수행했습니다.** 다이렉트 버퍼는 GC가 참조를 회수해야(**cleaner(클리너, 자바에서 다이렉트 버퍼 등의 네이티브 리소스를 가리키는 힙 내 객체 참조가 가비지 컬렉터에 의해 회수될 때 네이티브 메모리를 함께 해제하도록 예약하는 자원 정리 메커니즘)**) 네이티브 메모리가 반환되므로, 회수 가능한 참조가 있는지 마지막으로 확인한 것입니다

중요한 것은 결과입니다. **종료 코드 1, 스택 트레이스와 함께 원인이 로그에 명확히 기록되었습니다.** 원인 진단이 가능한 종료 형태입니다

### (B) 런타임 상한을 컨테이너 한도 위로 설정한 경우

동일한 조건에서 `-XX:MaxDirectMemorySize=2g` 옵션 하나만 변경합니다. 컨테이너 한도는 여전히 1GiB입니다

```bash
docker run --memory=1g offheap \
  java -Xmx512m -Xms512m -XX:MaxDirectMemorySize=2g -Xlog:gc Leak
```

```text
다이렉트  800 MiB 할당  |  힙 사용   6 / 494 MiB ( 1%)
다이렉트  880 MiB 할당  |  힙 사용   6 / 494 MiB ( 1%)
다이렉트  960 MiB 할당  |  힙 사용   6 / 494 MiB ( 1%)

ExitCode=137  OOMKilled=true
```

**프로세스가 즉시 강제 종료됩니다**

예외가 발생하지 않으며 스택 트레이스도 남지 않습니다. GC 로그에서 Full GC와 `OutOfMemoryError` 발생 건수는 **0건**입니다

종료 직전까지 **힙 사용률은 1%** 였습니다. 494MiB 중 6MiB만 사용 중이었습니다

![The same leak, two endings — the cap decides who kills you](/diagrams/offheap-oomkilled-nmt-2.svg)

두 시나리오의 메모리 증가 궤적은 동일합니다. 하단의 힙 영역(파란색)은 양쪽 모두 거의 변화가 없으며, 다이렉트 버퍼 영역(회색)만 지속적으로 증가합니다

결과가 갈리는 지점은 **한도선의 위치**입니다

**(A) 시나리오:** 런타임의 상한(`MaxDirectMemorySize` ≈ 495MiB)이 커널의 강제 종료 기준(1GiB)보다 **아래**에 위치합니다. 런타임이 먼저 한도에 도달해 예외를 발생시킵니다

**(B) 시나리오:** 상한을 2GiB로 높여 커널의 강제 종료 기준보다 **위**로 설정했습니다. 이 경우 커널이 먼저 한도에 도달합니다

**같은 코드, 같은 힙 상태, 같은 메모리 증가 패턴이지만, 상한을 어디에 두었는지가 종료 방식을 결정합니다**

여기서 운영 원칙이 도출됩니다. **힙 외부 영역에 명시적인 상한을 설정하여, 커널이 강제 종료하기 전에 런타임이 예외를 발생시키도록 해야 합니다.** 이를 통해 로그에 원인이 기록되고 장애 진단이 가능해집니다

## 힙 외부 메모리의 구조와 NMT 실측

[13편](/essays/jvm-tlab-bump-pointer-offheap)에서 `-Xmx` 외부의 구조를 다뤘습니다. Metaspace·코드 캐시·스레드 스택·다이렉트 메모리가 힙 외부에 존재하며, cgroup은 이 전체를 합산합니다

이번에는 이 구조를 **수치로** 확인합니다. **NMT(Native Memory Tracking, JVM이 자바 힙 외부에서 할당하고 사용하는 네이티브 메모리를 범주별로 추적하고 계측하는 자체 프로파일링 도구)** 를 활성화하면 JVM이 힙 외부에서 사용하는 메모리를 범주별로 확인할 수 있습니다

```bash
java -Xmx256m -XX:NativeMemoryTracking=summary \
     -XX:+UnlockDiagnosticVMOptions -XX:+PrintNMTStatistics Idle
```

유휴 상태 JVM의 측정 결과입니다(JDK 25.0.3 기준)

| 범주 | committed | reserved |
|---|---|---|
| Java Heap | 256 MiB | 256 MiB |
| **GC** | **53.4 MiB** | 53.4 MiB |
| Shared class space | 13.6 MiB | 16 MiB |
| Code | 7.5 MiB | **244 MiB** |
| Thread | 0.46 MiB | 26 MiB |
| Metaspace | 0.20 MiB | 64 MiB |
| Class | 0.19 MiB | **1.02 GiB** |
| **Total** | **334 MiB** | **1.65 GiB** |

이 표의 핵심은 `committed`와 `reserved` 두 수치 간의 차이입니다

### Reserved와 Committed, 그리고 커널의 실제 청구량

**`reserved`(예약 공간)**: 프로세스가 OS로부터 확보한 **가상 주소 공간**입니다. 표의 Class 범주 1.02GiB, Code 범주 244MiB가 이에 해당합니다

예약 공간은 **실제 물리 메모리를 소비하지 않습니다.** [20편](/essays/container-memory-accounting-cgroup)에서 확인했듯, 접촉되지 않은 페이지에는 물리 프레임이 할당되지 않습니다

**`committed`(커밋 공간)**: 애플리케이션이 실제 사용을 위해 OS에 확정적으로 요청한 용량입니다

**커널이 `memory.current`에 청구하는 값은 실제로 접촉된 페이지의 총합**입니다

![Reserved, committed, and what the kernel actually charges](/diagrams/offheap-oomkilled-nmt-3.svg)

그림은 이 세 지표의 포함 관계를 나타냅니다

**가장 바깥쪽 옅은 영역**이 reserved(1.65GiB)입니다. 범위는 넓지만 대부분 비어 있습니다

**가운데 파란 영역**이 committed(334MiB)입니다

**가장 안쪽 붉은 영역**이 실제로 접촉되어 커널이 청구하는 부분입니다

따라서 `ps`에서 VSZ가 수 GB로 표시되어도 정상입니다. **예약된 가상 공간일 뿐입니다**

실제로 위험한 것은 committed 영역이 물리적으로 채워지며 증가하는 부분입니다. Metaspace·다이렉트 버퍼·malloc arena가 이에 해당합니다

[21편](/essays/gc-headroom-bin-packing-density)에서 확인한 지표 간 불일치도 여기서 설명됩니다. **NMT는 JVM이 직접 관리하는 메모리만 측정합니다.** C 라이브러리가 할당한 영역, 실행 파일의 메모리 매핑, ZGC의 `memfd` 힙 등은 NMT의 측정 범위 밖에 있습니다. 커널의 회계와 런타임의 회계는 **측정 범위가 다릅니다**

### 유휴 상태에서도 발생하는 GC 오버헤드

유휴 상태임에도 힙 다음으로 가장 큰 항목이 **GC 범주(53.4MiB)** 입니다

카드 테이블, 마킹 비트맵, G1의 Remembered Set은 모두 힙 외부의 네이티브 자료구조입니다. **GC 동작을 지원하는 메타데이터 자체가 힙과 별도로 메모리를 점유합니다**

21편에서 확인한 "프로세스 단위로 고정 발생하는 오버헤드"의 상당 부분이 여기에 해당합니다

## 힙 외부 메모리 팽창의 주요 원인

`memory.max` 초과를 유발하는 대표적인 힙 외부 영역들입니다. 공통점은 **GC 로그에 기록되지 않는다**는 점입니다

### Metaspace — 기본 상한의 부재

클래스 메타데이터가 저장되는 영역입니다. 핵심 문제는 **기본적으로 상한이 사실상 무제한**이라는 점입니다

13편에서 확인했듯 `MaxMetaspaceSize`가 `18446744073709551615`, 즉 2⁶⁴−1입니다

따라서 **classloader leak(클래스로더 누수, 동적 클래스 로딩이나 리플렉션을 사용하는 프레임워크에서 기존 클래스로더 참조가 해제되지 않아 Metaspace 메모리가 계속 누적되는 메모리 누수 현상)**(동적 클래스 생성, 리로딩, 과도한 프록시 사용)이 발생하면 Metaspace가 지속적으로 증가합니다. 힙은 GC에 의해 정리되어 정상 범위를 유지하지만, Metaspace는 상한이 없어 커널 한도까지 계속 증가합니다

`-XX:MaxMetaspaceSize`를 **명시적으로 설정하면** 커널이 강제 종료하기 전에 JVM이 `OutOfMemoryError: Metaspace`를 발생시킵니다. 앞선 (A)/(B) 실험과 동일한 원리입니다. **진단 가능성이 크게 달라집니다**

### Code Cache — JIT 컴파일러 영역

[8편](/essays/jit-deopt-safepoint-tail-latency)에서 다룬 코드 캐시입니다. 기본 예약 공간은 약 240MB입니다

대규모 코드베이스나 프레임워크가 많은 애플리케이션에서는 committed 용량이 240MB까지 증가할 수 있습니다. 힙과 무관한 영역이므로 GC 로그에는 기록되지 않습니다

### 스레드 스택 — 스레드 수 × 스택 크기

각 스레드는 네이티브 스택 공간을 할당받습니다. JDK 25 기준 `ThreadStackSize` 기본값은 2048KB, 즉 **스레드당 2MiB**입니다

스레드가 수천 개로 증가하면(스레드풀 설정 오류, 블로킹 I/O에서 요청당 스레드 생성, 커넥션 폭증) 이 값이 그대로 네이티브 메모리 사용량이 됩니다

### Direct Buffer — 힙 상한에 종속되는 한도

[4편](/essays/runtime-weight-zero-cost-vs-fat)에서 다룬 내용입니다. `-XX:MaxDirectMemorySize`를 지정하지 않으면 기본 상한이 `-Xmx`와 동일하게 설정되므로, 힙을 3GB로 잡으면 다이렉트 메모리도 최대 3GB까지 추가로 사용할 수 있습니다

앞선 (A)/(B) 실험이 이를 실제로 재현한 것입니다

기본 상한이 `-Xmx`와 동일하다는 점이 위험 요인입니다. **힙을 크게 설정할수록 다이렉트 버퍼 한도도 함께 증가합니다**

`-Xmx`를 3GiB로 설정하면 다이렉트 메모리도 3GiB까지 사용할 수 있어 합계가 6GiB에 이릅니다. 컨테이너 한도가 4GiB라면 이 조합만으로 초과됩니다

게다가 다이렉트 버퍼는 GC가 참조를 회수해야 반환됩니다. **힙에 여유가 많아 GC가 자주 발생하지 않으면, 오히려 다이렉트 버퍼가 반환되지 않고 누적되는** 역설이 발생합니다

### glibc arena — MALLOC_ARENA_MAX에 대한 통념 검증

[11편](/essays/ptmalloc-rust-malloc-delegation)에서 다룬 ptmalloc의 **arena(아레나, ptmalloc 등 C/C++ 메모리 할당자가 멀티스레드 환경의 락 경합을 줄이기 위해 스레드별 또는 그룹별로 분할하여 독립적으로 관리하는 메모리 구획)** 구조입니다. 락 경합을 줄이기 위해 여러 개의 arena를 유지하며, 64비트 시스템에서 최대 개수는 **코어 수 × 8**입니다

이 구조가 컨테이너 환경의 RSS 지표에 미치는 영향을 확인합니다

16개의 스레드가 각각 64KiB 크기의 청크를 256개씩 할당한 후(총 256MiB) **전체를 `free()`** 하는 C 프로그램을 4코어 컨테이너에서 실행했습니다

| | 최대 점유 | `free()` 후 | `malloc_trim()` 후 |
|---|---|---|---|
| 기본 (arena 상한 32개) | 258 MiB | **3 MiB** | 3 MiB |
| **`MALLOC_ARENA_MAX=2`** | 258 MiB | **96~113 MiB** | **1 MiB** |

먼저 11편에서 예측한 현상이 확인됩니다. **`free()`를 호출했음에도 RSS가 즉시 감소하지 않습니다.** `MALLOC_ARENA_MAX=2` 환경에서는 100MiB에 가까운 메모리가 그대로 남아 있습니다

`malloc_trim(0)`을 호출하자 **점유량이 즉시 1MiB로 감소했습니다.** 메모리 누수가 아니라, **할당자가 재사용을 위해 보유하고 있던 것**임을 의미합니다

**오히려 `MALLOC_ARENA_MAX=2`를 설정한 쪽이 더 많은 메모리를 반환하지 못하고 보유했습니다**

이는 일반적으로 알려진 조언과 상반되는 결과입니다. "JVM 컨테이너의 RSS가 지속적으로 증가하면 `MALLOC_ARENA_MAX=2` 설정으로 해결할 수 있다"는 가이드가 흔히 통용되기 때문입니다

![Fewer arenas held more memory, not less](/diagrams/offheap-oomkilled-nmt-4.svg)

원인은 arena의 메모리 관리 방식에 있습니다

그림 왼쪽은 arena 개수 제한이 없는 기본 설정입니다. 11편에서 확인했듯 **thread arena는 `mmap`으로 할당받은 독립 구획**을 사용합니다. 각 스레드가 자신의 구획을 독점하므로, 내부 청크가 모두 해제되면 **구획 전체가 비워져** 커널에 반환될 수 있습니다

그림 오른쪽은 `MALLOC_ARENA_MAX=2` 설정입니다. 여덟 개의 스레드가 **하나의 구획을 공유**합니다. 서로 다른 스레드가 할당한 청크들이 **교차로 배치**됩니다

모두 해제되더라도 **완전히 비워진 연속 구획이 형성되지 않습니다.** 따라서 할당자가 메모리를 계속 보유하게 됩니다

**arena 개수 설정은 "락 경합"과 "단편화" 사이의 트레이드오프이며, 낮춘다고 항상 유리한 것은 아닙니다**

단, 이는 특정 할당 패턴(균일한 64KiB 청크를 일괄 할당·일괄 해제)에서의 결과입니다. 실제 JVM 컨테이너에서 `MALLOC_ARENA_MAX=2`가 효과를 보인 사례도 존재하며, 이는 할당 패턴의 차이에 기인합니다

결론적으로 **arena 상한 설정이 무조건적인 해결책은 아니며**, 워크로드에 따라 실효성이 달라지므로 **반드시 사전 실측 후 적용해야 합니다**

### GC 관련 자료구조 — 앞서 확인한 53MiB

NMT 표에서 힙 다음으로 컸던 항목입니다. GC 동작을 지원하는 자료구조가 힙 외부에 존재합니다

## Go 환경의 동일한 취약점 — GOMEMLIMIT의 cgo 영역 누락

Go 환경의 힙 외부 메모리 구조는 JVM보다 단순하지만, 동일한 취약점이 존재합니다

핵심적인 제약은 **`GOMEMLIMIT` 설정이 Go 런타임이 직접 관리하는 메모리(힙, 고루틴 스택, 런타임 내부 구조)에만 적용된다**는 점입니다

**cgo(C-Go 연동 메커니즘, Go 언어 코드 내부에서 C 언어로 작성된 라이브러리를 직접 호출하거나 C의 네이티브 메모리 할당 함수를 사용할 수 있게 해 주는 상호 운용 기능)를 통해 C 라이브러리가 `malloc`으로 할당한 메모리는 이 범위에서 제외됩니다**

Java의 다이렉트 버퍼 실험과 동일한 방식으로, C 언어 확장을 통해 8MiB씩 지속 할당하는 코드를 작성해 검증합니다

```go
/*
#include <stdlib.h>
#include <string.h>
static void* grab(size_t n) { void* p = malloc(n); if (p) memset(p, 1, n); return p; }
*/
import "C"

keep = append(keep, C.grab(C.size_t(8<<20))) // C 쪽에서 8MiB malloc
```

컨테이너 한도를 1GiB로 지정하고, **`GOMEMLIMIT=700MiB`** 로 명시적으로 설정한 뒤 실행합니다

```text
GOMEMLIMIT=700 MiB
C 힙  800 MiB 할당  |  Go 힙  0 MiB · GC 0회
C 힙  880 MiB 할당  |  Go 힙  0 MiB · GC 0회
C 힙  960 MiB 할당  |  Go 힙  0 MiB · GC 0회

ExitCode=137  OOMKilled=true
```

**Go 힙 0 MiB, GC 0회**

`GOMEMLIMIT`을 700MiB로 설정했음에도 프로세스는 960MiB를 사용하다 종료되었습니다. Go 런타임 입장에서는 **자신이 관리하는 메모리 사용량이 0이므로 문제를 감지하지 못한 것입니다**

`gctrace`와 `runtime.MemStats` 모두 정상 상태를 표시하지만, 프로세스는 `exit 137`로 종료됩니다

**JVM의 다이렉트 버퍼 문제와 정확히 동일한 구조입니다.** 메모리 회계를 런타임에 위임했으나, **런타임이 계량하지 않는 영역**이 존재하는 것입니다

참고로 Go에는 **OS 반환 정책**과 관련된 이력이 있습니다. [10편](/essays/kernel-lazy-allocation-page-fault)·[12편](/essays/go-allocator-mcache-contiguous-stack)에서 다뤘듯 Go 1.12에서 `MADV_FREE`로 전환했으나 컨테이너 모니터링과 충돌하는 문제가 있어 **1.16부터 `MADV_DONTNEED`가 다시 기본값**이 되었습니다. 이에 따라 현재 Go 프로세스의 RSS는 비교적 실제 사용량을 정확히 반영합니다

## 언어별 힙 외부 가시성 비교

![The wider the runtime, the more places to die behind a clean log](/diagrams/offheap-oomkilled-nmt-5.svg)

그림의 세로 점선은 **런타임이 감지할 수 있는 가시성 경계**를 나타냅니다. 점선 왼쪽은 런타임 계측(로그, 예외)이 가능하며, 오른쪽은 가시성 범위 밖입니다

반면 **커널의 계량 범위(`memory.current`)는 점선 오른쪽의 비가시 영역까지 모두 포함합니다.** 이 영역이 넓을수록 로그에 흔적을 남기지 않고 종료될 가능성이 높아집니다

| 환경 | 힙 외부 구조 | 징후 없는 종료 위험 | 사유 |
|---|---|---|---|
| **Rust** | 모든 동적 할당이 명시적이며 런타임 추상화가 최소화됨 | 낮음 | GC와 거대 런타임이 없어 비가시적으로 증가하는 영역이 적음. 단편화 문제는 있으나 해제 시점이 코드에 명확히 드러남 |
| **Go** | 고루틴 스택 + **cgo/C 할당** | 중간 | 순수 Go 환경에서는 가시성이 높으나, **cgo 사용 시** `GOMEMLIMIT` 범위 밖의 영역이 생겨 Java와 유사한 위험에 노출됨 |
| **Java** | Metaspace·코드캐시·다이렉트·스레드스택·arena·GC구조 | **높음** | 힙 외부 영역이 넓고 상당수가 기본 무제한이거나 큰 예약 공간을 가짐. NMT 없이는 추적이 어려움 |

표를 세로로 보면 다음과 같습니다. **런타임이 클수록 힙 외부 구조가 넓고, 넓을수록 '정상 로그' 뒤에서 종료될 가능성이 커집니다**

Rust는 모든 메모리 할당이 코드에 명시적으로 드러나므로 숨겨진 영역이 적습니다

Java는 런타임이 관리하는 영역이 넓어 편의성은 높지만, **그만큼 관리 범위 밖으로 벗어나는 경로도 많습니다**

Go는 순수 Go 코드일 때는 Rust에 가깝고 cgo를 사용하면 Java에 가까워지는, **설계 선택이 위험 수준을 결정하는** 중간 지점에 있습니다

[19편](/essays/gc-cost-conservation-no-silver-bullet)에서 다룬 "회계 주체의 위치"라는 축이 컨테이너 환경에서 다시 나타난 것입니다. 회계를 런타임에 위임할수록 개발 편의성은 높아지지만, **런타임이 계량하지 않는 메모리 영역이 사각지대**로 남습니다

## 운영 환경 권장 가이드라인

### 1. 추정 대신 실측 도구로 확인

**Java 환경:** `-XX:NativeMemoryTracking=summary` 옵션으로 실행하고, `jcmd <pid> VM.native_memory summary` 명령으로 범주별 committed 수치를 확인합니다

시간에 따라 증가하는 범주를 확인하면 원인을 특정할 수 있습니다. `jcmd ... VM.native_memory baseline`로 스냅샷을 캡처한 뒤 `... summary.diff`로 증가분을 추적하면 누수 지점을 빠르게 파악할 수 있습니다

**Go 환경:** `GODEBUG=gctrace=1`로 힙 상태가 정상인지 확인하고, cgroup `memory.current`와의 격차가 크다면 **cgo 영역을 우선 점검해야 합니다.** 앞서 확인했듯 Go 런타임 자체 지표는 정상으로 표시되기 때문입니다

### 2. 명시적 상한 설정으로 가시성 확보

본문의 (A)/(B) 실험이 보여준 전략입니다. **기본 상한이 없는 힙 외부 영역에 명시적인 상한을 설정하여, 커널이 강제 종료하기 전에 런타임이 예외를 발생시키도록** 해야 합니다

`-XX:MaxMetaspaceSize`로 Metaspace 증가를 `OutOfMemoryError: Metaspace`로 전환하고, `-XX:MaxDirectMemorySize`로 다이렉트 버퍼 상한을 명시적으로 설정하며, 스레드풀 크기와 `-Xss`로 스택 총량을 제어합니다

**단, 설정한 상한의 총합이 컨테이너 한도보다 작아야 합니다.** (B)에서 확인했듯 상한을 한도 위로 설정하면 이 전략은 무의미해집니다

### 3. 컨테이너 메모리에 여유 공간 확보

컨테이너 `limits.memory`는 **힙, 힙 외부 전체, 페이지 캐시 여유**를 모두 수용할 수 있는 크기로 설정해야 합니다

`-Xmx`(또는 `MaxRAMPercentage`)를 한도의 50~75% 수준으로 설정하고 나머지를 힙 외부 몫으로 남겨야 합니다. **"한도 = `-Xmx`"로 설정하면 본문에서 다룬 모든 요인이 곧바로 OOMKilled로 이어질 수 있습니다**

Go 환경도 마찬가지로 `GOMEMLIMIT`을 한도의 약 90% 수준으로 설정하되, **cgo를 사용한다면 그만큼 비율을 추가로 낮춰야** 합니다

## 트레이드오프 소결

한 문장으로 정리하면 이렇습니다

**커널은 힙과 힙 외부를 구분하지 않고 합산하지만, 런타임은 힙만 관리하고 로그에 기록합니다. 이 사각지대가 exit 137로 나타납니다**

따라서 진단 절차는 다음과 같이 정리됩니다

메모리 부족으로 종료된 경우, 먼저 **런타임 수준의 종료(예외·로그 존재)인지 커널 수준의 강제 종료(SIGKILL·로그 없음·exit 137)인지**를 구분해야 합니다

후자에 해당한다면 힙 지표 분석에 시간을 소모할 필요가 없습니다. 힙은 정상일 가능성이 높으므로, **NMT와 `memory.stat`을 통해 힙 외부 영역을 분석해야 합니다**

5부의 세 편이 여기서 이어집니다. 20편이 "커널은 모든 메모리를 합산한다"는 회계 규칙을, 21편이 "리소스 요구량을 정밀하게 잡아도"라는 전제를, 22편이 "힙 외 여백이 없으면 그 안에서 종료된다"는 결론을 다뤘습니다

남은 것은 시간 축입니다. 이 모든 현상이 **프로세스가 막 시작되는 순간**에는 어떻게 나타나는가입니다

2. **상한 총합 검증** — `-Xmx` + `MaxMetaspaceSize` + `MaxDirectMemorySize` + (스레드 수 × `-Xss`) + 코드 캐시의 합이 컨테이너 한도를 초과하면, 개별 상한 설정의 의미가 사라지는 이유는 무엇일까요? 이 총합을 계산해 한도와 비교하는 검증 스크립트를 배포 파이프라인에 추가한다면 어떤 값을 어디서 읽어와야 할까요?
3. **Metaspace 상한 산정** — 상한을 설정하지 않으면 커널 OOM으로, 지나치게 낮게 설정하면 `OutOfMemoryError: Metaspace`로 조기 종료됩니다. 배포마다 클래스 수가 증가하는 애플리케이션에서 이 값을 NMT의 `Class committed` 추이를 바탕으로 어떻게 산정해야 할까요?
4. **`MALLOC_ARENA_MAX` 실효성 검증** — 이번 편에서 `MALLOC_ARENA_MAX=2`가 오히려 메모리 반환을 저해하는 사례를 확인했습니다. 특정 워크로드에서 이 설정이 유리한지 불리한지를 어떤 실험으로 판정해야 할까요? 락 경합(처리량)과 RSS를 동시에 평가하려면 무엇을 계측해야 할까요?
5. **Go cgo 사각지대 대응** — cgo를 사용하는 Go 서비스에서 `GOMEMLIMIT`을 한도의 90%로 설정했음에도 OOMKilled가 발생합니다. C 언어 영역의 할당을 어떤 도구로 계측하고, `GOMEMLIMIT` 비율을 어떻게 재조정해야 할까요? cgo를 제거하는 선택과 비교했을 때의 트레이드오프는 무엇일까요?
6. **컨테이너 한도 산정 자동화** — "힙 + 힙 외부 + 페이지 캐시 여유"를 담는 `limits.memory`를 추정이 아닌 부하 테스트의 NMT·RSS 실측값으로 자동 산정하려면 어떤 파이프라인이 필요할까요? 배포 후 힙 외부 영역 증가를 어떤 지표로 감지해야 할까요?

## 핵심 요약

- **커널의 강제 종료와 런타임의 예외는 다른 현상입니다.** 런타임 종료는 `OutOfMemoryError`·스택 트레이스·GC 로그를 남기지만, 커널의 강제 종료는 `SIGKILL`이므로 **아무런 기록도 남기지 않습니다.** exit 137이 발생해도 GC 로그는 정상 상태입니다
- **동일한 코드라도 플래그 하나가 종료 방식을 결정합니다.** `MaxDirectMemorySize`가 컨테이너 한도 아래면 JVM이 예외를 발생시키고(exit 1), 한도 위로 설정하면 커널이 강제 종료합니다(exit 137, GC 로그 0건). 두 경우 모두 **힙 사용률은 1%** 였습니다
- 따라서 운영 전략은 명확합니다. **상한이 없는 힙 외부 영역에 명시적인 상한을 설정하여, 커널이 강제 종료하기 전에 런타임이 예외를 발생시키도록 해야 합니다.** 단, **상한의 총합이 컨테이너 한도보다 작아야** 의미가 있습니다
- **`reserved`는 물리 메모리가 아닙니다.** 유휴 상태 JVM의 reserved 1.65GiB 중 committed는 334MiB에 불과합니다. `ps`의 VSZ 수치에 과도하게 반응할 필요는 없으며, 실제로 주의해야 할 것은 committed 영역이 물리적으로 채워지는 부분입니다
- **NMT와 커널은 측정 범위가 다릅니다.** NMT는 JVM이 관리하는 메모리만 측정하며, C 라이브러리 할당과 ZGC의 `memfd` 힙은 이 범위 밖에 있어 21편에서 확인한 지표 불일치가 발생합니다
- 주요 원인: **Metaspace**(상한이 2⁶⁴−1로 사실상 무제한) · **코드 캐시**(예약 240MB) · **스레드 스택**(스레드당 2MiB) · **다이렉트 버퍼**(기본 상한이 `-Xmx`와 동일) · **glibc arena** · **GC 관련 자료구조**(유휴 상태에도 53MiB)
- **`MALLOC_ARENA_MAX=2`에 대한 일반적인 통념이 실측 결과와 다르게 나타났습니다.** 테스트한 워크로드에서는 이 설정을 적용한 쪽이 오히려 96~113MiB를 반환하지 못했습니다(기본 설정은 3MiB). thread arena는 `mmap` 구획을 통째로 반환할 수 있지만, arena를 2개로 제한하면 여러 스레드의 할당이 뒤섞여 이것이 불가능해지기 때문입니다. **arena 개수는 락 경합과 단편화 사이의 트레이드오프이며 일괄적인 해결책이 아닙니다.** `malloc_trim(0)` 호출로 즉시 1MiB까지 감소했으므로 **메모리 누수가 아니라 할당자의 보유 상태**였음을 확인했습니다
- **`GOMEMLIMIT`은 cgo 영역을 포함하지 않습니다.** `GOMEMLIMIT=700MiB`로 설정한 Go 프로세스가 C 힙 960MiB를 사용하다 exit 137로 종료되는 동안, **Go 힙은 0MiB, GC는 0회**였습니다. Java의 다이렉트 버퍼 문제와 동일한 구조입니다
- **런타임이 클수록 '정상 로그' 뒤에서 종료될 가능성이 큰 영역이 넓어집니다.** Rust는 이 영역이 좁고, Java는 넓으며, Go는 cgo를 사용하는 순간 Java에 가까워집니다
