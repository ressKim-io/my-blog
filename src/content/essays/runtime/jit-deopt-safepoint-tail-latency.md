> **시리즈 "커널과 런타임으로 깊이 파헤치는 Rust · Go · Java"의 8편**
> [7편](/essays/java-jit-c2-runtime-optimization)에서 C2가 MDO를 읽고 무엇을 하는지 다섯 가지로 봤습니다. 인라이닝하고, 다형 호출을 단형으로 좁히고, 루프를 벡터로 묶고, 객체를 통째로 지웠습니다
> 다섯 가지 전부 같은 성격을 공유합니다. 실측을 믿고 낙관적으로 베팅한다는 점입니다
> 이번 편은 그 베팅이 틀렸을 때 무슨 일이 벌어지는지부터 시작합니다. 그리고 이 웜업이라는 청구서를 실무에서 다스리는 법으로 2부를 매듭짓습니다

## 도박이 틀리면 무슨 일이 벌어질까

C2의 최적화는 전부 **낙관적 도박**입니다

"이 호출은 항상 `Cat`이다", "이 메서드는 오버라이드된 적이 없다"를 가정하고 인라인하고 역가상화합니다

그 가정이 깨지면 어떻게 될까요. 새 서브클래스가 클래스로더에 올라와 "오버라이드 없음" 가정이 무효화되거나, 타입 가드가 빗나가는 상황에서입니다

그 기계어는 그 순간 틀린 코드가 됩니다. 실행을 계속하면 안 됩니다

여기서 진짜 질문은 이것입니다. **이미 그 기계어를 실행하고 있던 스레드를 어떻게 인터프리터로 되돌리는가**

![Deoptimization rebuilds an interpreter frame from the scope descriptor](/diagrams/jit-deopt-safepoint-tail-latency-1.svg)

답은 그림 가운데의 보라색 상자에 있습니다

C2는 기계어를 만들 때 **scope descriptor**라는 디버그 정보를 함께 심어 둡니다. 지도라고 생각하면 됩니다

이 지도에는 "지금 이 기계어 지점은 원래 바이트코드 몇 번째 줄에 해당하고, 그 시점의 지역변수 `x`는 레지스터 `R8`에, `y`는 스택 `[SP+16]`에 들어 있다"가 적혀 있습니다

그림 왼쪽의 C2 프레임에 적힌 `x → R8`, `y → [SP+16]`이 그 매핑입니다

가정이 깨지는 순간(붉은 화살표), JVM은 이 지도를 읽습니다. 그리고 그대로 **인터프리터 프레임을 재구성**합니다

레지스터와 스택 슬롯에 흩어져 있던 값들을 인터프리터가 기대하는 자리에 옮겨 놓고, 바이트코드의 바로 그 지점부터 이어서 실행합니다

그림 오른쪽이 그 결과입니다. 스레드는 자기가 방금 기계어를 실행 중이었다는 사실조차 모른 채 인터프리터로 이어 달립니다

그다음 아래쪽 회색 띠의 뒤처리가 돕니다. 무효가 된 기계어를 폐기하고, MDO를 다시 수집하고, 언젠가 다시 컴파일합니다

컴파일된 코드가 "자기가 원래 바이트코드 어디에서 왔는지"를 지도로 들고 다니기 때문에, 언제든 안전하게 되짤 수 있습니다

**deoptimization(역최적화, JIT 컴파일러가 가정한 최적화 전제가 깨졌거나 상위 컴파일 코드가 확보되었을 때 기존 컴파일 코드를 폐기하고 하위 컴파일러나 인터프리터 모드로 되돌리는 안전장치)** 은 이렇게 C2가 마음 놓고 도박을 걸 수 있게 해 주는 **안전망**입니다. 되돌릴 수 없다면 애초에 그런 가정을 세우지 못했을 것입니다

동시에 그림 오른쪽 아래의 붉은 상자이기도 합니다. 그 순간에 걸린 요청은 **tail latency 스파이크**를 맞게 됩니다

특히 megamorphic 호출 지점에서 컴파일과 역최적화가 반복되는 **deopt storm**은 Java 성능 사고의 단골입니다. C2가 인라인을 시도하고, 새 타입이 와서 무효화되고, 다시 시도하고, 다시 무효화되는 순환입니다

## 세이프포인트 — GC가 아닌데도 생기는 멈춤

앞에서 "역최적화가 스파이크를 만든다"고 했습니다. 그런데 그것이 **어떻게 스레드를 멈추는지**는 아직 설명하지 않았습니다

이 조각이 세이프포인트입니다. Java tail latency를 이해하는 데 빠지면 안 되는데 자주 생략되는 메커니즘입니다

**safepoint(세이프포인트, JVM이 모든 애플리케이션 스레드를 안전하게 일시 정지시키고 힙과 스택의 정확한 메모리 참조 상태를 일관되게 파악할 수 있는 동기화 지점)** 는 JVM이 모든 스레드의 정확한 상태를 아는 순간입니다. 어떤 객체가 살아 있고, 각 스택 프레임의 어느 슬롯이 참조인지가 확정된 지점입니다

GC, 역최적화, 스레드 덤프, 클래스 재정의 같은 작업은 **모든 스레드가 세이프포인트에 모여 멈춰 있어야만** 안전하게 수행할 수 있습니다

힙에서 객체를 옮기는 도중에 어떤 스레드가 옛 주소를 레지스터에 들고 뛰어다니면 안 되기 때문입니다

단체 사진을 찍는 상황을 떠올려 보겠습니다. 사진사는 한 사람이라도 뛰어다니고 있으면 셔터를 누르지 못합니다. 모두가 동시에 멈춰야 비로소 찍을 수 있습니다. 세이프포인트도 똑같습니다. JVM이 안전하게 일할 수 있는 순간은 모든 스레드가 함께 멈춰 선 그 찰나뿐입니다

그렇다면 스레드들을 어떻게 모을 수 있을까요

JIT이 컴파일할 때 메서드 반환 지점과 루프의 백엣지마다 **poll(세이프포인트 폴, 스레드가 세이프포인트 진입 요청이 있는지 주기적으로 플래그를 확인하는 짧은 기계어 명령어)** 을 심어 둡니다. "지금 멈춰야 하나?" 플래그를 확인하는 짧은 기계어 코드입니다

각 스레드는 다음 폴에 도달하는 순간 멈춰 섭니다

![Time to safepoint is set by the slowest thread](/diagrams/jit-deopt-safepoint-tail-latency-2.svg)

그림의 보라색 세로 점선이 "세이프포인트를 요청한 순간"입니다. 여기서부터 각 스레드가 자기 폴에 도달할 때까지의 시간을 측정합니다

T1은 곧바로 폴을 만나 멈춥니다. T2도 몇 개 지나 멈춥니다

T3이 문제입니다. 노란 막대가 **긴 카운티드 루프**입니다

카운티드 루프란 `for (int i = 0; i < n; i++)`처럼 정수 인덱스로 도는, 메서드 호출이 없는 루프를 말합니다. JIT은 성능을 위해 이런 루프의 백엣지에서 세이프포인트 폴을 **지워 버립니다**

한 바퀴에 몇 사이클밖에 안 걸리는 루프에서 매번 플래그를 확인하는 것은 큰 오버헤드이기 때문입니다

그 결과가 그림 그대로입니다. T1과 T2는 이미 멈춰 서서 아무 일도 하지 못하는데, T3은 루프를 다 돌 때까지 계속 달립니다. 사진사가 T3 한 사람 때문에 계속 기다리는 셈입니다

요청 시점부터 마지막 스레드가 도착할 때까지의 시간을 **TTSP(Time-To-SafePoint, 런타임이 세이프포인트 일시 정지를 요청한 시점부터 마지막 애플리케이션 스레드가 폴링 지점에 도달하여 완전히 멈출 때까지 소요된 총 지연 시간)** 라고 합니다. 그림 아래의 화살표입니다

핵심은 이것입니다. TTSP는 **GC 일시정지 시간과 별개**입니다

GC 로그에 찍히는 pause가 5ms여도, TTSP가 200ms였다면 애플리케이션은 205ms 동안 멈춰 있었습니다. 그리고 대부분의 GC 대시보드는 앞의 5ms만 보여 줍니다

TTSP를 길게 만드는 전형적인 패턴이 몇 가지 있습니다

폴이 제거된 긴 카운티드 루프가 첫째입니다

`System.arraycopy`로 거대한 버퍼를 복사하는 중인 스레드도 마찬가지입니다. 복사가 끝날 때까지 폴을 만나지 못합니다

코어 수보다 스레드가 훨씬 많은 oversubscription(오버서브스크립션, 물리적 CPU 코어 수보다 훨씬 많은 수의 활성 스레드가 실행을 경쟁하여 스케줄링 지연이 발생하는 과구독 상태) 환경에서도 발생합니다. 멈춰야 할 스레드가 CPU를 할당받지 못해 도달 자체가 늦어집니다

여기서 이번 편의 정직한 결론 하나가 도출됩니다

**Java의 tail latency 스파이크는 GC만의 문제가 아닙니다.** 역최적화도, 세이프포인트 도달 지연도 전부 멈춤을 만듭니다

5편에서 본 Rust의 "p99가 p50에 거의 붙어 있는 결정론"과 대비되는, Java의 구조적 약점입니다

## 웜업을 다스리는 세 갈래

그래서 웜업을 어떻게 해결할 것인가. 선택지가 스펙트럼 위에 놓입니다

![Four points on one spectrum from AOT to JIT](/diagrams/jit-deopt-safepoint-tail-latency-3.svg)

그림의 가로축은 웜업 비용입니다. 왼쪽으로 갈수록 시작이 빠르고, 오른쪽으로 갈수록 느립니다

**웜업을 짧게 — C1에서 멈추기**

```bash
java -XX:TieredStopAtLevel=1 -jar app.jar
```

C2를 아예 포기하고 L1에서 멈춥니다. 웜업이 빠르고 peak가 낮습니다

어차피 C2의 긴 계단을 다 못 오르고 죽을 짧은 프로세스에는 이것이 이득입니다. CLI 도구, 짧은 배치 잡, 일부 서버리스 함수가 그렇습니다

**웜업을 앞당기기 — Project Leyden**

발상이 단순합니다. 매번 실행할 때마다 처음부터 배우지 말고, **훈련 실행(training run)에서 배운 것을 미리 구워 두자**

세 개의 JEP(JDK Enhancement Proposal, Java 개발 키트 기능 향상 제안서)가 차례로 이 기반을 마련했습니다

| JEP | JDK | 무엇을 |
| --- | --- | --- |
| 483 | 24 | AOT 클래스 로딩·링킹 — 클래스 로드와 링크를 미리 끝내 캐시에 저장 |
| 514 | 25 | AOT 캐시 원커맨드 — record와 assemble 2단계였던 것을 하나로 |
| 515 | 25 | AOT 메서드 프로파일 — MDO 프로파일을 미리 캐시에 저장 |

이 시리즈의 맥락에서 결정적인 것은 **JEP 515**입니다

앞에서 "MDO는 실행하면서 모으는 것"이라고 했습니다. JEP 515는 그 프로파일을 훈련 실행에서 미리 구워 둡니다

그러면 JVM이 시작하자마자 프로파일을 이미 손에 쥐고 있습니다. 핫 메서드가 어디인지 처음부터 알고, JIT이 즉시 기계어 생성을 시작할 수 있습니다

JEP 515 문서가 담은 예제에서, 프로파일 없는 AOT 캐시로 90ms 걸리던 프로그램이 프로파일을 넣자 73ms에 끝났습니다. 19% 개선이고, 캐시 크기는 250KB(약 2.5%) 늘었을 뿐입니다

캐시를 만드는 방법도 한 줄입니다

```bash
java -XX:AOTCacheOutput=app.aot -jar app.jar   # 훈련 실행
java -XX:AOTCache=app.aot -jar app.jar         # 실전 실행
```

여기서 짚어야 할 의미가 있습니다. **Java가 AOT 쪽으로 한 발 내디뎠습니다**

JIT과 AOT는 이제 이분법이 아닙니다. 다음 편에서 이 지형도를 제대로 그립니다

**웜업을 없애기 — GraalVM native-image**

아예 바이트코드를 빌드 타임에 기계어로 뽑아 버립니다. 실행될 수 있는 코드를 빌드 시점에 전부 안다고 가정하는 **closed-world(닫힌 세계, 애플리케이션 실행에 필요한 모든 클래스와 리플렉션 대상이 컴파일 시점에 완전히 확정되며 런타임 동적 로딩이 없다고 가정하는 정적 분석 모델) AOT**입니다

웜업이 0이고, 시동이 수 ms이며, RSS가 크게 줄어듭니다

대가로 **C2의 런타임 적응을 통째로 포기**합니다. MDO 기반 역가상화도, 프로파일 인라이닝도, 실행 CPU 감지도 없습니다

그것을 빌드 타임 프로파일(PGO)로 부분 보상합니다. 5편의 Rust PGO, 6편의 Go PGO와 정확히 같은 계열의 처방입니다

그림 아래쪽의 두 띠가 이 편의 뼈대입니다

왼쪽 절반(native-image, C1 정지)은 **빌드 이후 얼어붙습니다.** 오른쪽 절반(Leyden, 풀 티어)은 **실행하는 내내 다시 최적화합니다**

Leyden이 왼쪽에서 오른쪽으로 넘어와 있다는 점이 중요합니다. 시작은 AOT처럼 하고, 달리는 것은 JIT처럼 합니다

### 코드 캐시 — 세그먼트 구조를 알아야 고갈을 진단합니다

7편에서 JIT이 만든 기계어가 쌓이는 코드 캐시를 소개했습니다. 그런데 이 코드 캐시는 하나의 덩어리가 아닙니다

JDK 9의 JEP 197부터 세 개의 code heap(코드 힙, 용도와 수명에 따라 분리하여 관리되는 코드 캐시의 메모리 세그먼트 구역)으로 나뉩니다

| 구역 | 담기는 것 | 수명 | 기본 크기 |
| --- | --- | --- | --- |
| non-method | 인터프리터·컴파일러 버퍼 등 VM 내부 코드 | 영구 | 약 5MB |
| profiled | L2·L3의 프로파일된 코드 | 짧음 — 나중에 L4로 대체될 운명 | 약 122MB |
| non-profiled | L1·L4 코드 | 잠재적으로 영원히 | 약 122MB |

non-method 구역의 크기(`-XX:NonNMethodCodeHeapSize`)는 고정 상수가 아니라 컴파일러 스레드 수에 따라 산정되므로, 환경마다 조금씩 다릅니다

왜 나누었을까요

예전에는 안 쓰는 코드를 청소하는 sweeper가 캐시 전체를 훑어야 했습니다. 구역을 나누면 "절대 비우지 않는 non-method 구역"을 아예 건너뛸 수 있습니다

수명이 비슷한 코드끼리 모아 두니 스윕 시간이 짧아지고 단편화도 줄어듭니다

`-XX:ReservedCodeCacheSize`의 기본값이 240MB로 넉넉한 것도 이 세그먼트 구조와 티어드 컴파일을 함께 수용하기 위해서입니다. L3 코드와 L4 코드가 한동안 공존하기 때문입니다

세 구역이 이 240MB를 나눠 갖습니다. VM 내부 코드를 뺀 나머지를 profiled와 non-profiled가 대략 절반씩 사용합니다

여기에 조용한 함정이 있습니다

**코드 캐시가 고갈되면 JIT이 컴파일을 멈춥니다.** 그러면 새로 자주 불리게 된 메서드가 인터프리터에 눌러앉고, 처리량이 절벽처럼 떨어집니다

에러도 없고 예외도 없습니다. 로그 한 줄과 함께 그냥 느려질 뿐입니다

람다와 제네릭을 많이 쓰는 대형 애플리케이션에서 실제로 겪는 문제라, 감시 대상에 넣어 둘 만합니다

### 그 밖의 레버

- **인라이닝 튜닝** — `-XX:MaxInlineSize`(기본 35바이트)와 `-XX:FreqInlineSize`(기본 325바이트)로 인라인 후보의 바이트코드 크기 상한을 조정합니다. 6편의 Go 예산 80과 같은 성격의 손잡이인데, Java는 여기에 실측 빈도가 더해집니다
- **명시적 SIMD, Vector API** — JEP 508로, JDK 25 기준 **10번째 인큐베이터**입니다. 아직 preview 단계도 아니고 Project Valhalla의 값 클래스를 기다리는 중입니다. C2의 자동 SuperWord 위에서 개발자가 벡터 폭을 직접 제어하는 API입니다. 내부적으로는 FFM(Foreign Function & Memory, JNI 오버헤드 없이 JVM 외부의 네이티브 C/C++ 라이브러리 함수와 오프힙 메모리를 안전하고 효율적으로 호출·조작하는 표준 API)을 통해 네이티브 수학 라이브러리에 연결됩니다
- **관측** — JFR(JDK Flight Recorder, JVM 내부에 깊숙이 통합되어 극도로 낮은 런타임 오버헤드로 애플리케이션 및 가상 머신의 동작 이벤트를 기록하고 프로파일링하는 진단 도구)로 컴파일·역최적화·세이프포인트 이벤트를 뜨고, `-XX:+PrintCompilation`으로 티어 승격 로그를 봅니다. JITWatch는 인라인과 벡터화 결정을 시각화하고, `-Xlog:safepoint`는 TTSP를 남깁니다

## 세 언어 수렴 비교 — 프로파일을 언제 반영하는가

| 항목 | Java (7·8편) | Rust (5편) | Go (6편) |
| --- | --- | --- | --- |
| 기계어 생성 시점 | 실행 중 (JIT) | 빌드 타임 | 빌드 타임 |
| 프로파일이 사는 곳 | MDO — 실행 내내 기록 | PGO 프로파일 파일 — 빌드 시 소비 | `default.pgo` — 빌드 시 소비 |
| 역가상화 | 인라인 캐시 + 타입 가드, 실행 중 계속 | 단형화로 정적 제거, `dyn`은 vtable | PGO 역가상화, 빌드 1회 |
| 자동 벡터화 | 있음 (C2 SuperWord) | 있음 (LLVM 루프·SLP 벡터라이저) | 없음 (수동 어셈블리) |
| 탈출하지 않는 객체 | 스칼라 치환 — 객체가 증발 | 스택·힙을 정적으로 결정 | 이스케이프 분석, 정적 |
| 워크로드 변화 적응 | 자동 (MDO 재수집 → 재컴파일) | 재빌드해야 함 | 재빌드해야 함 |
| 시동 · 웜업 | 느림, 계단식 | 즉시 | 즉시 |
| tail latency | 스파이크 (역최적화·세이프포인트) | 안정 (p99≈p50) | 안정 |
| 메모리 | 큼 (코드 캐시·MDO·JIT 자료구조) | 작음 | 작음 |
| 데이터 레이아웃 | `Object[]`는 참조의 배열 | 값 타입이 연속 배치 | `[]struct`가 연속 배치 |
| 아키텍처 특화 | 실행 중인 CPU를 보고 그 아키텍처 기계어 | 빌드 시 `target-cpu` | 빌드 시 `GOARCH` |

표를 가로로 훑으면 Java가 대부분의 행에서 이기거나 지는 것이 아니라, **다른 시점에 결정한다**는 사실이 보입니다

관통하는 축은 "백엔드 품질"이 아닙니다. **프로파일을 언제 반영하는가**입니다

Rust와 Go는 프로파일을 빌드할 때 한 번 소비합니다. 그래서 예측 가능하고, 배포된 뒤에는 아무것도 배우지 못합니다

Java는 프로파일을 실행 내내 MDO에 모아 계속 반영합니다. 그래서 적응하고, 예측 불가능하고, 무겁습니다

5편에서 "AOT의 약점은 정적이라 재최적화를 못 하는 것이지, 프로파일을 못 보는 것이 아니다"라고 못 박아 두었습니다. 그 문장이 이 표에서 회수됩니다

Go에도 PGO는 있습니다. 없는 것은 **재최적화**입니다

그리고 이 이분법은 양쪽에서 무너지는 중입니다. Java는 Leyden으로 AOT 쪽에 다가서고, Rust와 Go와 GraalVM은 PGO로 프로파일 쪽에 다가섭니다

다음 편이 이 수렴의 지형도를 그립니다

### 덧 — 메모리 레이아웃이라는 숨은 축

지금까지의 이야기를 절반쯤 되돌려 놓는 반전이 하나 있습니다. 6편에서 예고하고 넘긴 내용입니다

Java에서 `Point[]`는 `Point`가 일반 클래스라면 **참조의 배열**입니다

실제 `Point` 객체들은 각자 헤더(마크워드와 클래스 포인터, 대략 12~16바이트)를 달고 힙 여기저기에 흩어져 있고, 배열은 그것들을 가리키는 포인터만 담습니다

순회하면 포인터를 따라 힙 곳곳으로 점프합니다(pointer chasing, 메모리상에서 객체 포인터 참조를 연쇄적으로 추적하여 힙 곳곳으로 무작위 접근을 반복하는 패턴). 하드웨어 프리패처가 무력화되고 캐시 미스가 잦아집니다

Go의 `[]struct` 연속 배치와 정반대입니다. 이것은 Java가 "모든 객체에 정체성과 헤더와 간접 참조를 부여한" 설계의 대가입니다

단, `int[]`나 `long[]` 같은 원시 타입 배열은 Java도 연속입니다. 문제는 **객체**의 배열입니다

이 격차를 좁히려는 것이 **Project Valhalla의 값 클래스(JEP 401)** 입니다. heap flattening(힙 평탄화, 객체 포인터 참조 대신 객체 내부 필드 데이터 자체를 상위 배열이나 구조체 메모리 블록 내부에 인라인하여 연속 배치하는 기법)으로 객체를 배열 안에 직접 박고, JIT 스칼라화로 할당 자체를 없애려 합니다

여기서 흔한 오해를 하나 정정해야 합니다. JEP 401은 **JDK 25에 프리뷰로 들어간 적이 없습니다**

JDK 25 시점까지 JEP 401은 메인라인 밖에 있었고, Valhalla 조기 접근(early-access) 빌드로만 시험할 수 있었습니다

Valhalla 팀은 JDK 28 메인라인에 프리뷰로 통합하는 것을 목표로 작업 중입니다. 다만 이 글을 쓰는 2026년 7월 현재 JEP 401의 상태는 여전히 `Submitted`이고, 특정 릴리스에 공식 타겟되지는 않았습니다

프리뷰를 벗어나는 시점은 더더욱 정해진 바 없습니다

그러니 이 시리즈의 기준 시점(JDK 25 LTS)에서 그 격차는 여전히 실재합니다

이 사실이 SuperWord 이야기를 절반쯤 되돌립니다

수치 커널이 메모리 바운드라면, 즉 병목이 계산이 아니라 데이터를 기다리는 데 있다면, 벡터화 우위가 캐시 미스로 상쇄될 수 있습니다

SIMD로 한 번에 여덟 개를 계산해 봤자 그 여덟 개가 캐시에 없으면 소용없기 때문입니다. 6편에서 Go의 값 의미론을 두고 했던 이야기의 정확한 뒷면입니다

## 트레이드오프 소결

Java JIT의 계약을 한 문장으로 줄이면 다음과 같습니다

> 가장 느린 시동과 가장 큰 메모리를 지불하는 대신, 실행 실측(MDO)에 근거한 최적화로 정적 컴파일이 원리적으로 놓치는 것을 잡는다

**얻는 것** — 충분히 오래 달리는 CPU 바운드 서버에서 peak throughput이 정적 언어를 따라잡거나 앞섭니다

다형 호출을 실측으로 단형화하고, 스칼라 루프를 벡터화하고, 탈출하지 않는 객체를 통째로 소거하고, 트래픽이 변하면 다시 적응합니다

**잃는 것** — 웜업(콜드 스타트 손해), tail latency(역최적화와 세이프포인트 스파이크), 메모리(코드 캐시와 MDO)

이 셋은 여전히 정적 언어에 열세입니다

그래서 4편에 심어 둔 반전, "가장 무거운 JVM이 왜 롱런 정점 처리량에서 앞서나"의 답은 이것입니다

**MDO라는 실행 실측을 C2가 기계어에 직접 녹여 넣기 때문입니다.** 무거움의 정체가 곧 앞서는 이유입니다. 코드 캐시도 MDO도 JIT 컴파일 스레드도, 전부 "실행 중에 배우기 위한 장비"입니다

다만 이 역전은 **무조건이 아니라 조건부**입니다

장수명 프로세스여야 하고, CPU 바운드여야 하고, 지표가 peak throughput이어야 합니다. tail latency나 메모리나 콜드 스타트로 저울을 바꾸면 결론이 뒤집힙니다

그 조건을 정밀하게 규정하는 것이 다음 편의 일입니다

## DevOps 관점에서 더 파고들 질문

1. **역최적화와 세이프포인트 폭풍 탐지** — JFR의 `Deoptimization` 이벤트와 `-Xlog:safepoint`의 TTSP를 상시 관측하려면 무엇이 필요할까요? megamorphic 호출, 긴 카운티드 루프, 대형 `arraycopy` 같은 유발 패턴을 코드 리뷰나 정적 분석에서 선제적으로 잡아낼 방법은 무엇일까요?
2. **코드 캐시 세그먼트 모니터링** — `ReservedCodeCacheSize` 고갈은 에러 없이 처리량만 떨어지는 조용한 절벽입니다. JMX `MemoryPoolMXBean`으로 profiled와 non-profiled 세그먼트를 따로 보고, 고갈 징후(컴파일 중단 로그, 처리량 급락)를 대시보드와 알림에 엮는 설계는 어떻게 될까요?
3. **Leyden AOT 캐시의 CI 파이프라인** — JEP 515의 프로파일을 구우려면 대표성 있는 훈련 실행이 필요합니다. 프로덕션 트래픽을 재현하는 training run을 CI에서 어떻게 구성하고, 코드가 바뀔 때 캐시를 무효화하고 재생성할까요? 훈련 프로파일이 실트래픽과 어긋나면 어떤 위험이 생길까요?
4. **티어 정책과 프로세스 수명 매칭** — 짧은 배치잡과 상시 API 서버는 최적 티어 정책이 다릅니다. `TieredStopAtLevel=1`(빠른 웜업), 풀 티어(높은 peak), Leyden, native-image(웜업 0) 중 무엇을 고를지 프로세스 수명과 트래픽 패턴으로 판단하는 의사결정 트리를 어떻게 세울까요?

## 핵심 요약

- 역최적화는 scope descriptor라는 지도로 이루어집니다. 컴파일된 코드가 "원래 바이트코드 어디에서 왔고 지역변수가 어느 레지스터에 있는지"를 들고 다니기 때문에 언제든 인터프리터 프레임을 재구성할 수 있습니다. C2가 도박을 걸 수 있게 해 주는 안전망이자, tail latency 스파이크의 근원입니다
- 세이프포인트와 TTSP는 GC와 별개의 멈춤입니다. JIT이 긴 카운티드 루프에서 세이프포인트 폴을 지워 버리면, 그 스레드가 루프를 다 돌 때까지 이미 멈춘 다른 스레드 전부가 기다립니다. GC 로그의 pause 시간만 봐서는 이 멈춤이 보이지 않습니다
- 웜업을 다스리는 세 갈래는 C1 정지(`TieredStopAtLevel=1`), Leyden AOT 캐시(JEP 483·514·515), native-image입니다. Leyden의 JEP 515는 MDO 프로파일을 훈련 실행에서 미리 구워, JVM이 시작하자마자 핫 메서드를 알고 출발하게 합니다
- 코드 캐시는 JDK 9의 JEP 197부터 non-method·profiled·non-profiled 세 구역입니다. 고갈되면 JIT이 멈추고 처리량이 절벽처럼 떨어지는데, 에러가 나지 않아 조용합니다
- 세 언어를 가로지르는 축은 "누가 코드를 더 잘 짜는가"가 아니라 "프로파일을 언제 반영하는가"입니다. Rust·Go는 빌드 시 1회, Java는 실행 내내입니다
- 4편의 복선에 대한 답은 "MDO라는 실행 실측을 C2가 기계어에 녹여 넣기 때문"입니다. 무거움의 정체가 곧 앞서는 이유입니다. 단, 이 역전은 장수명·CPU 바운드·peak throughput이라는 조건 아래에서만 성립합니다
