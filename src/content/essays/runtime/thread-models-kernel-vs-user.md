## 접속 10만 개, 스레드 10만 개?

동시 접속 10만 개를 처리하는 서버를 떠올려 보겠습니다. 접속이 들어올 때마다 스레드를 하나씩 새로 만들면 시스템에는 어떤 일이 벌어질까요?

이 단순해 보이는 질문이 지난 20년간 서버 아키텍처의 흐름을 갈라놓았습니다. 답을 찾으려면 먼저 스레드 하나를 만들고 굴리는 데 실제로 무엇이 얼마나 드는지, 운영체제가 청구하는 원가표부터 확인해야 합니다

## 커널 스레드의 실체 — 1:1 모델

"스레드"라는 단어는 사용자 영역과 커널 영역이라는 두 층위에서 섞여 쓰이기 때문에 자주 혼란을 줍니다. 이 글에서 **커널 스레드**라고 부르는 대상은 커널 스케줄러가 직접 인식하고 물리 코어(CPU)에 배정하는 실행 단위를 뜻합니다

리눅스 커널은 내부에서 프로세스와 스레드를 엄격하게 구분하지 않습니다. 둘 다 커널 작업 제어 구조체(`task_struct`)라는 동일한 자료구조로 표현되며, 운영체제 스케줄러가 CPU 시간을 나누어 배분하는 기준 단위가 바로 이 구조체입니다

**1:1 모델**은 프로그래밍 언어가 만드는 애플리케이션 스레드 1개를 운영체제의 커널 스레드 1개, 즉 `task_struct` 1개에 그대로 일대일 매핑하는 방식입니다. Rust의 `std::thread`와 Java의 전통적인 플랫폼 스레드가 이 방식에 해당합니다

병원 진료 체계에 비유하면 이해가 빠릅니다. 환자(작업)가 접수하면 병원이 그 환자 전담으로 의사 한 명(커널 스레드)을 즉시 배정하는 방식입니다. 담당 의사가 정해져 있으니 진료 흐름은 단순하고 예측하기 쉽습니다. 문제는 의사 한 명을 고용하고 유지하는 데 드는 비용이 결코 작지 않다는 데 있습니다

![One-to-one threading model — every app thread is one kernel task with its own bill](/diagrams/thread-models-kernel-vs-user-1.svg)

그림처럼 애플리케이션 영역의 스레드 셋은 커널 작업 제어 구조체 셋으로 매핑됩니다. 커널 스케줄러는 이 개체들을 물리 코어에 시간 조각 단위로 번갈아 가며 배정합니다

이 스케줄링 알고리즘은 리눅스 커널 6.6 버전부터 CFS(Completely Fair Scheduler, 완전 공정 스케줄러)에서 대기 지연 시간을 더욱 균등하게 분배하는 EEVDF(Earliest Eligible Virtual Deadline First) 스케줄러로 진화했습니다

이러한 1:1 구조에서 "OS 스레드가 무겁다"는 말은 막연한 인상이 아닙니다. 의사 한 명을 배정할 때마다 병원이 실제로 지불하는 세 가지 청구서로 구체화됩니다

**첫 번째 청구서는 메모리 원가, 그중에서도 호출 스택입니다.** 모든 스레드는 함수를 호출할 때마다 지역 변수와 복귀 주소를 쌓아 둘 자기만의 호출 스택(Call Stack) 메모리가 필요합니다. 진료 기록을 담을 개인 캐비닛을 의사 한 명마다 미리 확보해 두는 셈입니다. 리눅스 `pthread`의 기본 가상 스택 공간은 8MB 예약(실제 물리 메모리는 사용한 만큼만 상주)입니다

Rust의 `std::thread`는 기본 2MB를 예약하며, Java 플랫폼 스레드는 `-Xss` 옵션 기준으로 수백 KB에서 1MB를 기본으로 차지합니다. 세 수치 모두 스레드 하나당 고정 비용이라는 점이 핵심입니다. 스레드 수가 늘어나면 이 스택 용량도 정확히 그 배수만큼 선형으로 누적됩니다

**두 번째 청구서는 컨텍스트 스위치(Context Switch, 문맥 교환) 비용입니다.** 물리 코어에서 실행되던 스레드를 A에서 B로 교체하는 작업으로, 진료실 안의 의사를 통째로 바꿔치기하는 것에 해당합니다. 커널 모드 진입과 레지스터 상태 저장·복원, 스케줄러 연산 자체에만 약 1~2μs의 직접 비용이 듭니다

여기에 더해 간접 비용이 따라붙습니다. CPU 캐시와 TLB(Translation Lookaside Buffer, 변환 색인 버퍼), 분기 예측기(Branch Predictor)가 오염되면서 연산 효율이 급감하는 구조는 1편에서 다룬 시스템 콜 복귀 직후의 상황과 정확히 일치합니다. 의사를 바꿀 때마다 새 의사가 이전 환자의 차트를 처음부터 다시 훑어야 하는 것과 같습니다

**세 번째 청구서는 생성과 소멸 비용입니다.** OS 스레드 하나를 생성하는 것은 `clone()` 시스템 콜을 호출해 가상 메모리에 스택을 매핑하고 커널 내부에 `task_struct`를 할당하는 무거운 작업입니다. 사용자 요청마다 매번 새 의사를 채용하고 해고하는 셈이니, 이 비용은 감당할 수 없이 누적됩니다. 그래서 실무에서는 스레드를 미리 만들어 두고 재사용하는 스레드 풀(Thread Pool)이 사실상 필수로 쓰입니다

> **ARM 각주** — 문맥 교환 시 CPU에 저장하고 복원해야 할 레지스터 수는 아키텍처마다 다릅니다. AArch64는 범용 레지스터 31개에 SIMD(단일 명령 다중 데이터)/FP(부동소수점) 레지스터 32개를 더하고, x86-64는 범용 16개에 벡터 레지스터 16개(AVX-512 사용 시 32개)를 처리합니다. ARM 계열이 저장해야 할 상태가 약간 더 크지만 전체 문맥 교환 비용에서는 부차적이며, 1:1 모델 구조 자체가 무겁다는 본질은 두 아키텍처가 동일합니다

## 1:1의 벽 — C10K 문제

1999년, 동시 접속 1만 개를 하나의 서버에서 어떻게 감당할 것인가라는 화두에 C10K 문제(Concurrent 10,000 Connections, 1만 동시 접속 처리 한계)라는 이름이 붙었습니다. 접속마다 OS 스레드 하나를 배정하는 1:1 설계, 즉 환자마다 전담 의사를 새로 고용하는 병원은 여기서 두 겹의 벽에 차례로 부딪힙니다

**첫 번째 벽은 순수한 물리 메모리 한계입니다.** 앞서 확인한 스택 예약 비용을 접속 수만큼 곱해 보면 그 크기가 바로 드러납니다

| 접속 수 | 필요 스택 (스레드당 1MB 가정) | 결과 |
| --- | --- | --- |
| 1,000 | 약 1GB | 빠듯함 |
| 10,000 | 약 10GB | 메모리 폭발 |
| 100,000 | 약 100GB | 불가능 |

표가 보여 주듯 실제 비즈니스 로직 연산은 시작도 하기 전에, 의사들을 위한 캐비닛(스택 공간)을 확보하는 것만으로 병원(서버)의 창고가 가득 찹니다

**두 번째 벽은 커널 스케줄링 과부하입니다.** 이 벽이 세워지는 과정은 순서대로 따라가면 이해하기 쉽습니다

물리 코어 수는 서버를 아무리 키워도 보통 수십 개 수준에서 고정됩니다. 반면 접속이 늘수록 실행 후보가 되는 스레드 수는 수만 개까지 치솟습니다. 실행할 후보는 폭증하는데 실제로 일할 좌석(코어)은 그대로이니, 커널 스케줄러는 그 좁은 좌석에 후보들을 욱여넣기 위해 컨텍스트 스위치를 훨씬 더 자주 돌려야 합니다

문제는 컨텍스트 스위치 자체가 공짜가 아니라는 점입니다. 앞서 확인했듯 스위치 한 번마다 직접 비용과 간접 비용이 함께 청구됩니다. 스위치 횟수가 폭증하면 이 청구서도 같이 폭증합니다

그 결과 CPU는 실제 비즈니스 연산보다 스레드를 갈아 끼우고 스케줄링을 계산하는 데 대부분의 시간을 써 버리는 상태, 즉 스래싱(Thrashing, 시스템 과부하로 연산보다 자원 교체 비용이 압도하는 상태)에 빠집니다. 유효 연산 시간이 스위치 오버헤드에 잠식당하는 것입니다

더 큰 문제는 네트워크 서버에서 일어나는 동작의 실제 성격입니다. 웹 서버에 존재하는 스레드 대다수는 끊임없이 계산을 수행하는 것이 아니라, 데이터가 도착하기를 기다리는 I/O 대기(Blocking) 상태로 멈춰 있습니다

병원 비유로 보면, 대기실 의사 대부분이 실제 진료가 아니라 검사 결과지가 도착하기만을 멍하니 기다리고 있는 셈입니다. 소켓 하나에서 데이터가 오기를 기다리자고 1MB의 스택 공간과 커널 스케줄링 슬롯을 통째로 붙들어 두는 것은 명백한 낭비입니다. 이 근본적인 비효율을 해결하기 위해 등장한 개념이 바로 유저 레벨 스케줄링입니다

## 유저 레벨 스케줄링 — M:N 모델

이 구조를 지탱하는 핵심 발상은 한 줄로 요약됩니다. **동시성을 표현하는 단위를 무거운 커널 스케줄링 단위에서 완전히 분리하는 것입니다**

다시 병원 비유로 돌아가면, 환자마다 전담 의사를 새로 고용하는 대신 접수처에 숙련된 간호사를 두는 방식으로 바뀝니다. 간호사는 대기 중인 환자(태스크)들을 순서대로 관리하다가, 소수의 의사(OS 스레드)가 비는 즉시 다음 환자를 배정합니다. 의사 수는 늘리지 않은 채로도 훨씬 많은 환자를 동시에 받을 수 있습니다

이 비유를 기술 용어로 옮기면 다음과 같습니다. 메모리 비용이 매우 저렴한 유저 레벨 태스크(Go의 고루틴, Java의 가상 스레드, Rust의 비동기 태스크)를 수십만 개에서 수백만 개까지 자유롭게 생성합니다. 그리고 언어 런타임이 소수의 OS 스레드 위에서 이 태스크들을 직접 스케줄링해 다룹니다. 이 런타임이 바로 접수처의 간호사 역할입니다

만약 특정 태스크가 네트워크 입출력 등으로 I/O를 기다리게 되면, OS 스레드를 통째로 멈추는 대신 해당 태스크만 대기 상태로 재워 둡니다(Park). 그리고 OS 스레드는 즉시 대기열에 있는 다른 유저 태스크를 올려 실행을 이어 갑니다. 의사가 검사 결과를 기다리는 동안 그 의사를 놀리지 않고, 간호사가 곧바로 다음 환자를 진료실로 들여보내는 것과 같습니다

![M-to-N model — a user-level scheduler sits between cheap tasks and few OS threads](/diagrams/thread-models-kernel-vs-user-2.svg)

그림에서 가장 중요한 부분은 중간에 위치한 유저 레벨 스케줄러 층입니다. 이 스케줄러가 수많은 경량 태스크와 소수의 OS 스레드 사이에서, 커널이 전혀 모르는 채로 효율적인 배차 업무를 전담합니다

운영체제 커널은 여전히 자신이 할당한 소수의 OS 스레드만 인식합니다. 수백만 개에 달하는 유저 태스크의 존재는 오직 언어 런타임 내부에서만 관리됩니다. 병원 원무과(커널) 서류에는 의사 명단만 올라가 있을 뿐, 대기실에 몇 명이 앉아 있는지는 기록되지 않는 셈입니다

따라서 유저 태스크 간의 교체 작업에는 무거운 커널 모드 진입이나 하드웨어 레지스터를 통째로 비우는 OS 문맥 교환 비용이 발생하지 않습니다. 간호사가 환자를 바꿔 앉히는 데는 병원 시스템 전체를 재부팅할 필요가 없는 것과 같은 이치입니다

## 두 개의 축으로 M:N 모델 나누기

같은 M:N 모델이라도 언어 런타임마다 세부 구현과 설계 철학이 크게 다릅니다. 앞으로 3편에서 Go·Java·Rust 각각의 구현을 뜯어보기 전에, 그 차이를 미리 가늠할 수 있는 두 가지 질문을 세워 두겠습니다

**첫 번째 축은 선점형(Preemptive)인가, 협력형(Cooperative)인가입니다.** 런타임이 CPU를 오래 독점하는 태스크를 강제로 멈추고 제어권을 뺏을 수 있는지, 아니면 태스크가 스스로 양보 지점(`yield`, `.await`)을 지나며 제어권을 반납해야 하는지를 가르는 축입니다

간호사가 강제로 진료실 문을 열고 다음 환자를 들여보낼 수 있다면 선점형이고, 의사가 스스로 "이 환자는 여기까지"라고 알려 줘야만 다음 환자가 들어갈 수 있다면 협력형입니다. 협력형 런타임에서 만약 어떤 태스크가 양보 지점 없이 무거운 연산을 계속 붙들고 있으면, 같은 OS 스레드에 배정된 다른 태스크들은 그 태스크가 끝날 때까지 꼼짝없이 기다려야 합니다

**두 번째 축은 투명한가, 색칠되는가(Function Coloring)입니다.** 기존의 일반적인 블로킹 코드를 수정 없이 그대로 작성할 수 있는지, 아니면 비동기를 위해 `async/await` 키워드로 함수 생태계를 별도로 구분해야 하는지를 가르는 축입니다

색칠이 없는 런타임에서는 개발자가 평소 쓰던 방식으로 코드를 작성해도 런타임이 알아서 블로킹 지점을 감지해 처리합니다. 반대로 색칠이 있는 런타임에서는 함수 하나가 "동기 함수"와 "비동기 함수"라는 서로 다른 두 세계 중 한쪽에 속하게 되고, 두 세계를 섞어 쓰려면 명시적인 변환이 필요합니다

![Where each runtime sits on two axes — preemption style and code style](/diagrams/thread-models-kernel-vs-user-3.svg)

이 좌표계는 3편에서 실제 세 언어의 구현을 확인할 때 계속 참조할 지도입니다. 왼쪽 위로 갈수록 런타임이 더 많은 일을 대신 처리해 주는 대신 런타임 자체의 풋프린트가 무거워집니다. 오른쪽 아래로 갈수록 런타임은 가벼워지고 투명해지지만, 개발자가 명시적으로 관리해야 할 책임이 늘어납니다

## 이론에서 구현으로

지금까지 세운 것은 설계도입니다. OS 스레드 하나가 왜 무거운지 세 가지 청구서로 확인했고, 그 무게가 C10K라는 벽 앞에서 왜 무너지는지 봤고, 그 자리를 대신할 M:N 모델의 뼈대와 두 가지 분류 축을 세웠습니다

이 설계도 위에 실제로 무엇을 지을지는 언어마다 다른 답을 내놓습니다. 같은 문제, 같은 재료(OS 스레드·유저 태스크·스케줄러) 앞에서 Go는 선점형이면서 투명한 자리를, Java는 준선점형이면서 투명한 자리를, Rust는 협력형이면서 색칠되는 자리를 택했습니다. 왜 그 자리를 택했는지, 그리고 그 선택이 실무에서 어떤 결과를 낳는지는 3편에서 이어집니다

## 실무 운영 관점에서 짚어볼 질문

1. **컨텍스트 스위치 계측** — 리눅스 CLI에서 `pidstat -w`로 자발적(I/O 대기) 및 비자발적(타임슬라이스 소진) 문맥 교환 수치를, `perf sched` 명령으로 스케줄링 폭풍 지점을 정밀 계측할 수 있습니다. 스레드 풀 크기를 물리 코어 대비 과하게 설정해 비자발적 문맥 교환이 치솟고 있다면, 적정 스레드 풀 크기는 어떻게 산출해야 할까요?
2. **워크로드 판별** — 지금 운영 중인 서비스가 스레드 대부분을 I/O 대기로 소모하는 유형인지, 아니면 CPU 연산으로 실제 코어를 쉼 없이 쓰는 유형인지에 따라 1:1과 M:N 중 유리한 쪽이 갈립니다. `pidstat`이나 `top`으로 관찰한 스레드들이 R(실행) 상태와 D/S(대기) 상태 중 어느 쪽에 더 오래 머무는지 직접 확인해 보셨나요?

## 핵심 요약

- 리눅스 커널 관점에서 프로세스와 스레드는 동일한 `task_struct` 제어 블록 하나로 표현됩니다. 1:1 모델은 여기에 스택 공간 예약(수백 KB~8MB), 문맥 교환 비용(직접 비용 1~2μs 및 CPU 캐시 오염), 생성 비용이라는 세 가지 청구서를 반드시 동반합니다
- C10K 문제의 핵심은 메모리와 CPU의 구조적 낭비입니다. 네트워크 서버 스레드 대다수는 I/O 대기 상태로 멈춰 있으면서도 물리 스택 메모리와 커널 스케줄링 슬롯을 통째로 점유합니다
- M:N 모델은 동시성을 표현하는 유저 단위와 실제 실행되는 커널 스케줄링 단위를 완전히 분리합니다. 무거운 커널 모드 진입 없이 유저 레벨 스케줄러가 태스크 교체를 전담합니다
- 선점형/협력형, 투명/색칠이라는 두 축이 언어별 M:N 구현의 차이를 미리 가늠하는 지도가 됩니다. 이 지도 위에 각 언어가 실제로 무엇을 지었는지는 3편에서 확인합니다
