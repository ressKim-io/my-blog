## 접속 10만 개, 스레드 10만 개?

동시 접속 10만 개를 처리하는 서버를 떠올려 보겠습니다. 접속이 들어올 때마다 스레드를 하나씩 새로 만들면 시스템에는 어떤 일이 벌어질까요?

이 단순해 보이는 질문이 지난 20년간 서버 아키텍처의 흐름을 갈라놓았습니다. 그리고 Go, Java, Rust는 각자의 설계 철학에 따라 완전히 다른 답을 내놓게 되었습니다

세 언어가 선택한 답을 공정하게 비교하려면 먼저 운영체제가 청구하는 원가표부터 확인해야 합니다. 스레드 하나를 만들고 굴리는 데 실제로 무엇을 얼마나 소모하게 되는지부터 짚어보겠습니다

## 커널 스레드의 실체 — 1:1 모델

우리가 흔히 말하는 "스레드"라는 단어는 사용자 영역과 커널 영역이라는 두 층위에서 섞여 쓰이기 때문에 혼란을 줍니다. 이 글에서 커널 스레드라고 부르는 대상은 커널 스케줄러가 직접 인식하고 물리 코어(CPU)에 배정하는 실행 단위를 뜻합니다

리눅스 커널은 내부에 프로세스와 스레드를 엄격하게 구분하지 않습니다. 둘 다 커널 작업 제어 구조체(`task_struct`)라는 동일한 자료구조로 표현되며, 운영체제 스케줄러가 CPU 시간을 나누어 배분하는 기준 단위가 바로 이 구조체입니다

1:1 모델은 프로그래밍 언어가 만드는 애플리케이션 스레드 1개를 운영체제의 커널 스레드 1개, 즉 `task_struct` 1개에 그대로 일대일 매핑하는 방식입니다. Rust의 `std::thread`와 Java의 전통적인 플랫폼 스레드가 이 방식에 해당합니다

![One-to-one threading model — every app thread is one kernel task with its own bill](/diagrams/thread-models-kernel-vs-user-1.svg)

그림처럼 애플리케이션 영역의 스레드 셋은 커널 작업 제어 구조체 셋으로 매핑됩니다. 커널 스케줄러는 이 개체들을 물리 코어에 시간 조각 단위로 번갈아 가며 배정합니다

이 스케줄링 알고리즘은 리눅스 커널 6.6 버전부터 CFS(Completely Fair Scheduler, 완전 공정 스케줄러)에서 대기 지연 시간을 더욱 균등하게 분배하는 EEVDF(Earliest Eligible Virtual Deadline First) 스케줄러로 진화했습니다

이러한 1:1 구조에서 "OS 스레드가 무겁다"는 말은 막연한 느낌이 아니라 다음 세 가지 구체적인 물리적 청구서로 증명됩니다

1. **메모리 원가(호출 스택)** — 모든 스레드는 함수를 호출할 때마다 지역 변수와 복귀 주소를 쌓아 둘 자기만의 호출 스택(Call Stack) 메모리가 필요합니다. 리눅스 `pthread`의 기본 가상 스택 공간은 8MB 예약(실제 물리 메모리는 사용한 만큼 상주)이며, Rust의 `std::thread`는 기본 2MB, Java 플랫폼 스레드는 `-Xss` 옵션 기준으로 수백 KB에서 1MB를 기본 차지합니다. 스레드 수가 늘어나면 이 스택 용량도 선형으로 누적됩니다
2. **컨텍스트 스위치(Context Switch, 문맥 교환) 비용** — 물리 코어에서 실행되던 스레드를 A에서 B로 교체하는 작업입니다. 커널 모드 진입과 레지스터 상태 저장 및 복원, 스케줄러 연산 자체에만 약 1~2μs의 직접 비용이 소모됩니다. 여기에 더해 CPU 캐시와 TLB(Translation Lookaside Buffer, 변환 색인 버퍼), 분기 예측기(Branch Predictor)가 오염되면서 연산 효율이 급감하는 간접 비용이 따라붙는 구조는 1편에서 다룬 시스템 콜과 정확히 일치합니다
3. **생성 및 소멸 비용** — OS 스레드 하나를 생성하는 것은 `clone()` 시스템 콜을 호출하여 가상 메모리에 스택을 매핑하고 커널 내부에 `task_struct`를 할당하는 무거운 작업입니다. 사용자 요청마다 매번 스레드를 만들면 이 비용이 감당할 수 없이 누적되므로 이를 완화하기 위해 스레드 풀(Thread Pool)이 필수적으로 쓰입니다

> **ARM 각주** — 문맥 교환 시 CPU에 저장하고 복원해야 할 레지스터 수는 아키텍처마다 다릅니다. AArch64는 범용 레지스터 31개에 SIMD(단일 명령 다중 데이터)/FP(부동소수점) 레지스터 32개를 더하고, x86-64는 범용 16개에 벡터 레지스터 16개(AVX-512 사용 시 32개)를 처리합니다. ARM 계열이 저장해야 할 상태가 약간 더 크지만 전체 문맥 교환 비용에서는 부차적이며 1:1 모델 구조 자체가 무겁다는 본질은 두 아키텍처가 동일합니다

## 1:1의 벽 — C10K 문제

1999년, 동시 접속 1만 개를 하나의 서버에서 어떻게 감당할 것인가라는 화두에 C10K 문제(Concurrent 10,000 Connections, 1만 동시 접속 처리 한계)라는 이름이 붙었습니다. 접속마다 OS 스레드 하나를 할당하는 1:1 설계는 여기서 치명적인 두 겹의 벽에 부딪히게 됩니다

| 접속 수 | 필요 스택 (스레드당 1MB 가정) | 결과 |
| --- | --- | --- |
| 1,000 | 약 1GB | 빠듯함 |
| 10,000 | 약 10GB | 메모리 폭발 |
| 100,000 | 약 100GB | 불가능 |

첫 번째 벽은 위 표에 나타난 물리 메모리 한계입니다. 실제 비즈니스 로직 연산을 시작하기도 전에 스택 공간 예약만으로 서버의 메모리가 바닥납니다

두 번째 벽은 커널 스케줄링 과부하입니다. 실행 가능한 스레드 수가 물리 코어 개수를 크게 초과하면 커널 스케줄러가 수만 개의 스레드를 짧은 주기로 돌려야 하므로 문맥 교환이 폭증합니다

결국 CPU가 실제 비즈니스 연산보다 스레드 교체와 스케줄링 계산에 대부분의 시간을 소진하는 스래싱(Thrashing, 시스템 과부하로 연산보다 자원 교체 비용이 압도하는 상태)에 빠지게 됩니다

더 큰 문제는 네트워크 서버에서 일어나는 동작의 실제 성격입니다. 웹 서버에 존재하는 스레드의 대다수는 끊임없이 계산을 수행하는 것이 아니라 데이터가 도착하기를 기다리는 I/O 대기(Blocking) 상태로 멈춰 있습니다

소켓 하나에서 데이터가 오기를 기다리자고 1MB의 스택 공간과 커널 스케줄링 슬롯을 통째로 낭비하는 셈입니다. 이 근본적인 비효율을 해결하기 위해 등장한 개념이 바로 유저 레벨 스케줄링입니다

## 유저 레벨 스케줄링 — M:N 모델

이 구조를 지탱하는 핵심 발상은 한 줄로 요약됩니다. **동시성을 표현하는 단위를 무거운 커널 스케줄링 단위에서 완전히 분리하는 것입니다**

메모리 비용이 매우 저렴한 유저 레벨 태스크(Go의 고루틴, Java의 가상 스레드, Rust의 비동기 태스크)를 수십만 개에서 수백만 개까지 자유롭게 생성합니다. 그리고 언어 런타임이 소수의 OS 스레드 위에서 이 태스크들을 직접 스케줄링하여 다룹니다

만약 특정 태스크가 네트워크 입출력 등으로 I/O를 기다리게 되면 OS 스레드를 통째로 멈추는 대신 해당 태스크만 대기 상태로 재워 둡니다(Park). 그리고 OS 스레드는 즉시 대기열에 있는 다른 유저 태스크를 올려 실행을 이어갑니다

![M-to-N model — a user-level scheduler sits between cheap tasks and few OS threads](/diagrams/thread-models-kernel-vs-user-2.svg)

그림에서 가장 중요한 부분은 중간에 위치한 유저 레벨 스케줄러 층입니다. 이 스케줄러가 수많은 경량 태스크와 소수의 OS 스레드 사이에서 커널이 모르게 효율적인 배차 업무를 전담합니다

운영체제 커널은 여전히 자신이 할당한 소수의 OS 스레드만 인식합니다. 수백만 개에 달하는 유저 태스크의 존재는 오직 언어 런타임 내부에서만 관리됩니다

따라서 유저 태스크 간의 교체 작업에는 무거운 커널 모드 진입이나 하드웨어 레지스터를 통째로 비우는 OS 문맥 교환 비용이 발생하지 않습니다

같은 M:N 모델이라도 언어 런타임마다 세부 구현과 설계 철학이 크게 다릅니다. 다음 두 가지 질문을 기준으로 분류하면 언어별 차이가 선명하게 드러납니다

- **선점형(Preemptive)인가, 협력형(Cooperative)인가** — 런타임이 CPU를 오래 독점하는 태스크를 강제로 멈추고 제어권을 뺏을 수 있는지, 아니면 태스크가 스스로 양보 지점(`yield`, `.await`)을 지나며 제어권을 반납해야 하는지
- **투명한가, 색칠되는가(Function Coloring)** — 기존의 일반적인 블로킹 코드를 수정 없이 그대로 작성할 수 있는지, 아니면 비동기를 위해 `async/await` 키워드로 함수 생태계를 별도로 구분해야 하는지

![Where each runtime sits on two axes — preemption style and code style](/diagrams/thread-models-kernel-vs-user-3.svg)

이 좌표계 위에서 Go는 선점형이면서 투명한 위치에 서 있고, Java 가상 스레드는 준선점형이면서 투명한 영역에 자리합니다. 반면 Rust의 tokio 런타임은 협력형이면서 함수 색칠이 요구되는 위치를 택했습니다

왼쪽 위로 갈수록 런타임이 많은 복잡한 일을 대신 처리해 주는 대신 런타임 자체의 풋프린트가 무거워집니다. 반대로 오른쪽 아래로 갈수록 런타임은 가벼워지고 투명해지지만 개발자가 명시적으로 관리해야 할 책임이 늘어납니다

이 위치 선정이 곧 각 언어가 추구하는 철학입니다. 이제 세 언어가 왜 이 지점에 자리 잡았는지 구체적인 구현 체계를 살펴보겠습니다

## Go의 답 — GMP 스케줄러

Go는 정교한 M:N 스케줄러를 언어 런타임 내부에 기본 탑재했습니다. 개발자는 복잡한 비동기 문법 대신 `go f()` 단 한 줄만으로 고루틴을 생성하며 기존과 동일한 방식으로 동기화/블로킹 코드를 작성합니다

이 강력한 마법을 실현하는 중심 엔진이 바로 GMP 스케줄러입니다

- **G(Goroutine, 고루틴)** — 사용자 애플리케이션의 경량 실행 단위입니다. 최초 2KB의 작은 스택으로 시작하며 메모리가 더 필요해지면 런타임이 더 큰 스택을 할당하여 통째로 복사하는 방식으로 성장합니다
- **M(Machine, OS 스레드)** — 운영체제가 인식하는 실제 커널 스레드입니다. 물리 코어에서 기계어를 실행하는 실체입니다
- **P(Processor, 논리 프로세서)** — 고루틴(`G`)을 OS 스레드(`M`)에 태워 실행할 수 있는 논리적 권리이자 로컬 실행 대기열(Local Run Queue)을 소유하는 객체입니다. 생성되는 개수는 `GOMAXPROCS` 환경 변수(기본값은 호스트의 사용 가능 코어 수)에 의해 결정됩니다

스케줄러가 지키는 핵심 규칙은 하나입니다. 고루틴(`G`)이 물리 CPU에서 실행되려면 논리 프로세서(`P`)를 획득해야 하며 이 `P`가 OS 스레드(`M`)와 결합되어 있어야 합니다

따라서 어떤 순간에 실제로 동시에 연산을 수행하는 고루틴의 최대 개수는 `P`의 개수, 즉 `GOMAXPROCS` 설정값과 정확히 일치합니다

![Go GMP — per-P run queues, work stealing, global queue and the netpoller](/diagrams/thread-models-kernel-vs-user-4.svg)

그림에서 눈여겨볼 부분은 각 논리 프로세서 `P`가 독립적인 로컬 실행 대기열을 갖고 있다는 점입니다. 대부분의 스케줄링이 전역 락(Lock) 없이 각자의 로컬 대기열 안에서 빠르게 처리되므로 연산 오버헤드가 극히 낮습니다

로컬 대기열에 미처 담지 못한 고루틴은 전역 실행 대기열(Global Run Queue)로 이동하며 각 `P`가 주기적으로 이를 확인해 분배받습니다. 또한 네트워크 I/O를 기다리는 고루틴들은 별도의 네트워크 폴러(Netpoller) 장치에 잠들어 대기합니다

### 네 가지 핵심 메커니즘

**작업 훔치기(Work Stealing)** — 특정 `P`의 로컬 대기열이 비어 할 일이 없어지면 다른 `P`를 무작위로 선택해 대기열에 있는 고루틴의 절반을 훔쳐옵니다. 이 분산적이고 무작위적인 훔치기 메커니즘이 다중 코어 전체에 부하를 고르게 분산시킵니다

**시스템 콜 핸드오프(Syscall Handoff)** — 1편에서 설명한 "런타임이 커널 관문을 감싼다"는 개념의 실제 작동 방식입니다. 고루틴이 파일 읽기나 네트워크 외 커널 대기가 필요한 블로킹 시스템 콜을 호출하면 런타임의 `entersyscall` 훅이 즉시 개입합니다. 이때 커널에 묶이게 될 OS 스레드(`M`)로부터 논리 프로세서(`P`)를 분리하여 대기 중인 다른 물리 OS 스레드에 붙여 줍니다

![Syscall handoff — the P detaches from a blocked M and keeps scheduling on another M](/diagrams/thread-models-kernel-vs-user-5.svg)

그림의 왼쪽처럼 고루틴 `G1`이 시스템 콜로 커널 모드에 진입하면 `M0` 스레드도 커널 대기열에 묶이게 됩니다. 하지만 오른쪽을 보면 `P`가 묶여 버린 `M0`을 미련 없이 버리고 새로운 `M`으로 옮겨 가면서, 대기열에 있던 `G2`와 `G3`가 중단 없이 물리 코어에서 실행을 이어갑니다

시스템 콜 처리가 끝나고 돌아온 `G1`은 런타임의 `exitsyscall` 훅에서 다시 비어 있는 `P`를 획득해야만 실행을 재개할 수 있습니다. 스레드 하나가 블로킹되어도 전체 런타임이 멈추지 않고 매끄럽게 돌아가게 만드는 핵심 설계입니다

**네트워크 폴러(Netpoller)** — 소켓 통신을 포함한 네트워크 I/O는 훨씬 더 효율적인 샛길을 이용합니다. 리눅스의 `epoll`(macOS의 `kqueue`, Windows의 `IOCP`)은 수천 개의 소켓 이벤트를 커널 내부에서 감시하다가 준비된 소켓만 알려주는 비동기 통지 메커니즘입니다. Go 런타임은 네트워크 소켓을 전부 논블로킹(Non-blocking) 모드로 생성한 뒤 이를 네트워크 폴러에 등록해 감시합니다

`conn.Read()`로 소켓에서 데이터가 오길 기다리는 고루틴은 무거운 OS 스레드를 멈추지 않고 즉시 네트워크 폴러 대기 상태로 재워집니다(Park). 수백만 개의 소켓 커넥션이 대부분 잠든 상태로 단 몇 개의 OS 스레드만 효율적으로 공유하는 구조이며, 이것이 Go가 I/O 중심 네트워크 서버에서 탁월한 성능을 내는 원동력입니다

**비동기 선점(Asynchronous Preemption, Go 1.14+)** — 초기 Go 런타임은 오직 함수 호출 경계에서만 스케줄러가 개입할 수 있었습니다. 그래서 `for {}` 같은 무한 루프나 계산 중심 루프가 돌면 고루틴이 CPU를 독점해 다른 고루틴들이 굶주리는 현상이 발생했습니다. Go 1.14부터는 런타임이 10ms 이상 CPU를 독점한 `M` 스레드에 유닉스 `SIGURG` 시그널을 발송하고, 시그널 핸들러가 해당 고루틴의 실행 레지스터를 강제로 힙에 저장한 뒤 로컬 대기열로 강제 내리는 비동기 선점을 구현했습니다

이 메커니즘 덕분에 Go의 M:N 스케줄러는 완벽한 선점형(Preemptive) 런타임으로 완성되었습니다. 앞서 본 좌표계에서 Go가 가장 윗줄에 위치하는 이유입니다

### GOMAXPROCS와 컨테이너 환경의 주의점

```go
// 고루틴 10만 개 — 1:1 OS 스레드였다면 스택만 수십 GB를 소모합니다
for i := 0; i < 100_000; i++ {
    go func(id int) {
        resp, _ := http.Get(url) // 블로킹 함수처럼 보이지만
        _ = resp                 // 내부적으로 netpoller에 Park되면서 OS 스레드(M) 반환
    }(i)
}
```

이토록 훌륭한 `GOMAXPROCS` 설정에도 오랜 시간 운영 엔지니어를 괴롭힌 함정이 존재했습니다. Go 1.24 버전까지 런타임은 쿠버네티스 파드(Pod)에 설정된 컨테이너 CPU 제한(Limit)을 인식하지 못하고 호스트 노드의 전체 물리 코어 수를 그대로 읽었습니다

예를 들어 64코어 물리 노드에서 `resources.limits.cpu: "2"`로 2코어만 쓰도록 제한한 파드를 띄우면 Go 런타임은 논리 프로세서 `P`를 64개나 만들어 버립니다. 64개의 `P`가 동시에 연산을 시도하다가 cgroup(Linux Control Group, 컨테이너 리소스 격리 메커니즘)에 할당된 CFS 쿼터를 단숨에 소진해 버리면 남은 스케줄링 주기 동안 파드 전체가 멈춰 서는 CPU 스로틀링(Throttling) 벌칙을 받게 됩니다

이것이 컨테이너 환경에서 간헐적으로 응답 지연이 수백 ms까지 치솟는 꼬리 지연(Tail Latency, 상위 99% 구간의 응답 지연) 문제의 대표적인 원인이었습니다

Go 1.25 버전부터는 런타임 기본값으로 이 문제를 마침내 해결했습니다. 커널의 cgroup CPU 제한 설정(`cpu.cfs_quota_us` 및 cgroup v2의 `cpu.max`)을 런타임이 직접 읽어 `GOMAXPROCS`를 파드 리밋에 맞춰 자동으로 설정하며 약 30초마다 설정 변화를 감지해 동적으로 재조정합니다

이 자동 조절 기능을 끄고 싶다면 환경 변수 `GODEBUG=containermaxprocs=0`을 지정하면 됩니다. 단 런타임은 CPU 제한(`limit`)만 참고하며 최소 보장량(`request`)은 무시한다는 점은 유의해야 합니다

> **DevOps 함의** — Go 1.24 이하 버전을 쿠버네티스 클러스터에 배포한다면 Uber의 `automaxprocs` 라이브러리를 임포트하여 `GOMAXPROCS`를 수동 보정하는 것이 필수 정석이었습니다. Go 1.25부터는 언어 런타임이 알아서 처리해 줍니다. "여유로운 2코어 파드가 왜 간헐적으로 스로틀링에 걸릴까?"라는 의문의 정답이 바로 여기에 있습니다

## Java의 답 — 플랫폼 스레드에서 가상 스레드로

Java는 이제 두 가지 스레드 모델을 동시에 지원합니다. 오랜 시간 Java 생태계를 지배해 온 "JVM 스레드는 OS 스레드와 무조건 1:1"이라는 상식이 깨진 전환점입니다

기존의 전통적인 플랫폼 스레드(Platform Thread)는 여전히 OS 스레드와 일대일로 매핑됩니다. `new Thread()`나 기존의 `ThreadPoolExecutor`가 만드는 스레드이며 스택 하나당 수백 KB에서 1MB를 점유하는 무거운 물리적 비용도 그대로입니다

따라서 플랫폼 스레드 기반의 thread-per-request(요청당 1스레드) 모델로 구성된 전통적인 Java 서버는 수백에서 수천 개의 동시 요청이 물리적 한계였습니다. 이 한계를 극복하기 위해 WebFlux 같은 리액티브(Reactive) 스택이 광범위하게 도입되었으나, 콜백 중첩과 비동기 연산자 체인으로 인해 코드 읽기와 디버깅이 난해해지는 비용을 치러야 했습니다

가상 스레드(Virtual Thread, JDK 21 정식 출시)는 완전히 다른 길을 제시합니다. 무거운 OS 스레드 스택을 사용하는 대신 실행 중단 상태를 힙(Heap) 메모리에 가볍게 저장하는 초경량 유저 레벨 스레드입니다

여기서 중심이 되는 개념이 바로 컨티뉴에이션(Continuation, 실행을 멈춘 지점의 호출 스택과 레지스터 상태를 담은 힙 객체)입니다. 특정 시점까지 실행된 함수의 호출 스택을 그대로 복사해 JVM 힙 메모리에 일반 객체처럼 저장해 둔 것이라고 이해하면 직관적입니다

![Virtual thread lifecycle — mount on a carrier, unmount to the heap on blocking I/O, remount later](/diagrams/thread-models-kernel-vs-user-6.svg)

그림에 나타난 생애주기의 순환이 가상 스레드 작동 방식의 전부입니다. 가상 스레드가 실제로 물리 CPU에서 실행되려면 먼저 운송반 역할을 하는 캐리어 스레드(Carrier Thread, 내부적으로 코어 수만큼 생성된 `ForkJoinPool`의 플랫폼 스레드)에 올라타야(Mount) 합니다

실행 도중 네트워크 소켓 대기나 데이터베이스 조회 같은 블로킹 I/O를 만나면 JVM이 사전에 계측해 둔 지점에서 컨티뉴에이션 객체로 스택 상태를 힙에 떠 놓고 캐리어 스레드에서 즉시 내립니다(Unmount). 해방된 캐리어 스레드는 대기열에서 기다리는 다른 가상 스레드를 태워 쉼 없이 연산을 이어갑니다

I/O 응답이 도착해 연산을 재개할 때가 되면 가상 스레드는 비어 있는 임의의 캐리어 스레드에 다시 올라타(Remount) 중단되었던 지점부터 실행을 계속합니다. 초기 메모리 풋프린트가 단 수백 바이트에서 수 KB 수준에 불과하므로 단일 JVM 내에서 수백만 개의 스레드를 자유롭게 생성할 수 있습니다

```java
// 직관적인 thread-per-request 스타일 코드가 초경량으로 되살아납니다
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            var body = client.send(req, ofString()); // 블로킹 호출 → 컨티뉴에이션 힙 저장 후 캐리어에서 Unmount
            return body.body();
        });
    }
}
```

개발자 입장에서는 WebFlux처럼 복잡한 리액티브 코드를 짤 필요 없이 익숙하고 동기적인 블로킹 코드를 그대로 작성합니다. Go가 추구했던 "투명한 동시성 처리" 철학과 정확히 수렴하는 결실입니다

### 피닝(Pinning) 현상과 JEP 491의 해결

가상 스레드 도입 초기 가장 치명적인 아킬레스건은 피닝(Pinning, 고정) 문제였습니다. 가상 스레드가 블로킹 I/O를 만났음에도 정상적으로 내리지(Unmount) 못하고 물리 캐리어 스레드를 꽉 쥐어 잡고 있는 현상입니다

캐리어 스레드가 고정되어 묶여 버리는 만큼 M:N 스케줄링의 이점이 사라지며 극단적인 경우 대기 중인 다른 가상 스레드들이 굶주리다 서버 데드락으로 이어집니다

JDK 21에서 JDK 23까지 이 피닝을 유발했던 가장 큰 주범은 바로 Java 기본 동기화 키워드인 `synchronized` 블록이었습니다. `synchronized`는 객체마다 부여되는 모니터 락(Monitor Lock, JVM 내부 동기화 잠금)으로 임계 구역을 보호하는데, 당시 JVM은 이 락의 소유권을 가상 스레드가 아닌 하부 물리 캐리어 스레드 기준으로 추적했습니다

이 때문에 `synchronized` 블록 내부에서 데이터베이스 호출 등의 블로킹이 발생하면 JVM 안전성 문제로 Unmount가 차단되었습니다. 수많은 오픈소스 레거시 라이브러리가 `synchronized`로 보호되고 있었기 때문에 가상 스레드 전환 시 기대했던 성능 이점이 무력화되곤 했습니다

JDK 24에 적용된 JEP 491(Java Enhancement Proposal 491, 모니터 락 아키텍처 개편안)이 마침내 이 오래된 난제를 완전히 풀어냈습니다. 모니터 락 소유권과 내부 상태 추적을 가상 스레드 단위로 완벽하게 분리하도록 JVM 커널을 재작성하여 이제 `synchronized` 블록 내부에서 블로킹이 일어나도 정상적으로 Unmount되어 캐리어를 해방합니다

현재 장기 지원(LTS) 버전인 JDK 25에는 이 개선 사항이 기본 탑재되어 있습니다. 다만 JNI(Java Native Interface) 네이티브 C 메서드 호출 도중의 블로킹이나 클래스 초기화 시점(`static` 블록)의 블로킹 같은 예외적인 상황에서의 피닝은 여전히 주의해야 합니다

> **DevOps 함의** — 가상 스레드를 프로덕션에 도입할 때는 `-Djdk.tracePinnedThreads` JVM 옵션을 켜거나 JFR(JDK Flight Recorder, JVM 내장 성능 분석기)의 `jdk.VirtualThreadPinned` 이벤트를 모니터링하여 숨은 피닝 지점을 찾아내야 합니다. 만약 사용하는 Java 버전이 JDK 24 미만이라면 `synchronized` 락이 심각한 병목 지점이 되므로 라이브러리의 락을 `java.util.concurrent.ReentrantLock`으로 치환하거나 상위 JDK로의 업그레이드를 신중히 판단해야 합니다

### 컨테이너 리소스 인식은 Java가 훨씬 앞섰습니다

재미있는 역사적 사실이 하나 있습니다. JVM은 이미 JDK 10(2018년 정식 배포)부터 `UseContainerSupport` 플래그를 기본 활성화하여 리눅스 cgroup의 CPU 및 물리 메모리 제한을 스스로 정확히 파악해 왔습니다

`Runtime.availableProcessors()` 메서드가 파드의 CPU 제한 설정을 반영하여 값을 반환하므로 가상 스레드의 기반이 되는 `ForkJoinPool` 캐리어 스레드 개수와 가비지 컬렉터(GC) 스레드 수가 컨테이너 한계에 자동으로 맞춰집니다. Go 런타임이 2025년(1.25 버전)에야 기본 탑재한 컨테이너 자동 인식을 Java는 7년이나 일찍 갖추고 있었던 셈입니다

물론 JVM은 이 물리 메모리 인식 결과를 힙(`-Xmx`) 크기 자동 산정과 엮으면서 또 다른 형태의 컨테이너 OOMKilled 문제를 유발하기도 하는데, 이 흥미로운 트레이드오프는 5부에서 심도 있게 다룹니다

## Rust의 답 — 1:1 유지, 비동기는 라이브러리 선택

Rust는 다른 두 언어와 달리 무거운 M:N 스케줄러를 언어 내장 런타임에서 의도적으로 완전히 제거했습니다

1.0 버전 정식 출시 이전인 2014년까지는 Rust도 M:N 기반의 그린 스레드(`libgreen`)를 실험했습니다. 그러나 모든 컴파일된 바이너리에 무거운 유저 스케줄러와 메모리 관리 코드를 강제하는 것이 Rust의 핵심 철학인 제로코스트 추상화(Zero-Cost Abstraction, 쓰지 않는 기능에는 어떤 런타임 비용도 지불하지 않는다)와 정면으로 배치되어 과감히 삭제했습니다. 이 역사적 결정의 전모는 바로 다음 3편에서 낱낱이 해부합니다

그 결과 현재 Rust의 동시성 모델은 명확히 두 개의 층으로 분리되어 동작합니다

**`std::thread`** — 커널의 OS 스레드와 정확히 1:1로 일치하는 표준 스레드입니다. Java의 전통적인 플랫폼 스레드와 같은 무게(기본 2MB 스택)를 지니며 CPU 자원을 집약적으로 소모하는 병렬 수치 연산 작업에는 이 1:1 모델이 가장 이상적인 정답입니다(`rayon` 같은 고성능 데이터 병렬 라이브러리도 이 스레드 모델 위에서 작동합니다)

**`async`/`.await` 키워드** — 비동기 함수를 작성하면 언어 런타임이 개입하는 것이 아니라 컴파일러가 해당 함수를 상태 기계(State Machine, 실행 중단점마다의 변수 상태를 열거형 변수로 관리하는 기계어 변환 체계)로 재구성합니다. 이렇게 변환된 `Future`(비동기 연산의 최종 완료 상태를 표현하는 핵심 트레이트) 객체는 힙 메모리 할당이나 무거운 런타임을 절대 강제하지 않습니다

하지만 Rust 표준 라이브러리는 이 `Future`를 실제로 구동하고 배차해 줄 실행기(Executor)를 제공하지 않습니다. 개발자가 프로젝트 특성에 맞게 `tokio`, `async-std`, `smol` 같은 외부 비동기 런타임 라이브러리를 직접 선택해 조립해야 합니다

### tokio — 라이브러리로 구현한 M:N, 단 협력형 런타임

```rust
// tokio: 비동기 스케줄링은 협력형이므로 오직 .await 지점에서만 자발적으로 양보합니다
#[tokio::main]
async fn main() {
    let mut tasks = Vec::new();
    for _ in 0..100_000 {
        tasks.push(tokio::spawn(async {
            let resp = reqwest::get(URL).await.unwrap(); // .await 지점에서 스케줄러에게 양보
            resp.text().await
        }));
    }
}
```

Rust 생태계의 표준처럼 자리 잡은 `tokio`의 멀티스레드 런타임은 물리 코어 수만큼 생성한 워커 OS 스레드 풀 위에서 수많은 비동기 태스크를 작업 훔치기(Work Stealing) 방식으로 스케줄링합니다. 네트워크 및 파일 I/O는 `mio`라는 저수준 라이브러리를 거쳐 `epoll` 이벤트 루프로 처리합니다

이 기본 구조만 놓고 보면 Go의 GMP 스케줄러와 매우 닮아 있습니다. 하지만 실제 운영 측면에서는 두 가지 결정적인 차이점이 존재합니다

**함정 1: 함수 색칠 문제(Function Coloring)** — 비동기 함수(`async fn`)는 다른 비동기 함수(`.await`) 내부에서만 자연스럽게 호출됩니다. 이로 인해 코드베이스 전체가 동기(Sync) 세계와 비동기(Async) 세계로 명확히 단절되는 구조적 제약이 발생합니다

**함정 2: 협력형 스케줄링의 대가** — `tokio` 런타임에는 Go의 `SIGURG` 시그널 강제 선점이나 Java 가상 스레드의 자동 Unmount 같은 마법이 없습니다. 만약 어떤 태스크가 `.await` 지점 없이 CPU 연산을 오래 독점하거나 일반 블로킹 함수(`std::thread::sleep` 등)를 호출하면 해당 워커 스레드 전체가 즉시 멈춰 섭니다

```rust
// 비동기 태스크 내부에서 일반 블로킹 함수를 호출하면 워커 스레드 전체가 동결됩니다
tokio::spawn(async {
    std::thread::sleep(Duration::from_secs(1)); // .await가 아님 → 해당 워커 스레드의 스케줄러 정지
    // 무거운 연산이나 블로킹 호출이 필수라면 반드시 tokio::task::spawn_blocking으로 격리해야 합니다
});
```

런타임이 커널 블로킹이나 무거운 연산을 알아서 흡수해 주지 않으므로 개발자가 직접 `tokio::task::spawn_blocking` 같은 격리 스레드 풀로 작업을 떼어 내야 합니다. 편한 자동화를 포기한 대신 모든 시스템 제어권을 개발자에게 명시적으로 위임한 설계입니다

> **컨테이너 각주** — `tokio`가 기본적으로 생성할 워커 스레드 수는 하부의 `num_cpus` 크레이트(Crate, Rust 패키지)가 결정합니다. 이 크레이트와 Rust 표준 라이브러리의 `available_parallelism` API는 꾸준히 개선되어 현재는 리눅스 cgroup의 CPU 할당 쿼터를 정확히 읽어 옵니다. 따라서 최신 버전에서는 Go 1.24 이전과 같은 "호스트 전체 코어 수 오인" 문제가 기본적으로 해소되었습니다. 다만 어떤 계열의 라이브러리나 프레임워크가 코어 수를 산정하는지 프로젝트마다 다를 수 있으므로 컨테이너 배포 시에는 환경 변수 `TOKIO_WORKER_THREADS`를 명시하여 워커 스레드 수를 확정 짓는 것이 안전합니다

## 세 언어 수렴 비교

| 항목 | Rust (std) | Rust (tokio) | Go | Java (플랫폼) | Java (가상) |
| --- | --- | --- | --- | --- | --- |
| 모델 | 1:1 | M:N (협력형) | M:N (선점형) | 1:1 | M:N (준선점형) |
| 스케줄 주체 | 커널 스케줄러 | tokio 런타임 | Go 런타임 | 커널 스케줄러 | JVM (ForkJoinPool) |
| 태스크 무게 | 스택 약 2MB | 상태 기계(`Future`) | 스택 2KB (동적 성장) | 스택 약 0.5~1MB | 힙 컨티뉴에이션 |
| 선점 방식 | OS 타임슬라이스 | 없음 (`.await` 양보) | `SIGURG` 시그널 선점 | OS 타임슬라이스 | 블로킹·마운트 지점 |
| 코드 스타일 | 동기 블로킹 | 비동기 색칠 분리 | 투명한 동기 작성 | 동기 블로킹 | 투명한 동기 작성 |
| 블로킹 흡수 | 스레드 메모리 낭비 | `spawn_blocking` 수동 | `P` 핸드오프 자동 | 스레드 메모리 낭비 | 캐리어 Unmount 자동 |
| 컨테이너 인식 | cgroup 쿼터 반영 | `num_cpus` 쿼터 파싱 | Go 1.25부터 기본 | JDK 10부터 기본 | JDK 10부터 기본 |

표에서 Java 가상 스레드를 "준선점형"으로 표기한 명확한 이유가 있습니다. 가상 스레드는 물리 코어의 실행 시간(Time Slice)을 기준으로 강제 선점되는 것이 아니라 오직 I/O 블로킹이나 마운트 지점에서만 양보하므로 Go처럼 타이트 루프를 강제로 중단하는 완전 선점형과는 구별됩니다

이 비교표를 가로질러 관찰할 수 있는 거대한 수렴 현상이 이번 편의 핵심입니다. Go와 Java 가상 스레드는 출발점과 역사가 다름에도 결국 **"개발자에게는 투명한 기존 블로킹 코드를 제공하고, 하부 런타임이 I/O 대기 비용을 알아서 흡수한다"**는 동일한 궁극의 결론에 도달했습니다

오직 Rust만이 이 친절한 자동화를 거절했습니다. 제로코스트 추상화와 극도의 성능 예측 가능성을 수호하기 위해 비동기 함수 색칠과 수동 블로킹 격리라는 무거운 책임을 명시적으로 개발자에게 남겼습니다

결국 1편에서 내렸던 결론과 동일한 철학적 축 위로 귀결됩니다. 같은 기술적 난제를 두고 런타임이 모든 것을 책임질 것인가(Go, Java), 아니면 통제권을 개발자에게 온전히 넘길 것인가(Rust)의 선택입니다

## 트레이드오프 소결

1:1 모델은 구조가 단순하고 실행 경로가 명확하며 CPU 자원을 집약적으로 소모하는 고성능 병렬 연산 작업에 가장 훌륭한 선택지입니다. 하지만 대규모 동시 접속과 I/O 대기가 폭주하는 서버 환경에서는 메모리 폭발과 문맥 교환 과부하로 무너집니다

M:N 모델은 수십만 개의 I/O 대기 태스크를 극히 적은 물리적 비용으로 매끄럽게 흡수합니다. 대신 런타임 내부의 복잡도가 크게 증가하며 피닝(Pinning), 워커 스레드 블로킹, `GOMAXPROCS` 설정 오류 등 이전에는 없었던 새로운 형태의 장애 양식을 만들어 냅니다

그리고 어느 모델을 선택하든 서버가 가진 물리 코어 수라는 절대적 연산 천장은 변하지 않습니다. M:N 스케줄링은 데이터가 오기를 '기다리는' 수많은 태스크를 값싸게 다룰 뿐, 실제로 CPU를 써서 '계산하는' 태스크의 속도를 물리적으로 더 빠르거나 늘려 주지는 않습니다

## DevOps 관점에서 더 파고들 질문

1. **스로틀링 진단** — 컨테이너의 `/sys/fs/cgroup/cpu.stat` 파일에서 `nr_throttled`(스로틀링 횟수)와 `throttled_time`(누적 지연 시간)을 확인하면 CFS 스로틀링 여부를 즉시 포착할 수 있습니다. Go 1.24 이하 버전이나 워커 스레드 수가 과도하게 설정된 서비스가 CPU 제한에 걸려 간헐적인 꼬리 지연을 만들고 있지는 않나요?
2. **컨텍스트 스위치 계측** — 리눅스 CLI에서 `pidstat -w`로 자발적(I/O 대기) 및 비자발적(타임슬라이스 소진) 문맥 교환 수치를, `perf sched` 명령으로 스케줄링 폭풍 지점을 정밀 계측할 수 있습니다. 스레드 풀 크기를 물리 코어 대비 과하게 설정하여 비자발적 문맥 교환이 치솟고 있다면 적정 스레드 풀 크기는 어떻게 산출해야 할까요?
3. **가상 스레드 피닝 탐지** — JVM 옵션 `-Djdk.tracePinnedThreads`나 JFR의 `jdk.VirtualThreadPinned` 이벤트를 활성화하여 운영 중인 서비스의 숨은 피닝 지점을 식별할 수 있습니다. 만약 대상 시스템이 JDK 24 미만이라면 플랫폼 업그레이드 비용과 레거시 `synchronized`를 `ReentrantLock`으로 치환하는 리팩토링 비용 중 어느 쪽이 더 합리적일까요?
4. **가상 스레드 vs 리액티브 스택** — Java 가상 스레드(Project Loom) 출시 이후 WebFlux 같은 리액티브 프레임워크의 존재 이유가 상당 부분 희석되었습니다. 그럼에도 불구하고 백프레셔(Backpressure, 데이터 흐름 제어)나 정교한 데이터 스트리밍 조합이 핵심인 시스템에서는 여전히 리액티브 모델이 유리할까요?
5. **tokio 블로킹 격리 장애** — 비동기 핸들러 내부에서 무거운 암호화 해시 연산이나 파일 I/O를 `spawn_blocking` 없이 직접 수행하면 왜 서버 전체 처리량이 순식간에 반토막 날까요? 비동기 서버에서 별도 블로킹 스레드 풀 크기는 얼마로 잡아야 병목이 생기지 않을까요?
6. **최소 보장량(`request`)과 상한 제한(`limit`)의 불일치** — Go 1.25 런타임은 cgroup의 CPU 제한(`limit`)만 참고하며 쿠버네티스의 최소 보장량(`request`)은 완전히 무시합니다. 만약 `requests.cpu: "1"`과 `limits.cpu: "4"`로 설정된 파드에서 `GOMAXPROCS`는 4로 설정되는데, 이 고무줄 같은 간극이 노드의 빈패킹(5부에서 다룰 자원 밀집 배치) 전략과 충돌할 때 어떤 부작용이 발생할까요?

## 핵심 요약

- 리눅스 커널 관점에서 프로세스와 스레드는 동일한 `task_struct` 제어 블록 하나로 표현됩니다. 1:1 모델은 여기에 스택 공간 예약(수백 KB~8MB), 문맥 교환 비용(직접 비용 1~2μs 및 CPU 캐시 오염), 생성 비용이라는 세 가지 묵직한 청구서를 반드시 동반합니다
- C10K 문제의 핵심 본질은 메모리와 CPU의 구조적 낭비입니다. 네트워크 서버 스레드의 대다수는 I/O 대기 상태로 멈춰 있으면서도 물리 스택 메모리와 커널 스케줄링 슬롯을 통째로 점유합니다
- M:N 모델은 동시성을 표현하는 유저 단위와 실제 실행되는 커널 스케줄링 단위를 완전히 분리합니다. Go는 GMP 스케줄러(작업 훔치기, 시스템 콜 핸드오프, 네트워크 폴러, `SIGURG` 선점)로 이를 이뤄 냈고, Java는 가상 스레드(힙 컨티뉴에이션, Mount/Unmount, JEP 491 개편)로 구현해 냈습니다
- Rust는 제로코스트 추상화 철학을 지키기 위해 언어 내장 M:N 런타임을 1.0 이전에 과감히 포기했습니다. 외부 라이브러리인 `tokio`가 그 역할을 훌륭히 대신하지만 협력형 스케줄링 구조이므로 블로킹 작업을 격리하는 것은 전적으로 개발자의 의무입니다
- 컨테이너 CPU 제한 인식 기능은 Java(JDK 10)가 Go(1.25)보다 무려 7년 앞서 탑재했습니다. 컨테이너 CPU 리밋 설정과 언어 런타임의 논리 스레드(`GOMAXPROCS` 및 워커 수)가 어긋나면 CFS 스로틀링이 발생해 치명적인 꼬리 지연을 일으킵니다
- Go와 Java 가상 스레드는 "개발자에게 투명한 동기 블로킹 코드 제공과 런타임의 자동 I/O 흡수"라는 궁극의 해답으로 수렴했고, Rust는 극도의 제어력과 명시성을 선택했습니다. 1편의 결론과 마찬가지로 런타임이 무거운 책임을 질 것인가, 개발자가 제어권을 쥘 것인가의 선택입니다
