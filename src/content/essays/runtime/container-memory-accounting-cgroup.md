## 메모리 제한이 커널에 반영되는 방식

쿠버네티스 매니페스트의 `limits.memory: 512Mi` 설정이 리눅스에서 실제로 어떻게 반영되는지 확인합니다

### cgroup, 컨테이너 자원을 제한하는 커널 메커니즘

**cgroup(control group 혹은 컨트롤 그룹, 리눅스 커널이 CPU와 메모리 등 시스템 자원을 프로세스 그룹 단위로 계량하고 사용 상한을 강제하는 자원 통제 메커니즘)** 은 리눅스 커널이 프로세스 그룹의 자원 사용량을 계량하고 제한하는 기능입니다

컨테이너는 **네임스페이스(격리)** 와 **cgroup(제한)** 의 조합으로 구현됩니다. 도커나 쿠버네티스에서 메모리 한도를 설정하는 것은 결국 cgroup 파일에 값을 기록하는 동작입니다

현재 표준으로 쓰이는 **cgroup v2(unified hierarchy)** 환경에서, 512MiB 한도를 설정한 컨테이너 내부 상태를 확인합니다

```bash
docker run --rm --memory=512m alpine sh -c '
  cat /sys/fs/cgroup/cgroup.controllers
  echo "memory.max     = $(cat /sys/fs/cgroup/memory.max)"
  echo "memory.high    = $(cat /sys/fs/cgroup/memory.high)"
  echo "memory.current = $(cat /sys/fs/cgroup/memory.current)"
'
```

```text
cpuset cpu io memory hugetlb pids rdma misc
memory.max     = 536870912
memory.high    = max
memory.current = 1163264
```

`536870912`는 512MiB(512 × 1024²)와 정확히 일치합니다. 매니페스트의 `limits.memory` 설정이 **`memory.max`** 파일에 그대로 반영된 것을 확인할 수 있습니다

세 파일의 역할은 다음과 같습니다

**`memory.max`** 는 하드 한도입니다. 초과 시 프로세스가 강제 종료됩니다

**`memory.high`** 는 소프트 한도입니다. 초과 시 커널이 메모리 회수(reclaim)를 강하게 수행해 할당 속도를 늦추지만, 프로세스를 강제 종료하지는 않습니다. 쿠버네티스는 기본적으로 이 값을 설정하지 않으므로 위 출력처럼 `max`로 표시됩니다

**`memory.current`** 는 현재 커널이 이 cgroup에 집계한 총 사용량입니다. `kubectl top`이나 `docker stats`가 참조하는 수치입니다

## 커널의 메모리 측정 단위는 힙이 아니라 페이지입니다

애플리케이션 개발자는 흔히 메모리 사용량을 힙 크기와 동일시하지만, 커널의 계량 방식은 다릅니다

커널은 특정 언어의 '힙' 개념을 인지하지 않습니다. 커널은 메모리를 **page(페이지, 리눅스 커널이 메모리를 할당하고 관리하는 기본 가상 메모리 분할 단위로 보통 4KiB 크기를 가짐)** 단위로 관리하며, 프로세스 그룹이 사용하는 모든 유형의 페이지를 합산합니다

![The kernel charges every kind of page to one bill; the heap is only part of one column](/diagrams/container-memory-accounting-cgroup-1.svg)

그림 가운데의 큰 영역이 커널이 합산하는 메모리 총량, 즉 `memory.current`입니다

그 안에 여러 구성 요소가 있습니다. 가장 왼쪽의 **anon**(익명 페이지)이 애플리케이션의 힙과 스택에 해당합니다. 언어 런타임의 힙은 **이 anon 메모리의 일부일 뿐**입니다

나머지 구성 요소는 다음과 같습니다. **file**은 페이지 캐시로, 읽어 들인 파일과 메모리에 적재된 공유 라이브러리(`.so`)가 포함됩니다. **kernel**은 슬랩(커널 내부 자료구조용 할당자)·페이지 테이블·커널 스택입니다. **sock**은 소켓 송수신 버퍼이고, **shmem**은 공유 메모리 및 `/dev/shm`과 같은 tmpfs 영역입니다

런타임이 힙 크기를 200MB로 보고하더라도, 커널은 이 구성 요소 전부를 합산하여 관리합니다

### 컨테이너 내부 실측

단순한 알파인 셸 프로세스만 실행한 컨테이너의 `memory.stat`을 통해 실제 할당 내역을 확인합니다

```bash
docker run --rm --memory=512m alpine \
  grep -E '^(anon|file|kernel|kernel_stack|pagetables|percpu|sock|shmem|slab) ' \
  /sys/fs/cgroup/memory.stat
```

```text
anon 122880
file 0
kernel 1134592
kernel_stack 32768
pagetables 90112
percpu 2232
sock 0
shmem 0
slab 352384
```

`anon`은 122,880바이트, 즉 **120KiB**입니다. 일반적으로 '메모리 사용량'이라 부르는 값입니다

반면 `kernel` 관련 메모리는 1,134,592바이트, **1.08MiB**로 익명 메모리의 **9배**에 달합니다

셸 프로세스 하나만 실행한 환경임에도 페이지 테이블(88KiB)·커널 스택(32KiB)·슬랩(344KiB) 등 커널 구동에 필요한 메모리가 프로그램 힙보다 더 큽니다

실제 애플리케이션이 실행되면 `anon`의 비중이 가장 커집니다. 핵심은 각 비율이 아니라 **이 항목들이 모두 같은 한도 아래 합산된다는 사실**입니다. 힙 크기는 작은데 RSS가 큰 경우 가장 먼저 점검해야 할 파일이 `memory.stat`인 이유입니다

다만 완충 장치가 하나 있습니다. `file`(페이지 캐시) 중 **깨끗한(clean) 페이지**, 즉 디스크와 내용이 동일해 언제든 다시 읽어올 수 있는 페이지는 버려도 무방합니다. 한도에 근접하면 커널은 이 페이지를 먼저 회수합니다

반면 `anon`은 임의로 버릴 수 없습니다. 스왑이 없는 경우(컨테이너는 대부분 스왑을 비활성화합니다) **회수할 방법이 없으며**, 이는 OOM의 직접적인 원인이 됩니다

## 메모리 한도 초과 시 커널의 동작

`memory.max`를 초과하는 할당 요청이 발생하면 커널은 정해진 절차를 수행합니다

![Over the limit: reclaim first, then SIGKILL — exit 137](/diagrams/container-memory-accounting-cgroup-2.svg)

그림 상단은 `memory.current`가 `memory.max`를 초과하기 직전 상황을 나타냅니다

**첫 단계는 direct reclaim(직접 회수, 컨테이너 메모리 사용량이 한도에 도달했을 때 커널이 메모리 할당을 요청한 애플리케이션 스레드를 멈춰 세우고 동기적으로 페이지 캐시나 슬랩을 정리하여 빈 공간을 확보하는 동작)** 입니다. 커널은 깨끗한 페이지 캐시를 정리하고 회수 가능한 슬랩 메모리를 줄입니다. 충분한 공간이 확보되면 할당이 승인됩니다. 다만 이 회수 작업은 **할당을 요청한 스레드에서 동기적으로 수행**되므로, 그 지연이 애플리케이션의 응답 지연(latency tail)으로 이어집니다

**회수로도 부족하면 OOM killer(Out-Of-Memory killer, 시스템이나 컨테이너 가용 메모리가 고갈되어 직접 회수로도 메모리를 확보하지 못했을 때 커널이 프로세스를 강제 종료하여 메모리를 회수하는 커널 모듈)가 개입합니다.** cgroup OOM killer가 해당 cgroup 내 프로세스에 **SIGKILL(시그널 9 혹은 강제 종료 시그널, 프로세스가 가로채거나 무시할 수 없이 커널에 의해 즉각적으로 프로세스가 강제 종료되는 치명적 시그널)** 을 전송합니다. 이때 종료 코드는 **137**이 되는데, 시그널에 의해 종료된 프로세스의 종료 코드는 관례상 `128 + 시그널 번호`이기 때문입니다(128 + 9 = 137)

512MiB 컨테이너에서 1GiB 크기의 바이트 배열 할당을 시도해 재현합니다

```bash
docker run --memory=512m --name oomtest python:3-alpine \
  python3 -c "x = bytearray(1024*1024*1024); print('할당 성공')"

docker inspect oomtest --format 'ExitCode={{.State.ExitCode}} OOMKilled={{.State.OOMKilled}}'
```

```text
ExitCode=137  OOMKilled=true
```

"할당 성공" 메시지는 출력되지 않았으며, 파이썬의 `MemoryError` 예외도 발생하지 않았습니다

**SIGKILL은 애플리케이션에서 가로채거나(catch) 무시할 수 없습니다.** 런타임은 예외를 처리할 기회조차 얻지 못하고 즉시 종료됩니다. 이 특성은 5부 전체에서 반복되며, 3편 뒤에서 다룰 "GC 로그는 정상인데 파드가 종료된다" 문제의 배경이 됩니다

커널 측 기록은 `memory.events` 파일에 남습니다

```bash
docker run --rm --memory=256m python:3-alpine sh -c \
  'python3 -c "x = bytearray(512*1024*1024)"; cat /sys/fs/cgroup/memory.events'
```

```text
Killed
low 0
high 0
max 35
oom 1
oom_kill 1
oom_group_kill 0
```

`max 35`는 한도에 도달해 회수를 시도한 횟수이고, `oom_kill 1`은 실제로 프로세스를 강제 종료한 횟수입니다. OOMKilled 알람 발생 시 가장 먼저 확인해야 할 지표입니다

### 압축 가능한 자원과 그렇지 않은 자원

여기서 쿠버네티스 자원 관리의 중요한 전제 하나가 도출됩니다

CPU 한도를 초과하면 커널은 **throttling(스로틀링, CPU 사용량이 한도에 도달했을 때 프로세스를 강제 종료하는 대신 실행 속도를 인위적으로 제한하여 지연을 발생시키는 제어 방식)** 을 통해 처리 속도를 제한합니다. 프로세스는 유지되며, 속도만 느려집니다

메모리 한도를 초과하면 속도를 늦춰 대응할 방법이 없습니다. 그래서 프로세스를 **종료합니다**

메모리는 **incompressible(압축 불가능한 자원, CPU처럼 실행 속도를 늦춰 대응할 수 없어 한도 초과 시 메모리 확보를 위해 프로세스를 강제 종료할 수밖에 없는 자원 특성)** 입니다. **CPU 초과는 지연(slowdown), 메모리 초과는 종료(kill)** 로 이어집니다

매니페스트의 네 필드가 cgroup의 어떤 값으로 매핑되는지 정리하면 다음과 같습니다

| 쿠버네티스 필드 | 의미 | cgroup v2로 내려가는 곳 | 초과 시 |
|---|---|---|---|
| `requests.memory` | 스케줄링 보증 | (스케줄러 계산용, 직접 강제 아님) | — |
| `limits.memory` | 절대 상한 | `memory.max` | **OOMKilled** |
| `requests.cpu` | CPU 배분 가중치 | `cpu.weight` | (경합 시 비율 배분) |
| `limits.cpu` | CPU 상한 | `cpu.max` | **스로틀** |

이 표는 다음 편(빈패킹)의 출발점이 됩니다. **스케줄러는 `requests`를 기준으로 노드에 파드를 배치하고, 커널은 `limits`를 기준으로 종료 여부를 결정합니다.** 두 지표가 서로 다른 주체를 향한다는 사실 자체가 곧 비용으로 이어집니다

## 런타임은 컨테이너 한도를 어떻게 인식하는가

지금까지 커널의 메모리 회계 방식을 살펴봤습니다. 이제 런타임이 이 회계를 인식하는지를 확인합니다

### 런타임의 기본 전제와 그 한계

JVM과 Go 런타임은 기본적으로 단일 프로세스가 물리 서버 전체 자원을 독점한다는 전제로 설계되었습니다

이 때문에 가용 메모리를 확인할 때는 `/proc/meminfo`를, CPU 코어 수를 확인할 때는 `sched_getaffinity`를 참조했습니다. 문제는 이 값들이 **컨테이너 내부에서도 호스트(노드)의 수치**를 그대로 반환한다는 점입니다

노드가 64GiB·32코어 사양이면, 512MiB·0.5코어로 제한된 파드 내부의 런타임도 64GiB 메모리와 32코어를 사용할 수 있다고 인식합니다

그 결과는 두 가지로 나타납니다. 메모리 측면에서는 힙을 노드 크기 기준으로 과대 산정해 한도를 곧바로 초과하고, CPU 측면에서는 GC 스레드와 스레드풀을 32개 단위로 생성해 0.5코어 안에서 과도한 컨텍스트 스위칭을 유발합니다

**container awareness(컨테이너 인식, 런타임이 호스트 서버의 전체 메모리나 CPU가 아니라 cgroup 파일에 설정된 컨테이너의 제한치를 읽어와 가용 자원을 보정하는 기능)** 은 런타임이 `/proc/meminfo` 대신 **cgroup에 설정된 한도**를 읽도록 보정하는 기능입니다

### Java(JVM) — CPU와 메모리 한도를 자동으로 감지

HotSpot JVM은 `-XX:+UseContainerSupport` 옵션이 기본 활성화되어 있어, 부팅 시 cgroup을 감지해 CPU와 메모리 한도를 모두 인식합니다

512MiB·2코어로 제한된 컨테이너에서 감지 로그를 확인합니다

```bash
docker run --rm --memory=512m --cpus=2 eclipse-temurin:25-jdk \
  java -Xlog:os+container=trace -version
```

```text
[os,container] Detected cgroups v2 unified hierarchy
[os,container] Path to /cpu.max is /sys/fs/cgroup/cpu.max
[os,container] OSContainer::active_processor_count: 2
[os,container] Path to /memory.max is /sys/fs/cgroup/memory.max
[os,container] Memory Limit is: 536870912
```

JVM은 `memory.max`를 직접 읽어 **536870912**바이트를 가용 메모리로 인식했습니다. 또한 `cpu.max`를 기준으로 실효 코어 수를 2개로 계산했습니다

JVM은 이 값을 기준으로 힙 상한을 결정합니다. 같은 컨테이너에서 확인한 결과입니다

```bash
docker run --rm --memory=512m --cpus=2 eclipse-temurin:25-jdk \
  java -XX:+PrintFlagsFinal -version | grep MaxHeapSize
```

```text
size_t MaxHeapSize = 134217728  {product} {ergonomic}
```

134,217,728바이트, 즉 **128MiB**입니다. 한도인 512MiB의 정확히 25%에 해당합니다

`{ergonomic}` 표시는 사용자가 지정한 값이 아니라 **JVM이 컨테이너 환경을 인식해 동적으로 계산했다**는 의미입니다

### 소형 컨테이너에서 힙 크기가 고정되는 구간

실측 결과 흔히 알려진 규칙과 다른 지점이 발견됩니다

일반적으로 알려진 규칙은 다음과 같습니다. 힙 상한은 `memory.max` × `MaxRAMPercentage`(기본 25%)로 계산되며, 컨테이너가 매우 작을 경우 `MinRAMPercentage`(기본 50%)가 적용됩니다

기본값 자체는 정확합니다. JDK 25.0.3에서 확인한 값은 `MaxRAMPercentage=25.0`, `MinRAMPercentage=50.0`, `InitialRAMPercentage=1.5625`입니다

문제는 **"매우 작다"의 기준점**입니다. 컨테이너 메모리 한도를 세밀하게 조정하며 힙 크기 산정 결과를 측정했습니다

| 컨테이너 메모리 | 산정된 힙 | 한도 대비 |
|---|---|---|
| 2 GiB | 512 MiB | 25.0% |
| 1 GiB | 256 MiB | 25.0% |
| 700 MiB | 176 MiB | 25.1% |
| 600 MiB | 152 MiB | 25.3% |
| **512 MiB** | **128 MiB** | **25.0%** |
| **500 MiB** | **128 MiB** | 25.6% |
| **400 MiB** | **128 MiB** | 32.0% |
| **300 MiB** | **128 MiB** | **42.6%** |
| **250 MiB** | **128 MiB** | 51.2% |
| 240 MiB | 120 MiB | 50.0% |
| 128 MiB | 64 MiB | 50.0% |

굵게 표시한 구간을 보면, **250MiB부터 512MiB까지 컨테이너 크기가 두 배 이상 차이 나는데도 힙은 모두 128MiB로 동일합니다**

25%도 50%도 아닌 비율입니다. 300MiB 파드의 힙은 한도의 **42.6%** 를 차지합니다

![The heap floor flattens the 25% rule between 250 and 512 MiB](/diagrams/container-memory-accounting-cgroup-3.svg)

그림의 가로축이 컨테이너 메모리, 세로축이 산정된 힙입니다

**오른쪽의 기울어진 직선**이 25% 규칙에 해당합니다. 컨테이너가 커질수록 힙도 비례해 증가합니다

**가운데의 평평한 구간**이 앞서 확인한 고정 구간입니다. 힙이 128MiB에서 변하지 않습니다

**왼쪽 기울어진 직선**이 50% 규칙(`MinRAMPercentage`)입니다

이 평평한 구간은 **최소 힙 기준선(floor)** 때문에 발생합니다. HotSpot에는 에르고노믹 계산이 개입하기 전의 기본 `MaxHeapSize`가 존재하며, 64비트 환경에서 이 값은 약 **124.8MiB**(96MiB에 워드 크기 보정 1.3배)입니다. 힙 정렬(alignment) 과정을 거치면 128MiB가 됩니다

실측 결과를 바탕으로 JVM의 산정 방식을 역산하면 다음과 같습니다. 컨테이너의 50%가 최소 기준선보다 작으면 50% 비율을 적용하고, 그렇지 않으면 **25% 비율과 최소 기준선 중 큰 값**을 적용합니다. 25% 비율이 최소 기준선보다 작아지는 구간(약 250~500MiB)에서는 최소 기준선이 적용되어 힙이 128MiB로 고정됩니다

이는 **파드 크기가 작을수록 힙이 한도에서 차지하는 비중이 예상보다 커진다는 의미**입니다

300MiB 파드에 128MiB의 힙을 할당하면 나머지는 172MiB뿐입니다. 이 172MiB로 Metaspace·코드 캐시·스레드 스택·다이렉트 버퍼·페이지 캐시를 모두 감당해야 합니다. 다음 편에서 다루겠지만, **JVM은 요청을 하나도 처리하기 전부터 이미 수십 MiB를 힙 밖에서 사용합니다**

따라서 "기본값 25%는 지나치게 보수적이니 상향해야 한다"는 조언은 **대형 컨테이너에만 유효합니다**. 소형 컨테이너에서는 이미 42%, 51%에 해당하는 비율을 사용하고 있기 때문입니다

이 최소 기준선을 벗어나려면 비율을 직접 명시해야 합니다. 같은 300MiB 컨테이너에서 확인한 결과입니다

| `MaxRAMPercentage` | 산정된 힙 |
|---|---|
| 25 (기본) | 128 MiB (42.6%) |
| 50 | 152 MiB (50.6%) |
| 75 | 232 MiB (77.3%) |

비율을 상향하면 최소 기준선을 넘어서므로 지정한 비율대로 정상 작동합니다

> **참고 — 버전 관련 주의사항** 리눅스 커널 6.12에서 일부 컨트롤러 구조가 cgroup v1에서 v2로 변경되면서, 이전 버전의 JDK가 컨테이너 메모리 한도를 인식하지 못하고 호스트 값으로 대체되어 OOM이 발생한 사례가 보고되었습니다. 이 감지 버그는 **JDK 25에서 수정**되었습니다(OpenJDK 이슈 JDK-8348566). 정상 동작하던 파드가 커널 업그레이드 이후 종료되는 문제의 전형적인 원인입니다

CPU 자원에도 동일한 방식이 적용됩니다. `UseContainerSupport`는 `cpu.max`를 읽어 실효 코어 수를 계산하며, 이 값이 **GC 병렬 스레드 수**·**JIT 컴파일러 스레드 수**·`ForkJoinPool.commonPool` 병렬도까지 연쇄적으로 결정합니다. [15편](/essays/java-g1-region-full-gc-cliff)에서 GC 스레드 수가 코어 수에서 파생된다고 설명했는데, 컨테이너 환경에서는 이 실효 코어 수가 그 기준이 됩니다

### Go — CPU 한도는 인식하지만 메모리 한도는 인식하지 못합니다

Go 런타임의 컨테이너 인식 기능은 비교적 최근에 도입되었으며, **자원별로 지원 수준에 차이**가 있습니다. 이 비대칭은 이번 편에서 다루는 핵심 내용 중 하나입니다

**CPU 자원은 인식합니다.** Go 1.25부터 런타임이 cgroup의 CPU 한도를 자동으로 인식해 `GOMAXPROCS` 값을 결정합니다. `cpu.max`를 읽어 실효 CPU 수를 계산하고, 이 값이 물리 코어 수보다 작으면 그에 맞춰 조정합니다. 한도가 변경되면 주기적으로 재확인하여 갱신합니다

**메모리 자원은 인식하지 않습니다.** `GOMEMLIMIT`(소프트 메모리 한도, Go 1.19부터 도입)은 cgroup 한도를 자동으로 읽지 않습니다

이는 Go 1.26.5의 런타임 소스 코드를 통해 직접 확인할 수 있습니다

```bash
GOROOT=$(go env GOROOT)
grep -c "memory.max\|memory.limit_in_bytes" -r $GOROOT/src/runtime/
grep -n "func " $GOROOT/src/runtime/cgroup_linux.go
grep -n "containermaxprocs\|cgroupmemlimit" $GOROOT/src/internal/godebugs/table.go
```

```text
0

45:func defaultGOMAXPROCSInit() {
75:func defaultGOMAXPROCSUpdateGODEBUG() {
85:func defaultGOMAXPROCS(ncpu int32) int32 {
109:func adjustCgroupGOMAXPROCS(procs int32, cpu cgroup.CPU) int32 {

31:	{Name: "containermaxprocs", Package: "runtime", Changed: 25, Old: "0"},
```

확인 결과는 다음과 같습니다

런타임 소스 전체에서 `memory.max` 문자열이 **0건** 검색됩니다. cgroup 관련 파일인 `cgroup_linux.go`에는 **GOMAXPROCS 관련 함수만** 존재합니다. godebug 목록에는 `containermaxprocs`(Go 1.25에서 도입)만 있고, 메모리에 대응하는 항목은 없습니다

**Go 런타임은 컨테이너의 메모리 한도를 읽지 않습니다.** 이를 읽는 코드 자체가 존재하지 않습니다

따라서 `GOMEMLIMIT`을 명시하지 않으면 기본값은 사실상 무제한입니다. 컨테이너 내부에서 확인한 값입니다

```text
GOMEMLIMIT=9223372036854775807  GOGC=100
```

9223372036854775807은 `math.MaxInt64`로, 사실상 한도가 없다는 의미입니다. GC는 오직 `GOGC=100` 규칙, 즉 힙이 라이브 객체의 2배에 도달하면 수집을 시작하는 규칙만으로 동작합니다

### Go 컨테이너 메모리 초과 현상 재현

이로 인해 발생하는 현상을 직접 재현합니다

라이브 집합 300MiB를 유지하면서 그 위에서 추가로 메모리를 할당하는 Go 프로그램을 **512MiB 컨테이너**에서 실행합니다

```go
live := make([][]byte, 0, 300)
for i := 0; i < 300; i++ {
	b := make([]byte, 1<<20)
	touch(b)               // 페이지를 실제로 만진다
	live = append(live, b) // 300MiB를 계속 붙들고 있는다
}

for round := 1; round <= 30; round++ {
	for i := 0; i < 100; i++ {
		b := make([]byte, 1<<20)
		touch(b) // 즉시 버려질 쓰레기
	}
	runtime.ReadMemStats(&ms)
	fmt.Printf("round %2d  힙 %4d MiB  다음GC목표 %4d MiB  GC %d회\n",
		round, ms.HeapInuse>>20, ms.NextGC>>20, ms.NumGC)
}
```

`GOMEMLIMIT` 없이 돌린 결과입니다

```text
GOMEMLIMIT=9223372036854775807  GOGC=100
라이브 집합 300MiB 확보. 이제 쓰레기를 만든다
round  1  힙  400 MiB  다음GC목표  562 MiB  GC 7회
round  2  힙  500 MiB  다음GC목표  562 MiB  GC 7회

ExitCode=137  OOMKilled=true
```

두 번째 순회 만에 프로세스가 종료됩니다. 종료 원인은 **로그에 그대로 나타나 있습니다**

**다음 GC 목표가 562MiB입니다. 컨테이너 한도는 512MiB입니다**

![The GC's next target sits above the kernel's kill line](/diagrams/container-memory-accounting-cgroup-4.svg)

그림의 세로축이 메모리입니다

**아래쪽 파란 영역**이 라이브 집합 300MiB, 즉 실제로 참조되고 있는 객체입니다

**그 위 회색 영역**이 GC가 수집을 시작하기 전까지 허용되는 여유 공간입니다. `GOGC=100` 설정에 따라 라이브 객체의 2배까지 증가하도록 허용되므로, GC의 다음 목표는 562MiB 부근에서 결정됩니다

**빨간 가로선**이 커널의 강제 종료 기준선, 즉 `memory.max` = 512MiB입니다

**GC의 목표선이 커널의 강제 종료 기준선보다 위에 위치합니다.** 힙이 512MiB를 초과하는 순간, GC가 수집을 시작하기 전에 커널이 먼저 `SIGKILL`을 전송합니다

[17편](/essays/gc-cost-conservation-no-silver-bullet)에서 할당이 저렴한 대가는 GC 비용으로 돌아온다고 설명했습니다. 컨테이너 환경에서는 이 비용이 **GC가 아니라 커널의 OOM killer**에 의해 청구됩니다

### `GOMEMLIMIT` 설정의 효과

동일한 프로그램과 512MiB 컨테이너에 `GOMEMLIMIT=450MiB` 설정만 추가합니다

```text
GOMEMLIMIT=471859200  GOGC=100
round  1  힙  400 MiB  다음GC목표  431 MiB  GC 7회
round  2  힙  373 MiB  다음GC목표  431 MiB  GC 8회
...
round 29  힙  318 MiB  다음GC목표  431 MiB  GC 435회
round 30  힙  418 MiB  다음GC목표  431 MiB  GC 435회
살아남았습니다

ExitCode=0  OOMKilled=false
```

프로세스가 정상 종료됩니다. 다음 GC 목표가 562MiB에서 **431MiB로 낮아져** 한도 아래에서 유지됩니다

다만 이는 공짜로 얻어지지 않습니다. **GC 횟수가 7회에서 435회로 증가했습니다**

`gctrace`를 통해 그 대가를 구체적으로 확인할 수 있습니다

| | `GOMEMLIMIT` 미설정 | `GOMEMLIMIT=450MiB` |
|---|---|---|
| 결과 | **exit 137, OOMKilled** | 살아남음 |
| GC 횟수 | 7회 | **435회** |
| GC가 쓴 CPU | **0%** | **6%** |
| 다음 GC 목표 | 562 MiB (한도 초과) | 431 MiB |

`gctrace` 원문에서 각 줄 앞의 백분율은 GC가 사용한 CPU 비중을 나타냅니다. 미설정 상태는 `gc 7 @0.046s 0%`, 설정 후에는 `gc 278 @0.292s 6%`로 나타났습니다

**메모리를 한도 안에 묶는 대가로 CPU 6%를 사용했습니다.** 4부에서 다룬 "메모리 여유 ↔ GC 빈도"의 트레이드오프가 컨테이너 환경에서는 이런 방식으로 나타납니다. 비용은 사라지지 않고 다른 자원으로 옮겨갈 뿐입니다

### 아직 채택되지 않은 자동화

이를 런타임 기본 동작으로 도입하자는 제안이 논의 중입니다(golang/go 이슈 #75164). `GOMAXPROCS` 방식과 유사하게, `GODEBUG=cgroupmemlimit`으로 활성화하고 한도의 90%(최소 100MB)를 `GOMEMLIMIT`으로 설정하자는 안입니다

2026년 7월 기준 이 제안은 **아직 논의 중이며 정식 릴리스에는 반영되지 않았습니다.** 따라서 현재로서는 관리자가 직접 설정해야 합니다

### Rust — 런타임 차원의 힙 관리가 존재하지 않습니다

Rust에는 GC와 별도의 언어 런타임이 없어 "힙을 얼마로 설정할까"라는 질문 자체가 성립하지 않습니다

메모리 수명은 소유권(ownership) 모델에 따라 컴파일 타임에 결정되며, 상한 관리는 애플리케이션 코드의 책임입니다. 컨테이너 한도를 초과해 할당하면 Rust 프로세스 역시 OOM-kill 대상이 됩니다. 다만 이는 **런타임에 조절 수단이 없어서**가 아니라, 애초에 그런 수단이 개입할 지점 자체가 없기 때문입니다. 할당 지점이 코드에 명시적으로 드러나므로 개발자가 직접 제어합니다

CPU 자원의 경우 표준 라이브러리가 컨테이너 환경을 인식합니다. [2편](/essays/thread-models-kernel-vs-user)에서 확인했듯 `std::thread::available_parallelism()`은 cgroup v2의 CPU 쿼터를 반영한 값을 반환합니다

다만 이는 **정확한 수치를 제공하는 함수**일 뿐입니다. 이 값을 바탕으로 스레드풀 크기를 결정하는 것은 `tokio`·`rayon` 같은 라이브러리와 개발자의 몫입니다. 런타임이 자동으로 조정하는 것이 아니라, 정확한 수치를 라이브러리에 전달할 준비가 되어 있는 구조입니다

## 예약된 메모리와 실제로 접촉된 메모리의 차이

앞선 Go 실험을 준비하는 과정에서 확인해야 할 개념이 하나 있습니다

초기 버전의 데모는 슬라이스를 할당만 하고 **데이터를 기록하지 않았습니다**

```go
_ = make([]byte, 1<<20) // 할당만 하고 만지지 않음
```

이 프로그램은 힙이 **8300MiB까지 증가하는 동안에도 512MiB 컨테이너에서 정상적으로 실행되었습니다.** Go의 `MemStats`는 8300MiB를 보고했지만 커널은 프로세스를 종료하지 않았습니다

그 이유는 [8편](/essays/kernel-lazy-allocation-page-fault)에서 다룬 내용과 동일합니다

Go는 `mmap`으로 반환받은 영역이 이미 0으로 채워져 있음을 알고 있으므로, 새 span에 대해서는 **memset을 생략합니다.** 이 프로그램은 해당 슬라이스에 아무 데이터도 기록하지 않았으므로 **페이지가 한 번도 접촉되지 않았습니다**

접촉되지 않은 페이지에는 물리 프레임이 할당되지 않습니다. **커널이 계량하는 대상은 접촉된 페이지뿐입니다**

![Reserved address space costs nothing; only touched pages become physical](/diagrams/container-memory-accounting-cgroup-5.svg)

그림 왼쪽은 프로세스가 확보한 **가상 주소 공간**입니다. 범위는 넓지만 대부분 비어 있습니다

오른쪽은 **물리 메모리**이며, 두 공간을 연결하는 화살표는 **실제로 접촉된 페이지에만** 표시됩니다

커널이 `memory.current`에 집계하는 대상은 오른쪽뿐입니다

`touch()`를 추가해 페이지를 실제로 접촉하게 하자 비로소 프로세스가 종료되었습니다. 이 구분(**예약 vs 접촉**)은 3편 뒤에서 다룰 "가상 메모리는 수 GB인데 왜 종료되지 않는가"라는 질문의 답이 됩니다

## 세 언어 비교 — CPU 인식과 메모리 인식은 별개의 문제입니다

같은 질문, 즉 런타임이 컨테이너 한도를 인식하는지를 세 언어에 적용하면 다음과 같이 나뉩니다

![CPU awareness landed everywhere; memory awareness did not](/diagrams/container-memory-accounting-cgroup-6.svg)

| | CPU 한도 인식 | 메모리 한도 인식 | 회계의 층위 |
|---|---|---|---|
| **Rust** | `available_parallelism()`이 cgroup 쿼터 반영(힌트). 스레드풀은 tokio·rayon 몫 | GC가 없어 "힙 크기" 질문 자체가 없음 | 라이브러리·앱 |
| **Go** | `GOMAXPROCS` 자동(1.25+) | `GOMEMLIMIT` **수동** — 1.26에도 cgroup 자동 아님 | 런타임(반쪽) |
| **Java** | `ActiveProcessorCount` 자동 | `MaxRAMPercentage`로 힙 자동 산정 | 런타임(양쪽) |

표를 세로로 보면 4부에서 다룬 삼각형 구도가 클라우드 환경에서도 동일하게 나타납니다

**Java**는 런타임이 CPU와 메모리를 모두 자동으로 조정합니다. 관리 부담은 적지만, 자동 산정값이 워크로드에 맞지 않아도 이를 알아채기 어렵습니다. 앞서 확인한 128MiB 최소 기준선이 그 예입니다. **자동화로 인해 설정을 점검하지 않게 되는 것 자체가 비용**입니다

**Go**는 CPU 인식만 도입된 과도기 상태입니다. `GOMAXPROCS`는 신뢰할 수 있지만 `GOMEMLIMIT`은 여전히 수동 설정이 필요합니다. 이를 인지하지 못하면 "Go는 메모리 사용이 가벼운 언어인데 왜 파드가 반복적으로 종료되는가"라는 문제에 부딪힙니다. **프로덕션 환경에서 가장 흔하게 발생하는 함정입니다**

**Rust**는 인식할 런타임 자체가 존재하지 않습니다. 대신 메모리 수명이 소유권으로 결정되어 있어 예측하지 못한 증가가 발생할 여지가 적고, CPU 병렬도는 라이브러리에 정확한 수치를 전달할 준비만 되어 있습니다. **제어 수단이 없는 것이 아니라, 제어 수단이 코드 안에 위치한 것**입니다

## 운영 환경 설정 가이드

핵심 원칙은 다음과 같습니다. **커널의 강제 종료 기준(`limits.memory`)과 런타임의 힙 설정값은 서로 다른 지표이며, 그 사이에 여백을 반드시 남겨야 합니다**

**JVM에서는** `-Xmx` 고정과 `MaxRAMPercentage` 비율 중에 고릅니다

`-Xmx512m`처럼 절대값을 지정하면 컨테이너 한도와 무관하게 힙 크기가 고정되며, 이 설정이 `MaxRAMPercentage`보다 우선합니다. 한도가 고정되어 있고 힙과 네이티브 메모리 비율을 정밀하게 제어하고자 할 때 사용합니다

`-XX:MaxRAMPercentage=75.0`처럼 비율로 지정하면 `limits.memory`가 변경되어도 자동으로 반영됩니다. 여러 파드에 동일한 템플릿을 사용하거나 VPA로 한도가 동적으로 변하는 환경에 적합합니다

대형 컨테이너에서는 기본값 25%가 보수적이므로 50~75%로 상향하되, 나머지는 Metaspace·코드 캐시·스레드 스택·다이렉트 버퍼 몫으로 남겨야 합니다. **소형 컨테이너(250~500MiB)에서는 최소 기준선으로 인해 이미 40~50%를 사용하고 있다는 점을 감안해**, 비율을 올리기 전 힙 외 영역의 필요량을 먼저 측정해야 합니다

**Go에서는** `GOMEMLIMIT`을 명시하는 것에서 출발합니다

권장 패턴은 `GOMEMLIMIT`을 한도의 약 90%로 설정하고 `GOGC`는 기본값 100으로 유지하는 것입니다. 평상시에는 `GOGC` 규칙에 따라 여유 있게 동작하다가, 메모리가 한도에 근접하면 `GOMEMLIMIT`이 GC 주기를 강하게 당겨 조절합니다

`GOGC=off`로 설정하고 `GOMEMLIMIT`만으로 제어하는 방식도 있지만 권장하지 않습니다. 라이브 집합이 한도에 근접하면 GC가 끊임없이 실행되어 CPU 사용량이 크게 늘어납니다. 앞서 확인한 6%가 30%, 50%로 증가할 수 있는 방식입니다

값을 설정하는 방법은 세 가지입니다. 매니페스트에 상수로 지정하거나, downward API로 `limits.memory`를 환경변수로 전달받아 계산하거나, `automemlimit` 같은 라이브러리로 부팅 시 cgroup을 읽도록 하는 방법입니다. 한도가 자주 변경되는 환경에서는 뒤의 두 방식이 더 안전합니다

## 트레이드오프 소결

한 문장으로 정리하면 이렇습니다

**커널의 메모리 회계(전체 합산)와 런타임의 힙 회계(일부만 조절)는 층위가 다르며, 이 둘을 연결하는 방식은 언어마다 차이가 있습니다**

Java는 런타임이 이 연결을 자동으로 처리합니다. 다만 자동 산정값이 항상 최적은 아니며, 소형 컨테이너에서는 최소 기준선이라는 제약도 존재합니다

Go는 CPU 자원만 자동으로 연결하고, 메모리 자원은 관리자의 수동 설정에 의존합니다. 장애가 가장 발생하기 쉬운 지점입니다

Rust는 이러한 연결 자체가 필요 없도록, 소유권 모델로 메모리 수명을 컴파일 타임에 결정합니다

언어와 관계없이 공통으로 적용되는 원칙이 하나 있습니다. **힙 크기를 한도에 정확히 맞추더라도, 커널은 힙 외의 영역까지 합산하므로 여백이 없으면 프로세스가 종료됩니다.** 이 여백의 물리적 원리는 20편에서 다룹니다

## 더 파고들 질문

1. **`memory.stat`으로 원인 좁히기** — 파드의 `memory.current`는 한도에 근접했는데 애플리케이션이 보고하는 힙은 절반 수준입니다. `anon`·`file`·`slab`·`sock` 중 무엇이 증가했는지 어떻게 판별해야 할까요? `file`(페이지 캐시)이 큰 경우와 `anon`이 큰 경우, 대응 방식이 어떻게 달라져야 할까요?
2. **requests와 limits를 동일하게 둘 것인가** — 메모리 `requests < limits`로 설정하면 빈패킹 밀도는 높아지지만, 노드 메모리가 부족할 때 어떤 파드가 먼저 종료될까요(QoS class: Guaranteed vs Burstable)? 안정성이 중요한 서비스에서 두 값을 동일하게 두는 이유를 cgroup 관점에서 설명할 수 있을까요?
3. **`GOMEMLIMIT` 주입 자동화** — VPA가 파드의 `limits.memory`를 동적으로 변경하는 환경에서, `GOMEMLIMIT`을 매니페스트에 상수로 고정하면 어떤 문제가 발생할까요? downward API·`automemlimit`·(아직 미도입된) `GODEBUG=cgroupmemlimit` 중 무엇을 선택해야 할까요?
4. **CPU 스로틀링과 GC의 상호작용** — `limits.cpu`로 스로틀링이 걸린 파드에서 GC 스레드가 CPU 시간을 확보하지 못하면 어떤 문제가 발생할까요? `GOMAXPROCS`·`ActiveProcessorCount`가 컨테이너 값에 맞춰 **줄어드는 것**이 오히려 GC 지연 측면에서 유리한 이유는 무엇일까요?
5. **소형 파드의 힙 최소 기준선** — 동일한 이미지를 128Mi·300Mi·2Gi 파드에 배포합니다. 이번 편에서 확인한 128MiB 기준선으로 인해 300Mi 파드에서 힙 외 여백은 얼마나 남을까요? 이 파드에서 `MaxRAMPercentage`를 상향하는 것이 위험할 수 있는 이유는 무엇일까요?
6. **계측 파이프라인 대응 순서** — "OOMKilled가 증가했다"는 알람을 받았을 때, cgroup `memory.events`의 `oom_kill` 카운터, 커널 `dmesg`의 OOM 리포트, 쿠버네티스 이벤트, 런타임 GC 로그를 어떤 순서로 확인하여 원인을 "힙 초과 / 네이티브 누수 / 한도 과소설정" 중 하나로 좁혀야 할까요?

## 핵심 요약

- 매니페스트의 `limits.memory: 512Mi`는 커널에서 **`memory.max = 536870912`** 값으로 그대로 반영됩니다. 실측으로 확인한 결과입니다
- **커널은 힙 개념 없이 페이지 단위로 메모리를 계량합니다.** `anon`(힙·스택) 외에도 `file`(페이지 캐시)·`kernel`(슬랩·페이지 테이블)·`sock`·`shmem`이 모두 같은 한도에 합산됩니다. 아무 작업도 하지 않는 컨테이너에서도 커널 관련 메모리가 익명 메모리의 9배였습니다
- 한도를 초과하면 커널은 먼저 회수를 시도하고, 실패하면 **`SIGKILL`로 즉시 종료**합니다. 종료 코드는 **137**(=128+9)입니다. SIGKILL은 가로챌 수 없으므로 런타임은 예외를 처리할 기회조차 얻지 못합니다
- **메모리는 압축 불가능한 자원**입니다. CPU 초과는 스로틀링(지연)으로, 메모리 초과는 종료로 이어집니다
- **JVM은 CPU와 메모리 한도를 모두 자동으로 인식**합니다. 다만 힙 산정에 **128MiB 최소 기준선**이 있어, **250~512MiB 컨테이너는 크기와 무관하게 힙이 모두 128MiB로 고정**됩니다. 300MiB 파드의 힙은 25%가 아닌 **42.6%** 를 차지합니다
- **Go 런타임은 컨테이너 메모리 한도를 읽는 코드가 존재하지 않습니다.** 소스 전체에서 `memory.max` 문자열이 0건이며, `GOMEMLIMIT` 기본값은 `math.MaxInt64`입니다. 자동화 제안(#75164)은 아직 채택되지 않았습니다
- 따라서 라이브 300MiB인 Go 서비스를 512MiB 컨테이너에서 실행하면 **GC의 다음 목표(562MiB)가 커널의 강제 종료 기준(512MiB)보다 높게** 설정됩니다. GC가 동작하기 전에 프로세스가 종료되며, 실제로 exit 137을 재현했습니다
- `GOMEMLIMIT=450MiB` 설정만으로 정상 동작합니다. 다만 **GC 횟수가 7회에서 435회로 증가하고 CPU 6%를 사용합니다.** 비용은 사라지지 않고 메모리에서 CPU로 이전됩니다
- **접촉하지 않은 페이지는 물리 메모리로 계량되지 않습니다.** 힙이 8300MiB로 보고되더라도 페이지를 접촉하지 않았다면 커널은 이를 집계하지 않습니다. 예약과 접촉의 구분은 20편에서 다시 다룹니다
