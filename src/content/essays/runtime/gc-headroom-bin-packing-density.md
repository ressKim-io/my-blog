> **시리즈 "커널과 런타임으로 톺아보는 Rust · Go · Java"의 19편**
> [18편](/essays/container-memory-accounting-cgroup)에서 커널의 회계 규칙을 세웠습니다. 커널은 힙을 모르고 페이지를 세며, 한도를 넘으면 죽입니다
> 그 과정에서 런타임이 요구하는 **메모리 여유(headroom)** 를 여러 번 봤습니다. Go는 `GOGC`가 만드는 라이브의 2배, JVM은 힙 밖 여백
> 이번 편은 그 여유가 **어떻게 돈이 되는지**를 끝까지 따라갑니다. **GC가 남겨 둔 여유는 파드의 크기가 되고, 파드 크기가 노드 밀도를, 밀도가 청구서를 정합니다**

## 빈패킹 — 스케줄러는 '사용량'이 아니라 '요청'으로 채웁니다

**빈패킹(bin packing)** 은 크기가 정해진 상자에 물건을 되도록 빈틈없이 채워 상자 수를 최소화하는 문제입니다

여기서 상자는 노드, 물건은 파드입니다. 클라우드에서 노드 한 대는 시간당 요금이므로 **빈패킹 밀도가 그대로 비용**이 됩니다

그런데 스케줄러가 짐의 크기를 어디서 읽는지가 핵심입니다

![The scheduler packs by declared requests, not by actual usage](/diagrams/gc-headroom-bin-packing-density-1.svg)

그림 왼쪽이 노드이고, 안에 파드들이 쌓여 있습니다

각 파드에 상자가 두 개 겹쳐 그려져 있습니다. **바깥 점선 상자**가 파드가 선언한 `requests`이고, **안쪽 색칠된 상자**가 실제 사용량입니다

**스케줄러는 바깥 점선만 봅니다.** 2 CPU를 요청하고 실제로는 200m만 쓰는 파드도, 스케줄러 눈에는 그냥 "2 CPU짜리 짐"입니다

그래서 `requests`를 크게 잡으면 실제로는 놀고 있는 메모리라도 스케줄러는 "찼다"고 판단하고 다음 파드를 다른 노드로 보냅니다. **요청과 실사용의 간극이 곧 낭비된 노드**입니다

그림 오른쪽 위의 회색 영역이 그 간극입니다. 노드가 실제로는 절반이 비어 있는데도 스케줄러는 더 못 싣죠

### 노드가 파드에게 주는 몫은 물리값이 아닙니다

한 가지 더 짚고 갑니다. 파드가 나눠 갖는 건 노드의 물리 메모리 전부가 아니라 **Allocatable**입니다

물리 메모리(Capacity)에서 세 몫이 먼저 빠집니다. **system-reserved**(OS·커널·sshd 등 호스트 몫), **kube-reserved**(kubelet과 컨테이너 런타임 몫), 그리고 **eviction-threshold**(축출을 막기 위한 방어 여유)입니다

남은 것이 Allocatable이고, 파드는 이걸 `requests`로 나눠 갖습니다

즉 32GiB 노드라고 32GiB를 파드에 못 줍니다. 통상 몇 백 MiB에서 1GiB 정도가 먼저 빠지죠

여기서 노드 크기 선택 자체가 경제학이 됩니다. **노드가 작을수록 이 고정 상납분의 비율이 커집니다.** 같은 총량을 작은 노드 여러 대로 사면 상납분을 여러 번 내는 셈이니까요

### 압축 가능한 자원과 그렇지 않은 자원, 다시

[18편](/essays/container-memory-accounting-cgroup)의 비대칭이 빈패킹에서 이렇게 작동합니다

CPU는 `requests`로 스케줄하되 실사용이 넘치면 스로틀로 버티므로 **오버커밋(요청보다 많이 얹기)이 관대**합니다. 좀 느려질 뿐 아무도 안 죽으니까요

메모리는 넘치면 OOMKilled입니다. 그래서 오버커밋이 위험합니다

결국 **"메모리 `requests`를 얼마나 촘촘히 잡느냐"** 가 빈패킹 밀도와 안정성을 동시에 쥔 손잡이가 됩니다

## GC 헤드룸이 상자를 부풀립니다

같은 라이브 집합을 담더라도 어떤 런타임과 GC를 쓰느냐에 따라 필요한 컨테이너 상한이 달라집니다. 그 차액이 전부 **런타임 오버헤드**입니다

### Go — `GOGC=100`이 만드는 라이브의 2배

[18편](/essays/container-memory-accounting-cgroup)에서 직접 봤습니다. 라이브 300MiB인 Go 서비스의 다음 GC 목표가 **562MiB**로 찍혔죠

`GOMEMLIMIT`을 안 주면 이 2배가 상자의 기본 크기입니다. 여기에 고루틴 스택과 런타임 자체가 얹힙니다

상자를 줄이려면 `GOMEMLIMIT`을 라이브 근처로 당기면 됩니다. 다만 18편에서 그 대가도 함께 봤습니다. **GC가 7회에서 435회로, CPU가 0%에서 6%로** 늘었습니다

**상자 크기와 GC CPU가 맞바꿔지는 손잡이**인 셈입니다

### Java G1 — 힙 안 예비 + 회계 구조

G1의 헤드룸은 여러 겹입니다

**`G1ReservePercent = 10`** — 힙의 10%를 예비로 비워 둡니다. [15편](/essays/java-g1-region-full-gc-cliff)에서 본 이사(evacuation) 실패를 막기 위한 완충이죠. 즉 힙의 90%만 실사용에 쓴다고 봐야 합니다

**카드 테이블(card table)** — 세대 간 참조를 추적하는 자료구조로, 힙 512바이트당 1바이트, 곧 힙의 약 **0.2%** 를 상시 점유합니다

**Remembered Set** — region 간 참조를 기록합니다. 포인터가 많은 워크로드에선 상당히 커질 수 있어 고정 수치로 못 박기 어렵습니다

그리고 **힙 밖**이 있습니다. 이게 이번 편의 진짜 주제입니다

### 그래서 힙 밖을 직접 재 봤습니다

`-Xmx`로 힙을 묶어도 줄어들지 않는 그 영역이 얼마나 되는지, GC를 바꿔 가며 컨테이너 안에서 측정했습니다

**아무 일도 안 하는 JVM**입니다. 클래스 하나 로드하고 200ms 자다 끝나는 프로그램이죠

```bash
docker run --rm --memory=1g eclipse-temurin:25-jdk \
  java -XX:+UseG1GC -Xmx512m -Xms16m \
       -XX:NativeMemoryTracking=summary \
       -XX:+UnlockDiagnosticVMOptions -XX:+PrintNMTStatistics Idle
```

**NMT(Native Memory Tracking)** 는 JVM이 힙 밖에서 쓰는 메모리를 범주별로 보여 주는 핫스팟 기능입니다

그런데 여기서 함정에 한 번 빠졌습니다. 그 함정이 이번 편에서 가장 배울 게 많은 대목이라 그대로 적습니다

### 같은 JVM을 세 가지로 재면 순위가 서로 다릅니다

처음엔 NMT만 봤습니다. 그랬더니 이런 표가 나왔습니다

| GC | NMT 힙 밖 committed | 총 reserved |
|---|---|---|
| Serial | 24.3 MiB | 1.84 GiB |
| Parallel | 44.4 MiB | 1.86 GiB |
| **G1**(기본) | **65.0 MiB** | 1.89 GiB |
| ZGC | 26.4 MiB | **9.35 GiB** |

읽자마자 "통념이 뒤집혔다"고 생각했습니다. **ZGC가 커밋한 힙 밖 메모리(26.4MiB)가 G1(65.0MiB)보다 적으니까요.** ZGC의 거대한 숫자는 9.35GiB짜리 **예약**뿐인데, 18편에서 봤듯 커널은 예약을 세지 않습니다

절반은 맞습니다. 그런데 **정작 파드를 죽이는 지표로는 재지 않았습니다**

커널이 실제로 청구하는 값, 즉 `memory.current`를 같은 조건에서 함께 뽑아 보겠습니다

```bash
docker run --rm --memory=1g eclipse-temurin:25-jdk sh -c '
  java -XX:+UseZGC -Xmx512m -Xms16m Idle &
  sleep 1.8
  cat /sys/fs/cgroup/memory.current'
```

| GC | reserved (가상) | NMT committed | **memory.current (커널이 청구)** |
|---|---|---|---|
| Serial | 1.84 GiB | 40.3 MiB | 41.2 MiB |
| Parallel | 1.86 GiB | 60.4 MiB | 43.3 MiB |
| G1 | 1.89 GiB | **83.0 MiB** | 52.0 MiB |
| ZGC | **9.35 GiB** | 42.4 MiB | **77.9 MiB** |

**세 열의 순위가 전부 다릅니다**

reserved로는 ZGC가 압도적입니다. committed로는 **G1이 1위**입니다. 그런데 커널이 실제로 청구하는 값으로는 **다시 ZGC가 1위**입니다

![Three ways to measure the same JVM, three different rankings](/diagrams/gc-headroom-bin-packing-density-2.svg)

그림에 GC마다 막대가 셋입니다

**옅은 회색**이 예약된 가상 주소 공간, **파란색**이 JVM이 커밋한 양, **붉은색**이 커널이 실제로 청구하는 `memory.current`입니다

ZGC의 회색 막대만 화면 밖으로 뻗어 나가는데, 정작 **붉은 막대도 ZGC가 가장 큽니다**

그리고 G1을 보면 파란 막대(83.0)가 붉은 막대(52.0)보다 **큽니다.** 커밋했지만 아직 만지지 않은 페이지가 있다는 뜻이죠

반대로 ZGC는 붉은 막대(77.9)가 파란 막대(42.4)보다 큽니다. **NMT가 세지 않는 무언가를 커널이 청구하고 있습니다**

### ZGC의 힙은 `anon`에 없습니다

그 무언가를 찾아 `memory.stat`을 열었습니다

```text
G1     청구 51.9 = anon 49.0 + file  0.0 + 커널 2.9
ZGC    청구 77.9 = anon 58.8 + file 16.0 + 커널 3.1
```

ZGC만 **`file` 기둥에 16.0MiB**가 있습니다. G1은 0이고요

`file`은 페이지 캐시입니다. 아무 파일도 안 읽는 프로그램이 왜 16MiB의 페이지 캐시를 쓸까요

프로세스의 메모리 매핑을 직접 열어 봤습니다

```bash
grep -iE 'memfd|deleted' /proc/$PID/maps
```

```text
40000000000-40001000000 rw-s 00000000 00:01 3102   /memfd:java_heap (deleted)
```

**ZGC의 힙이 `memfd:java_heap`이라는 익명 파일로 매핑돼 있습니다.** 권한이 `rw-s`, 끝의 **`s`가 shared(공유)** 입니다. 사설 익명 매핑이 아니죠

그래서 cgroup 회계에서 이 페이지들은 `anon`이 아니라 **`shmem`** 으로 잡힙니다. `memory.stat`을 다시 보면 정확히 확인됩니다

```text
anon  61644800
file  16809984
shmem 16777216      ← 정확히 16 MiB = -Xms16m
```

`shmem`이 **초기 힙 크기와 정확히 일치**합니다. 우연이 아닙니다

[16편](/essays/generational-zgc-colored-pointer-stall)에서 본 colored pointer 때문입니다. ZGC는 포인터에 색을 칠해 쓰려고 힙을 특별한 주소 공간에 놓는데, 매핑 주소가 `40000000000`, 즉 **1TiB 언저리**인 것도 그래서입니다. 그 9.35GiB 예약의 정체이기도 하고요

**여기서 실무 함정이 나옵니다**

ZGC 파드가 메모리를 많이 쓴다고 `memory.stat`을 열어 `anon`에서 힙을 찾으면 **안 나옵니다.** `shmem`을 봐야 합니다

게다가 18편에서 "`file`(페이지 캐시)은 압박이 오면 커널이 먼저 버린다"고 했죠. 그 안심이 여기선 통하지 않습니다. **`shmem`은 `file` 기둥에 있지만 버릴 수 없습니다.** 뒷받침할 디스크 파일이 없으니까요

### 그래서 ZGC는 결국 무겁습니다. 다만 이유가 다릅니다

정리하면 **"ZGC는 메모리를 많이 쓴다"는 통념은 참**입니다. 커널이 청구하는 값으로 가장 크니까요

하지만 **사람들이 대는 이유는 대체로 틀렸습니다**

`ps`에서 VSZ 9GB를 보고 놀라는 건 **착시**입니다. 그 9GB는 예약이고 커널은 세지 않습니다. OOMKill을 일으키지 않습니다

ZGC가 실제로 무거운 이유는 둘입니다

하나는 [16편](/essays/generational-zgc-colored-pointer-stall)에서 플래그로 확인한 **compressed oops 미지원**입니다. G1은 `UseCompressedOops = true`, ZGC는 `false`죠. 32GiB 이하 힙에서 G1은 객체 참조를 4바이트로 압축하지만 ZGC는 주소 비트를 colored pointer에 쓰느라 **8바이트를 그대로** 씁니다. 포인터가 많은 객체 그래프일수록 **힙 안의 실사용**이 커집니다

다른 하나는 **동시 수집용 빈 공간**입니다. 뮤테이터를 안 멈추고 이사하려면 GC가 도는 동안 들어오는 할당을 받아 줄 여유가 힙 안에 있어야 합니다. [16편](/essays/generational-zgc-colored-pointer-stall)의 allocation stall이 바로 이 여유가 모자랄 때 나는 현상입니다

원본 정리에는 "ZGC가 힙의 약 3%를 고정 오버헤드로 점유한다"고 적혀 있었는데, **출처를 찾지 못해 싣지 않았습니다.** OpenJDK가 실제로 말하는 것은 "필요한 여유는 할당률과 라이브 집합에 따라 크게 다르다"이고, 문서가 주는 감각은 peak working set 대비 **15~25%** 입니다

## 아무도 말하지 않는 고정비 — 파드를 쪼갤 때마다 곱해집니다

여기서 실측 하나를 더 얹으면 빈패킹의 그림이 달라집니다

**힙 밖 고정비가 힙 크기와 거의 무관합니다**

`-Xmx`를 바꿔 가며 커널이 청구하는 값을 재 봤습니다. 힙은 `-Xms16m`으로 고정해 두고, 상한만 키웁니다

| `-Xmx` | `memory.current` |
|---|---|
| 256 MiB | 51.7 MiB |
| 512 MiB | 53.6 MiB |
| 1 GiB | 52.0 MiB |
| 2 GiB | 54.4 MiB |

힙 상한을 **8배로 키웠는데 청구액은 52~54MiB에서 움직이지 않습니다**

**힙 밖 고정비는 힙에 비례하는 세금이 아니라, 프로세스마다 한 번씩 내는 인두세**입니다

### 그럼 세 언어는 각각 얼마를 낼까요

여기까지 왔으니 JVM만 볼 이유가 없습니다. **아무 일도 안 하고 3초 자다 끝나는 프로그램**을 세 언어로 만들어, 같은 컨테이너에서 커널이 얼마를 청구하는지 나란히 쟀습니다

| 런타임 | `memory.current` (유휴) | Rust 대비 |
|---|---|---|
| **Rust** | **2.3 MiB** | 1배 |
| **Go** | **4.8 MiB** | 2.1배 |
| **Java** (기본 G1) | **41.2 MiB** | **18배** |
| **Java** (ZGC) | **77.9 MiB** | **34배** |

요청 하나 처리하기 전, 코드 한 줄 실행하기 전의 숫자입니다

[3편](/essays/runtime-weight-zero-cost-vs-fat)에서 "런타임의 무게"를 이야기하며 zero-cost와 fat runtime을 갈랐습니다. **그 무게가 컨테이너에서 청구되는 실제 금액이 이것입니다**

빈패킹에서 이건 반전을 만듭니다

![Splitting pods multiplies the per-process fixed cost](/diagrams/gc-headroom-bin-packing-density-3.svg)

그림 왼쪽은 큰 파드 하나입니다. 힙이 크고, 고정비는 **한 번만** 붙습니다

그림 오른쪽은 같은 총 힙을 작은 파드 넷으로 쪼갠 것입니다. 힙 총량은 같은데 **붉은 고정비 블록이 파드 수만큼 늘어났습니다**

산수는 단순합니다. 파드 10개를 띄우면 아무 일도 안 하는 상태에서

**Java는 412MiB, Go는 48MiB, Rust는 23MiB**를 냅니다. 같은 클러스터에서 **389MiB의 차이**가 순수하게 런타임 살림에서만 생깁니다

**밀도를 높이려고 파드를 잘게 쪼개는 행위가, 정작 런타임 고정비를 파드 수만큼 곱합니다**

마이크로서비스 논의에서 자주 빠지는 항목입니다. 서비스를 잘게 나누는 데는 아키텍처적 이유가 충분히 있을 수 있지만, **런타임이 무거운 언어에서는 나눌 때마다 고정비를 새로 냅니다.** Go나 Rust는 같은 결정의 대가가 한 자릿수 MiB죠

사이드카를 떠올려 보십시오. 파드마다 붙는 프록시가 Go나 Rust로 쓰인 데는 이유가 있습니다. **수천 개가 뜨는 프로세스의 고정비는 곱셈으로 돌아옵니다**

그리고 앞 표의 Serial GC(41.2MiB)와 G1(52.0MiB)을 비교해 보십시오. 작고 많은 파드를 굴리는 클러스터라면 **GC 선택만으로 파드당 10MiB를 아낄 수 있습니다.** 물론 그만큼 멈춤 시간을 내주지만요

## 상자를 줄이는 법과 그 대가

밀도를 높이는 손잡이는 셋입니다. 공짜는 하나도 없습니다

### 런타임 쪽 — 상자 자체를 줄이기

**Go에서는** `GOMEMLIMIT`을 라이브 집합 가까이 당겨 2배 상자를 압축합니다. 대가는 GC CPU입니다. 18편에서 6%를 봤고, 과하게 당기면 GC가 쉼 없이 도는 지경까지 갑니다

**Java에서는** `MaxRAMPercentage`를 워크로드에 맞게 조정합니다. 다만 18편에서 본 **128MiB 바닥**을 기억하십시오. 작은 파드는 이미 40~50%를 쓰고 있습니다

그리고 **G1을 쓴다면 힙을 32GiB 밑에 두는 것만으로 실사용이 줄어듭니다.** compressed oops가 참조를 4바이트로 압축하니까요. ZGC엔 이 이득이 없습니다

**GC 선택 자체가 상자 결정입니다.** 지연이 절박하지 않다면 Serial이나 Parallel이 G1보다 힙 밖이 가볍습니다. 밀도를 사려고 지연을 내주는 거래죠

### 상자를 어떻게 선언할 것인가

**`requests` = `limits`(Guaranteed)** 로 두면 스케줄러가 본 크기와 커널이 죽이는 선이 같아집니다. 밀도는 보수적이지만 노드 레벨 경합이 없어 안전합니다

**`requests` < `limits`(Burstable)** 로 두면 스케줄러엔 작게 보여 **더 촘촘히** 채워집니다. 대신 여러 파드가 동시에 `limits`로 치달으면 노드 메모리가 터지고 누군가 OOMKilled 당합니다

이때 죽는 순서는 QoS class를 따릅니다. **Guaranteed > Burstable > BestEffort** 순으로 보호되죠. 즉 **밀도를 사는 대신 안정성을 저당 잡는** 거래입니다

### 스케줄러에게 "빽빽이 채워"라고 말하기

마지막 함정입니다. **쿠버네티스 스케줄러의 기본 점수 전략은 빈패킹이 아닙니다**

기본은 `LeastAllocated`로, 가용 자원이 많은 노드를 선호해 파드를 **분산**시킵니다

빈패킹을 원하면 `NodeResourcesFit` 플러그인의 **`MostAllocated`** 또는 **`RequestedToCapacityRatio`** 를 스케줄러 프로파일에 명시해야 합니다

즉 **"촘촘히"는 기본값이 아니라 선택**입니다. Karpenter나 Cluster Autoscaler의 노드 통합(consolidation)도 같은 목표를 사후에 달성하는 접근입니다

## 세 언어 수렴 — 같은 워크로드, 다른 밀도

라이브 집합이 같은 서비스를 세 언어로 담았을 때의 경향입니다

| | 유휴 고정비 | 상자를 키우는 요인 | 무엇을 사고 무엇을 파나 |
|---|---|---|---|
| **Rust** | **2.3 MiB** | GC 없음. 할당자 단편화 여유 정도 | 밀도·예측가능성을 얻고 **개발 시간**을 냄 |
| **Go** | **4.8 MiB** | `GOGC=100`의 2배(또는 `GOMEMLIMIT` 여유) + 스택 | 단순함을 얻고 **헤드룸 또는 GC CPU**를 냄 |
| **Java G1** | **41.2 MiB** | 힙 예비 10% + RSet + 힙 밖 살림 | 성숙한 처리량을 얻고 **힙 밖 여백**을 냄 |
| **Java ZGC** | **77.9 MiB** | 압축 참조 없음 + 동시 수집 여유 | **sub-ms 지연**을 얻고 밀도·처리량을 냄 |

세로로 읽으면 **밀도는 곧 무엇을 포기했느냐의 기록**입니다

Rust는 런타임 여유가 없어 가장 빽빽하지만, 그 대가를 컴파일 타임과 개발자의 시간에 **미리** 냈습니다

ZGC는 지연을 극한까지 낮춘 대신 상자가 커져 노드를 더 씁니다

Go는 그 사이에서 손잡이 하나로 밀도와 CPU를 맞바꿉니다

중요한 건 **"어느 언어가 싸냐"가 아니라 "이 클러스터가 무엇을 최적화하느냐"** 입니다. 지연 SLO가 빡빡한 결제 API라면 ZGC의 성긴 밀도가 정당한 비용이고, 야간 배치라면 그 여유는 순수 낭비입니다

## 트레이드오프 소결

한 문장으로 정리하면 이렇습니다

**GC가 요구하는 메모리 여유는 파드 상자의 크기가 되고, 상자 크기가 노드 밀도를, 밀도가 청구서를 정합니다**

[17편](/essays/gc-cost-conservation-no-silver-bullet)의 삼각형이 클라우드에서 이렇게 번역됩니다

![The cloud triangle: density, safety margin, latency](/diagrams/gc-headroom-bin-packing-density-4.svg)

그림의 세 꼭짓점이 **밀도**(노드 비용)·**안전마진**(오버커밋을 안 할수록 안전)·**지연**입니다

**Rust**는 밀도 꼭짓점에 가장 가깝습니다. 개발 시간을 미리 낸 대가죠

**ZGC**는 지연 꼭짓점으로 끌려가 있고, 그만큼 밀도와 처리량에서 멀어졌습니다

**Guaranteed(`requests` = `limits`)** 는 안전마진 꼭짓점입니다. 밀도를 포기한 자리죠

밀도를 높이는 세 손잡이 — GC 헤드룸 압축, 오버커밋, 빈패킹 스케줄러 — 는 각각 **CPU·안정성·복잡도**를 대가로 요구합니다. **공짜 밀도는 없습니다**

그리고 이 모든 계산 밑에 18편의 전제가 깔려 있습니다. **상자 크기를 아무리 잘 잡아도, 커널이 세는 건 힙만이 아니므로 힙 밖 여백이 부족하면 상자 안에서 파드가 죽습니다**

그 순간을 다음 편에서 해부합니다

## 더 파고들 질문

1. **requests 산정의 근거** — 메모리 `requests`를 P50 사용량으로 잡을지, P99로 잡을지, GC 헤드룸을 포함한 최대 힙으로 잡을지에 따라 밀도와 OOM 위험이 어떻게 갈립니까? Go에서 `GOMEMLIMIT`을 먼저 정하고 그 값을 기준으로 `requests`를 잡는 순서가 왜 자연스러울까요?

2. **파드 쪼개기의 손익** — 이번 편에서 잰 유휴 고정비(Java 41.2 · Go 4.8 · Rust 2.3 MiB)를 근거로, 하나의 서비스를 파드 3개로 나눌 때와 1개로 둘 때의 총 메모리를 계산해 보십시오. 어느 지점부터 나누는 이득(격리·독립 배포)이 곱해지는 고정비를 넘어섭니까? 같은 계산을 Go로 하면 결론이 어떻게 달라집니까?

3. **오버커밋 비율 결정** — 메모리 `requests:limits`를 1:1로 둘 서비스와 1:1.5로 둘 서비스를 무엇으로 나누겠습니까? 노드에 Burstable 파드가 몰렸을 때 동시에 `limits`로 튀는 "상관된 스파이크"를 어떤 신호로 예측하겠습니까?

4. **32GiB 절벽** — G1을 쓰는 서비스의 힙을 30GiB에서 40GiB로 올렸더니 실사용이 비선형으로 늘었습니다. compressed oops 관점에서 무슨 일이 일어난 겁니까? 같은 상황에서 ZGC라면 왜 이 절벽이 없을까요?

5. **노드 크기 선택** — 같은 총 자원을 작은 노드 여러 대와 큰 노드 몇 대로 살 때, `system-reserved`·`kube-reserved` 고정 상납분과 빈패킹 파편(마지막 짐이 안 들어가는 낭비)이 각각 어떻게 달라집니까?

6. **밀도 회귀 감시** — 배포 후 "노드당 파드 수가 줄었다"를 어떤 지표로 잡겠습니까? requests 합계 대비 Allocatable, 노드별 파편률, 힙 설정 변경으로 인한 헤드룸 증가 중 무엇을 대시보드에 올리겠습니까?

## 핵심 요약

- **스케줄러는 `requests`로 채우고 커널은 `limits`로 죽입니다.** 두 숫자가 서로 다른 주체를 향하고, 요청과 실사용의 간극이 곧 낭비된 노드입니다
- 파드가 나눠 갖는 건 노드 물리값이 아니라 **Allocatable**입니다. system-reserved·kube-reserved·eviction-threshold가 먼저 빠지므로, 노드가 작을수록 상납분 비율이 커집니다
- **같은 JVM을 세 가지로 재면 순위가 전부 다릅니다.** 예약으로는 ZGC(9.35GiB)가, JVM이 커밋한 양으로는 G1(83.0MiB)이, **커널이 실제로 청구하는 `memory.current`로는 다시 ZGC(77.9MiB)** 가 1위입니다. **파드를 죽이는 건 세 번째 숫자입니다**
- **ZGC의 힙은 `memfd:java_heap`으로 매핑된 공유 메모리**라서, cgroup 회계에서 `anon`이 아니라 **`shmem`** 에 잡힙니다. ZGC 파드에서 `anon`을 보고 힙을 찾으면 안 나옵니다. 게다가 `shmem`은 `file` 기둥에 있지만 **버릴 수 없습니다**
- 그래서 "ZGC는 메모리를 많이 쓴다"는 통념 자체는 **참**입니다. 다만 흔히 대는 이유(VSZ 9GB)는 **착시**입니다. 진짜 이유는 **compressed oops 미지원**(참조가 4바이트가 아니라 8바이트)과 동시 수집용 여유입니다. 근거 없이 떠도는 "고정 3%" 수치는 싣지 않았습니다
- **힙 밖 고정비는 힙 크기와 무관합니다.** `-Xmx`를 8배 키워도 청구액은 52~54MiB에서 안 움직입니다. 힙에 비례하는 세금이 아니라 **프로세스마다 내는 인두세**입니다
- **아무 일도 안 하는 프로세스의 청구액**: Rust **2.3MiB** · Go **4.8MiB** · Java(G1) **41.2MiB** · Java(ZGC) **77.9MiB**. [3편](/essays/runtime-weight-zero-cost-vs-fat)에서 말한 "런타임의 무게"가 컨테이너에서 청구되는 실제 금액입니다
- 그래서 **파드를 쪼갤 때마다 이 고정비가 곱해집니다.** 파드 10개면 Java 412MiB · Go 48MiB · Rust 23MiB입니다. 밀도를 높이려는 행위가 밀도를 갉아먹는 지점이고, **사이드카 프록시가 Go나 Rust로 쓰인 이유**이기도 합니다
- 쿠버네티스 스케줄러의 기본은 빈패킹이 **아닙니다**(`LeastAllocated`는 분산). 촘촘히 채우려면 `MostAllocated`를 명시해야 합니다
- 밀도를 높이는 세 손잡이는 각각 **GC CPU·안정성·복잡도**를 대가로 요구합니다. **공짜 밀도는 없습니다**
