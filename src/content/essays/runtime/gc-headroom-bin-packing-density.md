> **시리즈 "커널과 런타임으로 톺아보는 Rust · Go · Java"의 19편**
> [18편](/essays/container-memory-accounting-cgroup)에서 커널의 메모리 회계 규칙을 확인했습니다. 커널은 힙이 아닌 페이지 단위로 메모리를 계량하며, 한도를 초과하면 프로세스를 강제 종료합니다
> 그 과정에서 런타임이 요구하는 **메모리 여유(headroom)** 를 여러 차례 확인했습니다. Go는 `GOGC`가 만드는 라이브 객체의 2배, JVM은 힙 외 여백입니다
> 이번 편에서는 이 여유가 **실제 비용으로 어떻게 이어지는지** 확인합니다. **GC가 남겨 둔 여유는 파드의 요구 크기가 되고, 이 크기가 노드 밀도를, 밀도가 인프라 비용을 결정합니다**

## 빈패킹(bin packing) — 스케줄러는 요청량 기준으로 배치합니다

**빈패킹(bin packing)** 은 제한된 용량의 컨테이너에 항목을 최대한 빈틈없이 배치해 필요한 컨테이너 수를 최소화하는 문제입니다

여기서 상자는 노드, 물건은 파드에 해당합니다. 클라우드에서는 노드 단위로 과금되므로 **빈패킹 밀도가 곧 비용**이 됩니다

핵심은 스케줄러가 파드의 크기를 어떤 기준으로 판단하는가입니다

![The scheduler packs by declared requests, not by actual usage](/diagrams/gc-headroom-bin-packing-density-1.svg)

그림 왼쪽이 노드이고, 안에 파드들이 쌓여 있습니다

각 파드는 두 겹의 영역으로 표시됩니다. **바깥쪽 점선 영역**이 파드가 선언한 `requests`이고, **안쪽 색칠된 영역**이 실제 사용량입니다

**스케줄러는 바깥쪽 점선(`requests`)만을 기준으로 동작합니다.** 2 CPU를 요청하고 실제로는 200m만 사용하는 파드도, 스케줄러 관점에서는 2 CPU를 점유한 워크로드로 취급됩니다

따라서 `requests`를 과다 산정하면 실제로는 여유 자원이 있어도 스케줄러는 노드가 포화 상태라고 판단하여 다음 파드를 다른 노드로 배치합니다. **요청량과 실사용량의 간극이 곧 노드 자원의 낭비로 이어집니다**

그림 오른쪽 위의 회색 영역이 이 간극에 해당합니다. 노드가 실제로는 절반가량 비어 있어도 스케줄러는 추가 배치를 하지 못합니다

### 노드 할당 가능 자원(Allocatable)과 고정 공제

파드가 할당받는 기준은 노드의 물리 메모리 전체가 아니라 **Allocatable** 자원입니다

물리 메모리(capacity)에서 다음 세 영역이 먼저 공제됩니다. **system-reserved**(OS·커널·sshd 등 호스트 시스템 몫), **kube-reserved**(kubelet 및 컨테이너 런타임 몫), **eviction-threshold**(축출 방지를 위한 여유분)입니다

이 공제 후 남은 용량이 Allocatable이며, 파드는 이를 `requests` 기준으로 나누어 사용합니다

32GiB 메모리의 노드라도 그 전부를 파드에 배분할 수는 없습니다. 통상 수백 MiB에서 1GiB가량이 먼저 차감됩니다

따라서 노드 규격 선택 자체가 비용에 영향을 미칩니다. **노드가 작을수록 이 고정 공제분이 차지하는 비율이 커집니다.** 동일한 총량을 소형 노드 여러 대로 구성하면 이 공제분을 그만큼 여러 번 부담하는 셈입니다

### 압축 가능한 자원과 그렇지 않은 자원, 다시 — 스케줄링 관점

[18편](/essays/container-memory-accounting-cgroup)에서 다룬 자원의 비대칭은 빈패킹 관점에서 다음과 같이 작용합니다

CPU는 `requests` 기준으로 스케줄링되며, 실사용이 초과되어도 스로틀링으로 처리 속도만 지연될 뿐 프로세스가 종료되지는 않으므로 **오버커밋 허용도가 상대적으로 높습니다**

반면 메모리는 한도를 초과하면 OOMKilled가 발생하므로 오버커밋의 위험이 큽니다

결국 **"메모리 `requests`를 얼마나 타이트하게 설정할 수 있는가"** 가 빈패킹 밀도와 서비스 안정성을 동시에 좌우하는 핵심 지표가 됩니다

## GC 여유 공간에 따른 파드 요구 크기 증가

동일한 라이브 집합을 처리하더라도 언어 런타임과 GC 종류에 따라 필요한 컨테이너 상한이 달라집니다. 이 차이가 **런타임 오버헤드**를 구성합니다

### Go — `GOGC=100`이 만드는 여유 공간

[18편](/essays/container-memory-accounting-cgroup) 실측에서 라이브 집합이 300MiB인 Go 서비스의 다음 GC 목표가 **562MiB**로 설정됨을 확인했습니다

`GOMEMLIMIT`을 명시하지 않으면 이 2배가 요구 메모리의 기본값이 되며, 여기에 고루틴 스택과 런타임 오버헤드가 추가됩니다

요구량을 줄이려면 `GOMEMLIMIT`을 라이브 집합에 가깝게 설정하면 됩니다. 다만 18편에서 확인했듯 **GC 횟수는 7회에서 435회로, GC CPU 점유율은 0%에서 6%로** 증가합니다

즉, **파드 메모리 크기와 GC CPU 사용량을 맞바꾸는 구조**입니다

### Java G1 — 힙 내부 여유 공간과 힙 외 메모리

G1의 여유 공간은 다음과 같이 구성됩니다

**`G1ReservePercent = 10`**: [15편](/essays/java-g1-region-full-gc-cliff)에서 다룬 객체 이동(evacuation) 실패를 방지하기 위해 기본적으로 힙의 10%를 완충 공간으로 비워둡니다. 실사용 가능한 힙은 90% 수준입니다

**카드 테이블(card table)**: 세대 간 참조 추적용 자료구조로 힙 512바이트당 1바이트를 상시 점유합니다(힙의 약 **0.2%**)

**Remembered Set**: region 간 참조를 기록하며, 포인터 참조가 많은 워크로드일수록 사용량이 증가합니다

여기에 추가로 **힙 외(off-heap) 메모리**가 발생합니다

### 유휴 상태 JVM의 힙 외 메모리 실측

`-Xmx`로 힙을 고정한 상태에서 각 GC가 힙 외부에 어느 정도의 메모리를 사용하는지 실측했습니다. 테스트 애플리케이션은 클래스 하나를 로드하고 200ms 대기 후 종료하는 유휴 상태의 프로그램입니다

```bash
docker run --rm --memory=1g eclipse-temurin:25-jdk \
  java -XX:+UseG1GC -Xmx512m -Xms16m \
       -XX:NativeMemoryTracking=summary \
       -XX:+UnlockDiagnosticVMOptions -XX:+PrintNMTStatistics Idle
```

JVM의 자체 프로파일링 도구인 **NMT(Native Memory Tracking)** 를 통해 힙 외부 메모리를 범주별로 확인할 수 있습니다

### 측정 기준에 따라 달라지는 순위

NMT 기준으로 측정한 결과는 다음과 같습니다

| GC | NMT 힙 밖 committed | 총 reserved |
|---|---|---|
| Serial | 24.3 MiB | 1.84 GiB |
| Parallel | 44.4 MiB | 1.86 GiB |
| **G1**(기본) | **65.0 MiB** | 1.89 GiB |
| ZGC | 26.4 MiB | **9.35 GiB** |

지표상으로는 **ZGC의 힙 외부 커밋 메모리(26.4MiB)가 G1(65.0MiB)보다 작게 나타납니다.** ZGC의 9.35GiB라는 큰 수치는 18편에서 확인했듯 커널이 실제로 청구하지 않는 예약된 가상 주소 공간입니다

다만 **이 지표는 커널이 파드를 강제 종료할 때 기준으로 삼는 값과는 다릅니다**

동일 조건에서 커널이 실제로 청구하는 값, 즉 `memory.current`를 함께 확인합니다

```bash
docker run --rm --memory=1g eclipse-temurin:25-jdk sh -c '
  java -XX:+UseZGC -Xmx512m -Xms16m Idle &
  sleep 1.8
  cat /sys/fs/cgroup/memory.current'
```

| GC | reserved (가상) | NMT committed | **memory.current (커널이 청구)** |
|---|---|---|---|
| Serial | 1.84 GiB | 40.3 MiB | 41.2 MiB |
| Parallel | 1.86 GiB | 60.4 MiB | 43.3 MiB |
| G1 | 1.89 GiB | **83.0 MiB** | 52.0 MiB |
| ZGC | **9.35 GiB** | 42.4 MiB | **77.9 MiB** |

**세 열의 순위가 전부 다릅니다**

reserved 기준으로는 ZGC가 압도적으로 크고, committed 기준으로는 **G1이 가장 큽니다.** 그러나 커널이 실제로 청구하는 값 기준으로는 **다시 ZGC가 가장 큽니다**

![Three ways to measure the same JVM, three different rankings](/diagrams/gc-headroom-bin-packing-density-2.svg)

그림에 GC마다 막대가 셋입니다

**옅은 회색**이 예약된 가상 주소 공간, **파란색**이 JVM이 커밋한 양, **붉은색**이 커널이 실제로 청구하는 `memory.current`입니다

ZGC의 회색 막대가 두드러지게 크지만, **붉은 막대 역시 ZGC가 가장 큽니다**

G1의 경우 파란 막대(83.0)가 붉은 막대(52.0)보다 **큽니다.** 커밋되었으나 아직 접촉되지 않은 페이지가 존재한다는 의미입니다

반면 ZGC는 붉은 막대(77.9)가 파란 막대(42.4)보다 큽니다. **NMT가 집계하지 않는 영역을 커널이 청구하고 있음을 의미합니다**

### ZGC 메모리 회계의 특성 — shmem

차이의 원인을 확인하기 위해 `memory.stat`을 분석합니다

```text
G1     청구 51.9 = anon 49.0 + file  0.0 + 커널 2.9
ZGC    청구 77.9 = anon 58.8 + file 16.0 + 커널 3.1
```

ZGC 환경에서만 **`file` 지표에 16.0MiB**가 점유되어 있습니다

`file`은 페이지 캐시 영역입니다. 파일 읽기 작업이 없는 프로그램임에도 페이지 캐시가 점유된 것입니다

프로세스의 메모리 매핑(`/proc/$PID/maps`)을 확인하면 이유를 알 수 있습니다

```bash
grep -iE 'memfd|deleted' /proc/$PID/maps
```

```text
40000000000-40001000000 rw-s 00000000 00:01 3102   /memfd:java_heap (deleted)
```

**ZGC의 힙은 `memfd:java_heap`이라는 공유 익명 파일로 매핑되어 있습니다.** 권한 표시 `rw-s`의 **`s`는 shared(공유)** 를 의미하며, 사설(private) 매핑이 아닙니다

따라서 cgroup 회계에서는 이 페이지들을 `anon`이 아닌 **`shmem`** 항목으로 분류합니다. `memory.stat`을 다시 확인하면 정확히 일치합니다

```text
anon  61644800
file  16809984
shmem 16777216      ← 초기 힙 크기(16 MiB, -Xms16m)와 정확히 일치
```

`shmem` 값이 **설정된 초기 힙 크기와 정확히 일치합니다.** 우연이 아닙니다

이는 [16편](/essays/generational-zgc-colored-pointer-stall)에서 다룬 colored pointer 방식 때문입니다. ZGC는 포인터 주소 비트를 식별자로 사용하기 위해 힙을 특정 가상 주소 공간(약 **1TiB** 부근)에 배치하며, 이것이 9.35GiB에 달하는 가상 메모리 예약 공간의 정체입니다

이는 실무 모니터링에서 다음과 같은 특성으로 이어집니다

ZGC 파드의 메모리 사용량을 분석할 때 `anon` 지표만 확인하면 실제 힙 사용량을 놓치게 됩니다. **`shmem` 지표를 함께 확인**해야 합니다

18편에서 확인했듯 일반적인 `file`(페이지 캐시)은 메모리 압박 시 커널이 회수할 수 있지만, **ZGC의 `shmem`은 실제 디스크 데이터와 연결되지 않은 익명 파일 매핑이므로 회수할 수 없습니다**

### ZGC가 무거운 실제 이유

정리하면 **"ZGC는 메모리를 많이 사용한다"는 통념 자체는 사실입니다.** 커널이 청구하는 값 기준으로 가장 크기 때문입니다

다만 **일반적으로 언급되는 이유는 정확하지 않은 경우가 많습니다**

`ps`에서 VSZ 9GB를 확인하고 우려하는 것은 **사실과 다릅니다.** 이 9GB는 예약된 가상 공간으로 커널이 집계하지 않으며, OOM Kill의 원인이 되지 않습니다

ZGC가 실제로 무거운 이유는 둘입니다

첫 번째는 [16편](/essays/generational-zgc-colored-pointer-stall)에서 확인한 **compressed oops 미지원**입니다. G1은 `UseCompressedOops = true`인 반면 ZGC는 `false`입니다. 32GiB 이하의 힙에서 G1은 객체 참조를 4바이트로 압축하지만, ZGC는 colored pointer 처리를 위해 주소 비트를 그대로 사용하므로 **8바이트를 유지**합니다. 포인터 참조가 많은 객체 그래프일수록 힙 사용량이 크게 증가합니다

두 번째는 **동시 수집(concurrent GC)을 위한 여유 공간**입니다. 애플리케이션을 중단하지 않고 객체를 이동시키려면, 수집 주기 동안 발생하는 신규 할당을 수용할 여유 공간이 힙 내부에 필요합니다. [16편](/essays/generational-zgc-colored-pointer-stall)에서 다룬 allocation stall은 이 여유가 부족할 때 발생하는 현상입니다

일부 자료에서는 "ZGC가 힙의 약 3%를 고정 오버헤드로 점유한다"는 수치가 언급되나, **명확한 근거를 확인할 수 없어 본문에는 반영하지 않았습니다.** OpenJDK 공식 문서에 따르면 필요한 여유 공간은 할당률과 라이브 집합에 따라 편차가 크며, 통상 peak working set 대비 **15~25%** 수준을 권장합니다

## 파드를 나눌수록 곱해지는 런타임 고정비

추가 실측을 통해 빈패킹에 영향을 미치는 요인을 하나 더 확인합니다

**힙 밖 고정비가 힙 크기와 거의 무관합니다**

힙 상한(`-Xmx`)을 변경하며 커널이 실제로 청구하는 값(`memory.current`)을 측정했습니다. 초기 힙 크기는 `-Xms16m`으로 고정했습니다

| `-Xmx` | `memory.current` |
|---|---|
| 256 MiB | 51.7 MiB |
| 512 MiB | 53.6 MiB |
| 1 GiB | 52.0 MiB |
| 2 GiB | 54.4 MiB |

힙 상한을 **8배로 늘려도 청구량은 52~54MiB 구간에 머무릅니다**

**즉, 힙 외부 메모리의 고정 비용은 힙 크기에 비례하여 증가하는 것이 아니라, 프로세스 실행 단위마다 고정적으로 발생하는 오버헤드입니다**

### 언어별 유휴 상태 고정비 비교

언어 런타임별 유휴 상태 고정비를 비교합니다. 단순 대기 후 종료되는 동일 로직의 프로그램을 세 언어로 구현해, 같은 컨테이너 환경에서 커널이 청구하는 값을 측정했습니다

| 런타임 | `memory.current` (유휴) | Rust 대비 |
|---|---|---|
| **Rust** | **2.3 MiB** | 1배 |
| **Go** | **4.8 MiB** | 2.1배 |
| **Java** (기본 G1) | **41.2 MiB** | **18배** |
| **Java** (ZGC) | **77.9 MiB** | **34배** |

요청을 하나도 처리하기 전, 즉 실질적인 작업이 시작되기 전의 수치입니다

[3편](/essays/runtime-weight-zero-cost-vs-fat)에서 "런타임의 무게"를 다루며 zero-cost와 fat runtime을 구분했습니다. **이 수치가 컨테이너 환경에서 청구되는 실제 비용에 해당합니다**

이 수치는 빈패킹 관점에서 다음과 같은 결과로 이어집니다

![Splitting pods multiplies the per-process fixed cost](/diagrams/gc-headroom-bin-packing-density-3.svg)

그림 왼쪽은 하나의 대형 파드로 운영되는 경우로, 고정비가 **한 번만** 발생합니다

그림 오른쪽은 동일한 힙 총량을 4개의 소형 파드로 분할한 경우로, 힙 총량은 같지만 **고정비가 파드 수만큼 누적됩니다**

계산은 단순합니다. 유휴 상태의 파드를 10개 구동할 경우

**Java는 약 412MiB, Go는 48MiB, Rust는 23MiB**를 사용합니다. 같은 클러스터에서 **389MiB의 차이**가 순전히 런타임 오버헤드에서 발생합니다

**밀도를 높이기 위해 파드를 작게 분할하는 결정이, 오히려 런타임 고정비를 파드 수만큼 곱하는 결과로 이어질 수 있습니다**

마이크로서비스 아키텍처(MSA) 논의에서 자주 간과되는 부분입니다. 서비스를 분할하는 데는 아키텍처적 이유가 충분히 있을 수 있지만, **런타임 오버헤드가 큰 언어에서는 분할할 때마다 고정비가 새로 발생합니다.** Go나 Rust에서는 같은 결정의 비용이 한 자릿수 MiB 수준입니다

사이드카(sidecar) 프록시를 예로 들 수 있습니다. 파드마다 배치되는 프록시가 대부분 Go나 Rust로 작성되는 데는 이유가 있습니다. **수천 개 단위로 실행되는 프로세스의 고정비는 곱셈으로 누적됩니다**

앞선 표의 Serial GC(41.2MiB)와 G1(52.0MiB)을 비교하면, 소규모 파드를 다수 운영하는 클러스터에서는 **GC 선택만으로 파드당 약 10MiB를 절감할 수 있습니다.** 다만 그만큼 정지 시간(pause time)이 늘어나는 트레이드오프가 있습니다

## 밀도 최적화 방안과 트레이드오프

노드 집적 밀도를 높이는 방법은 세 가지이며, 각각 대가가 따릅니다

### 1. 런타임 여유 공간 축소

**Go 환경:** `GOMEMLIMIT` 값을 라이브 집합에 가깝게 설정해 요구 메모리를 줄일 수 있습니다. 대가는 GC CPU 사용량입니다. 18편에서 6%를 확인했으며, 과도하게 낮추면 GC가 끊임없이 실행되는 상태에 이를 수 있습니다

**Java 환경:** `MaxRAMPercentage`를 워크로드에 맞게 조정할 수 있습니다. 다만 18편에서 확인한 **128MiB 최소 기준선**을 고려해야 합니다. 소형 컨테이너는 이미 40~50%를 사용하고 있을 수 있습니다

**G1을 사용한다면 힙을 32GiB 이하로 유지하는 것만으로 실사용량을 줄일 수 있습니다.** compressed oops가 참조를 4바이트로 압축하기 때문입니다. ZGC에는 이 이점이 적용되지 않습니다

**GC 선택 자체가 요구 메모리 크기를 결정하는 요인입니다.** 레이턴시 제약이 엄격하지 않다면 Serial이나 Parallel GC가 G1보다 힙 외 메모리가 가볍습니다. 밀도를 확보하는 대신 레이턴시를 양보하는 방식입니다

### 2. 쿠버네티스 리소스 오버커밋 설정

**`requests` = `limits`(Guaranteed QoS):** 스케줄링 요구치와 OOM Kill 기준선이 일치하여 노드 레벨의 자원 경합이 발생하지 않습니다. 안정성은 높지만 밀도는 낮아집니다

**`requests` < `limits`(Burstable QoS):** 스케줄러가 파드를 작게 인식하므로 노드에 더 많은 파드가 배치되어 밀도가 높아집니다. 다만 여러 파드가 동시에 `limits` 수준의 메모리를 요구하면 노드 메모리가 고갈되어 일부 파드가 OOMKilled 될 수 있습니다

이때 종료 우선순위는 QoS class를 따릅니다. **Guaranteed > Burstable > BestEffort** 순으로 보호됩니다. 즉 **리소스 효율성과 서비스 안정성을 맞교환하는 전략**입니다

### 3. 쿠버네티스 빈패킹 스케줄링 활성화

**쿠버네티스 스케줄러의 기본 할당 전략은 빈패킹이 아닙니다**

기본 점수 산정 전략은 `LeastAllocated`로, 가용 자원이 넉넉한 노드를 선호하여 워크로드를 고르게 **분산**시킵니다

밀집도를 높이려면 스케줄러 프로파일의 `NodeResourcesFit` 플러그인에서 **`MostAllocated`** 또는 **`RequestedToCapacityRatio`** 정책을 명시적으로 활성화해야 합니다

즉 **높은 밀도는 기본값이 아니라 별도의 설정이 필요한 선택 사항입니다.** Karpenter나 Cluster Autoscaler의 노드 통합(consolidation) 기능도 동일한 목표를 사후적으로 달성하는 접근 방식입니다

## 언어별 워크로드 밀도 비교

동일한 라이브 집합을 처리하는 서비스를 세 언어로 구현했을 때의 경향은 다음과 같습니다

| | 유휴 상태 고정비 | 스케줄링 요구 크기 증가 요인 | 비용과 이점(트레이드오프) |
|---|---|---|---|
| **Rust** | **2.3 MiB** | 런타임 없음(할당자 단편화 여유 정도) | 높은 밀도와 예측 가능성을 얻는 대신 **개발 시간**을 지불 |
| **Go** | **4.8 MiB** | `GOGC=100`의 2배(또는 `GOMEMLIMIT` 여유) + 스택 | 간결한 동시성 모델을 얻는 대신 **메모리 여유 또는 GC CPU**를 지불 |
| **Java G1** | **41.2 MiB** | 힙 예비 10% + RSet + 힙 외 메모리 | 검증된 처리량을 얻는 대신 **힙 외 메모리 점유**를 지불 |
| **Java ZGC** | **77.9 MiB** | 압축 참조 없음 + 동시 수집 여유 | **1ms 이하의 초저지연**을 얻는 대신 밀도와 처리량을 지불 |

표를 세로로 보면, **밀도는 결국 무엇을 포기했는지에 대한 기록**임을 알 수 있습니다

Rust는 런타임 여유 공간이 없어 밀도가 가장 높지만, 그 대가를 컴파일 타임과 개발 시간에 **사전에** 지불합니다

ZGC는 레이턴시를 최소화한 대신 요구 메모리가 커져 더 많은 노드를 사용합니다

Go는 그 사이에서 설정값 하나로 밀도와 CPU 사용량을 조정할 수 있습니다

중요한 것은 **"어떤 언어가 저렴한가"가 아니라 "이 클러스터가 무엇을 최적화하는가"** 입니다. 레이턴시 SLO가 엄격한 결제 API라면 ZGC의 낮은 밀도는 타당한 비용이며, 야간 배치 작업이라면 그 여유는 불필요한 낭비가 됩니다

## 트레이드오프 소결

한 문장으로 정리하면 이렇습니다

**GC가 요구하는 메모리 여유는 파드의 리소스 요구량이 되고, 이 요구량이 노드의 빈패킹 밀도를, 밀도가 인프라 비용을 결정합니다**

[17편](/essays/gc-cost-conservation-no-silver-bullet)에서 다룬 삼각 구도는 클라우드 환경에서 다음과 같이 나타납니다

![The cloud triangle: density, safety margin, latency](/diagrams/gc-headroom-bin-packing-density-4.svg)

그림의 세 꼭짓점은 **밀도**(노드 비용), **안전 마진**(오버커밋을 하지 않을수록 높아짐), **레이턴시**입니다

**Rust**는 밀도 꼭짓점에 가장 가깝습니다. 개발 시간을 사전에 지불한 결과입니다

**ZGC**는 레이턴시 꼭짓점에 치우쳐 있으며, 그만큼 밀도와 처리량에서는 멀어집니다

**Guaranteed(`requests` = `limits`)** 는 안전 마진 꼭짓점에 해당하며, 밀도를 양보한 결과입니다

밀도를 높이는 세 가지 방법 — GC 여유 공간 축소, 오버커밋, 빈패킹 스케줄러 — 은 각각 **CPU, 안정성, 복잡도** 중 하나 이상을 대가로 요구합니다

이 모든 계산의 바탕에는 18편에서 다룬 전제가 깔려 있습니다. **파드 리소스 설정을 아무리 정밀하게 조율해도, 커널이 합산하는 메모리는 힙에 국한되지 않으므로 힙 외 여백이 부족하면 결국 프로세스가 종료됩니다**

이 문제를 다음 편에서 자세히 다룹니다

## 더 파고들 질문

1. **`requests` 산정 기준** — 메모리 `requests`를 P50 사용량으로 설정할지, P99로 설정할지, GC 여유 공간을 포함한 최대 힙으로 설정할지에 따라 밀도와 OOM 위험이 어떻게 달라집니까? Go에서 `GOMEMLIMIT`을 먼저 정하고 이를 기준으로 `requests`를 산출하는 순서가 자연스러운 이유는 무엇일까요?

2. **서비스 분할 비용 산정** — 이번 편에서 측정한 유휴 상태 고정비(Java 41.2 · Go 4.8 · Rust 2.3 MiB)를 근거로, 하나의 서비스를 파드 1개로 운영할 때와 3개로 분할할 때의 총 메모리를 계산해 보십시오. 어느 지점부터 분할의 이점(격리·독립 배포)이 누적되는 고정비를 상회합니까? 같은 계산을 Go로 하면 결론이 어떻게 달라집니까?

3. **오버커밋 한도 설정** — 메모리 `requests:limits`를 1:1(Guaranteed)로 둘 서비스와 1:1.5(Burstable)로 둘 서비스를 어떤 기준으로 구분하겠습니까? 노드에 Burstable 파드가 밀집한 상황에서 동시에 `limits` 수준으로 증가하는 "상관된 스파이크"를 어떤 지표로 사전에 감지하겠습니까?

4. **Compressed Oops 임계점** — G1을 사용하는 서비스의 힙을 30GiB에서 40GiB로 늘렸더니 실사용량이 비선형적으로 증가했습니다. compressed oops 관점에서 어떤 일이 일어난 것입니까? 동일한 상황에서 ZGC라면 이러한 현상이 나타나지 않는 이유는 무엇입니까?

5. **노드 규격 선택** — 동일한 총 자원을 소형 노드 다수와 대형 노드 소수로 구성할 때, `system-reserved`·`kube-reserved` 고정 공제분과 빈패킹 파편화(마지막 워크로드가 배치되지 못하는 낭비)가 각각 어떻게 달라집니까?

6. **빈패킹 밀도 회귀 감지** — 배포 후 "노드당 평균 파드 수가 감소했다"는 문제를 어떤 지표로 감지하겠습니까? `requests` 합계 대비 Allocatable 여유분, 노드별 파편화 비율, 힙 설정 변경으로 인한 여유 공간 증가 중 무엇을 핵심 지표로 선정하겠습니까?

## 핵심 요약

- **스케줄러는 `requests`를 기준으로 배치하고, 커널은 `limits`를 기준으로 종료 여부를 결정합니다.** 두 지표가 서로 다른 주체를 향하며, 요청량과 실사용량의 간극이 곧 노드 자원의 낭비로 이어집니다
- 파드가 할당받는 것은 노드의 물리 메모리가 아니라 **Allocatable** 자원입니다. system-reserved·kube-reserved·eviction-threshold가 먼저 공제되므로, 노드가 작을수록 이 공제 비율이 커집니다
- **같은 JVM을 세 가지 기준으로 측정하면 순위가 모두 다릅니다.** 예약량 기준으로는 ZGC(9.35GiB), JVM 커밋량 기준으로는 G1(83.0MiB), **커널이 실제로 청구하는 `memory.current` 기준으로는 다시 ZGC(77.9MiB)** 가 가장 큽니다. **파드 종료를 결정하는 것은 세 번째 지표입니다**
- **ZGC의 힙은 `memfd:java_heap`으로 매핑된 공유 메모리**이므로, cgroup 회계에서 `anon`이 아닌 **`shmem`** 으로 분류됩니다. ZGC 파드에서 `anon` 지표만 확인하면 힙 사용량을 놓치게 됩니다. `shmem`은 `file` 항목에 속하지만 **회수할 수 없습니다**
- 따라서 "ZGC는 메모리를 많이 사용한다"는 통념 자체는 **사실입니다.** 다만 흔히 언급되는 이유(VSZ 9GB)는 **사실과 다릅니다.** 실제 원인은 **compressed oops 미지원**(참조가 4바이트가 아닌 8바이트)과 동시 수집을 위한 여유 공간입니다. 근거가 명확하지 않은 "고정 3%" 수치는 본문에 반영하지 않았습니다
- **힙 외 고정비는 힙 크기와 무관합니다.** `-Xmx`를 8배로 늘려도 청구량은 52~54MiB에서 유지됩니다. 힙 크기에 비례하는 비용이 아니라 **프로세스 단위로 고정 발생하는 오버헤드**입니다
- **유휴 상태 프로세스의 청구량**: Rust **2.3MiB** · Go **4.8MiB** · Java(G1) **41.2MiB** · Java(ZGC) **77.9MiB**. [3편](/essays/runtime-weight-zero-cost-vs-fat)에서 다룬 "런타임의 무게"가 컨테이너 환경에서 청구되는 실제 비용입니다
- **파드를 분할할 때마다 이 고정비가 누적됩니다.** 파드 10개 기준 Java 412MiB · Go 48MiB · Rust 23MiB입니다. 밀도를 높이려는 결정이 오히려 밀도를 낮추는 지점이며, **사이드카 프록시가 Go나 Rust로 작성되는 이유**이기도 합니다
- 쿠버네티스 스케줄러의 기본 전략은 빈패킹이 **아닙니다**(`LeastAllocated`는 분산 배치). 밀도를 높이려면 `MostAllocated`를 명시적으로 설정해야 합니다
- 밀도를 높이는 세 가지 방법은 각각 **GC CPU, 안정성, 복잡도** 중 하나 이상을 대가로 요구합니다
