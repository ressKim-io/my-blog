> **시리즈 "Rust와 함께 가는 CS 레이어"의 5편**
> Rust 공식 홈페이지에서 우리를 가장 먼저 반겨주는 문구, **"비용 없는 추상화(Zero-cost abstraction)"**. 이 우아한 말의 *진짜 실체*를 깨달으려면, 컴파일러가 우리가 짠 추상화 코드를 기계어로 번역할 때 도대체 무엇을 어떻게 걷어내는지 그 밑바닥을 들여다봐야 합니다.
> 똑같은 Trait을 쓰는데도 정적 디스패치(제네릭)냐, Trait Object(`dyn`)냐에 따라 생성되는 기계어는 하늘과 땅 차이입니다. 그리고 이 갈림길이 4편에서 슬쩍 예고했던 무시무시한 'Noalias 최적화'를 살리기도 하고, 처참하게 죽이기도 합니다.

## 🤔 "제네릭은 빠르고 dyn은 느리다"의 진짜 이유

Rust를 현업에서 조금 굴려보신 분들이라면 일종의 불문율 같은 경험칙을 하나 아실 겁니다.
*"성능이 쥐어짜져야 하는 핫 패스(Hot path)는 무조건 제네릭으로 짜고, 런타임 유연성이 필요할 때만 `dyn`을 써라."*

그런데 면접이나 코드 리뷰에서 **"왜 제네릭이 더 빠르죠?"**라고 물어보면, 십중팔구 **"동적 디스패치는 포인터를 타고 가는 간접 호출(Indirect Call) 비용이 한 번 더 들기 때문입니다"**라고 대답합니다.

절반만 맞는 정답입니다. 진짜 치명적인 성능 저하의 원인은 간접 호출 그 자체가 아닙니다. 진짜 이유는 바로 그 호출 경계에서 **컴파일러의 모든 강력한 최적화가 일제히 멈춰버린다**는 데 있습니다.
이 흐름을 끝까지 따라가다 보면 "왜 어떤 API는 굳이 제네릭만 강제하는지", "왜 핫 루프(Hot loop)에서 `dyn`을 썼더니 SIMD 벡터화가 박살 나는지", "왜 내 Rust 프로젝트는 빌드가 드럽게 느리고 바이너리는 산더미만 한지"가 하나의 선으로 쫙 연결됩니다.

출발점은 '똑같은 Trait이 두 갈래로 번역된다'는 사실입니다.

## 🧩 한 Trait, 두 갈래의 길

우리가 만든 `Sink`라는 Trait이 있다고 해봅시다. 이걸 함수의 인자로 받는 방식에 따라 디스패치(Dispatch) 전략이 완전히 갈립니다.

```rust
// 1. 제네릭 Bound → 정적 디스패치 (Monomorphization)
fn process<T: Sink>(s: T) { ... }

// 2. Trait Object → 동적 디스패치 (vtable)
fn process(s: &dyn Sink) { ... }
```

위쪽 코드는 컴파일 타임에 타입을 완전히 박아 넣어버리고, 아래쪽 코드는 런타임에 vtable(가상 함수 테이블)을 타고 넘어가며 호출합니다. 둘의 기계어와 최적화 결과는 어떻게 다를까요?

## 🏗️ 단형화(Monomorphization) — 타입별로 도장을 찍어내다

만약 `fn process<T: Sink>(s: T)`라는 제네릭 함수에 `FileSink`와 `KafkaSink` 두 가지 타입을 밀어 넣으면 무슨 일이 벌어질까요?

컴파일러는 **타입별로 함수의 복사본을 따로따로 찍어냅니다.** 마치 쿠키 틀로 반죽을 찍어내듯, 원본 코드는 하나지만 컴파일이 끝나면 `process_FileSink`, `process_KafkaSink`처럼 타입이 하드코딩된 여러 개의 함수가 생성됩니다. (C++의 템플릿과 완벽히 같은 방식입니다.)

각각의 복사본은 어떤 타입을 호출할지 컴파일 타임에 100% 확정되어 있습니다. 그래서 코드가 통째로 **인라인(Inline) 확장**될 수 있고, 그 타입에 딱 맞는 LLVM의 극한 최적화가 전부 적용됩니다.

대가는 혹독합니다. 바로 코드 팽창(Code Bloat)입니다. 만약 N개의 타입이 M개의 제네릭 함수를 호출한다면? 순식간에 N × M개의 복사본이 생깁니다. 바이너리 용량은 터져나가고, 컴파일 시간은 하염없이 늘어나며, CPU의 Instruction Cache(I-Cache)를 압박해 오히려 성능이 떨어질 수도 있습니다. 우리가 겪는 "Rust 빌드 속도 지옥"의 주범이 바로 이 녀석입니다.

## 📦 Trait Object와 vtable — 뚱뚱한 포인터 (Fat Pointer)

반면 `&dyn Sink`나 `Box<dyn Sink>`는 C언어의 평범한 포인터가 아닙니다. 크기가 무려 16바이트(두 워드)에 달하는 뚱뚱한 포인터(Fat Pointer)입니다.

첫 번째 8바이트는 실제 객체의 데이터 주소를 가리키고, 두 번째 8바이트는 vtable(가상 함수 테이블)을 가리킵니다. vtable은 (타입, Trait) 쌍마다 메모리에 딱 하나씩 정적으로 만들어지는 읽기 전용 테이블입니다. 여기엔 메모리를 해제하는 소멸자(`drop_in_place`), 크기, 메모리 정렬(Alignment), 그리고 Trait에 정의된 메서드들의 실제 함수 포인터들이 빽빽하게 담겨 있습니다.

![A &dyn Sink fat pointer and the vtable it points to](/diagrams/static-vs-dynamic-dispatch-1.svg)

그래서 `dyn`의 메서드 호출은 구조적으로 '이중 간접 참조 + 간접 호출'입니다.
포인터를 타고 가서 vtable 주소를 꺼내고 → vtable 안에서 메서드 주소를 찾은 다음 → 그 주소로 점프(`call rax`)해야 합니다.

*"아니, 이렇게 느리고 복잡한 걸 왜 씁니까?"*
코드 복사본이 단 **한 벌**만 생성되기 때문입니다. 더 중요한 건 `Vec<Box<dyn Sink>>`처럼 **서로 다른 타입들을 하나의 컬렉션(배열)에 욱여넣을 수 있다는 것**입니다. 런타임에 동적으로 조립되는 이종(Heterogeneous) 집합을 다루거나 플러그인 아키텍처를 만들 때, 컴파일 타임에는 타입을 알 수 없기 때문에 제네릭(단형화)은 아예 불가능합니다. 이때 `dyn`이 유일하고도 우아한 해답이 됩니다.

## 💡 진짜 악당은 간접 호출이 아니라 "인라인(Inline) 상실"이다

자, 이제 흔한 오해를 박살 낼 시간입니다. "동적 디스패치는 간접 호출이 한 번 더 발생해서 약간 느리다"라는 말은 현대 CPU 아키텍처에서는 반쯤 틀린 말입니다. 요즘 CPU의 분기 타깃 예측기(Branch Target Predictor)는 미친 듯이 똑똑해서 이런 뻔한 간접 호출 레이턴시는 기가 막히게 숨겨버립니다. 진짜 무서운 비용은 '최적화 장벽(Optimization Barrier)'입니다.

![Static dispatch inlines and unlocks the optimizer; dynamic dispatch blocks it at the call](/diagrams/static-vs-dynamic-dispatch-2.svg)

정적 디스패치(제네릭)는 컴파일러가 호출될 코드를 훤히 꿰뚫어 봅니다. 그래서 함수를 아예 호출부 내부로 욱여넣어 버립니다(Inline). 호출 경계라는 벽이 허물어지는 순간, 컴파일러의 최적화 파이프라인(GVN, LICM, Noalias, Vectorization)이 코드를 가로지르며 미쳐 날뛰기 시작합니다.

반면 동적 디스패치의 `call rax` 명령어는 컴파일러 입장에서 '안이 전혀 보이지 않는 블랙박스'입니다. 런타임에 어디로 튈지 모르니 인라인 확장은 꿈도 못 꾸고, 그 호출 경계를 기점으로 모든 최적화 분석이 강제로 멈춰버립니다.

결론적으로 정적 디스패치가 압도적으로 빠른 이유는 "간접 호출 2 사이클을 아껴서"가 아니라, "인라인 확장이 그 뒤에 숨어있던 수많은 LLVM 최적화의 빗장을 전부 열어젖혀 주기 때문"입니다. 특히 데이터가 쏟아지는 핫 루프 안에서 `dyn`을 쓰면, 원소마다 타입이 다를 수 있어 CPU의 필살기인 SIMD(단일 명령어 다중 데이터 처리) 벡터화가 완전히 박살 납니다.

> **💡 시니어의 빌드 팁: 크레이트(Crate) 경계와 인라인**
> 제네릭 함수와 `#[inline]` 매크로가 붙은 함수는 MIR(Mid-level IR) 형태로 라이브러리 메타데이터에 실려 배포되기 때문에, 이걸 가져다 쓰는 다른 크레이트에서도 인라인 확장이 가능합니다. 하지만 평범한 함수는 이 메타데이터가 없어서 **LTO(Link Time Optimization)를 켜지 않으면 크레이트 경계를 넘어 인라인되지 않습니다.**

## ⚡ `&mut T`가 미치도록 빠른 이유 — Noalias의 마법

4편에서 "컴파일러 최적화와 메모리 오더링"을 이야기하며 잠시 미뤄뒀던 떡밥을 회수하겠습니다.
Rust의 `&mut T`(가변 참조자)는 타입 시스템 레벨에서 "이 데이터를 가리키는 포인터는 전 우주에 나 하나뿐이다"라는 독점권을 완벽하게 보장합니다.

컴파일러는 이 강력한 보증 수표를 LLVM에게 `noalias`라는 태그로 달아서 넘겨줍니다.
*"LLVM아, 이 포인터가 가리키는 메모리는 코드 실행 중에 절대 다른 놈이 몰래 안 건드리니까, 마음 놓고 최적화해!"*

이 약속 하나가 풀어주는 최적화는 어마어마합니다.

1. **GVN (중복 로드 제거):** 메모리에서 똑같은 값을 두 번 읽어야 할 때, 중간에 누가 값을 안 바꿨다는 확신이 있으니 처음 한 번만 읽고 레지스터에 캐싱해서 씁니다.
2. **LICM (루프 불변 코드 이동):** 루프 안에서 값이 변하지 않는 로드(Load) 연산을 루프 바깥으로 시원하게 끌어올려 버립니다.
3. **SIMD 벡터화:** 배열 원소들 사이에 메모리 간섭(Aliasing)이 없다는 걸 확신하니 병렬로 묶어버립니다.

C나 C++에서는 포인터들이 같은 메모리를 가리킬 가능성(Aliasing) 때문에 컴파일러가 늘 방어적으로 메모리를 매번 다시 읽어야 합니다. Rust는 언어의 근간인 Borrow Checker 규칙이 이 가정을 타입으로 증명해 줍니다. (실제로 Rust가 LLVM의 `noalias`를 너무 하드코어하게 쓴 나머지, 그동안 아무도 몰랐던 LLVM 내부의 `noalias` 버그들을 줄줄이 터뜨리며 한동안 이 기능을 껐다 켰다 했던 웃지 못할 역사도 있습니다.)

결국 "제네릭이 Zero-cost다"라는 말의 진짜 실체는 "추상화가 컴파일 과정에서 인라인되어 완전히 흔적도 없이 사라져 버리고, 극한으로 튜닝된 기계어만 남는다"는 뜻입니다.

## 🔀 제3의 선택지 — Enum Dispatch

성능(정적)과 유연성(동적) 사이에서 고민하는 개발자를 위한 실용적인 제3의 길이 있습니다. 만약 처리해야 할 타입의 종류가 무한하지 않고 *유한하게 닫혀있다면*, 쿨하게 Enum으로 묶어버리는 겁니다.

```rust
enum AnySink {
    File(FileSink),
    Kafka(KafkaSink)
}
// match 문으로 분기 → 정적 디스패치 (vtable도 힙 할당도 없음!)
```

런타임에 `match` 문으로 분기하기 때문에 컴파일러가 안을 들여다볼 수 있고, 당연히 정적 디스패치라 인라인 최적화가 폭발합니다. 생태계의 `enum_dispatch` 크레이트가 이 지루한 보일러플레이트 코드를 자동화해 줍니다.

트레이드오프는 명확합니다. `dyn`은 나중에 외부에서 타입을 무한히 추가할 수 있는 *열린(Open)* 생태계지만 최적화를 손해 보고, Enum은 내가 정의한 타입만 쓸 수 있는 *닫힌(Closed)* 생태계지만 압도적으로 빠릅니다. (단, Enum은 내부 구조체 중 **가장 뚱뚱한 녀석의 크기로 전체 사이즈가 고정**되기 때문에 메모리를 약간 더 먹을 수 있습니다). 백엔드 서버의 Telemetry 파이프라인처럼 "처리할 이벤트 타입이 딱 정해져 있는" 워크로드라면 `dyn`보다 Enum Dispatch가 정답입니다.

## 🚧 Object Safety — 왜 아무 Trait이나 dyn이 될 수 없을까?

Rust를 쓰다 보면 특정 Trait을 `Box<dyn MyTrait>`으로 만들려는데 컴파일러가 "Object Safe 하지 않다"며 뺨을 때리는 경우가 있습니다. 그 이유는 철저하게 앞서 배운 **vtable의 물리적 한계** 때문입니다.

* **제네릭 메서드가 있는 경우 (`fn foo<U>(&self)`):** 제네릭은 타입별로 함수 복사본을 무한대로 찍어내야 하는데, 크기가 고정된 vtable 안에 무한대의 함수 포인터를 박아 넣을 수는 없습니다.
* **`Self`를 반환하는 메서드가 있는 경우:** `dyn`은 런타임에 구체적인 타입이 지워진 상태인데, 메서드가 `Self`(나 자신)를 반환해 버리면 호출자 입장에선 리턴되는 메모리 사이즈를 알 길이 없어 스택을 짤 수 없습니다.

"왜 어떤 라이브러리 API는 무조건 제네릭만 강제하나요?"의 답이 바로 이것입니다. 라이브러리 제작자가 Trait을 설계할 때 `dyn`을 허용할지 말지(Object Safety를 지킬지) 자체가 아주 중요한 아키텍처적 결단입니다.

## 🎛️ 빌드 노브 (Knob) — 컴파일러를 쥐어짜는 다이얼

컴파일러가 코드를 생성하는 방식은 `Cargo.toml`의 `[profile.release]` 세팅으로 튜닝할 수 있습니다.

| 노브 (옵션) | 효과 | 트레이드오프 |
| --- | --- | --- |
| **`opt-level`** | 0/1/2/3 = 속도 중심, s/z = 크기 중심 | `z`는 바이너리가 작아지는 대신 런타임 속도를 희생합니다. |
| **`lto`** | 크레이트 경계를 무시하고 통째로 인라인 병합 (thin/fat) | 바이너리 크기↓ 런타임 성능↑ / 대신 **컴파일 시간 대폭발** |
| **`codegen-units`** | 크레이트를 N조각으로 쪼개서 병렬로 컴파일 | 기본값은 16. `1`로 설정하면 병렬 컴파일을 포기하는 대신 최적화 한계 돌파. |
| **`panic`** | `"abort"`로 설정 시 Unwinding 테이블 삭제 | 바이너리 크기 급감. (6편의 FFI와 깊게 연결되는 설정입니다) |

여기서 주목할 건 `codegen-units`입니다. 컴파일 속도를 높이려고 기본적으로 코드를 16조각으로 찢어서 병렬로 번역하는데, 이렇게 조각이 나뉘면 **조각들 사이를 가로지르는 인라인 최적화가 끊어집니다.**
만약 프로덕션 환경에서 영혼까지 끌어모은 성능이 필요하다면 `codegen-units=1`에 `lto="fat"`을 줍니다. CI 파이프라인에서 "개발용 빠른 빌드"와 "배포용 극한 빌드"를 가르는 가장 핵심적인 다이얼입니다.

## 🛠️ 실무 모니터링: 진짜 눈으로 확인해 보기

내 바이너리를 뚱뚱하게 만든 주범이 어떤 제네릭인지, 기계어가 정말 다르게 나오는지 직접 사냥해 볼 수 있습니다.

```bash
# 함수별로 생성된 LLVM IR 줄 수 확인 — 제네릭 코드 팽창의 주범 찾기
cargo llvm-lines | head -20

# 컴파일된 바이너리에서 가장 용량을 많이 차지하는 함수/크레이트 색출
cargo bloat --release --crates
```

[Godbolt (Compiler Explorer)](https://godbolt.org)에 접속해서 `process<T>` 함수를 제네릭 버전과 `&dyn` 버전으로 각각 컴파일해 보세요. 제네릭은 호출부가 인라인되어 싹 사라진 깔끔한 코드가 나오고, `dyn`은 `call rax` 명령어가 덩그러니 남는 걸 두 눈으로 확인할 수 있습니다.
실무에서 `cargo llvm-lines`를 쳐봤더니 *"바이너리 팽창의 80%가 저 제네릭 함수 3개 때문이네?"*라는 데이터가 나오면, 성능을 약간 양보하더라도 그걸 `dyn`으로 바꿔서 컨테이너 이미지 크기와 콜드 스타트(Cold Start) 시간을 극적으로 줄일 수 있습니다. (수천 대의 K8s 노드에 배포되는 관측 에이전트 같은 경우 바이너리 크기가 곧 돈이고 성능이니까요.)

## 📌 오늘의 핵심 요약

* 같은 Trait이라도 제네릭(정적 디스패치, Monomorphization)과 `dyn`(동적 디스패치, vtable)에 따라 생성되는 기계어 생태계가 완전히 달라집니다.
* 제네릭은 타입별로 함수를 도장 찍듯 복제해 성능을 극대화하지만, 그 대가로 끔찍한 코드 팽창(Code Bloat)을 가져옵니다.
* 동적 디스패치의 진짜 치명적인 비용은 간접 호출이 아니라 "인라인 최적화 상실"입니다. 호출 경계에서 Noalias, GVN, LICM, 벡터화가 전부 멈춰버립니다.
* Rust의 `&mut T`가 미치도록 빠른 이유는 겹치지 않는다는 보장(**Noalias**)이 LLVM의 최적화 빗장을 전부 열어주기 때문입니다.
* 타입이 유한하다면 **Enum Dispatch**가 최고의 가성비를 내며, `codegen-units`와 `LTO`는 빌드 속도와 런타임 성능을 교환하는 마법의 다이얼입니다.
