> **시리즈 "Rust와 함께 가는 CS 레이어"의 4편**
> Rust가 자랑하는 '두려움 없는 동시성(Fearless Concurrency)'이 멀티스레드 버그를 *무엇을* 막아주는지는 튜토리얼에서 흔히 배웁니다. 하지만 이번 편에서는 그게 밑바닥에서 *어떻게* 동작하는지를 파헤칩니다.
> 1편에서 보았던 CPU의 하드웨어적 재배치(Store Buffer)와 5편에서 다룰 컴파일러의 소프트웨어적 최적화, 이 두 마리 통제 불능의 괴물을 동시에 길들이는 강력한 계약이 바로 **Memory Ordering(메모리 오더링)**입니다. `Send`와 `Sync`가 "스레드 간에 무엇을 넘겨도 되나"를 따지는 문지기라면, Ordering은 "그 넘겨준 데이터를 언제, 어떻게 동기화할 것인가"를 결정하는 교통경찰입니다.

## 🤔 "ready가 true인데, data가 0이라고요? 그게 말이 됩니까?"

멀티스레드 프로그래밍을 하다 보면 정말 귀신에 홀린 듯한 버그를 만나게 됩니다. 아주 전형적인 메시지 전달(Message Passing) 코드를 한 번 보겠습니다. 스레드 A가 데이터를 채워 넣고 다 했다는 깃발을 올리면, 스레드 B는 그 깃발을 보고 데이터를 읽어갑니다.

```rust
// Thread A (Producer)        // Thread B (Consumer)
data = 42;                    while !ready {}
ready = true;                 print(data);   // 당연히 42가 나오겠지?
```

우리의 순진한 직관은 이렇습니다. *"B가 `ready == true`를 보고 루프를 빠져나왔다면, A가 그보다 먼저 쓴 `data = 42`도 당연히 반영되어 있겠지."* **틀렸습니다.** 실제 환경에서는 B가 `ready == true`를 확인하고도 `data`를 읽었을 때 0(초기값)이 튀어나오는 끔찍한 일이 심심치 않게 벌어집니다. 코드는 위에서 아래로 순서대로 적었지만, 두 변수의 쓰기 순서가 뒤집혀서 실행되거나 다른 코어에 도달하는 가시성(Visibility) 순서가 어긋나버리기 때문입니다.

도대체 왜 이런 황당한 일이 가능한지, 그리고 이걸 어떻게 막을 수 있는지가 이번 4편의 전부입니다. 출발점은 이 쓰라린 명제입니다. **"코드는 위에서 아래로 실행된다는 보장은, 오직 단일 스레드 안에서만 유효하다."**

## 🧩 막 나가는 두 녀석: 컴파일러와 하드웨어

컴파일러와 CPU가 신봉하는 대원칙이 하나 있습니다. 바로 'As-if 규칙'입니다. "결과만 똑같다면, 속도를 위해 코드 실행 순서를 내 맘대로 뒤섞어도 무죄!"라는 무시무시한 규칙이죠. 문제는 멀티스레드 환경에서는 내가 혼자 섞어버린 순서가 *다른 스레드의 눈에는 그대로 적나라하게 노출된다*는 점입니다.

순서를 뒤섞는 범인은 두 명입니다.

1. **컴파일러 (소프트웨어 레벨 - 5편 예정):** `a=1; b=2;`라는 코드를 보고 레지스터를 아끼거나 루프를 최적화(LICM) 한답시고 `b=2; a=1;`로 슬쩍 바꿔버립니다. 싱글 스레드라면 아무도 눈치채지 못하니 합법입니다.
2. **하드웨어 (물리 계층 - 1편 참고):** CPU 코어가 값을 쓸 때 당장 L1 캐시에 쓰지 않고 'Store Buffer'라는 임시 공간에 던져둡니다. 내 코어는 다 썼다고 생각하지만, 다른 코어 입장에선 내 쓰기 작업이 아직 도착하지 않아 순서가 뒤바뀐 것처럼 보입니다.

Memory Ordering은 이 **위아래에서 날뛰는 두 놈의 재배치를 "어디까지 허용하고 어디서부터 막을 것인가"를 명시하는 엄격한 계약**입니다. 한쪽만 막아서는 안 되고, 컴파일러와 CPU 양쪽을 모두 묶어야만 앞서 본 황당한 버그가 사라집니다.

## 🔗 happens-before: 스레드 간 가시성을 잇는 유일한 다리

순서가 꼬이는 걸 막기 위해 컴퓨터 공학에서 쓰는 근본 개념이 'happens-before(발생 전)'입니다.
"연산 A가 연산 B보다 happens-before 관계에 있다면, A가 메모리에 남긴 흔적은 B에게 반드시, 100% 보인다"는 강력한 보장입니다. 이 관계는 딱 두 가지 방법으로만 만들어집니다.

* **sequenced-before:** 같은 스레드 안에서 코드가 쓰인 순서입니다. 당연히 `a=1` 다음에 `b=2`를 썼다면 A가 B보다 먼저 일어납니다.
* **💥 synchronizes-with:** 스레드 *사이*의 관계입니다. 이건 아무 때나 생기지 않고, **Release 쓰기**와 **Acquire 읽기**가 서로 '짝이 맞아서' 값이 전달되는 그 찰나의 순간에만 마법처럼 생겨납니다.

여기서 얻어야 할 가장 중요한 통찰은 "스레드 간의 happens-before 다리는 절대 공짜로 생기지 않는다"는 겁니다. 개발자가 명시적으로 동기화 다리를 놔주지 않으면, 내 스레드가 쓴 값이 남에게 언제 보일지 아무도 책임지지 않습니다. 이게 바로 앞선 예제에서 데이터가 0으로 출력된 이유입니다.

> **💡 잠깐! 데이터 레이스(Data Race)의 정확한 정의**
> ① 같은 메모리 주소를 두고 ② 최소 하나 이상의 스레드가 쓰기(Write)를 시도하는데 ③ happens-before 다리로 교통정리가 안 되어 있고 ④ 둘 다 일반 변수(비원자적)일 때 즉시 Undefined Behavior(UB)가 폭발합니다. Atomic으로 만들면 ④번 조건이 깨지고, Ordering으로 다리를 놓으면 ③번 조건이 깨져서 안전해집니다.

## 🚦 5가지 Ordering과 '단방향 울타리'의 마법

Rust는 `Atomic` 변수에 연산을 할 때마다 아래 5가지 Ordering 중 하나를 요구합니다.

| Ordering | 보장하는 것 | 실무 비유 |
| --- | --- | --- |
| **`Relaxed`** | 원자성만 챙김, 순서 보장은 **0** | 값만 안 깨지면 됨 (단순 카운터용) |
| **`Acquire`** | (Load용) 내 뒤에 있는 코드가 내 **위로 못 올라옴** | **구독 (수신)** |
| **`Release`** | (Store용) 내 앞에 있는 코드가 내 **아래로 못 내려감** | **발행 (송신)** |
| **`AcqRel`** | `fetch_add`처럼 읽고 쓰는 연산용. (Acquire + Release) | 받으면서 동시에 발행 |
| **`SeqCst`** | Acq/Rel 보장에 더해, 전체 시스템의 **단일 순서**를 강제 | 전 코어가 합의한 전역 시계 |

우선 가장 흔한 오해 하나부터 박살 내고 가겠습니다. **"`Relaxed`를 쓰면 값이 깨지거나 데이터 레이스가 나지 않나요?"** 아닙니다. `Relaxed`도 엄연한 원자적(Atomic) 연산입니다. 값이 반으로 찢어져서 읽히는 일(Torn read)은 없습니다. 단지 '순서'를 책임지지 않을 뿐입니다.

Acquire와 Release는 '단방향 울타리(One-way fence)'입니다. 아래 그림을 보면 아주 명확해집니다.

![Acquire/Release as one-way barriers, joined by a synchronizes-with bridge](/diagrams/memory-ordering-acquire-release-1.svg)

* **Release**는 "발행" 버튼입니다. 이 코드 윗줄에서 내가 지지고 볶았던 모든 메모리 작업들을 한 상자에 예쁘게 포장해서 내보냅니다. (위의 코드가 아래로 쏟아져 내리지 못하게 막습니다.)
* **Acquire**는 "구독" 버튼입니다. 발행된 상자를 열어보기 전까지는, 밑에 있는 코드가 함부로 위로 올라와서 미리 실행되는 걸 막아줍니다.

결국 앞서 봤던 버그 코드는, 스레드 A가 `ready`를 `Release`로 저장하고, 스레드 B가 `ready`를 `Acquire`로 읽어내는 순간 완벽하게 해결됩니다. 두 스레드 사이에 **synchronizes-with 다리**가 놓이고, `data = 42`라는 과거의 기억이 B에게 100% 전달되기 때문입니다.

## 🧱 SeqCst는 왜 필요한가? — Store Buffer의 끔찍한 귀환

"Acquire랑 Release만 있으면 완벽한 거 아닌가요?" 아쉽게도 이 둘만으로는 막지 못하는 패턴이 있습니다. 바로 1편에서 우리를 괴롭혔던 **Store Buffer**가 다시 악당으로 등장하는 순간입니다.

```rust
// Thread 1            // Thread 2
x.store(1);            y.store(1);
r1 = y.load();         r2 = x.load();
// 질문: r1 == 0 이고 r2 == 0 인 게 동시에 가능할까요?
```

우리의 직관으로는 "내가 1을 쓰거나, 쟤가 1을 썼을 테니 무조건 둘 중 하나는 1을 읽어야지!" 싶지만, 현실은 냉혹합니다.

![Store buffering — both loads see stale 0 because each store waits in a buffer](/diagrams/memory-ordering-acquire-release-2.svg)

각 스레드가 1을 쓰는 작업(`store`)을 일단 자기 코어의 Store Buffer에 쑤셔 넣고 쿨하게 다음 줄(`load`)을 실행해 버립니다. 두 코어 모두 서로의 값이 캐시에 도달하기도 전에 상대방의 변수를 읽어버려서, 둘 다 옛날 값인 0을 보게 되는 환각에 빠집니다. 이를 StoreLoad 재배치라고 합니다.

이 현상을 막으려면 전체 장벽(StoreLoad Fence)을 내려서 "버퍼 다 비울 때까지 다음 줄 실행하지 마!"라고 강제해야 하는데, 이 역할을 하는 게 바로 `SeqCst` (Sequential Consistency)입니다.

> **💡 시니어의 인프라 팁: AWS Graviton(ARM)으로 이사 갈 때 터지는 폭탄**
> 이 `SeqCst`의 하드웨어 비용이 CPU 아키텍처마다 극단적으로 다릅니다.
>
> | 아키텍처 | 메모리 모델 | 재배치 허용 범위 | Acq/Rel 비용 | SeqCst 비용 |
> | --- | --- | --- | --- | --- |
> | **x86 (Intel/AMD)** | TSO (강함) | StoreLoad만 허용 | 거의 0 (일반 `mov`와 같음) | 스토어에 강제 fence 1방 |
> | **ARM (Graviton, M1)** | WMM (약함) | 4종류 전부 다 허용! | **무거운 Barrier 명령어** | **더 무거운 Full Barrier** |
>
> 실전에서 아주 무서운 포인트가 이겁니다. x86 서버에서는 대충 짜도 하드웨어가 멱살 잡고 순서를 맞춰주기 때문에 버그가 안 터집니다. 그런데 인프라 비용 아끼겠다고 EKS 노드를 **AWS Graviton(ARM)으로 띄우는 순간, 완벽해 보이던 코드에서 처음 보는 환각 파티가 열리며 시스템이 붕괴**합니다. Lock-free 코드를 짤 때 아키텍처를 반드시 이해해야 하는 이유입니다.

## ♾️ Atomic이 없으면 루프가 영원히 끝나지 않는 이유

설령 하드웨어가 착해서 순서를 하나도 안 섞는다고 쳐볼까요? 그래도 **컴파일러가 기어코 코드를 부숴놓습니다.**

```rust
// 우리가 짠 원본 코드              // 컴파일러가 최적화(개악)해버린 코드
while !ready {}               let tmp = ready;   // 루프 밖으로 빼버림 (LICM)
                              if !tmp { loop {} } // 영원히 갇힘!
```

만약 `ready`를 Atomic 변수가 아니라 일반 `bool`로 둔다면, 똑똑한 컴파일러는 "어차피 루프 안에서 안 바뀌는 값이네? 레지스터에 한 번만 올려두고 계속 써야지!"라며 변수 읽기를 루프 밖으로 끄집어냅니다 (루프 불변 코드 이동, LICM). 결과는 **영원히 끝나지 않는 무한 루프**입니다.

`AtomicBool`을 `Acquire`로 읽겠다고 선언하는 순간, 컴파일러는 "아! 이 변수는 다른 스레드가 언제든 바꿀 수 있으니까 맘대로 루프 밖으로 빼거나 순서를 섞으면 안 되는구나!" 하고 최적화를 포기합니다.
Memory Ordering이 하드웨어뿐 아니라 **컴파일러의 소프트웨어 최적화까지 통제하는 이중 계약**이라는 게 여기서 명확히 드러납니다. (이 내용은 5편에서 더 딥하게 다룰 예정입니다.)

## ⚓ 가장 완벽한 표준 예제: Rust `Arc`의 구조

이 복잡한 개념이 가장 아름답게 구현된 실전 코드가 바로 Rust의 `Arc` (Atomic Reference Count) 소스코드입니다.

```rust
// clone: 나 참조 하나 더 할게! 카운트만 +1
fn clone() { count.fetch_add(1, Relaxed); }

// drop: 다 썼으니까 나갈게. 내가 마지막이면 진짜 지운다!
fn drop() {
    if count.fetch_sub(1, Release) == 1 {
        // 방금 내가 1을 0으로 만들었다!
        fence(Acquire);
        // 여기서 진짜 메모리 해제 (destructor 실행)
    }
}
```

* `clone`이 `Relaxed`인 이유: 내가 이미 원본 데이터를 들고 있으니 데이터가 삭제될 리는 없고, 그저 카운트 숫자만 안전하게 +1 올리면 되기 때문입니다. 순서는 중요하지 않죠.
* `drop`이 `Release`인 이유: 내가 이 객체를 가지고 놀았던 모든 흔적(메모리 수정)들이 해제 연산보다 먼저 세상에 퍼지도록 짐을 싸서 내보내는 겁니다.
* **`Acquire` Fence의 마법:** 카운트를 0으로 만든 '마지막 생존자'만이 `Acquire` 장벽을 발동시킵니다. 이 장벽은 앞서 다른 스레드들이 `Release`로 남겨둔 모든 과거의 기억들을 쫙 빨아들인 뒤, 완벽하게 동기화된 상태에서 객체를 소멸시킵니다. (Use-after-free를 막는 예술적인 설계입니다.)

## 🗑️ Lock-free의 진짜 끝판왕: 쓰레기 수거(Reclamation)

Lock-free 자료구조를 짤 때 CAS(Compare-And-Swap)로 노드를 연결하는 것보다 수백 배 어려운 난관이 있습니다. 바로 "이 메모리(노드), 언제 메모리에서 지워도 안전한가?"입니다.

스레드 A가 큐에서 노드 N을 뽑아내서 `free(N)`을 하려는데, 스레드 B가 그 찰나의 순간에 N의 주소를 읽어서 접근하려 한다면? A가 메모리를 날려버리는 순간 B는 끔찍한 Use-After-Free(메모리 오염)를 겪게 됩니다. 여기에 노드가 재사용되면서 주소가 같아지는 악명 높은 **ABA 문제**까지 엮이면 지옥이 펼쳐집니다.

Java나 Go 같은 언어는 가비지 컬렉터(GC)가 아무도 안 쳐다볼 때 알아서 치워주니까 공짜지만, Rust나 C++ 개발자는 이걸 직접 설계해야 합니다.

| 기법 | 어떻게 동작하는가? | 트레이드오프 |
| --- | --- | --- |
| **레퍼런스 카운팅 (`Arc`)** | 참조하는 놈이 몇 명인지 센다. | 쉽다. 하지만 매번 카운트를 올리고 내릴 때마다 엄청난 Atomic 병목(Contention)이 발생함. |
| **에폭 기반 회수 (EBR)** | 시스템 시간을 '에폭(시대)' 단위로 쪼개고, 그 시대에 작업하던 스레드가 다 나가면 쓰레기통을 한 번에 비운다. (`crossbeam-epoch`) | 접근할 때 비용이 거의 **무료**에 가까움. 대규모 동시성에 최강. |
| **해저드 포인터 (HP)** | 스레드가 "나 이거 보고 있으니까 지우지 마!" 하고 전역 배열에 팻말을 꽂아둔다. | 팻말을 꽂고 확인하는 비용이 듦. 대신 메모리 사용 상한선이 아주 타이트하게 관리됨. |

> **💡 이전 편(3편)과의 소름 돋는 연결고리**
> EBR(에폭 기반 회수) 방식에서 만약 한 스레드가 어떤 작업을 하느라 에폭을 오랫동안 붙잡고(Pin) 안 놔주면 어떻게 될까요? 뒤에 쌓인 쓰레기 노드들이 메모리에서 해제되지 못하고 계속 쌓입니다. **Lock-free 큐가 '느려져서' 죽는 게 아니라 '메모리가 터져서' OOMKill을 맞는** 참사가 벌어집니다. 3편에서 다뤘던 '메모리 단편화와 RSS 누수'의 악몽과 완벽하게 똑같은 모습이죠.

## 🛠️ 실전: 직접 눈으로 확인해 보기

이 모든 하드웨어의 마법과 재배치를 직접 확인해 볼 수 있는 도구들이 있습니다. 특히 `loom`은 일반 테스트로는 10년에 한 번 터질까 말까 한 메모리 재배치 버그를 전수 탐색해서 잡아내는 강력한 무기입니다.

```bash
# loom: 코드가 꼬일 수 있는 모든 스레드 인터리빙과 메모리 재배치 경우의 수를 전수 탐색!
RUSTFLAGS="--cfg loom" cargo test --release

# Miri로 런타임 UB(Undefined Behavior)와 데이터 레이스 잡기
cargo +nightly miri test
```

시간이 되실 때 [Godbolt(Compiler Explorer)](https://godbolt.org)에 접속해서 `AtomicU64`를 `Acquire`로 읽는 코드를 작성해 보세요. 컴파일 타깃을 `x86-64`로 했을 때는 평범한 `mov` 명령어가 나오지만, `aarch64(ARM)`로 바꾸는 순간 `ldar` 같은 무거운 배리어 명령어로 번역되는 걸 두 눈으로 확인할 수 있습니다. 인프라와 하드웨어의 차이가 코드로 다가오는 짜릿한 순간입니다.

## 📌 오늘의 핵심 요약

* 코드 순서 재배치는 컴파일러(소프트웨어)와 **하드웨어(Store Buffer)** 양쪽에서 일어나며, 이를 묶는 계약이 바로 Memory Ordering입니다.
* 스레드 간에 데이터가 확실히 보이려면 **happens-before 다리**가 필요하고, 이 다리는 **Release ↔ Acquire 짝맞추기**로만 생성됩니다.
* **`Relaxed`도 원자적(Atomic)입니다.** 단지 앞뒤 코드의 순서만 안 지켜줄 뿐이며, 단순 카운터에는 최고 존엄의 성능을 냅니다.
* StoreLoad 재배치를 막으려면 **SeqCst**를 써야 하며, 이 비용은 x86 서버보다 **AWS Graviton(ARM) 환경에서 훨씬 뼈아프게 다가옵니다.**
* Lock-free 프로그래밍의 끝판왕은 락을 피하는 게 아니라 안전한 메모리 해제(Reclamation)이며, 여기서도 메모리 누수(OOM)를 조심해야 합니다.
