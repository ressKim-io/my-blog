> **시리즈 "패킷은 어떻게 사용자에게 닿는가"의 3편**
> 2편이 서버 내부(East-West)였다면, 이번엔 바깥입니다. 사용자 기기에서 우리 서버까지 패킷이 거치는 모든 정거장을 따라갑니다

## 🗺️ 패킷은 직선으로 가지 않는다

도메인을 입력하고 화면이 뜨기까지, 패킷은 7\~8개의 정거장을 거칩니다. 이 경로를 모르면 "어디서 느린지"를 영원히 추측만 하게 됩니다. 전체 지도부터 봅니다

![Full North-South path from device to Pod with CDN cache-hit branch|xtall](/diagrams/north-south-journey-1.svg)

이 지도의 핵심은 두 가지입니다. 첫째, 패킷은 기기에서 출발해 DNS·접속망·CDN edge를 차례로 거치고, 그 뒤에야 클라우드 리전 안의 ALB → Ingress → Service → Pod로 들어갑니다. 둘째, **CDN edge에서 요청이 끝나 버릴 수도 있다**는 점입니다. 캐시에 있으면(HIT) edge가 즉시 응답하고 origin(우리 서버)까지는 가지 않습니다

즉 같은 도메인이라도 요청의 성격에 따라 경로 길이가 완전히 달라집니다. 정적 파일은 edge에서 끝나고, 동적 요청만 Pod까지 내려갑니다. 응답은 다시 역순으로 사용자에게 돌아갑니다

## 🚏 정거장 하나씩

### (1) DNS — 가까운 IP를 고르는 단계

도메인 이름(`example.com`)을 IP 주소로 바꾸는 과정입니다. 단순한 변환 같지만, 여기서 이미 "가까운 서버 고르기"가 시작됩니다

조회는 보통 로컬 캐시 → OS 리졸버 → ISP 리졸버 → 권한 서버(예: Route 53) 순서로 거슬러 올라갑니다. 그런데 권한 서버가 GeoDNS나 Anycast를 쓰면, **요청이 어디서 왔는지** 보고 답을 바꿉니다. 한국 사용자에겐 한국 IP를, 미국 사용자에겐 미국 IP를 돌려주는 식입니다

그래서 DNS는 단순 주소록이 아니라, **사용자를 가장 가까운 입구로 안내하는 첫 분기점**입니다. 이 "가까운 곳 고르기" 원리는 5편의 Anycast에서 더 깊게 다룹니다

### (2) 접속망 — 모바일과 유선이 갈린다

여기가 앱 개발자에게 특히 중요합니다. 사용자가 휴대폰이냐 PC냐에 따라 경로가 다릅니다

| 접속 유형 | 경로 | 특징 |
|----------|------|------|
| 모바일 | 기기 → 기지국 → 통신사 코어망 → 인터넷 | 무선 구간 탓에 패킷 손실·지연 변동(jitter) 큼 |
| 유선 PC | 기기 → 공유기 → ISP → 인터넷 | 상대적으로 안정적 |

모바일은 무선 구간 때문에 패킷 손실과 지연 변동이 큽니다. 신호 세기, 기지국 혼잡, 핸드오버 같은 변수가 끼어들기 때문입니다. 이 특성이 HTTP/3(QUIC)가 모바일에서 강한 이유로 이어집니다. QUIC은 한 패킷이 손실돼도 다른 스트림이 막히지 않아(head-of-line blocking 완화), 손실이 잦은 무선 환경에서 TCP보다 덜 취약합니다

### (3) CDN Edge — 끝날 수도 있는 지점

사용자와 가장 가까운 CDN 거점(PoP)입니다. 위 지도에서 봤듯, 여기서 요청이 **끝나 버릴 수도** 있습니다. 이게 CDN의 핵심입니다. 캐시에 있으면 origin까지 가지 않고 edge가 바로 응답하니, 거리가 그 자리에서 잘려 나갑니다

## 🔀 경우의 수 4가지

같은 서비스여도 요청 성격에 따라 경로 길이가 완전히 달라집니다

| 경우 | 경로 | 결과 |
|------|------|------|
| ① 정적 + CDN HIT | 기기 → edge → 기기 | 가장 빠름. origin 부담 0 |
| ② 정적 + CDN MISS | 기기 → edge → origin → (edge 캐싱) → 기기 | 첫 사용자만 느리고, 이후엔 ①이 됨 |
| ③ 동적 API | 기기 → edge(가속만) → origin → Pod → 역순 | edge-origin이 warm 연결이라 그냥 인터넷보다 빠름 |
| ④ 리전 위치 vs 사용자 위치 | 서버가 서울인데 미국 사용자 | 태평양 왕복 ~150ms → CDN edge를 미국에 둬 거리를 잘라냄 |

①과 ②의 차이는 "처음이냐 아니냐"입니다. 첫 요청은 캐시에 없으니 origin까지 다녀와야 하지만(MISS), 그때 edge가 응답을 저장해 두므로 다음 사용자부터는 HIT가 됩니다. 그래서 인기 있는 정적 콘텐츠일수록 대부분의 사용자는 ①의 빠른 경로를 탑니다

③은 캐싱이 안 되는 동적 API입니다. 매번 계산이 필요하니 결국 Pod까지 가야 합니다. 다만 edge가 origin과 미리 맺어 둔 따뜻한(warm) 연결을 재사용하므로, 사용자가 직접 origin에 붙는 것보다는 빠릅니다(1편의 핸드셰이크 왕복 절감)

④는 캐싱 여부와 무관한 거리의 문제입니다. 동적이라 Pod까지 가야 하는데 서버가 멀면 왕복이 길어집니다. 이때 CDN edge를 사용자 근처에 두면, 적어도 사용자–edge 구간은 짧아지고 edge–origin 구간만 긴 거리로 남습니다. ①과 ③의 차이, 즉 "정적은 거리를 잘라낼 수 있고 동적은 결국 Pod까지 간다"는 점이 곧 4편의 주제입니다

## 🛠️ CLI로 직접 보기

경로상의 정거장(홉)들과 DNS 응답을 직접 확인합니다

```bash
# 경로상의 정거장(홉)들을 순서대로 보기
traceroute example.com        # 리눅스/맥
tracert  example.com          # 윈도우

# DNS가 어떤 IP를 돌려주는지 (지역에 따라 다를 수 있음)
dig example.com +short
nslookup example.com

# 같은 URL을 CDN 경유로 찍어 1편 curl과 연결
curl -w "connect:%{time_connect} ttfb:%{time_starttransfer}\n" \
-o /dev/null -s https://cdn.example.com/static/img.png
```

`traceroute`로 보이는 홉들이 위 지도의 "접속망 → ISP → edge" 구간을 그대로 펼친 모습입니다. `dig`로 돌아오는 IP는 GeoDNS가 골라 준 "가장 가까운 입구"입니다

## 📌 정리

- 패킷은 **기기 → DNS → 접속망 → CDN edge → ALB → Ingress → Service → Pod**의 정거장을 거칩니다
- **DNS**부터 이미 "가까운 IP 고르기"가 시작됩니다(GeoDNS/Anycast)
- **모바일과 유선**은 접속망 경로가 다르고, 모바일은 손실·지연 변동이 큽니다
- 요청은 **4가지 경우(HIT/MISS/동적/리전위치)**로 갈리고, 캐시되면 경로가 극적으로 짧아집니다
