> **시리즈 "패킷은 어떻게 사용자에게 닿는가"의 4편**
> 1~3편에서 "거리가 지연을 지배한다"고 했습니다. 그런데 거리가 같아도 속도 차이가 납니다. 이번엔 그걸 파헤칩니다

## 🎛️ 거리를 똑같이 고정하면, 남는 차이는?

이런 경험이 있을 겁니다. 집에 서버를 깔아 파일을 호스팅했더니, 같은 한국 안 사용자인데도 상용 CDN보다 한참 느립니다. **거리는 비슷한데(둘 다 한국) 왜 그럴까요?**

실험 설계의 기본을 쓰겠습니다. **변수를 통제합니다.** "거리"를 똑같이 고정시켜 놓으면, 남는 속도 차이는 전부 *거리 외의 요인*입니다. 그게 무엇인지 6가지로 쪼갭니다

| 요인 | 영향 크기 | 한 줄 요약 |
|------|:--:|------|
| ① 네트워크 위치 (peering) | ★★★★★ | ISP 경계를 몇 번 넘나 |
| ② 가정용 회선의 본질적 한계 | ★★★★ | 업로드/best-effort/공유 |
| ③ 캐시 계층 (메모리/SSD) | ★★★ | 디스크 안 읽고 RAM에서 |
| ④ 서버 HW + 커널/TCP 튜닝 | ★★★ | NIC offload, BBR 등 |
| ⑤ 연결 관리 (warm pool) | ★★ | 핸드셰이크 재사용 |
| ⑥ 분산/용량 (Anycast, 대수) | ★★ | 한 대 vs 수십 대 |

![Relative impact of six factors behind CDN speed|tall](/diagrams/why-cdn-is-fast-1.svg)

핵심 결론을 먼저 말하면 이렇습니다. **CDN이 빠른 건 "더 좋은 컴퓨터라서"가 아닙니다.** ①②(네트워크 위치와 회선 등급)가 압도적이고, 박스 성능(③④)은 의외로 작은 부분입니다. 위 막대에서 위쪽 두 요인이 길고 아래로 갈수록 짧아지는 모양이 그 메시지입니다. 그래서 이 글은 영향이 큰 ①②를 먼저, 작은 것들을 뒤에 둡니다

## 📍 ① 가장 큰 것: 네트워크 위치

이건 5편에서 ISP·임베디드 캐시로 깊게 다루니, 여기선 결론만 잡겠습니다. **같은 한국이어도 "인터넷 안에서 몇 개의 ISP 경계를 넘느냐"가 다릅니다.**

가정용 연결은 보통 `집 → ISP A → (중간 경유) → ISP B → 목적지`처럼 여러 네트워크를 거칩니다. 거치는 네트워크가 많다는 건 홉이 많다는 뜻이고, 홉마다 처리지연과 큐잉지연이 붙습니다(1편의 4성분). 반면 CDN은 `사용자 → ISP → CDN(직결)`처럼 중간 경유를 줄이고, 때로는 사용자의 ISP 망 안에 직접 들어가 있습니다

거치는 네트워크 수가 줄면 처리·큐잉 지연이 함께 줄고, 중간 어딘가에서 막힐 확률도 줄어듭니다. 집 PC는 구조상 항상 망의 변두리 끝에 있고, CDN은 그 길목의 좋은 자리에 있습니다. 이 위치 차이가 6요인 중 가장 큽니다

## 🧵 ② 가정용 회선의 본질적 한계

집 PC를 아무리 좋은 사양으로 깔아도 못 넘는 벽이 있습니다. **회선 자체의 성질**입니다

| 한계 | 설명 |
|------|------|
| 비대칭 (asymmetric) | 가정용은 다운로드↑ 업로드↓. 서버는 업로드로 서빙 → 콘텐츠 주는 방향이 느림 |
| best-effort | 품질 보장(SLA)이 없음. 혼잡하면 그냥 느려짐 |
| 공유/초과가입 (contention) | 같은 동네가 한 회선을 공유 → 저녁 황금시간엔 이웃과 경쟁해 큐잉지연 ↑ |
| NAT/공유기 | 공유기가 NAT·세션테이블을 처리. 동시연결이 많으면 공유기가 먼저 뻗음 |
| 단일 회선 | 회선 하나가 끊기면 끝. 우회 경로가 없음 |

특히 비대칭이 치명적입니다. 가정용 회선은 "받는" 쪽에 맞춰 설계돼 다운로드는 빠르고 업로드는 느립니다. 그런데 서버 역할은 콘텐츠를 *주는*, 즉 업로드 방향입니다. 호스팅에 쓰는 순간 가장 느린 방향으로 일하게 됩니다

반면 CDN/IDC 회선은 대칭·전용·SLA 보장·다중 업링크입니다. 출발선이 다릅니다. 이건 박스 성능이 아니라 *회선 등급*의 문제라, 집에 아무리 비싼 서버를 놔도 극복되지 않습니다

## 🧊 ③ 캐시 계층 — 디스크를 안 읽는다

이제 "구성"이 실제로 다른 부분입니다. CDN은 캐시를 계층으로 쌓습니다

- **L1 (RAM 캐시)** — 있으면 메모리 속도(µs)로 즉시 응답합니다
- **L2 (NVMe SSD)** — RAM에 없으면 빠른 SSD에서 꺼냅니다
- **origin shield** — 여러 edge의 같은 콘텐츠 요청을 한데 묶어 origin에는 한 번만 보냅니다(origin 보호)
- **Origin (S3/ALB)** — 위 어디에도 없을 때의 최후 출처입니다

핫한 콘텐츠는 거의 **RAM에서** 나갑니다. 디스크를 읽지 않으니 I/O 지연이 사라집니다. 집 PC의 웹서버도 OS 페이지 캐시가 있긴 하지만, 이렇게 의도적으로 다계층을 설계해 놓은 것과는 다릅니다. 게다가 origin shield가 origin으로 가는 요청을 흡수해, 인기 콘텐츠가 몰려도 origin은 평온합니다

## ⚙️ ④ 서버 HW + 커널/TCP 튜닝

박스 자체도 다릅니다. 다만 ①②보다 영향은 작습니다

| 항목 | CDN 노드는 이렇게 |
|------|------|
| NIC offload | TSO/GRO/RSS/checksum offload로 NIC이 패킷을 처리 → CPU 부담 ↓ |
| 멀티큐 + RSS | NIC 큐를 CPU 코어별로 분산 → 단일코어 병목 제거 |
| 커널 우회 (일부) | DPDK/AF_XDP로 커널 스택을 건너뛰어 지연 ↓ |
| 혼잡제어 BBR | 손실 기반 CUBIC 대신 대역폭·RTT 측정 기반으로 동작 |
| initcwnd 튜닝 | 초기 전송창을 키워 slow start 초반을 가속 |
| 소켓 버퍼 autotune | BDP에 맞춰 버퍼를 자동 조정 → 긴 RTT도 채움 |

여기서 1편의 TCP slow start가 다시 등장합니다. BBR과 initcwnd 튜닝은 결국 "slow start를 빨리 벗어나기" 위한 장치입니다. BBR은 패킷 손실을 신호로 삼는 CUBIC과 달리, 실제 대역폭과 RTT를 측정해 보낼 양을 정하므로 긴 거리에서도 회선을 더 빨리 채웁니다. initcwnd를 키우면 처음부터 더 많이 보내 초반 왕복 낭비를 줄입니다

## 🔗 ⑤⑥ 연결 관리와 분산

- **⑤ warm pool** — CDN edge는 origin과 미리 맺어 둔 연결 풀을 유지합니다. 사용자 요청이 와도 그 자리에서 핸드셰이크를 새로 하지 않습니다(1편의 RTT 절감). 여기에 TLS resumption, HTTP/2, HTTP/3까지 다 켜져 있습니다
- **⑥ 분산** — 집 PC는 한 대입니다. 동시 접속이 몰리면 CPU·NAT·업로드 회선이 차례로 뻗습니다. CDN은 여러 PoP와 Anycast로 부하를 분산하고 DDoS도 흡수합니다. **부하가 올라갈수록 격차가 더 벌어집니다**

## 🔄 핵심 반전

질문은 "CDN이 빠른 건 박스(구성)가 좋아서인가?"였습니다. 답은 "구성도 다르지만(③④⑤), 진짜 큰 건 위치와 회선(①②)"입니다

> 집 PC를 CDN급 서버로 바꿔 깔아도 여전히 느립니다 — 가정용 회선과 ISP 경계는 못 바꾸니까요. 반대로 평범한 서버라도 ISP 안에, IXP 1홉 거리에, 대칭 전용회선으로 두면 그것만으로 집 PC를 압도합니다

한 문장으로 요약하면 이렇습니다. CDN이 빠른 결정적 이유는 "더 좋은 컴퓨터"가 아니라, *사용자와 같은 ISP 안 또는 IXP 1홉 거리에, 대칭 전용회선으로 꽂혀 있고, 핫 콘텐츠를 RAM에 들고 있어서 ISP 경계와 디스크 I/O를 둘 다 건너뛴다*는 것입니다

## ☁️ CDN이 아니라 그냥 클라우드 서버여도

여기까지는 빠른 쪽을 CDN으로 놓고 비교했습니다. 그런데 "캐싱을 안 하는 평범한 클라우드 서버(EC2 같은 VM)는 어떤가?"라는 질문이 남습니다. 결론부터 말하면 클라우드 VM도 집 PC 서버보다 빠릅니다. 단 이유의 구성이 조금 다릅니다 — 위 6요인 중 **③(캐시 계층)만 빠지고 ①②④⑤는 그대로** 적용됩니다

캐시는 CDN 고유의 장치입니다. 순수 VM은 핫 콘텐츠를 RAM 다계층으로 들고 있지 않으니, 같은 동적 응답이라면 클라우드 VM과 캐시 미스 상태의 CDN origin은 결이 비슷합니다. 그래서 클라우드 VM이 집 PC를 이기는 힘은 거의 **①위치와 ②회선**에서 나옵니다. 이게 앞 절 "핵심 반전"이 말한 바로 그 지점입니다

①②가 실제로 무엇을 뜻하는지, 데이터센터와 가정용을 나란히 두면 이렇습니다

| 구분 | 클라우드 데이터센터 | 가정용 회선 |
|------|------------------|------------|
| 위치(①) | carrier-neutral DC·IXP에서 주요 ISP·Tier-1 망과 직결(peering) | 망의 leaf 끝 — 목적지까지 여러 AS 경계 경유 |
| 상위 연결(②) | BGP multihoming으로 여러 상위 ISP에 동시 연결 → 한 곳 죽어도 우회 | 보통 단일 ISP·단일 업링크 |
| 대역 성격(②) | 대칭·전용, oversubscription 거의 없음 | 비대칭 — 업로드가 전체의 10%대 |
| 혼잡(②) | 전용이라 피크에도 일정 | 25:1\~100:1 수준 초과가입 → 저녁 피크에 이웃과 경쟁 |
| 공인 IP(②) | 고정 공인 IP 직접 보유 | 다수 가입자가 CGNAT로 공인 IP 공유 → 인바운드 호스팅 자체가 까다로움 |

가정용 회선의 이 성질들은 대부분 **설계상 의도**입니다. ISP는 저가 가정 회선으로 서버를 돌리는 걸 원치 않아 업로드를 의도적으로 좁게 잡고, "모두가 동시에 최대로 쓰지는 않는다"는 가정 위에 회선을 초과 판매합니다. 평소엔 문제없지만, 서버처럼 꾸준히 업로드하는 용도에는 구조적으로 불리합니다

정리하면, 클라우드에 CDN을 안 붙이고 origin VM만 올려도 집 PC 호스팅보다 빠릅니다. 빠른 이유가 "더 좋은 컴퓨터"가 아니라 "더 좋은 위치와 회선(①②)"이기 때문입니다. CDN은 여기에 ③ 캐시 계층을 한 겹 더 얹어 거리를 한 번 더 잘라내는 추가 레이어입니다

## 🛠️ CLI로 직접 보기

origin과 CDN을 같은 명령으로 찍어 차이를 직접 봅니다

```bash
# 1편 curl을 origin vs CDN에 각각 찍어 비교 — 이 글의 핵심 실측
curl -w "connect:%{time_connect} tls:%{time_appconnect} ttfb:%{time_starttransfer}\n" \
-o /dev/null -s https://origin.example.com/img.png
curl -w "connect:%{time_connect} tls:%{time_appconnect} ttfb:%{time_starttransfer}\n" \
-o /dev/null -s https://cdn.example.com/img.png

# 내 리눅스 서버의 혼잡제어 확인 / BBR 켜기 (④ 실험)
sysctl net.ipv4.tcp_congestion_control
sudo modprobe tcp_bbr
sudo sysctl -w net.ipv4.tcp_congestion_control=bbr

# NIC offload 상태 확인 (④)
ethtool -k eth0 | grep -E 'tso|gso|gro|rx-checksum'

# 인위적 지연/손실 주입 후 CUBIC vs BBR 비교 (홈랩 실험)
sudo tc qdisc add dev eth0 root netem delay 150ms loss 2%
# ...측정 후 원복
sudo tc qdisc del dev eth0 root
```

origin과 CDN의 `time_connect`·`time_appconnect`를 나란히 보면, 거리가 비슷해도 위치와 연결 관리(①⑤)가 만든 차이가 숫자로 드러납니다. `netem`으로 지연·손실을 주입한 뒤 CUBIC과 BBR을 비교하면 ④의 효과도 손으로 확인할 수 있습니다

## 📌 정리

- 거리를 통제하면 남는 차이는 **6요인**입니다 — 위치·회선·캐시계층·HW튜닝·연결관리·분산
- **압도적인 건 ①네트워크 위치 ②회선 등급**입니다. 박스 성능(③④)은 생각보다 작습니다
- 가정용 회선은 **비대칭·best-effort·공유·CGNAT**라는 구조적 한계를 가집니다
- CDN은 핫 콘텐츠를 **RAM에서** 내보내고, **warm 연결**로 핸드셰이크를 건너뜁니다
- CDN이 아니라 **그냥 클라우드 VM**이어도 집 PC보다 빠릅니다 — 캐시(③)만 빠질 뿐 ①②④⑤가 그대로이기 때문입니다
