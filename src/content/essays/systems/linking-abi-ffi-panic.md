> **시리즈 "Rust와 함께 가는 CS 레이어"의 6편**
> 우리가 에디터에서 아무리 우아하게 `fn add(a: u32, b: u32)`라고 타입을 명시해도, 컴파일러를 거쳐 기계어가 되고 나면 `u32`라는 타입은 허공으로 증발해 버립니다. 남는 건 그저 "`rdi` 레지스터의 4바이트, `rsi` 레지스터의 4바이트"라는 차갑고 익명화된 바이트 더미뿐이죠.
> 그렇다면 전혀 다른 철학을 가진 C언어(gcc)와 Rust(rustc)는 도대체 *무엇을 어떻게 합의하고* 서로의 함수를 호출하는 걸까요? 이 질문의 답이 바로 **ABI(Application Binary Interface)**이며, 이번 편의 하이라이트인 "왜 panic이 C 경계를 넘으면 시스템이 박살(UB)나는가"를 이해하는 열쇠가 됩니다.

## 🤔 "서로 다른 언어는 어떻게 서로를 부를까?"

C언어로 짠 레거시 라이브러리 함수를 Rust에서 호출한다고 상상해 봅시다. "첫 번째 인자는 무조건 `rdi` 레지스터에 넣을게!"라는 걸 이 두 언어는 도대체 어떻게 합의한 걸까요? 놀랍게도 하드웨어(CPU)는 이런 규칙을 전혀 강제하지 않습니다.

여기서 아주 중요한 개념이 등장합니다. **ABI(Application Binary Interface)는 컴파일 과정에서 사라져 버린 타입과 의미를, 바이트 더미 위에 다시 부여하는 '국가 간의 조약문'**입니다.
그리고 **링킹(Linking)은 이 조약문을 전제로, 각자 따로 컴파일된 파일 조각(`.o`)들 사이의 이름(Symbol)을 실제 메모리 주소로 연결해 하나의 거대한 실행 이미지를 만드는 과정**이죠.

이 합의가 어디서 오고 컴파일러가 어디까지 지켜주는지를 따라가다 보면, 그 악랄한 FFI(Foreign Function Interface) 버그들이 왜 발생하는지 명확히 보이게 됩니다.

## 🧩 하드웨어는 '정책'을 모른다 — 호출 규약 (Calling Convention)

x86-64 아키텍처 CPU가 함수 호출에 대해 아는 명령어는 딱 두 개뿐입니다.
복귀할 주소를 스택에 푸시하고 목표 지점으로 점프하는 `call`, 그리고 스택에서 복귀 주소를 팝(pop)해서 돌아오는 `ret`입니다.

결정적인 사실은, **CPU에는 "함수 인자"라는 개념 자체가 아예 없습니다.** `call` 명령어는 인자를 옮겨주지 않습니다. "첫 번째 정수 인자는 `rdi`에 넣는다" 같은 규칙은 실리콘 칩 그 어디에도 박혀있지 않죠.
왜냐고요? 하드웨어는 오직 **메커니즘**만 제공할 뿐, **정책**은 소프트웨어의 몫으로 남겨두기 때문입니다. 정책을 칩에 하드코딩해버리면 영원히 바꿀 수 없으니까요.

그래서 똑같은 인텔 CPU 위에서도 리눅스의 System V ABI와 윈도우의 Microsoft x64 ABI가 서로 전혀 다른 방식으로 인자를 넘깁니다. **호출 규약(Calling Convention)**이란, CPU가 비워둔 이 '정책 슬롯'을 OS와 컴파일러 생태계가 함께 채워 넣은 신사협정입니다.
이 협정을 어기면 컴파일러는 에러를 뱉지 않습니다. 그저 `rdi` 레지스터에 엉뚱한 쓰레기 값이 들어가 런타임에 소리 없이 크래시가 날 뿐이죠. FFI 버그가 악몽 같은 이유가 바로 이겁니다. 언어 차원의 깐깐한 타입 검증이 아니라, 바이너리 레벨의 눈치 게임이기 때문입니다.

## 📐 System V ABI 조문 톺아보기 — 레지스터, 스택, 그리고 정렬

리눅스/macOS 생태계를 지배하는 System V ABI 조약문의 핵심 조항은 대략 이렇습니다.

| 항목 | 규칙 | 비고 |
|---|---|---|
| **정수·포인터 인자** | `rdi, rsi, rdx, rcx, r8, r9` | 7번째 인자부터는 스택으로 밀려납니다. |
| **반환값 (Return)** | `rax` (부동소수점은 `xmm0`) | 128비트 크기면 `rax`와 `rdx`를 합쳐서 씁니다. |
| **Caller-saved** | `rax, rcx, rdx, rsi, rdi...` | **호출된 함수(Callee)가 맘대로 덮어써도 무죄.** 보존하고 싶으면 호출한 쪽(Caller)이 미리 대피시켜야 합니다. |
| **Callee-saved** | `rbx, rbp, r12 ~ r15` | **호출된 함수가 이 레지스터를 쓰려면, 무조건 기존 값을 스택에 백업했다가 리턴 직전에 원복**해야 합니다. |

인자를 6개나 넉넉하게 레지스터로 넘기는 이유는 간단합니다. 레지스터는 메모리(스택)보다 수십 배 빠르기 때문에, 대부분의 함수(인자 6개 이하)에서 스택 접근 비용을 통째로 날려버리기 위해서죠. Caller-saved와 Callee-saved로 레지스터를 반반 가른 것은 **"레지스터 백업 비용을 누가 낼 것인가?"를 타협한 완벽한 균형의 결과**입니다.

> **💡 시니어의 어셈블리 삽질 노트: 스택 정렬(Alignment)의 저주**
> ABI 조약에는 아주 무서운 숨은 조항이 있습니다. **"함수를 부르기 직전(`call` 직전), 스택 포인터 `rsp`는 반드시 16바이트 단위로 정렬되어 있어야 한다"**는 것입니다. (SSE 같은 최신 벡터 명령어들이 16바이트 정렬된 메모리를 강요하기 때문이죠).
> C나 Rust 컴파일러는 알아서 이 박자를 맞춰주지만, JIT 컴파일러를 직접 만들거나 인라인 어셈블리를 손으로 짤 때 이 16바이트 스텝을 삐끗하면 가차 없이 `#GP (General Protection Fault)`를 맞고 프로세스가 즉사합니다.

## 🧱 구조체는 어떻게 넘어갈까 — `#[repr(C)]`의 진짜 의미

FFI 경계에서 초보자들이 가장 많이 밟는 지뢰가 바로 '구조체(Struct)'입니다.
System V ABI는 구조체를 넘길 때 **8바이트 단위(Eightbyte)**로 쪼개서 이리저리 분류합니다. 16바이트 이하의 깔끔한 구조체는 레지스터 2개에 나눠 담아 보내지만, 덩치가 크거나 정렬이 안 맞으면 가차 없이 스택(메모리)으로 던져버립니다.

이게 왜 Rust에서 폭탄이 될까요? **Rust의 기본 레이아웃(`#[repr(Rust)]`)은 C언어와 동일한 메모리 구조를 보장하지 않기 때문입니다.** Rust 컴파일러는 메모리 낭비를 줄이려고 구조체 필드의 순서를 자기 맘대로 섞어버립니다(Field Reordering). 이걸 C언어로 냅다 넘기면? C언어는 A 필드가 있을 거라 기대하고 레지스터를 읽었는데, Rust가 거기에 B 필드를 쑤셔 넣어둔 상황이 발생합니다. 컴파일은 멀쩡히 되는데 데이터가 모조리 쓰레기 값으로 변하는 대참사가 터지죠.

그래서 구조체에 **`#[repr(C)]`**를 붙이는 겁니다. 이건 단순한 매크로가 아니라, **"내 Rust 구조체를 건방지게 최적화하지 말고, C언어의 엄격한 ABI 조약문에 정식으로 등록하겠다"**는 강력한 선언입니다. (이건 1편에서 봤던 거짓 공유(False Sharing)를 막기 위해 패딩을 직접 통제하는 것과도 맥락이 닿아 있습니다.)

## 🔤 네임 맹글링(Name Mangling) — 왜 이름을 암호문으로 꼴 보기 싫게 비틀까?

링커(Linker)가 바라보는 세상은 오직 **단순한 문자열(Symbol)**들의 세계입니다.
C언어는 네임스페이스가 평평해서 `add`라는 함수는 링커에게도 그냥 `add`입니다. 하지만 C++나 Rust처럼 오버로딩, 제네릭, 모듈 시스템이 있는 언어는 어떨까요? `Math::add(int)`와 `Math::add(double)`은 명백히 다른 함수인데, 링커 눈에는 똑같은 `add`라는 문자열로 보여서 충돌이 납니다.

이 딜레마를 해결하기 위해 컴파일러는 함수의 경로, 파라미터 타입 등을 영문자와 숫자를 뒤섞어 **하나의 길고 기괴한 심볼 이름으로 압축(인코딩)**해버립니다. 이를 네임 맹글링이라고 합니다. (예: `_ZN4Math3addEi` 같은 형태죠).

여기서 실무적인 혼동 포인트 하나 짚고 갑니다. FFI 코드를 짤 때 쓰는 **`extern "C"`와 `#[no_mangle]`은 완전히 직교하는 별개의 설정입니다.**
* **`extern "C"`:** "이 함수의 인자와 반환값을 **C언어의 ABI 규약**대로 처리해라!" (호출 규칙)
* **`#[no_mangle]`:** "링커가 찾기 쉽게 내 함수 이름을 **괴상하게 비틀지 마라!**" (심볼 이름)

C언어에서 Rust 함수를 호출하고 싶다면 이 두 개를 반드시 같이 써야 합니다. 하나라도 빼먹으면? 규약이 안 맞아서 크래시가 나거나, 링커가 함수를 못 찾아서 빌드가 터집니다.

## 🗂️ ELF — 하나의 파일을 바라보는 두 개의 시선

리눅스의 목적 파일(`.o`), 공유 라이브러리(`.so`), 실행 파일은 모두 **ELF(Executable and Linkable Format)**라는 규격을 따릅니다. 이 ELF 구조의 가장 아름다운 통찰은 **"하나의 파일을 두 가지 시선으로 바라본다"**는 점입니다.

1. **섹션 뷰 (Linking View):** 링커를 위한 시선입니다. 링커는 파편화된 `.o` 파일들에서 코드는 `.text` 섹션끼리, 전역 변수는 `.data` 섹션끼리 끼리끼리 모아서 합치고 심볼의 주소를 메워 넣습니다.
2. **세그먼트 뷰 (Execution View):** 커널과 로더(Loader)를 위한 시선입니다. 파일을 메모리에 띄울 때 커널은 자잘한 섹션에는 관심이 없습니다. 오직 **권한(읽기/쓰기/실행)** 단위로 뭉쳐진 세그먼트만 봅니다. (예: 코드가 담긴 `.text`와 읽기 전용 데이터 `.rodata`를 묶어서 `R-X` 권한으로 한 번에 메모리에 `mmap` 해버립니다.)

이 뷰의 분리가 왜 중요할까요? 프로그램 실행을 극도로 효율적으로 만들기 위해서입니다. 초기화되지 않은 전역 변수들이 들어가는 `.bss` 섹션은 굳이 파일에 0으로 꽉 채워 용량을 낭비하지 않습니다. 크기만 적어두고, 나중에 커널이 "요구 기반 페이징(Demand Paging, 2편 참고)"을 통해 메모리에 0으로 채워진 페이지를 툭 던져주는 우아한 구조를 취합니다.

## 🔗 지연 바인딩(Lazy Binding)의 마법과 GOT/PLT

동적 링킹(Dynamic Linking)은 실행 파일 크기를 줄이고 메모리를 절약해 줍니다.
하지만 내가 짠 코드에서 `libc.so`에 있는 `printf`를 호출하려면 런타임에 그 주소를 어떻게 찾을까요? 이 동적 링킹의 심장이 바로 **GOT(Global Offset Table)**와 **PLT(Procedure Linkage Table)**가 만들어내는 **지연 바인딩(Lazy Binding)**입니다.

![GOT/PLT lazy binding — first call resolves through the dynamic linker, then caches in the GOT](/diagrams/linking-abi-ffi-panic-1.svg)

최초로 `printf`를 호출할 때, 프로그램은 진짜 주소를 몰라서 일단 PLT라는 작은 점프대(Stub)로 뜁니다. PLT는 다시 동적 링커(Dynamic Linker)를 호출해서 "야, `printf` 주소 좀 찾아줘!"라고 부탁하죠.
주소를 찾은 동적 링커는 그 결과값을 **GOT 테이블에 캐싱(저장)**해 두고 함수를 실행합니다. 두 번째 호출부터는? 지루한 탐색 과정 없이 GOT에 적힌 주소로 다이렉트로 점프합니다.

왜 굳이 첫 호출까지 미루는 'Lazy' 방식을 쓸까요? 수천 개의 함수가 들어있는 거대 라이브러리를 로드할 때, 쓰지도 않을 함수 주소까지 전부 찾고 앉아있으면 프로그램 시작 시간(Cold Start)이 끔찍하게 느려지기 때문입니다.
(단, 보안이 빡빡한 컨테이너 환경에서는 해커가 이 GOT 테이블을 덮어써서 흐름을 가로채는 걸 막기 위해, 프로그램 시작 시점에 모든 주소를 다 찾고 GOT를 '읽기 전용'으로 잠가버리는 **Full RELRO + BIND_NOW** 방식을 기본으로 씁니다.)

## 🤝 언어를 넘는 국경선, FFI에서 합의해야 할 5가지

자, 이제 C와 Rust가 경계를 넘으려면 다음 5가지를 합의해야 합니다. 앞의 3개는 우리가 설정만 잘하면 컴파일러가 알아서 해줍니다.

1. **호출 규약:** `extern "C"`로 System V 레지스터 룰에 맞춘다.
2. **데이터 레이아웃:** `#[repr(C)]`로 구조체 패딩을 맞춘다.
3. **심볼 이름:** `#[no_mangle]`로 이름을 암호화하지 않는다.

하지만 진짜 무서운 건 컴파일러가 도와주지 못하는 나머지 2개, **언어 차원의 계약이 닿지 않는 무법 지대**입니다.

4. **소유권과 수명 (Lifetimes):** ABI 조약문에는 "이 메모리를 다 쓰고 누가 `free` 할 것인가?"에 대한 조항이 없습니다. Rust에서 할당한 힙 메모리를 C언어가 `free()`로 해제해 버리면 어떻게 될까요? 할당자(Allocator)의 메타데이터가 꼬이면서 런타임이 통째로 붕괴합니다 (2편의 Allocator와 직결됨). **"할당한 놈이 해제한다"**는 철칙을 손수레처럼 끌고 다녀야 해서 FFI가 늘 `unsafe`인 것입니다.
5. **💥 에러 처리 (Panic):** 그리고 대망의 패닉(Panic)입니다.

## 💥 패닉(Panic)이 C 경계를 넘으면 왜 시스템이 터질까 (UB)

이 글의 하이라이트입니다. Rust 코드에서 `panic!`이 터지면 기본적으로 **Unwind(되감기)**를 시작합니다.
스택을 거꾸로 타고 올라가면서 각 프레임에 있는 변수들의 소멸자(`Drop`)를 깔끔하게 실행해 주고, 스레드 최상단까지 올라가 안전하게 죽는 아주 우아한 방식이죠. 이 Unwind 과정은 C++의 예외 처리(Exception) 메커니즘과 완전히 동일한 산업 표준 ABI(Itanium C++ ABI)를 사용합니다.

컴파일러는 각 함수를 빌드할 때 **`.eh_frame` (어디로 돌아가야 하는지)**과 **`.gcc_except_table` (어떤 소멸자를 불러야 하는지)**이라는 지도(Metadata)를 꼼꼼하게 남겨둡니다.
언와인더(Unwinder)는 에러가 터지면 이 지도를 더듬으며 스택을 위로 올라가는 일종의 '상태 기계(State Machine)'입니다.

자, 그런데 중간에 C언어로 짠 함수 프레임이 끼어있다면? (보통 C 코드는 예외를 안 쓰니까 `-fexceptions` 플래그 없이 빌드됩니다.)

![Panic unwinds through Rust frames but a C frame without unwind info is undefined behavior](/diagrams/linking-abi-ffi-panic-2.svg)

Rust의 패닉이 되감기를 하다가 이 C언어 프레임을 만나는 순간, **언와인더는 지도가 끊겨버린 칠흑 같은 암흑 속에 갇혀버립니다.** 이게 바로 그 끔찍한 **UB(Undefined Behavior, 정의되지 않은 동작)**의 정체입니다. 그냥 크래시가 나고 끝나는 게 아니라 시스템 전체가 썩어버립니다.

1. **지도를 잃어버린 폭주:** 다음 프레임 위치를 계산하지 못해 엉뚱한 메모리 주소를 스택으로 착각하고 거기로 점프(실행)해 버릴 수 있습니다. 해커들의 놀이터가 열리는 셈이죠.
2. **C언어의 불변식 파괴:** 기적적으로 되감기에 성공해서 C 프레임을 건너뛰었다 쳐도, C언어는 "함수가 `ret`로 얌전히 복귀할 것"을 전제로 락(Lock)을 걸거나 메모리를 잡고 있습니다. 되감기가 이걸 강제로 찢고 지나가버리면 영구적인 데드락(Deadlock)과 메모리 누수가 발생합니다.

이 치명적인 문제에 대한 Rust 진영의 해답은 무엇이었을까요?
**Rust 1.81(2024년 9월)부터 `extern "C"` 경계 밖으로 새어나가는 패닉은 아예 '프로세스 강제 종료(Abort)'로 못을 박아버렸습니다.** UB로 시스템이 좀비가 되는 것보단, 그 자리에서 즉사하는 게 훨씬 안전하다는 철학이죠. (만약 양쪽 언어 모두 예외 처리 지도를 그렸다면 `extern "C-unwind"`를 통해 통제된 언와인딩을 할 수도 있습니다.)

## 🦀 Rust 훅 + 실무 DevOps 팁

링킹과 ABI의 개념은 그대로 우리의 인프라 관측성(Observability) 도구로 직결됩니다.

* **프레임 포인터(Frame Pointer)의 귀환:** 예전에는 최적화 한답시고 `-fomit-frame-pointer`를 켜서 1개의 레지스터를 더 썼습니다. 하지만 이렇게 하면 앞서 말한 언와인딩 지도를 풀기 전에는 스택 트레이스를 그릴 수가 없죠. 요즘처럼 eBPF 기반 프로파일러(Pyroscope 등)가 1밀리초 단위로 커널에서 스택을 떠보는 시대에는 이 값싼 프레임 포인터가 필수입니다. (그래서 최근 우분투나 페도라는 이 옵션을 다시 켜는 추세입니다.)
* **정적 링킹의 함정 (DNS 쿼리):** 도커 이미지 줄인다고 `musl` libc로 바이너리를 완전 정적 링킹(Static Link)하는 경우가 많습니다. 아주 좋지만, 가끔 C언어의 `getaddrinfo`(DNS 해석기) 류의 함수들은 `dlopen`을 통해 런타임에 동적으로 라이브러리를 물고 옵니다. 분명 정적 링크를 했는데 컨테이너 안에서 조용히 네트워크 호출이 실패하는 호러 스토리가 여기서 나옵니다.
* **에이전트의 패닉 격리:** Vector나 Fluent Bit 같은 장기 실행 에이전트에서, 특정 로그 하나 파싱하다가 터진 Panic이 전체 파이프라인을 다 죽이면 안 되겠죠? `catch_unwind`로 철저히 방화벽을 쳐야 합니다. 단, `panic = "abort"`로 빌드했다면 이 방화벽도 뚫리고 에이전트가 통째로 죽어버리니 주의가 필요합니다.

## 🛠️ 실무 모니터링: 내 바이너리 직접 뜯어보기

개발 환경이나 배포된 도커 이미지 안에서 아래 명령어들로 링킹 상태를 직접 해부해 볼 수 있습니다.

```bash
# 심볼 암호문(Mangling) 풀기 — Rust는 rustfilt, C++는 c++filt
nm ./my_rust_app | rustfilt | head

# 이 앱이 런타임에 어떤 공유 라이브러리(.so)를 물고 오는지 확인
readelf -d ./my_rust_app | grep -E 'NEEDED|INTERP'
ldd ./my_rust_app

# 컨테이너 보안 상태(RELRO, PIE, 스택 카나리) 하드닝 점검
checksec --file=./my_rust_app
```

`readelf`를 쳐봤을 때 `libc.so`가 걸려 있다면 동적 링킹이 된 상태입니다. `checksec`에서 RELRO가 `Full`로 뜬다면, 앞서 말한 GOT 지연 바인딩을 포기하고 시작할 때 주소를 다 묶어버린 '보안 몰빵' 모드라는 걸 확인할 수 있습니다.

## 📌 오늘의 핵심 요약

* 코드의 타입은 기계어가 되며 증발하고, **ABI(호출 규약)는 그 빈자리에 의미를 다시 부여하는 바이트 레벨의 합의문**입니다.
* 구조체의 메모리 순서를 지키기 위해 `#[repr(C)]`를 쓰고, 링커가 함수를 찾을 수 있게 `#[no_mangle]`로 이름 꼬임을 방지합니다.
* 동적 링킹의 마법인 **GOT/PLT**는 함수 주소 찾기를 런타임 첫 호출 시점까지 미루는 지연 바인딩(Lazy Binding)으로 시작 성능을 끌어올립니다.
* FFI 경계에서는 메모리 소유권과 에러(Panic) 처리가 보장되지 않으므로 철저히 수동으로 방어해야 합니다.
* **Rust의 패닉이 Unwind 지도(`.eh_frame`)가 없는 C 프레임으로 넘어가면 길을 잃고 UB(미정의 동작)를 일으킵니다.** (Rust 1.81부터는 안전을 위해 이를 Abort로 차단합니다.)
