> **시리즈 "Rust와 함께 가는 CS 레이어"의 3편**
> 2편에서 우리는 아주 충격적인 사실을 하나 알게 되었습니다. 분명 코드에서 `free`(해제)를 꼼꼼히 했는데도, 물리 메모리(RSS)는 즉시 OS로 돌아가지 않는다는 사실이었죠. 그 이유를 "유저 스페이스 할당자(Allocator)가 쥐고 있기 때문"이라며 3편으로 미뤄두었습니다.
> 내 코드의 `malloc()` 호출과 리눅스 커널의 시스템 콜 사이, 그 보이지 않는 틈새에서 도대체 무슨 일이 벌어지고 있는 걸까요? Rust에서 `#[global_allocator]` 한 줄을 jemalloc으로 바꿨을 때 성능이 요동치는 이유, 오늘 그 뚜껑을 열어보겠습니다.

## 🤔 "할당자 코드 한 줄 바꿨더니 서버가 쌩쌩해졌어?"

Rust로 백엔드 서버나 데이터 파이프라인을 구축해 본 분들이라면 아마 이런 경험, 한 번쯤 있으실 겁니다.
로직은 단 한 글자도 건드리지 않고 `#[global_allocator]`를 `jemalloc`이나 `mimalloc`으로 쓱 바꿨을 뿐인데, 뜬금없이 처리량(Throughput)이 오르고 널뛰던 메모리(RSS)가 안정화됩니다.

분명 똑같은 `malloc`과 `free`를 호출하는데 왜 할당자에 따라 이런 극적인 차이가 날까요?
그 답은 할당자가 내부적으로 짊어지고 있는 두 가지 극한의 미션 — **"요청한 메모리를 락(Lock) 없이 최대한 빨리 떼어줄 것"**과 **"파편화(낭비) 없이 메모리를 꽉꽉 눌러 담아 재활용할 것"** — 을 멀티스레드 환경에서 어떻게 풀어내느냐에 있습니다.

## 🧩 할당자는 '도매상'이다 (커널과 앱 사이의 간극)

2편에서 배웠듯, 리눅스 커널은 메모리를 무조건 **페이지(보통 4KB)** 단위로만 큼직하게 던져줍니다.
그런데 우리 프로그램은 어떨까요? 당장 String 하나, 객체 하나를 만들기 위해 33바이트, 17바이트, 때로는 5,000바이트 등 아주 자잘한 크기를 쉴 새 없이 요구합니다.

이때마다 커널을 부르면 비싼 시스템 콜(Syscall) 비용과 Page Fault 오버헤드 때문에 시스템은 그 자리에서 멈춰버릴 겁니다. 그래서 유저 스페이스의 메모리 할당자는 OS한테 한 번에 거대한 땅을 '도매'로 떼어온 뒤, 그걸 아주 잘게 쪼개서 우리 코드에 '소매'로 넘기고 재활용합니다.

즉, 우리가 호출하는 **대부분의 `malloc`은 커널을 부르지 않습니다.** 할당자 내부에서 얼마나 빠르고 조용히 이 거래를 끝내느냐가 할당자 성능의 절반을 결정합니다.

## 🗂️ Size Class와 Free List: alloc/free를 O(1)로 만들기

가장 원시적인 할당자는 거대한 빈 공간을 연결 리스트(Free list)로 쭉 묶어두고, 요청이 올 때마다 쭈욱 순회하며 크기가 맞는 조각을 찾아 떼어줍니다. 당연히 검색 시간이 너무 느리겠죠?
그래서 현대의 할당자들은 크기를 **'Size Class(크기 계급)'**로 반올림해버리는 전략을 취합니다.

| 요청 크기 | 반올림된 Size Class | 할당자 내부 구조 |
|---|---|---|
| 1 ~ 8B | **8B** | |
| 9 ~ 16B | **16B** | 각 크기별로 '전용 빈 슬롯 리스트(Bin)'가 있음 |
| 17 ~ 32B | **32B** | 할당: 리스트에서 툭 떼어줌 (O(1)) |
| **33 ~ 48B** | **48B** | 해제: 리스트에 툭 던져넣음 (O(1)) |
| 49 ~ 64B | **64B** | |

요청을 가장 가까운 Size Class로 올림 처리하면, 해당 크기의 슬롯들만 모아둔 리스트에서 O(1)의 속도로 넣고 뺄 수 있게 됩니다.
하지만 대가가 따릅니다. 내가 33바이트를 요청했는데 할당자가 48바이트 슬롯을 줘버리면, 남는 **15바이트는 영영 쓰지 못하고 버려집니다**. 이를 **내부 단편화(Internal Fragmentation)**라고 부르며, 속도를 얻는 대신 낭비를 감수하는 첫 번째 트레이드오프입니다.

참고로 glibc malloc 등은 자잘한 요청은 이렇게 처리하지만, 매우 큰 요청(보통 128KB 초과)은 이런 복잡한 과정을 거치지 않고 곧바로 `mmap`으로 커널에서 직수입해옵니다. 그래서 이런 덩치 큰 녀석들은 `free`를 호출하면 `munmap`을 통해 OS로 즉각 환불(반납)됩니다. (2편에서 "작은 메모리는 안 돌려주는데 큰 건 바로 돌려준다"고 했던 이유가 바로 이것입니다.)

## 🧵 jemalloc의 진짜 무기: tcache (락을 지워버리다)

*"그럼 도대체 jemalloc은 왜 이렇게 빠른 건가요?"* 이 질문에 대한 핵심 답변이 바로 **tcache(Thread-Local Cache)**에 있습니다.

순진한 메모리 할당 구조에서는 힙(Heap) 메모리가 프로세스 전체의 '공유 자원'입니다. 16개의 스레드가 동시에 객체를 만들려고 하면, 다 같이 하나의 글로벌 락(Lock) 앞에 줄을 서야 합니다. 1편에서 봤던 끔찍한 캐시 핑퐁 지옥이 또 펼쳐지는 거죠. 코어를 아무리 늘려도 성능이 오르지 않습니다.

이 병목을 박살 내기 위해 고안된 것이 **"스레드마다 자기만의 작은 전용 창고(tcache)를 주자"**는 아이디어입니다.

![Allocator funnel — tcache, then sharded arenas, then the kernel](/diagrams/allocator-internals-fragmentation-1.svg)

1. **tcache (1차 방어선):** 거의 99%의 `malloc`/`free`가 여기서 끝납니다. 내 스레드 전용 창고니까 락(Lock)도 필요 없고, 무거운 Atomic 연산도 없습니다.
2. **Arena (2차 방어선):** 내 창고가 텅 비었을 때만 공용 풀장(Arena)으로 내려갑니다. 이때도 전체 락을 걸지 않고, 공용 풀장을 여러 개(jemalloc 기준 코어 수의 약 4배)로 쪼개어 충돌 확률을 획기적으로 낮춥니다.
3. **OS 커널 (최후의 수단):** Arena마저 비어있을 때 비로소 커널에 `brk`나 `mmap`을 때려 새 땅을 받아옵니다. 제일 비싼 경로죠.

멀티스레드 환경에서 할당자 성능을 가르는 제일 결정적인 요인이 바로 이 락을 우회하는 `tcache` 구조입니다.
(물론 리눅스의 기본인 glibc도 2.26 버전부터 이 tcache를 도입했습니다. 그래서 단발성 워크로드에서는 glibc도 꽤 빠르지만, 트래픽이 치열하고 객체 생성이 폭발하는 장기 실행 서버 환경에서는 여전히 jemalloc이나 mimalloc의 구조적 우위가 크게 드러납니다.)

## 🧨 진짜 악당: 외부 단편화 (그리고 억울한 OOMKill)

다시 2편의 미스터리로 돌아가 보겠습니다. 코드를 완벽히 짜서 메모리를 전부 `free` 했는데도 RSS가 줄어들지 않았던 그 현상. 근본 원인은 **'외부 단편화(External Fragmentation)'**입니다.

| 종류 | 현상 | 결과 |
|---|---|---|
| **내부 단편화 (Internal)** | Size Class 올림 처리로 인해 *슬롯 내부에서* 발생하는 빈 공간 | 속도를 위한 어쩔 수 없는 희생 |
| **외부 단편화 (External)** | 전체 빈 공간은 충분한데, **파편처럼 여기저기 흩어져 있어서** 큰 요청을 못 맞추는 끔찍한 상태 | **살아있는 데이터는 줄었는데 RSS는 터지기 직전인 상태가 됨** |

예를 들어 힙 영역이 `[사용중][빈공간 16B][사용중][빈공간 16B][사용중]` 상태라고 해보겠습니다. 총 32바이트가 비어있지만, 누군가 연속된 32바이트를 요구하면 할당자는 이 공간을 내어줄 수 없어 커널에 새로 메모리를 요구하게 됩니다.
결국 할당자가 이 자잘하게 해제된 빈 슬롯들을 나중에 재활용하겠다고 꼭 쥐고 놔주지 않으니, 페이지가 OS로 반납되지 않아 RSS는 요지부동인 것입니다.

여기서 쿠버네티스(K8s) 환경의 치명적인 함정이 나타납니다.
**컨테이너의 실제 메모리 점유량(RSS) = (진짜 사용 중인 Live Heap) + (단편화로 낭비되는 공간) + (할당자가 쥐고 안 놔주는 페이지)** 입니다.
만약 K8s Limit을 개발 환경에서 측정한 순수 'Live Heap' 기준으로 타이트하게 잡아버리면, 운영 환경에서 파편화가 누적되는 순간 영문도 모른 채 OOMKill의 낫에 목이 날아갑니다.

## ♻️ jemalloc은 어떻게 OS에 램을 돌려주는가 (Decay 상태 기계)

그렇다면 좋은 할당자는 이 딜레마를 어떻게 풀까요? jemalloc은 빈 페이지를 쥐고 있다가, **시간이 지나면 '부패(Decay)'했다고 판단해 OS에 스무스하게 돌려주는 정책**을 씁니다.

![jemalloc decay — DIRTY to MUZZY to CLEAN, RSS drops only at the last step](/diagrams/allocator-internals-fragmentation-2.svg)

메모리가 해제되면 다음의 '식어가는' 과정을 거칩니다.

1. **DIRTY:** 방금 막 해제되었지만 아직 물리 메모리를 점유하고 있는 뜨끈뜨끈한 상태입니다. **모니터링 툴에 보이는 RSS 수치가 바로 이 녀석들입니다.**
2. **MUZZY:** `dirty_decay_ms` 설정 시간이 지나면 `MADV_FREE` 딱지가 붙습니다. 커널에게 "나 이거 당장은 안 쓰는데, 너 급하면 먼저 가져가서 써라"라고 알려주는 상태입니다. (하지만 커널이 여유로우면 안 가져가기 때문에 여전히 RSS 메트릭에 잡혀서 우리를 혼란스럽게 합니다.)
3. **CLEAN:** `muzzy_decay_ms`마저 지나면 `MADV_DONTNEED` 시스템 콜을 날려 커널에 물리 페이지를 강제로 반납해버립니다. **이 순간 비로소 RSS 그래프가 뚝 떨어집니다.**

> **💡 실무 튜닝 팁**
> OOM 킬러가 주시하는 진짜 타겟은 **DIRTY 페이지**입니다. 만약 파드의 메모리 Limit이 빡빡해서 OOMKill이 자주 난다면, 환경 변수 튜닝을 통해 `dirty_decay_ms` 값을 극단적으로 짧게 줘서 빈 페이지를 빨리 OS로 반납하게 만들 수 있습니다. (단, 잦은 반납은 필연적으로 잦은 재할당(`madvise` syscall 발생)을 부르기 때문에 응답 지연 시간(p99 Latency)이 튈 수 있다는 트레이드오프를 명심해야 합니다.)

## ⚖️ 할당자 3대장 비교: jemalloc vs tcmalloc vs mimalloc

최근 잘 나가는 3대장 모두 앞서 말한 tcache와 Arena 개념을 쓰지만, 각자의 철학과 강조점이 다릅니다.

| 할당자 | 철학과 핵심 전략 | 특징 |
|---|---|---|
| **jemalloc** | 촘촘한 Size Class와 Decay 기반의 깐깐한 반납 로직 | 파편화 방어력이 뛰어나 장기 실행되는 서버(ex. Meta/Facebook)에 강력함. |
| **tcmalloc** | per-CPU 기반 캐싱과 주기적 Release | 처리량(Throughput) 극대화에 초점. Google의 철학이 담김. |
| **mimalloc** | 스레드별 원격 해제(thread-free) 큐 분리 | 로컬 스레드 해제와 남이 해제해 주는 것을 분리해 락 없는(Lock-free) 해제를 지원. 극단적으로 작고 빠름 (Microsoft). |

특히 **mimalloc**의 트릭이 아주 예술입니다. A 스레드가 만든 객체를 B 스레드가 해제할 때 발생하는 무거운 동기화(Atomic) 비용을 피하려고, 스레드마다 **'원격 해제용 우편함(thread-free list)'**을 따로 둡니다. 생산자가 만들고 소비자가 버리는 구조, 즉 **Vector나 OpenTelemetry 같은 파이프라인 아키텍처**에서 캐시 지역성을 극대화하고 락 경합을 지워버리는 치트키로 작용합니다.

## 🪣 아예 묶어서 버리자: 특수 목적 할당자 (Bump / Arena)

만약 "할당 지연 시간 0초"에 도전하고 싶다면 튜닝을 넘어 아예 할당 방식을 바꿔야 합니다.

* **Bump Allocator (Arena):** 커널에서 거대한 메모리 운동장을 하나 떼어놓고, 할당 요청이 올 때마다 **포인터만 1칸씩 앞으로 미는(Bump)** 극단적인 방식입니다. 더하기 연산 한 번이면 할당이 끝납니다. 개별 객체의 `free`는 지원하지 않으며, 작업 배치가 끝나면 포인터를 맨 앞으로 돌려 한 번에 `reset` 해버립니다. 로그 파서 루프처럼 **"객체 1만 개를 순식간에 만들었다가 한 번에 다 버리는"** 패턴에 완벽한 무기입니다.

## 🦀 Rust에서 내 맘대로 주물러보기

Rust 생태계에서는 이 모든 밑바닥 기술을 놀랍도록 쉽게 제어할 수 있습니다.

* **전역 교체:** `#[global_allocator] static A: Mimalloc = Mimalloc;` 딱 한 줄이면 프로세스 전체의 룰을 바꿀 수 있습니다.
* **`allocator_api` (최신 트렌드):** 전역 교체가 부담스럽다면, `Vec::new_in(&my_arena)`처럼 특정 작업 루프 안에서만 개별 컬렉션에 특수 할당자를 주입할 수도 있습니다. (2026년 기준, 이 기능이 점점 표준화되는 추세입니다.)

## 🛠️ 실무 모니터링: 진짜 좋아졌는지 눈으로 확인하기

할당자를 바꿨다면 LGTM(Loki, Grafana, Tempo, Mimir) 대시보드에 증거를 남겨야겠죠? jemalloc은 자체 통계 API를 제공합니다.

```bash
# 빈 페이지 반납 주기를 1초로 짧게 당김 (RSS 방어 특화)
MALLOC_CONF="dirty_decay_ms:1000,muzzy_decay_ms:0" ./your_rust_server

# 종료 시 통계를 뱉어내도록 설정
MALLOC_CONF="stats_print:true" ./your_rust_server
```

Rust에서는 `tikv-jemallocator`와 `jemalloc-ctl` 크레이트를 써서 런타임에 이 통계를 Prometheus 메트릭으로 빼낼 수 있습니다.
대시보드에서 `active`(진짜 사용 중), `resident`(현재 RSS), `retained`(OS에 안 돌려준 메모리) 수치를 비교해 보세요. 만약 RSS가 터질 것 같은데 `active`는 바닥을 기고 있다면, 할당자 튜닝이 시급하다는 아주 강력한 시그널입니다.

## 📌 오늘의 핵심 요약

* 유저 스페이스 할당자는 커널(4KB 페이지)과 우리 코드(몇 바이트) 사이의 갭을 메우는 도매상입니다. **대부분의 `malloc`은 커널을 부르지 않습니다.**
* **Size Class**로 묶어 할당을 빛의 속도(O(1))로 처리하지만, 빈 공간이 흩어지는 **외부 단편화**가 컨테이너 OOM 킬러의 진짜 배후입니다.
* jemalloc 등 현대 할당자가 빠른 이유는 **tcache**를 통해 스레드 간의 락(Lock) 경합과 Atomic 연산을 원천 차단했기 때문입니다.
* jemalloc은 Decay 상태 기계 (DIRTY → MUZZY → CLEAN)를 통해 안 쓰는 메모리를 시간차를 두고 OS에 스무스하게 반납합니다.
