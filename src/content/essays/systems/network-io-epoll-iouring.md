> **시리즈 "Rust와 함께 가는 CS 레이어"의 7편 (마지막)**
> Rust의 꽃인 `async/await`가 궁극적으로 딛고 서 있는 바닥은 결국 운영체제(OS)의 I/O 모델입니다. 스레드 단 몇 개로 수만 개의 커넥션을 쥐락펴락할 때, 시스템 내부에서는 두 가지 거대한 긴장감이 흐릅니다. **"누가 기다릴 것인가(다중화)"**, 그리고 **"누가 일할 것인가(통지 모델)"**.
> 이 두 축의 진화 과정을 따라가다 보면, Rust의 Async 생태계(Tokio)가 왜 지금의 모습을 띠게 되었는지, 그리고 차세대 `io_uring` 생태계가 왜 패러다임을 통째로 뒤흔들고 있는지 완벽하게 이해할 수 있습니다. 7부작의 마지막 여정, 지금 시작합니다.

## 🤔 "스레드 한 줌으로 1만 개의 연결을 어떻게 다룰까?"

서버가 1만 개의 소켓 연결을 맺고 있다고 쳐봅시다. 사실 이 중 대부분의 시간은 '대기 상태(Idle)'고, 아주 가끔씩만 데이터를 주고받습니다.
가장 순진한 구현은 연결 1개당 스레드 1개를 할당해서 `read()` 함수에 블로킹(대기) 시켜두는 겁니다. 코드는 아주 짜기 쉽겠지만, 1만 개의 스레드가 갉아먹는 컨텍스트 스위칭 오버헤드와 메모리(스택) 때문에 서버는 비명을 지르며 뻗어버릴 겁니다. 이게 그 유명한 **C10k 문제**입니다.

그래서 시스템 엔지니어들은 고민하기 시작했습니다.
"스레드 하나가 수만 개의 소켓(fd)을 동시에 감시하려면 **어떻게 기다려야 할까?**" (select → poll → epoll의 역사)
그리고 "통지를 받은 뒤에 **누가 실제로 데이터를 퍼 나를 것인가?**" (epoll의 '준비 통지' vs io_uring의 '완료 통지').

이 역사를 관통하는 아주 중요한 숨은 변수가 하나 있습니다. 바로 **"시스템 콜(Syscall)은 절대 공짜가 아니며, 심지어 2018년 이후로 엄청나게 비싸졌다"**는 사실입니다. 이걸 알아야만 `io_uring`이 왜 그런 기괴하고 복잡한 구조로 설계되었는지 가슴으로 이해할 수 있습니다.

## 🧩 패킷 한 개의 험난한 여정 — NIC DMA와 NAPI

우리 앱이 `read()`를 호출하기도 *전에*, 네트워크로 날아온 데이터는 도대체 어디에 있을까요?

랜카드(NIC)는 CPU를 귀찮게 하지 않고 **DMA(Direct Memory Access)**라는 기술을 써서 시스템 RAM에 패킷을 직접 꽂아 넣습니다. 이를 위해 커널과 NIC는 미리 'RX 디스크립터 링(Ring)'이라는 빈 버퍼 목록을 약속해 둡니다.
NIC가 패킷을 받아 빈 버퍼에 채워 넣으면, CPU에게 "나 일 다 했어!"라며 **인터럽트(IRQ)**를 쏩니다. 그러면 커널이 깨어나서 TCP/IP 스택을 거쳐 찌꺼기를 떼어내고, 순수한 페이로드만 '소켓 수신 버퍼'에 예쁘게 쌓아둡니다. 나중에 우리 앱이 `read()`를 부르면 이 수신 버퍼의 데이터를 유저 스페이스로 복사해 가는 구조죠.

문제는 트래픽이 미쳐 날뛰어 초당 수백만 개의 패킷이 쏟아질 때 터집니다.
CPU가 패킷을 처리할 새도 없이 "나 일 다 했어!"라는 인터럽트만 처리하다가 시스템이 멈춰버리는 **Receive Livelock** 현상이 발생합니다.

이걸 해결하기 위해 리눅스 커널이 내놓은 기가 막힌 타협안이 **NAPI(New API)**입니다.
* 트래픽이 한산할 땐: 평소처럼 인터럽트(IRQ)를 써서 레이턴시를 최소화합니다. ("일 생기면 깨워줘!")
* 폭주하기 시작하면: **인터럽트를 아예 꺼버리고, 커널이 직접 주기적으로 버퍼를 확인하는 폴링(Polling) 모드로 전환**하여 링 버퍼를 뭉텅이로(Batch) 비워버립니다.

이 **"인터럽트(지연 최소) vs 폴링(처리량 최대)"**의 트레이드오프는 시스템 프로그래밍 내내 메아리치는 철학입니다. NAPI가 하드웨어와 커널 사이의 타협이었다면, 잠시 후 살펴볼 `io_uring`의 SQPOLL은 유저 스페이스와 커널 사이에서 똑같은 선택을 반복한 결과물입니다.

## 💸 2018년의 비극: 시스템 콜(Syscall) 청구서가 폭등하다

우리가 코드에 적는 `read(fd, buf, len)`은 단순한 함수 호출이 아닙니다. 유저 모드(Ring 3)에서 커널 모드(Ring 0)로 **특권 경계를 넘나드는 무거운 사건**입니다.
`syscall` 명령을 내리면 커널 진입점으로 점프하고, 유저 스택에서 커널 스택으로 전환하고, 끝나면 `sysret`으로 돌아와야 하죠. 이 과정에서 CPU 파이프라인이 깨지고 레지스터를 대피시키느라 수백 사이클이 날아갑니다.

그런데 **2018년, 이 비용이 끔찍하게 비싸지는 대사건**이 터집니다.

바로 하드웨어 투기 실행(Speculative Execution) 취약점인 **Meltdown(멜트다운)** 사태입니다. 해커가 이 버그를 이용해 유저 모드에서 커널 메모리를 훔쳐볼 수 있게 되자, 보안 진영은 **KPTI (Kernel Page Table Isolation)**라는 극약처방을 내립니다. 유저 페이지 테이블과 커널 페이지 테이블을 물리적으로 완전히 찢어버린 겁니다.
그 결과, 이제 시스템 콜을 호출할 때마다 **CR3 레지스터(페이지 테이블 베이스 주소)를 매번 통째로 갈아끼워야 하는** 대참사가 벌어졌습니다.

> **💡 잠깐! 2편과의 연결고리 (PCID)**
> CR3 레지스터를 바꾸면 원칙적으로 TLB(주소 변환 캐시)가 전부 날아가 버립니다. 2편에서 배웠던 그 끔찍한 4단계 페이지 워크(Page Walk)를 처음부터 다시 해야 하죠. 이 엄청난 성능 저하를 방어해 준 은인이 바로 2편에서 언급했던 **PCID (Process-Context ID)**입니다. TLB에 ID 꼬리표를 달아서, 페이지 테이블을 바꿔도 남의 캐시를 날리지 않게 막아준 덕분에 우리는 지금 그나마 버틸 만한 성능을 누리고 있습니다.

하지만 아무리 방어해도 KPTI 도입 이후 시스템 콜 자체가 눈에 띄게 비싸진 것은 사실입니다.
여기가 이 글의 핵심 척추입니다. **"하드웨어 취약점(Meltdown) → 소프트웨어 방어(KPTI) → 시스템 콜 단가 폭등 → 시스템 콜 횟수를 극한으로 줄이는 `io_uring`의 시대 도래"**라는 역사의 인과관계가 만들어진 것이죠.

## 🪜 기다림의 진화: select와 poll이 무너진 이유

하나의 스레드로 여러 소켓을 감시하는 다중화(Multiplexing)의 조상님은 `select`와 `poll`입니다.
`select`는 감시하고 싶은 파일 디스크립터(fd) 목록을 비트맵으로 커널에 넘기면, 커널이 0번부터 끝까지 쫙 훑어보고 "이거 이거 데이터 왔어"라고 체크해서 돌려줍니다.

하지만 치명적인 결함이 3가지나 있습니다.
1. 한 번에 감시할 수 있는 개수가 1024개로 하드코딩되어 있습니다 (FD_SETSIZE).
2. **O(n) 스캔 문제:** 1만 개의 커넥션 중 딱 1개만 데이터가 왔어도, 커널은 누가 보냈는지 알기 위해 1만 개를 다 뒤져봐야 합니다.
3. 매번 루프를 돌 때마다 유저 스페이스에서 커널 스페이스로 그 거대한 비트맵을 복사(Copy)해서 넘겨줘야 합니다.

## 🎯 epoll — "이벤트가 생기면 네가 직접 줄을 서라"

이 비효율의 극치를 구원하기 위해 리눅스에 **`epoll`**이 등장합니다.
가장 큰 철학적 변화는 **"관심 있는 소켓 목록을 커널에 영구적으로 저장(영속화)한다"**는 점입니다. `epoll_ctl`로 한 번만 커널에 등록해 두면, 루프를 돌 때마다 전체 목록을 넘길 필요가 없습니다.

![epoll internals — an interest RB-tree, a ready list, and a callback that moves fds onto it](/diagrams/network-io-epoll-iouring-1.svg)

`epoll`의 진짜 마법은 **콜백(Callback)**에 숨어 있습니다.
NIC로 패킷이 들어와서 특정 소켓이 읽기 가능한 상태가 되면, 네트워크 스택이 해당 소켓의 Wait Queue를 깨웁니다. 이때 `epoll`이 미리 심어둔 콜백 함수가 발동하면서, **준비된 소켓(fd) 스스로가 자신을 `epoll`의 준비 목록(Ready List)에 쓱 밀어 넣습니다.**

결정적인 역전입니다. 커널이 1만 개를 훑어보는 게 아니라, 이벤트가 발생한 소켓만 Ready List에 올라옵니다. 1만 개의 연결 중 5개만 활성화되었다면 딱 5개의 일만 하면 됩니다. 일의 양이 *전체 소켓 수*가 아니라 *실제 발생한 이벤트 수*에 비례하게 되면서 C10k 문제가 우아하게 해결되었습니다.

> **💡 레벨 트리거(LT) vs 엣지 트리거(ET)**
> * **LT (기본):** 소켓에 읽을 데이터가 남아있으면 `epoll_wait`을 부를 때마다 계속 "데이터 있어!"라고 알려줍니다. (안전하지만 귀찮음)
> * **ET (Edge Trigger):** 데이터가 없는 상태에서 **새로 들어온 그 찰나의 순간(Edge)에 딱 한 번만** 알려줍니다. 효율의 극치죠.
> Rust의 `tokio`가 의존하는 저수준 라이브러리인 `mio`는 **ET**를 기본으로 씁니다. 대신 ET는 아주 깐깐한 규칙을 강제합니다. 한 번 통지를 받으면 `EAGAIN` 에러(더 이상 읽을 게 없음)가 뜰 때까지 루프를 돌며 영혼까지 데이터를 빨아먹어야(Drain) 합니다. 안 그러면 큐에 데이터가 썩어 있어도 영원히 통지를 받지 못합니다.

## 🧱 epoll이 부딪힌 한계: "알려만 줄게, 일은 네가 해"

`epoll`은 기적 같았지만, 패러다임의 한계가 명확했습니다. 어디까지나 **준비 통지(Readiness Notification)** 모델이라는 점입니다.
"이제 블로킹 없이 읽을 수 있어!"라고 신호만 줄 뿐, 실제로 데이터를 퍼 나르는 건 여전히 유저 스페이스의 몫입니다.

이벤트 1개가 발생하면:
1. `epoll_wait` (Syscall)로 신호를 받고,
2. `read` (Syscall)를 호출해서 커널 버퍼의 데이터를 유저 버퍼로 복사하고,
3. `write` (Syscall)로 응답을 보냅니다.

앞서 말했죠? KPTI 이후 시스템 콜은 엄청나게 비싸졌습니다. `epoll`은 '감시 비용'은 0에 가깝게 줄였지만, **'실제 I/O 작업을 위한 Syscall 횟수와 복사 비용'**은 단 1원도 깎아주지 못했습니다. 이 두꺼운 벽을 때려 부수기 위해 등장한 것이 바로 **완료(Completion) 모델**입니다.

## 🚀 io_uring — 시스템 콜의 종말, 그리고 공유 메모리 링

2019년 리눅스 5.1 커널에 Jens Axboe가 혜성처럼 내놓은 **`io_uring`**은 다음 4가지 무기로 I/O의 판도를 뒤집어 엎어버립니다.

![io_uring — submission and completion rings shared between user and kernel via mmap](/diagrams/network-io-epoll-iouring-2.svg)

1. **준비가 아니라 완료(Completion) 통지:** "읽을 수 있으니까 네가 알아서 퍼가"가 아닙니다. **"저 소켓에서 데이터 1KB 읽어서 이 메모리 주소에 갖다 놔. 다 되면 알려줄게"**라고 작업 전체를 커널에 외주(위임)를 줍니다.
2. **mmap된 두 개의 링(Ring):** 커널과 유저 스페이스 사이에 메모리를 공유(`mmap`)해서 큐를 만듭니다. 우리는 '제출 큐(SQ)'에 주문서를 적어두고, '완료 큐(CQ)'에서 결과물을 꺼내옵니다.
3. **극한의 Syscall 상각(Amortization):** 주문서를 SQ에 100개 적어두고 `io_uring_enter` 시스템 콜 딱 한 번만 부르면 커널이 100개를 다 처리해 줍니다.
4. **SQPOLL (Syscall 0번의 마법):** 아예 커널 쪽에 폴링 스레드를 하나 상주시킬 수도 있습니다. 우리는 그냥 공유 메모리(SQ)에 주문서만 쓱 적어두면, 커널 스레드가 알아서 눈치채고 가져갑니다. **시스템 콜 횟수가 말 그대로 '0번'이 됩니다.**

> **💡 소름 돋는 연결고리: 4편 Memory Ordering의 실전 무대**
> 유저 스페이스와 커널 스페이스는 서로 다른 CPU 코어에서 돕니다. 한쪽은 SQ에 주문을 넣고(생산자), 한쪽은 가져가죠(소비자). 이 링 버퍼의 Head/Tail 포인터를 업데이트할 때 어설프게 동기화하면 어떻게 될까요? 데이터가 써지기도 전에 포인터가 넘어가서 커널이 쓰레기 값을 읽어갑니다!
> 그래서 `io_uring`의 C 구조체를 뜯어보면 데이터를 기록한 *후에* Tail을 올리는 **Release**, Tail을 읽은 *후에* 데이터를 읽는 **Acquire** 오더링이 글자 그대로 박혀있습니다. 4편에서 배운 흑마법이 여기서 빛을 발하는 겁니다.

## 📦 완료 모델의 치명적 아킬레스건 — 버퍼(Buffer)를 잡아먹는 괴물

완료 모델은 완벽해 보이지만, 아키텍처적으로 끔찍한 약점이 하나 있습니다.

`epoll`(준비 모델)은 데이터가 들어왔다는 알람이 울린 *후에* 버퍼를 할당해서 `read`를 호출하면 됩니다. 10만 개의 커넥션이 있어도 실제로 활성화된 몇 백 개만큼의 버퍼 메모리만 있으면 충분하죠.
하지만 `io_uring`(완료 모델)은 **"주문을 넣을 때 미리 빈 버퍼를 같이 바쳐야"** 합니다. 10만 개의 소켓에 "데이터 오면 읽어주세요"라고 예약을 걸려면, 데이터가 당장 오지도 않을 텐데 10만 개 분량의 메모리 버퍼를 미리 잡아두고 파킹(Parking) 시켜놔야 합니다. 시스템 메모리(RSS)가 순식간에 거덜 나는 폭탄입니다.

이 한계를 부수기 위해 `io_uring`은 **'제공된 버퍼 링(Provided Buffer Ring)'**이라는 엔지니어링 마법을 부립니다.
버퍼를 소켓마다 매칭해서 던져주는 게 아니라, 커널 쪽에 "빈 버퍼 풀장"을 아예 던져줍니다. 나중에 진짜 데이터가 도착하면 커널이 알아서 풀장에서 빈 버퍼를 쏙 뽑아 쓰고, CQ에 "내가 42번 버퍼에 데이터 채워놨어~"라고 번호표만 돌려주는 방식입니다. 이로써 `io_uring`은 메모리 폭발을 막고 다시 실전 최강자로 군림하게 되었습니다.

## 🦀 Rust Async와 소유권(Ownership)의 웅장한 충돌

자, 이제 이 긴 여정의 결론이자, 이 시리즈 전체를 관통하는 하이라이트입니다.
Rust의 `async/await` 시스템은 왜 지금의 형태를 갖게 되었을까요?

**기본적으로 Rust의 `tokio`는 완벽한 epoll(준비) 모델입니다.**
Future의 `poll()` 함수가 `Poll::Pending`을 돌려준다는 것 자체가 다분히 "아직 소켓이 준비 안 됐어"라는 의미입니다. "대기 → 알람 울림 → Waker로 태스크 깨움 → 다시 `poll()` 시도". 이 폴링 패턴은 `epoll`의 동작 원리와 형태학적으로 완벽하게 똑같습니다(동형, Isomorphic).

하지만 `io_uring`(완료 모델)을 Rust에 얹으려고 하자 **엄청난 딜레마**가 터져버렸습니다.
커널에게 "이 버퍼 주소(`&mut buf`)에 데이터 채워줘!"라고 비동기 위임을 맡겼는데, 만약 유저 스페이스에서 타임아웃 등으로 Future가 취소(Cancel/Drop)되면 어떻게 될까요?
Rust 컴파일러는 메모리를 안전하게 해제해 버렸지만, **커널은 아직 쥐고 있던 주소인 줄 알고 거기다 냅다 네트워크 패킷을 들이붓습니다.** 이게 바로 힙 메모리가 오염되는 최악의 재앙, **Use-After-Free** 현상입니다. (이른바 취소 안전성, Cancellation Safety 문제).

이 문제를 해결할 유일한 방법은 **'Owned Buffer(소유된 버퍼)'**를 사용하는 것뿐입니다.
기존처럼 버퍼를 살짝 빌려주는(`&mut buf`) 게 아니라, `Vec<u8>`의 **소유권(Ownership) 자체를 런타임(커널 쪽)으로 통째로 넘겨버리는 겁니다.** 작업이 끝날 때까지 버퍼가 소멸되는 걸 원천 차단하고, 작업이 성공하면 CQE와 함께 버퍼의 소유권을 다시 반환받습니다. (현재 `tokio-uring`이나 `glommio`, `monoio` 같은 io_uring 기반 런타임들이 모두 이 방식을 강제하고 있습니다.)

이것이 1편에서 시작했던 소유권(Ownership)의 종착지입니다.
Borrow Checker는 컴파일러가 보는 세계 안에서만 유효합니다. **Borrow Checker의 눈이 닿지 않는 저 너머의 '커널 경계'에서 수명을 강제하려면, 물리적으로 소유권 자체를 이전해야만 합니다.** 이것이 바로 시스템 프로그래밍에서 소유권 철학이 물리적 런타임 구현으로 발현되는 궁극의 형태입니다.

## 🌐 성능 스펙트럼 — 커널을 아예 우회해버리자 (Kernel Bypass)

지금까지의 여정을 시스템 콜(Syscall) 비용이라는 축으로 세워보면 완벽한 스펙트럼이 완성됩니다.

`블로킹` → `select/poll` → `epoll` → `io_uring (배치)` → `io_uring (SQPOLL)` → **`AF_XDP / DPDK`**

`io_uring`이 "커널 I/O를 극한으로 싸게 만들자"는 철학이라면, 오른쪽 끝의 **커널 바이패스(Kernel Bypass)**는 **"아예 커널을 무시해버리자!"**는 철학입니다.
DPDK는 유저 스페이스 애플리케이션이 직접 NIC 하드웨어를 통제하고 코어 하나를 100% 점유하며 폴링해버립니다. AF_XDP는 eBPF 기술을 이용해 패킷이 커널 TCP/IP 스택에 올라타기도 전에 NIC 드라이버 맨 앞단에서 가로채버립니다. K8s의 Cilium CNI 플러그인이 엄청난 성능을 내는 원리가 바로 이 지점에 닿아있습니다.

## 🛠️ 실무 운영 팁: 내 서버 병목 진단하기

지금 내 서버의 I/O가 어디서 병목인지 눈으로 확인해 보세요.

```bash
# 1. 시스템 콜 횟수와 종류 집계 (syscall-bound 확인)
strace -c -p <pid>

# 2. 코어별 softirq 비중 점검 (특정 코어에 네트워크 인터럽트가 몰리는지 확인)
mpstat -P ALL 1

# 3. 소켓 수신 큐(Recv-Q) 적체 확인
ss -tan state established
```

만약 `strace`를 켰을 때 `epoll_wait`나 `read`/`write`의 호출 수와 점유 시간이 터무니없이 높다면, 당신의 워크로드는 Syscall-bound 상태입니다. 이때가 바로 `io_uring` 도입을 진지하게 고민할 시점입니다.
만약 `mpstat`에서 특정 코어의 `%soft` 수치가 100%를 치고 있다면, NIC 인터럽트가 한 코어에 몰려 죽어가고 있는 겁니다. (이때는 인프라 레벨에서 RSS/RFS 설정을 통해 패킷 수신을 여러 코어로 분산시켜야 합니다.)

> **⚠️ 보안 주의보:** `io_uring`은 커널 깊숙한 곳과 공유 메모리로 통신하기 때문에 공격 표면(Attack Surface)이 굉장히 넓습니다. 실제로 권한 탈취 취약점(CVE)이 자주 터졌고, K8s나 Docker의 기본 `seccomp` 프로파일이 `io_uring` 관련 syscall을 막아두는 경우가 많습니다. 운영 환경에 도입하기 전에 클러스터의 seccomp 정책을 반드시 확인하세요!

## 📚 시리즈를 마치며: 왜 Rust로 CS 밑바닥을 파야 하는가?

7개의 레이어를 쉴 새 없이 관통한 여정이 마침내 끝났습니다.
처음엔 파편화된 주제 같았지만, 사실 이 모든 건 하나의 거대한 톱니바퀴처럼 맞물려 있습니다.

* 1편에서 봤던 캐시 라인 핑퐁과 Store Buffer는 **4편 Memory Ordering**의 근본 원인이었습니다.
* 2편의 가상 메모리 매핑과 OOMKill은 **3편 할당자 파편화**와 4편 Lock-free 회수 정체(EBR)가 만든 환장의 콜라보였습니다.
* 5편의 인라인(Noalias) 최적화는 **6편 ABI 경계**에서 차갑게 멈춰 섰습니다.
* 6편에서 "Borrow Checker는 FFI 경계를 넘지 못한다"던 그 한계는, 7편에서 OS 커널 경계의 Owned Buffer(소유권 이전)라는 형태로 극복되었습니다.

이 시리즈의 핵심 명제는 처음부터 끝까지 동일합니다.
**"Rust는 다른 언어들이 런타임 뒤에 예쁘게 숨겨두었던 CS(컴퓨터 구조/OS)의 끔찍한 실체들을 문법의 표면 위로 강제로 끄집어냅니다."**

`#[repr]`, `Send/Sync`, `unsafe`, 그리고 `소유권(Ownership)`.
처음엔 컴파일러의 억지스러운 잔소리 같던 이 문법들이 사실은 CPU 캐시, 페이지 테이블, 메모리 할당자, 동시성 오더링, ABI, 커널 I/O 모델이라는 거대한 시스템 위에 정교하게 얹혀 있는 '안전벨트'였다는 걸 이제 우리는 압니다.

Rust를 끝까지 집요하게 파고들면, 결국 내 손에 남는 건 '새로운 언어 문법' 하나가 아닙니다. **시스템 전체를 수직으로 투시할 수 있는 '엔지니어의 거대한 시야'가 남습니다.** 그리고 이 시야를 한 번이라도 얻게 된 개발자는, 훗날 Go 언어의 Escape Analysis를 튜닝하든, Java의 G1GC 병목을 잡든, C++의 메모리 릭을 쫓든, 어떤 환경에서도 두려움 없이 시스템의 심장부를 향해 메스를 들이댈 수 있는 진짜 '시니어'로 성장하게 될 것입니다.

그동안 기나긴 시리즈를 함께해 주셔서 감사합니다! 🦀
