> **시리즈 "Rust와 함께 가는 CS 레이어"의 2편**
> 1편에서 CPU 캐시라는 하드웨어의 밑바닥을 찍고 왔다면, 이번엔 한 칸 위인 운영체제(OS) 계층으로 올라옵니다. 우리가 무심코 호출하는 `Box`나 `Vec` 할당이 결국 어디로 가는지, 그리고 가상 주소에서 물리 페이지까지 이어지는 매핑 과정을 모르면 컨테이너가 왜 OOMKill을 맞는지 영원히 이해할 수 없습니다.
> 이번 편의 핵심 질문은 단 하나입니다. **"분명 로직에서 `free`를 다 했는데, 왜 서버 메모리 그래프는 내려가지 않을까요?"**

## 🤔 "분명 메모리를 해제했는데, 왜 그대로지?"

K8s 환경에서 파드를 운영하다 보면 꽤 자주, 그리고 가장 혈압이 오르는 상황을 마주하게 됩니다. 컨테이너의 메모리 사용량 그래프가 톱니바퀴처럼 오르내리는 게 아니라, 슬금슬금 우상향하며 Limit을 향해 차오르다 어느 순간 파드가 비명을 지르며 죽어버립니다. 로그를 보면 선명하게 찍혀 있는 `OOMKilled`.

"아니, 내가 짠 코드엔 분명 다 쓴 메모리를 `free`(Rust라면 `Drop`)로 꼬박꼬박 해제하고 있는데 도대체 왜?!"

직관과 정면으로 충돌하는 이 기묘한 현상의 해답은 애플리케이션 코드가 아니라, 그 밑에 깔린 OS의 메모리 관리 정책에 있습니다. "메모리를 할당한다"는 행위의 진짜 의미와, "해제한다"고 해서 **물리 메모리를 즉시 돌려주는 게 아니라는 사실**을 깨닫는 순간, OOMKill은 미스터리가 아니라 아주 예측 가능한 시스템의 동작으로 보이게 됩니다. 그 여정의 첫걸음은 '가상 메모리'에서 시작합니다.

## 🧩 가상 메모리 — 할당은 "약속"일 뿐, "실체"가 아니다

우리 프로그램(프로세스)이 바라보는 메모리 주소는 100% **'가상 주소(Virtual Address)'**입니다. 실제 RAM에 박혀 있는 물리 주소가 아니죠. 이 둘 사이를 '페이지 테이블(Page Table)'이 매핑해주고, CPU의 MMU가 메모리에 접근할 때마다 가상 주소를 물리 주소로 실시간 번역해 줍니다.

여기서 반드시 짚고 넘어가야 할 핵심은, 이 OS의 매핑 작업이 **상상 이상으로 극단적인 게으름(Lazy)**을 피운다는 점입니다.

우리가 메모리를 할당(malloc)하면 OS는 가상 주소 영역에 공간을 '예약'만 해둡니다. "어, 너 나중에 쓸 공간 여기 확보해 뒀어~" 하고 장부(페이지 테이블)에만 적어둘 뿐, 실제 물리적 RAM 공간(페이지)은 단 한 장도 내어주지 않습니다.
그러다가 우리 프로그램이 그 주소에 **'처음으로 값을 쓰는 순간(First Touch)'**, 그제야 Page Fault라는 인터럽트가 발생하며 커널이 화들짝 놀라 실제 물리 페이지 한 장을 매핑해 줍니다. 이를 요구 기반 페이징(Demand Paging)이라고 부릅니다.

결국 메모리 할당이란 "필요하면 줄게"라는 커널의 부도 수표 같은 약속이고, 실제 RAM 점유는 우리가 그 메모리를 건드리는 찰나의 순간에 이루어집니다. 그래서 리눅스 프로세스의 메모리는 늘 두 가지 지표로 쪼개서 봐야 합니다.

- **VmSize (가상 크기):** 예약한 가상 주소 공간의 총량입니다. 할당(malloc)하는 순간 즉각적으로 치솟습니다.
- **RSS (Resident Set Size):** 실제로 RAM을 점유한 물리 페이지의 진짜 크기입니다. 예약해 둔 공간 중 **'처음으로 건드린'** 공간만큼만 잡힙니다.

여기서 가장 중요한 팩트 하나. OOM 킬러(OOMKiller)의 눈길을 끄는 범인은 VmSize가 아니라 **RSS**입니다. "10GB를 할당했다"가 곧 "RAM 10GB를 썼다"가 아닌 이유가 바로 이 게으른 할당 방식 때문입니다.

## 🧱 Box::new 한 줄이 커널까지 가는 험난한 길

Rust에서 무심코 작성하는 `Box::new(v)`나 `Vec::push`는 결국 힙(Heap) 메모리 할당입니다. 이 코드 한 줄이 커널 깊숙한 곳까지 어떻게 도달하는지 그 경로를 따라가 볼까요?

![From Box::new to a physical frame — allocation reserves virtual space, first touch commits RAM](/diagrams/virtual-memory-and-oomkill-1.svg)

1. **`Box::new(v)`:** 언어 레벨에서 값을 힙에 올려 달라고 할당자(Allocator)에게 요청합니다.
2. **할당자 (malloc) → `brk` / `mmap`:** 유저 스페이스의 할당자가 커널에게 가상 주소 영역을 요구합니다. 작은 크기는 기존 힙의 끝을 미는 `brk`로, 큰 크기(보통 128KB 이상)는 `mmap`을 통해 새로운 영역을 큼직하게 받아옵니다. **이 단계에서 VmSize가 오릅니다.**
3. **가상 페이지 확보:** 가상 주소의 공간은 잡혔지만, 물리 페이지(RAM)는 아직 텅 비어있는 상태입니다.
4. **최초 쓰기 (First Write) → Page Fault:** 우리 코드가 그 주소에 실제 값을 밀어 넣는 순간, 페이지 폴트(#PF)가 터지며 커널 모드로 트랩(Trap)됩니다.
5. **물리 프레임 매핑:** 커널이 부랴부랴 물리 메모리(RAM) 페이지 한 장을 할당해서 가상 주소와 연결해 줍니다. **비로소 이 순간에 RSS가 오릅니다.**

> **💡 시니어의 팁: NUMA와 First-Touch의 비밀**
> 1편에서 배웠던 NUMA 아키텍처 기억나시나요? 5단계에서 커널이 물리 페이지를 떼어줄 때, **"어느 CPU 소켓(NUMA 노드)에 붙은 RAM을 줄 것인가?"**를 결정하는 기준이 바로 "이 메모리를 '처음 건드린(First-Touch)' 스레드가 어느 노드에 속해 있는가"입니다. 할당만 잔뜩 해둔 스레드와, 나중에 실제로 값을 읽고 쓰는 스레드가 물리적으로 멀리 떨어져 있다면? 매번 원격 노드의 RAM을 긁어오느라 성능이 나락으로 떨어집니다.

이 일련의 과정을 이해해야만 Rust에서 `#[global_allocator]`를 jemalloc이나 mimalloc으로 교체하는 행위가 단순한 마법이 아니라, `brk`와 `mmap`을 제어하는 유저 스페이스의 지휘관을 바꾸는 일이라는 걸 깨닫게 됩니다.

## 💥 분명 free 했는데, RSS 그래프는 왜 그대로일까?

이제 진짜 우리를 괴롭혔던 핵심 질문으로 돌아옵시다. 로직 처리가 끝난 메모리를 해제하면 무슨 일이 벌어질까요?

`free`(또는 Rust의 `Drop`)를 호출한다고 해서, 그 메모리가 **즉시 커널(OS)에 반납되는 게 절대 아닙니다.** 다 쓴 메모리는 OS가 아니라 **'유저 스페이스의 할당자(Allocator)'**에게 돌아갑니다. 할당자는 나중에 또 메모리 달라고 `brk`나 `mmap` 시스템 콜을 부르면 비용이 너무 비싸니까, 이 해제된 메모리를 일단 자기 주머니(Free List)에 꽉 쥐고 있습니다.

문제는 할당자가 메모리를 쥐고 있는 동안, 그 물리 페이지는 여전히 우리 프로세스에 매핑된 채로 남아있다는 점입니다. 즉, **"코드 레벨에서 해제(Free)를 완벽하게 했어도 시스템 메트릭의 RSS 수치는 절대 떨어지지 않는다"**는 충격적인 사실입니다.

할당자가 이 메모리를 진짜 OS에 돌려주는 건 아주 까다로운 조건이 맞을 때뿐입니다. `mmap`으로 통째로 빌려온 큰 덩어리는 `munmap`으로 깔끔하게 반납되지만, 잘게 쪼개서 힙(`brk`)으로 쓴 영역들은 힙의 '맨 끝 연속된 공간'이 완전히 비워져야만 크기를 줄일 수 있습니다. 중간중간 듬성듬성 이빨이 빠진 상태(외부 단편화)라면 끝부분을 반납할 수가 없습니다.

이게 바로 컨테이너 OOMKill의 가장 흔한 시나리오입니다.
메모리 릭(Leak)이 없더라도, 트래픽 스파이크 때 최대로 할당받았던 메모리를 할당자가 파편화(Fragmentation) 때문에 꽉 쥐고 놓지 않으면 RSS는 계속 Limit 근처를 배회하게 됩니다. (할당자가 도대체 어떻게 생겨먹었길래 이러는지, 그리고 이걸 jemalloc이 어떻게 해결하는지는 3편에서 아주 딥하게 파헤칩니다!)

## ♻️ 진짜 RAM이 바닥나면? — 리눅스 커널의 회수와 OOM Killer

프로세스들이 메모리를 쥐고 안 놔줘서 진짜 시스템의 물리 RAM 여유분이 워터마크 밑으로 떨어지면 리눅스는 어떻게 대응할까요? 무작정 다짜고짜 프로세스를 죽이진 않습니다. 단계별로 칼을 뽑아 듭니다.

![Kernel reclaim escalation — kswapd, then direct reclaim, then the OOM killer](/diagrams/virtual-memory-and-oomkill-2.svg)

1. **메모리 압박 (Memory Pressure):** 여유 RAM이 부족해진 걸 커널이 감지합니다.
2. **kswapd 백그라운드 회수:** 커널의 청소부 스레드인 `kswapd`가 조용히 깨어나 안 쓰는 페이지들을 스왑(Swap) 아웃시키거나 캐시를 날리며 회수합니다. 이때 애플리케이션은 멈추지 않습니다.
3. **💥 Direct Reclaim (직접 회수):** 백그라운드 청소부의 속도보다 애플리케이션이 메모리를 처묵처묵하는 속도가 더 빠르면? 커널은 "안 되겠다, 너네가 직접 치워!"라며 **메모리를 요청한 애플리케이션 스레드 자체를 멈춰 세우고 강제로 메모리 회수 작업에 투입**시킵니다.
4. **OOM Killer 강림:** 3번까지 했는데도 답이 없으면, 커널은 `oom_score`를 계산해 가장 뚱뚱하고 쓸모없는 프로세스 하나를 골라 `SIGKILL` 낫을 휘두릅니다.

> **💡 실무 장애 추적 팁:**
> 운영 환경에서 가장 무서운 건 4번(OOM)이 아니라 **3번(Direct Reclaim)**입니다. 평소엔 쌩쌩하던 API 응답 속도(p99 Latency)가 가끔씩 이유 없이 수십~수백 ms씩 튀어 오르나요? 십중팔구 여러분의 스레드가 멈춰 서서 Direct Reclaim으로 남의 쓰레기를 치우고 있었을 확률이 높습니다.

K8s 환경에서는 파드의 Memory Limit을 넘는 순간, 노드 전체가 멀쩡해도 해당 cgroup 내부에서 자체적인 OOM 킬러가 파드를 쏴 죽입니다. 앞서 말한 "할당자의 잦은 파편화 → 반납 불가 → RSS 팽창 → Direct Reclaim → OOMKill"이라는 끔찍한 연쇄 작용의 퍼즐이 이제 완벽히 맞춰지셨을 겁니다.

## 🧬 COW (Copy-On-Write) — 운영체제의 게으름이 Rust의 타입으로

리눅스에서 프로세스를 복제하는 `fork()` 시스템 콜은, 부모 프로세스의 거대한 메모리를 즉시 통째로 복사하는 미련한 짓을 하지 않습니다. 대신 부모와 자식이 동일한 물리 페이지를 'Read-Only(읽기 전용)' 상태로 같이 공유하게 둡니다.
그러다 둘 중 하나가 그 데이터를 **'수정(Write)'**하려고 시도하는 찰나의 순간에만 Page Fault를 발생시켜, 그때 딱 그 페이지만 살포시 복사해 줍니다. 이것이 그 유명한 **Copy-On-Write(COW)**입니다.

이 OS 레벨의 메커니즘을 그대로 소프트웨어 디자인 패턴으로 끌고 온 것이 바로 Rust의 표준 라이브러리 타입인 `Cow<T>`(Clone-On-Write)입니다! 평소에는 빌림(`Borrowed`) 상태로 가볍게 데이터를 같이 쓰다가, 무언가 변경이 필요할 때만 복제(`clone`)를 수행해 소유권(`Owned`)을 가집니다.

OS 커널의 밑바닥 최적화 철학이, 현대적인 언어의 타입 시스템(Type System) 이름에 그대로 아로새겨져 있다는 사실. CS 레이어 전체를 관통하는 이런 연결고리를 발견할 때의 짜릿함이 바로 시스템 프로그래밍의 진짜 묘미가 아닐까요?

## 🔀 스레드를 늘린다고 무조건 빨라지지 않는 이유 (Context Switch)

쓰레드를 10개에서 100개, 1000개로 늘리면 처리량(Throughput)이 오를까요? OS 레벨에서 보면 완벽한 오산입니다. 스레드가 전환되는 '컨텍스트 스위치(Context Switch)'의 청구서는 생각보다 비쌉니다.

- **직접 비용:** CPU 레지스터 백업/복원, 유저 모드와 커널 모드 간의 전환(Syscall), TLB 갱신 오버헤드.
- **간접 비용 (캐시 오염 ☠️):** 더 치명적인 건 이겁니다. 새로운 스레드가 들어와서 L1/L2 캐시를 자기 데이터로 싹 다 덮어써버립니다. 나중에 쫓겨났던 원래 스레드가 돌아와 보면 자기 캐시는 다 날아가고 싸늘하게 식어버린(Cold Cache) 상태입니다.

그래서 스레드 개수가 CPU 코어 수를 아득히 초과하면 시스템은 일은 안 하고 스레드 교체와 캐시 복구만 하다가 죽어버립니다. 이를 해결하기 위해 Rust의 `tokio` 같은 비동기(Async) 런타임들이 수만 개의 코루틴(Task)을 단 몇 개의 OS 스레드 위에서 굴리는 멀티플렉싱 구조를 취하는 겁니다. (이 흥미진진한 비동기 I/O 모델에 대해서는 5편과 7편에서 본격적으로 해부하겠습니다!)

## 🛠️ 실무에서 내 파드 메모리 직접 눈으로 확인하기

백문이 불여일견입니다. 지금 운영 중인 리눅스 서버나 컨테이너에서 가상 메모리와 물리 메모리의 괴리를 직접 숫자로 확인해 보세요.

```bash
# 내 프로세스의 가상 크기(VmSize)와 실제 물리 점유(VmRSS) 나란히 비교하기
grep -E 'VmSize|VmRSS' /proc/<pid>/status

# K8s 컨테이너(cgroup v2)의 현재 메모리 사용량과 한계치 확인
cat /sys/fs/cgroup/memory.current
cat /sys/fs/cgroup/memory.max

# 메모리 할당이 진짜 brk와 mmap으로 분기되는지 커널 콜 추적하기
strace -e trace=brk,mmap,munmap ./your_program
```

큰 버퍼 배열을 할당만 해두고 값을 채우지 않으면 `VmSize`만 치솟고 `VmRSS`는 꼼짝도 하지 않는 걸 볼 수 있습니다. 그러다 버퍼를 전부 0으로 채우는(First Touch) 순간 `VmRSS`가 급발진하며 따라붙습니다.
또한 메모리를 해제했는데도 `VmRSS`가 즉시 떨어지지 않는 현상을 두 눈으로 직접 본다면, "free ≠ RSS 즉시 반납"이라는 오늘의 주제가 뼈저리게 와닿을 겁니다.

## 📌 오늘의 핵심 요약

* 메모리 할당(`malloc`, `Box::new`)은 가상 주소 공간만 **예약**할 뿐, 실제 물리 페이지는 **처음 건드릴 때(First Touch)** Page Fault를 통해 붙습니다.
* VmSize(가상)와 RSS(물리)는 완전히 다른 지표입니다. K8s의 OOM 킬러는 **RSS**를 보고 목을 칩니다.
* `free`나 `Drop`을 호출해도 메모리는 유저 스페이스의 할당자(Allocator)에게 돌아갈 뿐, 커널에 즉시 반납되지 않아 RSS는 떨어지지 않습니다.
* 메모리가 부족하면 커널은 `kswapd` → **`Direct Reclaim`** → `OOM Killer` 순으로 대응하며, `Direct Reclaim`이 p99 지연 시간 스파이크의 숨은 주범입니다.
* OS의 게으른 복제 기법인 **COW**는 Rust의 `Cow<T>` 타입으로 이어지며, 무분별한 스레드 증가는 캐시 오염(Cache Pollution)을 유발해 성능을 갉아먹습니다.
