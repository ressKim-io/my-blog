## 한 줄 요약

> 컨테이너 이미지의 재현성은 다이제스트(SHA256 해시)에 의해 보장됩니다 — 중앙 레지스트리에서 한 번 검증한 이미지를 각 CSP 네이티브 레지스트리로 복제하면, 공급망 보안과 pull 지연 최소화를 동시에 달성할 수 있습니다

---

## 🤔 무엇을 푸는 기술인가

컨테이너 이미지 공급망(supply chain)이란 소스 코드부터 실행 중인 컨테이너까지의 전 과정입니다 코드 커밋 → 빌드 → 이미지 레지스트리 저장 → 클러스터 pull → 실행 컨테이너, 이 각 단계에서 이미지가 변조되거나 예상과 다른 내용으로 실행될 위험이 존재합니다

공급망 보안이 다루는 핵심 문제는 두 가지입니다

- **재현성(reproducibility)**: 어느 환경, 어느 시점에 pull해도 같은 이미지가 실행된다는 보장
- **무결성(integrity)**: 레지스트리에 저장된 이미지가 빌드된 이후 변조되지 않았다는 보장

이 두 문제는 멀티 CSP 환경에서 더 복잡해집니다 CSP마다 독립적으로 이미지를 빌드하면 레이어 내용이 같더라도 빌드 환경 차이로 해시가 달라질 수 있습니다 이 때문에 "어떤 레지스트리에 어떤 이미지를 어떻게 배치할 것인가"는 단순한 인프라 배치 결정을 넘어 보안·비용·운영 복잡도를 동시에 고려하는 아키텍처 결정입니다

## 🔧 동작 원리

### 이미지 다이제스트가 재현성을 보장하는 방법

컨테이너 이미지는 레이어(layer)의 스택으로 구성됩니다 각 레이어는 파일시스템 변경분을 담은 tar 아카이브이며, 이 tar를 SHA256으로 해시한 값이 **레이어 다이제스트**입니다

이미지 전체는 여러 레이어 다이제스트와 이미지 설정(config)을 담은 **매니페스트(manifest)**로 표현됩니다 이 매니페스트를 다시 SHA256으로 해시한 값이 **이미지 다이제스트**입니다

```text
이미지 다이제스트 = SHA256(매니페스트)
매니페스트       = { config, [layer1_digest, layer2_digest, ...] }
레이어 다이제스트 = SHA256(레이어 tar)
```

다이제스트 기반 pull의 예시는 아래와 같습니다

```bash
# 태그 기반 pull — 같은 명령이 다른 이미지를 가리킬 수 있음
docker pull myregistry/api:v1.2.3

# 다이제스트 기반 pull — 절대 같은 이미지를 가리킴
docker pull myregistry/api@sha256:a3f8c2e1d7b94...
```

태그(`v1.2.3`)는 언제든 다른 다이제스트를 가리키도록 덮어쓸 수 있습니다 반면 다이제스트는 내용에서 파생되므로, 다이제스트가 같으면 내용도 반드시 같습니다 이것이 **다이제스트 기반 배포가 재현성을 보장하는 이유**입니다

Kubernetes는 이 원리를 이용해 이미지 고정을 권고합니다 `image: myregistry/api:v1.2.3` 대신 `image: myregistry/api@sha256:a3f8c2e1d7b94...`로 지정하면, 태그 재지정이나 레지스트리 오염으로 인한 불일치를 방지할 수 있습니다

### CSP별 독립 빌드 시 해시 불일치 문제

두 개의 CSP에서 각각 이미지를 빌드한다고 가정합니다 소스 코드는 완전히 동일하더라도, 빌드 결과 이미지의 다이제스트가 달라질 수 있는 원인이 있습니다

- **타임스탬프 차이**: 파일시스템 타임스탬프가 레이어에 포함되면, 빌드 시각이 다르면 해시가 달라집니다
- **빌드 환경 차이**: 빌더 버전, OS 패키지 버전이 미세하게 다를 경우 레이어 내용이 달라집니다
- **base 이미지 차이**: `FROM golang:1.22` 같은 태그를 쓰면, 두 빌드 사이에 base 이미지가 업데이트될 경우 결과 해시가 달라집니다

결과적으로 ECR의 이미지 다이제스트와 GAR의 이미지 다이제스트가 서로 다른 상태가 됩니다 배포 YAML에 `@sha256:...` 다이제스트를 고정해 두면 한쪽 CSP에서는 pull 실패가 발생하고, 고정하지 않으면 두 CSP에서 실제로 다른 이미지가 실행되는 상태가 됩니다

```text
ECR 이미지 다이제스트:  sha256:aaa...
GAR 이미지 다이제스트:  sha256:bbb...  ← 다른 해시 — 재현성 훼손
```

이 상태에서는 "동일한 코드"라는 보장이 레지스트리 레벨에서 성립하지 않습니다 디버깅 시 "AWS에서는 재현 안 됨, GCP에서만 발생"이라는 환경 의존 버그가 숨어있을 수 있습니다

### 중앙 레지스트리 1차 검증 후 멀티 리전 복제

이 문제를 해결하는 패턴은 **단일 빌드 → 중앙 레지스트리 → 복제** 입니다

![중앙 레지스트리 검증 후 멀티 CSP 복제 흐름|tall](/diagrams/goti-deepdive-image-supply-chain-1.svg)

위 다이어그램은 전체 공급망 흐름을 보여줍니다

빌드는 한 번만 수행됩니다 CI 파이프라인(GitHub Actions 등)이 이미지를 빌드해 중앙 레지스트리(Harbor)에 push합니다 Harbor에서는 취약점 스캔, 서명 검증 등 1차 검증을 수행합니다 검증을 통과한 이미지는 원본 그대로 각 CSP 네이티브 레지스트리(ECR, GAR)로 **복제**됩니다

복제된 이미지는 원본과 **동일한 다이제스트**를 가집니다 복제는 이미지 레이어를 그대로 옮기는 작업이므로, 새로운 빌드가 발생하지 않습니다 이미 해시로 고정된 내용이 그대로 전달됩니다

```text
Harbor:  sha256:ccc...  (1차 검증 완료)
ECR:     sha256:ccc...  (복제 — 동일한 다이제스트)
GAR:     sha256:ccc...  (복제 — 동일한 다이제스트)
```

EKS 클러스터는 ECR에서, GKE 클러스터는 GAR에서 pull합니다 두 클러스터 모두 같은 다이제스트의 이미지를 실행합니다 "Harbor 해시 == ECR 해시 == GAR 해시"가 구조적으로 보장됩니다

### 네이티브 레지스트리에서 pull하는 이점

중앙 레지스트리 단일화와 복제를 조합하는 이유는 공급망 보안만이 아닙니다 **cross-cloud pull의 지연과 egress 비용** 문제도 중요한 동기입니다

GCP에서 AWS에 있는 Harbor로 이미지를 pull하면, 이미지 레이어 데이터가 AWS → GCP 인터넷 구간을 통과합니다 컨테이너 이미지는 수십~수백 MB가 일반적이며, 신규 배포나 오토스케일링 시 여러 노드가 동시에 pull을 시도할 수 있습니다

| 시나리오 | pull 경로 | 지연 | egress 비용 |
|---|---|---|---|
| Harbor 단일 + cross-cloud pull | AWS Harbor → GCP GKE | 인터넷 구간 통과, 수 초 이상 | AWS → 외부 egress 발생 |
| 네이티브 레지스트리 복제 후 pull | GAR → GCP GKE | 같은 리전 내, 밀리초 단위 | 리전 내 트래픽 무료 또는 저렴 |

네이티브 레지스트리(ECR, GAR)는 같은 클라우드 내 같은 리전에서 운영되도록 설계되어 있습니다 동일 리전 pull은 전용 백플레인을 사용하므로 인터넷 구간을 통과하지 않습니다 첫 배포 지연이 줄고, 반복 pull도 안정적입니다

오토스케일링 이벤트처럼 단시간에 다수 노드가 이미지를 필요로 하는 상황에서 이 차이는 더 두드러집니다 cross-cloud pull이 병목이 되면 Pod가 ContainerCreating 상태에 오래 머무를 수 있습니다

### 레지스트리 복제의 동작 방식

이미지 복제는 원본 이미지의 모든 레이어와 매니페스트를 대상 레지스트리로 업로드하는 작업입니다 컨텐츠 어드레서블(content-addressable) 구조 덕분에 이미 존재하는 레이어는 재전송하지 않습니다

Harbor는 자체 복제 기능(Replication)을 내장하고 있습니다 Harbor에 복제 규칙(replication rule)을 설정하면, 새 이미지가 push될 때마다 지정한 대상 레지스트리로 자동 복제가 트리거됩니다

```text
[Harbor 복제 규칙]
트리거:   이미지 push 이벤트
소스:     harbor.go-ti.internal/goti/*
대상 1:   ECR (AWS ap-northeast-2)
대상 2:   GAR (GCP asia-northeast3)
필터:     태그 패턴 v*
```

복제 완료 후에는 각 레지스트리에서 동일한 다이제스트로 이미지를 확인할 수 있습니다 Renovate나 GitOps 파이프라인이 이 다이제스트를 읽어 배포 YAML을 갱신하면, 전체 공급망이 단방향 흐름으로 연결됩니다

## 📐 세부 동작과 옵션

### Harbor vs CSP 네이티브 레지스트리 단독 사용 비교

![CSP 독립 빌드(해시 불일치) vs 중앙 검증 후 복제(해시 일치) 비교|tall](/diagrams/goti-deepdive-image-supply-chain-2.svg)

위 다이어그램은 두 전략의 핵심 차이를 나란히 보여줍니다 왼쪽은 CSP별 독립 빌드로 인해 해시가 달라지는 경우이고, 오른쪽은 중앙에서 한 번 빌드·검증한 이미지를 동일 다이제스트로 복제하는 경우입니다

왼쪽 전략에서는 CI 파이프라인이 AWS용과 GCP용으로 각각 실행됩니다 소스는 같아도 앞서 설명한 타임스탬프·환경 차이로 인해 두 이미지의 다이제스트가 달라질 수 있습니다 이 경우 다이제스트 기반 배포가 불가능하고, 태그 기반 배포로 후퇴하게 됩니다 취약점 스캔도 두 곳에서 각각 수행해야 합니다

오른쪽 전략에서는 CI 파이프라인이 한 번만 실행됩니다 Harbor에서 취약점 스캔과 서명 검증이 단일 지점에서 이루어집니다 Harbor를 통과한 이미지만 ECR·GAR로 복제되므로, 검증을 우회한 이미지가 배포될 수 없습니다

| 항목 | CSP별 독립 빌드 | 중앙 Harbor + 복제 |
|---|---|---|
| 이미지 다이제스트 일치 | 보장 불가 | 구조적 보장 |
| 빌드 횟수 | CSP 수 × 이미지 수 | 이미지 수 |
| 취약점 스캔 위치 | CSP별 각각 | Harbor 단일 |
| cross-cloud pull | 발생 없음 | 발생 없음 (복제 후 native pull) |
| 빌드 비용 | 2배 이상 | 1배 |
| SSoT | 없음 | Harbor |

### 이미지 서명(Cosign)과 다이제스트 결합

다이제스트만으로는 "빌드된 이미지가 공식 CI 파이프라인에서 나온 것"을 증명하지 못합니다 누군가 임의로 같은 내용의 이미지를 Harbor에 push할 수 있습니다

이 문제는 **Cosign**을 이용한 이미지 서명으로 보완합니다 CI 파이프라인이 빌드 완료 후 이미지 다이제스트에 서명을 첨부합니다 Harbor나 배포 파이프라인이 서명을 검증해 서명되지 않은 이미지의 복제·배포를 차단합니다

```bash
# CI 파이프라인: 이미지 빌드 후 서명
cosign sign --key cosign.key harbor.go-ti.internal/goti/api@sha256:ccc...

# 배포 시 서명 검증
cosign verify --key cosign.pub harbor.go-ti.internal/goti/api@sha256:ccc...
```

다이제스트는 "이 이미지가 변조되지 않았음"을, 서명은 "이 이미지가 신뢰된 파이프라인에서 생성되었음"을 각각 보장합니다 둘을 결합하면 supply chain 공격의 주요 벡터를 차단할 수 있습니다

### 비용 실측 참고

ADR-0025에서 공개된 비용 수치를 참고하면, 이미지 복제(ECR/GAR egress) 비용은 월 약 $2로 측정되었습니다 이는 cross-cloud pull 비용과 비교했을 때 적은 수준입니다 Harbor가 AWS EKS 위에서 운영되므로 GCP에서 복제를 받을 때는 AWS → 외부 egress가 발생하지만, 복제는 배포 시마다 아닌 이미지 변경 시에만 발생하므로 트래픽 규모가 작습니다

## 🧩 go-ti에서는

go-ti는 AWS EKS와 GCP GKE에 동일한 Go MSA 6종을 배포합니다 ADR-0025에서 이미지 레지스트리 전략을 결정할 때, CSP별 독립 빌드(ECR + GAR 각각 빌드)는 이미지 해시 불일치로 재현성을 손상한다는 이유로 기각되었습니다 Harbor 단일 레지스트리에서 cross-cloud pull하는 안은 GCP → AWS 구간의 첫 배포 지연과 네트워크 비용 때문에 기각되었습니다

결정된 구조는 **Harbor(EKS 위)에서 1차 검증 → ECR·GAR로 복제 → 각 CSP 네이티브 pull**입니다 이 구조에서 "Harbor 해시 == ECR 해시 == GAR 해시" 가 설계상 보장됩니다 AWS EKS는 ECR에서, GCP GKE는 GAR에서 pull하므로 cross-cloud 지연도 발생하지 않습니다

다만 Harbor가 AWS EKS에 의존하므로, AWS가 전면 장애일 경우 새 이미지를 빌드·배포할 수 없다는 리스크가 ADR에 명시되어 있습니다 이미 복제된 이미지는 GAR에서 정상 pull 가능하므로, 운영 중인 클러스터는 새 배포 없이 계속 동작합니다

> 이 기술을 go-ti에 도입하기까지의 의사결정 과정(검토한 대안과 기각 이유)은
> [Multi-Cloud 상위 아키텍처 — Active-Passive + CF Worker 팀 코드 라우팅을 택한 이유](/logs/goti-multicloud-high-level-architecture-adr)에 정리했습니다

## 📚 핵심 정리

- 이미지 다이제스트는 매니페스트 전체의 SHA256 해시입니다 다이제스트가 같으면 레이어 내용도 반드시 같으므로, 다이제스트 기반 배포는 구조적으로 재현성을 보장합니다
- CSP별 독립 빌드는 타임스탬프·빌드 환경 차이로 인해 동일 소스에서도 다이제스트가 달라질 수 있습니다 재현성 보장이 필요하다면 빌드를 한 번만 수행해야 합니다
- 중앙 레지스트리(Harbor)에서 1차 검증한 이미지를 각 CSP 네이티브 레지스트리로 복제하면, 공급망 SSoT와 네이티브 pull 성능을 동시에 달성합니다
- 레지스트리 복제는 이미지를 재빌드하지 않고 레이어를 그대로 전송하므로, 복제 전후 다이제스트가 동일합니다 ECR·GAR 모두 Harbor와 같은 해시를 가집니다
- 이미지 서명(Cosign)을 다이제스트와 결합하면 "변조 없음"에 더해 "신뢰된 파이프라인에서 생성됨"까지 보장하는 완전한 supply chain 보안을 구성할 수 있습니다
