## 한 줄 요약

> 서킷 브레이커는 전기 차단기처럼 실패가 임계를 넘으면 연결을 끊고, 일정 시간 후 조심스럽게 탐색해 스스로 복구하는 3-상태 자동화 장치입니다

---

## 🤔 무엇을 푸는 기술인가

분산 시스템에서 서비스 A가 서비스 B를 호출할 때, B가 느려지거나 죽으면 어떤 일이 일어날까요

**장애 전파(Cascading Failure)**가 발생합니다 B에 묶인 A의 요청이 타임아웃까지 기다리고, 그 사이 새 요청이 계속 들어와 A의 스레드/커넥션 풀이 고갈됩니다 결국 A까지 응답 불능 상태가 됩니다 B 하나의 장애가 A, 그리고 A에 의존하는 C까지 연쇄 마비시키는 구조입니다

서킷 브레이커(Circuit Breaker)는 이 연쇄 마비를 막기 위한 패턴입니다 핵심 아이디어는 단순합니다

- 실패가 잦으면 **더 이상 시도하지 않는다** — 연결을 끊어 빠른 실패를 유도
- 일정 시간이 지나면 **조심스럽게 다시 시도한다** — 자동 복구 경로를 남김

마이클 니가드의 저서 *Release It!* (2007)에서 정형화된 이 패턴은 이후 Hystrix, Resilience4j, Istio 등 다양한 라이브러리·플랫폼에서 구현됩니다

## 🔧 동작 원리

### 3가지 상태

서킷 브레이커는 세 가지 상태를 오갑니다

| 상태 | 역할 | 요청 처리 방식 |
|---|---|---|
| **Closed** | 정상 운영 | 모든 요청을 원래 대상으로 통과시킴 |
| **Open** | 장애 격리 | 모든 요청을 즉시 fallback으로 돌림 (원래 대상에 접근 안 함) |
| **Half-Open** | 복구 탐색 | 첫 요청만 원래 대상으로 보내 결과를 확인 |

![서킷 브레이커 3-상태 머신 다이어그램|tall](/diagrams/goti-deepdive-circuit-breaker-1.svg)

위 상태 머신 다이어그램에서 세 상태가 전이하는 경로를 볼 수 있습니다 Closed에서 실패 임계를 넘으면 Open으로 전이합니다 Open에서 TTL이 만료되면 Half-Open으로 자동 전이하고, probe 결과에 따라 Closed(복귀) 또는 Open(재차단)으로 갈립니다

각 상태를 차례로 살펴봅니다

### Closed — 정상 통과, 실패를 센다

Closed는 서킷 브레이커의 기본 상태입니다 모든 요청이 원래 대상으로 흘러가고, 서킷 브레이커는 실패 여부를 조용히 기록합니다

기록하는 항목은 구현마다 다르지만 일반적으로 다음을 추적합니다

- 최근 N개 요청의 실패 비율 (슬라이딩 윈도우)
- 또는 최근 T초 내 실패 횟수
- 또는 연속 실패 횟수 (Consecutive Failures)

실패로 집계되는 조건도 구현에서 명시합니다 HTTP 5xx 응답, connection refused, 타임아웃 등이 대표적입니다 중요한 것은 **connection timeout과 5xx를 구분**할 수 있어야 한다는 점입니다 connection timeout은 서버가 아예 응답하지 않는 상황이므로, 타임아웃 자체를 실패로 집계하지 않으면 open 전이 조건을 놓칩니다

### Open — 연결을 끊고 TTL을 기다린다

실패 카운터가 임계치(threshold)를 넘으면 서킷이 **Open 상태로 전이**됩니다

Open 상태에서 서킷 브레이커는 원래 대상에 **아예 연결을 시도하지 않습니다** 요청이 들어오면 즉시 오류를 반환하거나 fallback 경로로 보냅니다 이것이 장애 격리의 핵심입니다

타임아웃을 기다리지 않기 때문에 응답 지연이 수 밀리초로 줄어듭니다 연결 시도 자체가 없으니 스레드와 커넥션 풀도 소비되지 않습니다 장애가 있는 대상에 계속 부하를 주지 않아 대상 서비스가 회복할 여유를 줍니다

Open 상태는 영구적이지 않습니다 **TTL(Time To Live) 타이머**가 시작됩니다 타이머가 만료되면 Half-Open 상태로 자동 전이합니다

### Half-Open — probe로 복구를 확인

Half-Open은 "조심스럽게 열어본다"는 탐색 상태입니다

TTL이 만료된 후 들어오는 **첫 번째 요청(probe)**만 원래 대상으로 보냅니다 그 결과를 보고 전이를 결정합니다

- **probe 성공(2xx 등 정상 응답)**: Closed로 전이합니다 이후 요청은 다시 원래 대상으로 통과합니다
- **probe 실패(5xx 또는 타임아웃)**: 다시 Open으로 전이합니다 TTL이 재설정되어 다음 탐색까지 기다립니다

probe가 진행되는 동안 나머지 요청은 여전히 fallback으로 보냅니다 probe 하나의 결과가 나올 때까지 전체 트래픽을 전환하지 않기 때문에, 복구 중인 서버에 갑자기 부하가 몰리지 않습니다

### Open → Half-Open의 자동 전이가 중요한 이유

수동 개입 없이 자동으로 복구를 탐색하는 것이 서킷 브레이커 패턴의 핵심 가치입니다

단순 타임아웃에 비해 서킷 브레이커가 나은 이유가 이 자동화에 있습니다 타임아웃만 있으면 매 요청마다 실패할 때까지 기다려야 합니다 서킷 브레이커는 임계를 넘은 뒤에는 **기다리는 행위 자체를 멈추고**, 일정 시간 후 딱 한 번만 확인합니다

복구 사이클은 다음 순서로 반복됩니다

1. open 전이 → TTL 60초 대기 → 첫 probe 허용
2. probe 성공 → closed 복귀 (자동)
3. probe 실패 → open 재진입 → TTL 재설정 → 1번으로 돌아감

서버가 살아날 때까지 이 사이클이 반복됩니다 알람을 받고 담당자가 수동으로 재활성화할 필요 없이, 서버가 정상화된 순간 자연스럽게 트래픽이 복귀합니다

### 요청 흐름 — Closed와 Open의 차이

![서킷 브레이커 요청 흐름 — Closed/Open 상태별 동작 비교|tall](/diagrams/goti-deepdive-circuit-breaker-2.svg)

위 흐름도는 Closed와 Open 두 상태에서 같은 요청이 어떤 경로를 밟는지 나란히 비교합니다

왼쪽 Closed 상태에서는 클라이언트 요청이 Worker를 거쳐 배정된 AWS 오리진으로 직접 흘러갑니다 타임아웃 1.5초 안에 정상 응답이 오면 응답 헤더에 `circuit: closed`가 기록됩니다 실패 카운터는 조용히 누적됩니다

오른쪽 Open 상태에서는 Worker가 circuit 상태를 먼저 확인합니다 AWS에 대한 circuit이 open임을 확인하면 AWS로 연결 시도 없이 즉시 GCP fallback으로 요청을 보냅니다 AWS 서버가 아예 죽어 있어도 클라이언트는 타임아웃 없이 GCP에서 빠른 응답을 받습니다 응답 헤더에 `circuit: open/aws`와 `failover: true`가 남습니다

이 두 경로의 차이가 장애 격리의 실체입니다 Open 상태에서 클라이언트가 체감하는 지연은 fallback 오리진의 응답 시간뿐입니다 이미 죽은 서버를 향한 연결 시도와 그 타임아웃 대기가 사라졌습니다

## 📐 세부 동작과 옵션

### 실패 감지 조건 — connection timeout vs 5xx

서킷 브레이커가 "실패"로 집계하는 조건은 구현에서 명시적으로 지정합니다 크게 두 유형이 있습니다

| 실패 유형 | 원인 | 집계 여부 |
|---|---|---|
| **5xx 응답** | 서버가 응답하지만 오류 반환 | 집계 (기본) |
| **Connection Timeout** | 서버가 TCP 연결에 응답 안 함 | 집계 필요 — 명시하지 않으면 누락 |
| **Request Timeout** | 서버가 연결은 하지만 응답 지연 | 집계 필요 |
| **4xx 응답** | 클라이언트 오류 (잘못된 요청) | 집계 안 함 (서버 문제가 아님) |

connection timeout을 실패로 집계하지 않으면, 서버가 완전히 다운됐을 때 서킷이 열리지 않습니다 매 요청마다 연결 대기 전체를 소비하게 됩니다

### TTL과 probe 횟수

Half-Open에서 probe를 몇 번 허용하느냐도 구현마다 다릅니다

- **probe 1회**: 가장 단순합니다 하나가 성공하면 즉시 Closed로 전이합니다 복구가 불완전할 경우 Closed 전이 직후 다시 Open으로 돌아올 수 있습니다
- **probe N회**: 여러 번 연속 성공을 요구합니다 복구 안정성을 높이지만, 전이까지 시간이 더 걸립니다
- **비율 방식**: N개 요청 중 성공 비율이 임계를 넘으면 Closed로 전이합니다

TTL 설정은 장애 유형에 따라 달리 합니다 짧은 TTL(10~30초)은 빠른 복구 탐색에 유리하지만 probe 실패 시 잦은 재차단이 발생합니다 긴 TTL(60초+)은 서버가 충분히 회복할 시간을 주지만 복귀가 느립니다

### Circuit Breaker의 저장 계층

Circuit 상태는 어딘가에 저장되어야 합니다 저장 계층의 선택이 범위(scope)를 결정합니다

| 저장 계층 | 범위 | 특징 |
|---|---|---|
| 인스턴스 메모리 | 인스턴스 단독 | 가장 단순, 인스턴스별 독립 상태 |
| 외부 캐시 (Redis 등) | 서비스 전체 공유 | 모든 인스턴스가 동일 circuit 상태 |
| CDN 캐시 (PoP 단위) | PoP 단위 독립 | 엣지에서 구현 시 사용, 지역별 독립 |

외부 캐시를 쓰면 인스턴스가 여러 개여도 circuit 상태가 통일됩니다 CDN PoP 캐시를 쓰면 PoP마다 독립적인 상태를 가지며, 전 세계 각 지점이 독자적으로 장애를 탐지하고 복구합니다

## 🧩 go-ti에서는

go-ti는 Cloudflare Worker에서 팀 코드 기반 AWS/GCP 라우팅을 운영하고 있었습니다 AWS cost freeze로 ASG가 0으로 내려간 상황에서, 기존 5xx 기반 failover는 서버가 아예 응답하지 않는 connection timeout을 잡지 못해 매 요청마다 약 10초 지연이 발생했습니다

ADR-0016에서 채택한 대안 C는 `PRIMARY_TIMEOUT_MS = 1500` + AbortController로 connection timeout을 명시적으로 1.5초로 제한하고, 이 타임아웃도 실패로 집계하는 서킷 브레이커를 Worker에 구현했습니다 Circuit 상태는 Cloudflare Cache API에 `max-age=60`으로 저장해 PoP 단위 독립 상태를 유지합니다

`isCircuitOpen(cloud)` 호출로 먼저 circuit 상태를 확인합니다 open이면 AWS 접근 없이 GCP를 primary로 교체합니다 primary 실패 시 `openCircuit(cloud)`를 호출합니다 60초 TTL 만료 후 첫 요청이 자동으로 probe 역할을 하며, probe 성공 시 circuit이 닫혀 정상 5:5 분배로 복귀합니다 응답 헤더 `x-goti-route-circuit`으로 현재 상태를 관측합니다

> 이 기술을 go-ti에 도입하기까지의 의사결정 과정(검토한 대안과 기각 이유)은
> [Multi-cloud 장애 대응 — Circuit Breaker + HPA ADR](/logs/goti-multicloud-circuit-breaker-hpa-adr)에 정리했습니다

## 📚 핵심 정리

- **서킷 브레이커는 실패 격리와 자동 복구 두 가지를 모두 해결합니다** Open에서 연결 자체를 차단해 장애 전파를 막고, Half-Open을 통해 수동 개입 없이 복귀 경로를 확보합니다
- **connection timeout도 명시적으로 실패로 집계해야 합니다** 서버가 완전히 다운된 상황에서 5xx만 감지하면 서킷이 열리지 않고 매 요청이 타임아웃을 소비합니다
- **Half-Open의 probe는 단 한 번입니다** 나머지 요청은 fallback을 유지하므로 회복 중인 서버에 부하가 몰리지 않습니다
- **TTL 만료 후 자동 전이가 핵심 가치입니다** 알람·수동 복구 없이 서버가 정상화되는 순간 트래픽이 복귀합니다
- **Circuit 저장 계층이 격리 범위를 결정합니다** 인스턴스 메모리는 독립 상태, 공유 캐시는 전체 통일, PoP 캐시는 지역별 독립 상태를 만듭니다
