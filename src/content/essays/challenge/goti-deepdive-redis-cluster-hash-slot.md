## 한 줄 요약

> Redis Cluster는 CRC16 해시로 키를 16384 슬롯 중 하나에 배치하고, hash tag `{...}`를 사용하면 여러 키를 같은 슬롯에 강제 배치해 멀티키 연산과 Lua 원자성이 가능해집니다

## 🤔 무엇을 푸는 기술인가

### 단일 Redis 인스턴스의 한계

단일 Redis 인스턴스는 메모리 상한과 처리량 한계가 있습니다 한 서버에 데이터를 무한히 늘릴 수 없고, 초당 명령 처리 건수도 단일 스레드 구조의 물리적 한계가 있습니다 이를 수평 확장하는 방법이 **Redis Cluster**입니다

Redis Cluster는 데이터를 여러 노드에 자동으로 분산합니다 클라이언트 쪽에서 샤딩 로직을 직접 구현하지 않아도, 클러스터가 어떤 키가 어떤 노드에 있는지 관리합니다

### 분산 환경에서 생기는 새로운 문제

데이터를 여러 노드에 나누면 새로운 제약이 생깁니다 서로 다른 노드에 있는 키를 하나의 연산으로 묶을 수 없습니다 `MGET key1 key2 key3`에서 `key1`이 Node A에, `key2`가 Node B에 있다면, 단일 요청으로 처리하기 어렵습니다

이 제약은 Lua 스크립트에서도 동일하게 적용됩니다 `EVALSHA`로 실행하는 스크립트의 `KEYS[]`에 넘기는 키들이 서로 다른 슬롯에 있으면 오류가 발생합니다

**hash tag**는 이 문제를 해결하는 메커니즘입니다 키에 `{...}` 패턴을 포함시키면, Redis Cluster는 키 전체가 아닌 중괄호 안쪽 문자열만 해싱해 슬롯을 결정합니다 같은 hash tag를 쓰는 키들은 반드시 같은 슬롯에 배치됩니다

## 🔧 동작 원리

### 16384 슬롯과 CRC16 해싱

Redis Cluster는 키 공간을 **16384개의 슬롯**으로 나눕니다 슬롯 번호는 0부터 16383까지입니다 각 노드는 이 슬롯의 일정 범위를 담당합니다 예를 들어 3노드 클러스터라면 Node A가 0~5460, Node B가 5461~10922, Node C가 10923~16383을 맡는 식입니다

키가 어느 슬롯에 속하는지는 **CRC16 해시 함수**로 계산합니다

```text
slot = CRC16(key) % 16384
```

`CRC16`은 키 문자열을 입력 받아 0~65535 범위의 정수를 반환합니다 이 값을 16384로 나눈 나머지가 슬롯 번호입니다 키가 어떤 슬롯에 속하는지는 순수하게 키 이름에 의해 결정됩니다 데이터 내용이나 타입과 무관합니다

16384라는 숫자는 임의적인 선택이 아닙니다 Redis 설계 문서에 따르면 1000 노드 이상의 클러스터를 상정할 때 노드당 슬롯 비율이 적절하고, 클러스터 상태 정보를 전파하는 gossip 메시지 크기를 16384비트(2KB) 비트맵으로 처리할 수 있어 네트워크 오버헤드가 합리적인 수치입니다

![Redis Cluster 키 분산 구조도 — CRC16과 16384 해시 슬롯|tall](/diagrams/goti-deepdive-redis-cluster-hash-slot-1.svg)

위 구조도에서 키가 슬롯으로, 슬롯이 노드로 이어지는 두 단계 매핑을 볼 수 있습니다

첫 번째 단계에서 `seat:101:A1`, `inv:101:S1`, `hold:101:A1` 같은 키 문자열이 CRC16 함수를 통과합니다 각 키에 대해 독립적으로 해시 값이 계산되고, `% 16384` 연산으로 슬롯 번호가 나옵니다 왼쪽의 game_id 101 관련 키들은 같은 슬롯(4092)으로 향하고, 다른 game_id나 다른 엔티티의 키는 별도 슬롯(9105, 7331)으로 분산됩니다

두 번째 단계에서 슬롯 번호가 노드로 매핑됩니다 슬롯 4092는 0~5460 범위를 담당하는 Node A에 배치되고, 슬롯 9105와 7331은 5461~10922 범위의 Node B에 배치됩니다 클라이언트는 키 이름에서 슬롯을 계산하고, 슬롯-노드 매핑 테이블을 조회해 어느 노드로 요청을 보낼지 결정합니다

### 클라이언트 리다이렉트: MOVED와 ASK

클라이언트가 잘못된 노드에 요청을 보내면, 해당 노드는 `MOVED` 응답으로 올바른 노드 주소를 알려줍니다

```text
-MOVED 4092 10.0.0.1:6379
```

이 응답을 받은 클라이언트는 슬롯-노드 매핑 테이블을 갱신하고 올바른 노드에 재요청합니다 `redis.UniversalClient`(Go) 같은 Cluster-aware 클라이언트는 이 과정을 자동으로 처리합니다 애플리케이션 코드는 단일 Redis와 거의 동일하게 작성할 수 있습니다

슬롯 마이그레이션 중에는 `ASK` 응답이 사용됩니다 `MOVED`는 영구적인 슬롯 재배치를 의미하고, `ASK`는 마이그레이션 진행 중인 임시 상태임을 의미합니다

### hash tag: 슬롯 강제 배치

hash tag는 키 이름에 `{...}` 패턴을 포함시켜 슬롯 계산에 사용할 문자열 범위를 지정하는 기능입니다

파싱 규칙은 단순합니다

- 키에 `{`와 `}`가 모두 있고, 그 사이에 문자가 하나 이상 있으면 → 해당 구간 문자열만 CRC16 입력으로 사용
- 그 외의 경우(중괄호 없음, 빈 중괄호, 첫 `{` 이후 `}` 없음) → 키 전체를 CRC16 입력으로 사용

```text
seat:{101}:A1   →  CRC16("101") % 16384
inv:{101}:S1    →  CRC16("101") % 16384
hold:{101}:A1   →  CRC16("101") % 16384
order:7823      →  CRC16("order:7823") % 16384
```

`seat:{101}:A1`과 `inv:{101}:S1`은 중괄호 안이 모두 `101`이므로 정확히 같은 슬롯에 배치됩니다 `order:7823`은 hash tag가 없으므로 키 전체가 해싱 대상이 되어 다른 슬롯으로 갑니다

![hash tag co-location 구조도 — {game_id}로 같은 슬롯에 모으기|tall](/diagrams/goti-deepdive-redis-cluster-hash-slot-2.svg)

Before/After 비교 구조도를 보면 hash tag의 효과가 명확합니다

BEFORE(왼쪽)에서는 hash tag가 없습니다 `seat:101:A1`, `inv:101:S1`, `hold:101:A1`은 키 전체가 CRC16 대상이 됩니다 세 키가 각각 다른 슬롯 번호(4923, 12047, 8301)를 가져 서로 다른 노드(A, C, B)에 흩어집니다 이 상태에서 `MGET seat:101:A1 inv:101:S1 hold:101:A1`을 실행하면 Redis Cluster는 `CROSSSLOT Keys in request don't hash to the same slot` 오류를 반환합니다

AFTER(오른쪽)에서는 game_id에 hash tag를 붙입니다 `seat:{101}:A1`, `inv:{101}:S1`, `hold:{101}:A1` 모두 중괄호 안이 `101`이므로 동일한 슬롯 4092로 계산됩니다 세 키가 모두 Node A에 공존합니다 이제 `MGET seat:{101}:A1 inv:{101}:S1 hold:{101}:A1`이 정상 동작합니다 단일 노드 왕복 한 번으로 세 키를 한꺼번에 읽습니다

Lua EVALSHA와의 관계도 중요합니다 다이어그램 하단 안내처럼, Lua 스크립트에 넘기는 `KEYS[]`는 모두 같은 슬롯이어야 합니다 hash tag로 co-location이 보장된 키끼리만 하나의 Lua 스크립트로 묶을 수 있습니다 이 제약은 Redis Cluster 모드에서만 적용됩니다 단일 인스턴스나 Sentinel 모드에서는 슬롯 개념이 없으므로 해당 오류가 발생하지 않습니다

### slot rebalancing

노드를 추가하거나 제거할 때 슬롯 범위가 재배분됩니다 이를 **slot rebalancing** 또는 **reshard**라고 합니다

슬롯 이동은 노드 간 키를 하나씩 옮기는 과정입니다 Redis Cluster 프로토콜은 이 마이그레이션 중에도 서비스를 중단하지 않도록 설계되었습니다 마이그레이션 중인 슬롯에 대한 요청은 `ASK` 응답으로 클라이언트를 임시 대상 노드로 안내합니다 마이그레이션이 완료되면 `MOVED` 응답으로 영구 변경을 공지합니다

hash tag를 처음부터 사용하면 rebalancing 시 이점이 있습니다 같은 hash tag를 가진 키들은 항상 같은 슬롯에 있으므로, 슬롯 마이그레이션 한 번으로 관련 키 묶음 전체가 이동합니다 hash tag 없이 설계했다가 나중에 추가하면, 기존 키들의 슬롯 배치가 바뀌어 클라이언트 코드와 Lua 스크립트를 모두 재작성해야 합니다

## 📐 세부 동작과 옵션

### 멀티키 연산 제약 정리

Redis Cluster에서 여러 키를 다루는 명령의 동작 방식을 정리합니다

| 명령 | 같은 슬롯 | 다른 슬롯 |
|---|---|---|
| `MGET key1 key2` | 정상 | CROSSSLOT 오류 |
| `MSET key1 v1 key2 v2` | 정상 | CROSSSLOT 오류 |
| `EVALSHA sha 2 key1 key2 ...` | 정상 | CROSSSLOT 오류 |
| `SUNIONSTORE dest k1 k2` | 정상 | CROSSSLOT 오류 |
| `RENAME key1 key2` | 정상 | CROSSSLOT 오류 |
| `pipeline.GET(key1); pipeline.GET(key2)` | 정상 | 노드별 분리 실행 (클라이언트 처리) |

파이프라인은 예외입니다 Go의 `redis.UniversalClient`처럼 Cluster-aware 파이프라인 구현은 키를 슬롯별로 분류해 각 노드에 따로 전송합니다 개별 GET·SET은 오류 없이 처리됩니다 다만 여러 노드를 거치므로 원자성은 없습니다

### hash tag 설계 시 주의점

**키 핫스팟 문제**: 같은 hash tag를 너무 많은 키에 적용하면 특정 슬롯에 데이터가 몰립니다 `{game_id}`를 쓰면 하나의 game_id에 관련된 모든 키가 한 슬롯에 집중됩니다 특정 경기가 인기가 높아 트래픽이 집중되면, 그 경기의 슬롯을 담당하는 노드에 부하가 쏠립니다 이를 **슬롯 핫스팟**이라 합니다

**최소 범위 원칙**: hash tag의 범위는 co-location이 필요한 키끼리만 묶는 수준으로 제한하는 것이 좋습니다 `{}`로 모든 키를 한 슬롯에 넣거나, 도메인과 무관하게 같은 tag를 공유하면 클러스터의 분산 효과가 사라집니다

**단일 인스턴스 호환**: hash tag는 클러스터 모드에서만 효과가 있습니다 단일 인스턴스나 Sentinel 구성에서는 `{...}` 패턴이 포함된 키 이름 그대로 저장됩니다 클러스터 전환을 고려한다면 처음부터 hash tag를 포함한 키 명명 규칙을 사용해두는 것이 이점입니다

## 🧩 go-ti에서는

go-ti 티켓팅 플랫폼은 초기에 GCP Memorystore BASIC(단일 인스턴스)으로 운영했습니다 Redis Cluster가 아니었기 때문에 슬롯 제약이 없었지만, 키 명명 규칙에 hash tag를 처음부터 적용했습니다 `seat:{game_id}:{section_id}`, `inv:{game_id}:{section_id}`, `hold:{game_id}:{seat_id}` 형태로 game_id를 hash tag로 감쌌습니다

이유는 장기 운영 전환 시 재작성 비용을 줄이기 위해서입니다 ADR에 명시된 인프라 로드맵의 Target 단계에서 Memorystore Cluster(3~6 shard)로 전환하는 경우, `redis.UniversalClient` 추상만 유지하면 코드 재작성 없이 전환이 가능하도록 설계했습니다 hash tag가 없는 키들을 나중에 일괄 마이그레이션하는 것은 Lua 스크립트 전체를 다시 작성하는 부담이 따르기 때문입니다

> 이 기술을 go-ti에 도입하기까지의 의사결정 과정(검토한 대안과 기각 이유)은
> [Redis를 SoT로 채택한 이유 — SDD-0005 전면 승격 (ADR)](/logs/goti-redis-sot-adoption-adr)에 정리했습니다

## 📚 핵심 정리

- Redis Cluster는 16384 슬롯으로 키 공간을 나누고, `CRC16(key) % 16384`로 각 키를 슬롯에 배치합니다 슬롯은 노드에 범위로 할당됩니다
- 멀티키 연산(`MGET`, `MSET`, `EVALSHA` 등)은 관련된 키가 모두 같은 슬롯에 있어야 합니다 다른 슬롯의 키를 묶으면 `CROSSSLOT` 오류가 발생합니다
- hash tag `{...}`는 키의 슬롯을 결정할 때 중괄호 안쪽 문자열만 CRC16에 입력합니다 같은 hash tag를 쓰는 키는 반드시 같은 슬롯에 배치됩니다
- hash tag는 처음부터 적용하는 것이 중요합니다 나중에 추가하면 기존 키의 슬롯이 바뀌어 Lua 스크립트와 클라이언트 코드를 전면 수정해야 합니다
- 슬롯 핫스팟에 주의합니다 hash tag 범위가 넓을수록 특정 노드에 트래픽이 집중될 수 있으며, 클러스터의 수평 확장 효과가 줄어듭니다
