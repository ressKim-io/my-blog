## 한 줄 요약

> SETNX는 "키가 없을 때만 쓴다"는 원자 연산 하나로 분산 환경의 경합을 메모리 레벨에서 끊어냅니다 NX·EX 옵션으로 set-and-expire를 원자화하고, value에 소유자를 담아 무단 해제를 막습니다

---

## 🤔 무엇을 푸는 기술인가

### 분산 환경의 중복 실행 문제

단일 프로세스에서 공유 자원을 보호할 때는 OS 뮤텍스나 synchronized 블록으로 충분합니다
하지만 API 서버가 여러 인스턴스로 수평 확장되면 프로세스 경계를 넘는 잠금이 필요합니다

좌석 예매 시스템에서 가장 흔히 발생하는 문제는 다음 두 가지입니다

- **중복 점유**: 사용자 A와 B가 동시에 같은 좌석을 선택하고 둘 다 성공 응답을 받음
- **결제 중 재점유**: 사용자 A가 결제 중인 동안 다른 요청이 같은 좌석을 hold하고 결제 완료

RDB에서는 `SELECT FOR UPDATE`나 `UNIQUE` 제약으로 방어하지만, 고트래픽 환경에서 DB 행 잠금은 connection 풀을 빠르게 소진시킵니다

Redis SETNX는 이 문제를 **메모리 레벨 원자 연산**으로 해소합니다
"키가 없을 때만 값을 쓴다"는 단 하나의 의미론이 경쟁 조건을 구조적으로 차단합니다

### SETNX 명령의 핵심 의미론

```bash
SETNX key value
# 키가 존재하지 않으면 → 값을 쓰고 1 반환
# 키가 이미 존재하면  → 아무것도 하지 않고 0 반환
```

반환값 1은 "내가 최초로 이 키를 점유했다", 0은 "이미 누군가 점유 중이다"를 의미합니다
**이 분기가 경쟁 조건 없이 일어나는 것**이 핵심입니다 Redis는 단일 스레드로 모든 명령을 직렬 처리하기 때문에, 두 클라이언트가 동시에 같은 키에 SETNX를 보내도 하나는 반드시 먼저 실행되고 다른 하나는 그 다음에 실행됩니다

## 🔧 동작 원리

### SETNX의 원자성 — 단일 스레드가 보장

Redis 명령 처리기는 [단 하나의 스레드로 모든 명령을 처리](/essays/goti-deepdive-redis-single-thread-lua)합니다
명령 실행 도중에는 어떤 명령도 끼어들 수 없습니다

SETNX의 내부 의사 코드는 다음과 같이 이해할 수 있습니다

```text
function SETNX(key, value):
    if key가 keyspace에 없다면:
        keyspace[key] = value
        return 1
    else:
        return 0
```

이 "확인 후 쓰기" 과정이 **단 하나의 명령으로 원자적으로 실행**됩니다
두 클라이언트가 동시에 SETNX를 보내도, Redis 내부에서 이 두 명령은 줄을 서서 하나씩 처리됩니다
첫 번째 명령이 키를 쓰면, 두 번째 명령이 실행될 시점에는 이미 키가 존재하므로 0을 반환합니다

### 동시 요청 시퀀스

![두 사용자가 같은 좌석에 SETNX를 동시에 요청할 때 하나만 락을 획득하는 시퀀스|tall](/diagrams/goti-deepdive-redis-setnx-lock-1.svg)

위 시퀀스를 단계별로 따라가겠습니다

**T1~T2** — 사용자 A와 B가 거의 동시에 같은 좌석(3A)에 대한 hold 요청을 API 서버로 보냅니다 두 요청은 서로 다른 API 서버 인스턴스나 같은 인스턴스의 다른 고루틴에서 처리될 수 있습니다 어느 쪽이든 두 요청이 Redis에 SETNX를 보냅니다

**T3** — Redis가 두 명령 중 먼저 도착한 A의 명령을 단일 스레드에서 처리합니다 `seat:held:1:3A` 키가 존재하지 않으므로 `userA_id`를 쓰고 **1**을 반환합니다 이 시점에 키가 Redis keyspace에 등록됩니다

**T4** — A는 반환값 1을 받아 lock 획득을 확인하고 결제 흐름으로 진입합니다

**T5** — B의 SETNX가 이어서 실행됩니다 이미 `seat:held:1:3A` 키가 존재하므로 Redis는 쓰기 없이 **nil**을 반환합니다 B가 아무리 빠르게 요청을 보냈어도, Redis의 직렬 처리 보장 덕분에 A와 B 중 하나는 반드시 실패합니다

**T6** — B는 nil 응답을 받아 좌석이 이미 점유 중임을 알고 다른 좌석을 선택하도록 안내받습니다

이 시퀀스에서 핵심은 **"중복 점유 경쟁을 Redis 레벨에서 차단한다"**는 점입니다 API 서버나 DB에서 복잡한 충돌 감지 로직을 구현하지 않아도 됩니다

### NX·EX 옵션 — set-and-expire 원자화

초기 SETNX 명령에는 한 가지 치명적 결함이 있었습니다
락을 획득한 클라이언트가 crash하거나 응답하지 않으면, 키가 영원히 남아 **데드락**이 발생합니다

이를 해결하기 위해 초기 구현들은 SETNX 후 EXPIRE를 별도로 호출했습니다

```bash
SETNX seat:held:1:3A userA_id   # 1단계: 락 획득 시도
EXPIRE seat:held:1:3A 600       # 2단계: TTL 설정
```

그러나 이 두 명령 사이에 서버가 crash하면 EXPIRE가 실행되지 않아 키가 영구 잔존합니다
**두 명령이 원자적으로 실행되지 않기 때문**에 생기는 문제입니다

Redis 2.6.12부터 `SET` 명령의 `NX`·`EX` 옵션이 이를 완전히 해결합니다

```bash
SET seat:held:1:3A userA_id NX EX 600
# NX: 키가 없을 때만 쓴다 (SETNX 의미론)
# EX: 성공 시 TTL을 원자적으로 함께 설정
# 성공: OK 반환
# 실패(키 존재): nil 반환
```

"키 존재 확인 → 값 쓰기 → TTL 설정"의 세 동작이 **단 하나의 원자 명령**으로 실행됩니다
서버 crash, 네트워크 단절 등 어떤 장애가 발생해도 partial 상태가 없습니다
명령이 실행되면 TTL이 붙은 키가 존재하거나, 아무것도 없거나 둘 중 하나입니다

### value에 소유자를 담는 이유

락 키의 value는 단순한 더미 값이 아닙니다 **소유자 식별자**를 담아야 합니다

```bash
# 잘못된 패턴 — value가 고정 문자열
SET seat:held:1:3A "1" NX EX 600

# 올바른 패턴 — value가 소유자 식별자
SET seat:held:1:3A "userA_id_or_request_uuid" NX EX 600
```

소유자를 담지 않으면 **무단 해제** 문제가 생깁니다

시나리오를 따라가겠습니다

1. 사용자 A가 락 획득, TTL 600초
2. 어떤 이유로 처리가 길어지고 TTL이 만료됨
3. 사용자 C가 같은 좌석 락 획득 (C의 lock)
4. A의 처리가 뒤늦게 완료되어 lock 해제 코드를 실행
5. value 확인 없이 DEL을 실행하면 C의 lock을 삭제해버림
6. 이제 좌석이 열려서 또 다른 경쟁 발생

소유자 식별자를 사용하면 해제 전에 "내가 가진 락인가"를 확인할 수 있습니다

```lua
-- 안전한 lock 해제 Lua 스크립트
-- value가 내 소유자 ID와 일치할 때만 삭제
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

이 확인-삭제 과정도 Lua 스크립트로 원자화합니다
GET과 DEL 사이에 다른 명령이 끼어들면 확인 결과가 의미 없어지기 때문입니다

### TTL 만료 시 발생하는 race

TTL은 데드락을 방지하지만 새로운 엣지케이스를 만듭니다
**TTL이 너무 짧으면 처리 중인 락이 만료**될 수 있습니다

```text
T=0s    : A가 락 획득 (TTL 600s)
T=600s  : TTL 만료 — 키 자동 삭제
T=601s  : B가 같은 좌석 락 획득 (새 락)
T=602s  : A의 결제 처리 완료 — 락 해제 시도
          GET 반환: "userB_id"  ← A의 value와 다름
          Lua 스크립트가 삭제 거부 (소유자 불일치)
```

소유자 확인 덕분에 A가 B의 락을 실수로 삭제하지 않습니다
그러나 TTL 만료 구간에 B가 점유를 시작했으므로 **A의 결제는 실패 처리**되어야 합니다
애플리케이션 레벨에서 "내 락이 이미 만료되었다"는 상황을 감지해 보상 처리를 수행해야 합니다

이 때문에 TTL은 **"정상 처리가 끝날 충분한 시간 + 여유분"**으로 설정합니다
go-ti ADR에서는 hold에 600초, reservation(결제 lock)에 300초를 사용했습니다

## 📐 세부 동작과 옵션

### SET NX EX 옵션 조합 정리

| 옵션 | 의미 | 사용처 |
|---|---|---|
| `NX` | 키가 없을 때만 쓰기 | 락 획득 경쟁 |
| `XX` | 키가 있을 때만 쓰기 | 락 갱신(TTL 연장) |
| `EX <초>` | 초 단위 TTL | 일반 락 |
| `PX <밀리초>` | 밀리초 단위 TTL | 짧은 임계 구간 |
| `GET` | 기존 값 함께 반환 | 소유자 확인 + 갱신 원자화 (Redis 6.2+) |

`NX` + `EX`가 분산 락의 표준 조합입니다 `XX` + `EX`는 이미 획득한 락의 TTL을 원자적으로 연장할 때 사용합니다

```bash
# 락 TTL 연장 (소유자일 때만)
SET seat:held:1:3A "userA_id" XX EX 600
# XX: 키가 있어야 갱신 (없으면 nil — 만료 후 갱신 시도 방어)
# EX: TTL 재설정
```

### 단일 Redis vs Redlock

단일 Redis 인스턴스 SETNX 락은 Redis 노드 장애 시 **모든 락 정보가 소실**됩니다
장애 발생 직전에 락을 획득한 클라이언트는 해제 없이 락 정보가 사라지고, 다른 클라이언트가 즉시 락을 획득할 수 있습니다

Redlock 알고리즘(Redis 공식 제안)은 독립 Redis 노드 과반수에 동시에 락을 획득해 이 문제를 완화합니다
그러나 구현 복잡도가 높아 단순 use case에는 과잉입니다

| 방식 | 장애 내성 | 구현 복잡도 | 적합 상황 |
|---|---|---|---|
| 단일 SETNX | 낮음 (노드 장애 = 락 소실) | 낮음 | 비동시 클리티컬이 아닌 good-enough 락 |
| Redis Sentinel/Cluster | 중간 (failover 간 gap) | 중간 | 가용성 SLA가 있는 서비스 |
| Redlock | 높음 | 높음 | 금융·재고 등 엄격한 mutual exclusion |

go-ti 티켓팅 context에서는 Redis 장애 자체를 전체 장애로 분류했기 때문에 단일 인스턴스 SETNX 패턴을 사용했습니다

## 🧩 go-ti에서는

go-ti 티켓팅 플랫폼에서 3000 VU 부하 테스트 결과 RDS 행 잠금 경합으로 ticket_success 비율이 13~16%까지 떨어졌습니다
ADR 0014는 이 병목을 해소하기 위해 좌석 hold와 reservation 경로를 Redis SETNX 락으로 전환하는 결정을 담고 있습니다

hold 경로에는 `SETNX seat:held:{gameID}:{seatID} userID EX=600`, reservation 경로에는 `SETNX reserve:{gameID}:{seatID} userID EX=300` 패턴을 적용했습니다
두 패턴 모두 value에 userID를 담아 소유자 식별을 보장했고, Lua 스크립트로 확인-삭제를 원자화했습니다
RDS `UNIQUE` 제약 race와 중복 좌석 점유 문제가 Redis 메모리 레벨에서 차단되어 DB lock 경합이 크게 감소했습니다

> 이 기술을 go-ti에 도입하기까지의 의사결정 과정(검토한 대안과 기각 이유)은
> [Redis-first 전환 — 부하 테스트 후 좌석 경합 해소 결정 (ADR)](/logs/goti-redis-first-ticketing-adr)에 정리했습니다

## 📚 핵심 정리

- SETNX는 "키가 없을 때만 쓴다"는 원자 연산입니다 Redis의 단일 스레드 직렬 처리가 이 원자성을 보장해, 동시 요청 중 하나만 반드시 성공합니다
- `SET key value NX EX <초>` 형식이 현대적 표준입니다 SETNX + EXPIRE 두 명령 분리는 중간 crash 시 데드락을 유발하므로 반드시 원자 옵션을 사용합니다
- value에 소유자 식별자를 담아야 합니다 TTL 만료 후 다른 클라이언트가 락을 획득한 상황에서, 이전 소유자가 실수로 새 락을 삭제하는 사고를 막습니다
- 락 해제는 Lua 스크립트로 원자화합니다 GET(소유자 확인) + DEL(삭제) 사이에 다른 명령이 끼어드는 것을 막기 위함입니다
- TTL은 처리 완료에 충분한 시간으로 설정합니다 TTL이 처리보다 짧으면 유효한 락이 만료되어 중복 점유가 재현될 수 있습니다
