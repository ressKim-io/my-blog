## 한 줄 요약

> Redis는 메모리 저장소지만 RDB 스냅샷과 AOF 명령 로그 두 가지 영속화 메커니즘을 제공합니다 — fsync 주기와 스냅샷 주기에 따라 장애 시 유실할 수 있는 데이터 윈도우가 결정됩니다

---

## 🤔 무엇을 푸는 기술인가

### 메모리 저장소의 근본 약점

Redis는 모든 데이터를 RAM에 보관합니다 처리 속도가 빠른 이유가 바로 여기 있습니다 그러나 프로세스가 종료되거나 서버가 재기동되면 RAM의 내용은 사라집니다 전원이 꺼진 메모리는 빈 판입니다

이 약점이 Redis를 "캐시 전용"으로만 쓸 수 있게 만드는 것처럼 보입니다 캐시라면 데이터를 잃어도 됩니다 — 원본이 데이터베이스에 있으니까요 그러나 Redis를 티켓 재고나 좌석 상태의 **원본 저장소(SoT)** 로 쓰려면 이야기가 달라집니다 Redis가 죽어도 데이터가 살아있어야 합니다

Redis는 두 가지 영속화 메커니즘을 제공합니다

- **RDB (Redis Database)**: 특정 시점의 메모리 전체를 이진 파일로 덤프
- **AOF (Append-Only File)**: 메모리를 바꾸는 모든 쓰기 명령을 텍스트 로그로 기록

둘 다 파일 시스템에 기록하므로 재기동 후에도 데이터를 복원할 수 있습니다

## 🔧 동작 원리

### RDB 스냅샷: 특정 시점 전체 덤프

RDB는 특정 조건이 충족될 때 메모리 전체를 하나의 이진 파일(`dump.rdb`)로 저장합니다 조건은 `save` 설정으로 지정합니다

```text
save 900 1      # 900초(15분) 안에 쓰기 1회 이상 → 스냅샷
save 300 10     # 300초(5분) 안에 쓰기 10회 이상 → 스냅샷
save 60 10000   # 60초 안에 쓰기 10,000회 이상 → 스냅샷
```

조건이 충족되면 Redis는 `BGSAVE` 명령을 내부적으로 실행합니다 흐름은 다음과 같습니다

1. Redis 메인 프로세스가 `fork()`로 자식 프로세스를 생성
2. 자식 프로세스가 부모의 메모리를 읽어 디스크에 기록
3. 기록 완료 후 임시 파일을 `dump.rdb`로 원자적으로 교체

`fork()` 이후 부모는 계속 요청을 처리합니다 자식이 스냅샷을 쓰는 동안 메인 스레드가 블로킹되지 않습니다 Linux의 **Copy-on-Write(CoW)** 덕분입니다 자식 프로세스는 부모 메모리를 물리적으로 복사하지 않고, 부모가 특정 페이지를 수정할 때만 그 페이지를 실제로 복사합니다 스냅샷 도중 쓰기가 많으면 CoW 복사가 늘어나 메모리 사용량이 순간적으로 증가할 수 있습니다

재기동 시에는 `dump.rdb`를 읽어 메모리를 채웁니다 이진 형식이라 로드 속도가 빠릅니다 다만 스냅샷 시점 이후 변경된 데이터는 복원되지 않습니다 마지막 스냅샷 이후 장애가 나면 그 사이 쓰기는 사라집니다

### AOF: 명령 로그 이어붙이기

AOF는 메모리를 바꾸는 모든 쓰기 명령을 텍스트 파일(`appendonly.aof`)에 순서대로 기록합니다

```text
*3
$4
HSET
$14
seat:{101}:A1
$6
status
...
```

RESP(Redis Serialization Protocol) 형식입니다 명령이 들어올 때마다 이 형태로 파일에 이어붙입니다 재기동 시에는 이 파일을 처음부터 읽어 명령을 순서대로 재실행해 메모리 상태를 복원합니다

AOF의 내구성은 **`appendfsync`** 설정이 결정합니다

```text
appendfsync always    # 매 명령 직후 fsync — 성능 부담 가장 큼
appendfsync everysec  # 1초마다 fsync — 기본값, 유실 ≤ 1초
appendfsync no        # OS 맡김 — 빠르지만 유실 가변
```

`fsync`는 OS 버퍼 캐시의 내용을 물리 디스크에 실제로 내려쓰는 시스템 콜입니다 Redis는 쓰기 명령을 먼저 인메모리 AOF 버퍼에 넣고, `appendfsync` 설정에 따라 버퍼를 파일에 append합니다 그 후 `fsync`로 디스크에 확정합니다

`fsync`를 호출하기 전에 Redis가 죽으면 버퍼의 내용은 사라집니다 이것이 유실 윈도우입니다

![AOF appendfsync 모드별 유실 윈도우 타임라인|tall](/diagrams/goti-deepdive-redis-persistence-1.svg)

위 타임라인은 세 모드에서 쓰기 명령이 디스크에 확정되는 시점과 장애 발생 시 잃는 데이터 범위를 보여줍니다

**always 모드**(초록)에서는 매 명령 직후 `fsync`를 호출합니다 각 명령 박스 바로 밑에 fsync 마커가 붙어 있습니다 장애가 나도 직전 명령까지는 안전합니다 유실이 0에 가깝지만 매 명령마다 디스크 I/O가 발생해 처리량이 크게 줄어듭니다 쓰기가 많은 환경에서는 Redis의 속도 장점을 상쇄합니다

**everysec 모드**(보라)는 1초 단위 점선으로 fsync 시점이 표시됩니다 빨간 장애 선이 두 fsync 사이에 끼어 있을 때 유실 구간(분홍 박스)은 최대 1초입니다 성능과 내구성의 균형점으로, Redis 공식 문서가 권장하는 기본값입니다

**no 모드**(주황)에서는 Redis가 `fsync`를 직접 호출하지 않습니다 OS가 적절한 시점에 버퍼를 내려씁니다 OS 버퍼 비우는 주기는 통상 30초 이상입니다 타임라인에서 유실 구간(분홍 박스)이 훨씬 넓습니다 장애 전 OS가 fsync를 한 번도 안 했다면 그 사이 모든 쓰기가 사라집니다

### RDB와 AOF의 비교 구조

두 방식은 파일 구조와 복구 방식이 근본적으로 다릅니다

![RDB 스냅샷 vs AOF 로그 구조 및 재기동 복구 비교|tall](/diagrams/goti-deepdive-redis-persistence-2.svg)

왼쪽 RDB 흐름에서 Redis 메인 프로세스가 `fork()`를 호출하면 자식 프로세스가 생성됩니다 자식은 CoW 메커니즘으로 스냅샷 시점의 메모리를 읽어 `dump.rdb` 이진 파일에 씁니다 완료되면 임시 파일을 원자적으로 교체합니다 재기동 시에는 이 이진 파일 하나를 메모리에 로드하면 됩니다 명령 개수에 관계없이 파일 크기에 비례해 빠르게 복원됩니다

오른쪽 AOF 흐름에서는 쓰기 명령마다 AOF 버퍼에 RESP 텍스트가 쌓입니다 `appendfsync` 주기에 맞춰 `fsync`가 호출되어 `appendonly.aof` 파일에 확정됩니다 파일 내부 예시에서 볼 수 있듯 RESP 형식 명령이 순서대로 이어붙어 있습니다 재기동 시에는 이 파일을 처음부터 읽어 명령을 하나씩 재실행합니다 명령 수가 많을수록 복구 시간이 늘어납니다

다이어그램 하단 파란 박스가 혼용 모드입니다 RDB와 AOF를 동시에 활성화할 수 있습니다 재기동 시 Redis는 AOF를 우선 로드합니다 AOF가 RDB보다 더 최신 상태를 갖고 있기 때문입니다 RDB는 정기 스냅샷으로 복구 베이스라인 역할을 하고, AOF가 그 이후 변경분을 보완합니다

## 📐 세부 동작과 옵션

### fsync 주기별 트레이드오프

| 모드 | 유실 윈도우 | 성능 영향 | 적합한 상황 |
|---|---|---|---|
| `always` | ~0 (직전 명령까지 보장) | 높음 — 매 명령 디스크 I/O | 금융 트랜잭션, 절대 유실 불가 |
| `everysec` | 최대 1초 | 낮음 — 백그라운드 스레드 fsync | SoT 패턴, 대부분의 일반 사용 |
| `no` | 가변 (수십 초~분 단위) | 없음 | 캐시 전용, 재기동 복구 불필요 |

`everysec`에서 `fsync`는 별도 백그라운드 스레드가 처리합니다 메인 스레드가 블로킹되지 않습니다 1초 주기가 지났을 때 이전 fsync가 아직 완료되지 않았다면(디스크가 느릴 때), Redis는 최대 2초를 기다린 후 그래도 완료되지 않으면 `fsync`를 건너뜁니다 이 경우 유실 윈도우가 2초로 늘어날 수 있습니다

### AOF 파일 비대화와 rewrite

AOF는 명령을 계속 이어붙이므로 파일이 무한히 커집니다 동일 키를 100번 수정하면 100개의 명령이 쌓이지만, 실제 필요한 것은 최종 상태 한 줄입니다 Redis는 **AOFREWRITE**를 통해 이 문제를 해결합니다

```bash
# 수동 트리거
redis-cli BGREWRITEAOF

# 자동 트리거 설정
auto-aof-rewrite-percentage 100   # AOF 파일이 기준 크기의 2배 되면
auto-aof-rewrite-min-size 64mb    # 최소 64MB 이상일 때만
```

`BGREWRITEAOF`를 실행하면 자식 프로세스가 현재 메모리 상태를 최소 명령 집합으로 재작성합니다 100번 수정된 키는 최종 상태의 `SET` 명령 하나로 압축됩니다 재작성 완료 후 기존 AOF를 교체합니다 복구 시간과 파일 크기를 동시에 줄입니다

### RDB 트리거 방식 정리

RDB 스냅샷은 세 가지 방법으로 실행됩니다

```bash
SAVE      # 동기 — 메인 스레드 블로킹 (운영 중 금지)
BGSAVE    # 비동기 — fork 후 자식이 처리
save 설정 # 조건 충족 시 자동 BGSAVE
```

`SAVE`는 메인 스레드를 블로킹합니다 스냅샷 완료 전까지 모든 요청이 멈춥니다 운영 환경에서는 절대 쓰지 않습니다 `BGSAVE`가 안전한 선택입니다

Redis가 복제(replica)를 사용하는 경우 replica에서 `BGSAVE`를 실행하는 것이 권장 패턴입니다 primary의 `fork()` 비용과 CoW 메모리 증가를 피할 수 있습니다

### 재기동 시 복구 우선순위

AOF와 RDB가 모두 존재할 때 Redis는 다음 순서로 복구를 진행합니다

```text
AOF 활성화 여부 확인
  → AOF 활성화: appendonly.aof 로드 (더 최신 상태)
  → AOF 비활성화: dump.rdb 로드
```

AOF가 손상된 경우 `redis-check-aof --fix appendonly.aof`로 복구를 시도할 수 있습니다 손상 지점 이후를 잘라내고 그 전까지의 상태로 복원합니다

## 🧩 go-ti에서는

go-ti 티켓팅 플랫폼은 Redis를 단순 캐시가 아닌 좌석 상태·재고·주문의 원본 저장소로 운영합니다 `maxmemory-policy: noeviction`으로 설정해 메모리 압박 시 eviction 대신 오류를 반환하도록 했습니다 티켓 데이터를 캐시처럼 조용히 날리는 것은 허용할 수 없기 때문입니다

영속화는 **AOF everysec** 모드를 사용합니다 ADR에 명시된 리스크 분석에서 "AOF 최대 1초 유실 → outbox-replay, redis-rebuild-from-rds 운영 스크립트 준비 필수"로 기록되어 있습니다 1초 유실 윈도우는 수용 가능한 범위로 판단했습니다 장애 시 Redis Stream outbox를 통해 RDS에 보관된 이벤트를 재실행해 복구하는 경로가 설계에 포함되어 있기 때문입니다 always 모드는 처리량 목표(수만 TPS)와 충돌하므로 선택하지 않았습니다

> 이 기술을 go-ti에 도입하기까지의 의사결정 과정(검토한 대안과 기각 이유)은
> [Redis를 SoT로 채택한 이유 — SDD-0005 전면 승격 (ADR)](/logs/goti-redis-sot-adoption-adr)에 정리했습니다

## 📚 핵심 정리

- Redis는 RDB 스냅샷(특정 시점 이진 덤프)과 AOF 명령 로그(쓰기 명령 텍스트 기록) 두 메커니즘으로 영속화를 제공합니다
- RDB는 주기적 스냅샷이라 마지막 스냅샷 이후 변경 사항을 잃을 수 있습니다 복구 속도가 빠르고 파일 크기가 작습니다
- AOF는 `appendfsync` 설정으로 유실 윈도우를 제어합니다 `everysec`는 최대 1초 유실, `always`는 유실 0에 가깝지만 성능 부담이 큽니다
- `fsync`는 OS 버퍼 캐시를 물리 디스크에 확정하는 시스템 콜입니다 Redis가 죽어도 fsync 완료 시점까지는 복구됩니다
- AOF와 RDB를 혼용하면 재기동 시 Redis가 AOF를 우선 로드합니다 RDB는 복구 베이스라인, AOF가 이후 변경분을 보완합니다
- Redis를 SoT로 쓴다면 noeviction + AOF everysec 조합이 기본 출발점입니다 유실 1초가 허용되지 않으면 always 또는 복구 경로(outbox replay 등)를 함께 설계해야 합니다
