## 🎯 한 줄 요약

> 순수 대기열 성능 비교에서 B가 안정성 6관왕이었지만, **대용량 트래픽 흡수와 무중단 운영**이라는 목표 하에 확장 가능한 구조가 결정 기준이 되었습니다. CDN 캐싱으로 status 926ms→220ms를 달성한 A 구현체가 origin 부하를 VU와 분리할 수 있는 유일한 구조였기에 최종 선정되었습니다

---

## 🎬 이 결정의 컨텍스트

### 우리가 풀려는 문제

go-ti는 KBO 인기 경기 오픈 시점의 트래픽을 시뮬레이션합니다
실제 KBO 한국시리즈나 개막전은 오픈 직후 단시간에 수만 명이 동시에 폴링을 던지는 패턴이고, 이 트래픽을 흘려보내지 못하면 origin 서버가 즉시 무너집니다

이 PoC가 풀어야 했던 목표는 셋이었습니다

- **대용량 흡수**: 3,000 VU에서도 status 폴링이 흔들리지 않을 것. 가능하면 그 이상으로 확장 가능할 것
- **무중단**: 부하가 올라가는 구간에서도 5xx 에러가 거의 발생하지 않을 것
- **운영 가능한 구조**: VU가 10배 늘어나도 origin이 비례해서 무너지지 않을 것 (확장성)

세 번째 항목이 이 결정의 무게중심입니다
"지금 3000 VU에서 잘 도는가"보다 "VU가 10,000, 100,000으로 갈 때 어떤 구조가 살아남는가"가 PoC의 핵심 질문이었습니다

### 왜 3개 PoC를 비교했는가

세 명이 각자 다른 전략으로 구현한 이유는 어느 전략이 위 목표에 부합하는지 미리 알 수 없었기 때문입니다

- **A**: status 호출 시 동적 계산 + JWE 토큰 → 처리량 우선
- **C**: Scheduler 1초 주기 + Hash 카운터 → 단순함 우선
- **B**: 이벤트 기반 승격 + Heartbeat → 응답·안정성 우선

세 방식 모두 일정 부하까지는 작동합니다
**문제는 "한계점"과 "확장 곡선"이 다르다는 점**이고, 이를 같은 부하 조건에서 직접 비교해야 의사결정의 근거가 만들어집니다

### 비교 조건

2편에서는 3개 구현체의 Saturation 결과를 구현체별로 살펴봤습니다
이번 편에서는 **동일 축으로 정렬해서 직접 비교**하고, 최종 선정까지 진행합니다

| 항목 | Kind 클러스터 (2편) | EKS Prod (이번 편) |
|------|-------------------|-------------------|
| 인프라 | Kind 7노드 | EKS 10노드 (Karpenter) |
| 네트워크 | ALB 직접 | ALB 직접 + Cloudflare CDN |
| 공통 | maxCapacity=100, Pod 1개, 동일 경기 | maxCapacity=100, setup 토큰 일괄 발급 |

## 🔍 순수 대기열 성능 비교

### API 레이턴시

1000 VU와 3000 VU에서 모든 대기열 API 레이턴시를 비교했습니다

**1000 VU:**

| API (p95) | A | C | B | 최고 |
|-----------|------|------|------|------|
| queue_status | 2.18s | 361ms | **279ms** | B |
| queue_seat_enter | - | 452ms | **295ms** | B |
| http_req 전체 | 2.18s | 390ms | **282ms** | B |

**3000 VU:**

| API (p95) | A | C | B | 최고 |
|-----------|------|------|------|------|
| queue_status | 2.13s | 3.10s | **500ms** | B |
| queue_seat_enter | 2.29s | 2.99s | **562ms** | B |
| http_req 전체 | 2.13s | 3.10s | **501ms** | B |

B가 **모든 API 레이턴시에서 압도적 1위**입니다
3000 VU에서 status p95가 500ms — C(3.1s)의 1/6, A(2.13s)의 1/4 수준입니다

ZSET의 O(log N) 연산과 이벤트 기반 승격이 서버 부하를 최소화하는 구조가 레이턴시에 직접적으로 반영된 결과입니다

### 대기열 통과율

| VU | A | C | B |
|----|------|------|------|
| 300 | - | 99.26% | **100%** |
| 1000 | 81.56% | **100%** | **100%** |
| 3000 | 57.90% | **100%** | **100%** |

C와 B는 **전 구간 100% 통과**를 유지합니다
A는 3000 VU에서 57.90%로 하락합니다
빠른 슬롯 회전이 동시 좌석 경합을 유발하면서 예매 실패 → 재진입 비율이 높아지는 구조적 특성입니다

이 시점만 놓고 보면 **A는 무중단 목표를 만족하지 못합니다**
다만 이 통과율 문제는 슬롯 회전 속도의 부수효과이고, 운영 단계에서 좌석 락 정책으로 보완할 여지가 있습니다 (이 부분은 후속 단계에서 다룹니다)

### 대기 시간: 중간값 vs 꼬리

대기 시간은 p50(중간값)과 p95(꼬리)를 분리해서 봐야 합니다

**1000 VU:**

| 지표 | A | C | B |
|------|------|------|------|
| queue_wait p50 | 56.3s | 66.4s | **11.7s** |
| queue_wait p95 | **1m17s** | 2m16s | 2m54s |

**3000 VU:**

| 지표 | A | C | B |
|------|------|------|------|
| queue_wait p50 | 2m19s | 53s | **23s** |
| queue_wait p95 | 2m54s | **2m31s** | 4m48s |

흥미로운 패턴이 보입니다

**p50(대부분 유저)은 B가 압도적**입니다. 3000 VU에서도 23초
하지만 **p95(꼬리)는 B가 가장 깁니다.** 3000 VU에서 4분 48초

이것은 이벤트 기반 승격의 특성입니다
초반에 빠르게 통과하는 유저가 많지만, 대기열이 길어지면 이벤트 발생 간격이 불규칙해지면서 뒤쪽 유저의 대기가 길어집니다

### 슬롯 회전 속도 (처리량)

| VU | A | C | B |
|----|------|------|------|
| 300 iter | - | **4,476** | 457 |
| 1000 iter | **5,495** | 3,713 | 417 |
| 3000 iter | **6,574** | 763 | 290 |

처리량은 A가 압도적입니다
3000 VU에서 A(6,574)는 C(763)의 **8.6배**, B(290)의 **22.7배** 처리합니다

## 🔍 서버 부하 비교

성능만큼 중요한 것이 **서버가 얼마나 부담을 받는가**입니다
확장성 목표를 평가하려면 이 축이 결정적입니다

### Redis 연산 추정

| 구현체 | polling 간격 | heartbeat | VU당 Redis ops/sec | 3000 VU 합계 |
|--------|-------------|-----------|-------------------|-------------|
| A | 2초 | 없음 | 0.5 | ~1,500 ops/s |
| C | 1초 | 없음 | 1.0 | ~3,000 ops/s |
| B | 5초 | 10초 | 0.3 | ~900 ops/s |

B는 polling 5초 + heartbeat 10초로 **VU당 Redis 부하가 가장 낮습니다**
C는 1초 polling으로 3000 VU에서 Redis에 3,000 ops/s가 들어갑니다

### 총 HTTP 요청

| 구현체 | 3000 VU 총 요청 | recv | sent |
|--------|-----------------|------|------|
| A | 445,086 | 453MB | 32MB |
| C | 329,262 | 204MB | 22MB |
| B | **309,449** | **172MB** | 25MB |

B가 서버 요청 수와 네트워크 트래픽 모두 **가장 적습니다**
A는 빠른 회전으로 총 요청이 가장 많습니다

### 스케일링 내성

VU가 10배 증가할 때 레이턴시가 얼마나 올라가는지를 보면

| 구현체 | status p95 증가율 (300→3000 VU) | 특성 |
|--------|-------------------------------|------|
| B | 281ms → 500ms (**1.8x**) | 거의 선형 |
| A | - → 2.13s (기준 없음) | pod 한계에서 안정 |
| C | 404ms → 3.10s (**7.7x**) | 비선형 포화 |

B는 VU가 10배 늘어도 레이턴시가 1.8배만 증가합니다
C는 7.7배 — 3000 VU에서 서버가 포화 상태에 진입합니다

이 표만 보면 B가 확장성 목표에도 가장 부합해 보입니다
**그런데 이게 함정이었습니다.** 뒤에서 다시 다룹니다

## 📊 승격 메커니즘 비교

세 구현체의 내부 동작을 나란히 놓으면

| 항목 | A | C | B |
|------|------|------|------|
| **자료구조** | Hash + Lua script | Hash (O(1) 카운터) | ZSET + HASH |
| **승격 트리거** | status 호출 시 동적 계산 | Scheduler 1초 주기 | 이벤트 기반 (leave/expire) |
| **Heartbeat** | 없음 | 없음 | 있음 (30s TTL, 10s 갱신) |
| **만료 처리** | Scheduler 30초 주기 | Redis TTL 자동 | Key Expiration Event |
| **토큰** | JWE (AES-256-GCM) | JWT | Custom AES (@RequestParam) |
| **분산 락** | Lua script (원자적) | 없음 | Redisson |
| **Redis 키 수/VU** | ~2개 | ~2개 | ~5개 |

A의 핵심은 **"status 호출 = 승격 판단"**입니다
별도 Scheduler 없이 매 polling에서 publishedRank를 동적으로 계산해서 즉시 승격합니다
이것이 슬롯 회전이 빠른 이유입니다

B는 **이벤트 기반**이라 서버 부하는 적지만 승격 타이밍이 이벤트 발생에 의존합니다
leave나 expire가 없으면 승격이 일어나지 않아서 슬롯 회전이 느립니다

## 🏆 최종 선정: A + CDN 캐싱

### EKS Prod 3000 VU 테스트

Kind 클러스터에서의 비교를 마친 후, **EKS Prod 환경**에서 최종 검증을 진행했습니다
실제 운영 환경에 가까운 인프라(EKS + Karpenter)에서 같은 패턴이 유지되는지 확인하기 위해서입니다

**ALB 직접 (CDN 미적용):**

| 항목 | A | C | B |
|------|------|------|------|
| queue_pass_rate | 87.44% | **100%** | **100%** |
| queue_timeouts | 157건 | **0건** | **0건** |
| queue_status avg | 926ms | 2.66s | **158ms** |
| queue_status p95 | 1.98s | 7.48s | **719ms** |
| 처리량 (req/s) | 655 | 500 | 602 |
| 1K→3K status 증가율 | 16x | 24x | **4x** |

EKS Prod에서도 Kind와 같은 패턴이 확인됩니다
B가 레이턴시와 안정성에서 최고, C가 3000 VU에서 포화(status avg 2.66s)

### CDN 캐싱의 결정적 차별점

A 구현체에 Cloudflare CDN 캐싱을 적용한 결과입니다

| 항목 | ALB 직접 | CDN 캐싱 | 변화 |
|------|---------|---------|------|
| **queue_status avg** | 926ms | **220ms** | **4.2x 개선** |
| **queue_status p95** | 1.98s | **334ms** | **5.9x 개선** |
| **처리량 (req/s)** | 655 | **1,044** | **1.6x 증가** |
| **처리량 (iterations/5min)** | 907 | **1,889** | **2.1x 증가** |
| **http_req_failed** | 1.26% | **0.08%** | **16x 감소** |

이 숫자들이 말하는 바를 생각해보겠습니다

CDN 캐싱을 적용하면 status API의 응답 시간이 926ms에서 **220ms**로 떨어집니다
이 220ms는 B의 ALB 직접 결과(158ms)에 근접하는 수치입니다

하지만 핵심 차이가 있습니다
B의 158ms는 **origin 서버가 직접 응답**한 것입니다
A의 220ms는 **Cloudflare 엣지가 캐시에서 응답**한 것입니다

VU가 3000에서 10,000, 100,000으로 올라가면 어떻게 될까요?

![B 안과 A 안의 확장 부하 곡선 비교](/diagrams/queue-poc-loadtest-part3-selection-1.svg)

**polling 요청을 엣지에서 흡수**하면 origin 부하는 VU 수와 거의 무관하게 일정합니다
CDN `max-age=1`이라서 대기열 상태는 최대 1초 지연되지만, 사용자 체감에는 미미한 수준입니다

### 왜 B가 아니라 A를 골랐는가

순수 성능 표만 보면 B가 6관왕이고, 스케일링 내성도 B가 최고입니다
그런데 결국 A를 택한 이유는 **확장성 곡선의 모양이 달랐기 때문**입니다

**B의 한계: 확장은 origin에 종속됩니다**
B의 polling 5초 + heartbeat 10초 구조는 응답을 항상 origin에서 만들어야 합니다
heartbeat는 사용자 세션 상태에 결합되어 있어서 캐싱이 불가능하고, polling 응답도 사용자별 위치 정보(rank)를 담기 때문에 의미 있는 캐시 키가 만들어지지 않습니다
스케일링 내성이 1.8x로 좋다고 해도, 그건 **origin의 한계 안에서 선형성**일 뿐 origin 자체의 한계를 깨지는 못합니다

**A의 가능성: origin과 polling을 분리할 수 있습니다**
A의 JWE 토큰은 사용자 상태를 토큰 자체에 담는 stateless 구조입니다
이 덕분에 status 응답이 사용자별로 거의 동일한 형태가 되고, 짧은 TTL(max-age=1)로도 캐시 히트율이 매우 높게 나옵니다
**polling 트래픽을 엣지가 흡수**하면 origin은 좌석 선택·결제 같은 무거운 트랜잭션에만 집중할 수 있습니다

대용량/무중단 목표를 만족하려면 단순히 "지금 빠른 것"이 아니라 **"VU가 10배 늘어도 origin이 비례해서 무너지지 않는 구조"**가 필요했고, 그게 가능한 건 A뿐이었습니다

### 선택 근거

1. **CDN 캐싱이 결정적 차별점**: polling 요청을 엣지에서 흡수 → status 926ms → 220ms
2. **확장성**: VU 증가 시 CDN 캐시 히트율이 높아져 origin 부하는 거의 일정
3. **처리량 최고**: CDN 경유 1,044 req/s, 1,889 iterations/5min
4. **에러율 최저**: http_req_failed 0.08% (ALB 직접 대비 16배 개선)
5. **timeout 이슈는 해결 가능**: Cloudflare Free plan rate limit으로 signup이 실패한 것 — signup을 ALB 직접으로 분리하면 해결

### 트레이드오프 — 무엇을 포기했는가

솔직히 순수 성능만 보면 B가 더 안정적입니다. Timeout 0건, 에러 0건

이번 결정으로 우리는 다음을 포기했습니다

- **3000 VU에서의 즉각적 안정성**: A는 ALB 직접 모드에서 87.44% 통과율
- **B의 단순한 운영 모델**: B는 추가 인프라(CDN) 없이도 안정적으로 작동
- **꼬리 대기 시간이 짧은 경험**: B의 p95 4분 48초는 길지만, A의 일부 사용자는 통과 자체에 실패하는 게 사실

대신 다음을 얻었습니다

- **VU 수와 origin 부하의 분리**: 확장성 목표 달성

## 📚 핵심 포인트

**첫째, 순수 성능만으로는 최선을 판단할 수 없습니다**
B가 9개 카테고리 중 6개에서 1위였지만, CDN이라는 변수 하나가 판도를 뒤집었습니다
아키텍처 결정에서는 "현재 성능"보다 **"확장 가능한 구조"**가 더 중요합니다

**둘째, polling 요청을 엣지에서 흡수하면 origin 부하는 VU와 무관하게 거의 일정합니다**
대기열 상태 `max-age=1`은 사용자 체감에 미미한 지연이지만, 서버 부하 측면에서는 엄청난 차이를 만듭니다

**셋째, CDN 캐싱 가능 여부는 설계 단계에서 결정됩니다**
C(Redis 직접 조회)와 B(heartbeat)는 구조적으로 CDN 캐싱이 어렵습니다
A의 JWE 토큰 방식은 상태를 토큰에 담아 stateless하게 만들기 때문에 CDN 캐싱이 자연스럽게 가능했습니다

**넷째, 의사결정의 무게중심을 명시하지 않으면 비교가 흔들립니다**
"대용량 흡수 + 무중단 + 확장 가능한 구조"라는 목표를 명확히 두지 않았다면, 6관왕인 B를 그냥 골랐을 가능성이 높습니다
PoC 비교는 항상 **무엇을 위한 비교인지**가 결과 해석을 좌우합니다
