## 🔥 문제: 페이지 간 사실관계 충돌로 일관성 붕괴

### 배경

`slides/` 메인 포트폴리오(21장) 전반에 걸쳐 톤과 사실관계를 일괄 점검했습니다.

문제는 개별 페이지만 볼 때는 자연스럽게 읽히는 표현들이 다른 페이지의 사실과 충돌한다는 점이었습니다.

| 표현 | 충돌 지점 |
|------|-----------|
| "DevOps / Infra 단독 책임" | 7직군 16명 팀, 클라우드 네이티브 팀장으로 주도한 것이 사실 |
| "Java→Go 전환" (본인 기여로 강조) | Go 전환은 팀 작업, 발표자 기여는 인프라 영역에 한정 |
| "대시보드 팀 15명 적용" | 실제 일상 사용자는 백엔드만 해당, 전체 팀 사용은 사실과 다름 |

페이지별 검토로는 잡히지 않는 유형의 오류였습니다. 면접에서 다른 페이지와 대조하면 모순으로 노출됩니다.

## 🤔 원인: 페이지 단위 작성의 맹점

포트폴리오를 페이지별로 나눠 작성하면, 각 페이지 담당자(또는 각 작업 세션)가 해당 페이지의 맥락에서만 검토합니다.

"단독"이라는 표현은 인프라 구현 책임의 관점에서는 사실이지만, 팀 구성(7직군 16명)과 나란히 읽히면 협업 기여를 지워버립니다.

"Go 전환" 표현도 마찬가지입니다. KPI 카드에 "Java→Go" 메트릭을 넣으면 이 마이그레이션이 발표자의 성과처럼 읽힙니다. 그러나 실제 기여는 KEDA + Karpenter 스파이크 대응, PgBouncer 도메인 Proxy 등 인프라 영역이었습니다.

**09-m04 통째 재작성 (가장 큰 변경)**

이 페이지가 사실관계 오류의 핵심이었습니다.

기존 서사는 "JVM 한계 + Thundering Herd → 5 MSA 전면 Go 전환"이었습니다. Go 전환을 인프라 기여와 동등하게 묶어 발표자의 성과로 제시하는 구조였습니다.

수정 후 서사는 "스파이크 트래픽 대응 — 인프라 영역 기여"입니다.

발표자의 실제 기여만 남겼습니다.

1. **KEDA + Karpenter 30분 선대응** — Cron Job 기반 Pod·Node 사전 확장

KPI는 5xx 0%대, 10만 동시접속, RDS 커넥션 300 미만으로 정리했습니다.

회고 문장도 수정했습니다.

> "성능은 팀이 함께 풀었습니다 — 인프라가 먼저 받쳐주도록 책임졌습니다"

### AI Dev Platform Addendum (14 슬라이드) 재평가

Addendum을 처음 검토할 때 "AI 활용은 기본기"라는 일반론으로 평가했다가, 피드백을 받고 정정했습니다.

실제 내용을 다시 읽은 평가는 다음과 같습니다.

이 Addendum은 단순 AI 활용이 아니라 **Internal AI Platform 구축** 사례입니다.

주요 신호를 정리하면 다음과 같습니다.

- **Harness Engineering 선행**: Anthropic이 2026-02에 명명한 프레임워크(tools/context/guardrails/workflow/sensors/self-correcting)를 그 전에 이미 구현했습니다
- **자기 개선 사이클 구축**: log-trouble → pattern analysis → skill design → skill update 사이클이 89건 누적 → Rules 5 + Skills 5 형태로 환원됩니다
- **OSS 공개 검증**: 160+ skills, 26 agents를 외부 공개한 상태입니다
- **메타 증명**: 포트폴리오 자체가 시스템 산출물입니다 — architect-agent / code-reviewer / log-decision agent로 이 문서를 빌드했습니다

제출 전략은 다음과 같이 결정했습니다.

메인 포트폴리오 + Addendum 둘 다 제출합니다. Addendum은 차별화 자산이므로 제거하지 않습니다. 컨설턴트 피드백을 받기 전에 본인 판단으로 깎는 것은 위험합니다.

## 📚 배운 점

- **페이지 간 일관성 점검이 페이지별 검토보다 중요합니다** — "단독", "팀 15명 적용" 같은 표현은 한 페이지만 볼 때는 자연스럽습니다. 그러나 다른 페이지의 사실(7직군 협업, 백엔드만 일상 사용)과 나란히 읽히면 면접에서 모순으로 노출됩니다. 마지막 전수 점검은 페이지 순서가 아닌 표현 단위로 진행하는 것이 효과적입니다
- **사실관계와 톤은 다릅니다** — "단독 → 주도", "Go 전환 P50 → API 응답 P50"은 어필 톤을 약하게 만드는 것이 아닙니다. 반증 가능한 표현을 제거하고 실제 기여 영역을 선명하게 드러내는 쪽이 더 강한 피칭입니다
- **차별화 자산은 시장 반응으로 검증해야 합니다** — AI Addendum을 처음에 일반론으로 평가절하했다가 정정했습니다. 본인 판단으로 자산을 깎으면 컨설턴트나 면접관이 줄 수 있는 피드백 기회를 없애버립니다
- **여러 사이클을 한 번에 뽑지 않습니다** — 한 줄 피드백이 들어올 때마다 1개 페이지씩 사이클을 돌리는 것이 결과적으로 빠르고 정확합니다. 5건을 일괄 적용하면 검토가 누락됩니다
