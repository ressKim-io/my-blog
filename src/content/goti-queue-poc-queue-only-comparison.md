## 한 줄 요약

> 티켓팅/결제 속도를 제외하고 **대기열 자체 성능만** 비교했습니다. 구현 A(Redis 폴링), 구현 B(Kafka 이벤트 기반), 구현 C(CDN 캐싱) 셋을 9개 카테고리로 정리하면, 구현 B가 레이턴시·안정성·서버 부하·스케일링 내성에서 가장 우수하고 구현 A가 처리량에서 앞섭니다.

---

## 🔥 배경: 왜 "순수 큐 성능"만 비교하나

지금까지 시리즈 4·5·6편에서 3개 POC의 포화 결과를 각각 정리했습니다.
그 결과에는 **대기열 통과 이후의 티켓팅/결제 처리 시간**이 섞여 있었습니다.

대기열 아키텍처를 선정하려면 **큐 자체의 특성**만 분리해서 봐야 합니다.
티켓팅 SQL 최적화나 결제 연동 상태는 구현마다 다르고, 이 부분은 나중에 공통 구현으로 통일할 수 있기 때문입니다.

**테스트 조건**:

- 테스트일: 2026-03-30
- `maxCapacity=100`, 동일 경기, 동일 인프라(Kind 7노드)
- Queue Pod 1개, CDN 미적용
- 비교 범위: 대기열 enter/status/seat_enter/heartbeat API만

구현 A는 Redis 폴링 기반(Hash + Scheduler), 구현 B는 이벤트 기반(ZSET + 분산 락 + Key Expiration), 구현 C는 Lua script 기반(Hash + 동적 순번 계산)으로 각각 다른 접근을 취합니다.
어떤 구조가 순수 큐 성능에서 우위에 있는지 차례로 확인하겠습니다.

## 1. 대기열 API 응답 시간

### 1,000 VU

| API | 구현 C | 구현 A | 구현 B |
|-----|------|------|------|
| enter p95 | - | - | **293ms** |
| status p95 | 2.18s | 390ms | **279ms** |
| seat_enter p95 | - | 452ms | **295ms** |
| heartbeat p95 | 없음 | 없음 | 278ms |
| http 전체 p95 | 2.18s | 390ms | **282ms** |

### 3,000 VU

| API | 구현 C | 구현 A | 구현 B |
|-----|------|------|------|
| enter p95 | - | - | **917ms** |
| status p95 | 2.13s | 3.10s | **500ms** |
| seat_enter p95 | 2.29s | 2.99s | **562ms** |
| heartbeat p95 | 없음 | 없음 | 482ms |
| http 전체 p95 | 2.13s | 3.10s | **501ms** |

두 표를 합쳐서 보면 구현 B가 전 구간 전 API에서 레이턴시 1위입니다.
특히 3,000 VU 구간에서 차이가 벌어집니다.

구현 B의 status p95는 500ms로, 구현 A(3.10s)의 약 1/6, 구현 C(2.13s)의 약 1/4 수준입니다.
seat_enter p95도 562ms로 나머지 둘보다 5배 이상 빠릅니다.

구현 A는 1,000 VU에서는 390ms로 양호했지만, 3,000 VU에서 3.10s로 급증합니다.
구현 C는 1,000 VU에서도 이미 2.18s로 높았고, 3,000 VU에서도 비슷한 수준(2.13s)이 유지됩니다.
구현 C는 Lua script에서 `publishedRank`를 매 호출마다 다시 계산하는 구조라 부하가 꾸준히 높게 유지되는 것으로 보입니다.

## 2. 대기열 통과율 + 안정성

| VU | 구현 C pass | 구현 A pass | 구현 B pass | 구현 C err | 구현 A err | 구현 B err |
|----|-----------|-----------|-----------|----------|----------|----------|
| 300 | - | 100% | 100% | - | 5.26% | **0.00%** |
| 1,000 | 81.56% | 100% | 100% | 0.51% | 0.013% | **0.00%** |
| 3,000 | 57.90% | 100% | 100% | 0.63% | 0.003% | **0.00%** |

통과율과 에러율을 같이 보면 구현별 안정성이 명확히 드러납니다.

**구현 B**는 전 구간에서 통과율 100% + 에러 0건입니다.
300, 1,000, 3,000 VU 모두 단 한 건의 요청도 실패하지 않았습니다.

**구현 A**는 통과율은 100%를 유지하지만, 300 VU에서 에러율 5.26%가 관측됐습니다.
1,000 VU 이상에서는 에러율이 0.01% 수준으로 떨어지지만 0으로 수렴하지는 않습니다.

**구현 C**는 VU가 늘수록 통과율이 급격히 하락합니다.
1,000 VU에서 81.56%, 3,000 VU에서는 57.90%로 **24pp 하락**합니다.
대기열에 들어간 유저의 약 42%가 정상 통과하지 못했다는 의미입니다.

## 3. 대기 시간

### 1,000 VU

| 지표 | 구현 C | 구현 A | 구현 B |
|------|------|------|------|
| wait avg | 56.4s | 67.7s | **40.2s** |
| wait p50 | 56.3s | 66.4s | **11.7s** |
| wait p95 | **1m17s** | 2m16s | 2m54s |
| wait max | **1m26s** | 2m38s | - |

### 3,000 VU

| 지표 | 구현 C | 구현 A | 구현 B |
|------|------|------|------|
| wait avg | 1m55s | **53.5s** | 1m48s |
| wait p50 | 2m19s | 53s | **23s** |
| wait p95 | 2m54s | **2m31s** | 4m48s |
| wait max | 3m5s | **3m49s** | - |

대기 시간은 평균·중간값(p50)·꼬리(p95) 지표가 서로 다른 이야기를 합니다.

**중간값(p50)** 기준으로는 구현 B가 가장 빠릅니다.
1,000 VU에서 11.7s, 3,000 VU에서 23s로, 대부분의 유저가 매우 빠르게 통과합니다.

**꼬리(p95)** 기준으로는 구현 C와 구현 A가 우세합니다.
구현 B는 3,000 VU에서 p95가 4m48s까지 늘어나 꼬리가 깁니다.

이 현상은 구현 B의 이벤트 기반 승격 방식에서 비롯됩니다.
leave/expire 이벤트가 발생할 때만 다음 유저가 승격되므로, 초반 유저는 빠르게 빠져나가지만 뒤쪽 유저는 이벤트가 몰릴 때까지 기다려야 합니다.
**초반에 빠르지만 꼬리가 길어지는 특성**을 보입니다.

반면 구현 A와 구현 C는 Scheduler 또는 매 호출 시 계산 방식이라 대기 시간 분포가 더 균등합니다.

## 4. 슬롯 회전 속도 (처리량)

| VU | 구현 C iter | 구현 A iter | 구현 B iter | 구현 C/s | 구현 A/s | 구현 B/s |
|----|-----------|-----------|-----------|--------|--------|--------|
| 300 | - | **4,476** | 457 | - | **11.84** | 1.21 |
| 1,000 | **5,495** | 3,713 | 417 | **14.5** | 9.84 | 1.10 |
| 3,000 | **6,574** | 763 | 290 | **12.2** | 2.0 | 0.77 |

처리량 지표는 앞의 결과를 완전히 뒤집습니다.
구현 B가 처리량에서는 꼴찌입니다.

3,000 VU 기준 구현 C는 초당 12.2 iterations, 구현 A는 2.0 iterations, 구현 B는 0.77 iterations입니다.
구현 C가 구현 B 대비 **약 16배**, 구현 A는 약 2.6배 처리량이 높습니다.

1,000 VU 구간에서도 구현 C 14.5/s 대 구현 B 1.1/s로 **약 13배** 차이입니다.
구현 B의 이벤트 기반 승격은 안정성과 낮은 서버 부하를 가져오지만, 단위 시간당 처리할 수 있는 유저 수에는 분명한 한계가 있습니다.

이 부분은 단일 Pod 기준의 수치입니다.
Pod 스케일아웃으로 처리량을 보완할 수 있는지는 시리즈 후속 글에서 다시 다룹니다.

## 5. 서버 부하 추정 (Redis 부하)

### VU당 Redis 연산 추정

| 연산 | 구현 C | 구현 A | 구현 B |
|------|------|------|------|
| **enter** | HASH write + Lua script | HASH write | ZSET zadd + HASH hset + STRING set (3건) |
| **status polling** | Lua script (읽기) | HASH hget | HASH hget + ZSET zrank (2건) |
| **heartbeat** | 없음 | 없음 | STRING setex (1건, 10초마다) |
| **seat_enter** | Lua script (쓰기) | HASH hset | ZSET zrem + STRING set + HASH hincrby (3건) |
| **leave** | Lua script | HASH update | ZSET zrem + STRING del + HASH hincrby (3건) |
| **만료 체크** | Scheduler 30초 | Redis TTL 자동 | Redis Key Expiration 이벤트 |

구현 B는 ZSET·HASH·STRING을 조합해 한 동작당 여러 연산을 수행합니다.
언뜻 보면 연산 수가 가장 많지만, 실제 서버 부하는 **polling 빈도**가 결정합니다.

### Polling 기반 Redis 읽기 부하

| 구현 | polling 간격 | heartbeat 간격 | VU당 Redis ops/sec | 3,000 VU Redis ops/sec |
|------|-------------|---------------|-------------------|----------------------|
| 구현 C | 2초 | 없음 | 0.5 (status만) | ~1,500 |
| 구현 A | 1초 | 없음 | 1.0 (status만) | ~3,000 |
| 구현 B | 5초 | 10초 | 0.3 (status+heartbeat) | ~900 |

**구현 B**는 polling 간격 5초 + heartbeat 10초로, VU당 Redis 부하가 가장 낮습니다.
3,000 VU 환경에서 Redis ops/sec가 약 900으로, 구현 A(3,000) 대비 1/3 수준입니다.

**구현 A**는 polling 1초로 가장 공격적입니다.
3,000 VU에서 초당 약 3,000 읽기 연산이 Redis로 쏟아집니다.
레이턴시가 3,000 VU 구간에서 급증하는 이유를 여기서 찾을 수 있습니다.

### 총 HTTP 요청 비교 (서버 부하 직결)

| VU | 구현 C reqs | 구현 A reqs | 구현 B reqs |
|----|-----------|-----------|-----------|
| 300 | - | - | 34,894 |
| 1,000 | - | - | 106,987 |
| 3,000 | 445,086 | 329,262 | 309,449 |

3,000 VU 기준 구현 B 309K, 구현 A 329K, 구현 C 445K로, 구현 B가 **서버 요청 수가 가장 적습니다**.
구현 C는 구현 B 대비 약 44% 더 많은 요청을 보냅니다.

### 네트워크 트래픽

| VU | 구현 C recv/sent | 구현 A recv/sent | 구현 B recv/sent |
|----|---------------|---------------|---------------|
| 3,000 | 453MB / 32MB | 204MB / 22MB | 172MB / 25MB |

네트워크 트래픽도 구현 B가 가장 적습니다.
recv 기준 구현 B는 172MB로, 구현 C(453MB)의 약 38% 수준입니다.

## 6. 스케일링 내성 (VU 증가 시 성능 저하율)

### API 레이턴시 증가율 (300 → 3,000 VU)

| 구현 | status p95 변화 | 비율 |
|------|----------------|------|
| 구현 C | - → 2.13s | - |
| 구현 A | 404ms → 3.10s | **7.7배 증가** |
| 구현 B | 281ms → 500ms | **1.8배 증가** |

구현 B는 VU가 10배(300 → 3,000) 증가해도 레이턴시는 1.8배만 증가합니다.
구현 A는 같은 조건에서 7.7배로 급증합니다.

이 차이는 **구조적 차이**에서 옵니다.
구현 A는 매 1초마다 전체 활성 VU가 동시에 polling하므로, VU가 증가할수록 Redis에 가해지는 부하가 선형으로 증가합니다.
구현 B는 polling 간격이 5초이고 heartbeat도 10초마다이기 때문에, VU 증가가 부하 증가로 직접 이어지지 않습니다.

### 통과율 저하율

| 구현 | 1,000 VU → 3,000 VU |
|------|--------------------|
| 구현 C | 81.56% → 57.90% (**-24pp 하락**) |
| 구현 A | 100% → 100% (유지) |
| 구현 B | 100% → 100% (유지) |

통과율 기준으로는 구현 A와 구현 B 둘 다 스케일링에 안정적입니다.
구현 C만 VU 증가 시 급격히 통과율이 떨어집니다.

## 7. Redis 자료구조 + 승격 메커니즘 비교

| 항목 | 구현 C (방식3) | 구현 A (방식1) | 구현 B (방식2) |
|------|-------------|-------------|-------------|
| **대기열 자료구조** | Hash + Lua script | Hash (O(1) 카운터) | **ZSET** (score=순번) + HASH |
| **순번 관리** | Hash 필드 | Hash 카운터 | **ZSET score** (정렬 보장) |
| **승격 트리거** | status 호출 시 동적 계산 | Scheduler 1초 주기 | **이벤트 기반** (leave/expire) |
| **승격 방식** | publishedRank 매번 재계산 | 빈 슬롯만큼 승격 | **분산 락 획득 → allowedNum 갱신** |
| **Heartbeat** | 없음 | 없음 | **있음** (TTL 30초, 10초 갱신) |
| **만료 처리** | Scheduler 30초 주기 | Redis TTL 자동 | **Redis Key Expiration Event** |
| **분산 락** | Lua script (원자적) | 없음 | **있음** (Redisson) |
| **토큰 방식** | JWT (JWE AES-256) | JWT | Custom AES (@RequestParam) |
| **Redis 키 수 (VU당)** | ~2개 | ~2개 | **~5개** (entry, ZSET, heartbeat, active, status) |

세 구현의 근본적인 차이는 **승격 트리거**에 있습니다.

**구현 C**는 매 status 호출 시 publishedRank를 Lua script로 재계산합니다.
polling 빈도만큼 Redis에서 무거운 연산이 반복됩니다.

**구현 A**는 Scheduler가 1초마다 깨어나 빈 슬롯 수만큼 일괄 승격합니다.
Redis 자료구조는 단순하지만, polling이 Scheduler 주기와 무관하게 계속 돌아갑니다.

**구현 B**는 leave/expire 이벤트가 발생할 때만 승격합니다.
분산 락(Redisson)으로 동시성을 제어하고, Key Expiration Event로 만료를 자동 감지합니다.
Redis 키 수는 VU당 약 5개로 가장 많지만, 연산 빈도 자체가 훨씬 낮습니다.

## 8. 버그/이슈 발생 건수

| 구현 | 테스트 중 발견된 버그 |
|------|---------------------|
| 구현 C | **3건** — Lua ARGV 직렬화, race condition, Redis 직렬화 혼용 |
| 구현 A | **2건** — SCAN 병목(→ O(1) 개선), queueComplete API 누락 |
| 구현 B | **0건** — MeshSecurityConfig 미적용은 인프라 설정 이슈(코드 버그 아님) |

구현 C의 3건 버그는 모두 **Lua script 관련**입니다.
Lua script는 원자성을 보장하는 대신, 디버깅이 어렵고 직렬화 이슈가 자주 발생합니다.

구현 A는 초기 구현에서 SCAN을 사용하다 병목을 만났고, O(1) 카운터로 개선했습니다.
queueComplete API 누락은 기능 측면의 이슈입니다.

구현 B는 큐 로직 자체에서는 버그가 없었습니다.
MeshSecurityConfig 관련 이슈는 Istio 인프라 설정 문제로, 큐 구현과 무관합니다.

## 9. 종합 평가 (대기열 전용)

| 카테고리 | 1위 | 2위 | 3위 |
|----------|-----|-----|-----|
| API 레이턴시 | **구현 B** | 구현 A | 구현 C |
| 통과율 (3,000 VU) | **구현 A/B 공동** | - | 구현 C |
| 대기 시간 중간값 | **구현 B** | 구현 A | 구현 C |
| 대기 시간 꼬리 | **구현 C** | 구현 A | 구현 B |
| 슬롯 회전 (처리량) | **구현 C** | 구현 A | 구현 B |
| 서버 부하 (Redis) | **구현 B** | 구현 C | 구현 A |
| 스케일링 내성 | **구현 B** | 구현 C | 구현 A |
| 안정성 (에러율) | **구현 B** | 구현 A | 구현 C |
| 코드 안정성 (버그) | **구현 B** (0건) | 구현 A (2건) | 구현 C (3건) |

9개 카테고리 중 구현 B가 6개에서 1위를 차지했습니다.
구현 C는 꼬리 대기 시간과 슬롯 회전에서 1위, 구현 A는 단독 1위 없이 전 구간 2위 근방입니다.

### 한 줄 평가

| 구현 | 한 줄 | 적합 시나리오 |
|------|-------|-------------|
| **구현 C** | 처리량 최고, 안정성 약함 | 대규모 트래픽 + 빠른 회전이 핵심 |
| **구현 A** | 균형형, 3,000 VU에서 레이턴시 급증 | 1,000 VU 이하 중규모 |
| **구현 B** | 안정성+응답 최고, 처리량 최저 | 안정성 우선 + Pod 스케일아웃으로 처리량 보완 가능 |

구현 선정은 **트래픽 규모와 운영 우선순위**에 따라 달라집니다.
처리량이 압도적으로 중요하다면 구현 C, 중규모 안정 운영이 목표라면 구현 A, 안정성과 낮은 서버 부하를 우선하고 Pod 스케일아웃으로 처리량을 보완할 수 있다면 구현 B가 적합합니다.

## 📚 배운 점

- **순수 큐 성능 비교는 티켓팅/결제 처리를 제외하고 측정해야 구현별 특성이 드러납니다.** 대기열 단독 벤치마크에서는 polling 간격·승격 트리거·자료구조 같은 큐 고유의 설계가 성능을 좌우합니다.
- **p50과 p95는 서로 다른 이야기를 합니다.** 구현 B는 중간값이 빠르지만 꼬리가 길고, 구현 C는 꼬리가 짧지만 전체 레이턴시가 높습니다. 단일 지표로 우열을 가리지 않아야 합니다.
- **처리량과 안정성은 동시에 얻기 어렵습니다.** 구현 C는 처리량 1위지만 통과율과 에러율이 가장 나쁘고, 구현 B는 안정성 1위지만 처리량이 1/16 수준입니다.
- **Redis polling 간격이 서버 부하를 결정합니다.** 동일 자료구조라도 polling 1초 대 5초는 VU당 Redis ops/sec를 3배 이상 차이 나게 합니다.
- **Lua script의 원자성은 강점이지만, 디버깅과 직렬화 이슈가 자주 발생합니다.** 구현 C의 버그 3건이 전부 Lua script 관련이었다는 점은 설계 선택 시 참고할 만합니다.
