## 🧭 포지셔닝 결정: 4단계 시도

포지셔닝 키워드를 확정하기까지 4단계를 거쳤습니다.

| 단계 | 후보 | 결정 |
|------|------|------|
| 1차 | "Internal AI Platform" + Cloudflare iMARS 비교 | **기각** — 본인 영역(K8s 한정)과 Cloudflare(사내 전체 dev productivity)의 레이어가 다름 |
| 2차 | "인프라 도메인 특화 AI Platform" | **기각** — 본인 정체성("DevOps · Infra Lead")과 채용 시장 키워드 불일치 |
| 3차 | "DevOps 워크플로우 자동화" | **기각** — 너무 광범위, 다시 generic으로 회귀 |
| **4차** | **"K8s 운영 DevOps"** | **채택** — 본인 무게중심, agents/skills 자산과 정렬, 시장에서 specific 키워드 |

### 기각 이유

**1차 탈락**: Cloudflare iMARS 사례는 회사 전체 dev productivity 플랫폼입니다. 본인 영역은 K8s 운영에 집중되어 있어 레이어가 달라 비교 자체가 어색합니다.

**2차 탈락**: "AI Platform"은 채용 시장에서 ML 플랫폼 엔지니어 키워드와 충돌합니다. 본인이 강조하고 싶은 건 인프라와 DevOps 운영 경험입니다.

**3차 탈락**: "DevOps 워크플로우 자동화"는 범위가 너무 넓어 CI/CD 전문가와 구별이 안 됩니다. K8s 운영 경험이 묻히는 문제가 있었습니다.

### 결정 기준

결정 기준은 다음 우선순위였습니다.

1. **본인 무게중심**: 실제로 가장 많은 시간을 쏟은 도메인인가
2. **assets 정렬**: agents/skills 자산이 이 키워드와 일치하는가
3. **시장 specific**: 채용 담당자가 "어떻게?"를 물어볼 수 있는 구체성인가

"K8s 운영 DevOps"가 세 기준을 모두 만족했습니다.

## 🧭 덱 순서 결정: D안 → B안 → B-final

포지셔닝 확정 이후에도 덱 순서를 두 번 더 변경했습니다.

### 1단계: D안 (Hook + Climax 분산)

처음에는 AI 4장을 덱 뒷부분에 배치하는 D안(Hook 1장 + Climax 3장 분산)을 제안했습니다.

이에 대해 핵심 우려가 제기됐습니다.

> "이거 끝까지 안볼수도 있다는게 중요해. 이사람이 AI도 잘쓰네!! 이런게 20페이지 넘어 있으면 그까지 안본다면 이 포트폴리오 자체가 무의미해져서 그래"

채용 담당자는 페이지당 약 5초만 봅니다. **2분 deck 사고**를 도입했습니다.

| 원칙 | 적용 |
|------|------|
| 앞 5장 = 자체 완결 deck | AI 4장 + About = 그것만 봐도 핵심 다 전달 |
| 본문은 깊이 보고 싶은 사람용 detail | 6번부터 안 봐도 핵심 메시지 보존 |
| 헤드라인이 본문 역할 | 5초만 봐도 메시지 전달 |

### 2단계: B안 (AI 4장 앞부분)

D안을 폐기하고 AI 4장을 앞부분(1-4번)으로 배치하는 B안으로 전환했습니다.

```text
01. AI · HERO (Before/After · 표지 겸함)
02. AI · CYCLE (원형 다이어그램)
03. AI · HARNESS 매핑
04. AI · META (포트폴리오 = 작동 증거)
────────────────────────────────────────
05. About (이력서식)
────────────────────────────────────────
06~21. 본문 (Overview · Architecture · M02 · M04 · M05 · M06)
22. Closing
23. Impact Summary
24. Contact
```

### 3단계: B-final (About을 1번으로)

B안 검토 중 추가 우려가 제기됐습니다.

- "AI 끝나고 6번이 어정쩡하다"
- "Goti 컨텍스트 없이 AI가 시작된다"

두 우려를 동시에 해결하는 B-final 순서로 확정했습니다.

```text
01. About (이력서식 + Goti 정보 보강)
02-05. AI 4장
06. Overview ("한 순간에 10만 명이 들어옵니다" — transition)
07-21. 본문
22-24. Conclusion
```

1번 About에서 "이 사람 누구 + 이 프로젝트 뭐"를 동시에 전달합니다. AI 4장이 그 뒤에 오면 누가 왜 이 시스템을 만들었는지 맥락이 잡힌 상태에서 보게 됩니다.

## 🧭 AI Hero 디자인: 옵션 5개 비교

AI 첫 번째 페이지 디자인은 검색 인사이트로 5개 옵션을 도출하고 그 중 하나를 채택했습니다.

| 옵션 | 유형 | 특징 |
|------|------|------|
| 1. Big Number | 데이터형 | 숫자 임팩트 우선 |
| **2. Before/After** | **사고 깊은 사람** | **← 채택** |
| 3. Single Statement | 단호형 | 한 문장으로 마무리 |
| 4. Provocation | 도발형 | "정말 AI를 쓰고 있나?" 질문 |
| 5. Visual Proof | 시각 우선 | 증거 화면 중심 |

**옵션 2 채택 사유**: 본인 정체성("실시간 운영 중인 사람")과 일치합니다. agents 레포 매주 갱신 = 현재진행형 운영 증거가 있기 때문에 Before/After 대비가 가장 진정성 있게 보입니다.

디자인 핵심 요소입니다.

| 요소 | 내용 |
|------|------|
| 헤드라인 | "AI를 어디까지 운영할 수 있을까요?" (질문형) |
| 좌 카드 (BEFORE) | 쓴 사람 · one-shot · 매번 재학습 · 같은 실수 반복 (회색 처리) |
| 우 카드 (AFTER) | 운영한 사람 · cycle · sensors · self-correcting (브랜드 강조 + shadow-pop) |
| 중앙 화살표 | 대비를 시각화 |
| 하단 live signal | 실시간 운영 중 — agents 레포 매주 사이클로 갱신 |

## 📚 배운 점

**메모리를 먼저 읽고 시작해야 합니다.** 1차 제안에서 "AI를 도구가 아닌 환경으로" 슬로건 + Skills 카운트 카드가 메모리 원칙 위반이었습니다. 사용자가 직접 지적해줘서 발견했습니다. 메모리 단순 저장이 아니라 작업 시작 전 명시적으로 체크하는 습관이 필요합니다.

**외부 검색이 어필 강화에 결정적입니다.** "Cloudflare iMARS" 사례, "Skill = 1 job 원칙", "2026 채용 시그널" 모두 검색으로 발굴했습니다. 본인 직감("AI 카운트는 누구나 가능")이 외부 가이드와 정확히 일치함을 확인했습니다. 검색이 자기 직감의 근거를 제공합니다.

**포지셔닝은 좁혀가는 과정입니다.** 처음부터 정확한 답을 찾으려 하면 1차에서 막힙니다. 사용자 직감을 기준으로 단계마다 우려를 받아 좁혀가는 게 결과적으로 빠릅니다.

**"끝까지 안 본다"는 직감이 narrative arc 설계의 정답입니다.** 이론상 D안(분산)이 정교하지만, 현실적으로 차별화 메시지가 앞부분에 없으면 채용 담당자에게 닿지 않습니다. 사용자 우려를 추상 이론보다 우선시해야 합니다.

**"슬라이드 = 면접관 prompt" 사고가 워딩 점검의 기준입니다.** 헤드라인 작성 시 "이걸 본 면접관이 뭐 묻고 싶어할까?"를 1차 질문으로 삼습니다. 답변 거리가 본문에 있으면 OK, 없으면 워딩을 다시 씁니다.

**본인이 만든 비표준 표현은 면접관에게 통하지 않습니다.** "환원"은 본인 메타 키워드였지만 외부 검색 결과 IT 산업 용어가 아닙니다. 본인 머릿속에서 자연스러운 표현이 외부에서는 외계어가 될 수 있습니다. 면접관 관점 검증이 필수입니다.

**알파벳 순 빌드 함정에 주의해야 합니다.** `19-contact.html`이 새 `20-ai-*.html` 앞에 위치해 캡처 순서가 틀렸습니다. 파일명 prefix가 빌드 순서를 결정하는 시스템에서는 파일 추가/제거 시 항상 정렬 순서를 검증해야 합니다.
