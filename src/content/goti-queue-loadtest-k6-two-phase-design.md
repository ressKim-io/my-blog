## 한 줄 요약

> 대기열 구현체 3종을 비교하기 위해 K6 부하테스트 스크립트를 설계했습니다. E2E는 사람별 파일로, 비교 시나리오는 `QUEUE_IMPL` 환경변수 패턴으로 분리한 2-Phase 구조입니다.

---

## 배경

goti-server에 3명의 팀원이 각자 다른 방식으로 대기열을 구현했습니다.
PR #309, #311, #312가 순서대로 올라왔고, 이튿날 오전 부하테스트가 예정되어 있었습니다.

부하테스트의 목적은 두 가지였습니다.

1. **E2E 검증**: 각 구현체가 대기열 → 예매 → 결제 전체 플로우에서 정상 동작하는지 확인합니다.
2. **성능 비교**: 구현체 간 순수 대기열 처리량, Redis 명령 분포, TTL 회수 속도 차이를 측정합니다.

두 목적은 성격이 다릅니다.
E2E는 각 구현체 고유의 API 차이를 반영해야 하고, 비교 시나리오는 구현체만 바꿔가며 같은 부하를 줘야 공정한 비교가 됩니다.
이 차이를 어떻게 코드 구조에 반영할지가 설계의 핵심이었습니다.

---

## 🤔 선택지 비교

### Option A: 사람별 × 시나리오별 개별 파일 (9+6=15파일)

사람 3명과 시나리오 여러 종을 곱해 모든 조합을 개별 파일로 만드는 방식입니다.

**장점**: 파일마다 독립적이라 한쪽을 수정해도 다른 파일에 영향이 없습니다.

**단점**: 시나리오 로직이 거의 동일하고 import만 달라지는 탓에, 코드 중복이 극심해집니다.
파일 수도 15개까지 늘어나 관리 부담이 큽니다.

### Option B: 시나리오 타입별 1파일 + `QUEUE_IMPL` 환경변수 (3+6=9파일)

시나리오 종류마다 파일을 하나씩 두고, 환경변수로 어떤 구현체를 쓸지 선택하는 방식입니다.

**장점**: 코드 중복이 최소화됩니다. 3개 파일로 9개 조합을 커버할 수 있습니다.

**단점**: K6는 static import만 지원합니다.
따라서 비교 시나리오에서 구현체 3개 헬퍼를 모두 import해야 하고, 초기화 오버헤드가 약간 발생합니다.

### Option C: E2E는 사람별 파일 + 비교 시나리오만 `QUEUE_IMPL` 패턴 (혼합)

E2E는 사람별 API 차이가 커서 파일을 나누고, 비교 시나리오는 공정성을 위해 `QUEUE_IMPL` 환경변수로 묶는 혼합 방식입니다.

**장점**: E2E에서는 PR별 API 특성을 자유롭게 커스터마이징할 수 있고, 비교 시나리오에서는 중복이 제거됩니다.

**단점**: 두 가지 패턴이 한 디렉터리에 공존합니다.

---

## ✅ 결정: Option C 채택

E2E는 사람별 파일로, 비교 시나리오는 `QUEUE_IMPL` 패턴으로 만드는 혼합 구조를 선택했습니다.

E2E와 비교 시나리오는 목적 자체가 다릅니다.
E2E는 "이 구현체가 실제로 돌아가는가"를 검증하고, 비교 시나리오는 "같은 조건에서 어느 쪽이 빠른가"를 측정합니다.
목적이 다른 두 가지를 하나의 패턴으로 묶으려 하면 어느 쪽이든 불편해집니다.
Option C는 이 불편함을 감수하지 않고 각 목적에 맞는 구조를 택한 결과입니다.

## Phase 1 — E2E 대기열 스크립트

### 공통 인터페이스

사람별로 API는 다르지만, 부하테스트 관점에서 대기열 흐름은 `enterQueue() → waitForAdmission() → enterSeat() → [ticketing] → leaveQueue()` 네 단계로 추상화할 수 있습니다.

각 헬퍼는 이 인터페이스를 구현하면서, 내부에서 PR별 API 차이를 흡수합니다.

### PR별 API 차이

3개 PR은 같은 대기열 개념을 상당히 다르게 풀었습니다.

- **POC C (#309)**: `POST /queue/enter`로 진입하고, 응답의 `queueNumber`가 서버의 `publishedRank` 이하가 되면 통과로 판단합니다.
- **기여자 (#311)**: `POST /queues/enter?gameId=`로 진입하고 heartbeat를 병행합니다. `secureToken` 기반으로 통과 여부를 확인합니다.
- **POC A (#312)**: `POST /queue/validate`로 검증합니다. `enterSeat`은 Interceptor가 처리하므로 no-op, `leaveQueue`는 Spring Event로 처리되므로 no-op입니다.

헬퍼 안에서 이 차이를 모두 흡수하면, 시나리오 쪽 코드는 세 구현체 모두 동일한 모양이 됩니다.

## Phase 2 — 비교 시나리오 3종

비교 시나리오는 `QUEUE_IMPL` 환경변수 하나로 구현체를 스위치합니다.
스크립트 초기화 시 세 헬퍼를 모두 import하고, 환경변수 값에 따라 실제 호출할 함수만 바꿔치기하는 구조입니다.

### 3개 시나리오

- **`spike-queue.js`** — 5000 VU 스파이크를 한꺼번에 넣어 순수 대기열 처리 한계를 봅니다.
- **`queue-only.js`** — 순수 대기열 처리량 측정과 50% TTL 회수 테스트를 겸합니다.
- **`soldout-queue.js`** — 매진 시뮬레이션으로, 대기열 생명주기 전체를 재현합니다.

### 핵심 환경변수: `POLL_INTERVAL`

폴링 간격은 Redis 부하 차이를 가장 크게 벌리는 축입니다.
하드코딩 1초로 두면 구현체 간 차이를 제대로 드러낼 수 없습니다.

그래서 `POLL_INTERVAL` 환경변수로 폴링 간격을 조절할 수 있게 만들었습니다.
기본값은 1초이고, 0.5초에서 2초까지 범위에서 움직이며 Redis의 SCAN과 O(1) 접근 차이를 극명하게 드러내는 것이 목적입니다.

## run.sh 확장

실행 스크립트에는 다음을 추가했습니다.

- **`help` 명령**: 시나리오별 상세 설명을 출력합니다. 다음 날 아침 부하테스트 직전에 팀원들이 어떤 시나리오가 무엇을 측정하는지 빠르게 확인할 수 있어야 합니다.
- **9개 case 추가**: `spike/queueonly/soldout` 3종 × POC C / 기여자 / POC A 3명 조합입니다.
- **환경변수 전달**: `QUEUE_IMPL`, `POLL_INTERVAL`, `SOLDOUT_AT_SEC`를 K6 프로세스에 전달합니다.

## 최종 파일 구조

```text
load-tests/
  helpers/
    queue-poc-c.js       # PR #309 대기열 액션
    queue-contributor.js # PR #311 대기열 액션
    queue-poc-a.js       # PR #312 대기열 액션
  scenarios/
    queue-poc-c.js       # E2E (대기열→예매→결제)
    queue-contributor.js # E2E
    queue-poc-a.js       # E2E
    spike-queue.js       # 비교: 스파이크 (QUEUE_IMPL 패턴)
    queue-only.js        # 비교: 대기열만 (QUEUE_IMPL 패턴)
    soldout-queue.js     # 비교: 매진 (QUEUE_IMPL 패턴)
  run.sh                 # help 명령 + 18개 시나리오 지원
```

helpers는 사람별 3개, scenarios는 E2E 3개 + 비교 3개로 6개입니다.
E2E는 사람별 파일 이름을 그대로 두고, 비교 시나리오는 시나리오 성격을 파일명에 담았습니다.
이렇게 하면 파일 이름만 봐도 어떤 패턴인지 바로 구분할 수 있습니다.

## 실행 방법

E2E는 사람별 명령으로 실행합니다.

```bash
# E2E (사람별)
./load-tests/run.sh queue-poc-c
```

비교 시나리오는 `QUEUE_IMPL`이 자동으로 설정되는 명령을 씁니다.
환경변수로 VU 수와 폴링 간격을 조정할 수 있습니다.

```bash
# 비교 시나리오 (QUEUE_IMPL 자동 설정)
./load-tests/run.sh spike-poc-c
VUS=2000 POLL_INTERVAL=0.5 ./load-tests/run.sh queueonly-poc-a
```

## 제약 사항

이 설계를 쓰면서 감수한 제약이 몇 가지 있습니다.

- **K6 static import 제약**: 비교 시나리오에서 헬퍼 3개를 모두 import해야 합니다. 실제 실행 시 한 가지만 쓰더라도 초기화 오버헤드가 발생합니다.
- **매진 시나리오**: 매진 신호 자체는 외부에서 수동으로 트리거해야 합니다. K6는 측정만 담당합니다.
- **API 필드명은 PR diff 기반 추정**: 테스트 전에 curl로 실제 응답 형태를 한 번 더 검증해야 합니다.

## 후속 작업

이번 설계에서 제외한 것들은 별도 Phase로 넘겼습니다.

- **goti-load-observer(metrics-collector) 모니터링 연동**: Redis 메트릭을 수집하는 별도 Phase로 분리합니다.
- **redis-cli commandstats before/after diff**: 자동화 없이 수동 측정으로도 충분합니다.
- **대기열 진입 → 통과만 하는 단축 시나리오**: 필요해지면 추후 추가합니다.

## 📚 배운 점

### 목적이 다르면 구조를 섞지 않습니다

E2E와 비교 시나리오를 한 패턴으로 통합하려 했다면 어느 쪽이든 불편한 코드가 나왔을 것입니다.
사람별 파일은 E2E의 커스터마이징을 살리고, `QUEUE_IMPL` 패턴은 비교의 공정성을 살립니다.
혼합 구조의 단점(두 패턴이 공존)은 파일 이름 규칙으로 충분히 식별 가능합니다.

### 비교 축이 되는 변수는 환경변수로 뺍니다

`POLL_INTERVAL`을 하드코딩했다면 Redis 부하 차이를 제대로 측정하지 못했을 것입니다.
성능 비교의 핵심 축이 되는 값은 반드시 런타임에 바꿀 수 있어야 합니다.

### K6 static import 제약은 설계 시점에 인지해야 합니다

K6는 dynamic import를 지원하지 않습니다.
비교 시나리오처럼 "런타임에 구현체를 바꿔 끼우는" 패턴에서는 모든 후보를 미리 import해야 하고, 이 오버헤드가 스크립트 실행에 영향을 줍니다.
설계 시점에 이 제약을 알고 있어야 파일 구조 선택이 엇나가지 않습니다.

### 부하테스트 직전에는 `help` 명령이 설명서보다 빠릅니다

시나리오가 18개까지 늘어나면 README를 찾는 것보다 `./run.sh help`가 빠릅니다.
실행 스크립트 자체에 설명을 담아두면 바쁜 시점에 오류를 줄일 수 있습니다.
