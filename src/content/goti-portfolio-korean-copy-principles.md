## 한 줄 요약

> 포트폴리오 슬라이드 24장을 "PDF 단독 전달" 기준으로 점검했을 때 드러난 영어 직역체 문제를 7개 한국어 카피 원칙으로 정리하고, 5개 슬라이드 카피와 슬라이드 순서를 재정비했습니다.

## 🔥 문제: PDF로만 보면 어색한 카피 4가지

포트폴리오 컨설팅 Round 2 이틀째, 발표와 구두 보충이 아니라 **PDF만으로 면접관에게 전달되는 케이스**를 가정하고 24장 슬라이드를 다시 읽었습니다.

발견된 문제는 크게 네 가지였습니다.

**문제 1 — AI 4장이 뜬금없이 등장합니다.**

슬라이드 순서가 `About → AI 4장 → Overview → 본문` 이었습니다. 구두 설명 없이 PDF만 보는 독자에게는 Goti가 어떤 프로젝트인지 알기도 전에 AI hero 슬라이드가 나타납니다. "왜 갑자기 AI?"라는 의문이 생깁니다.

**문제 2 — 단순 양적 자랑 카피입니다.**

"대시보드 37개 / 패널 380개" 같은 숫자만 박힌 카피가 있었습니다. 비기술자 독자는 물론 기술 면접관도 "그래서 어떤 문제를 해결했나?"가 바로 와닿지 않습니다.

**문제 3 — 한국어가 영어 직역체입니다.**

명사화 과다, 한자어 동사 남발, 무생물 주어 직역, "진짜 사건" 같은 비교 암시 강조어가 섞여 있었습니다.

**문제 4 — 헤드라인과 본문이 모순됩니다.**

"메트릭·로그·트레이스·수집기, 각각 따로 골랐습니다"가 헤드라인인 슬라이드에서 본문은 Exemplar 체인, 쿼리 통일, OTel 표준 준수를 강조하고 있었습니다. 헤드와 본문이 반대 메시지를 냅니다.

## 🤔 원인: 한국어 카피 원칙 7가지

외부 테크니컬 라이팅 가이드([natural-kor-expression](https://technical-writing.dev/sentence/natural-kor-expression.html))와 이번 세션에서 식별한 패턴을 정리했습니다.

### 원칙 1. 명사화 → 동사화

영어는 동사를 명사형(`-tion`, `-ing`)으로 바꿔 쓰는 것이 자연스럽지만, **한국어는 동사를 그대로 쓰는 것이 자연스럽습니다.**

| 영어식 명사형 | 한국어 동사형 |
|---|---|
| API 키를 이용한 사용자 인증 처리가 완료된 후 | API 키로 사용자를 인증한 후 |
| CDN이 이를 Origin 도달 전에 흡수한 효과입니다 | Origin 도달 전에 흡수되는 것을 확인했습니다 |
| 관측이 진짜 사건 4건을 잡았습니다 | 운영 중 4건을 대시보드가 먼저 알려줬습니다 |

명사형으로 끝나는 문장은 서술어가 사라져 독자에게 "이 문장이 결론인지 근거인지"를 모호하게 만듭니다.

### 원칙 2. 한자어 동사 제거

"수행합니다", "실행합니다", "진행합니다"는 의미 없는 한자어 동사입니다. 뒤에 따라오는 동작 동사로 바로 연결하면 더 간결합니다.

| 한자어 동사 포함 | 간결한 표현 |
|---|---|
| 로그 파일을 삭제하는 작업을 수행합니다 | 로그 파일을 삭제합니다 |
| 모니터링을 실행하여 시스템을 측정합니다 | 모니터링으로 시스템을 측정합니다 |

### 원칙 3. 무생물 주어 회피

영어 "monitoring caught X" 같은 무생물 주어 직역은 한국어에서 어색합니다. 도구를 주어로 쓰거나, 도구를 수단으로 표현하는 것이 자연스럽습니다.

| 무생물 주어 직역 | 도구로 표현 |
|---|---|
| 관측이 사건을 잡았습니다 | 대시보드로 운영 중 4건을 잡았습니다 |
| 시스템이 결정합니다 | 시스템에서 결정합니다 / 시스템 덕에 결정됐습니다 |

### 원칙 4. 비교 암시 강조어 회피

"진짜 X"라고 하면 자동으로 "가짜 X"라는 비교 대상을 암시합니다. 독자 입장에서는 "가짜 사건이 뭔가?"라는 의문이 생겨 어색합니다.

| 비교 암시 표현 | 대안 |
|---|---|
| 진짜 사건 4건을 잡았습니다 | 운영 중 마주친 문제 4건 |
| 실제 운영 사례 | 운영 중 발견한 사례 / 실 운영 사례 |
| 진짜 효과가 있었습니다 | 효과가 있었습니다 (강조어 제거) |

### 원칙 5. 격식·모호 단어 → 구체 단어

"사건", "처리", "운영"은 맥락에 따라 의미가 달라지는 모호한 단어입니다. 시스템 장애 맥락에서는 더 구체적인 단어로 바꾸면 독자가 한 번에 이해합니다.

| 모호한 표현 | 구체적인 표현 |
|---|---|
| 사건 | 문제 / 이상 / 장애 / 이슈 |
| 처리 | 발견 / 해결 / 막음 |
| 운영 | 측정 / 관측 / 감시 (맥락에 따라) |

### 원칙 6. 헤드라인은 본문이 증명하는 것을 요약

헤드라인이 본문 내용과 모순되면 독자가 혼란을 느낍니다. 본문에서 실제로 보여주는 내용이 헤드라인에 먼저 나와야 합니다.

슬라이드 18번이 전형적인 사례였습니다. 헤드라인은 "각각 따로 골랐습니다"인데 본문은 OTel 통합 강조였습니다. "컴포넌트는 따로, 시스템은 OTel 답게 하나입니다"로 바꾸면 헤드 안에 분리와 통합 두 메시지를 모두 담을 수 있습니다.

### 원칙 7. 도메인 시작 → 기술 결과로 (포트폴리오 한정)

기술 수치만 박힌 카피는 비기술자에게 "그래서 뭐?"가 됩니다. **도메인 맥락 → 설계 의도 → 검증 결과** 순서로 흐르는 것이 자연스럽습니다.

> 이전: "588K 요청 중 390K (66%)가 Edge에서 처리되었습니다 — polling으로 인한 요청 밀집도의 증거이자, CDN이 Origin 도달 전에 흡수한 효과입니다."
>
> 이후: "티켓팅에서 사용자가 서버로 가장 많이 보내는 요청은 **대기열에서 발생하는 polling**입니다. 이를 CDN Edge에서 받아내도록 설계했고, **588K 요청 중 97%가 Origin 도달 전에 흡수**되는 것을 확인했습니다."

"이후" 문장은 독자에게 도메인 맥락(티켓팅 polling)을 먼저 주고, 설계 의도("받아내도록 설계했고")를 명시하고, 검증 수치(97%)로 마무리합니다.

## ✅ 해결: 5개 슬라이드 + 순서 재정비

### 1. 슬라이드 순서 재배치

| 이전 순서 | 이후 순서 |
|---|---|
| 01 About → 02~05 AI 4장 → 06 Overview → 본문 | 01 About → 02 Overview → 03~06 AI 4장 → 본문 |

Overview를 앞으로 당겼습니다. "Goti가 무슨 프로젝트인지" → "그걸 어떻게 만들었는지(AI)" → "구체 모듈" 흐름이 PDF 단독 독자에게 자연스럽습니다.

파일명 prefix를 변경(`02-overview.html`로 이동, 이전 02~05가 03~06으로 시프트)했습니다. 빌드 스크립트(`capture.mjs`)가 디렉토리 정렬 순서로 PPTX를 생성하므로 파일명 변경만으로 순서가 자동 갱신됩니다.

### 2. AI 4장 — 소속 framing 변경

AI 4장이 어색하게 느껴진 진짜 원인은 AI가 "별도 메타 토픽"처럼 보였기 때문이었습니다. eyebrow에 `GOTI에서의 AI` prefix를 통일하니 위치 고민 자체가 해소됐습니다.

| 위치 | 이전 | 이후 |
|---|---|---|
| Hero 헤드라인 | AI를 어디까지 운영할 수 있을까요? | **Goti에서 AI를 이렇게 운영했습니다** |
| Hero eyebrow | AI · K8s 운영 자동화 | **GOTI에서의 AI · 운영 방식의 차별화** |
| AI Cycle eyebrow | AI · 자기 개선 사이클 | **GOTI에서의 AI · 자기 개선 사이클** |
| AI Harness eyebrow | AI · 산업 framework 매핑 | **GOTI에서의 AI · 표준보다 먼저** |
| AI Meta eyebrow | DEVOPS 운영 · 시스템은 도구로 활용 | **GOTI에서의 AI · 운영 영역 정의** |
| index.html 섹션 라벨 | ▸ AI · 차별화 | **▸ GOTI에서의 AI 활용** |

Hero 폰트 크기도 84px → 64px로 조정했습니다. 이전 짧은 카피 기준으로 설정된 크기라 새 카피에서는 너무 컸습니다.

### 3. 슬라이드 02 (Overview) — 모니터링 카드

단순 양적 수치(37 / 380)를 의미 영역으로 교체했습니다.

| 항목 | 이전 | 이후 |
|---|---|---|
| 제목 | 모니터링 직접 구축 | **비즈니스 + 스파이크 관측** |
| 설명 | LGTM을 우리 시스템에 맞춰 직접 구축... 오진을 막는 대시보드 37종·패널 380개 | **야구 티켓팅에 맞춘 대시보드를 직접 만들었습니다. 실시간 대기열 · 분당 판매 · 스파이크 RPS를 한눈에 봅니다.** |
| 강조 메트릭 | 37 / 380 | **4건** (운영 중 먼저 발견한 문제) |

다른 카드(97.7%, 30s)와 단위가 다양해져 시각적 차별성도 함께 확보됩니다.

### 4. 슬라이드 13 (CDN 검증)

원칙 7 (도메인 시작 → 기술 결과)을 가장 직접적으로 적용한 슬라이드입니다.

**이전**

```text
588K 요청 중 390K (66%)가 Edge에서 처리되었습니다 — 클라이언트 polling으로 인한 요청 밀집도의 증거이자, CDN이 이를 Origin 도달 전에 흡수한 효과입니다.
```

**이후**

```text
티켓팅에서 사용자가 서버로 가장 많이 보내는 요청은 대기열에서 발생하는 polling입니다. 이를 CDN Edge에서 받아내도록 설계했고, 588K 요청 중 97%가 Origin 도달 전에 흡수되는 것을 확인했습니다.
```

동시에 수치 정합성도 수정했습니다. 이전 본문 66%는 헤드라인 97.7%와 충돌하는 수치였습니다. 이후 본문 97%로 맞춰 헤드라인과 일치시켰습니다.

### 5. 슬라이드 18 (LGTM 선정)

원칙 6 (헤드라인 ↔ 본문 일관성)을 적용했습니다.

| 위치 | 이전 | 이후 |
|---|---|---|
| eyebrow | 선정 · 4개 컴포넌트 독립 평가 | **선정 · 컴포넌트는 따로, 운영은 OTel 답게** |
| 헤드라인 | 메트릭 · 로그 · 트레이스 · 수집기, 각각 따로 골랐습니다 | **컴포넌트는 따로, 시스템은 OTel 답게 하나입니다** |
| 서브헤드 | (헤드라인 풀어쓰기 반복) | **삭제** |

서브헤드는 헤드라인의 풀어쓰기에 불과하고 정보 추가가 없어 제거했습니다. 대조 구조 "컴포넌트는 따로 ↔ 시스템은 하나"로 분리와 통합을 동시에 담았습니다.

### 6. 슬라이드 21 (모니터링 검증)

원칙 3 (무생물 주어 회피)과 원칙 4 (강조어 회피)를 적용했습니다.

| 항목 | 이전 | 이후 |
|---|---|---|
| 헤드라인 | 대시보드가 진짜 사건 4건을 잡았습니다 | **운영 중 마주친 문제 4건, 대시보드가 먼저 알려줬습니다** |
| stats 좌 | `37` 대시보드 | **비즈니스 — 실시간 대기열 · 분당 티켓 판매 · 매진까지** |
| stats 우 | `380` 패널 | **스파이크 — 순간 RPS · Pod 스케일 추이 · 큐 대기 깊이** |

"관측이 잡았다"(무생물 주어)를 "대시보드가 알려줬다"(도구 주어)로, "진짜 사건"을 "마주친 문제"로 바꿨습니다. 37/380 단순 수량도 비즈니스/스파이크 두 영역 카테고리로 교체했습니다.

### PPTX 빌드

```bash
# stale PNG 정리 후 전체 빌드
rm -f slides/build/out/*.png && cd slides && npm run all
```

파일명 변경(02-ai-hero → 03-ai-hero 등)으로 이전 캡처 PNG가 stale 상태로 남습니다. `make-pptx.mjs`가 `out/` 디렉토리를 정렬 순서로 PPTX를 만들기 때문에, stale 파일이 섞이면 잘못된 슬라이드가 들어갑니다. 파일명을 바꿀 때는 반드시 `out/` 디렉토리를 먼저 비워야 합니다.

## 📚 배운 점

- **명사화 문장은 결론이 흐릿합니다** — 동사형으로 바꾸면 독자가 "이 문장이 무엇을 말하는지" 바로 파악합니다
- **무생물 주어는 설계 의도를 지웁니다** — "CDN이 흡수했다"보다 "CDN Edge에서 받아내도록 설계했다"가 작성자의 의도를 명확히 전달합니다
- **수치보다 영역이 더 기억에 남습니다** — "37개 대시보드"보다 "비즈니스/스파이크 두 영역"이 독자에게 무엇을 만들었는지 더 잘 전달합니다
- **헤드라인 ↔ 본문 일관성은 슬라이드 단위로 점검해야 합니다** — 각 슬라이드를 독립적으로 읽었을 때 헤드라인과 본문이 같은 방향을 가리키는지 확인합니다
- **"진짜/실제" 강조어는 비교 대상이 없을 때 제거하는 것이 낫습니다** — 강조어가 없어도 사실 자체로 충분히 설득됩니다
- **eyebrow prefix 통일이 섹션 소속 문제를 해결합니다** — AI 4장처럼 "어디에 놓아도 어색한 슬라이드"는 소속 표시가 없어서인 경우가 많습니다
