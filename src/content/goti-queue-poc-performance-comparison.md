## 한 줄 요약

> 동일한 경기·동일한 인프라에서 3개 POC(방식1/방식2/방식3)의 대기열 성능을 5가지 축으로 비교했습니다. 한 명이 모든 축에서 1위가 아니었고, **각자 강점이 다른 트레이드오프 구조**였습니다

---

## 🔥 배경: 왜 3개 POC를 한 자리에서 비교해야 했는가

### 프로젝트 처리량 목표

go-ti는 KBO 야구 경기 예매를 시뮬레이션하는 티켓팅 시스템입니다
실제 KBO 인기 경기는 오픈 직후 수만 명이 동시에 몰리는 패턴이 일반적이고, 우리는 단일 경기 기준으로 다음과 같은 목표를 잡았습니다

- **동시 접속**: 3,000 VU 이상에서 대기열이 무너지지 않을 것
- **통과율**: 대기열에 들어온 사용자가 좌석 선택 단계까지 100%에 가깝게 도달할 것
- **응답 시간**: status 폴링 p95가 1초 이내일 것
- **에러율**: 부하가 올라가도 5xx 에러가 거의 없을 것

이 목표를 정해놓고 봐야 비교 결과가 의미를 가집니다
숫자만 나열하면 "어느 게 빠르네/느리네" 정도로 끝나지만, 목표 대비로 보면 **어느 구현이 우리가 풀려는 문제에 부합하는지** 판단할 수 있습니다

### 세 명이 다른 방식으로 구현한 이유

대기열 POC는 백엔드 세 명이 각자 다른 방식으로 구현했습니다
이게 우연이 아니라, 다음과 같은 의도가 있었습니다

- **응답 속도를 우선시한 사람** → 이벤트 기반 승격 (방식2/POC B): 사용자가 status 폴링할 때 서버 계산을 최소화하고, 승격은 백그라운드 이벤트로 처리해 응답을 가볍게 만드는 전략
- **단순함과 빠른 전환을 우선시한 사람** → SCAN 폴링 후 O(1) 개선 (방식1/POC A): 자료구조를 자주 갈아끼우면서 점진적으로 성능을 끌어올리는 전략
- **처리량을 우선시한 사람** → 동적 계산 (방식3/POC C): status 호출 시 즉시 승격까지 마무리해 슬롯 회전을 최대화하는 전략

세 방식이 모두 작동은 하지만, **"무엇을 희생하고 무엇을 얻는가"가 명백히 다릅니다**
그 차이를 정성적 토론이 아니라 같은 부하 조건의 수치로 확인하기 위해 이번 비교가 필요했습니다

### 이 비교의 목적

이 비교는 다음 의사결정을 위한 입력입니다

1. 실제 운영용으로 어느 구현체를 채택할지
2. 채택된 구현체의 어떤 약점을 후속 작업으로 보완할지 (CDN 캐싱, 인프라 튜닝 등)
3. 채택되지 않은 구현체의 강점 중 흡수할 만한 것이 있는지

각 구현은 개별 포화 테스트로 한계를 먼저 확인한 뒤, 이번에는 **같은 조건에서 한꺼번에 돌려 동일 축으로 비교**해야 했습니다

### 테스트 조건

| 항목 | 값 |
|------|----|
| 테스트일 | 2026-03-30 |
| maxCapacity | 100 |
| 경기 | 2026-04-01 KIA vs 두산 (동일) |
| 인프라 | Kind 7노드 (동일) |
| VU | 300 / 1000 / 3000 |
| CDN 캐시 | **미적용** |

CDN 캐시를 적용하지 않은 이유는, 티켓팅 API 레이턴시가 CDN 적용 후 크게 바뀌기 때문입니다
**이 비교에서 보고 싶은 건 백엔드 자체의 성능**이지 CDN을 끼고 본 가공된 수치가 아닙니다
CDN 적용 후의 결과는 다음 단계(part3 selection)에서 별도로 다룹니다

### 비교 대상

- **방식1 (POC A)**: Redis SCAN 기반 폴링 → O(1) 개선
- **방식2 (POC B)**: 이벤트 기반 승격 (leave → 분산락 → allowedNum)
- **방식3 (POC C)**: status API 호출 시 동적 계산

각 구현의 포화 테스트 결과는 앞선 S5 글에서 다뤘고, 이 글은 **셋을 한 표에 놓고 비교**하는 글입니다

## 🤔 비교 1: 대기열 API 레이턴시

### 1000 VU

| API | 방식3 | 방식1 | 방식2 | 최고 |
|-----|-------|-------|-------|------|
| queue_enter avg | 1.52s | - | **268ms** | 방식2 |
| queue_enter p95 | - | - | **293ms** | 방식2 |
| queue_status avg | 942ms | - | **267ms** | 방식2 |
| queue_status p95 | 2.18s | 361ms | **279ms** | 방식2 |
| queue_seat_enter avg | 1.27s | - | **269ms** | 방식2 |
| queue_seat_enter p95 | - | 452ms | **295ms** | 방식2 |
| http_req p95 (전체) | 2.18s | 390ms | **282ms** | 방식2 |

### 3000 VU

| API | 방식3 | 방식1 | 방식2 | 최고 |
|-----|-------|-------|-------|------|
| queue_enter avg | 1.53s | - | **350ms** | 방식2 |
| queue_enter p95 | - | - | **917ms** | 방식2 |
| queue_status avg | 1.55s | 2.42s | **275ms** | 방식2 |
| queue_status p95 | 2.13s | 3.10s | **500ms** | 방식2 |
| queue_seat_enter avg | 1.68s | - | **294ms** | 방식2 |
| queue_seat_enter p95 | 2.29s | 2.99s | **562ms** | 방식2 |
| http_req p95 (전체) | 2.13s | 3.10s | **501ms** | 방식2 |

숫자를 나란히 놓고 보면, **방식2가 모든 API 레이턴시에서 1위**입니다

1000 VU에서 http_req p95는 방식2가 282ms, 방식1이 390ms, 방식3이 2.18s입니다
방식3은 방식2 대비 약 7.7배 느립니다

3000 VU로 부하를 올려도 방식2는 queue_status p95를 500ms로 유지합니다
같은 지표에서 방식1은 3.10s, 방식3은 2.13s로 급격히 늘어납니다
방식2의 **스케일링 내성**이 가장 뚜렷하게 드러나는 지점입니다

목표("status p95 < 1초")와 대조하면 **3000 VU에서 이 기준을 만족하는 건 방식2뿐**입니다

## 🤔 비교 2: 대기열 통과율

| VU | 방식3 | 방식1 | 방식2 |
|----|-------|-------|-------|
| 300 | - | 99.26% | **100%** |
| 1000 | 81.56% | **100%** | **100%** |
| 3000 | 57.90% | **100%** | **100%** |

통과율은 대기열에 들어온 사용자 중 좌석 선택까지 도달한 비율입니다

방식1과 방식2는 1000 VU와 3000 VU에서 모두 **100% 통과**입니다
대기열이 길어져도 모든 사용자가 최종적으로 승격되어 좌석 선택 단계까지 갔다는 의미입니다

반면 방식3은 VU가 커질수록 통과율이 떨어집니다
1000 VU에서 81.56%, 3000 VU에서 57.90%로 하락합니다
3000 VU에서는 대기열에 들어온 사용자의 42%가 좌석 선택까지 가지 못한 셈입니다

이 차이는 승격 메커니즘에서 옵니다
방식3은 status API 호출 시점에 동적으로 승격 여부를 계산합니다
부하가 높아지면 status 자체가 느려지고, 그 사이에 사용자가 대기열에서 이탈하거나 타임아웃에 걸려 통과율이 떨어집니다

목표("통과율 100% 근접") 기준으로는 **방식1과 방식2가 통과**, **방식3은 3000 VU에서 명백한 미달**입니다

## 🤔 비교 3: 대기 시간

대기 시간은 사용자가 대기열에 들어온 뒤 좌석 선택 단계로 승격되기까지 걸린 시간입니다
중간값(p50)과 꼬리(p95) 둘 다 봐야 합니다

### 1000 VU

| 지표 | 방식3 | 방식1 | 방식2 | 최고 |
|------|-------|-------|-------|------|
| queue_wait avg | 56.4s | 67.7s | **40.2s** | 방식2 |
| queue_wait p50 | 56.3s | 66.4s | **11.7s** | 방식2 |
| queue_wait p95 | **1m17s** | 2m16s | 2m54s | 방식3 |
| queue_poll avg | **27회** | 52.2회 | 7.3회 | 방식2 |

### 3000 VU

| 지표 | 방식3 | 방식1 | 방식2 | 최고 |
|------|-------|-------|-------|------|
| queue_wait avg | 1m55s | 53.5s | **1m48s** | 방식1 |
| queue_wait p50 | 2m19s | 53s | **23s** | 방식2 |
| queue_wait p95 | **2m54s** | 2m31s | 4m48s | 방식1 |
| queue_poll avg | 42.5회 | 17.4회 | **19.6회** | 방식1 |

p50과 p95를 분리해서 봐야 해석이 정확합니다

**p50 (중간값)**: 1000 VU와 3000 VU 모두 방식2가 압도적으로 빠릅니다
1000 VU에서 방식2는 11.7초, 방식3은 56.3초, 방식1은 66.4초입니다
3000 VU에서도 방식2는 23초, 방식1은 53초, 방식3은 2분 19초입니다
**대부분의 사용자가 빠르게 통과**한다는 의미입니다

**p95 (꼬리)**: 반대로 뒤쪽 대기자 기준으로는 방식2가 가장 느립니다
1000 VU에서 방식2 p95는 2m54s, 방식3은 1m17s, 방식1은 2m16s입니다
3000 VU에서도 방식2 p95는 4m48s로 가장 깁니다

해석하자면, 방식2의 이벤트 기반 승격은 **초반 사용자를 빠르게 처리하지만 대기열이 길어질수록 꼬리가 늘어나는** 구조입니다
반면 방식3과 방식1은 상대적으로 중간값과 꼬리의 편차가 작습니다

## 🤔 비교 4: 슬롯 회전 속도 (핵심 지표)

슬롯 회전은 **예매 완료 → leave → 다음 사람 입장** 사이클이 초당 몇 회 도는지입니다
대기열 시스템의 궁극적인 목표는 이 회전을 빠르게 돌려 더 많은 예매를 처리하는 것입니다

| VU | 방식3 | 방식1 | 방식2 | 최고 |
|----|-------|-------|-------|------|
| **300 iterations** | - | **4,476** | 457 | 방식1 |
| **300 iter/s** | - | **11.84** | 1.21 | 방식1 |
| **1000 iterations** | **5,495** | 3,713 | 417 | 방식3 |
| **1000 iter/s** | **14.5** | 9.84 | 1.10 | 방식3 |
| **3000 iterations** | **6,574** | 763 | 290 | 방식3 |
| **3000 iter/s** | **12.2** | 2.0 | 0.77 | 방식3 |

순서는 **방식3 > 방식1 > 방식2**입니다

3000 VU에서 방식3의 초당 회전은 12.2회, 방식1은 2.0회, 방식2는 0.77회입니다
방식3이 방식2 대비 약 16배 많은 예매를 처리합니다

방식2의 iterations가 상대적으로 적은 원인은 세 가지입니다

1. **Heartbeat 오버헤드**: 10초마다 추가 요청이 나갑니다
2. **이벤트 기반 승격의 비동기 지연**: Scheduler 없이 Redis Key Expiration에 의존합니다
3. **@RequestParam 방식의 토큰 전달**: URL 길이 제한은 없지만 인코딩 오버헤드가 있습니다

응답 속도가 가장 빠른 방식2가 총 처리 건수는 가장 적다는 점이, 이 비교에서 가장 흥미로운 부분입니다

## 🤔 비교 5: 안정성 (에러율)

| VU | 방식3 http_failed | 방식1 http_failed | 방식2 http_failed | 최고 |
|----|-------------------|-------------------|-------------------|------|
| 300 | - | 5.26% | **0.00%** | 방식2 |
| 1000 | 0.51% | 0.013% | **0.00%** | 방식2 |
| 3000 | 0.63% | 0.003% | **0.00%** | 방식2 |

방식2는 300/1000/3000 VU 모든 구간에서 **에러 0건**입니다
방식1은 300 VU에서 5.26%의 에러율을 보이다가 1000 VU 이상에서 0.01% 수준으로 안정화됩니다
방식3은 1000/3000 VU에서 0.5~0.6% 수준의 에러율을 유지합니다

목표("부하가 올라가도 5xx 거의 없음") 기준으로는 셋 모두 합격선 근처지만, **에러 0%를 유지하는 건 방식2뿐**입니다

## 🤔 승격 메커니즘 비교

성능 차이의 근본 원인은 각 구현의 승격 메커니즘입니다

| 항목 | 방식3 (POC C) | 방식1 (POC A) | 방식2 (POC B) |
|------|-------------|-------------|-------------|
| **승격 트리거** | status API 호출 시 동적 계산 | SCAN 기반 (→ O(1) 개선) | 이벤트 기반 (leave → 분산락 → allowedNum) |
| **Heartbeat** | 없음 (polling만) | 없음 | 있음 (10초 간격, TTL 30초) |
| **만료 처리** | Scheduler 30초 주기 | Redis TTL | Redis Key Expiration Event |
| **토큰** | JWT (JWE AES-256-GCM) | JWT | Custom TokenEncryptor (@RequestParam) |
| **분산 락** | 모든 쓰기 | - | 이탈 이벤트 처리 |
| **장점** | 처리량 최고, Scheduler 안정 | 전환 간단, 응답 빠름 | 응답 최저, 에러 0, 중간값 대기 최단 |
| **약점** | 3000 VU에서 통과율 58% | 3000 VU에서 응답 급증 (3.1s) | 슬롯 회전 느림, 꼬리 대기 길음 |

세 방식은 승격을 계산하는 시점이 다릅니다

**방식3**은 사용자가 status API를 호출할 때마다 동적으로 승격 여부를 판단합니다
Scheduler 30초 주기로 만료 처리를 돌리고, 모든 쓰기에 분산 락을 겁니다
**쓰기 경로가 무거운 대신 처리량이 높게 나옵니다**

**방식1**은 Redis SCAN으로 대기열 상태를 주기적으로 훑다가, O(1) 자료구조 개선을 거쳤습니다
**별도 Heartbeat나 이벤트 없이 TTL만으로 만료**를 처리해 단순합니다
1000 VU까지는 좋지만, 3000 VU에서 응답이 3.1초로 급증합니다

**방식2**는 이탈 이벤트를 받아 분산락 안에서 allowedNum을 올리는 이벤트 기반 구조입니다
Heartbeat 10초 + TTL 30초 조합으로 유저 생존을 확인하고, Redis Key Expiration Event로 만료를 처리합니다
**응답이 매우 빠르고 에러가 없지만 슬롯 회전이 느린** 특성이 여기서 나옵니다

| 카테고리 | 1위 | 2위 | 3위 |

| API 레이턴시 | 방식2 | 방식1 | 방식3 |

| 대기 시간 (중간값) | 방식2 | 방식1 | 방식3 |

각 축에서 1위가 다릅니다
**"가장 좋은 POC"는 없었고, 축마다 다른 구현이 1위**였습니다

### 한 줄 요약

- **방식2 (POC B)**: 가장 안정적이고 빠르게 응답하지만, 슬롯 회전이 느려 총 처리 건수가 적습니다
- **방식3 (POC C)**: 가장 많은 예매를 처리하지만, 3000 VU에서 대기열 통과율이 낮아집니다
- **방식1 (POC A)**: 균형형 — 1000 VU까지는 좋지만 3000 VU에서 API 레이턴시가 급증합니다

### 그래서 어느 것을 택할 것인가

이 비교만으로 답이 나오지는 않습니다
방식2는 응답·안정성 6관왕이지만 처리량이 부족하고, 방식3은 처리량 1위지만 통과율이 무너집니다

다음 단계(part3 selection)에서는 **CDN 캐싱이라는 새로운 변수**를 도입해 이 트레이드오프를 다시 평가합니다
순수 성능만으로는 보이지 않던 "확장 가능한 구조"가 그 단계에서 결정적 차이를 만듭니다

## 📚 배운 점

- **목표를 먼저 정의해야 비교가 의미를 갖습니다**. "어느 게 빠르네"가 아니라 "목표 기준으로 어느 게 합격선이네"가 의사결정에 쓰일 수 있는 정보입니다
- **단일 축으로 POC를 판단하지 않습니다**. API 레이턴시만 보면 방식2가 압도적이지만, 슬롯 회전까지 보면 순위가 뒤집힙니다
- **p50과 p95를 분리해 해석합니다**. 중간값이 빠르다고 꼬리도 빠른 것은 아닙니다. 방식2의 p50은 최단이지만 p95는 최장이었습니다
- **응답 속도와 처리량은 독립 지표**입니다. 빠른 응답이 곧 많은 처리량을 의미하지 않습니다. Heartbeat, 분산 락, 이벤트 전파 비용이 처리량에 영향을 줍니다
- **스케일링 내성은 별도 축입니다**. 1000 VU에서 좋은 구현이 3000 VU에서도 좋으리라는 보장이 없습니다. 방식1이 대표 사례입니다
- **CDN 캐시 미적용 상태의 수치는 순수 백엔드 성능**입니다. CDN 적용 후의 비교는 다음 단계에서 이어집니다
