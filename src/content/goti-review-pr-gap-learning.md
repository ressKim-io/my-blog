## 한 줄 요약

> PR #11~#14 연속 리뷰에서 Claude가 놓치고 Gemini가 잡은 항목 5건이 누적되었습니다. 공통 패턴을 분석해 `/review-pr` 커맨드의 3개 에이전트 프롬프트에 "학습된 추가 체크" 12개를 명시적으로 주입했습니다

---

## 🔥 문제: Claude 리뷰가 반복적으로 놓치는 패턴이 있다

go-ti 레포의 K8s PR #11~#14를 Claude와 Gemini로 교차 리뷰하던 중이었습니다
매 PR마다 Gemini가 짚고 Claude는 놓치는 항목이 1~2건씩 나왔고, 이 갭을 `docs/review-gaps.md`에 누적 기록했습니다

네 개의 PR을 거치자 총 **5건의 갭**이 쌓였습니다
개별 사례로 보면 무관해 보였지만, 묶어서 보니 세 가지 근본 약점이 드러났습니다

### 발견한 근본 약점 3가지

**1. 보안 에이전트의 "works = OK" 편향**

동작하기만 하면 통과시키는 경향이 있었습니다
습관·컨벤션 수준의 보안 경고를 누락했습니다

- PR #12: `image.tag: "latest"`가 그대로 머지 대상이었지만 Claude는 지적하지 않았습니다
- PR #11: dev 환경 `insecure: true` 설정이 암묵적으로 통과됐습니다

**2. 패턴 에이전트의 "파일 내부 일관성" 미체크**

cross-file DRY는 검사하지만, 같은 파일 내부의 형식 혼용은 발견하지 못했습니다

- PR #11: FQDN(`svc.cluster.local`)과 short name이 한 파일에 섞여 있었습니다
- PR #14: Makefile 안에서만 하드코딩된 namespace/포트가 반복됐습니다

**3. 보안 에이전트의 "패턴 집계" 부재**

개별 항목은 지적하되, 여러 항목을 묶은 전체 패턴을 인식하지 못했습니다

- PR #13: AppProject에서 `kind: "*"` 와일드카드가 여러 API 그룹에서 반복됐는데, 개별 건만 보고 "전체적 최소 권한 위반"이라는 상위 패턴을 놓쳤습니다

---

## 🤔 원인: 프롬프트에 "실패 사례 기반 체크"가 없었다

Claude의 에이전트 프롬프트는 일반적인 원칙(보안/성능/일관성)만 나열하고 있었습니다
프로젝트에서 **실제로 놓쳤던 구체 사례**가 프롬프트에 들어가 있지 않았습니다

Gemini는 Google 쪽의 대규모 학습 데이터로 이런 패턴을 이미 내재화하고 있는 반면,
Claude는 프로젝트 컨텍스트 기반 리뷰에 강하지만 "이 프로젝트에서 반복 발견된 실수"를 명시하지 않으면 놓치는 경향이 있었습니다

---

## ✅ 해결: 3개 에이전트 프롬프트에 학습된 체크 12개 주입

`.claude/commands/review-pr.md`에 3개 에이전트 각각 **"학습된 추가 체크 (review-gaps 기반)"** 섹션을 추가했습니다

### Agent 1 (보안 + 권한) — 4개 체크

- `image.tag: "latest"`는 무조건 경고합니다. 의도적이라면 사유 주석을 요구합니다
- `kind: "*"` 와일드카드가 2개 이상 API 그룹에서 사용되면 **"전체적 최소 권한 위반"** 상위 경고로 승격합니다
- dev 환경의 보안 비활성화(`insecure: true`, TLS off)는 "dev 한정" 명시 또는 대안 제시를 요구합니다
- stdout에 비밀번호·토큰을 직접 출력하는 스크립트는 경고합니다

### Agent 2 (운영 + 성능) — 4개 체크

- Makefile·스크립트 내 하드코딩 설정값(namespace, 서비스명, 포트)은 변수 중앙화 가능 여부를 검토합니다
- 스크립트 안전성: `trap` cleanup, background process 정리, 변수 quoting(`"${VAR}"`)을 체크합니다
- 환경변수 fallback 패턴 `${VAR:-default}` 미사용 시 경고합니다
- Makefile 변수 선언에서 `?=` vs `:=` 적절성을 검토합니다

### Agent 3 (패턴 + 일관성) — 4개 체크

- 파일 내부 서비스 주소 형식 일관성(FQDN vs short name 혼용)을 검출합니다
- DRY 범위를 확장해 cross-file뿐 아니라 **file-internal 하드코딩 문자열**도 변수 추출 대상으로 봅니다
- 변수 quoting 일관성: 같은 파일 내 `$VAR` vs `"$VAR"` vs `${VAR}` 혼용을 경고합니다
- Makefile ↔ 스크립트 간 설정값 동기화 여부를 확인합니다

### 적용 방식

각 에이전트 프롬프트의 "리뷰 항목" 다음 줄에 다음 블록을 삽입했습니다

```markdown
### 학습된 추가 체크 (review-gaps.md 기반, 필수)

이 프로젝트에서 Gemini가 잡고 Claude가 놓친 실제 패턴입니다.
일반 원칙으로 커버되지 않을 수 있으니 반드시 별도로 체크합니다.

1. <체크 항목 1>
2. <체크 항목 2>
...
```

"필수" 표기와 "Gemini가 잡고 Claude가 놓친 패턴"이라는 문맥을 넣은 이유는, 일반 원칙에 묻혀 스킵되지 않도록 프롬프트에서 명시적 우선순위를 주기 위함이었습니다

## 📚 배운 점

- **AI 리뷰 갭은 반드시 기록합니다.** 단건으로는 무의미해 보여도 3~5건이 쌓이면 패턴이 드러납니다. `docs/review-gaps.md`처럼 표준 경로에 누적해야 분석이 가능합니다
- **"일반 원칙"보다 "실패 사례"가 강합니다.** Claude에 "보안을 체크하라"보다 "`image.tag: latest`는 무조건 경고하라"가 훨씬 잘 작동합니다. 구체 사례는 모호성을 제거합니다
- **파일 내부 일관성은 별도 지시가 필요합니다.** DRY/일관성 같은 추상 원칙은 cross-file 관점으로만 해석되기 쉬우므로, "file-internal 하드코딩", "한 파일 내 혼용"을 별도로 명시합니다
- **패턴 집계는 명시적으로 요구해야 합니다.** "여러 항목이 같은 축을 공유하면 상위 경고로 승격"이라는 문장을 넣어야 Claude가 개별 건에서 상위 패턴을 도출합니다
- **갭 추적 → 분석 → 프롬프트 개선의 사이클은 주 단위로 돌 만합니다.** 이번 1차 사이클에서 12개 체크가 주입됐고, 다음 PR부터 자동 적용됩니다
