## 한 줄 요약

> SDD-lite(검증 중심 축소형) + Acceptance Gate 8개 + Audit-only 범위(prod 10% 드라이런까지)로 Phase 7 Go Readiness Audit 설계를 확정했습니다. 1차·2차 코드 리뷰에서 제기된 P0/P1 지적을 모두 반영한 결과입니다.

---

## 배경

go-ti 프로젝트의 Java→Go 마이그레이션 Phase 1~6(stadium·user·queue·resale·payment·ticketing)은 2026-04-11까지 구현이 완료되었습니다. 다음 단계는 컷오버이지만, "Java에서는 되던 것이 Go로 바꾸자마자 안 된다"는 상황을 사전에 차단해야 했습니다.

원래 Phase 7 자리에는 Cleanup이 배치되어 있었습니다. 그러나 컷오버 전 검증이 선행되어야 한다는 판단 아래, Phase 7을 **"Go Readiness Audit"**으로 재정의하고 Cleanup을 Phase 8로 이동하기로 했습니다. 이 재정의와 함께 검증 작업의 **문서 형태·게이트 구조·컷오버 범위** 세 가지를 결정해야 했습니다.

## 🧭 선택지 비교

### SDD 형태: 세 가지 옵션

| 옵션 | 핵심 아이디어 | 장점 | 한계 |
|------|---------------|------|------|
| A. 풀 SDD | Phase 6 ticketing-deploy 956줄과 동일한 신규 구현 SDD 형태 | 정합성·완전성 높음, 다른 Phase와 형식 통일 | 신규 구현용 템플릿이 audit 성격에 맞지 않음, 비대해짐 |
| B. SDD-lite | 게이트·체크리스트 위주의 검증 중심 축소형 | audit 성격에 적합, phase-workflow PWG1~5 요건 충족 | 기존 템플릿과 다른 구조 — 처음 접하는 사람 적응 필요 |
| C. dev-logs 노트 | 한 장짜리 audit 메모 | 가장 가볍고 빠름 | 게이트 추적·매트릭스 관리 불가, phase-workflow Gate 미충족 |

**기각 이유**

- **A 탈락**: 풀 SDD 템플릿은 신규 구현을 기록하는 형식입니다. 이미 완성된 코드를 검증하는 audit 단계에는 구조가 맞지 않고, 작성 비용 대비 실질적 추적 가치가 낮습니다
- **C 탈락**: 한 장짜리 노트는 G1~G8 게이트 추적과 합격 기준 매트릭스를 담을 수 없습니다. phase-workflow.md의 PWG 요건을 충족하지 못합니다

### 게이트 구조: 6게이트 vs 8게이트

1차 리뷰에서 **G7(데이터 정합성)·G8(보안 동등성)** 누락이 P0 지적으로 올라왔습니다.

기존 6게이트 안에서는 G4(부하) 게이트에 이 두 검증을 묶는 방안을 검토했습니다. 그러나 좌석 이중선점·JWT·AuthZ 같은 보안/정합성 검증은 **부하 측정과 검증 축 자체가 다릅니다**. 한 게이트에 묶으면 각 축의 합격 기준이 서로를 가리게 됩니다. G7/G8을 G4에서 분리하여 독립 검증축으로 세웠습니다.

### 컷오버 범위: prod 100% 포함 vs Audit-only

1차 SDD가 "staging/prod 대상"으로 범위를 모호하게 정의한 것이 2차 리뷰에서 P1로 지적됐습니다. §1.2에는 "Java 코드 제거 제외"가 명시되어 있는데, prod 100% 컷오버와 양립하지 않는 내용입니다.

**prod 100% 컷오버**는 본 SDD 범위에서 제외했습니다. Phase 7에서는 **staging 컷오버 리허설 + prod 10% 드라이런**까지만 수행하고, prod 100%는 별도 운영 체인지 티켓으로 처리합니다. Java 코드 제거(Cleanup)는 Phase 8에서 이루어집니다.

### 결정 기준과 최종 선택

**SDD-lite(옵션 B) + 8게이트 + Audit-only 범위**를 채택했습니다.

결정 기준은 다음 우선순위였습니다.

1. **phase-workflow PWG 요건 충족**: 게이트 추적과 체크리스트 구조가 필수
2. **검증 축 독립성**: 보안·데이터 정합성을 부하와 동일 게이트에 묶으면 합격 기준이 흐려짐
3. **범위 명확화**: SDD 내부 조항 간 모순 제거, 이후 운영 체인지 티켓과 책임 분리

SDD-lite는 audit 목적에 맞는 가장 적합한 형태이며, 처음 접하는 사람의 적응 비용은 템플릿 변형 이유를 문서 앞에 명시하는 것으로 해소했습니다.

## ✅ 결정 내용

### 8개 Acceptance Gate

확정된 게이트 구조는 다음과 같습니다.

| Gate | 검증 항목 | 비고 |
|------|-----------|------|
| G1 | API 계약 동등성 | endpoint·응답 스키마·상태 코드 비교 |
| G2 | E2E 시나리오 | 티켓팅 핵심 플로우 자동화 |
| G3 | 정적 분석 | Go vet·staticcheck·race detector |
| G4 | 부하 성능 | `max(Java p95 × 1.2, SLO 상한)` |
| G5 | 관측성 | 라벨 정책 (필수 누락 0, cardinality ≤ 10K/메트릭, 삭제·이름변경 금지) |
| G6 | 운영 준비 | 헬스체크·드레인·ConfigMap·시크릿 동등성 |
| G7 | **데이터 정합성** | 공유 락(DB UNIQUE·Redis SETNX) 동등성, weighted routing 50:50 시 이중선점 검증 |
| G8 | **보안 동등성** | JWT·AuthZ 등가성 |

G4의 합격 기준으로 `max(Java p95 × 1.2, SLO 상한)` 공식을 채택했습니다. Java 서비스 자체가 SLO를 위반하는 상황에서 Go도 동일 기준으로 통과하는 것을 차단하기 위해서입니다.

G5 라벨 정책은 세 축으로 정리됩니다.

- **필수 라벨 누락 0건**: 누락 시 G5 미통과
- **추가 허용**: cardinality ≤ 10K/메트릭 조건 하에 신규 라벨 추가 가능
- **삭제·이름변경 금지**: 기존 대시보드·알람의 쿼리가 깨지지 않도록

### 11단계 실행 계획(S0~S11)

Phase 7은 S0부터 시작합니다.

```text
S0  — API 비교 도구 결정 + 부하 도구·시드 합의
S1  — API 계약 비교 (G1)
S2  — E2E 시나리오 (G2)
S3  — 정적 분석 (G3)
S4  — 부하 (G4)
S5  — 관측성 (G5)
S6  — 운영 준비 (G6)
```

## 📚 배운 점

- **audit용 SDD는 신규 구현용 템플릿과 다릅니다.** 형식 통일보다 문서 목적에 맞는 구조가 우선입니다. 기존 템플릿을 억지로 따르면 비대해지고, 정작 필요한 게이트 추적 구조가 빠집니다
- **게이트를 묶으면 합격 기준이 흐려집니다.** G7(데이터 정합성)과 G8(보안 동등성)은 G4(부하)와 검증 축이 다릅니다. 같은 게이트에 묶으면 각각의 합격 조건이 서로를 가리고 추후 책임 소재도 모호해집니다
- **SLO 기준선이 낮으면 부하 합격이 무의미해집니다.** `max(Java p95 × 1.2, SLO 상한)` 공식은 Java 서비스가 이미 SLO를 위반 중인 경우에도 Go가 동등 이상임을 보장합니다. 단순 SLO 비교만으로는 이 케이스를 놓칩니다
- **범위 모호성은 두 번째 리뷰에서도 잡힙니다.** "staging/prod 대상"이라는 한 줄이 내부 조항과 모순을 만들었고, 결국 P1 지적으로 이어졌습니다. 컷오버 범위는 SDD 첫 섹션에서 경계를 명확히 그어야 합니다
- **Phase 7은 PAUSED 상태로 프로젝트가 종료됩니다.** S0 진입 직전에 Phase 6.5(Go prod 인프라 신설)가 추가되었고, 이후 전체 인프라 destroy(2026-04-21)로 컷오버 미완료 상태로 프로젝트가 마무리되었습니다. 게이트 검증 자체는 설계 완료 단계까지 진행되었습니다
