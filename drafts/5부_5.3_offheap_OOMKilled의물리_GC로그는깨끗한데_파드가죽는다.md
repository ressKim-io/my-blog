## ① 이 절에서 도달할 지점

이 절은 5부에서 여러 번 미뤄 둔 청구서를 받는 자리입니다. 상황은 늘 이렇게 시작합니다.

> GC 로그는 깨끗합니다. Full GC도 없고, 힙 사용량은 한도의 절반. `-Xmx`도 넉넉히 줬습니다.
> 그런데 파드가 `exit 137 (OOMKilled)`로 죽습니다. 재시작해도 몇 시간 뒤 또 죽습니다.

이 절을 다 읽으면 다음을 물리로 설명할 수 있게 됩니다.

- 왜 **힙은 멀쩡한데** 프로세스가 죽는가 — 커널이 죽일 때 JVM은 `OutOfMemoryError`를
  던질 기회조차 없다는 것.
- `-Xmx` 밖에 있는 메모리 지형 전체 지도 — Metaspace, 코드 캐시, 스레드 스택, 다이렉트 버퍼,
  **glibc malloc arena**, GC 자료구조 — 이 각각이 어떻게 조용히 자라 `memory.max`를 넘기는지.
- NMT(Native Memory Tracking)로 이 지형을 **실제로 재는** 법, 그리고 `reserved`와 `committed`가
  왜 다른 숫자이고 cgroup은 그중 무엇을 세는지.
- Go에서도 같은 일이 왜 일어나는지 — `GOMEMLIMIT`이 **cgo/C 할당은 안 덮는다**는 것.

1.3(다이렉트 메모리), 2.4(cgroup이 힙+네이티브 합산), 3.2(ptmalloc arena), 3.4(힙 밖 지형)에서
여러 번 스쳐 온 조각들이 여기서 하나의 그림으로 수렴합니다. `[복선 확인 필요: 1.3·2.4·3.2·3.4]`

## ② 개념/실체 — 커널의 죽음과 런타임의 죽음은 다른 사건이다

### 두 종류의 OutOfMemory (정의부터, 이게 핵심)

"메모리가 부족해서 죽었다"에는 **완전히 다른 두 사건**이 섞여 있습니다.

- **런타임의 죽음 — `OutOfMemoryError`(JVM)**: 힙(`-Xmx`)이 꽉 차서 GC를 아무리 돌려도 공간을
  못 만들면, JVM이 스스로 `java.lang.OutOfMemoryError: Java heap space`를 **던집니다.** 이건
  자바 예외라서 스택 트레이스가 남고, 그 직전 GC 로그엔 연속된 Full GC가 찍힙니다. 즉
  **로그에 흔적이 남습니다.**
- **커널의 죽음 — OOMKilled(SIGKILL)**: 프로세스 전체(힙+힙 밖 전부)의 물리 메모리 합이 cgroup
  `memory.max`를 넘으면, 리눅스 OOM killer가 **`SIGKILL`(9)**로 프로세스를 즉살합니다. SIGKILL은
  **잡을 수도, 무시할 수도 없습니다.** 그래서 셧다운 훅도, `OutOfMemoryError`도, 자바 스택
  트레이스도 없습니다. 종료코드만 `137`(=128+9)로 남고, GC 로그는 **마지막까지 깨끗**합니다.

```
힙이 꽉 참                          프로세스 전체 RSS > memory.max
   │                                       │
   ▼                                       ▼
JVM이 OutOfMemoryError 던짐          커널 OOM killer가 SIGKILL
   │                                       │
   ├ 자바 예외 · 스택 트레이스          ├ 잡을 수 없음(핸들러 무효)
   ├ 직전 GC 로그에 Full GC 연속        ├ OutOfMemoryError 없음
   └ 힙이 원인 → 힙 로그로 추적 가능    └ GC 로그 깨끗 · exit 137 · "왜?"
```

**"GC 로그는 깨끗한데 파드가 죽는다"는 정확히 두 번째 사건입니다.** 힙은 건강하니 GC는 할 일을
다 하고 있고(그래서 로그가 깨끗), 정작 한도를 넘긴 범인은 **힙 밖**에 있습니다. 커널은 힙과 힙
밖을 구분하지 않고 합산하므로(5.1), 힙 밖이 부풀면 힙이 아무리 멀쩡해도 죽습니다.

### 힙 밖 지형을 실제로 재 보면 — NMT

핫스팟의 **NMT(Native Memory Tracking)**는 JVM이 힙 밖에서 쓰는 메모리를 범주별로 보여 줍니다
(`-XX:NativeMemoryTracking=summary`, 운영 중엔 `jcmd <pid> VM.native_memory`). 아무 일도 안 하는
빈 프로그램을 `-Xmx256m`으로 띄우고 종료 시점 NMT를 찍어 봤습니다.

**측정: OpenJDK 11.0.31, `-Xmx256m`, 트리비얼 프로그램 (committed 기준)**

| 범주 | committed | 비고 |
|---|---|---|
| Java Heap | 65 MB | 실제 힙(256MB 중 사용분) |
| GC | 45 MB | 카드 테이블·마킹 비트맵 등 GC 자료구조 |
| Code | 7.7 MB | JIT 코드 캐시(예약은 240MB) |
| Class(Metaspace) | 4.7 MB | 클래스 메타데이터(예약은 1GB) |
| Thread | 0.7 MB | 스레드 15개의 스택 |
| Symbol/Internal/기타 | ~4 MB | 심볼 테이블·내부 구조 |
| **Total committed** | **≈ 126 MB** | 이 중 **힙 밖만 ≈ 61 MB** |

핵심은 이겁니다. **힙을 256MB로 설정했고 실제 힙은 65MB뿐인데, 아무 일도 안 한 JVM이 이미 힙
밖으로 약 61MB를 커밋**했습니다. 컨테이너 한도를 `-Xmx`만 보고 잡으면, 이 61MB(그리고 부하가
붙으면 더 커질 이 영역)를 통째로 무시하는 셈입니다.

### `reserved` vs `committed` — cgroup은 무엇을 세는가

NMT는 두 숫자를 보여 줍니다. 이 구분을 모르면 숫자를 오해합니다.

- **`reserved`(예약)** — 프로세스가 확보해 둔 **가상 주소 공간**. 실제 물리 페이지가 아닙니다.
  위 측정에서 Total reserved는 1.69GB였지만(Class space 1GB, 코드 캐시 240MB 등 예약분 포함),
  이건 **물리 메모리를 안 씁니다.**
- **`committed`(커밋)** — OS에 "이 범위를 실제로 쓰겠다"고 약속한 부분. 실제로 **접촉(touch)한
  페이지**가 물리 메모리(RSS)가 됩니다.

cgroup `memory.current`가 세는 건 **접촉된 물리 페이지(RSS 계열)**이지 `reserved`가 아닙니다.
그래서 "가상 메모리(VSZ)는 몇 GB인데 왜 안 죽지?"는 정상이고(예약일 뿐), 정작 위험한 건
**committed가 실제로 채워지며 늘어나는** Metaspace·다이렉트 버퍼·malloc arena입니다.

## ③ 비용/메커니즘 해부 — 범인 목록

힙 밖에서 `memory.max`를 넘기는 단골 범인들입니다. 각각 "왜 조용한가(GC 로그에 안 남는가)"가
공통점입니다.

### 범인 1: Metaspace — 기본이 무제한이다

클래스 메타데이터(클래스 구조·메서드·상수 풀)는 힙이 아니라 **Metaspace**(네이티브)에 있습니다.
그런데 기본 상한이 사실상 무제한입니다(측정 JDK 11.0.31: `MaxMetaspaceSize` ≈ 2^64 = 무제한,
`CompressedClassSpaceSize` = 1GB 예약). 그래서 **클래스로더 누수**(동적 클래스 생성, 리로딩,
프록시·CGLIB 남발, 잘못 캐시된 클래스로더)가 있으면 Metaspace가 끝없이 자랍니다. 힙은 GC가
계속 청소해 깨끗한데, Metaspace는 상한이 없어 커널 한도까지 밀고 올라가 OOMKilled. `MaxMetaspaceSize`를
**명시적으로 캡**하면, 커널이 죽이기 전에 JVM이 `OutOfMemoryError: Metaspace`를 던져 **로그에
흔적을 남깁니다** — 진단 가능성이 완전히 달라집니다.

### 범인 2: 코드 캐시 — JIT이 쌓은 기계어 (기본 240MB)

JIT이 컴파일한 네이티브 코드가 사는 **코드 캐시**는 예약 240MB입니다(측정 JDK 11.0.31:
`ReservedCodeCacheSize` = 251658240 = 240MB, 2.3에서 본 그 값). 코드가 많은 대형 앱·프레임워크
과다·프로파일 폭주로 여기가 차면 committed가 240MB까지 오릅니다. 힙과 무관하니 GC 로그엔 안
남습니다.

### 범인 3: 스레드 스택 — 스레드 수 × 스택 크기

스레드마다 네이티브 스택을 가집니다(`-Xss`). 위 측정에서 스레드 15개에 스택 예약 ~31MB였는데,
**스레드가 수천 개**로 폭발하면(스레드풀 오설정, 블로킹 IO에 스레드-per-request, 커넥션 폭주)
이 곱이 그대로 네이티브 메모리입니다. 스택은 힙 밖이라 GC와 무관합니다. 4부에서 본 대로 코어
수에서 파생되는 GC/컴파일러 스레드도 여기에 얹힙니다.

### 범인 4: 다이렉트 버퍼 — 힙 밖의 IO 버퍼 (1.3의 회수)

`java.nio.ByteBuffer.allocateDirect()`, Netty·gRPC·카프카 클라이언트 등이 쓰는 **다이렉트
메모리**는 힙 밖입니다(1.3에서 예고). 기본 상한이 특이합니다(측정 JDK 11.0.31:
`MaxDirectMemorySize` = 0). **0은 "무제한"이 아니라 "상한을 최대 힙(-Xmx)에 맞춤"**을
뜻합니다. 즉 `-Xmx`가 크면 다이렉트도 그만큼 더 쓸 수 있어, 힙+다이렉트 합이 한도를 넘길 수
있습니다. 다이렉트 버퍼는 GC가 참조를 회수해야(Cleaner) 풀리므로, 힙에 여유가 많으면 GC가
안 돌아 **다이렉트가 안 풀린 채 쌓이는** 역설이 생깁니다.

### 범인 5: glibc malloc arena — 코어 수만큼 늘어나는 여유 (3.2의 회수)

3.2에서 본 glibc `ptmalloc`은 락 경합을 줄이려고 **arena(할당 풀)를 여러 개** 둡니다. 64비트에서
기본 arena 수 = **8 × CPU 코어**입니다. 그리고 여기서 5.1과 고리가 걸립니다 — **arena 수는
프로세스가 인식한 코어 수에 비례**합니다. 컨테이너가 CPU를 많게 인식하면 arena가 수십 개로
불어나고, arena마다 해제된 메모리를 OS에 안 돌려주고 쥐고 있어(단편화) 각 10~30MB씩 RSS를
차지할 수 있습니다(출처: glibc 동작 분석 글들, 아래 참조). 8코어면 arena가 최대 64개 → 수백 MB가
"쓰지도 않는데 잡혀 있는" 상태.

```
컨테이너가 인식한 코어 수 = 8            (CPU 한도를 안 걸어 호스트 코어를 봄)
                │
glibc arena 최대 = 8 × 8 = 64개
   ┌────┬────┬────┬────┬─── ... ───┐
   │ A0 │ A1 │ A2 │ A3 │           │   각 arena가 해제분을 쥐고 안 돌려줌
   └────┴────┴────┴────┴─── ... ───┘   → RSS 수백 MB, 힙과 무관
   해결: MALLOC_ARENA_MAX=2  또는  jemalloc/tcmalloc 로 교체
```

이게 "JDK 컨테이너가 힙은 안정적인데 RSS가 스멀스멀 오르다 OOMKilled"의 대표 범인입니다.
`MALLOC_ARENA_MAX=2` 한 줄이 종종 가장 효과적인 처방입니다(락 경합이 약간 늘 수 있으나 대개
처리량 영향은 미미). 많은 조직이 JVM 컨테이너 표준으로 jemalloc을 씁니다.

### 범인 6: GC 자료구조 — GC가 GC 아닌 메모리를 쓴다

위 측정에서 GC 범주가 committed 45MB로 힙(65MB) 다음으로 컸습니다. 카드 테이블(5.2), 마킹
비트맵, G1의 RSet, ZGC의 forwarding 테이블 등은 전부 힙 밖 네이티브입니다. GC를 돌리는 살림이
힙과 별도로 메모리를 먹습니다.

### Go도 같은 병에 걸린다 — `GOMEMLIMIT`은 cgo를 안 덮는다

Go는 힙 밖 지형이 JVM보다 좁지만 없지 않습니다. 그리고 결정적으로 **`GOMEMLIMIT`은 Go 런타임이
관리하는 메모리(힙·고루틴 스택·런타임 구조)만 셈**합니다. 다음은 그 밖입니다.

- **cgo/C 할당** — cgo로 부른 C 라이브러리(SQLite, 이미지·압축·ML 라이브러리 등)가 `malloc`한
  메모리는 Go 힙이 아니고 `GOMEMLIMIT`에도 안 잡힙니다. `runtime.MemStats`도 GC 로그
  (`GODEBUG=gctrace=1`)도 깨끗한데 RSS만 오르다 OOMKilled — JVM의 다이렉트 버퍼와 판박이입니다.
- **고루틴 스택** — 고루틴이 수십만 개면 스택(각 최소 2KB, 3.3)의 합이 커집니다.
- **OS 반납 지연(madvise)** — Go는 안 쓰는 페이지를 `madvise`로 OS에 돌려주는데, 정책이
  버전에 따라 달랐습니다. **Go 1.12에서 `MADV_FREE`로 바꿨다가**(RSS가 실제보다 높게 보여
  컨테이너 모니터링·OOM과 궁합이 나빴음) **1.16부터 다시 `MADV_DONTNEED`가 기본**입니다. 그래서
  현재(2026-07, Go 1.26)는 RSS가 대체로 정직합니다. 다만 `GODEBUG=madvdontneed=0`으로
  `MADV_FREE`로 되돌리면 다시 RSS가 부풀어 보일 수 있습니다(출처: golang/go #42330, #44554).

## ④ 회피/최적화 — 재고, 캡을 씌우고, 여백을 남긴다

### 먼저 재라 — 추측 금지

- **JVM**: `-XX:NativeMemoryTracking=summary`로 띄우고 `jcmd <pid> VM.native_memory summary`로
  범주별 committed를 봅니다. 어느 범주가 시간에 따라 자라는지(Metaspace? Thread? Internal?)가
  범인을 지목합니다. `jcmd ... VM.native_memory baseline` 후 `... summary.diff`로 증가분만
  추적하면 누수 탐지가 쉽습니다.
- **Go**: `GODEBUG=gctrace=1`로 힙은 정상인지 확인하고, RSS(cgroup `memory.current`)와의 격차가
  크면 cgo/스택/반납지연을 의심합니다. cgo 힙은 `jemalloc` 프로파일링이나 `valgrind`/`pprof`
  (C 쪽)로 별도 계측합니다.

### 캡을 씌워 '커널의 죽음'을 '런타임의 죽음'으로 바꿔라

핵심 전략은 **상한 없는 힙 밖 영역에 상한을 줘서, 커널이 죽이기 전에 JVM이 `OutOfMemoryError`를
던지게** 만드는 것입니다. 그래야 로그에 흔적이 남고 원인이 특정됩니다.

- `-XX:MaxMetaspaceSize=<값>` — Metaspace 폭주를 `OOME: Metaspace`로 전환.
- `-XX:MaxDirectMemorySize=<값>` — 다이렉트 버퍼를 명시 캡.
- `-XX:ReservedCodeCacheSize` 조정, 스레드풀·`-Xss`로 스레드 스택 총량 제어.
- `MALLOC_ARENA_MAX=2` 또는 jemalloc/tcmalloc으로 arena 단편화 억제.

### 여백을 남기고 한도를 잡아라 (5.1·5.2와 합류)

컨테이너 `limits.memory`는 **힙 + 힙 밖 전부 + 페이지 캐시 여유**를 담아야 합니다. 실무 감각으로
`-Xmx`(또는 `MaxRAMPercentage`)를 한도의 50~75%에 두고, 나머지를 위 힙 밖 몫으로 남깁니다.
"한도 = -Xmx"로 잡으면 5.3의 모든 범인이 곧장 OOMKilled로 돌아옵니다. Go도 마찬가지로
`GOMEMLIMIT`을 한도의 ~90%로 두되, **cgo가 있으면 그 만큼 더 여백**을 빼야 합니다.

## ⑤ 세 언어 수렴 — 힙 밖 지형의 넓이

| | 힙 밖 지형 | `GC 로그 깨끗한데 죽음` 위험 | 왜 |
|---|---|---|---|
| **Rust** | 개념상 "힙 밖"이 자연스러움 — 모든 할당이 명시적, 런타임 여백 없음 | 낮음 | GC도 런타임도 없어 "보이지 않게 자라는" 영역이 원천적으로 적음. malloc 단편화는 있으나(3.2) 소유권으로 해제 시점이 명확 |
| **Go** | 고루틴 스택 + **cgo/C 할당** + 반납지연 | 중간 | 순수 Go면 좁지만, cgo를 쓰는 순간 `GOMEMLIMIT` 밖 영역이 생겨 JVM과 같은 병 |
| **Java** | Metaspace·코드캐시·다이렉트·스레드스택·malloc arena·GC구조 — 가장 넓음 | 높음 | 힙 밖 영역이 많고 그중 다수가 기본 무제한/큰 예약. NMT 없이는 안 보임 |

세로로 읽으면 **"런타임이 클수록 힙 밖 지형이 넓고, 넓을수록 '깨끗한 로그' 뒤에서 죽을 자리가
많다"**입니다. Rust는 모든 메모리가 코드에 드러나 있어 숨을 곳이 적고, Java는 편의를 위해 런타임이
관리하는 영역이 넓은 만큼 그 관리 밖에서 새는 경로도 많습니다. Go는 순수하면 Rust에 가깝고 cgo를
쓰면 Java에 가까워지는, 선택이 위험을 결정하는 중간 지대입니다.

이건 4부의 "회계의 위치" 축(사람/컴파일러/런타임)이 컨테이너에서 다시 나타난 것입니다. 회계를
런타임에 위임할수록 편하지만, **런타임이 세지 않는 메모리**(cgo, 다이렉트, arena)가 사각지대가
됩니다.

## ⑥ 트레이드오프 소결

한 문장: **커널은 힙과 힙 밖을 구분하지 않고 합산하는데, 런타임은 힙만 관리하고 로그에 남긴다 —
그 사각지대가 exit 137이다.** 그래서 진단의 요령은 하나로 모입니다. "메모리로 죽었다"를 보면
먼저 **런타임의 죽음(OOME·로그 있음)인지 커널의 죽음(SIGKILL·로그 없음·exit 137)인지**를 가르고,
후자면 힙이 아니라 **힙 밖을 NMT/RSS로 재라.**

그리고 5부의 세 절이 여기서 합류합니다. 5.1이 "커널은 전부 합산한다"는 회계 규칙을, 5.2가
"상자 크기를 잘 잡아도"라는 전제를, 5.3이 "힙 밖 여백이 없으면 그 안에서 죽는다"는 결말을
맡았습니다. 남은 건 시간 축입니다 — 이 모든 게 **시작하는 순간**엔 어떻게 보이는가. 그게 5.4.

## ⑦ DevOps 관점 심화 질문

1. **죽음의 종류 가르기** — 파드가 `exit 137`로 죽었을 때, 그게 커널 OOM인지 kubelet 축출인지
   liveness 실패인지를 `dmesg`의 OOM 리포트, cgroup `memory.events`의 `oom_kill`, kubelet 이벤트
   중 무엇으로 구분하겠습니까? 각 신호가 가리키는 원인이 어떻게 다릅니까?

2. **NMT 상시화의 비용** — `NativeMemoryTracking=summary`는 약간의 오버헤드가 있습니다. 프로덕션에
   상시 켤지, 재현 파드에서만 켤지, 어떤 기준으로 정하겠습니까? `summary`와 `detail`의 트레이드오프는?

3. **Metaspace 캡 정하기** — `MaxMetaspaceSize`를 안 씌우면 커널 OOM, 너무 낮게 씌우면
   `OOME: Metaspace`로 조기 사망입니다. 클래스 수가 배포마다 늘어나는 앱에서 이 값을 어떤
   데이터(NMT Class committed 추이)로 정하겠습니까?

4. **malloc arena vs jemalloc** — `MALLOC_ARENA_MAX=2`와 jemalloc 교체 중 무엇을 택하겠습니까?
   멀티스레드 할당이 많은 워크로드에서 arena를 2로 줄일 때의 락 경합을 어떻게 계측해 판단합니까?
   이 결정이 5.1의 컨테이너 CPU 인식과 어떻게 얽힙니까?

5. **Go cgo 사각지대** — cgo를 쓰는 Go 서비스에서 `GOMEMLIMIT`을 한도의 90%로 뒀는데도
   OOMKilled가 납니다. C 쪽 할당을 어떤 도구로 계측하고, `GOMEMLIMIT`을 몇 %로 다시 잡겠습니까?

6. **한도 산정 자동화** — "힙 + 힙 밖 + 페이지 캐시 여유"를 담는 `limits.memory`를 사람이
   감으로 잡는 대신, NMT/RSS 실측을 CI나 부하 테스트에서 뽑아 자동 산정하려면 어떤 파이프라인이
   필요합니까? 배포 후 힙 밖 증가(회귀)를 어떤 알람으로 잡겠습니까?

## ⑧ 다음 절: 5.4 콜드 스타트와 웜업의 청구서

지금까지는 **정상 상태(steady state)**의 메모리 회계였습니다. 그런데 파드가 **막 뜨는 순간**은
또 다른 세계입니다 — JIT은 아직 인터프리터로 굴러 느리고(2.3의 웜업), GC는 아직 힙을 못 잡았고,
클래스 로딩·코드 캐시·arena가 한꺼번에 부풀어 오릅니다. 오토스케일링이 트래픽 급증에 파드를
새로 띄울 때, 이 콜드 스타트 구간의 느림과 메모리 스파이크가 **오토스케일링·서버리스의 경제성**을
직접 때립니다. 5.4는 그 시작 구간의 청구서를 해부하고, native-image·CRaC 같은 처방이 웜업을
어떻게 없애는지(그 대가는 무엇인지) 봅니다.
