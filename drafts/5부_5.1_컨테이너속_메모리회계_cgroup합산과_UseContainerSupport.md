# 5부 5.1 — 컨테이너 속 런타임의 메모리 회계: 커널은 전부 합산하고, 런타임은 컨테이너를 뒤늦게 배웠다

> **이 문서의 성격** — 최종 블로그 원고가 아니라, 최종 집필 에이전트에게 넘길 1차 심화
> 정리(reference-level draft)입니다. `개념 → 하드웨어/런타임 동작 → 비용 해부 → 회피 전략
> → 3개 언어 연결 → 트레이드오프 → 심화 질문`의 순서로 아래에서 위로 쌓아 올립니다. 그림은
> 모두 ASCII로 그렸고, 본문의 수치는 (해당 절 기준 버전)이며 환경/버전에 따라 달라질 수
> 있음을 전제로 읽어 주세요.

## ① 이 절에서 도달할 지점

4부까지 우리는 세 언어의 런타임이 메모리를 **어떻게 할당하고 회수하는가**를 봤습니다. 5부는
같은 이야기를 한 층 위, 즉 **컨테이너와 노드**라는 무대에서 다시 봅니다. 첫 절의 목표는
하나입니다.

> **"파드에 준 메모리 한도(limit)와, 런타임이 스스로 잡는 힙 크기는 서로 다른 층위의 숫자다.
> 그리고 이 둘을 이어 주는 다리를 언어마다 다르게(혹은 아예 안) 놨다."**

이 절을 다 읽으면 다음을 스스로 설명할 수 있게 됩니다.

- 쿠버네티스에서 `limits.memory: 512Mi`라고 적으면 리눅스 커널의 무엇이 바뀌는가 (cgroup v2
  `memory.max`).
- 커널이 "이 프로세스가 512Mi를 넘었다"고 판단할 때, **정확히 무엇무엇을 합산**해서 그렇게
  판단하는가 (힙만이 아니다).
- JVM은 왜 `-Xmx`를 안 줘도 컨테이너 한도에 맞춰 힙을 줄이는데, Go는 왜 `GOMEMLIMIT`을 안 주면
  한도를 무시하고 커지다가 죽는가.
- Rust에는 왜 이 질문("힙을 얼마로 잡을까") 자체가 없는가.

그리고 이 절은 5.3(off-heap OOMKilled — "GC 로그는 깨끗한데 파드가 죽는다")의 **회계 장부를
미리 펼쳐 두는** 역할을 합니다. 커널이 무엇을 합산하는지 여기서 정확히 세워 두면, 5.3에서
"힙은 멀쩡한데 왜 죽었나"가 저절로 풀립니다.

## ② cgroup v2의 메모리 회계 — 커널은 '힙'을 모른다, '페이지'를 센다

### cgroup이란 무엇인가 (정의부터)

**cgroup(control group)**은 리눅스 커널이 프로세스 무리를 하나로 묶어 **자원(CPU·메모리·IO
등)을 계량하고 제한**하는 장치입니다. 컨테이너 = "네임스페이스(격리) + cgroup(제한)"이라고
보면 됩니다. 도커든 쿠버네티스든, 컨테이너에 한도를 걸면 결국 이 cgroup 파일에 숫자를 쓰는
일로 귀결됩니다.

2026년 현재 리눅스는 **cgroup v2(unified hierarchy)**가 표준입니다(커널 7.x, 그 이전 6.x도
동일). v2에서 메모리 관련 핵심 파일은 셋입니다.

- **`memory.max`** — 하드 한도. 이 cgroup이 쓸 수 있는 메모리의 절대 상한. 쿠버네티스의
  `limits.memory`가 여기로 내려옵니다. **넘으면 죽습니다(뒤에서 설명).**
- **`memory.high`** — 소프트 한도(스로틀). 넘으면 커널이 회수 압력을 강하게 걸어 할당을
  **느리게** 만들지만, 죽이지는 않습니다. 쿠버네티스는 기본적으로 이걸 잘 안 씁니다.
- **`memory.current`** — 지금 이 순간 커널이 이 cgroup에 **청구한 총량**. 우리가 `kubectl top`
  이나 `docker stats`에서 보는 그 숫자의 뿌리입니다.

### 커널이 `memory.current`에 합산하는 것 — 여기가 핵심

개발자가 흔히 하는 착각이 "메모리 사용량 = 내 힙(heap)"입니다. 커널의 회계는 그렇지 않습니다.
커널은 언어의 '힙'이라는 개념을 모릅니다. 커널이 아는 건 **페이지(page, 보통 4KiB)**뿐이고,
이 프로세스 무리가 만지는 모든 종류의 페이지를 합산합니다.

```
Pod: limits.memory: 512Mi  ─────►  cgroup v2 /sys/fs/cgroup/.../memory.max = 536870912
                                                                              (512 × 1024²)
┌──────────────────────────────────────────────────────────────────────────┐
│  memory.current  =  커널이 이 cgroup에 청구한 총 페이지 (memory.stat 분해)   │
│                                                                            │
│   ┌──────────────┐  anon        익명 페이지 = 힙 + 스택 + 익명 mmap         │
│   │              │              (여기 '언어 런타임의 힙'이 들어간다)        │
│   ├──────────────┤  file        파일 backed = 페이지 캐시(읽은 파일·.so·mmap)│
│   ├──────────────┤  kernel      slab(dentry·inode) + page table + 커널 스택 │
│   ├──────────────┤  sock        소켓 송수신 버퍼                            │
│   ├──────────────┤  shmem       공유메모리 · tmpfs(/dev/shm) · memfd        │
│   └──────────────┘  ...                                                    │
│                                                                            │
│   Σ(이 전부)  >  memory.max   ──►   커널이 회수 시도 → 실패하면 OOM kill     │
└──────────────────────────────────────────────────────────────────────────┘
       ▲
       └── 언어 런타임이 "내 힙은 200MB"라고 말해도, 커널이 세는 건 이 기둥 전부다.
           힙은 anon 기둥의 '일부'일 뿐이다.
```

`anon`(익명 메모리)이 우리가 아는 힙·스택입니다. 하지만 `file`(페이지 캐시), `kernel`(슬랩·
페이지 테이블), `sock`(소켓 버퍼), `shmem`까지 전부 같은 장부에 올라갑니다. 이 분해는
`memory.stat` 파일에서 그대로 읽을 수 있습니다.

> 실무 확인용: `cat /sys/fs/cgroup/memory.stat`을 파드 안에서 찍으면 `anon`, `file`,
> `slab`, `sock`, `kernel_stack`, `pagetables`, `percpu` 항목이 바이트로 나옵니다. "힙은
> 작은데 RSS가 크다"의 범인을 여기서 찾습니다.

한 가지 완충 장치는 있습니다. `file`(페이지 캐시) 중 **깨끗한(clean) 페이지**는 언제든 버릴
수 있으므로, 한도에 근접하면 커널이 먼저 이걸 회수합니다. 그래서 페이지 캐시는 "압박이 오면
줄어드는" 종류입니다. 반면 `anon`(익명)은 함부로 못 버립니다 — 스왑이 없으면(컨테이너는 보통
스왑 off) **회수 불가**이고, 이게 OOM의 직접 원인이 됩니다.

### 한도를 넘으면 — reclaim → OOM kill → exit 137

`memory.max`를 넘어서는 할당이 일어나면 커널은 이 순서를 밟습니다.

```
할당 요청 → memory.current 가 memory.max 초과 예정
        │
        ├─ 1) 직접 회수(direct reclaim): 깨끗한 페이지 캐시 버림, 회수 가능한 슬랩 축소
        │        회수로 여유가 나면 → 할당 성공(단, 이 지연이 tail latency로 새어 나온다)
        │
        └─ 2) 회수해도 부족 → cgroup OOM killer 발동
                 → 이 cgroup 안에서 프로세스를 SIGKILL(9)
                 → 컨테이너 종료, 종료코드 137 = 128 + 9
                 → 쿠버네티스가 Reason: OOMKilled 로 기록, restartPolicy 따라 재시작
```

여기서 **메모리는 "압축 불가능(incompressible) 자원"**이라는 쿠버네티스의 대전제가 나옵니다.
CPU 한도를 넘으면 커널은 그냥 **스로틀(throttle)**, 즉 늦춰서 버팁니다(프로세스는 삽니다).
메모리 한도를 넘으면 늦출 방법이 없으므로 **죽입니다.** 이 비대칭이 5부 내내 따라다닙니다:
CPU 초과 = 느려짐, 메모리 초과 = 죽음.

### 쿠버네티스의 requests와 limits — 무엇이 cgroup의 무엇이 되나

| K8s 필드 | 의미 | cgroup v2로 내려가는 곳 | 초과 시 |
|---|---|---|---|
| `requests.memory` | 스케줄링 보증(이만큼은 준다) | (예약·스케줄러 계산용, 직접 강제 아님) | — |
| `limits.memory` | 절대 상한 | `memory.max` | **OOMKilled** |
| `requests.cpu` | CPU 배분 가중치 | `cpu.weight` | (경합 시 비율 배분) |
| `limits.cpu` | CPU 상한 | `cpu.max`(쿼터/주기) | **스로틀(throttle)** |

이 표가 5.2(빈패킹)의 출발점입니다. **스케줄러는 `requests`를 보고 노드에 파드를 채우고,
커널은 `limits`를 보고 죽일지 말지를 정합니다.** 런타임이 요구하는 메모리 여유가 이 두 숫자를
얼마나 부풀리는지가 곧 노드당 파드 밀도, 즉 비용이 됩니다.

## ③ 런타임은 컨테이너를 어떻게 (못) 보나 — 인식의 비대칭

### 문제의 뿌리: 런타임은 원래 '기계 전체'가 제 것인 줄 알았다

JVM도 Go 런타임도, 태생은 "한 대의 물리 서버 위에서 내가 주인"이라는 세계관입니다. 그래서
가용 메모리를 알아낼 때 `/proc/meminfo`를, CPU 개수를 알아낼 때 `/proc/cpuinfo`나
`sched_getaffinity`를 봤습니다. 그런데 이 값들은 **컨테이너 안에서도 여전히 호스트(노드)의
숫자**를 반환합니다. 노드가 64GiB·32코어면, 512MiB·0.5코어짜리 파드 안의 런타임도 "나 64GiB에
32코어 쓸 수 있네"라고 착각합니다.

이 착각의 대가는 둘입니다. 메모리 쪽에서는 힙을 노드 크기 기준으로 크게 잡아 **한도를 순식간에
넘겨 OOM**, CPU 쪽에서는 GC 스레드·스레드풀을 32개씩 만들어 **0.5코어 안에서 서로 스로틀**.
'컨테이너 인식(container awareness)'이란 결국 이 착각을 고쳐, 런타임이 `/proc/meminfo` 대신
**cgroup 한도**를 읽게 만드는 일입니다.

### JVM: CPU도 메모리도 자동으로 읽는다 (`UseContainerSupport`)

핫스팟 JVM은 `-XX:+UseContainerSupport`가 **기본값(true)**이고, 이게 켜져 있으면 부팅 시
cgroup을 감지해 CPU와 메모리 한도를 둘 다 읽습니다. 샌드박스에서 실제 감지 로그를 찍어
보면(측정: OpenJDK 11.0.31, `-Xlog:os+container=trace`) 이렇게 나옵니다.

```
[os,container] OSContainer::init: Initializing Container Support
[os,container] Detected cgroups v2 unified hierarchy
[os,container] Path to /memory.max is /sys/fs/cgroup/.../memory.max
[os,container] Raw value for memory limit is: max          ← 이 환경은 한도 없음
[os,container] Memory Limit is: Unlimited
[os,container] container memory limit unlimited: -1, using host value 4093874176
[os,container] OSContainer::active_processor_count: 4
```

여기서는 한도가 `max`(무제한)라 호스트 값(약 3.9GiB)으로 폴백했지만, `memory.max`에 숫자가
있으면 그 값을 "이 JVM이 아는 물리 메모리"로 삼습니다. 그다음 힙 상한을 이렇게 정합니다.

```
MaxHeapSize(힙 상한) = memory.max × MaxRAMPercentage
                     = memory.max × 25%   (기본값)
```

`MaxRAMPercentage`의 기본값과, 컨테이너가 작을 때 대신 쓰이는 `MinRAMPercentage`를 샌드박스에서
직접 재 봤습니다.

**측정: OpenJDK 11.0.31 (`-XX:+PrintFlagsFinal`), `MaxRAM`을 바꿔 가며 `MaxHeapSize` 관측**

| 컨테이너 메모리(MaxRAM) | 산정된 힙(MaxHeapSize) | 한도 대비 | 적용된 규칙 |
|---|---|---|---|
| 4 GiB | 1024 MiB | 25.0% | MaxRAMPercentage |
| 2 GiB | 512 MiB | 25.0% | MaxRAMPercentage |
| 1 GiB | 256 MiB | 25.0% | MaxRAMPercentage |
| 512 MiB | 128 MiB | 25.0% | MaxRAMPercentage |
| 200 MiB | 100 MiB | 50.0% | MinRAMPercentage |
| 128 MiB | 64 MiB | 50.0% | MinRAMPercentage |

관측된 규칙은 명확합니다(얼버무리지 않고 조건으로): **컨테이너 메모리가 약 256MiB를 넘으면
힙 = 25%(`MaxRAMPercentage`), 그 이하로 작아지면 힙 = 50%(`MinRAMPercentage`)**. 경계
근처(≈250–300MiB)엔 두 규칙이 갈아타는 전이 구간이 있습니다.

관련 플래그 기본값(측정: JDK 11.0.31 — `MaxRAMPercentage=25.0`, `MinRAMPercentage=50.0`,
`InitialRAMPercentage=1.5625`, `UseContainerSupport=true`). 이 기본값들은 JDK 25에서도
동일합니다(핫스팟 `gc_globals.hpp` 기준). 다만 **버전에 따라 달라질 수 있는 값이므로, 프로덕션
기준 JDK에서 `java -XX:+PrintFlagsFinal -version | grep RAMPercentage`로 재확인**하는 걸
전제로 두시길 바랍니다.

> **버전 주의(운영 함정)** — 커널 6.12 이후 일부 컨트롤러가 cgroup v1에서 v2로 이동하면서,
> 예전 JDK가 컨테이너 메모리 한도를 못 읽고 호스트 값으로 폴백해 OOM이 나는 사례가 보고됐고,
> 이 감지 버그는 JDK 25에서 고쳐졌습니다(OpenJDK 이슈 JDK-8348566). "예전엔 잘 되던 -Xmx
> 미지정 파드가 커널 올리고 나서 죽더라"의 전형적 배경입니다.

CPU도 같은 방식입니다. `UseContainerSupport`는 `cpu.max`(쿼터/주기)를 읽어 실효 코어 수
(`ActiveProcessorCount`)를 계산하고, 이 값이 **GC 병렬 스레드 수(`ParallelGCThreads`),
G1 동시 스레드(`ConcGCThreads`), JIT 컴파일러 스레드(`CICompilerCount`),
`ForkJoinPool.commonPool` 병렬도**까지 연쇄적으로 정합니다. 4부에서 본 GC 스레드 수가
"코어 수에서 파생된다"고 했던 그 코어 수가, 컨테이너에선 이 실효 코어 수입니다.

### Go: CPU는 배웠고(1.25+), 메모리는 아직 못 배웠다(수동)

Go는 최근에야, 그리고 **절반만** 컨테이너를 배웠습니다. 이 비대칭이 5.1에서 가장 중요한
사실입니다.

**CPU 쪽 — 배웠다.** Go 1.25부터 런타임이 리눅스 cgroup의 CPU 한도를 자동 인식해
`GOMAXPROCS`를 정합니다. `cpu.max`(v2) 또는 `cpu.cfs_quota_us`/`cpu.cfs_period_us`(v1)를 읽어
실효 CPU를 계산하고, 그 값이 머신 코어 수보다 작으면 `GOMAXPROCS`를 거기에 맞춥니다(올림).
컨테이너가 한도를 바꾸면 주기적으로 다시 읽어 갱신합니다. 끄려면
`GODEBUG=containermaxprocs=0`, 그리고 `GOMAXPROCS` 환경변수나 `runtime.GOMAXPROCS()`로 값을
직접 주면 자동 감지는 비활성화됩니다. (출처: Go 블로그 "Container-aware GOMAXPROCS",
go.dev/doc/go1.25)

**메모리 쪽 — 아직이다.** `GOMEMLIMIT`(소프트 메모리 한도, 1.19+)은 **cgroup을 자동으로
읽지 않습니다.** 2026-07 기준 최신인 **Go 1.26 릴리스 노트에도 이 자동화는 없습니다.** 즉
`GOMEMLIMIT`을 명시적으로 주지 않으면 기본값은 사실상 무제한(`math.MaxInt64`)이고, GC는
`GOGC=100` 규칙(힙이 라이브의 2배가 되면 수집)만으로 돕니다. 컨테이너 한도라는 개념이 Go
런타임의 메모리 결정에는 **전혀 안 들어옵니다.**

```
                    컨테이너 cgroup v2
         cpu.max = "50000 100000"      memory.max = 536870912
                 │(0.5코어)                   │(512MiB)
     ┌───────────┴───────────┐                │
     │                       │                │
   [JVM]                   [Go]                │
  Active                 GOMAXPROCS            │   ◀── CPU: JVM·Go 둘 다 자동으로 읽음
  ProcessorCount         (1.25+ 자동)          │
     │                       │                │
     ├───────────────────────┼────────────────┤
     │                       │                │
   [JVM]                   [Go]                │   ◀── MEMORY: 여기서 갈린다
  MaxRAMPercentage        GOMEMLIMIT           │
  25% × memory.max        미설정 = 무제한       │
  → 힙 자동 축소           → 한도 무시하고 성장   │
     │                       │                │
   한도 인지               한도 몰라서          │
   (OOM 회피)              커널이 먼저 죽임      ▼
                                          exit 137
```

왜 이게 위험한가는 물리로 설명됩니다. Go의 기본 GC는 "힙이 지난 수집 후 라이브의 2배가 되면
수집"합니다(GOGC=100). 라이브 집합이 300MiB인 워크로드는 힙을 600MiB까지 키운 **다음에야** GC를
돕니다. 그런데 컨테이너 `memory.max`가 512MiB라면? 힙이 512를 넘는 순간 커널이 먼저
SIGKILL합니다. **GC가 "이제 수집할 때"라고 느끼기도 전에 죽는** 겁니다. 4부에서 "할당이
싸구려인 대가는 GC로 청구된다"고 했는데, 컨테이너에선 그 청구서가 **GC가 아니라 커널의 OOM
killer**로 날아옵니다.

해법은 셋입니다. (a) `GOMEMLIMIT=450MiB`처럼 **직접** 주기 — 그러면 GC가 그 근처에서 훨씬
공격적으로 돌아 한도 안에 머뭅니다. (b) `KimMachineGun/automemlimit` 같은 라이브러리로 부팅 시
cgroup 한도의 약 90%를 `GOMEMLIMIT`으로 설정. (c) 매니페스트에서 downward API로 `limits.memory`를
환경변수로 받아 넣기. 참고로 이걸 런타임 기본값으로 만들자는 제안(golang/go #75164,
`GODEBUG=cgroupmemlimit`, 한도의 90%·최소 100MiB)이 GOMAXPROCS 방식을 그대로 본떠 논의
중이지만, **아직 채택·출시되지 않았습니다.**

### Rust: 이 질문 자체가 없다

Rust에는 GC가 없고 런타임도 없으니 "힙을 얼마로 잡을까"라는 질문이 성립하지 않습니다. 메모리
크기는 **소유권으로 이미 컴파일 타임에 결정**돼 있고, 실제 상한 관리는 애플리케이션의 몫입니다.
컨테이너 한도를 넘게 할당하면 Rust 프로세스도 물론 OOM-kill 당하지만, 그건 **런타임이 조절할
수 있는 손잡이가 없어서**가 아니라 애초에 그런 손잡이가 없기 때문입니다 — 대신 할당 지점이
코드에 명시적으로 드러나 있으니 사람이 제어합니다.

CPU 쪽은 흥미롭게도 표준 라이브러리가 컨테이너를 꽤 인식합니다. `std::thread::available_parallelism()`은
cgroup v2의 CPU 쿼터(`cpu.max`)를 `sched_getaffinity`에 섞어 반영합니다(rust-lang/rust
PR #92697). 다만 이건 **힌트를 주는 함수**일 뿐, 이걸 읽어 스레드풀을 몇 개로 잡을지는
`tokio`·`rayon` 같은 라이브러리와 개발자의 결정입니다. "런타임이 알아서"가 아니라 "라이브러리에게
정확한 숫자를 알려줄 준비는 돼 있다"는 쪽입니다.

## ④ 그래서 어떻게 설정하나 — 실무 매핑

핵심 원칙 하나로 요약됩니다. **`limits.memory`(커널이 죽이는 선)와 런타임의 힙 손잡이는 서로
다른 숫자이고, 그 사이 여백(native·페이지 캐시·스택)을 반드시 남겨야 한다.**

### JVM: `-Xmx` vs `MaxRAMPercentage`, 무엇을 쓸까

| 방식 | 특징 | 언제 |
|---|---|---|
| `-Xmx512m` (고정 절대값) | 컨테이너 한도와 무관하게 힙을 못 박음. `MaxRAMPercentage`를 **덮어씀**. | 한도가 고정이고, 힙/네이티브 비율을 손으로 정밀 제어할 때 |
| `-XX:MaxRAMPercentage=75.0` (비율) | `limits.memory`가 바뀌어도 **자동으로 따라감**(VPA·재조정 친화). | 한도가 변할 수 있고, 여러 파드에 같은 템플릿을 쓸 때 |

기본값 25%는 컨테이너 대부분에서 **너무 보수적**입니다(512MiB 파드에 힙 128MiB). 자바
프로세스는 힙 외 네이티브 소비가 크므로 통상 `MaxRAMPercentage`를 **50–75%** 사이로 올려
잡되, 나머지(25–50%)를 Metaspace·코드캐시·스레드 스택·다이렉트 버퍼·malloc arena 몫으로
남깁니다. 이 "나머지"가 정확히 5.3의 주제입니다. `-Xmx`를 쓰든 비율을 쓰든, **한도 전부를
힙에 주면 안 된다**는 게 요점입니다.

### Go: `GOMEMLIMIT`을 손으로, 단 GOGC와 함께

Go에서 컨테이너 메모리 안전은 **거의 항상 `GOMEMLIMIT`을 명시적으로 주는 것**에서 출발합니다.
권장 패턴은 `GOMEMLIMIT`을 한도의 약 90%로 두고(`limits.memory` 512MiB → `GOMEMLIMIT≈460MiB`),
`GOGC`는 그대로 100을 두는 것입니다. 그러면 평상시엔 GOGC 규칙으로 여유 있게 돌다가, 메모리가
한도 근처로 차오르면 `GOMEMLIMIT`이 GC를 강하게 당겨 한도를 넘지 않게 잡아 줍니다. `GOGC=off`로
두고 `GOMEMLIMIT`만으로 모는 방식도 있지만, 라이브 집합이 한도에 근접하면 GC가 쉼 없이 돌아
CPU를 태우는 **"GC 데스 스파이럴"** 위험이 있어 권하지 않습니다(4부 Mark Assist 논의의 연장).

### 남는 그림: 힙만 맞춰도 끝이 아니다 (→ 5.3 씨앗)

```
limits.memory = 512 MiB (memory.max, 커널이 죽이는 선)
├────────────────────────────────────────────────────────┤
│  힙 (JVM -Xmx / Go GOMEMLIMIT 로 조절 가능한 부분)         │  ← 우리가 흔히 신경 쓰는 곳
├──────────────────────────┐                               │
│                          │  ← 여기가 '보이지 않는 여백'    │
│  Metaspace · 코드캐시      │     힙 손잡이로는 안 줄어든다    │
│  스레드 스택 · 다이렉트버퍼 │     [복선 확인 필요: 1.3·2.4·3.4]│
│  malloc arena · 페이지캐시 │                               │
└──────────────────────────┴───────────────────────────────┘
      이 여백이 부족하면: "GC 로그는 깨끗한데 파드가 죽는다" → 5.3
```

`[복선 확인 필요: 1.3·2.4·3.4]` 표시를 남겨 둡니다. 1.3(Direct Memory), 2.4(cgroup이 힙+네이티브
합산), 3.4(힙 밖 지형)에서 이 여백을 여러 번 스쳤다고 알고 있지만, 실제 원고에서 그 대목을
확인한 뒤 이어 주세요. 확인이 안 돼도 이 절과 5.3은 그 자체로 성립합니다.

## ⑤ 세 언어 수렴 — 'CPU 인식'과 '메모리 인식'은 다른 문제다

같은 주제(컨테이너 한도를 런타임이 아는가)를 세 언어로 수렴시키면 이렇게 갈립니다.

| | CPU 한도 인식 | 메모리 한도 인식 | 층위 |
|---|---|---|---|
| **Rust** | `available_parallelism()`이 cgroup v2 쿼터 반영(힌트). 실제 스레드풀은 tokio/rayon 몫 | GC 없음 → "힙 크기" 질문 자체가 없음. 상한은 소유권+앱이 관리 | 라이브러리/앱 |
| **Go** | `GOMAXPROCS` 자동(1.25+, 기본 on) ✅ | `GOMEMLIMIT` **수동** — 1.26에도 cgroup 자동 아님 ❌ | 런타임(반쪽) |
| **Java** | `ActiveProcessorCount` 자동(`UseContainerSupport`) ✅ | `MaxRAMPercentage`로 힙 자동 산정(기본 25%) ✅ | 런타임(양쪽) |

세로로 읽으면 4부의 삼각형이 클라우드 축에서 다시 보입니다.

- **Java**는 런타임이 CPU·메모리 둘 다 알아서 맞춥니다. 편하지만, 그 자동값(25%)이 워크로드에
  안 맞으면 오히려 힙이 너무 작아 GC가 잦아지는 식으로 조용히 손해를 봅니다. **"알아서 해 주니
  안 건드림"의 대가.**
- **Go**는 CPU만 배운 과도기입니다. `GOMAXPROCS`는 이제 안심해도 되지만, `GOMEMLIMIT`은
  **여전히 사람이 넣어야** 합니다. 이걸 모르면 "Go는 메모리 가벼운 언어인데 왜 파드가 자꾸
  죽지?"에 빠집니다. **가장 흔한 프로덕션 함정.**
- **Rust**는 런타임이 없으니 인식할 주체도 없습니다. 대신 메모리가 소유권으로 결정돼 있어
  애초에 폭주할 여지가 적고, CPU 병렬도는 라이브러리에 정확한 숫자를 넘겨줄 준비만 돼
  있습니다. **손잡이가 없는 게 아니라, 손잡이가 코드 안에 있는 것.**

## ⑥ 트레이드오프 소결

한 문장으로: **커널의 메모리 회계(전부 합산)와 런타임의 힙 회계(일부만 조절)는 층위가 다르고,
이 둘을 잇는 다리를 언어마다 다르게 놨다.** Java는 런타임이 다리를 자동으로 놓아 주고(대신
자동값이 최적은 아니고), Go는 CPU 다리만 놓고 메모리 다리는 사람에게 맡기고(가장 사고 나기
쉬운 지점), Rust는 다리라는 개념 없이 애초에 소유권으로 강을 건너 둡니다.

그리고 어느 언어든 공통으로 남는 진실 하나 — **힙을 한도에 완벽히 맞춰도, 커널이 세는 건 힙만이
아니므로 여백이 없으면 죽습니다.** 이 여백의 물리를 5.3에서 해부합니다.

## ⑦ DevOps 관점 심화 질문

1. **`memory.stat`으로 범인 찾기** — 파드의 `memory.current`는 한도에 붙어 있는데 애플리케이션이
   보고하는 힙은 절반뿐입니다. `anon`/`file`/`slab`/`sock` 중 무엇이 부풀었는지 `memory.stat`
   으로 어떻게 좁혀 들어가겠습니까? `file`(페이지 캐시)이 큰 경우와 `anon`이 큰 경우, 대응이
   어떻게 달라져야 합니까?

2. **requests와 limits를 같게 vs 다르게** — 메모리 `requests < limits`로 두면 빈패킹은
   빽빽해지지만 노드 메모리 초과 시 어떤 파드가 먼저 OOM 됩니까(QoS class: Guaranteed vs
   Burstable)? 안정성이 중요한 서비스에서 둘을 같게 두는 이유를 cgroup 관점에서 설명해
   보십시오.

3. **`GOMEMLIMIT` 주입 자동화** — 오토스케일러(VPA)가 파드의 `limits.memory`를 동적으로 바꾸는
   환경에서, `GOMEMLIMIT`을 매니페스트 환경변수로 박아 두면 어떤 문제가 생깁니까? downward API,
   `automemlimit`, (미래의) `GODEBUG=cgroupmemlimit` 중 무엇을 택하겠습니까?

4. **CPU 스로틀과 GC의 상호작용** — `limits.cpu`로 스로틀이 걸리는 파드에서 GC 스레드가
   실행될 CPU 시간을 못 얻으면 무슨 일이 벌어집니까? `GOMAXPROCS`/`ActiveProcessorCount`가
   컨테이너 값으로 줄어드는 것이 GC 지연에 오히려 유리한 이유는 무엇입니까?

5. **`-Xmx` 못 박기 vs 비율** — 같은 이미지를 128Mi·512Mi·2Gi 파드에 배포한다면 `-Xmx` 고정과
   `MaxRAMPercentage` 중 무엇이 운영상 안전합니까? `MinRAMPercentage`(50%) 전환 구간(≈256MiB)이
   작은 파드에서 왜 함정이 되는지 설명해 보십시오.

6. **계측 파이프라인** — "OOMKilled가 늘었다"는 알람을 받았을 때, cgroup `memory.events`의
   `oom`/`oom_kill` 카운터, 커널 `dmesg`의 OOM 리포트, 쿠버네티스 이벤트, 런타임 GC 로그를
   각각 어떤 순서로 확인해 원인을 "힙 초과 / native 누수 / 한도 과소설정" 중 하나로 좁히겠습니까?

## ⑧ 다음 절: 5.2 빈패킹의 경제학

5.1에서 런타임이 요구하는 메모리 **여유(headroom)**가 어디서 오는지를 봤습니다. Java는 힙 밖
여백, Go는 GOGC가 만드는 라이브의 2배, ZGC는 복사용 빈 page — 이 여유가 그대로 파드의
`requests`/`limits`가 되고, 그 숫자가 **노드에 파드가 몇 개 들어가는가**를 정합니다. 5.2는 이
여유를 비용으로 환산합니다. "GC가 편하자고 남겨 둔 2배의 메모리가, 노드 열 대에서 몇 대분의
청구서가 되는가."
