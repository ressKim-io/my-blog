# 1부 · 1.3 런타임(Runtime)의 무게: Zero-Cost (Rust) vs Fat Runtime (Go, Java)

> **이 문서의 성격**: 최종 집필 에이전트에게 넘길 **1차 심화 정리(reference-level draft)**.
> 1.1(관문 비용)·1.2(그 관문을 넘는 스레드 모델)에서 "런타임이 개입하는 정도"가 언어마다 다름을 봤다. 이 절은 **그 런타임 자체의 무게가 얼마이고 무엇으로 이루어지는가**를 해부한다. 1.2에서 미뤄둔 "Rust가 그린 스레드를 버린 이유"가 여기서 완결된다.
> 그림은 ASCII, 코드는 개념 예시. **기준 버전(2026년 7월)**: JDK 25(LTS, Project Leyden AOT 캐시 포함), Go 1.25, GraalVM(native-image).

---

## 0. 이 절을 읽고 나면 이해하게 되는 범위

- "런타임"이라는 단어의 정확한 의미와, 그 안에 무엇이 들어갈 수 있는가
- "무겁다"를 **5개 축**(바이너리 크기 / 베이스라인 메모리 / 시동 시간 / 정상상태 오버헤드 / 예측가능성)으로 분해하는 법
- Rust의 **제로코스트**가 "런타임을 거의 두지 않는다"는 뜻임을, 그리고 그 대가
- Go의 런타임이 **Fat이지만 자족적(self-contained)** 인 이유
- JVM이 **가장 무겁지만 가장 영리한** 런타임인 이유, 그리고 지금 **Project Leyden·GraalVM으로 스스로 무게를 되돌리려는** 흐름
- "런타임에 박았나(Go) vs 라이브러리로 뺐나(Rust)"가 왜 제로코스트 vs Fat Runtime 철학의 가장 순수한 사례인가
- 이 무게 차이가 컨테이너 이미지·콜드 스타트·베이스라인 RSS로 이어지는 지점 (5부 복선)

## 1. "런타임"이란 정확히 무엇인가

용어가 두 가지로 혼용된다. 구분하고 가자.

- **런타임 라이브러리(runtime library)**: `printf`, 문자열, 컬렉션처럼 표준 라이브러리 함수들. 이건 모든 언어에 있다.
- **언어 런타임(language runtime)**: 개발자가 부르지 않아도 **프로그램과 함께/아래에서 자동으로 도는 관리 코드**. 이 절의 주제는 이쪽이다.

언어 런타임에 **들어갈 수 있는 것들:**

```
 애플리케이션 코드 (내가 짠 것)
 ────────────────────────────────
 │  ▲ 이 아래가 '언어 런타임'                     ┌ 무게의 정체 ┐
 │  • 메모리 관리: GC, allocator                  │             │
 │  • 스케줄러: 고루틴/가상스레드 M:N             │  이 층이     │
 │  • JIT 컴파일러 (실행 중 기계어 생성)          │  두꺼울수록  │
 │  • 리플렉션 / 런타임 타입정보(RTTI)            │  '무거운     │
 │  • 예외 언와인딩 / 패닉 처리                    │  런타임'     │
 │  • 동적 디스패치(vtable) / 인터페이스           │             │
 │  • 클래스 로더 / 바이트코드 인터프리터          └─────────────┘
 ────────────────────────────────
 OS 커널 (1.1의 세계)
```

핵심 질문은 이거다: **이 중간 층에 무엇을, 얼마나 넣었는가?** Rust는 거의 비웠고, Go는 실속 있게 채웠고, JVM은 가상머신 하나를 통째로 넣었다.

## 2. 무게를 재는 축 — "무겁다"의 5가지 의미

"무겁다"를 뭉뚱그리면 오해가 생긴다. 다섯 축으로 분해해야 한다.

```
 축                     Rust      Go        JVM(전통)   JVM(native-image)
 ─────────────────      ────      ────      ─────────   ─────────────────
 ① 바이너리/이미지 크기  아주작음  중간      큼(+JRE)    작음
 ② 베이스라인 RSS       아주작음  작음      큼          작음
 ③ 시동(cold start)     ms        ms        초(warmup)  ms
 ④ 정상상태 오버헤드    거의0     GC+스케줄  GC+JIT스레드 GC(있음)
 ⑤ 정점 처리량(peak)    높음      중상      최고(JIT)   중(-10~25%)
```

무게 스펙트럼 한 장:

```
 가벼움 ◄──────────────────────────────────────────────► 무거움
 Rust(no_std) < Rust(std) < Go < JVM(native-image) < JVM(전통 JIT)

 ※ 단 ⑤ 정점 처리량은 이 순서와 반대로 뒤집힐 수 있다(오른쪽이 유리).
   → 무게는 '죄'가 아니라 '교환'이라는 게 이 절의 결론이자 2부의 씨앗.
```

이 표에서 이미 보이는 반전: **가장 무거운 JVM(전통)이 ⑤ 정점 처리량에선 앞선다.** 무게를 지불하고 런타임 적응성을 산 것 — 이게 2부·4부의 큰 줄기다.

## 3. Rust — Zero-Cost: "런타임이 거의 없다"

### 3.1 제로코스트 추상화 원칙

C++에서 온 원칙을 Rust가 채택했다:

> **"쓰지 않는 것에는 비용을 내지 않는다. 쓰는 것은 손으로 짜도 그보다 더 잘 짤 수 없다."**

즉 추상화(제네릭, 이터레이터, async)가 **런타임 비용을 0으로** 컴파일돼야 한다. 이 원칙이 Rust에 **관리형 런타임을 두지 못하게** 강제한다.

### 3.2 Rust "런타임"의 실체 — 거의 빈 칸

```
 애플리케이션 코드
 ────────────────────────────────
 │  Rust 런타임:                    ← 매우 얇음
 │   • main 전 lang_start 셤:
 │       - 스택 오버플로 가드 페이지 설정
 │       - 패닉 핸들러/언와인딩 준비
 │       - 인자/TLS 초기화
 │   • 그게 (거의) 전부.
 │   • GC 없음. M:N 스케줄러 없음. JIT 없음.
 ────────────────────────────────
 OS 커널
```

- **메모리 관리를 컴파일 타임으로 옮겼다.** 소유권(ownership)·빌림(borrow) 검사기가 "언제 해제할지"를 컴파일 때 결정 → `drop`이 정해진 지점에 삽입됨(RAII). **런타임 GC가 필요 없다.** (3부에서 malloc/free 라이프사이클로 이어짐)
- **제네릭은 monomorphization**: `Vec<T>`는 사용된 타입마다 전용 코드로 복제된다. 가상 디스패치(vtable)는 `dyn Trait`를 명시할 때만. → 추상화가 인라인되어 사라진다.
- **`#![no_std]`**: 표준 라이브러리(힙 할당·OS 의존)까지 걷어내면 베어메탈에서 돈다. 임베디드 펌웨어, 부트로더, **eBPF, 커널 모듈, OS 커널**(Rust for Linux)까지 Rust가 들어가는 이유.

### 3.3 [1.2 완결] Rust가 그린 스레드를 버린 이야기

1.0 이전(2014)의 Rust는 **libgreen(M:N 그린 스레드) + libnative(1:1)** 를 둘 다 가졌다. 하지만:

- 그린 스레드 스케줄러는 **모든 Rust 프로그램에 무거운 런타임을 강제**했다 — 제로코스트 위반.
- **FFI(C 호출) 때마다 스택 전환** 비용이 붙었다. Rust를 C 라이브러리/OS에 심으려는 목표와 정면충돌.
- 결론: **RFC 230(2014)** 으로 그린 스레드를 언어에서 제거. **1:1 네이티브 스레드를 기본**으로 하고, M:N이 필요하면 **라이브러리(tokio 등)로 각자 선택**하게 했다.

→ 그래서 1.2에서 본 "Rust std = 1:1, async = 라이브러리" 구도는 **이 제로코스트 결정의 직접적 귀결**이다. 여기서 1.2의 실이 매듭지어진다.

### 3.4 대가

가벼움은 공짜가 아니다. Rust는 무게를 **개발 시점의 복잡도**로 지불한다 — borrow checker와 씨름, 긴 컴파일 시간, async 함수 색칠, 수동 메모리 수명 관리. (7부 인건비 폭발 논의의 근거)

## 4. Go — Fat이지만 Self-Contained

Go는 Rust와 반대 선택을 했다. **실속 있는 런타임을 바이너리에 정적으로 박아 넣었다.**

```
 Go 바이너리 하나 안에:
 ────────────────────────────────
 │  애플리케이션 코드
 │  ── Go 런타임 (정적 링크) ──
 │   • GMP 스케줄러 (1.2)
 │   • GC: 동시 3색 마크-스윕 (4부에서 딥다이브)
 │   • allocator: TCMalloc 계열 mcache/mcentral (3부)
 │   • 고루틴 스택 관리(성장/축소)
 │   • netpoller(epoll) (1.2)
 │   • sysmon: 백그라운드 감시 스레드(선점·GC 트리거)
 ────────────────────────────────
 → 결과물: 외부 의존 없는 단일 실행 파일
```

- **무게**: `hello world` 정적 바이너리가 대략 **1.5~2MB**. 런타임을 통째로 담기 때문. GC 고루틴·sysmon이 앱과 **동시에** 돈다(정상상태 오버헤드 ④ 존재).
- **그러나 자족적**: 외부 라이브러리·인터프리터 불필요 → 컨테이너에서 `FROM scratch`/distroless로 **수 MB 이미지**. AOT 컴파일이라 **웜업 없이 ms 단위 시동**(③ 유리).

```dockerfile
# Go: 런타임이 바이너리에 있으니 베이스 이미지가 필요 없다
FROM scratch
COPY app /app
ENTRYPOINT ["/app"]     # 최종 이미지 ≈ 바이너리 크기(수 MB)
```

- **철학**: Rust만큼 가볍진 않지만, "충분히 가볍고 + 자족적 + 빠른 시동 + GC의 편안함". 이 **균형점**이 6부에서 볼 클라우드 네이티브 인프라(Docker/K8s)가 Go를 택한 이유다.

## 5. JVM — 가장 무겁고 가장 영리한 런타임

JVM은 런타임이 아니라 **가상 머신 하나를 통째로** 얹은 것이다.

```
 JVM 프로세스 메모리 구성 (앱 힙은 일부일 뿐)
 ────────────────────────────────
 │  Java Heap (내 객체들) ←── 흔히 이것만 계산하다 5.2에서 터진다
 │  ─────────────────────── ↓ 여기부터 전부 Off-Heap(-Xmx 밖) ───
 │  Metaspace              : 클래스 메타데이터
 │  JIT Code Cache         : C1/C2가 생성한 기계어
 │  Thread Stacks          : 스레드마다 -Xss
 │  Direct Memory(NIO/Netty): 힙 밖 DirectByteBuffer ←── -Xmx에 안 잡힘! K8s OOMKilled 단골
 │  GC / Compiler Threads  : 백그라운드 작업 스레드
 │  ── JVM 엔진 ──
 │   • 클래스 로더 + 바이트코드 인터프리터
 │   • C1/C2 JIT (계층형 컴파일, 2부)
 │   • GC: G1/ZGC/Parallel (4부)
 ────────────────────────────────
```

- **바이트코드 실행**: `.class`는 네이티브가 아니라 JVM용 중간 코드. 실행하려면 JVM 프로세스가 필요하다.
- **베이스라인 메모리 ②가 큼**: 내 힙이 0이어도 Metaspace·코드캐시·스레드스택·**Direct Memory(NIO/Netty)**·GC/컴파일러 스레드가 먼저 자리를 먹는다. 특히 `DirectByteBuffer`는 **`-Xmx` 밖(off-heap)** 이라 힙 모니터링엔 안 잡히고 RSS만 조용히 부풀다 컨테이너 리밋에 걸린다. 게다가 `-XX:MaxDirectMemorySize`를 안 주면 기본 상한이 대략 `-Xmx`와 같아, **"힙 3GB면 다이렉트도 최대 3GB 더"** 라는 이중 함정이 된다. → **"힙만 계산하면 K8s에서 OOMKilled"** 라는 5.2의 근원.
- **웜업 ③**: 처음엔 인터프리트 → 자주 도는 코드를 C1 → 더 뜨거우면 C2로 재컴파일. **정점 성능까지 시간이 걸린다.**(2.3·5.3 복선)
- **그런데 이 무게가 사는 것**: 실행 중 프로파일을 보고 **런타임에 기계어를 다시 짜는 적응성(JIT 재최적화)**, 20년 쌓인 생태계, 세계 최고 수준의 GC들. → 무게 ↔ 적응성의 교환.

### 5.1 [최신] JVM이 자기 약점을 공격한다 — Leyden & GraalVM

JVM의 두 약점(무거운 시동·웜업)을 정면으로 줄이는 두 흐름:

- **Project Leyden (AOT 캐시, JVM은 유지)**:
  - **JEP 483(JDK 24)**: 클래스 로딩·링킹을 "훈련 실행(training run)" 때 미리 해서 캐시 → 프로덕션 시동 때 반복 안 함.
  - **JEP 514/515(JDK 25)**: 워크플로우 단순화 + **AOT 메서드 프로파일**을 캐시해 JIT 웜업까지 앞당김.
  - 효과 예: Spring PetClinic 시동 **4.486s(JDK23) → 2.604s(JDK24, AOT 캐시)**, 약 **42% 단축**. JVM을 버리지 않고 개선.
- **GraalVM native-image (JVM을 버리고 AOT)**:
  - **얻는 것(런타임)**: closed-world 가정으로 **미리 네이티브 실행 파일**을 만든다. 시동 **3~5초 → 50~200ms(10~50배)**, 메모리 **약 4배 절감(400MB → 100MB급)**. 즉 **실행은 빠르다 — 그게 목적이다.**
  - **⚠ 진짜 대가는 '빌드타임'이다 (실행이 아니라)**: points-to 정적 분석 + 전체 AOT 컴파일이라 **빌드가 분 단위(수 분~10분+)** 로 오래 걸리고 **RAM을 폭식**한다. CI에선 시스템 메모리의 **85%(최대 30GiB)** 까지 쓰고, 메모리가 부족하면 빌드 도중 GC로 **시간의 절반 이상**을 날리기도 한다. → CI/CD가 느려지고 개발 반복(iteration) 속도가 떨어진다. **"런타임은 ms, 빌드타임은 분."**
  - **closed-world 제약**: 리플렉션·동적 프록시·JNI·리소스·직렬화는 **reachability 메타데이터(힌트)** 를 명시해야 바이너리에 포함된다. "JVM에선 되는데 네이티브에선 `ClassNotFound`로 깨짐" 류의 디버깅이 생긴다. Spring AOT·Quarkus·Micronaut가 힌트를 자동 생성해 완화하지만 **제약 자체는 남는다.**
  - **GC·관측 제한**: 기본이 **Serial GC**(작은 힙·낮은 풋프린트용). **G1은 Community Edition에 없고 Linux AMD64/AArch64만**, **ZGC는 미지원** → **대용량 힙·저지연 워크로드엔 오히려 불리**. 일부 JVM 에이전트/툴링도 제약.
  - **정점 처리량**: JIT보다 **10~25% 낮음**(런타임 적응 포기). **PGO**를 먹이면 격차의 30~50%를 회복(정점 대비 5~15% 이내).
  - **정리**: native-image는 "전부 해결"이 아니라 **런타임 비용(시동·풋프린트)을 빌드타임·유연성·대용량 힙 GC와 맞바꾼 것**이다. 자주 죽고 켜지며(서버리스·CLI) 힙이 작을 땐 순이득, **대용량 힙으로 롱런하는 서버엔 오히려 손해**일 수 있다.

> 이 두 흐름이 곧 2부(AOT vs JIT)와 5부(콜드 스타트·MSA 경제학)의 정확한 예고편이다. 다만 어느 쪽도 은탄환이 아니다 — Leyden은 훈련 실행 부담을, native-image는 빌드타임·유연성 비용을 대신 문다. JVM은 "무거움"을 **공짜가 아니라 '선택지'** 로 만들려 하고 있을 뿐이다.

## 6. [정면 비교 박스] 런타임에 박았나(Go) vs 라이브러리로 뺐나(Rust)

같은 문제(수백만 I/O 태스크를 소수 스레드로 굴리기)를 두 언어가 **런타임 무게 철학대로** 반대로 풀었다. 이게 이 절의 정수다.

```
 Go (통합 Fat 런타임)                    Rust + tokio (제로코스트 유저랜드 조립)
 ────────────────────                    ──────────────────────────────────
  goroutine (스택풀 코루틴)               async task = Future (스택리스 상태기계)
        │                                       │
  ┌────────────────┐                      ┌────────────┐  Waker  ┌────────────┐
  │ GMP 스케줄러    │  ← 런타임에 내장     │  Executor   │◄──────▶│ Reactor(mio)│
  │ netpoller(epoll)│                      │ (워커풀/steal)│        │  epoll      │
  │ (둘 다 언어 안) │                      └────────────┘         └────────────┘
  └────────────────┘                      ← 언어 런타임은 비어 있고, 둘을 라이브러리로 조립
        │                                     Waker가 유일한 이음새 → 통째로 교체 가능
        ▼                                       ▼
  OS 스레드(M) ≈ 코어 수                    OS 워커 스레드 ≈ 코어 수
```

- **Go**: 스케줄러+netpoller가 **런타임에 박혀** 있다. 개발자는 조립할 게 없고 편하다. 대신 그 무게(GC·런타임)를 모든 바이너리가 진다. 아키텍처를 **바꿀 수 없다**.
- **Rust**: 언어 런타임을 **비워 두고** tokio(Executor) + mio(Reactor)를 유저랜드에서 조립한다. `Waker`라는 이음새 덕에 **런타임을 통째로 교체**할 수 있다:
  - 예: **glommio** — work-stealing 대신 **thread-per-core + io_uring** 아키텍처. 코어 간 공유·동기화를 없애 캐시 지역성을 극대화. **같은 언어에서 런타임 철학 자체를 갈아끼우는 것** — Go에선 불가능한 자유.

한 줄: **Go는 "정답을 하나 정해서 박았고", Rust는 "런타임을 비워 선택지를 열었다."** 이것이 Fat Runtime ↔ Zero-Cost의 가장 순수한 표현이다. (스택풀↔스택리스, 선점↔협력의 3축 차이는 1.2 참조)

## 8. 트레이드오프 소결 — 무게는 죄가 아니라 교환

- **Rust**: 런타임을 비워 **가벼움·예측가능성·임베더빌리티**를 얻고, 그 값을 **개발 복잡도**로 낸다.
- **Go**: 실속 런타임을 박아 **자족성·생산성·빠른 시동**을 얻고, **Rust보다 무겁고 GC가 있다**는 값을 낸다. 균형점.
- **JVM**: 가상머신을 통째로 얹어 **런타임 적응성(JIT)·최고 정점 처리량·생태계**를 얻고, **최대 무게·웜업**을 낸다. 그리고 지금은 Leyden/Graal로 **그 무게조차 선택지로 되돌리는** 중.

핵심: 무게 자체가 나쁜 게 아니다. **"이 무게로 무엇을 샀는가"** 를 물어야 한다. 다음 2부는 그 대표 구매품 — **실행 중 기계어를 다시 짜는 JIT의 적응성**이 정말 정적 컴파일을 이길 수 있는지를 판별한다.

## 9. DevOps 관점에서 더 파고들 만한 질문들

1. **컨테이너 이미지 다이어트**: Go/Rust는 `FROM scratch`로 수 MB, JVM은 JRE 베이스가 필요하다. jlink로 커스텀 런타임을 깎거나 GraalVM native-image로 갈 때, 이미지 크기·빌드 복잡도·디버깅 난이도의 교환은?
2. **콜드 스타트와 서버리스**: Lambda/Knative처럼 자주 죽고 켜지는 환경에서 JVM 웜업(초)은 치명적이다. Leyden AOT 캐시(42% 단축)로 충분한가, 아니면 native-image(50~200ms)까지 가야 하나? (5.3 직결)
3. **베이스라인 RSS와 빈패킹**: 200MB 리밋 파드에 Spring Boot(JVM 350~550MB)를 못 올리는 이유가 힙이 아니라 Metaspace·코드캐시·스레드스택이라면, `request/limit`을 어떻게 잡아야 노드에 촘촘히 담기나? (5.1·5.2 직결)
4. **native-image의 함정**: closed-world 가정 때문에 리플렉션·동적 프록시(스프링 등)가 깨진다. 힌트 설정 비용 ↔ 처리량 -10~25% ↔ 시동 이득, 어느 워크로드에서 순이득인가?
5. **Leyden training run의 CI/CD 부담**: AOT 캐시는 "대표 워크로드로 훈련 실행"을 전제한다. 이 훈련 단계를 파이프라인에 넣는 비용과, 캐시가 실제 트래픽과 어긋날 때의 리스크는?
6. **멀티아치(ARM) 배포**: Go/Rust는 arm64로 크로스 컴파일이 쉬워 Graviton용 멀티아치 이미지를 쉽게 만든다. JVM 바이트코드는 아키텍처 중립이지만 JIT는 arch별 기계어를 낸다 — native-image는 타깃 arch별로 따로 빌드해야 한다. Graviton 전환 시 세 언어의 빌드/배포 파이프라인 차이는?
