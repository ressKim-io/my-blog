## 슬롯 회전 속도 비교 (핵심 지표)

| VU | 수연 | 준상 | 성전 | 최고 |
|----|------|------|------|------|
| **300 iterations** | - | **4,476** | 457 | ⭐ 준상 |
| **300 iter/s** | - | **11.84** | 1.21 | ⭐ 준상 |
| **1000 iterations** | **5,495** | 3,713 | 417 | ⭐ 수연 |
| **1000 iter/s** | **14.5** | 9.84 | 1.10 | ⭐ 수연 |
| **3000 iterations** | **6,574** | 763 | 290 | ⭐ 수연 |
| **3000 iter/s** | **12.2** | 2.0 | 0.77 | ⭐ 수연 |

> - 슬롯 회전 = 예매 완료 → leave → 다음 사람 입장 사이클
> - **수연 >> 준상 >> 성전** 순서
> - 성전님의 iterations가 적은 이유:
>   1. Heartbeat 오버헤드 (매 10초마다 추가 요청)
>   2. 이벤트 기반 승격의 비동기 지연 (Scheduler 없이 KeyExpiration 의존)
>   3. @RequestParam 방식의 토큰 전달 (URL 길이 제한 없지만 인코딩 오버헤드)

## 승격 메커니즘 비교

| 항목 | 수연 (방식3) | 준상 (방식1) | 성전 (방식2) |
|------|-------------|-------------|-------------|
| **승격 트리거** | status API 호출 시 동적 계산 | SCAN 기반 (→ O(1) 개선) | 이벤트 기반 (leave → 분산락 → allowedNum) |
| **Heartbeat** | 없음 (polling만) | 없음 | 있음 (10초 간격, TTL 30초) |
| **만료 처리** | Scheduler 30초 주기 | Redis TTL | Redis Key Expiration Event |
| **토큰** | JWT (JWE AES-256-GCM) | JWT | Custom TokenEncryptor (@RequestParam) |
| **분산 락** | 모든 쓰기 | - | 이탈 이벤트 처리 |
| **장점** | 처리량 최고, Scheduler 안정 | 전환 간단, 응답 빠름 | 응답 최저, 에러 0, 중간값 대기 최단 |
| **약점** | 3000 VU에서 통과율 58% | 3000 VU에서 응답 급증 (3.1s) | 슬롯 회전 느림, 꼬리 대기 길음 |

## 종합 순위 (대기열 성능 기준)

| 카테고리 | 1위 | 2위 | 3위 |
|----------|-----|-----|-----|
| **API 레이턴시** | ⭐ 성전 | 준상 | 수연 |
| **대기열 통과율** | ⭐ 준상/성전 (100%) | - | 수연 |
| **대기 시간 (중간값)** | ⭐ 성전 | 준상 | 수연 |
| **대기 시간 (꼬리)** | ⭐ 수연/준상 | - | 성전 |
| **슬롯 회전 (처리량)** | ⭐ 수연 | 준상 | 성전 |
| **안정성 (에러율)** | ⭐ 성전 | 준상 | 수연 |
| **스케일링 내성** | ⭐ 성전 (레이턴시 유지) | 수연 (처리량 유지) | 준상 (3000에서 급증) |

### 한 줄 요약
- **성전**: 가장 안정적이고 빠르게 응답하지만, 슬롯 회전이 느려 총 처리 건수가 적음
- **수연**: 가장 많은 예매를 처리하지만, 3000 VU에서 대기열 통과율이 낮아짐
- **준상**: 균형형 — 1000 VU까지 좋지만 3000 VU에서 API 레이턴시 급증
