# 1부 · 1.1 유저 모드와 커널 모드: 제어권 전환(System Call)의 숨은 비용

> **이 문서의 성격**: 최종 블로그 원고가 아니라, 최종 집필 에이전트에게 넘길 **1차 심화 정리(reference-level draft)** 입니다.
> 개념 → 하드웨어 동작 → 비용 해부 → 회피 전략 → 3개 언어 연결 → 트레이드오프 → 심화 질문 순으로 쌓아 올립니다.
> 그림은 모두 ASCII. 수치는 측정 예시이며 CPU 세대/커널 설정에 따라 달라짐을 명시합니다.

---

## 0. 이 절을 읽고 나면 이해하게 되는 범위

- CPU가 **왜** 두 개의 실행 모드를 하드웨어로 강제하는가 (보호 링)
- `syscall` 명령어 한 줄이 실제로 하드웨어에서 무엇을 하는가 (MSR, 레지스터 규약)
- 리눅스 커널 진입/복귀 경로가 밟는 단계들 (swapgs → CR3 스왑 → 레지스터 저장 → 투기적 실행 완화 → 실제 syscall)
- syscall 비용이 **직접 비용(고정)** 과 **간접 비용(마이크로아키텍처 교란)** 으로 나뉘는 이유
- KPTI(멀티다운 완화)가 왜 syscall마다 세금을 매기는가
- Rust / Go / Java가 이 "커널의 관문"을 각각 어떻게 대하는가 → **이 차이가 2·3부의 성능 서사 전체의 씨앗**

## 1. 왜 두 개의 모드가 존재하는가 — 특권 분리 (x86 보호 링 & ARM 예외 레벨)

애플리케이션이 디스크 컨트롤러에 직접 명령을 쏘거나, 임의 물리 메모리를 읽거나, 페이지 테이블을 바꿀 수 있다면 OS의 격리는 존재할 수 없다. 그래서 **모든 현대 CPU는 특권 분리를 하드웨어로 강제한다.** 다만 이름과 계층 구조는 아키텍처마다 다르다 — x86-64는 **보호 링(CPL, Current Privilege Level)**, ARM(AArch64)은 **예외 레벨(EL, Exception Level)** 이라 부른다. 클라우드에서 ARM 서버(AWS Graviton, Ampere Altra, Apple Silicon)가 주류가 된 지금은 두 모델을 같이 봐야 한다.

```
 x86-64 (보호 링 / CPL)              ARM AArch64 (예외 레벨 / EL)
 ─────────────────────              ────────────────────────────
 Ring 3  User    (CPL=3) ◄────────► EL0  User (앱: Rust/Go/Java)
 Ring 2  (거의 미사용)               EL1  OS 커널
 Ring 1  (거의 미사용)               EL2  하이퍼바이저
 Ring 0  Kernel  (CPL=0) ◄────────► EL3  시큐어 모니터(펌웨어/TrustZone)

 • 실질적으로 Ring 0/3 두 개만 사용   • EL0~EL3 계층을 그대로 활용
 • 가상화는 VMX root/non-root라는     • 가상화(EL2)·시큐어월드(EL3)가
   '별도의 축'으로 뒤늦게 얹음          권한 사다리에 처음부터 내장
```

두 모델의 공통점과 차이:

- **공통**: 앱은 최하위 권한(Ring 3 / EL0)에서 돈다. 커널은 특권 레벨(Ring 0 / EL1)에서 돌며 하드웨어를 독점한다. 앱이 특권 작업을 하려면 반드시 커널에게 요청해야 한다.
- **특권 명령 위반 시**:
  - x86: `HLT`, `MOV to CR3`, `IN/OUT`(포트 I/O) 등을 Ring 3에서 실행하면 **General Protection Fault(#GP)** → 커널이 프로세스를 강제 종료.
  - ARM: 시스템 레지스터를 건드리는 `MSR` 등 특권 명령을 EL0에서 실행하면 **동기 예외(Synchronous Exception)** 로 EL1(커널)에 트랩.
- **구조적 차이(중요)**: x86은 가상화를 링과 직교하는 별도 모드(VMX root/non-root)로 뒤늦게 붙였다. ARM은 **EL2(하이퍼바이저)·EL3(시큐어 모니터)를 권한 사다리에 처음부터 설계**해 넣었다. 그래서 뒤에서 볼 KPTI(멀티다운 완화)도 arm64에선 구조적으로 더 싸게 먹힌다(4.3절 각주).

이 모드 구분은 소프트웨어 관례가 아니라 **CPU 회로가 매 명령마다 검사하는 하드웨어 상태**다. 따라서 파일 읽기, 소켓 전송, 메모리 매핑처럼 특권이 필요한 작업은 반드시 **커널에게 대신 요청**해야 한다. 이 요청 통로가 시스템 콜이다.

> 핵심: 시스템 콜은 "함수 호출"이 아니라 **특권 레벨(Ring 3→0 / EL0→1)을 넘나드는 제어권 이양**이다. 이 레벨 전환이 뒤에서 볼 모든 비용의 근원이다 — x86이든 ARM이든.

## 2. 커널로 넘어가는 관문 — `syscall`/`svc` 명령어의 실제 동작 (x86-64 & AArch64)

과거에는 `int 0x80`(소프트웨어 인터럽트)로 커널에 진입했지만 느렸다. 현대 x86-64는 전용 명령 `syscall` / `sysret`을 쓴다.

**호출 규약 (Linux x86-64):**

```
 레지스터   역할
 ────────   ───────────────────────────────
 rax        시스템 콜 번호 (예: read=0, write=1, mmap=9)
 rdi        1번 인자
 rsi        2번 인자
 rdx        3번 인자
 r10        4번 인자   (※ 일반 함수 호출의 rcx가 아니라 r10 — syscall이 rcx를 덮어씀)
 r8         5번 인자
 r9         6번 인자
 ────────
 rax        (복귀 시) 반환값 / 음수면 -errno
```

**`syscall` 명령이 하드웨어에서 하는 일 (한 사이클성 동작):**

1. 현재 `RIP`(복귀 주소)를 `rcx`에 저장 — 그래서 인자 4번이 rcx가 아니라 r10인 것.
2. 현재 `RFLAGS`를 `r11`에 저장한 뒤, `IA32_FMASK` MSR로 지정된 플래그를 마스킹(예: 인터럽트 플래그 정리).
3. `IA32_LSTAR` MSR에 미리 등록된 **커널 진입점 주소**를 `RIP`에 로드 → 실행 흐름이 커널 코드로 점프.
4. `IA32_STAR` MSR 값으로 `CS`/`SS`(코드/스택 세그먼트)를 커널용으로 교체 → **CPL이 3→0으로 전환**.

주목할 점: `syscall`은 **스택 포인터(rsp)를 자동으로 바꿔주지 않는다.** 진입 순간 `rsp`는 여전히 유저 스택을 가리킨다. 스택 교체·페이지 테이블 교체는 뒤에서 **커널 소프트웨어가 손수** 해야 한다. 이게 다음 절의 진입 경로다.

**AArch64에서는 (같은 개념, 다른 명령):**

```
 항목        x86-64                     ARM AArch64
 ─────       ──────────────────         ──────────────────
 명령        syscall                    svc #0  (Supervisor Call)
 콜 번호     rax                        x8
 인자        rdi rsi rdx r10 r8 r9      x0 ~ x5
 반환값      rax                        x0
 복귀 주소   rcx (syscall이 저장)       ELR_EL1
 상태 저장   r11 (RFLAGS)               SPSR_EL1
 진입점      IA32_LSTAR MSR             예외 벡터 테이블(VBAR_EL1)
```

- x86의 `syscall`은 MSR(LSTAR)에 등록된 **단일 고속 진입점**으로 직접 점프한다. ARM의 `svc`는 **예외 벡터 테이블(VBAR_EL1)** 의 정해진 오프셋으로 트랩하는 **동기 예외**다.
- 철학 차이: x86은 "syscall 전용 빠른 경로 명령", ARM은 "일반 예외 메커니즘 재사용". 하지만 EL0→EL1 제어권 이양이라는 본질과 비용 구조는 동일하다.

## 3. 커널 진입 경로를 한 스텝씩 (`entry_64.S`)

`syscall`로 점프한 직후, 커널은 실제 작업(`do_syscall_64`)을 부르기 전에 아래 준비 작업을 순서대로 수행한다. 이 "계단"의 각 칸이 전부 사이클을 먹는다.

```
 USER (Ring 3)                         KERNEL (Ring 0)
 ─────────────                         ───────────────
 rax=번호, 인자 세팅
 syscall  ──────────────►  [1] swapgs
                              GS 베이스를 유저(TLS)→커널(per-cpu)로 교체
                           [2] SWITCH_TO_KERNEL_CR3   ★KPTI
                              CR3(페이지 테이블 루트)를 커널용으로 스왑
                           [3] 유저 rsp 저장 → 커널 스택으로 rsp 교체
                           [4] 유저 레지스터 전부 커널 스택에 push + 살균
                           [5] IBRS_ENTER / UNTRAIN_RET / CLEAR_BRANCH_HISTORY
                              (투기적 실행 공격 완화 — Spectre/Retbleed)
                           [6] call do_syscall_64  ← ★여기서 진짜 read/write 실행
                           ───── 복귀 시 위를 역순으로 ─────
                           [5'] IBRS_EXIT
                           [4'] 레지스터 복원
                           [2'] SWITCH_TO_USER_CR3 (다시 유저 페이지 테이블)
                           [1'] swapgs
 sysret ◄──────────────    CPL 0→3, RIP←rcx 로 복귀
 다음 유저 명령 재개
```

- **[1] swapgs**: GS 세그먼트 베이스를 교체. 유저에선 TLS, 커널에선 per-CPU 변수(현재 태스크 포인터 등) 접근에 쓰인다.
- **[2] CR3 스왑 (KPTI의 핵심)**: 멀티다운 완화를 위해 유저용/커널용 페이지 테이블을 분리했기 때문에, 진입·복귀 때마다 CR3를 갈아끼운다. **이게 syscall을 특히 비싸게 만든 주범**(4.3에서 상세).
- **[4] 레지스터 저장 + 살균**: 커널 코드가 레지스터를 덮어쓰므로 유저 값을 스택에 백업하고, 정보 유출 방지를 위해 일부를 0으로 지운다.
- **[5] 투기적 실행 완화**: 분기 예측기 상태를 리셋/제한. 논리적 결과는 안 바뀌지만 캐시 사이드채널 유출을 막는다. **간접 비용의 원천**(4.2).

## 4. 비용의 해부 — 직접 비용 vs 간접 비용

syscall이 "느리다"는 말은 두 층위를 섞어 부르는 것이다. 이걸 분리해야 최적화 판단이 선다.

### 4.1 직접 비용 (Direct / 고정 오버헤드)

진입·복귀 경로에서 항상 실행되는 명령들의 비용. syscall 종류와 무관하게 대체로 일정하다.

측정 예 (`clock_gettime`을 커널 경유 vs vDSO 경유로 비교, 2.3GHz 노트북):

```
 방식                         호출당 사이클
 ──────────────────────       ─────────────
 직접 syscall (커널 진입)     ~1428 cycles
 vDSO (유저 공간 함수처럼)    ~157 cycles
 ──────────────────────
 차이                         ~1270 cycles  (약 9배)
```

- 절대 시간으로는 모드 전환 바닥값이 대략 **수백 ns**(빠른 하드웨어는 100ns 미만, 완화 켜진 서버는 더 큼).
- 통념: **syscall ≈ 일반 함수 호출의 약 10배 이상**.
- 이 수치는 "루프 안에서 반복 측정"이라 간접 비용이 섞여 다소 부풀 수 있음 → 그래서 아래 간접 비용을 따로 본다.

### 4.2 간접 비용 (Indirect / 마이크로아키텍처 교란)

진짜 무서운 부분. 커널 진입이 CPU의 **투기적 실행 최적화 상태를 무너뜨린다.** syscall이 끝나고 유저 코드로 돌아온 뒤에도 한동안 느려진다.

```
 교란 대상            무슨 일이 일어나나                          체감 효과
 ─────────────────    ────────────────────────────────────      ─────────────────────
 명령 파이프라인       syscall은 직렬화 지점. 진입 전 CPU가        복귀 후 파이프라인이
 (20~30 스테이지)      파이프라인을 비움(drain).                   거의 빈 상태 → 재충전
                                                                  까지 throughput 하락
 BHB (분기 히스토리)   완화로 clear됨                             복귀 후 분기 오예측 증가
 RSB (리턴 스택)       untrain됨                                  함수 리턴 예측 실패 →
                                                                  느린 간접 예측기로 폴백
 IBRS                  진입/복귀 시 on/off 명령 추가              직접 비용 가중
```

핵심 통찰: **syscall의 진짜 비용은 커널 안에서 실행되는 몇 개 명령이 아니라, CPU가 쌓아둔 '가속 상태(파이프라인·예측기)'를 리셋시키고 유저 코드가 그걸 다시 데우는 데 드는 시간이다.** 그래서 프로파일 flame graph에서 syscall이 핫스팟으로 튄다.

> 단, 최신 CPU는 enhanced IBRS 같은 하드웨어 완화를 넣어 BHB clear/RSB untrain을 no-op으로 만든다. → 비용은 **CPU 세대·커널 설정에 따라 크게 달라진다.**

### 4.3 KPTI(멀티다운) 세금 — syscall마다 붙는 상납금

```
 KPTI OFF                          KPTI ON (기본값)
 ────────                          ────────
 프로세스 1개 = 페이지 테이블 1개   유저용 / 커널용 페이지 테이블 2개
 커널 매핑이 유저 공간에 상주       유저 테이블엔 커널 매핑 최소만
 syscall 시 CR3 교체 없음           syscall 진입·복귀마다 CR3 스왑 → TLB 부담
```

- 왜 생겼나: 멀티다운 취약점은 유저가 투기적 실행으로 커널 메모리를 훔쳐볼 수 있게 했다. 아예 유저 페이지 테이블에서 커널 매핑을 걷어내(=KPTI) 물리적으로 차단.
- 대가: CR3 스왑은 TLB(주소 변환 캐시) 플러시를 유발. **PCID/INVPCID** 하드웨어로 전면 플러시는 피하지만 여전히 비용이 남는다.
- 규모 감: **초당 5만 syscall/CPU 수준에서 ~2%**, syscall·인터럽트 집약 워크로드에선 **1~10%** 까지. syscall 호출 빈도에 비례해 세금이 커진다.

> **ARM 각주 (arm64의 KPTI가 더 싼 이유)**: AArch64는 유저/커널 주소 변환을 **TTBR0_EL1(유저) / TTBR1_EL1(커널)** 두 레지스터로 원래부터 분리해 둔다. 그래서 KPTI를 적용해도 x86의 CR3 전면 스왑보다 구조적으로 가볍다. 게다가 멀티다운에 취약한 ARM 코어는 일부(예: Cortex-A75)뿐이라, arm64에선 KPTI가 **취약 코어에 한해 선택적으로** 켜진다. Graviton 같은 서버 코어의 syscall 세금이 x86 대비 상대적으로 낮은 배경 중 하나다. (단 Spectre 계열 분기 예측 완화는 ARM도 동일하게 부담)

> DevOps 함의: "이 서비스가 왜 CPU를 더 먹지?"의 답이 애플리케이션 코드가 아니라 **syscall 빈도 × 커널 완화 세금**일 수 있다. `perf stat`의 컨텍스트 스위치/syscall 카운트를 봐야 보인다.

## 5. 그래서 나온 회피 전략 — "커널을 덜 넘는다"

모든 고성능 I/O 설계의 공통 목표는 **crossing(모드 전환) 횟수 줄이기**다.

```
 전략            원리                                     대표 사례
 ─────────       ──────────────────────────────────       ─────────────────────
 vDSO            자주 쓰는 read-only syscall(clock_        clock_gettime,
                 gettime 등)을 유저 공간에 매핑 →          gettimeofday
                 커널 진입 없이 함수처럼 호출
 버퍼드 I/O      raw read/write 대신 유저 버퍼에 모아      stdio, BufReader
                 한 번에
 scatter/gather  여러 버퍼를 한 syscall로                  readv / writev
 io_uring        유저-커널 공유 큐에 I/O 요청을 쌓아       고성능 네트워크/디스크
                 두면 커널이 배치로 처리 → crossing 급감
 eBPF            로직 일부를 커널 안으로 올려 crossing      필터링/관측/정책
                 자체를 제거
```

이 표가 중요한 이유: 2~3부에서 **Go의 netpoller(epoll)**, **Java NIO(epoll)**, **Rust의 tokio(epoll/io_uring)** 를 볼 때, 전부 "이벤트 하나마다 커널을 넘지 않으려는" 같은 동기에서 나온 설계임을 알게 된다.

## 6. 세 언어는 이 관문을 어떻게 대하는가 (커널 → 런타임으로 올라오기)

**이 절이 이 시리즈의 척추다.** 같은 syscall 관문을 세 언어가 다르게 대하고, 그 차이가 이후 모든 성능 서사로 번진다.

```
                    앱 코드가 read()를 부르면...

 Rust  ─►  std::fs::read → libc read 래퍼 → syscall
           런타임 개입 없음. 스레드가 그냥 커널에서 블록.
           "직거래". 예측 가능하고 얇다. (대신 블록되면 그 OS 스레드는 논다)

 Go    ─►  runtime.entersyscall → (libc 없이 직접 syscall) → runtime.exitsyscall
           블로킹 syscall이면 런타임이 P(프로세서)를 M(OS 스레드)에서 떼어
           다른 고루틴에 넘김(handoff). 네트워크는 netpoller(epoll)로
           논블로킹 전환 → 수만 고루틴이 소수 OS 스레드를 공유.
           "런타임이 관문을 감싸고 스케줄링한다." → 1.2, 1.3의 핵심

 Java  ─►  JVM 네이티브 코드 → libc → syscall  (nanoTime은 vDSO)
           전통적 blocking I/O는 플랫폼 스레드(1:1)를 붙잡음 →
           NIO(Selector=epoll)로 우회 → 이제 Virtual Thread(Loom)가
           블로킹 syscall 시 캐리어 스레드에서 unmount.
           "무거운 런타임이 관문 위에 여러 추상화를 얹는다." → 1.2, 1.3
```

한 줄 요약:
- **Rust**: 커널과 직거래하는 얇은 층. 비용이 투명하고 예측 가능.
- **Go**: 런타임이 syscall을 가로채 **유저 레벨 스케줄링**으로 블로킹을 흡수. (M:P:G의 씨앗)
- **Java**: 런타임이 관문 위에 **여러 세대의 추상화**(blocking → NIO → Loom)를 겹쳐 얹음.

## 7. 트레이드오프 정리

```
 관점                Rust                 Go                    Java
 ─────────────       ─────────────        ─────────────         ─────────────
 커널 접근 층        얇음(libc 직거래)    런타임이 감쌈         런타임+추상화 겹층
 블로킹 syscall 시   OS 스레드가 블록     P 분리로 흡수         Loom이 unmount(신)
 syscall 비용 투명성 높음(예측가능)       중간(런타임 개입)     낮음(추상화 뒤에 숨음)
 최적화 여지         수동/명시적          런타임이 자동 처리    JIT+런타임이 자동
 대가                블록된 스레드 낭비   스케줄러 복잡도       런타임 무게/웜업
```

**소결**: syscall이라는 물리적 비용은 세 언어 모두 동일하게 문다. 차이는 **"그 비용을 누가, 어느 층에서 관리하느냐"** 다 — 개발자(Rust), 런타임 스케줄러(Go), 무거운 런타임+JIT(Java). 이 관리 주체의 차이가 곧 체급·웜업·클라우드 비용의 차이로 이어진다.

## 8. DevOps 관점에서 더 파고들 만한 질문들

> (지식을 실무 계측/운영 감각으로 연결하는 훈련용. 최종 원고에 넣어도 되고, 본인 학습용으로 남겨도 됨.)

1. **계측**: `strace -c ./app` 와 `perf stat -e raw_syscalls:sys_enter ./app` 로 syscall 횟수를 세면, 애플리케이션 로직 대비 "커널 왕복"이 얼마나 되는지 보인다. 내 서비스의 syscall/sec는 KPTI 세금(2~10%)을 걱정할 수준인가?
2. **컨테이너 보안 vs 비용**: 컨테이너 런타임은 `seccomp-bpf`로 허용 syscall을 필터링한다. 이 필터가 **모든 syscall에 BPF 검사 비용**을 추가한다. 보안(공격 표면 축소) ↔ 성능(추가 오버헤드) 트레이드오프를 어떻게 잡을 것인가?
3. **완화 스위치**: 신뢰된 워크로드만 도는 노드에서 `mitigations=off`(Spectre/Meltdown 완화 해제)로 부팅하면 syscall 비용이 크게 준다. 성능 이득 ↔ 사이드채널 위험을 어떤 노드에서 감수할 수 있나? (예: 격리된 배치 처리 노드)
4. **vDSO와 가상화**: 클라우드 VM/컨테이너에서 vDSO의 `clock_gettime`이 항상 유저 공간으로 처리되는가, 아니면 하이퍼바이저/클록소스 설정에 따라 커널로 떨어지는가? (`clocksource`가 `tsc`가 아니면 vDSO 이점이 사라질 수 있음)
5. **Go 스케줄러와 syscall 폭주**: Go 서비스에서 블로킹 syscall이 급증하면 왜 P 분리·M 증가로 이어져 스케줄러 지연과 컨텍스트 스위치가 늘어나는가? (→ 1.2에서 GMP로 이어짐)
6. **io_uring 도입 판단**: 초당 수십만 I/O를 하는 프록시라면 io_uring으로 crossing을 줄이는 게 얼마의 CPU를 되찾아 주는가? 반대로 복잡도·커널 버전 의존성이라는 비용은?
