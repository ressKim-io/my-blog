## ① 이 절에서 도달할 지점

5.1~5.3은 **정상 상태(steady state)**의 메모리 이야기였습니다. 이 절은 시간 축을 앞으로 감아,
파드가 **막 뜨는 순간**을 봅니다. 이 시작 구간은 정상 상태와 물리가 다릅니다.

> **"Java 프로세스는 차게 태어나 데워진다(warm up). Go·Rust는 태어날 때 이미 데워져 있다.
> 그 데워지는 몇 초/몇십 초가 오토스케일링과 서버리스의 경제성을 직접 때린다."**

이 절을 다 읽으면 다음을 설명할 수 있습니다.

- **웜업(warm-up)이란 무엇인가** — 왜 JVM은 시작 직후 느리고, 무엇이 지나야 빨라지는가
  (티어드 컴파일의 계단을 실측 임계값으로).
- 시작 구간이 **지연**만이 아니라 **메모리 스파이크**도 만든다는 것, 그래서 5.2의 상자 크기를
  정상 상태가 아니라 **피크**로 잡아야 하는 이유.
- 이 웜업 세금이 **오토스케일링·서버리스**에서 왜 곧 돈인지.
- 그 세금을 없애는 세 처방 — **Project Leyden(AOT 캐시)**, **GraalVM native-image**,
  **CRaC / AWS Lambda SnapStart** — 의 원리와 각각의 대가.

이건 2.3(JIT 웜업·티어드 컴파일)과 4부(GC)를 시간 축에서 다시 만나는 자리이기도 합니다.

## ② 개념/실체 — 시작 구간에는 세 가지가 한꺼번에 일어난다

### 콜드 스타트와 웜업 (정의부터)

- **콜드 스타트(cold start)** — 실행 중인 인스턴스가 없어 **새 프로세스를 처음부터 띄워** 첫
  요청을 처리하는 상황. 프로세스 시작 + 초기화 + (JVM이면) 웜업까지 걸리는 시간이 첫 요청의
  지연에 그대로 얹힙니다.
- **웜업(warm-up)** — 프로세스가 살아난 뒤에도 **한동안 피크 성능이 안 나오는** 구간. JVM에서
  이건 주로 **JIT 컴파일이 아직 안 끝나서**입니다.

시작 구간에는 세 가지가 겹칩니다. (1) JIT 웜업, (2) GC가 아직 힙을 못 잡은 상태, (3) 클래스
로딩·코드 캐시·malloc arena가 한꺼번에 부풀어 오르는 메모리 스파이크. 하나씩 봅니다.

### (1) JIT 웜업 — 최적 코드까지 '수천 번의 호출'이 필요하다

2.3에서 봤듯 HotSpot은 처음엔 바이트코드를 **인터프리터**로 실행하고, 자주 불리는 메서드만
**C1(빠른 컴파일·보통 코드) → C2(느린 컴파일·최고 코드)**로 승급시킵니다(티어드 컴파일). 문제는
"자주"의 기준이 **수천 번**이라는 것입니다(측정: AArch64 OpenJDK 11.0.31, 기본값).

| 플래그(측정 JDK 11.0.31) | 값 | 의미 |
|---|---|---|
| `TieredCompilation` | true | 티어드 켜짐(기본) |
| `Tier3InvocationThreshold` | 200 | 이 정도 호출되면 C1(프로파일링 포함)로 |
| `Tier4InvocationThreshold` | 5000 | **이 정도 호출돼야 C2(피크 코드)로** |
| `Tier4CompileThreshold` | 15000 | C2 승급 보조 임계 |
| `CompileThreshold`(비티어드) | 10000 | 티어드 끄면 이 값 |

즉 어떤 메서드가 **최고 성능 코드(C2)로 도는 건 대략 5000번쯤 불린 뒤**입니다. 시작 직후엔
모든 게 인터프리터/C1이라 피크의 몇 분의 일 속도로 깁니다. 이 계단이 웜업 곡선을 만듭니다.

```
성능(1/지연)
  ▲
  │                                  ┌────────── C2(Tier4) 정착 = 피크
  │                         ┌────────┘            (핫 메서드 호출 ~5000회 후)
  │                ┌────────┘  C1(Tier3, 프로파일링) = 중간
  │        ┌───────┘            (호출 ~200회 후)
  │────────┘  인터프리터 = 시작 직후, 피크의 몇 분의 일
  └──────────────────────────────────────────────────────▶ 시간(누적 호출)
   Go·Rust: ═══════════════════════════════════════════   t=0부터 이 높이
            (AOT 네이티브 기계어 — 웜업 계단 없음)
   * 곡선은 개념도이나, 계단이 갈리는 지점은 위 측정 임계값(200 / 5000)에서 나온다.
```

참고로 **JVM 바이너리 자체의 부팅**은 빠릅니다(측정: `java -version` ≈ 0.02초). 콜드 스타트가
느린 건 JVM이 켜지는 데 오래 걸려서가 아니라, 그 뒤 **클래스 수천 개 로딩 + 프레임워크 초기화 +
JIT 웜업**이 쌓여서입니다. 그래서 실제 앱(예: 스프링 부트)의 시작~피크 도달은 초 단위가 됩니다
(보고 예: JVM 모드 3~5초 — 아래 출처).

### (2) GC 미정착 & (3) 메모리 스파이크

시작 구간엔 힙이 아직 안정 크기에 안 왔고(초기 힙에서 커지는 중), 클래스 메타데이터가
Metaspace에 밀려 들어오고(5.3), 코드 캐시가 JIT 결과로 차오르고, glibc arena가 스레드 기동과
함께 늘어납니다(5.3). 그래서 **시작 시점의 RSS가 정상 상태보다 높이 튀는** 경우가 흔합니다.
5.2에서 상자(`limits.memory`)를 "정상 상태 사용량"으로 잡으면, 이 시작 스파이크에서 곧장
OOMKilled가 납니다. **상자는 피크로 잡아야 합니다.**

## ③ 비용/메커니즘 해부 — 웜업 세금이 왜 '돈'인가

웜업 세금은 두 얼굴입니다. **지연**(첫 요청들이 느림)과 **메모리**(시작 스파이크). 이게
클라우드의 두 과금 모델을 각각 때립니다.

### 오토스케일링(K8s HPA)에서

HPA는 부하가 오르면 파드를 **사후에** 늘립니다. 그런데 새 Java 파드가 **효력(피크 성능)을
내는 데 수십 초**가 걸리면, 오토스케일러가 용량을 추가해도 그 용량이 한동안 반값짜리입니다.
그래서 실무에선 (a) 항상 여분 파드를 데워 두거나(over-provision), (b) 트래픽보다 미리 스케일
아웃(예측 스케일링)하는데, **둘 다 노는 자원 = 비용**입니다. Go·Rust 파드는 ~1초면 제 성능을
내므로 스케일링을 더 촘촘히, 여분 없이 굴릴 수 있습니다. **웜업이 길수록 안전 여분이 커지고,
여분이 곧 청구서**입니다(5.2의 밀도 논리가 시간 축으로 확장된 것).

또한 K8s `readinessProbe`를 웜업 완료 후에 통과하도록 잡지 않으면, 아직 인터프리터로 도는 파드에
트래픽이 라우팅돼 **배포 직후 지연 스파이크**가 사용자에게 샙니다. 준비성 게이트를 "떴다"가
아니라 "데워졌다"에 맞추는 게 핵심입니다.

### 서버리스(Lambda·Knative scale-to-zero)에서

여기선 더 직접적입니다. 요청이 없으면 인스턴스를 0으로 내리고, 요청이 오면 새로 띄웁니다
(scale-to-zero). 이때 **콜드 스타트 지연이 곧 사용자 지연**입니다. Java의 콜드 스타트가
초 단위면 대화형 API에선 치명적이라, 역사적으로 Java는 scale-to-zero 서버리스에 불리했습니다.
아래 세 처방은 바로 이 지점을 공략합니다.

## ④ 회피/최적화 — 웜업을 없애는 세 처방

세 처방은 접근이 다릅니다. **하나는 캐시로 데우고(Leyden), 하나는 아예 네이티브로 굽고
(native-image), 하나는 데워진 상태를 스냅샷으로 떠서 복원합니다(CRaC).**

### 처방 1: Project Leyden — 훈련 실행을 캐시한다 (JVM은 그대로)

Leyden은 JVM을 버리지 않습니다. 대신 **훈련 실행(training run)**에서 얻은 것을 캐시해 다음
실행의 시작·웜업을 당깁니다. JDK 25에서 세 JEP로 정식 사용 가능해졌습니다(출처: OpenJDK Leyden,
inside.java).

- **JEP 483(JDK 24): AOT 클래스 로딩·링킹** — 클래스를 읽고 파싱·로드·링크한 결과를 AOT
  캐시에 저장해 다음 실행에서 재사용 → 시작 시간 단축.
- **JEP 514(JDK 25): AOT 커맨드라인 인체공학** — 캐시 생성 워크플로를 단순화
  (`-XX:AOTCacheOutput=app.aot` 한 번으로 관측+생성).
- **JEP 515(JDK 25): AOT 메서드 프로파일링** — **핫 메서드 프로파일을 캐시에 담아**, 프로덕션
  기동 시 JIT이 프로파일을 처음부터 갖고 시작 → 핫 메서드를 훨씬 일찍 C2로 컴파일. **웜업 자체를
  공략**합니다.

**대가:** 대표 워크로드로 훈련 실행을 해야 하고, 캐시를 빌드·배포·버전 관리해야 합니다. 대신
**JVM의 동적성과 피크 처리량은 그대로** 유지됩니다(닫힌 세계로 굽는 게 아니므로). GraalVM 없이
Java의 콜드 스타트를 줄이는 길입니다.

### 처방 2: GraalVM native-image — 빌드타임에 네이티브로 굽는다 (닫힌 세계)

native-image는 빌드 시점에 **도달 가능한 코드 전부를 정적 분석해 네이티브 실행 파일**로
컴파일합니다. 런타임에 JVM도, JIT도, 클래스 로딩도 없습니다. 그 결과가 극적입니다(보고 수치,
출처 아래).

- **시작**: JVM 모드 3~5초 → 네이티브 50~200ms(전형적으로 10~100ms).
- **메모리**: 250~400MB → 50~100MB 수준으로 감소.
- **웜업**: 없음(처음부터 네이티브 코드).

**대가(중요):**

- **닫힌 세계 가정(closed-world)** — 빌드 시점에 안 보이는 코드는 바이너리에 없습니다. **리플렉션·
  동적 프록시·JNI·런타임 클래스 로딩**은 빌드 메타데이터(reachability hints)로 명시해야 하고,
  안 하면 런타임에 깨집니다("리플렉션이 2026년에도 깨진다"는 말이 여기서 나옵니다).
- **피크 처리량 하락** — 런타임 프로파일 없이 빌드타임 분석만으로 최적화하므로, 장기 실행에서
  JIT보다 처리량이 낮습니다. 보고는 **피크 약 8%↓**부터 **정상 상태 10~25%↓**까지 워크로드에
  따라 다양합니다. 즉 **오래 도는 배치·고TPS 서비스에선 native-image가 오히려 손해**일 수
  있습니다(4부의 "롱런에서 Java가 역전"과 정확히 반대 방향의 트레이드오프).
- **빌드 복잡도·시간 증가.**

### 처방 3: CRaC / AWS Lambda SnapStart — 데워진 프로세스를 스냅샷으로 뜬다

**CRaC(Coordinated Restore at Checkpoint)**는 **이미 웜업된 실행 중 JVM을 특정 시점에
체크포인트(스냅샷)로 뜨고**, 다음엔 그 스냅샷에서 **복원(restore)**해 시작합니다. 바탕 기술은
리눅스 CRIU(Checkpoint/Restore In Userspace)입니다. 복원된 프로세스는 **이미 JIT 컴파일되고 힙도
잡힌** 상태라 시작하자마자 피크에 가깝습니다. **AWS Lambda SnapStart**가 이 CRaC API를 매니지드
Java 런타임에 구현한 것입니다(출처 아래).

- 효과: SnapStart는 콜드 스타트를 **80~90%** 줄이고 **P99 콜드 스타트 94%↓**로 보고됩니다 —
  Java를 Python·Node 수준의 콜드 스타트로 끌어내립니다.

**대가:** 스냅샷은 **외부 상태**(열린 커넥션·파일 핸들·난수 시드·시간·임시 시크릿)를 그대로
얼립니다. 그래서 `beforeCheckpoint()`/`afterRestore()` 훅으로 커넥션을 닫았다 다시 열고,
**난수·시크릿을 복원 후 재생성**해야 합니다(스냅샷에 시크릿이 박제되는 보안 이슈). 복원도
완전 공짜는 아니며 스냅샷 로딩 시간이 듭니다.

### 세 처방 비교

| 처방 | 무엇을 | 웜업 | 피크 처리량 | 동적성(리플렉션 등) | 주된 대가 |
|---|---|---|---|---|---|
| **Leyden(AOT 캐시)** | 훈련 실행 캐시(클래스+프로파일) | 크게 단축 | **유지** | 유지 | 훈련 실행·캐시 관리 |
| **native-image** | 빌드타임 네이티브(닫힌 세계) | 없음 | **8~25%↓(보고)** | 제약(힌트 필요) | 리플렉션 설정·빌드 복잡·피크 손해 |
| **CRaC/SnapStart** | 웜업된 프로세스 스냅샷 | 없음 | 유지(이미 웜업) | 유지 | 외부상태 훅·스냅샷 보안 |

운영 쪽 회피책도 병행됩니다. 서버리스의 **provisioned/pre-warmed 인스턴스**, K8s의 **웜업
완료 기준 readiness gate**, 예측 스케일링 등 — 다만 이들은 웜업을 없애는 게 아니라 **비용으로
가리는** 쪽입니다.

## ⑤ 세 언어 수렴 — '태생 웜업' vs '데워짐'

| | 시작 직후 성능 | 웜업 | 왜 |
|---|---|---|---|
| **Rust** | 피크 | 없음 | AOT 네이티브 + 런타임 초기화 최소. "태어날 때 데워져 있음" |
| **Go** | 피크에 근접 | 없음 | AOT 네이티브. 런타임 초기화(GC·스케줄러)만 ~ms. 웜업 계단 없음 |
| **Java(기본)** | 피크의 몇 분의 일 | 있음(수천 호출) | 인터프리터 시작 → JIT 승급. 대신 **롱런 피크는 최고**(4부) |
| **Java + 처방** | 처방에 따라 피크/근접 | 없음~단축 | Leyden·native-image·CRaC로 Go/Rust의 성질을 **대가를 치르고 수입** |

세로로 읽으면 이렇게 됩니다. **Go·Rust는 웜업이 없다는 이점을 태생으로 갖는 대신, JIT처럼
런타임 프로파일로 코드를 재최적화하는 능력이 없습니다.** Java는 반대로 차게 시작하는 대가를
치르는 대신, 오래 돌면 런타임 프로파일 기반 C2로 **가장 빠른 코드**에 도달합니다(4부 소결의
"롱런 역전"). 세 처방은 이 둘 사이에서 위치를 옮기는 장치입니다 — native-image는 Go/Rust 쪽
(빠른 시작·낮은 피크)으로, Leyden·CRaC는 "빠른 시작을 얻되 JIT 피크는 지키는" 절충으로.

즉 **"콜드 스타트가 잦은가, 오래 도는가"**가 선택을 가릅니다. scale-to-zero 서버리스·이벤트
함수처럼 **자주 차게 시작**하면 native-image/SnapStart가 유리하고, 며칠씩 도는 고TPS 서비스면
기본 JVM(+Leyden)의 JIT 피크가 유리합니다. 이건 6부(K8s가 Go를 택한 이유)와 7부(의사결정
매트릭스)로 이어집니다.

## ⑥ 트레이드오프 소결

한 문장: **웜업은 "런타임에 코드를 재최적화하는 능력"의 대가다.** JIT은 그 능력으로 롱런 피크를
사지만, 그 대가를 시작 구간의 느림·메모리 스파이크로 냅니다. Go·Rust는 그 능력을 포기해 웜업을
면제받았고, Java의 세 처방은 능력의 어느 부분을 얼마나 포기할지를 정하는 다이얼입니다.

```
        피크 처리량(롱런)
             ▲
            ╱ ╲   기본 JVM(JIT): 피크 최고, 대신 웜업 세금
           ╱   ╲
  콜드스타트 ◀─╱─────╲─▶ 동적성(리플렉션·런타임 로딩)
  (빠른 시작)          (native-image는 이걸 닫힌 세계로 포기)
  native-image        CRaC·Leyden: 시작을 사되 피크·동적성 지킴
```

그리고 5부의 시간 축이 여기서 닫힙니다. 5.1(회계 규칙) → 5.2(밀도) → 5.3(힙 밖 죽음) →
5.4(시작 구간). 남은 5.5는 이 넷을 4부의 GC 삼각형과 함께 **하나의 의사결정 그림**으로
포갭니다.

## ⑦ DevOps 관점 심화 질문

1. **readiness를 '데워짐'에 맞추기** — `readinessProbe`를 프로세스 기동이 아니라 웜업 완료
   기준으로 잡으려면 무엇을 신호로 쓰겠습니까(예열 요청 N회 후 통과, JIT 컴파일 큐 소진 등)?
   이걸 안 하면 배포 직후 어떤 지연 패턴이 사용자에게 보입니까?

2. **처방 선택** — 하루 수천 번 차게 시작하는 이벤트 함수와, 3일씩 도는 고TPS 게이트웨이가
   있습니다. 각각 native-image / CRaC(SnapStart) / Leyden 중 무엇을 택하고, 그 근거를 "콜드
   스타트 빈도 × 피크 처리량 민감도"로 설명해 보십시오.

3. **native-image의 리플렉션 부채** — 스프링/하이버네이트처럼 리플렉션이 많은 스택을
   native-image로 굽는 비용(메타데이터 유지·빌드 시간·CI)을 어떻게 관리하겠습니까? 어떤 서비스는
   native-image가 오히려 손해입니까?

4. **CRaC 스냅샷의 상태 관리** — 체크포인트에 얼린 DB 커넥션·난수 시드·시크릿을 복원 후 어떻게
   안전하게 갱신하겠습니까? 스냅샷에 시크릿이 박제되는 위험을 배포 파이프라인에서 어떻게
   차단합니까?

5. **시작 스파이크와 limits** — 시작 시 RSS 피크가 정상 상태보다 높습니다(5.3). `limits.memory`를
   피크로 잡으면 밀도가 나빠지고, 정상값으로 잡으면 기동 중 OOMKilled입니다. NMT/부하 테스트로
   이 피크를 어떻게 재고, 어떤 값을 택하겠습니까?

6. **오토스케일링 웜업 지연 보정** — HPA가 파드를 늘려도 웜업 때문에 실효 용량이 지연됩니다.
   이 지연을 어떤 지표(웜업 시간 × 스케일 이벤트 빈도)로 모델링해, 예측 스케일링이나 여분 파드
   수를 정하겠습니까? 이 여분의 비용을 어떻게 정당화합니까?

## ⑧ 다음 절: 5.5 소결 — 세 언어를 클라우드 경제 축에 얹기

5부의 네 절이 각각 하나의 축을 세웠습니다. 5.1은 **회계**(커널은 전부 합산), 5.2는
**밀도**(GC 헤드룸이 노드 비용), 5.3은 **안전마진**(힙 밖 여백이 없으면 죽음), 5.4는
**시작 구간**(웜업 세금이 오토스케일·서버리스 비용). 5.5는 이 네 축을 4부의 GC 트레이드오프
삼각형과 포개, Rust·Go·Java(G1/ZGC/native-image)를 **어떤 클라우드에서 무엇을 최적화할 때
무엇을 고르나**의 한 장짜리 그림으로 정리하고, 6부(쿠버네티스는 어떻게 Go의 한계를 극복했나)로
다리를 놓습니다.
