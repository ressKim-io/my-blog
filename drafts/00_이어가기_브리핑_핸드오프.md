## 1. 프로젝트 개요

**제목**: 커널과 런타임으로 톺아보는 Rust, Go, Java의 성능과 비용 트레이드오프
**목적**: 커널(syscall)에서 시작해 런타임 → 언어 → 클라우드로 **아래에서 위로** 쌓아 올리는 심화 기술 글.
**성격**: 이 세션의 산출물은 **최종 원고가 아니라 1차 심화 정리(reference-level draft)**. 최종 집필은 별도 Claude Code 에이전트가 함.
**진행 단위**: 한 채팅에 **1부씩**.

- **깊이**: 레퍼런스/교과서 수준. 커널·메모리 흐름까지 깊게. 얕은 개론 금지.
- **각 절 공통 구조**: ① 도달 지점(학습 목표) → ② 개념/물리적 실체 → ③ 비용·메커니즘 해부 → ④ 회피/최적화 전략 → ⑤ **3개 언어(Rust/Go/Java) 수렴 비교** → ⑥ 트레이드오프 소결 → ⑦ **DevOps 관점 심화 질문** → ⑧ 다음 절 연결.
- **3언어 수렴**: 모든 절 끝에서 같은 주제를 Rust/Go/Java로 수렴시켜 비교(이 시리즈의 척추).
- **ARM 병행**: x86-64뿐 아니라 **ARM(AArch64)** 도 같이 다룸(클라우드 = Graviton/Ampere/Apple Silicon).
- **그림**: 전부 **ASCII 아트**(코드펜스 안).
- **코드**: 코드 블록 **허용**(필요할 때 짧고 illustrative하게).
- **DevOps 질문**: 각 절 끝에 "발전하고 싶은" 독자를 위한 실무 계측/운영 심화 질문 5~6개.
- **언어**: 한글 설명. 사용자 선호 = 간결·직접적, 불필요한 실용성/년차 판단 금지.
- **사실확인**: 버전 민감 주제는 반드시 웹 검색으로 최신 확인 후 작성.

**기준 버전(2026-07)**: JDK 25(LTS), Go 1.25, Linux 6.x(EEVDF 스케줄러), GraalVM native-image, Rust(제로코스트).

## 3. 완료 현황

### ✅ 1부 완료 (파일 3개, outputs 폴더)
- `1부_1.1_유저모드_커널모드_syscall비용.md`
- `1부_1.2_스레드모델_1대1_vs_유저레벨스케줄링.md`
- `1부_1.3_런타임의_무게_ZeroCost_vs_FatRuntime.md`

**1.1 요약**: 보호 링(x86 CPL) / 예외 레벨(ARM EL0-EL3), `syscall`/`svc` 명령, 커널 진입 경로(swapgs→CR3 스왑(KPTI)→레지스터 저장→투기적 완화→do_syscall_64). 직접 비용(~1400 cycle vs vDSO ~157) + 간접 비용(파이프라인 drain, BHB/RSB/IBRS). KPTI 세금(2~10%), arm64는 TTBR0/1로 더 쌈. 회피: vDSO/io_uring/eBPF. 3언어: Rust 직거래 / Go 런타임 핸드오프 / Java 추상화 겹층.

**1.2 요약**: 1:1 커널스레드(무게=스택+컨텍스트스위치, C10K 벽) vs M:N 유저스케줄링. **Go GMP**(work stealing, syscall handoff, netpoller, SIGURG 선점, **Go 1.25 컨테이너 인식 GOMAXPROCS**). **Java 가상스레드(Loom, JDK21 GA)** mount/unmount·pinning·**JEP491(JDK24)로 synchronized pinning 해소**. **Rust** 1:1 + tokio(협력형 M:N, 색칠). 숨은 대비: JVM은 JDK10부터 컨테이너 인식, Go는 1.25에야.

**1.3 요약**: 런타임을 5축(바이너리/베이스라인RSS/시동/정상상태오버헤드/정점처리량)으로 분해. Rust=제로코스트(런타임 거의 없음, 그린스레드 RFC230 제거로 1.2 완결). Go=Fat이지만 자족적(정적 바이너리, scratch 이미지). JVM=최대 무게·최대 영리(JIT 적응성). JVM 메모리 구성표에 **Direct Memory(NIO/Netty) off-heap OOMKilled** 포함. 최신: **Leyden AOT 캐시(JEP483/514/515)**, **GraalVM native-image**(빌드타임·closed-world·Serial GC 트레이드오프 정직하게 명시).

## 4. 다음: 2부 (기계어 생성과 런타임 최적화) — 이번엔 이걸 씀

**2부. CPU를 쥐어짜는 전략: 기계어 생성과 런타임 최적화**
- 2.1 Rust (AOT): 컴파일러의 엄격함 → 예측 가능한 하드웨어 직거래
- 2.2 Go (AOT): 빠른 빌드의 대가, 보수적 기계어의 한계 — **⚠ 정정: "PGO 부재"는 틀림. Go 1.21+ PGO 정식. "정적이라 런타임 재최적화 불가"라는 각도로 재구성.**
- 2.3 Java (JIT): 느린 웜업의 반전, C2의 런타임 돌연변이 최적화(인라이닝, 벡터화, 역가상화)
- 2.4 [소결] 롱런 서버에서 Java가 정적 컴파일 언어의 CPU 효율을 역전하는 이유 — **⚠ "역전"은 조건부(peak throughput 한정, tail latency·메모리·웜업은 여전히 열세)로 다듬기.**

**2부에서 회수할 복선(1부에서 심어둠)**:
- 1.3의 반전: "가장 무거운 JVM이 왜 롱런 정점 처리량에서 앞서는가" → 2.3/2.4가 답.
- Leyden(JEP515 AOT 프로파일)·GraalVM PGO → JIT vs AOT 경계 흐리기.
- ARM 병행: JIT는 arch별 기계어 생성, native-image는 arch별 빌드.

## 5. 목차 초안의 낡은/부정확 지점 (전체적으로 주의)

작성 시 아래는 최신 사실로 교정할 것:
1. **Go "PGO 부재"** → 사실 아님(1.21+ 정식). "정적 한계"로 재프레임.
2. **"Go=M:N vs Java=1:1" 이분법** → 깨짐. Java 가상스레드(Loom) 반영.
3. **Rust가 brk/mmap "직접 호출"** → 부정확. libc malloc 위임(3부에서 정확히).
4. **"Java 역전"** → peak throughput 한정 조건부.
5. **ZGC** → JDK21+ Generational ZGC 표준. sub-ms STW + allocation stall/처리량 세 정확히(4.4).
