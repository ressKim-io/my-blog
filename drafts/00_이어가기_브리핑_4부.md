# 📌 이어가기 브리핑 (핸드오프) — 4부. 새 채팅에서 이 파일부터 읽히세요

> 3부(메모리 할당)까지 끝났습니다. 이 파일은 **4부(GC의 한계와 환상)**를 새 채팅에서
> 동일한 깊이·톤·형식으로 이어 쓰기 위한 핸드오프입니다. 4부 집필 에이전트는 **이 파일을
> 먼저 통독**한 뒤 아래 규칙대로 4.1부터 한 절씩 작성하세요.

---

## ▶ 사용법 & 추천 재개 멘트 (그대로 복사해서 쓰세요)

새 채팅을 열고 이 파일을 첨부(또는 "outputs 폴더의 00_이어가기_브리핑_4부 읽어줘")한 뒤,
아래 멘트를 붙여넣으면 동일한 깊이·형식·톤으로 재개됩니다.

```
outputs 폴더의 '00_이어가기_브리핑_4부'를 먼저 읽어줘.
그 '집필 규칙(2장)'대로 4부부터 한 절씩 이어서 작성해줘. 특히 아래를 반드시 지켜줘:

1) 톤·문체: 존댓말(–습니다체), 자연스러운 한글. 번역투 금지(대시 남발·"~라는 점" 반복·
   영어 문장구조 직역 금지). 각 절 맨 앞에 "이 문서의 성격" 블록쿼트를 넣고 시작.
2) 깊이=자립성: 개념은 반드시 '정의부터' 풀 것(예: 삼색 마킹이 뭔지, 쓰기 배리어가 뭔지,
   allocation stall이 뭔지). 그 문단만 읽어도 따로 검색 없이 이해되게. 밀도≠축약 — 깊되
   풀어서. 실제 자료구조·정확한 수치(임계값·기본값)·런타임 기전까지.
3) 형식 분업(이번이 핵심): 진짜 공간/구조 그림은 ASCII 코드펜스, 표 형태 데이터(비교·수치)는
   마크다운 표, 열거·개념은 풀어쓴 산문. 구조 있는 대목엔 ASCII를 아끼지 말고 그릴 것.
4) 버전 민감 사실은 웹 검색으로 최신 확인 후 쓸 것(가능하면 샌드박스 java/go로 실측).
   리뷰 피드백이 오면 감으로 수긍하지 말고 웹/소스로 교차검증한 뒤 정확히 반영/정정.
5) 한 절씩 써서 보여주고 깊이·톤을 확인받은 뒤 다음 절로(한 응답에 여러 절 몰아쓰기 금지).

3.5에서 세운 '트레이드오프 삼각형(단편화 ↔ 이동 비용 ↔ 회계의 위치)'을 4부의 뼈대로 삼고,
"GC는 은탄환이 아니다"를 검증적으로 풀어줘. 2.2의 Mark Assist·디스코드 복선을 4.2에서
회수하며 시작해줘.
```

**왜 이 멘트가 필요한가:** 3부를 진행하며 문체·형식이 크게 진화했습니다. 처음엔 평서체
번역투로 써서 "밀도는 높은데 읽기 어렵다"는 피드백을 받았고, 이후 **① 존댓말+자연스러운 한글,
② 정의부터 풀어쓰기, ③ 형식 분업(ASCII는 그림에만·표는 마크다운·개념은 산문), ④ 사실
교차검증, ⑤ 한 절씩 검토**를 규칙으로 확정했습니다. 이 멘트가 그 진화의 종착점을 처음부터
강제합니다. (규칙을 안 박으면 다시 번역투/축약으로 돌아갑니다.)

## 1. 프로젝트 개요

- **제목:** 커널과 런타임으로 톺아보는 Rust · Go · Java의 성능과 비용 트레이드오프
- **목적:** 커널(syscall)에서 시작해 런타임 → 언어 → 클라우드로, 아래에서 위로 쌓아 올리는
  심화 기술 글.
- **성격:** 이 세션 산출물은 최종 원고가 아니라 **1차 심화 정리(reference-level draft)**.
  최종 집필은 별도 에이전트가 함.
- **진행 단위:** 한 채팅에 1부씩. (4부는 새 채팅에서.)

## 2. 집필 규칙 (확정된 컨벤션 — 반드시 유지) ★★★

### 2-A. 톤·문체 (★이번 세션에서 확정 — 최우선)

- **존댓말(–습니다/합니다체)**로 씁니다. 설명 문장은 존댓말, 표·다이어그램 캡션·불릿 조각은
  명사형으로 끝내도 됩니다(예: "락 없음", "8KB 페이지").
- **번역투를 걷어냅니다.** 다음을 피하세요: 대시(—)로 절을 잇는 영어식 구조 남발, "~라는
  점", "~것이다"의 반복, "A는 B이다 — C이기 때문이다" 식 직역체. 자연스러운 연결어(그래서,
  따라서, 이때, 반대로, 결국)를 쓰고 문장을 끊어 읽기 쉽게.
- **각 절 맨 앞에 "이 문서의 성격" 블록쿼트**를 답니다. 형식(그대로 재사용):
  > **이 문서의 성격** — 최종 블로그 원고가 아니라, 최종 집필 에이전트에게 넘길 1차 심화
  > 정리(reference-level draft)입니다. `개념 → 하드웨어/런타임 동작 → 비용 해부 → 회피 전략
  > → 3개 언어 연결 → 트레이드오프 → 심화 질문`의 순서로 아래에서 위로 쌓아 올립니다. 그림은
  > 모두 ASCII로 그렸고, 본문의 수치는 (해당 절 기준 버전)이며 환경/버전에 따라 달라질 수
  > 있음을 전제로 읽어 주세요.

### 2-B. 깊이 = 자립성 (풀어서 설명)

- 개념을 처음 도입할 때 **정의부터** 풉니다. (예: "삼색 마킹이란, 객체를 흰/회/검 세 색으로
  칠하며 살아있음을 추적하는 방식인데…"처럼 용어의 뜻부터.)
- 독자가 **따로 검색·공부하지 않아도 그 문단만으로 이해**되게 씁니다. 축약해서 밀도만 올리면
  실패. **밀도 ≠ 축약 — 깊되, 풀어서.**
- 항상 "왜 중요한가 / 물리적으로·런타임에서 무슨 일이 일어나는가"를 붙입니다.
- **정확한 수치**를 씁니다("대략"으로 얼버무리지 말 것). 실제 자료구조·상수·기본값·기전까지.

### 2-C. 형식 분업 (★이번 세션에서 확정)

3부 초반엔 모든 그림을 ASCII로 밀어넣었다가, "표 형태는 마크다운 표가 낫다"는 피드백으로
아래로 정리했습니다. **역할별로 도구를 나눕니다.**

- **진짜 공간·구조·흐름 → ASCII 코드펜스.** (페이지 워크, 자료구조 바이트 배치, 3층 구조,
  스택 성장, GC evacuation, 결정 흐름, 트레이드오프 삼각형 등.) **구조 있는 대목엔 ASCII를
  아끼지 말고 그릴 것.** 사용자가 "그림 있으면 이해·레퍼런스에 좋다"고 명시적으로 요청함.
- **표 형태 데이터(비교·수치 매트릭스) → 마크다운 표.** ASCII 표로 억지로 만들지 말 것.
- **열거·개념 → 풀어쓴 산문(+ 필요시 불릿).** 불릿은 굵은 라벨 + 한두 문장으로.
- 어셈블리/구조체/코드는 짧고 illustrative하게 코드펜스에. (4부는 어셈블리보다 **GC 알고리즘
  의사코드·상태 전이·바이트 레이아웃**이 주로 쓰일 것.)

### 2-D. 각 절 공통 8단 구조

① 도달 지점(학습 목표) → ② 개념/물리적·런타임 실체 → ③ 비용·메커니즘 해부 → ④ 회피/최적화
전략 → ⑤ 3개 언어(Rust/Go/Java) 수렴 비교 → ⑥ 트레이드오프 소결 → ⑦ DevOps 관점 심화 질문
(5~6개) → ⑧ 다음 절 연결.

### 2-E. 그 밖의 확정 규칙

- **3언어 수렴:** 모든 절 끝에서 같은 주제를 Rust/Go/Java로 수렴시켜 비교(시리즈의 척추).
  4부는 특히 **Rust=GC 없음(회수를 컴파일러/사람이)**, **Go=비이동 동시 마크-스윕**,
  **Java=이동 세대별(G1/ZGC)**의 대비가 중심.
- **숨은 축도 함께:** codegen뿐 아니라 데이터 지역성(메모리 레이아웃), GC의 CPU/지연 점유,
  호출 규약, 컨테이너 현실(OOMKilled·cgroup·GOMEMLIMIT/-Xmx)까지 엮을 것.
- **ARM 병행:** x86-64뿐 아니라 AArch64도. 클라우드 = Graviton/Ampere/Apple Silicon. (GC
  쓰기 배리어의 약한 메모리 모델 상호작용, colored pointer의 주소 비트 등.)
- **DevOps 질문:** 각 절 끝 5~6개. 실무 계측(GC 로그·JFR·pprof·eBPF)·운영·클라우드 네이티브
  (K8s/cgroup) 관점.
- **언어/톤:** 한글. 존댓말. 간결·직접적이되 풀어서. 실용성/년차 판단 금지, 지식 중심.
- **사실확인:** 버전 민감 주제는 반드시 웹 검색으로 최신 확인 후 작성. 가능하면 샌드박스에서
  실측(`java -XX:+PrintFlagsFinal`, `go version` 등). **리뷰 피드백은 감으로 수긍하지 말고
  교차검증 후 반영**(3부에서 이 방식으로 다수 교정·강화됨).
- **미래 떡밥은 슬쩍:** 최신/차기 방향(예: Valhalla, Green Tea GC, ZGC 진화)은 한두 문장으로
  짚되 본론은 해당 절에서. 과하게 벌리지 말 것.

### 2-F. 작업 프로세스

- **한 절씩** 써서 보여주고 **깊이·톤을 확인받은 뒤** 다음 절로. 한 응답에 여러 절을 몰아
  쓰지 말 것(품질 저하).
- 각 절: (1) 버전 사실 웹/샌드박스 검증 → (2) 8단 구조로 집필 → (3) outputs에 md 저장 →
  (4) `present_files`로 공유 → (5) 깊이·톤 확인 요청.
- 파일명 규칙(3부 예): `3부_3.3_Go자체할당자_P별캐시와_연속스택.md`. 4부도 동일 패턴.

### 2-G. 기준 버전 (2026-07 최신 — ★3부에서 갱신·정정됨)

- **JDK 25 (LTS)** — HotSpot. G1 기본 GC. Generational ZGC는 **JDK 23부터 ZGC의 기본 모드**
  (JEP 439/474). compact object headers **JEP 519**(JDK 25 정식, 아직 opt-in). Valhalla
  **JEP 401** Value Classes는 **프리뷰·JDK 28 목표**.
- **Go 1.26** (2026-02, 최신 패치 1.26.5). **Green Tea GC가 1.26에서 기본**(작은 객체 마킹
  지역성 개선, GC 오버헤드 10~40%↓ 보고, 여전히 **비이동식**). `GOEXPERIMENT=simd` 실험.
  컨테이너 인식 `GOMAXPROCS`는 1.25+. `GOMEMLIMIT`은 1.19+지만 **아직 cgroup 자동 설정 아님
  (수동)**.
- **Linux 7.x** — ★정정: 커널이 2026년 7.x로 넘어감(7.0 4월, 7.1 6월). MM 내부(brk/mmap·페이지
  폴트·TLB·overcommit)는 6.x/7.x 동일. EEVDF 스케줄러.
- **Rust** — 제로코스트, GC 없음. 기본 `System` 할당자=libc malloc(1.32부터). `Rc`/`Arc`는
  옵트인 참조 계수.

## 3. 완료 현황

### ✅ 1부 (커널·런타임 기초) — 파일 3개
- 1.1 유저/커널 모드·syscall 비용(보호링·`syscall`/`svc`·KPTI 세금·vDSO/io_uring).
- 1.2 스레드 모델 1:1 vs 유저레벨(C10K, Go GMP·work stealing·netpoller, Java 가상스레드/Loom,
  Rust 1:1+tokio).
- 1.3 런타임의 무게 ZeroCost vs FatRuntime(5축 분해, Rust=제로코스트, Go=Fat이나 자족, JVM=
  최대무게·최대영리). Direct Memory off-heap OOMKilled 예고.

### ✅ 2부 (컴파일·codegen) — 파일 4개
- 2.1 Rust(AOT): MIR·borrow check·단형화(제로코스트, 코드팽창)·`dyn` fat pointer·ThinLTO·PGO.
  ★"Rust가 brk/mmap 직접 호출" 복선 심음(→ 3.2에서 malloc 위임으로 정정 완료).
- 2.2 Go(AOT): 빠른 빌드·얕은 최적화·인라인 예산 80·이스케이프 분석·레지스터 ABI·데이터
  지역성. ★GC CPU 복선(25% 백그라운드 + Mark Assist), 디스코드 사례 예고(→ 4.2).
- 2.3 Java(JIT): MDO·티어드 컴파일·인라인 캐시·스칼라 치환·deopt·세이프포인트/TTSP·세그먼트
  코드캐시(240MB)·Leyden·Valhalla 예고.
- 2.4 소결: Java 역전은 **조건부**(peak throughput 한정). ★deopt storm·OOMKilled(cgroup이
  힙+네이티브 합산)·JIT↔AOT 연속 스펙트럼.

### ✅ 3부 (메모리 할당 밑바닥) — 파일 5개 (이번 세션, 최신·완결)
- 3.1 커널의 메모리: 가상주소·페이지 테이블(4단계 워크)·TLB·PCID/ASID(KPTI 고리)·brk/mmap
  (어셈블리)·VMA·페이지 폴트(minor/major)·zero page·overcommit·OOM·cgroup v2·huge page.
- 3.2 유저 공간 할당자: glibc ptmalloc(arena/bin/chunk·boundary tag)·tcache(2.26)·
  `M_MMAP_THRESHOLD`(128KB→32MB)·단편화. ★"Rust brk/mmap" 복선 **완전 정정**(malloc 위임,
  jemalloc→System 1.32). ARM 약한 메모리 모델.
- 3.3 Go 할당자: mcache(P별 락 없음)/mcentral/mheap·size class(~67)·span·tiny(16B)·큰 것
  직행·**고루틴 스택 2KB·copystack**·GOMEMLIMIT. ★Green Tea GC(1.26)·비이동식.
- 3.4 JVM TLAB: bump-the-pointer·moving GC의 선물·retire/Eden/**Humongous(50% region)**·
  객체 헤더(12~16B, compact 8B/JEP519)·**compressed oops 32GB 절벽**·힙 밖 지형(Metaspace·
  코드캐시·다이렉트)·OOMKilled·스칼라 치환(JIT 조건부)·**Generational ZGC 예고**·Valhalla.
- 3.5 소결: **세 할당자 수렴(하드웨어가 강제한 캐시 계층)** + **회계 보존 법칙**(GC=조건부
  필연, Rust 반례) + **할당이 GC를 결정**(bump→이동식, 슬롯→비이동식) + **트레이드오프 삼각형**
  + 비용 보존 표. → 4부로 다리.

## 4. 다음: 4부 (GC의 한계와 환상)

**한 줄 목표:** "GC는 은탄환"이라는 프레이밍을, 3.5의 **트레이드오프 삼각형(단편화 ↔ 이동
비용 ↔ 회계의 위치)**에 각 GC를 얹어 **검증적으로** 해부한다.

### 절 구성 (초안 — 8단 구조로)

- **4.1 GC의 기본 원리.** GC가 실제로 하는 일(정의부터): 루트(스택·전역) → **추적(mark)** →
  **회수(sweep)** 또는 **압축/이사(compact/evacuate)**. **삼색 마킹**(흰/회/검)과 불변식,
  **쓰기 배리어(write barrier)**가 왜 필요한지(동시 마킹 중 뮤테이터가 그래프를 바꾸는 문제),
  **STW vs 동시(concurrent)**의 차이. → 이후 세 GC를 읽을 공용 어휘를 세운다.
- **4.2 Go의 GC.** 동시 마크-스윕(비이동), **GC 페이싱**(`GOGC` 기본 100, `GOMEMLIMIT`),
  백그라운드 마킹(25% = `gcBackgroundUtilization`), ★**Mark Assist**(할당하는 뮤테이터를 GC
  노동에 징집) → **디스코드 사례**(2.2 복선 회수: 2분마다 강제 GC로 지연 스파이크), **Green
  Tea GC**(1.26). "왜 Go는 tail latency가 튀는가"의 물리.
- **4.3 Java G1.** region 기반·young/mixed/**Full GC**, **동시 마킹**(IHOP 기본 45%),
  evacuation pause(-XX:MaxGCPauseMillis 기본 200 목표), **humongous → 동시 마킹 트리거**(3.4
  연결), Full GC STW 폭발이 언제 터지나. "왜 G1은 대체로 안정적이나 가끔 크게 멈추나".
- **4.4 Generational ZGC.** **colored pointer**(주소에 상태 비트)·**load/store barrier**로
  **동시 evacuation**(뮤테이터 안 멈추고 이사). sub-ms STW의 실체와, 그 뒤에 숨은 **처리량
  세금(barrier 비용)**·**allocation stall**(회수보다 할당이 빠르면 뮤테이터가 잠깐 막힘).
  3.4에서 예고한 그 청구서. vs G1 비교.
- **4.5 소결.** 세 GC를 트레이드오프 삼각형에 얹어 최종 비교(단편화/이동 비용/회계 위치 +
  tail vs throughput + 메모리 오버헤드). "은탄환은 없다" 검증 마무리 → **5부(MSA·클라우드
  경제학: 빈패킹·OOMKilled·콜드스타트)**로 다리.

### 4부에서 회수할 복선 (앞 부에서 심음)

- **2.2 → 4.2:** Go GC의 CPU 점유(25% 백그라운드 + **Mark Assist**), **디스코드 사례**.
- **3.3 → 4.2:** "할당 싸구려의 청구서가 GC로" + 비이동식 + **Green Tea GC**의 정체.
- **3.4 → 4.3/4.4:** **Generational ZGC**의 sub-ms STW·**allocation stall**·처리량 세금,
  **humongous → concurrent marking** 유발.
- **2.4 → 4.4:** ZGC generational 표준, allocation stall.
- **3.5 → 4부 전체:** **트레이드오프 삼각형**과 "GC=조건부 필연"이 4부의 뼈대·평가 잣대.
- **5부로 넘길 씨앗:** GC 지연/처리량 세금이 빈패킹·콜드스타트·off-heap OOMKilled(1.3/2.4/3.4)
  의 클라우드 경제학으로 어떻게 번지는지.

### 4부의 뼈대 = 3.5의 트레이드오프 삼각형

```text
                     회계의 위치
                (사람 / 컴파일러 / 런타임)
                         ▲
                        ╱ ╲
                       ╱   ╲
       단편화 ◀───────╱─────╲───────▶ 이동 비용
   (비이동의 대가)                  (이동의 대가: barrier·pause)

   4부는 여기에 각 GC를 얹는다:
     · Go G1아님(비이동 마크-스윕) → 이동비용 0, 단편화 감수, 대신 Mark Assist 지연
     · Java G1(이동)             → 단편화 0(압축), 대신 evacuation pause·Full GC 위험
     · Java ZGC(동시 이동)        → pause는 sub-ms로, 대신 barrier 처리량 세금·allocation stall
```

## 5. 사실·목차 교정 사항 (★이번 세션에서 확인·정정 — 4부에서 유지)

1. **커널 6.x → 7.x.** Linux는 2026년 7.x 계열(7.0 4월, 7.1 6월). MM 내부는 동일.
2. **Go 1.25 → 1.26.** Green Tea GC가 1.26 기본. 4부 4.2에서 정확히 반영.
3. **고루틴 초기 스택 8KB → 2KB.** Go 1.4부터 연속 스택으로 2KB. (세그먼트 스택 시절이 8KB.)
4. **Generational ZGC = JDK 23부터 ZGC 기본**(JEP 474). 4.4의 핵심.
5. **compressed oops 32GB 절벽**(3.4). 4부에서 힙 사이징·ZGC(colored pointer는 64비트 주소를
   쓰므로 이 제약과 별개) 논의 시 참고.
6. **GOMEMLIMIT은 아직 cgroup 자동 아님**(수동). 4.2/5부 컨테이너 논의에서.
7. **GC "은탄환" 프레이밍은 반드시 검증적으로** — 3.5의 삼각형과 보존 법칙으로. 각 GC의 "공짜
   같아 보이는" 지점 뒤의 청구서(Mark Assist·Full GC·barrier 세금)를 반드시 드러낼 것.
