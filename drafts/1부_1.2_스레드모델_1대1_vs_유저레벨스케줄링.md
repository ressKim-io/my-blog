# 1부 · 1.2 스레드 모델의 진화: 커널 스레드 1:1 매핑 vs 유저 레벨 스케줄링 (Go GMP)

> **이 문서의 성격**: 최종 블로그 원고가 아니라, 최종 집필 에이전트에게 넘길 **1차 심화 정리(reference-level draft)**.
> 1.1(syscall 비용)에서 "블로킹 syscall을 누가 흡수하느냐"가 언어별로 갈렸다. 그 흡수 메커니즘의 정체가 이 절이다.
> 그림은 ASCII, 코드는 개념 예시. 수치는 대략치이며 CPU/커널/버전에 따라 달라짐을 명시.
> **기준 버전(2026년 7월)**: JDK 25(LTS), Go 1.25, Linux 6.x(EEVDF 스케줄러).

---

## 0. 이 절을 읽고 나면 이해하게 되는 범위

- OS 스레드(커널 스레드)의 물리적 실체와 "무겁다"는 말의 정확한 의미 (task_struct·스택·컨텍스트 스위치)
- 1:1 모델이 부딪히는 벽 — C10K 문제
- 유저 레벨 스케줄링(M:N)이 "동시성 단위"를 "커널 스케줄링 단위"에서 떼어내는 원리
- **Go GMP 스케줄러**의 내부 (work stealing, netpoller, async 선점, 컨테이너 인식 GOMAXPROCS)
- **Java 가상 스레드(Loom)** 의 mount/unmount와 pinning, 그리고 JDK 24 JEP 491의 해결
- **Rust**가 왜 그린 스레드를 버리고 1:1 + 선택적 async(tokio)를 택했는가 (제로코스트 철학)
- 이 차이가 컨테이너 CPU 제한·꼬리 지연(tail latency)으로 이어지는 지점

## 1. 커널 스레드란 무엇인가 — 1:1 모델의 물리적 실체

"스레드"라는 단어가 두 층위에서 쓰여 혼란을 준다. 여기서 말하는 **커널 스레드**는 **커널 스케줄러가 인식하고 CPU에 직접 배정하는 실행 단위**다. 리눅스에선 프로세스든 스레드든 커널 입장에선 `task_struct`(스케줄 가능 엔티티)로 표현된다.

1:1 모델이란 **언어의 스레드 1개 = OS 스레드 1개 = task_struct 1개**로 매핑되는 것이다. Rust의 `std::thread`, Java의 플랫폼 스레드가 여기 해당한다.

```
 1:1 모델 (Rust std::thread, Java 플랫폼 스레드)

  앱 스레드        OS 스레드(M)        커널               하드웨어
  ────────         ───────────         ──────             ────────
   T1 ───────────── kT1 ───────┐
   T2 ───────────── kT2 ───────┤─► [커널 스케줄러]  ───► CPU 코어
   T3 ───────────── kT3 ───────┘    CFS→EEVDF(6.6+)       (코어 수만큼만
                                     타임슬라이스 배분       진짜 동시 실행)

  스레드 1개의 원가:
   • task_struct (커널 메모리)
   • 커널 스택 + 유저 스택 (수백 KB ~ MB 예약)
   • 스케줄러 자료구조 등록
```

**"무겁다"의 세 가지 정확한 의미:**

1. **메모리 원가 (스택)**: 스레드는 자기 호출 스택이 필요하다. 리눅스 pthread 기본 스택은 8MB(예약, 실제 상주는 touch한 만큼), Rust `std::thread` 기본 2MB, Java 플랫폼 스레드는 `-Xss` 기본 수백 KB~1MB. 스레드 수가 늘면 이 스택이 선형으로 쌓인다.

2. **컨텍스트 스위치 비용 (1.1과 직결)**: 커널이 스레드 A→B로 전환하려면 syscall/인터럽트로 커널에 진입(모드 전환), 레지스터 파일 저장·복원, 스케줄러 실행, 주소공간이 다르면 TLB 부담까지. **직접 비용 ~1~2μs 수준**이지만, 진짜 비용은 1.1에서 본 것처럼 **캐시·TLB·분기예측기 상태가 오염되는 간접 비용**이다.

3. **생성/소멸 비용**: 스레드 생성은 `clone()` syscall + 스택 매핑 + 커널 자료구조 할당. 요청마다 스레드를 새로 만들면 이 비용이 반복된다(그래서 스레드 풀을 쓴다).

> **ARM 각주**: 컨텍스트 스위치 시 저장할 레지스터 파일이 arch마다 다르다. AArch64는 범용 레지스터 31개(x0~x30) + SIMD/FP 32개(v0~v31, 128비트), x86-64는 GPR 16개 + 벡터 16개(AVX-512면 32개). ARM이 정수 레지스터가 더 많아 저장 상태가 약간 크지만, 컨텍스트 스위치 총비용에서 이 차이는 부차적이다. **1:1 모델의 무게라는 본질은 두 아키텍처 동일.**

## 2. 1:1 모델의 벽 — C10K 문제

동시 접속 1만 개(C10K), 나아가 100만 개(C10M)를 스레드 하나씩으로 감당하려 하면 두 벽에 부딪힌다.

```
 스레드 1개 = 1MB 스택 가정

   접속 수      필요 스택          결과
   ───────      ─────────          ────
   1,000        ~1 GB              빠듯
   10,000       ~10 GB             메모리 폭발
   100,000      ~100 GB            불가능

 게다가 실행 가능한 스레드가 코어 수를 크게 넘으면:
   → 커널 스케줄러가 수만 개를 라운드로빈 → 컨텍스트 스위치 폭풍
   → CPU가 '일'보다 '스위칭'에 시간을 씀 (thrashing)
```

핵심 모순: **네트워크 서버의 스레드 대부분은 사실 "일"을 하지 않고 I/O를 기다린다.** 소켓 하나 기다리자고 1MB 스택 + 커널 스케줄링 슬롯을 통째로 점유하는 건 극도로 비효율적이다. 이 낭비를 없애려는 게 유저 레벨 스케줄링이다.

## 3. 유저 레벨 스케줄링의 아이디어 — M:N

발상의 핵심 한 줄: **"동시성의 단위"를 "커널 스케줄링의 단위"에서 분리한다.**

- 값싼 유저 태스크(고루틴/가상 스레드/async 태스크)를 **수백만 개** 만든다.
- 이들을 **소수의 OS 스레드(대개 코어 수 근처)** 위에서 **런타임이 직접 스케줄링**한다.
- 태스크가 I/O로 블로킹되면, OS 스레드를 붙잡는 대신 **런타임이 그 태스크만 재워두고(park) OS 스레드는 다른 태스크에 넘긴다.**

```
 M:N 모델 (유저 레벨 스케줄러가 커널과 태스크 사이에 낀다)

   G1 G2 G3 ... G(수백만)     ← 값싼 유저 태스크 (2KB~, 힙/작은 스택)
      \   |   /
   ┌──────────────────┐
   │ 유저 레벨 스케줄러 │       ← Go 런타임 / Java Loom / Rust tokio
   └──────────────────┘
      /   |   \
    M1   M2   M3               ← 소수의 OS 스레드 (≈ 코어 수)
      \   |   /
   [커널 스케줄러] → CPU 코어
```

두 축으로 M:N 런타임을 구분하면 언어별 차이가 선명해진다:

- **선점(preemptive) vs 협력(cooperative)**: 런타임이 태스크를 강제로 뺏을 수 있나, 아니면 태스크가 스스로 양보(yield/await)해야 하나?
- **투명(transparent) vs 색칠(colored)**: 평범한 블로킹 코드를 그대로 쓰나, 아니면 `async/await`로 함수를 "물들여야" 하나?

이 두 축 위에 Go / Java / Rust를 놓으면 아래처럼 갈린다. 하나씩 보자.

## 4. Go의 답 — GMP 스케줄러 (Deep Dive)

Go는 **언어 런타임에 M:N 스케줄러를 내장**했다. 개발자는 `go f()` 한 줄로 고루틴을 만들고, 블로킹 코드를 그대로 쓴다(투명). 이 마법의 정체가 GMP다.

### 4.1 G, M, P의 정의

```
 G (Goroutine)  : 고루틴. 실행 단위. 초기 스택 2KB, 필요 시 복사하며 성장.
 M (Machine)    : OS 스레드. 실제로 CPU에서 도는 것.
 P (Processor)  : 논리 프로세서. G를 M에 태울 '권리'이자 로컬 런큐 소유자.
                  개수 = GOMAXPROCS (기본 = 사용 가능 코어 수)

 규칙: G가 실행되려면 P가 필요하고, P는 M에 붙어야 CPU를 쓴다.
       즉 동시에 '진짜 실행 중'인 고루틴 최대 개수 = P 개수 = GOMAXPROCS.
```

```
 Go GMP 한 장 요약

  P0[runq: G G G] ── M0 ── 코어0
  P1[runq: G G  ] ── M1 ── 코어1
  P2[runq: G    ] ── M2 ── 코어2
        ▲
        └─ P2 런큐가 비면 다른 P의 런큐에서 절반을 훔침 (work stealing)

  [전역 런큐(global runq)]   ← 로컬에 못 담긴 G, 주기적으로 확인
  [netpoller = epoll]        ← I/O 대기 중인 G를 여기 park (M 점유 안 함)
```

### 4.2 네 가지 핵심 메커니즘

**(1) 로컬 런큐 + Work Stealing.** 각 P는 자기 런큐(고루틴 대기열)를 갖는다. 대부분의 스케줄링은 락 없이 로컬에서 처리돼 빠르다. 어떤 P가 놀게 되면 **다른 P를 무작위 순회하며 런큐의 절반을 훔쳐온다.** 이 "제한적·무작위" 훔치기가 부하를 코어에 고르게 편다.

**(2) syscall 핸드오프 (1.1과 연결).** 고루틴이 블로킹 syscall에 들어가면 런타임의 `entersyscall`이 개입한다. **그 M은 커널에 붙잡히지만, P를 떼어내(handoff) 다른 M에 붙여** 나머지 고루틴들이 계속 돌게 한다. syscall이 끝나면 `exitsyscall`로 복귀. → 1.1에서 예고한 "Go는 런타임이 관문을 감싸 블로킹을 흡수한다"의 실체.

**(3) netpoller.** 네트워크 I/O는 syscall 블로킹 대신 **논블로킹 + epoll(리눅스)/kqueue/IOCP**로 처리된다. `conn.Read`로 대기 중인 고루틴은 OS 스레드를 점유하지 않고 netpoller에 park된다. **수백만 커넥션이 대부분 park 상태로 소수 스레드를 공유** — Go가 네트워크 서버에 강한 이유.

**(4) 비동기 선점 (async preemption, Go 1.14+).** 초기 Go는 함수 호출 경계에서만 선점 가능해서, `for{}` 같은 타이트 루프가 스케줄러를 독점했다. 1.14부터 런타임이 **hot한 M에 SIGURG 시그널을 보내** 시그널 핸들러에서 고루틴 상태를 저장하고 강제 선점한다. → Go의 M:N은 **선점형**.

### 4.3 GOMAXPROCS와 컨테이너 (DevOps 핵심, Go 1.25 변화)

```go
// 고루틴 10만 개 — OS 스레드였다면 스택만 수십 GB
for i := 0; i < 100_000; i++ {
    go func(id int) {
        resp, _ := http.Get(url) // 블로킹처럼 보이지만
        _ = resp                 // netpoller가 FD로 park → M 반환
    }(i)
}
```

- `GOMAXPROCS` = 동시에 도는 고루틴 수의 상한 = P 개수. 기본은 사용 가능 코어 수.
- **Go 1.24 이전의 함정**: 컨테이너에 `CPU limit=2`를 걸어도 Go는 **호스트의 전체 코어 수**(예: 64)로 GOMAXPROCS를 잡았다. → 런타임이 64개 P로 일하려다 커널 CFS 쿼터에 걸려 **스로틀링(throttling: 남은 주기 동안 실행 완전 정지)** → **꼬리 지연 폭증**.
- **Go 1.25의 해결**: 런타임이 cgroup의 CPU 제한(v1 `cpu.cfs_quota_us`/`cfs_period_us`, v2 `cpu.max`)을 읽어 **GOMAXPROCS를 CPU limit에 맞춰 기본 설정**하고, 약 30초마다 재확인해 동적으로 조정한다. 끄려면 `GODEBUG=containermaxprocs=0`. 단 **CPU limit만 보고 CPU request는 무시**한다.

> DevOps 함의: Go 1.24 이하를 K8s에 올렸다면 `automaxprocs`(uber 라이브러리)로 수동 보정했어야 한다. 1.25부터는 런타임 기본값이 대신 해준다. "왜 2코어 파드가 스로틀되지?"의 고전적 원인.

## 5. Java의 답 — 플랫폼 스레드에서 가상 스레드로 (Project Loom)

Java는 **두 종류의 스레드**를 갖게 됐다. 이게 목차 초안의 "Java=1:1" 이분법이 깨진 지점이다.

### 5.1 플랫폼 스레드 (기존, 1:1)

`new Thread()`, 스레드 풀의 스레드 = OS 스레드 1:1. 무겁다(스택 수백 KB~1MB). 그래서 전통적 Java 서버는 **스레드 풀 + thread-per-request**로 수백~수천 동시성이 한계였고, 이를 넘으려 **리액티브(WebFlux 등) 콜백 지옥**으로 우회했다.

### 5.2 가상 스레드 (JDK 21 GA, M:N)

```
 가상 스레드 mount/unmount

  VT(가상스레드) ──mount──► Carrier(플랫폼 스레드, 기본 ForkJoinPool, parallelism=코어수)
       │
       ├─ 블로킹 I/O(계측된 JDK API) 만나면:
       │     VT의 실행 상태(Continuation)를 힙에 저장 ──unmount──► Carrier 즉시 해방
       │     Carrier는 대기 중인 다른 VT를 태움
       │
       └─ I/O 완료되면 VT를 아무 Carrier에 다시 mount하여 이어서 실행
```

- **가상 스레드**는 OS 스레드가 아니다. 실행 상태가 **힙에 Continuation 객체**로 저장돼 초기 풋프린트가 수백 바이트~KB, 필요 시 증감. **수백만 개** 생성 가능.
- 블로킹 호출을 만나면(계측된 `java.net`/`java.nio` 등) **캐리어에서 unmount**되어 캐리어를 해방한다. 개발자는 **평범한 블로킹 코드(thread-per-request)를 그대로 쓴다**(투명). → Go의 철학과 수렴.
- 캐리어 풀은 work-stealing ForkJoinPool.

```java
// 가상 스레드: thread-per-request 스타일을 되살린다
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            var body = client.send(req, ofString()); // 블로킹 → 캐리어 unmount
            return body.body();
        });
    }
}
```

### 5.3 Pinning 문제와 JEP 491 (JDK 24 → 현 LTS JDK 25)

- **Pinning(고정)**: 가상 스레드가 unmount되지 못하고 캐리어를 붙잡는 현상. 캐리어가 고정되면 그만큼 M:N의 이점이 사라지고, 심하면 **기아·데드락**.
- **JDK 21~23의 아킬레스건**: `synchronized` 블록/메서드 안에서 블로킹하면 캐리어가 pinning됐다. 레거시 라이브러리가 `synchronized`를 많이 쓰면 가상 스레드 이점이 무력화.
- **JDK 24 JEP 491의 해결**: 모니터 소유권을 **캐리어가 아니라 가상 스레드 단위로 추적**하도록 JVM을 바꿔, `synchronized` 안에서 블로킹해도 unmount가 된다. → **현 LTS인 JDK 25에는 이 개선이 포함**.
- **여전히 남는 pinning**: 네이티브(JNI) 코드가 콜백으로 자바 블로킹을 호출하는 경우, 클래스 초기화(class initializer) 중 블로킹 등 예외적 상황.

> DevOps 함의: 가상 스레드 도입 시 `-Djdk.tracePinnedThreads` 또는 JFR의 `jdk.VirtualThreadPinned` 이벤트로 pinning을 계측해야 한다. JDK 버전이 24 미만이면 `synchronized` 병목이 그대로 살아있다.

### 5.4 Java는 원래 컨테이너 인식이 됐다 (Go보다 먼저)

**JVM은 JDK 10(2018)부터 `UseContainerSupport`로 cgroup의 CPU/메모리 제한을 읽는다.** `Runtime.availableProcessors()`가 CPU limit을 반영하므로, ForkJoinPool 캐리어 수·GC 스레드 수 등이 컨테이너 제한에 맞춰진다. → **Go가 1.25(2025)에야 얻은 컨테이너 인식을 Java는 7년 먼저 갖고 있었다.** (반대로 Java는 이걸 힙 계산과 엮으며 다른 함정을 만드는데, 그건 5부에서.)

## 6. Rust의 답 — 1:1 유지 + 선택적 async (제로코스트 철학)

Rust는 **의도적으로 M:N 런타임을 언어에서 뺐다.** 1.0 이전(2014)에 그린 스레드(libgreen)를 실험했지만, "모든 바이너리에 무거운 런타임을 강제"하는 것이 **제로코스트 추상화 철학**에 어긋나 제거했다. → 1.3의 Zero-Cost 주제로 이어짐.

### 6.1 두 가지 동시성 층

**(1) `std::thread` = 1:1 OS 스레드.** Rust의 기본 스레드는 OS 스레드 그대로. Java 플랫폼 스레드와 같은 무게(기본 스택 2MB). CPU 바운드 병렬 작업엔 이게 정답(rayon 등).

**(2) `async`/`.await` = 컴파일된 상태 기계.** async 함수는 런타임이 아니라 **컴파일러가 상태 기계(`Future`)로 변환**한다. 힙 할당·런타임이 강제되지 않는다(제로코스트). 다만 이 `Future`를 **실제로 굴리는 실행기(executor)는 언어에 없다** — tokio/async-std/smol 같은 라이브러리를 골라야 한다.

```rust
// tokio: async는 협력적(cooperative) — .await 에서만 양보한다
#[tokio::main]
async fn main() {
    let mut tasks = Vec::new();
    for _ in 0..100_000 {
        tasks.push(tokio::spawn(async {
            let resp = reqwest::get(URL).await.unwrap(); // .await 에서 양보
            resp.text().await
        }));
    }
}
```

### 6.2 tokio = 라이브러리로 구현한 M:N (협력형)

- tokio 멀티스레드 런타임은 **워커 스레드 풀(기본=코어 수) 위에서 async 태스크를 work-stealing으로 스케줄링**한다. I/O는 `mio`를 통해 epoll. 구조적으로 Go의 GMP와 닮았지만 **라이브러리 레벨**이고 **협력형**이다.
- **함정 1 — 색칠(coloring)**: `async fn`은 `async fn`에서만 자연스럽게 호출된다. 동기/비동기 세계가 갈라지는 "함수 색칠" 문제.
- **함정 2 — 협력형의 대가**: async 태스크가 `.await` 없이 오래 CPU를 쥐거나 **블로킹 호출을 하면 워커 스레드 전체가 멈춘다.** Go/Loom 같은 선점·자동 unmount가 없다.

```rust
// 함정: async 안에서 블로킹 호출 → 그 워커 스레드가 통째로 마비
tokio::spawn(async {
    std::thread::sleep(Duration::from_secs(1)); // ❌ .await가 아님 → 협력 스케줄러 정지
    // 블로킹이 꼭 필요하면 tokio::task::spawn_blocking 으로 격리해야 함
});
```

> **컨테이너 인식 주의**: tokio 기본 워커 수는 `available_parallelism()`(리눅스에서 `sched_getaffinity` 기반)이라 **cpuset이 아닌 CFS quota만 걸린 컨테이너에선 호스트 코어 수를 볼 수 있다.** Go 1.25 이전과 같은 함정 — 필요하면 `worker_threads`를 명시적으로 CPU limit에 맞춰야 한다.

## 7. 세 언어 수렴 비교

```
 항목           Rust std       Rust tokio      Go             Java 플랫폼    Java 가상(Loom)
 ───────        ────────       ──────────      ──             ──────────     ──────────────
 모델           1:1            M:N (협력)      M:N (선점)     1:1            M:N (준선점*)
 스케줄 주체    커널           tokio 런타임    Go 런타임      커널           JVM(ForkJoinPool)
 태스크 스택    ~2MB           상태기계(Future) 2KB~복사성장   ~0.5~1MB       힙 Continuation, 성장
 선점           OS 타임슬라이스 없음(.await)    SIGURG 선점    OS 타임슬라이스 마운트/블로킹 지점
 코드 스타일    블로킹          async 색칠      투명 블로킹    블로킹         투명 블로킹
 블로킹 흡수    스레드 낭비     spawn_blocking  P 핸드오프     스레드 낭비    캐리어 unmount
 컨테이너 인식  부분(affinity)  부분(affinity)  1.25부터 O     JDK10부터 O    JDK10부터 O
```

`*` 가상 스레드는 시간 기반 선점이 아니라 블로킹/마운트 지점에서 양보하므로 완전 선점은 아님.

핵심 통찰: **Go와 Java 가상 스레드는 "투명한 블로킹 코드 + 런타임이 알아서 흡수"로 수렴**했다. Rust는 이 자동화를 **제로코스트 철학을 지키려 포기**하고, 대가로 명시성(async 색칠, 수동 블로킹 격리)을 개발자에게 넘겼다. 같은 문제(블로킹 흡수)에 **"런타임이 책임진다(Go/Java)" vs "개발자가 책임진다(Rust)"** 로 갈린 것 — 1.1의 결론과 정확히 같은 축.

## 8. 트레이드오프 소결

- **1:1(무거운 스레드)** 은 단순하고 예측 가능하며 CPU 바운드 병렬에 최적. 대신 대규모 I/O 동시성에서 메모리·컨텍스트 스위치로 무너진다.
- **M:N(유저 레벨)** 은 대규모 I/O 동시성을 값싸게 흡수. 대신 런타임 복잡도, 스케줄러 관련 미묘한 버그(pinning, 워커 블로킹, GOMAXPROCS 오설정)라는 새로운 실패 양식을 만든다.
- 어느 쪽이든 **물리 코어 수라는 천장은 동일**하다. M:N은 "기다리는 태스크"를 값싸게 다룰 뿐, "계산하는 태스크"를 더 빠르게 하진 않는다.

## 9. DevOps 관점에서 더 파고들 만한 질문들

1. **스로틀링 진단**: 컨테이너의 `cpu.stat`에서 `nr_throttled`/`throttled_time`을 보면 CFS 스로틀링 여부가 보인다. Go 1.24 이하 또는 tokio 워커 미설정 서비스가 CPU limit에 걸려 꼬리 지연을 만들고 있지 않은가?
2. **컨텍스트 스위치 계측**: `pidstat -w`(자발/비자발 스위치), `perf sched`로 스위치 폭풍을 관측할 수 있다. 스레드 풀 크기를 코어 수 대비 과도하게 잡으면 비자발 스위치가 치솟는다 — 적정 크기는?
3. **가상 스레드 pinning 탐지**: `-Djdk.tracePinnedThreads` / JFR `jdk.VirtualThreadPinned`로 pinning을 잡는다. 도입 대상 JDK가 24 미만이면 `synchronized` 병목이 그대로다 — 업그레이드 vs `ReentrantLock` 치환, 무엇이 싼가?
4. **가상 스레드 vs 리액티브**: Loom 이후 WebFlux 같은 리액티브 스택의 존재 이유가 줄었나? 어떤 경우(백프레셔, 스트리밍 조합)엔 여전히 리액티브가 유리한가?
5. **tokio 블로킹 격리**: async 핸들러 안에서 파일 I/O·CPU 무거운 작업을 `spawn_blocking` 없이 호출하면 왜 전체 처리량이 무너지는가? 블로킹 풀 크기는 어떻게 잡나?
6. **request vs limit**: Go 1.25는 CPU **limit**만 보고 **request**는 무시한다. request=1, limit=4인 파드에서 GOMAXPROCS는 몇이 되며, 그게 노드 빈패킹(5부)과 어떻게 충돌하는가?
