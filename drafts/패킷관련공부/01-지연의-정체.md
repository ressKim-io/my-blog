# 1편. 느린 건 왜 느린가 — 지연(latency)의 정체

> **시리즈 "패킷은 어떻게 사용자에게 닿는가"의 1편 (허브 글)**
> 이 글에서 잡은 개념(RTT, 지연 4성분, slow start)을 이후 2~6편에서 계속 다시 꺼내 쓴다.

## 들어가며: "10Gbps인데 왜 느려?"

회선 속도가 10Gbps라고 적혀 있는데 페이지가 느리게 뜬다. 흔한 오해 하나부터 깨고 시작하자.

**느린 것의 원인은 거의 항상 대역폭(Gbps)이 아니다. 거리와 왕복 횟수(RTT)다.**

대역폭은 "한 번에 얼마나 굵게 보내느냐(throughput)"고, 지연은 "한 번 갔다 오는 데 얼마나 걸리느냐(latency)"다. 이 둘은 다른 축이다. 굵은 파이프(대역폭)가 있어도, 파이프가 길거나(거리) 물을 여러 번 왕복시켜야 하면(RTT) 체감은 느리다.

## 지연은 4가지로 쪼개진다

패킷이 A에서 B로 가는 총 지연은 네 성분의 합이다.

```
패킷 A→B 총 지연 = 전파지연 + 전송지연 + 처리지연 + 큐잉지연

┌────────────────┬─────────────────────────────────────────────┐
│ 전파(propagation) │ 거리 / 빛의 속도. 광케이블 속 빛 ≈ 진공의 2/3.  │
│                │ → 거리가 멀수록 ↑. 물리 법칙이라 못 줄임.       │
├────────────────┼─────────────────────────────────────────────┤
│ 전송(transmission)│ 패킷크기 / 대역폭. 큰 데이터 = 시간 ↑.        │
│                │ → 대역폭 키우면 ↓.                            │
├────────────────┼─────────────────────────────────────────────┤
│ 처리(processing)  │ 라우터/스위치가 헤더 보고 경로 결정.           │
│                │ → 거치는 홉(hop)이 많을수록 ↑.                │
├────────────────┼─────────────────────────────────────────────┤
│ 큐잉(queuing)     │ 혼잡할 때 버퍼에서 대기. → 트래픽 몰리면 ↑.    │
└────────────────┴─────────────────────────────────────────────┘
```

> 🎨 **그래픽 추천**: 이 4성분 표는 "패킷 하나가 출발→도착하는 타임라인 막대"로 그리면 직관적이다. 각 구간이 막대의 다른 색 구획으로 나뉘는 형태.

여기서 대역폭을 키우면 줄어드는 건 **전송지연 하나뿐**이다. 나머지 셋(전파·처리·큐잉)은 대역폭과 무관하다. 그래서 "회선만 빠르면 다 빠를 것"이라는 직관이 깨진다.

## 빛의 속도라는 천장

전파지연은 거리에 비례하고, 그 비례 상수는 빛의 속도다. 광케이블 안에서 빛은 진공의 약 2/3 속도로 움직인다. 이걸로 계산하면:

```
서울 ↔ 미국 서부, 편도 직선거리 ≈ 9,000km 이상
광속(2/3)으로 왕복하면        ≈ 최소 ~140ms

→ 아무리 좋은 회선, 아무리 비싼 장비를 써도
  이 ~140ms 밑으로는 내려갈 수 없다. 물리 법칙이다.
```

이게 핵심이다. **거리는 latency의 하한선을 정한다.** 그래서 뒤(4·5편)에서 다룰 CDN의 본질이 "거리를 물리적으로 잘라내는 것"이 된다. 더 좋은 컴퓨터가 아니라 더 가까운 위치가 답인 이유다.

## 진짜 범인: RTT와 왕복 횟수

지연을 한 번의 편도로만 생각하면 안 된다. 연결 하나를 맺는 데 **왕복(Round Trip Time, RTT)이 여러 번** 필요하기 때문이다.

HTTPS로 첫 페이지를 받기까지 드는 왕복을 세어보자.

```
DNS 조회             : 1 RTT  (캐시 없을 때)
TCP 3-way handshake  : 1 RTT  (SYN → SYN-ACK → ACK)
TLS 1.2 handshake    : 2 RTT  (TLS 1.3이면 1 RTT)
────────────────────────────────
그제서야 첫 HTTP 요청  : 1 RTT

RTT가 100ms라면? 데이터 받기도 전에 핸드셰이크만 400~500ms.
```

TLS 1.2 핸드셰이크를 펼쳐 보면 이렇게 왔다 갔다 한다.

```
클라이언트                          서버
   │── ClientHello ───────────────>│  ┐
   │<─ ServerHello, Cert, Done ────│  ┘ 1 RTT
   │── KeyExchange, Finished ─────>│  ┐
   │<─ Finished ───────────────────│  ┘ 2 RTT
   │── (드디어) GET / ─────────────>│  ┐
   │<─ 200 OK ─────────────────────│  ┘ +1 RTT
```

> 🎨 **그래픽 추천**: 이 핸드셰이크 시퀀스는 "사다리 다이어그램(sequence diagram)"으로 그리면 왕복이 눈에 확 들어온다. 좌우 두 세로선 사이를 화살표가 지그재그로 오가는 형태. 1편에서 가장 그래픽 효과가 큰 그림.

왕복 횟수를 줄이는 기법들이 곧 "빠른 서비스"의 정체다. (이 표는 4편에서 CDN이 실제로 어떻게 적용하는지로 이어진다.)

```
┌──────────────────────┬────────────────────────────────────┐
│ 기법                  │ 효과                                │
├──────────────────────┼────────────────────────────────────┤
│ Keep-Alive(연결 재사용) │ 핸드셰이크 1번 후 같은 연결로 N개 요청  │
│ TLS Session Resumption│ 이전 세션키 재사용 → TLS 왕복 생략     │
│ TLS 1.3 0-RTT         │ 재방문 시 첫 요청을 핸드셰이크와 동시에 │
│ HTTP/2 multiplexing   │ 한 연결로 여러 요청 동시 처리          │
│ HTTP/3 (QUIC, UDP기반) │ TCP+TLS 핸드셰이크를 1개로 통합        │
└──────────────────────┴────────────────────────────────────┘
```

## 잘 안 알려진 핵심: TCP Slow Start

여기가 사람들이 잘 모르는 부분이다. TCP는 처음부터 전속력으로 보내지 않는다. 조금씩 보내고(slow start), 잘 도착하면 보내는 양(혼잡 윈도우, congestion window)을 점점 키운다. 그런데 이 **"키우는 주기"가 RTT에 묶여 있다.**

```
RTT 짧음 (가까운 서버, 5ms):
  win: 10 → 20 → 40 → 80 ...   (5ms마다 2배)
  → 순식간에 대역폭을 꽉 채움

RTT 김 (먼 서버, 150ms):
  win: 10 → 20 → 40 ...        (150ms마다 2배)
  → 같은 회선이어도 한참 동안 느릿느릿
```

> 🎨 **그래픽 추천**: 두 곡선(짧은 RTT vs 긴 RTT)이 시간 축에서 대역폭 천장에 도달하는 속도를 비교하는 라인 차트. "거리가 멀면 throughput도 늦게 찬다"가 한눈에 보인다.

즉 **거리가 멀면 빛의 속도 때문만이 아니라, TCP가 속도를 끌어올리는 데도 오래 걸려서** 체감이 훨씬 느려진다. 가까운 곳에서 받으면 RTT가 짧아 slow start를 금방 벗어나고 throughput이 빨리 올라간다. 이걸 BDP(Bandwidth-Delay Product)로도 설명하는데, "회선을 가득 채우려면 RTT만큼의 데이터가 항상 공중에 떠 있어야 한다"는 개념이다.

## CLI로 직접 보기

이 시리즈의 고정 코너. 매 편 "직접 재보는 명령어" 하나씩.

```bash
# 페이지를 받는 데 각 단계가 몇 초 걸리는지 분해
curl -w "dns:%{time_namelookup} connect:%{time_connect} \
tls:%{time_appconnect} ttfb:%{time_starttransfer} total:%{time_total}\n" \
-o /dev/null -s https://example.com

# 읽는 법:
#   time_namelookup  → DNS (1 RTT)
#   time_connect     → TCP handshake까지
#   time_appconnect  → TLS handshake까지
#   time_starttransfer → 첫 바이트 도착(TTFB)
```

가까운 사이트와 먼 해외 사이트를 각각 찍어보면, `time_connect`(거리·홉)와 `time_appconnect`(TLS 왕복) 숫자가 이 글의 모든 내용을 그대로 보여준다.

## 정리

- 느린 건 대역폭이 아니라 **거리와 왕복(RTT)** 탓이다.
- 지연은 **전파·전송·처리·큐잉** 4성분이고, 대역폭이 줄여주는 건 전송 하나뿐이다.
- 거리는 빛의 속도로 latency의 **하한선**을 정한다 → 빠르게 하려면 거리를 줄여야 한다(CDN의 본질).
- 연결 하나에 **왕복이 여러 번** 든다 → 왕복을 줄이는 게 곧 최적화다.
- **TCP slow start** 때문에 거리가 멀면 throughput까지 늦게 찬다.
