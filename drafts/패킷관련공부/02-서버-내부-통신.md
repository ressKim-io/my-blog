# 2편. 서버 안에서의 거리 — East-West 통신 해부

> **시리즈 "패킷은 어떻게 사용자에게 닿는가"의 2편**
> 1편에서 "거리가 지연을 지배한다"고 했다. 이번엔 거리를 데이터센터 안으로 좁혀, *서버끼리 주고받는 트래픽(East-West)*을 본다.

## East-West가 뭐야

트래픽 방향을 두 가지로 부른다.

- **North-South**: 외부 사용자 ↔ 서버 (위아래 방향). 3편 주제.
- **East-West**: 서버 ↔ 서버, 내부끼리 (좌우 방향). 이번 주제.

마이크로서비스가 많아질수록 East-West 트래픽이 폭증한다. 서비스 A가 B를 부르고, B가 C를 부르고… 이 내부 호출 한 번 한 번에 지연이 붙는다. 그래서 "내부니까 빠르겠지"는 위험한 가정이다. **같은 거리라도 어느 레이어를 거치느냐에 따라 µs에서 ms까지 갈린다.**

## 거리 사다리: 가까운 것부터

```
가까움(빠름) ───────────────────────────────────> 멈(느림)

localhost   같은노드      다른노드        같은AZ    다른AZ     다른리전
 ~µs        ~수십µs    오버레이/네이티브   ~0.5ms    ~2ms      수십~수백ms
```

> 🎨 **그래픽 추천**: 이 사다리는 "동심원" 또는 "계단형 막대"로 그리면 좋다. 가운데(localhost)에서 바깥으로 갈수록 지연이 커지는 동심원 형태가 직관적.

하나씩 내려가 보자.

### ① 같은 Pod 안 (localhost)

```
┌─────────── Pod (하나의 network namespace) ───────────┐
│  [컨테이너 A] ──localhost(lo)──> [컨테이너 B]          │
│         물리 NIC 안 거침. 커널 loopback. 거의 메모리 복사 │
└──────────────────────────────────────────────────────┘
지연: ~µs. 가장 빠름.
```

사이드카 패턴(예: Istio sidecar)이 빠른 이유가 이거다. 앱과 Envoy 프록시가 같은 Pod 안 localhost로 통신하니까. 물리 네트워크를 아예 안 탄다.

### ② 같은 노드의 다른 Pod

```
┌──────────────── Node (워커) ─────────────────┐
│  [Pod A] eth0 ─veth─┐                         │
│                     ├─ 가상브리지 or eBPF ─┐    │
│  [Pod B] eth0 ─veth─┘                      │    │
│                                            └→ [Pod B] │
│  물리 NIC 밖으로 안 나감. 커널 내부에서 처리.   │
└───────────────────────────────────────────────┘
지연: ~수십 µs.
```

여기서 **kube-proxy(iptables) vs eBPF(Cilium)** 차이가 나온다. 전통 kube-proxy는 iptables 룰을 선형 탐색해서 서비스가 많아지면 O(n)으로 느려지지만, Cilium의 eBPF는 해시 기반 조회 + 브리지/iptables 우회로 더 짧은 경로로 직행한다.

### ③ 다른 노드의 Pod (여기서 갈린다)

Pod IP는 보통 하부 네트워크에서 그대로 라우팅되지 않는다. 그래서 노드 경계를 넘을 때 두 방식 중 하나를 쓴다.

```
방식 A) 오버레이 캡슐화 (VXLAN/Geneve)
  원래 패킷[Pod A→Pod B]을 UDP로 한 번 더 감쌈
  ┌Node1─────────┐         ┌Node2─────────┐
  │ Pod패킷       │         │              │
  │  ↓ 캡슐화      │  물리망  │  ↓ 역캡슐화    │
  │ [UDP[Pod패킷]]│ ──────> │ [Pod패킷]     │
  └──────────────┘         └──────────────┘
  + 하부 네트워크가 Pod IP 몰라도 됨 (어디든 깔림)
  - 헤더 오버헤드, MTU 감소, 캡/디캡 CPU 비용

방식 B) 네이티브 라우팅 (eBPF / 클라우드 VPC CNI)
  캡슐화 없이 Pod IP를 그대로 라우팅
  ┌Node1─────────┐         ┌Node2─────────┐
  │ [Pod패킷]     │ ──────> │ [Pod패킷]     │
  └──────────────┘         └──────────────┘
  + 오버헤드 없음, 더 빠름
  - 하부 네트워크/CNI 지원 필요
```

> 🎨 **그래픽 추천**: 캡슐화(방식 A)는 "패킷이 봉투 안에 또 봉투로 들어가는" 양파/러시아인형 일러스트가 효과적. 오버헤드 개념이 시각적으로 와닿는다.

Cilium의 native routing 모드가 방식 B고, AWS VPC CNI는 ENI에 Pod IP를 직접 할당해 캡슐화를 생략한다. 캡슐화를 없애는 것만으로 노드 간 지연과 CPU 비용이 줄어든다.

### ④ EC2 간 — k8s 추상화를 벗기면

쿠버네티스 레이어를 걷어내면 물리적 거리가 그대로 드러난다.

```
┌─────────────────┬──────────────┬──────────────────────────┐
│ 구간            │ 대략 RTT      │ 특징                       │
├─────────────────┼──────────────┼──────────────────────────┤
│ 같은 AZ         │ ~0.1~0.5ms   │ 사실상 같은(또는 인접) DC.  │
├─────────────────┼──────────────┼──────────────────────────┤
│ 다른 AZ(같은리전) │ ~1~2ms      │ AZ = 물리적으로 떨어진 DC.  │
├─────────────────┼──────────────┼──────────────────────────┤
│ 다른 리전       │ 수십~수백ms   │ 백본 또는 인터넷 경유.       │
└─────────────────┴──────────────┴──────────────────────────┘
```

여기서 중요한 통찰: **AZ(가용영역)는 일부러 물리적으로 떨어뜨린 데이터센터다.** 한 곳이 화재·정전으로 죽어도 다른 AZ가 살아남게 하려는 고가용성 설계다. 그 대가로 AZ를 넘는 통신에는 ~1~2ms의 지연이 붙는다. **가용성과 지연의 트레이드오프**가 여기서 발생한다.

## cross-AZ를 줄이는 법

서비스가 3개 AZ에 흩어져 있으면, Service 호출이 무작위로 AZ를 넘나들며 지연을 만든다. 이걸 줄이는 장치가 있다.

```
[기본] Service → 무작위 엔드포인트 선택 → AZ를 마구 넘나듦

[개선] Topology Aware Routing 켜기
       service.kubernetes.io/topology-mode: Auto
       → "같은 AZ 안의 엔드포인트 우선"으로 라우팅
       → cross-AZ 지연 ↓

[Istio]  locality-aware load balancing
       → 같은 zone 우선, 같은 region 차선, 그다음 타지역
```

> 참고: cross-AZ 트래픽은 지연뿐 아니라 클라우드 사업자(AWS 등) 기준으로 데이터 전송 *과금* 대상이기도 하다. (이 비용 축은 시리즈 본론에서 빼지만, "같은 AZ로 모으면 지연도 비용도 준다"는 한 줄은 기억해 둘 만하다.)

## CLI로 직접 보기

```bash
# Pod 안에서 다른 Pod/노드까지 지연 재기
kubectl exec -it <pod> -- ping -c 5 <대상-pod-ip>

# 노드 간 지연 (EC2 사설 IP끼리)
ping -c 10 <다른-노드-사설IP>

# 어떤 CNI 모드인지 (Cilium 예시) — 오버레이 vs 네이티브 확인
cilium status | grep -i "routing\|tunnel"

# 같은 AZ vs 다른 AZ 비교: 두 노드가 어느 AZ인지 보고
kubectl get nodes -L topology.kubernetes.io/zone
```

같은 AZ 노드끼리와 다른 AZ 노드끼리 ping을 비교하면, 위 표의 ~0.5ms vs ~2ms가 네 클러스터에서 그대로 재현된다.

## 정리

- East-West(내부 통신)도 거리가 지배한다: **localhost(µs) < 같은노드 < 다른노드 < 같은AZ(~0.5ms) < 다른AZ(~2ms) < 다른리전.**
- 사이드카가 빠른 이유 = localhost 홉. 노드 경계는 **오버레이(캡슐화) vs 네이티브 라우팅**으로 갈린다.
- **AZ는 일부러 떨어뜨린 DC** → 가용성을 얻는 대신 지연이 붙는다.
- **Topology Aware Routing / locality LB**로 같은 AZ를 우선시켜 cross-AZ를 줄인다.
