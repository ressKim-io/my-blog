# 3편. 휴대폰에서 Pod까지 — North-South 여정 전체 지도

> **시리즈 "패킷은 어떻게 사용자에게 닿는가"의 3편**
> 2편이 서버 내부(East-West)였다면, 이번엔 바깥. 사용자 기기에서 우리 서버까지 패킷이 거치는 모든 정거장을 따라간다.

## 패킷은 직선으로 가지 않는다

도메인을 입력하고 화면이 뜨기까지, 패킷은 7~8개의 정거장을 거친다. 이 경로를 모르면 "어디서 느린지"를 영원히 추측만 하게 된다. 전체 지도부터 보자.

```
[사용자 기기]
  │ (1) 도메인 입력 → DNS 조회
  │     로컬캐시 → OS리졸버 → ISP리졸버 → 권한서버(예: Route53)
  │     ※ GeoDNS/Anycast로 "가까운 IP"를 돌려줌
  ▼
[접속망]
  │ 모바일: 기기 → 기지국 → 통신사 코어망 → 인터넷
  │ 유선PC: 기기 → 공유기 → ISP → 인터넷
  ▼
[ISP ↔ IXP/Peering]  ← 여러 네트워크(AS)를 경유할 수 있음
  ▼
[CDN Edge / PoP]  (사용자와 가장 가까운 지점)
  │
  ├── ★캐시 HIT → 여기서 즉시 응답! origin 안 감. (정적 파일/이미지)
  │
  └── 캐시 MISS or 동적 요청 ↓
        ▼
   [클라우드 리전 - VPC]
        │ ALB (L7 로드밸런서)
        ▼
   [Ingress Controller]  (또는 Istio Ingress Gateway)
        │ kube-proxy(iptables) or eBPF(Cilium)
        ▼
   [Service (ClusterIP)] → 엔드포인트 선택
        ▼
   [Pod]  ← 종착지
        │ (응답은 역순으로 돌아감)
        ▼
   (다시 사용자에게)
```

> 🎨 **그래픽 추천**: 이 전체 경로도가 시리즈에서 **가장 그래픽으로 만들 가치가 큰 그림**이다. 휴대폰 아이콘 → 기지국 → 클라우드 → Pod로 이어지는 가로 흐름도(flow diagram). CDN HIT 분기점을 색으로 강조하면 "여기서 끝날 수도 있다"가 한눈에 보인다.

## 정거장 하나씩

### (1) DNS — 가까운 IP를 고르는 단계

도메인 이름(`example.com`)을 IP 주소로 바꾸는 과정이다. 단순한 변환 같지만, 여기서 이미 "가까운 서버 고르기"가 시작된다.

```
GeoDNS / Anycast:
  같은 도메인이어도 "요청이 어디서 왔는지" 보고
  → 한국 사용자에겐 한국 IP, 미국 사용자에겐 미국 IP를 돌려줌
```

즉 DNS는 단순 주소록이 아니라, **사용자를 가장 가까운 입구로 안내하는 첫 분기점**이다. (이 "가까운 곳 고르기" 원리는 5편의 Anycast에서 더 깊게 다룬다.)

### (2) 접속망 — 모바일과 유선이 갈린다

여기가 앱 개발자에게 특히 중요하다. 사용자가 휴대폰이냐 PC냐에 따라 경로가 다르다.

```
모바일:  기기 ──무선──> 기지국 ──> 통신사 코어망 ──> 인터넷
         (셀룰러는 패킷 손실·지연 변동이 큼)

유선PC:  기기 ──> 공유기 ──> ISP ──> 인터넷
         (상대적으로 안정적)
```

> 🎨 **그래픽 추천**: 모바일 경로와 유선 경로를 위아래 두 갈래로 나란히 보여주는 분기 다이어그램. 모바일 쪽에 "손실·지연 변동 큼" 라벨을 붙이면 차이가 명확.

모바일은 무선 구간 때문에 패킷 손실과 지연 변동(jitter)이 크다. 이게 나중에(시리즈 외) HTTP/3(QUIC)가 모바일에서 강한 이유로 이어진다 — QUIC은 패킷 손실에 TCP보다 덜 취약하다.

### (3) CDN Edge — 끝날 수도 있는 지점

사용자와 가장 가까운 CDN 거점(PoP)이다. 여기서 요청이 **끝나버릴 수도** 있다. 이게 CDN의 핵심이다.

## 경우의 수 4가지

같은 서비스여도 요청 성격에 따라 경로 길이가 완전히 달라진다.

```
① 정적 콘텐츠 + CDN HIT  (이미지·JS·CSS, 이미 캐시됨)
   기기 → edge → 기기.   가장 빠름. origin 부담 0.

② 정적 콘텐츠 + CDN MISS  (첫 요청이라 캐시에 없음)
   기기 → edge → origin → edge가 캐싱 → 기기.
   첫 사용자만 느리고, 이후엔 ①이 됨.

③ 동적 API  (캐싱 불가, 매번 계산 필요)
   기기 → edge(통과/가속만) → origin → Pod → 역순.
   단, edge-origin이 warm 연결이라 그냥 인터넷보다 빠름.

④ 리전 위치 vs 사용자 위치
   서버가 서울인데 미국 사용자면 태평양 왕복 ~150ms.
   → CDN edge를 미국에 둬서 거리를 잘라냄.
```

> 🎨 **그래픽 추천**: 4가지 경우를 각각 짧은 경로/긴 경로 길이로 시각화하면(①이 가장 짧고 ④가 가장 김) "캐시되면 길이가 확 준다"가 직관적으로 보인다.

①과 ③의 차이가 곧 4편의 주제다. 정적인 것은 캐시로 거리를 잘라내지만, 동적인 것은 결국 Pod까지 가야 한다.

## CLI로 직접 보기

```bash
# 경로상의 정거장(홉)들을 순서대로 보기
traceroute example.com        # 리눅스/맥
tracert  example.com          # 윈도우

# DNS가 어떤 IP를 돌려주는지 (지역에 따라 다를 수 있음)
dig example.com +short
nslookup example.com

# 같은 URL을 CDN 경유 vs origin 직접으로 각각 찍어 비교 (1편 curl과 연결)
curl -w "connect:%{time_connect} ttfb:%{time_starttransfer}\n" \
-o /dev/null -s https://cdn.example.com/static/img.png
```

## 정리

- 패킷은 **기기 → DNS → 접속망 → CDN edge → ALB → Ingress → Service → Pod**의 정거장을 거친다.
- **DNS**부터 이미 "가까운 IP 고르기"가 시작된다(GeoDNS/Anycast).
- **모바일과 유선**은 접속망 경로가 다르고, 모바일은 손실·지연 변동이 크다.
- 요청은 **4가지 경우(HIT/MISS/동적/리전위치)**로 갈리고, 캐시되면 경로가 극적으로 짧아진다.
