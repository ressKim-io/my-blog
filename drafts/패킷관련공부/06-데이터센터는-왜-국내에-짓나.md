# 6편. 데이터센터는 왜 국내에 짓나 — 속도는 이유의 일부일 뿐

> **시리즈 "패킷은 어떻게 사용자에게 닿는가"의 6편(마지막)**
> 5편에서 "CDN 박스가 ISP 건물 안에 들어간다"고 했다. 이번엔 한 단계 더 큰 그림. 데이터센터를 통째로 짓고 입주 경쟁을 하는 진짜 이유들.

## "데이터센터에 들어간다"는 한마디에 4가지가 섞여 있다

뉴스에서 "어디가 데이터센터에 입찰해 들어갔다"는 말을 들으면, 사실 성격이 완전히 다른 여러 가지가 한 단어에 뭉뚱그려져 있다. 규모 순으로 4층위로 나눠야 보인다.

```
규모 작음 ──────────────────────────────────────> 규모 큼

① 임베디드 캐시      ② 코로케이션         ③ 리전/대형 자체 DC   ④ 자체 신축
  (박스 몇 대)        (랙 몇~수십 개)       (건물 전체 임대/구축)   (땅 사서 직접)

[ISP 건물 안에       [중립 DC 사업자       [클라우드가 DC를        [하이퍼스케일러가
 서버 박스 위탁]      건물에 랙 임대해       통째로 빌리거나         부지 매입해
 (5편 내용)          장비 반입]            지어서 운영]            직접 건설]

목적: 콘텐츠 캐싱     목적: 망 연결+호스팅   목적: 클라우드 서비스   목적: 초대형 용량
```

> 🎨 **그래픽 추천**: 4층위를 왼→오른쪽으로 규모가 커지는 단계 막대로. 각 단계에 대표 이미지(박스 1개 → 랙 → 건물 → 단지)를 두면 규모 차이가 직관적.

### ① 임베디드 캐시
5편에서 다뤘다. CDN 박스를 ISP 건물에 위탁 설치. 목적은 순수하게 "콘텐츠를 사용자 가까이". 규모 작음.

### ② 코로케이션 (Colocation) — "입찰해서 들어간다"의 진짜 정체

**중립적인 데이터센터 사업자가 지은 건물에, 여러 회사가 랙 공간을 빌려 자기 장비를 들여놓는 것**이다.

```
[중립 데이터센터 건물]
┌─────────────────────────────────────────────────┐
│ 랙구역 A: 게임회사 서버                            │
│ 랙구역 B: 핀테크 서버                              │
│ 랙구역 C: ☁️ 클라우드 Direct Connect 장비         │  ← 클라우드 "온램프"
│ 랙구역 D: ☁️ 다른 클라우드 연결 장비               │
│ 랙구역 E: 📦 CDN PoP (edge 노드)                  │
│ ─────────────────────────────────────────────── │
│ 🔌 IXP(인터넷 교환점): 모든 ISP가 여기서 1홉 직결  │  ← ★ 핵심
└─────────────────────────────────────────────────┘
```

왜 다들 들어오려 경쟁하나 — 이 건물 안에 **IXP(인터넷 교환점)** 가 있어서, 입주하면 주요 ISP·해외망과 **전부 1홉으로 직결(peering)** 되기 때문이다. 4편에서 "ISP 경계를 적게 넘을수록 빠르다"고 한 그 일이 *물리적으로 일어나는 장소*가 바로 이 건물이다.

```
비유: 공항 면세점 입점 경쟁.
  공항(IXP) 자체는 중립 사업자가 운영하고,
  거기 입점하면 전 세계 항공편(모든 ISP)과 바로 연결.
  그래서 다들 좋은 자리를 두고 경쟁한다.
```

### ③ 리전/대형 자체 DC
클라우드 사업자가 **클라우드 서비스(컴퓨팅·스토리지)를 제공하려고** DC를 통째로 빌리거나 짓는다. 이건 캐싱이 아니라 *연산·저장 자체를 국내에 두는 것*이다.

### ④ 자체 신축 (하이퍼스케일)
땅 사서 직접 건설. 요즘 "데이터센터 엄청 짓는다" 뉴스의 상당수가 이것 + AI 수요다.

## 진짜 질문: 왜 국내에 짓나 — 속도는 5개 중 1개

"사용자에게 빠르게"는 이유 중 하나일 뿐이다. 더 강한 이유들이 따로 있다. 중요도 순으로.

```
┌──────────────────────────┬────────────────────────────────────┐
│ 이유                      │ 설명                                 │
├──────────────────────────┼────────────────────────────────────┤
│ ① 데이터 주권/규제 (법)    │ ★최강. "한국인 데이터는 한국 안에"     │
│                          │ 라는 법·규제. 금융·공공·의료 특히.     │
│                          │ → 국내에 없으면 사업 자체가 불가        │
├──────────────────────────┼────────────────────────────────────┤
│ ② latency (속도)          │ 사용자 가까이 = 빠름. 게임·실시간·     │
│                          │ 스트리밍에 결정적. (이 시리즈 주제)     │
├──────────────────────────┼────────────────────────────────────┤
│ ③ AI / GPU 수요 폭증       │ 요즘 신축 러시의 진짜 원인.            │
│                          │ GPU 클러스터는 전력·냉각이 막대 →      │
│                          │ 전용 대형 DC 필요. (지금 제일 뜨거움)   │
├──────────────────────────┼────────────────────────────────────┤
│ ④ 가용성/재해복구 (DR)     │ 리전 여러 개 → 한 곳 터져도 버팀.       │
│                          │ AZ 분산(2편)도 같은 맥락.             │
└──────────────────────────┴────────────────────────────────────┘
```

> 🎨 **그래픽 추천**: 4가지 이유를 아이콘과 함께(법 = 저울, 속도 = 번개, AI = 칩, 가용성 = 우산) 카드 형태로. "속도는 그중 하나"라는 메시지를 위해 속도를 강조하지 *않는* 균형 잡힌 배치가 좋다.

### ① 데이터 주권 (data residency) — 왜 속도보다 센가

클라우드를 하면 계속 만나는 개념이다. **"특정 데이터는 국경 밖으로 못 나간다"**는 법·규제가 있다.

```
- 개인정보보호법 / 위치정보법  → 위치·개인 데이터
- 금융권 클라우드 규제         → 금융사 데이터는 국내 보관 강제
- 공공기관 (CSAP 인증)        → 공공 클라우드는 국내 리전 필수
```

그래서 글로벌 클라우드가 한국 리전을 *반드시* 둔다. 속도 때문이 아니라, **국내 리전이 없으면 금융·공공·의료 고객을 아예 못 받기 때문**이다. 속도는 "있으면 좋은 것", 규제는 "없으면 사업 불가"라 무게가 다르다.

### ③ 요즘 신축 러시의 진짜 주인공 — AI/GPU

"데이터센터 엄청 짓는다"는 뉴스 상당수는 CDN/속도가 아니라 **AI 때문**이다.

```
전통 DC 서버 랙: 5~10 kW
AI GPU 랙:      40~100+ kW   ← 전력·발열이 자릿수 단위로 다름

→ 기존 DC로 감당 불가 → 전력 끌어오기 쉬운 곳에 초대형 신축 러시
→ 입지 경쟁(전력·부지·냉각수)이 치열 = 뉴스의 "입찰" 분위기
```

> 🎨 **그래픽 추천**: 전통 랙(5~10kW)과 AI 랙(40~100kW)의 전력을 막대 높이로 비교. 자릿수 차이가 시각적으로 충격적이다.

이건 "사용자에게 빠르게"가 아니라 **"연산 능력 자체를 국내에 확보"**하는 게임이라, 속도 논리와는 결이 다르다.

## 정리 — "다 속도 때문이야?"에 대한 답

```
✅ 맞는 부분:
   IXP·코로케이션 입주 경쟁 → "망 1홉 직결 + 사용자 가까이"
   = 이 시리즈에서 배운 속도/위치 논리. 정확하다.

➕ 더 큰 부분:
   리전·대형 DC를 통째로 짓는 건 속도보다
   ① 규제(데이터 국내 보관 강제) — 없으면 사업 불가
   ③ AI/GPU 연산 수요 — 요즘 신축 러시의 실제 동력

➡ 결론:
   - CDN 박스 ISP 입주    → 순수 속도 (5편)
   - 코로케이션/IXP 입주   → 속도 + 망 연결 효율
   - 리전·대형 DC 신축     → 규제 + AI + 가용성 (속도는 부차적)
```

**한 문장**: 작은 캐시 박스를 ISP에 넣는 건 순수하게 속도 때문이 맞지만, 클라우드가 *리전·대형 DC를 통째로 짓는* 건 속도보다 **규제(데이터 주권)와 AI 연산 수요**가 더 큰 동력이다.

## 클라우드 한국 계층 매핑 (시리즈 총정리)

이 시리즈 전체를 한 장에 모으면, 클라우드 사업자가 한국에 두는 것들이 이렇게 대응한다.

```
[리전]            ← 컴퓨팅·스토리지 (③, 2편 EC2/Pod)
  └─[여러 AZ]      ← 물리적으로 떨어진 DC (2편, 가용성↔지연 트레이드오프)
[CDN edge PoP]    ← 사용자 가까이 콘텐츠 캐싱 (3·4·5편)
[Direct Connect]  ← 코로케이션 DC 안의 전용 연결점 (이번 편 ②)
[임베디드 캐시]    ← ISP 건물 안 (5편)
```

> 🎨 **그래픽 추천**: 한국 지도 위에 이 5가지를 점으로 찍는 지도 인포그래픽. 시리즈 전체를 한 장으로 마무리하는 "총정리 그림"으로 강력하다.

## CLI로 직접 보기

```bash
# 클라우드 edge가 정말 국내(가까이)에 있는지 — CDN URL vs origin 비교
traceroute <CDN-도메인>     # 몇 홉, 몇 ms에 도착하는지
traceroute <origin-도메인>  # 리전 origin과 비교

# 응답 헤더에서 어느 edge가 응답했는지 흔적 보기
curl -sI https://<CDN-도메인> | grep -iE 'x-cache|via|server|x-amz-cf-pop'
```

## 시리즈를 마치며

여섯 편을 관통한 한 줄은 이것이다.

> **속도는 거리와 왕복(RTT)이 지배하고, 빠르게 만드는 일은 결국 "거리를 줄이거나(캐시·CDN·임베디드) 왕복을 줄이는(연결 재사용·프로토콜)" 것이다.**

그리고 그 "거리"는 물리적 km만이 아니라, *인터넷이라는 네트워크의 연합 안에서 내가 어디에 꽂혀 있느냐*라는 위상적 거리이기도 하다. 집 PC는 망의 변두리 끝에, CDN은 중심에 있다 — 같은 케이블을 쓰면서도.

## 정리

- "DC에 들어간다"는 **4층위**(임베디드/코로케이션/리전/신축)로 나뉜다.
- 코로케이션 입주 경쟁의 핵심은 **IXP 1홉 직결**(공항 면세점 비유).
- 국내 건설의 진짜 동력은 속도뿐 아니라 **①규제(데이터 주권) ③AI/GPU 수요 ④가용성**이다.
- 특히 **규제**는 "없으면 사업 불가"라 속도보다 무겁고, **AI/GPU**가 요즘 신축 러시의 실제 원인이다.
