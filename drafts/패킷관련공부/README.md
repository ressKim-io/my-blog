# 시리즈: 패킷은 어떻게 사용자에게 닿는가 — 네트워크 지연의 해부

네트워크 지연(latency)을 밑바닥(빛의 속도)부터 꼭대기(데이터센터 입지 전략)까지 한 줄기로 따라가는 6부작.

**관통하는 한 문장**
> 속도는 거리와 왕복(RTT)이 지배하고, 빠르게 만드는 일은 결국 "거리를 줄이거나(캐시·CDN) 왕복을 줄이는(연결 재사용)" 것이다.

## 목차

| 편 | 제목 | 한 줄 |
|----|------|-------|
| **1** | [지연의 정체](./01-지연의-정체.md) | 느린 건 대역폭이 아니라 거리·왕복(RTT) 탓 *(허브 글)* |
| **2** | [서버 내부 통신 (East-West)](./02-서버-내부-통신.md) | 같은 거리도 레이어 따라 µs~ms로 갈린다 |
| **3** | [사용자까지의 여정 (North-South)](./03-사용자까지의-여정.md) | 휴대폰에서 Pod까지의 정거장 지도 |
| **4** | [CDN은 왜 빠른가](./04-CDN은-왜-빠른가.md) | 같은 한국인데도 빠른 이유, 변수 통제로 분해 |
| **5** | [인터넷의 물리적 실체](./05-인터넷의-물리적-실체.md) | ISP·AS·"ISP 건물 안의 CDN" |
| **6** | [데이터센터는 왜 국내에 짓나](./06-데이터센터는-왜-국내에-짓나.md) | 속도는 이유의 일부, 진짜 동력은 따로 *(총정리)* |

## 읽는 순서 / 묶음

```
[기초]   1 → 2 → 3      (지연 원리 + 내부/외부 경로)
[심화]   4 → 5          (CDN과 인터넷의 물리적 실체)
[조망]   6              (데이터센터 전략, 독립 완결)
```

- **1편이 허브**: RTT·지연 4성분·slow start를 정의하고, 2~6편에서 계속 역참조.
- **4·5·6편은 단독 검색 유입**도 잘 되는 주제라 개별 완성도를 높여 둠.

## 시리즈 고정 코너

매 편 끝에 **"CLI로 직접 보기"** 가 있다. 따라 해보면서 글 내용을 자기 환경에서 재현하는 코너.

| 편 | 핵심 명령어 |
|----|-------------|
| 1 | `curl -w` (단계별 시간 분해) |
| 2 | `ping`, `kubectl get nodes -L zone` (AZ 비교) |
| 3 | `traceroute`, `dig` |
| 4 | `curl -w` origin vs CDN, `tc netem`, `ethtool`, BBR |
| 5 | `traceroute`, `mtr`, `whois` (AS 확인) |
| 6 | `traceroute`, `curl -sI` (edge 헤더) |

## 그래픽 작업 메모

각 글 본문에 `🎨 그래픽 추천` 표시를 달아 뒀다. ASCII로 일단 그려 뒀지만, 실제 발행 시 그래픽으로 바꾸면 가시성이 크게 오르는 지점들이다. 우선순위가 높은 것:

1. **3편 — 전체 경로도** (기기→Pod 흐름도): 시리즈에서 가장 그래픽 가치 큼
2. **1편 — 핸드셰이크 시퀀스 다이어그램**: 왕복이 눈에 확 들어옴
3. **5편 — 망 계층 트리 + 데이터센터 랙 일러스트**: 충격 포인트 전달
4. **6편 — 한국 지도 인포그래픽**(총정리): 시리즈를 한 장으로 마무리
5. **4편 — 6요인 영향크기 막대**: "위치가 압도적, 박스는 작다" 메시지

## 별도 시리즈로 뺀 것

"네트워크 비용은 물리(거리)가 아니라 제도(피어링 모델)가 정한다" — 망사용료·상호접속고시·트위치 철수·클라우드 egress 비용 축은 의도적으로 제외했다. 이 시리즈와 겹치지 않으면서 또 한 묶음이 나온다.
