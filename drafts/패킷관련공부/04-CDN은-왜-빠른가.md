# 4편. 같은 한국인데 왜 CDN이 빠른가 — 변수를 통제해 보면

> **시리즈 "패킷은 어떻게 사용자에게 닿는가"의 4편**
> 1~3편에서 "거리가 지연을 지배한다"고 했다. 그런데 거리가 같아도 속도 차이가 난다. 이번엔 그걸 파헤친다.

## 거리를 똑같이 고정하면, 남는 차이는?

이런 경험이 있을 거다. 집에 서버를 깔아 파일을 호스팅했더니, 같은 한국 안 사용자인데도 상용 CDN보다 한참 느리다. **거리는 비슷한데(둘 다 한국) 왜?**

실험 설계의 기본을 쓰자. **변수를 통제한다.** "거리"를 똑같이 고정시켜 놓으면, 남는 속도 차이는 전부 *거리 외의 요인*이다. 그게 뭔지 6개로 쪼갠다.

```
집 로컬 PC vs 국내 CDN — 둘 다 한국. 거리 비슷. 그런데 왜 차이?

┌───────────────────────────┬─────────┬──────────────────────┐
│ 요인                       │ 영향크기 │ 한 줄 요약            │
├───────────────────────────┼─────────┼──────────────────────┤
│ ① 네트워크 위치 (peering)  │ ★★★★★  │ ISP 경계를 몇 번 넘나   │
│ ② 가정용 회선의 본질적 한계 │ ★★★★   │ 업로드/best-effort/공유 │
│ ③ 캐시 계층 (메모리/SSD)   │ ★★★    │ 디스크 안 읽고 RAM에서   │
│ ④ 서버 HW + 커널/TCP 튜닝  │ ★★★    │ NIC offload, BBR 등    │
│ ⑤ 연결 관리 (warm pool)    │ ★★     │ 핸드셰이크 재사용       │
│ ⑥ 분산/용량 (Anycast,대수) │ ★★     │ 한 대 vs 수십 대        │
└───────────────────────────┴─────────┴──────────────────────┘
```

> 🎨 **그래픽 추천**: 6요인을 영향 크기 순으로 쌓은 가로 막대(별점 길이)로 그리면 "위치·회선이 압도적이고 박스 성능은 의외로 작다"는 메시지가 시각적으로 명확해진다.

핵심 결론을 먼저: **"CDN이 빠른 건 더 좋은 컴퓨터라서"가 아니다. ①②(네트워크 위치와 회선 등급)가 압도적이고, 박스 성능(③④)은 의외로 작은 부분이다.**

## ① 가장 큰 것: 네트워크 위치

이건 5편에서 ISP·임베디드 캐시로 깊게 다루니, 여기선 결론만. **같은 한국이어도 "인터넷 안에서 몇 개의 ISP 경계를 넘느냐"가 다르다.**

```
가정용:  너집 → ISP A → (중간 경유) → ISP B → 목적지
         (거치는 네트워크가 많음 = 홉 많음 = 처리·큐잉 지연 ↑)

CDN:     사용자 → ISP ─직결─> CDN
         (중간 경유 생략 = 홉 ↓, 때론 사용자의 ISP 망 안에 직접 위치)
```

거치는 네트워크 수가 줄면 처리지연·큐잉지연이 같이 줄고, 중간에서 막힐 확률도 준다. 집 PC는 구조상 항상 망의 변두리 끝에 있다.

## ② 가정용 회선의 본질적 한계

집 PC를 아무리 좋은 사양으로 깔아도 못 넘는 벽이 있다. **회선 자체의 성질**이다.

```
┌──────────────┬─────────────────────────────────────────────┐
│ 한계          │ 설명                                          │
├──────────────┼─────────────────────────────────────────────┤
│ 비대칭(asym)  │ 가정용은 다운로드↑ 업로드↓. 서버는 "업로드"로   │
│              │ 서빙함 → 콘텐츠 주는 방향이 느린 회선.          │
├──────────────┼─────────────────────────────────────────────┤
│ best-effort  │ 품질 보장(SLA) 없음. 혼잡하면 그냥 느려짐.       │
├──────────────┼─────────────────────────────────────────────┤
│ 공유/초과가입  │ 같은 동네가 한 회선 공유. 저녁 황금시간엔        │
│ (contention) │ 이웃들과 경쟁 → 큐잉지연 ↑.                    │
├──────────────┼─────────────────────────────────────────────┤
│ NAT/공유기    │ 공유기가 NAT·세션테이블 처리. 동시연결 많으면   │
│              │ 공유기가 먼저 뻗음.                            │
├──────────────┼─────────────────────────────────────────────┤
│ 단일 회선     │ 회선 하나 끊기면 끝. 우회 경로 없음.            │
└──────────────┴─────────────────────────────────────────────┘
```

반면 CDN/IDC 회선은 **대칭·전용·SLA 보장·다중 업링크**다. 출발선이 다르다. 이건 박스 성능이 아니라 *회선 등급*의 문제다.

## ③ 캐시 계층 — 디스크를 안 읽는다

이제 "구성"이 실제로 다른 부분. CDN은 캐시를 계층으로 쌓는다.

```
요청 → [L1: RAM 캐시]  ── HIT ──> 즉시 응답 (메모리 속도, µs)
          │ MISS
          ▼
       [L2: NVMe SSD]  ── HIT ──> 빠른 응답
          │ MISS
          ▼
       [origin shield] ── 여러 edge의 같은 콘텐츠 요청을 묶어
          │              origin엔 1번만 (origin 보호)
          ▼
       [Origin (S3/ALB)]  ← 최후
```

> 🎨 **그래픽 추천**: 캐시 계층을 깔때기(funnel) 또는 피라미드로 그리면 "위로 갈수록 빠르고 좁다"가 직관적. 대부분 요청이 RAM 층에서 빠져나가는 흐름을 화살표로.

핫한 콘텐츠는 거의 **RAM에서** 나간다. 집 PC의 웹서버는 보통 매번 디스크에서 읽는다(OS 페이지 캐시는 있지만 의도적 다계층 설계는 아니다). 게다가 origin shield로 origin 부담까지 흡수한다.

## ④ 서버 HW + 커널/TCP 튜닝

박스 자체도 다르다. 단, ①②보다 영향은 작다.

```
┌────────────────────┬──────────────────────────────────────┐
│ 항목                │ CDN 노드는 이렇게                       │
├────────────────────┼──────────────────────────────────────┤
│ NIC offload         │ TSO/GRO/RSS/checksum offload로 CPU가   │
│                    │ 패킷 처리를 안 함. NIC이 처리.           │
│ 멀티큐 + RSS         │ NIC 큐를 CPU 코어별 분산 → 단일코어 병목 제거│
│ 커널 우회(일부)      │ DPDK/AF_XDP로 커널 스택 건너뛰어 지연↓   │
│ 혼잡제어 BBR        │ 손실 기반 CUBIC 대신, 대역폭·RTT 측정기반 │
│ initcwnd 튜닝       │ 초기 전송창 키워 slow start 초반 가속    │
│ 소켓 버퍼 autotune  │ BDP에 맞춰 버퍼 자동조정 → 긴 RTT도 채움 │
└────────────────────┴──────────────────────────────────────┘
```

여기서 1편의 TCP slow start가 다시 등장한다. BBR과 initcwnd 튜닝이 결국 "slow start를 빨리 벗어나기" 위한 것이다.

## ⑤⑥ 연결 관리와 분산

- **⑤ warm pool**: CDN edge는 origin과 미리 맺어둔 연결 풀을 유지한다. 사용자 요청이 와도 그 자리에서 핸드셰이크를 새로 안 한다(1편 RTT 절감). + TLS resumption, HTTP/2, HTTP/3까지 다 켜져 있다.
- **⑥ 분산**: 집 PC는 한 대다. 동시 접속이 몰리면 CPU·NAT·업로드 회선이 차례로 뻗는다. CDN은 여러 PoP + Anycast로 부하를 분산하고 DDoS도 흡수한다. **부하가 올라갈수록 격차가 더 벌어진다.**

## 핵심 반전

```
질문: CDN이 빠른 건 박스(구성)가 좋아서?
답:   구성도 다르다(③④⑤). 그런데 진짜 큰 건 위치와 회선(①②).

      집 PC를 CDN급 서버로 바꿔 깔아도 ──> 여전히 느림
      (가정용 회선 + ISP 경계는 못 바꾸니까)

      반대로 평범한 서버라도 ISP 안에 + IXP 직결 +
      대칭 전용회선에 두면 ──> 그것만으로 집 PC를 압도.
```

**한 문장**: CDN이 빠른 결정적 이유는 "더 좋은 컴퓨터"가 아니라, *사용자와 같은 ISP 안/IXP 1홉 거리에, 대칭 전용회선으로 꽂혀 있고, 핫 콘텐츠를 RAM에 들고 있어서 ISP 경계와 디스크 I/O를 둘 다 건너뛴다*는 것이다.

## CLI로 직접 보기

```bash
# 1편 curl을 origin vs CDN에 각각 찍어 비교 — 이 글의 핵심 실측
curl -w "connect:%{time_connect} tls:%{time_appconnect} ttfb:%{time_starttransfer}\n" \
-o /dev/null -s https://origin.example.com/img.png
curl -w "connect:%{time_connect} tls:%{time_appconnect} ttfb:%{time_starttransfer}\n" \
-o /dev/null -s https://cdn.example.com/img.png

# 내 리눅스 서버의 혼잡제어 확인 / BBR 켜기 (④ 실험)
sysctl net.ipv4.tcp_congestion_control
sudo modprobe tcp_bbr
sudo sysctl -w net.ipv4.tcp_congestion_control=bbr

# NIC offload 상태 (④ 확인)
ethtool -k eth0 | grep -E 'tso|gso|gro|rx-checksum'

# 인위적 지연/손실 주입 후 CUBIC vs BBR 비교 (홈랩 실험)
sudo tc qdisc add dev eth0 root netem delay 150ms loss 2%
# ...측정 후 원복
sudo tc qdisc del dev eth0 root
```

## 정리

- 거리를 통제하면 남는 차이는 **6요인**: 위치·회선·캐시계층·HW튜닝·연결관리·분산.
- **압도적인 건 ①네트워크 위치 ②회선 등급.** 박스 성능(③④)은 생각보다 작다.
- 가정용 회선은 **비대칭·best-effort·공유·NAT**라는 구조적 한계가 있다.
- CDN은 핫 콘텐츠를 **RAM에서** 내보내고, **warm 연결**로 핸드셰이크를 건너뛴다.
