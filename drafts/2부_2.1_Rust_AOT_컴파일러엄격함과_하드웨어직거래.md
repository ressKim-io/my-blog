# 2부. CPU를 쥐어짜는 전략: 기계어 생성과 런타임 최적화
## 2.1 Rust (AOT): 컴파일러의 엄격함 → 예측 가능한 하드웨어 직거래

> 1부에서 우리는 "런타임의 무게"를 5축으로 분해했고, Rust를 **제로코스트(런타임이 거의 없음)** 극단에 놓았다.
> 그런데 "런타임이 얇다"는 것과 "실행되는 기계어가 좋다"는 것은 다른 이야기다.
> 2부는 축을 바꾼다. 관심사는 이제 **누가, 언제, 어떤 정보로 기계어를 만드는가**이다.
> 2.1은 그 스펙트럼의 한쪽 끝 — "빌드 타임에 모든 것을 확정하고, 런타임에는 아무것도 바꾸지 않는" AOT의 순수 형태다.

---

### ① 도달 지점 (이 절을 읽으면 이해하게 되는 범위)

- `rustc → MIR → LLVM IR → 타깃 기계어` 파이프라인에서 **각 단계가 무엇을 결정**하고, borrow check가 왜 런타임 비용 0인지
- **단형화(monomorphization)** 가 제네릭을 어떻게 실제 기계어(`cmov`/`csel` 한 줄)로 녹여 없애는지 — 어셈블리로 직접 확인
- 제네릭/`#[inline]`이 **크레이트 경계를 넘는 물리적 메커니즘**(rlib 메타데이터에 MIR 직렬화)과, 그렇지 않은 함수가 왜 경계에서 막히는지
- `codegen-units`(릴리스 16 / 디버그 256)와 **ThinLTO의 함수 요약 인덱스**가 최적화 품질 ↔ 빌드 시간을 맞바꾸는 기전
- `dyn Trait`의 **fat pointer + vtable 레이아웃**과 간접 호출의 물리 비용
- `target-cpu`의 마이크로아키 레벨(x86-64-v1~v4 / armv8-a)과 SIGILL(#UD)의 실체, **SVE VLA**가 AOT에 주는 선물(Graviton3 2×256b vs Graviton4 4×128b 정밀 비교)
- AOT 언어의 프로파일 최적화 실측(PGO ~10%+, rustc 자체 ~15%, BOLT 코드 재배치) — JIT만의 전유물이 아니라는 복선

---

### ② 개념 / 물리적 실체 — 기계어는 "언제" 존재하게 되는가

Rust는 순수 **AOT**다. `cargo build --release`가 끝나는 순간, CPU가 실행할 **모든 기계어**는 이미 ELF `.text`에 확정돼 있다. 실행 중 새 명령이 생기는 일은 없다.

```
[소스 .rs]
    │  파싱 · 타입 추론 · trait 해석
    ▼
[HIR]  문법에 가까운 고수준 IR
    │  desugaring (for→loop, ?→match, 클로저→구조체+호출 …)
    ▼
[MIR]  기본블록(BB)의 CFG + statement/terminator  ★rustc 고유
    │   • borrow check = MIR 위 데이터플로 분석(NLL). 통과하면 흔적 0.
    │   • drop 자동 삽입 (각 값의 소멸 지점을 terminator로 확정 = RAII)
    │   • MIR 최적화: const-prop, MIR inlining, dataflow DCE
    │     └─ 목적: LLVM에 넘길 IR "부피 자체"를 줄인다 (뒤의 컴파일 시간과 직결)
    ▼
[LLVM IR]  단형화 인스턴스가 여기서 폭발적으로 증식
    │   ┌──────────────────────────────────────────────┐
    │   │  LLVM 최적화(-O3): 진짜 성능의 대부분이 여기서 결정   │
    │   │  inline · loop-unroll · SLP/loop 벡터화          │
    │   │  GVN · 레지스터 할당(그리디) · 명령 스케줄링          │
    │   └──────────────────────────────────────────────┘
    ▼
[타깃 기계어]  x86-64  또는  AArch64
```

**핵심: 컴파일러가 둘이고, 역할이 갈린다.**
- **rustc 프런트엔드(MIR까지)** = Rust 고유의 안전성·의미 결정. borrow checker는 여기 살며, 통과 시 **런타임 비용 0인 컴파일 타임 계약**만 남긴다("이 참조는 유효", "drop은 여기").
- **LLVM 백엔드(LLVM IR부터)** = 인라이닝·루프·벡터화·레지스터 할당 등 성능을 좌우하는 고전 최적화 전부. → **Rust의 런타임 속도는 상당 부분 LLVM 품질에 위임**돼 있다. (그래서 "rustc가 느리다"는 대개 "LLVM에 넘긴 IR이 방대하다"는 뜻이다.)

> 물리 확인: `cargo build --release && objdump -d target/release/<bin>` 하면 `.text`에 박힌 확정 기계어가 보인다. 몇 번을 돌려도 이 바이트는 불변이다. **이 불변성**이 2.1 전체의 주제 — 그리고 2.3 JVM 코드 캐시의 "실행마다 달라지는 기계어"와 정확히 대척점이다.

### ③ 비용·메커니즘 해부 — 단형화라는 파우스트의 거래

#### (1) 단형화: 추상화가 기계어에서 사라지는 순간 (어셈블리로 확인)

`fn max<T: Ord>(a: T, b: T) -> T`는 **쓰인 구체 타입마다 별도 함수로 복제**된다. `T = i32` 인스턴스를 `-O`로 뽑으면:

```
// Rust
pub fn max_i32(a: i32, b: i32) -> i32 { if a > b { a } else { b } }

; x86-64 (SysV: a=edi, b=esi, ret=eax)      ; AArch64 (AAPCS64: a=w0, b=w1, ret=w0)
max_i32:                                     max_i32:
    cmp   edi, esi                               cmp   w0, w1
    mov   eax, esi                               csel  w0, w0, w1, gt
    cmovg eax, edi   ; 분기 없는 조건이동            ret
    ret
```

제네릭·`Ord` 트레이트·`if/else`가 **분기 없는 `cmov`/`csel` 한 줄**로 증발했다. vtable 조회도, 런타임 디스패치도 없다. 이것이 "제로코스트"의 물리적 실체다. `T = String`이면 별도 함수(`call <str as Ord>::cmp`)가 또 생성된다.

**대가는 두 가지, 그리고 둘 다 정량적이다.**
- **코드 팽창**: 같은 로직이 타입마다 복제 → `.text` 증가 → **i-cache/iTLB 압력**. (제로코스트 추상화가 캐시 미스로 되돌아오는 역설.) `Vec<T>`·`HashMap<K,V>`·이터레이터 어댑터 체인이 수십 인스턴스로 불면 바이너리가 수 MB씩 커진다.
- **컴파일 시간 폭발**: LLVM이 삼켜야 할 IR 부피 ≈ (단형화 인스턴스 수 × 함수 크기). `serde`·`diesel`처럼 제네릭이 깊은 크레이트가 느린 근본 원인이 이것이다.

#### (2) 크레이트 경계를 넘는 인라이닝의 물리 — 왜 어떤 건 넘고 어떤 건 막히나 ★

앞 절에서 "제네릭·`#[inline]`은 메타데이터로 넘어간다"고 했는데, 그 **메커니즘**은 이렇다.

```
크레이트 A를 컴파일 → libA.rlib 생성. rlib 안에는 두 종류가 공존:
  (1) 이미 기계어가 된 함수  → 객체코드. 다른 크레이트에서 "속을 못 봄"(불투명).
  (2) MIR로 직렬화된 함수    → 제네릭 함수 + #[inline] 함수(+작은 함수 휴리스틱).
        └ 크레이트 B가 이 MIR을 읽어 "자기 쪽에서" 단형화/인라인/codegen.

∴ 기본(LTO off)에서:
    - 일반 non-generic·non-#[inline] 함수 호출 = 크레이트 경계에서 CALL로 남음(인라인 불가)
    - 제네릭/#[inline] 함수 = 경계 무관하게 호출부에 인라인 가능
```

즉 "경계를 넘는다"는 건 비유가 아니라 **rlib에 함수의 MIR이 실제로 직렬화되어 하위 크레이트가 재컴파일하는 것**이다. 그래서 라이브러리 저자가 핫 함수에 `#[inline]`을 붙이는 건 "경계 너머 인라인을 허가"하는 물리적 스위치다. 경계를 광범위하게 허물려면 → LTO(아래 (4)).

#### (3) 정적 vs 동적 디스패치 — fat pointer와 vtable의 실제 레이아웃

`dyn Trait`는 코드 팽창을 막는 대신 **런타임 간접성**을 남긴다.

```
&dyn Draw  =  ( data_ptr , vtable_ptr )   ← 2워드 "fat pointer"

vtable (.rodata에 정적 생성, 타입마다 1개):
  [0] drop_in_place 함수 포인터
  [1] size            (바이트)
  [2] align           (바이트)
  [3] Draw::method0 함수 포인터
  [4] Draw::method1 함수 포인터
  ...

호출 x.method0():                 ; AArch64 예
    ldr  x8, [x_vtable, #24]       ; 슬롯 로드 (오프셋 3*8)
    blr  x8                        ; 간접 호출 → BTB 예측 실패 시 파이프라인 손해, 인라인 불가
```

Rust가 "제로코스트"인 건 **당신이 정적 디스패치를 골랐을 때** 이야기다. `dyn`을 쓰면 그 비용(간접 분기 + 인라인 차단)이 **명시적으로 보인다** — "비용을 숨기지 않는다"는 철학의 실물. (참고: 이 vtable 간접 호출은 2.2 Go의 `itab`, 2.3 Java의 megamorphic 호출과 같은 계열의 비용이다. 셋을 나란히 두면 "다형성의 물리 가격표"가 보인다.)

#### (4) codegen-units와 LTO — 최적화 지평 ↔ 빌드 시간의 물리

**codegen-units(CGU)**: rustc는 크레이트의 단형화 아이템들을 **N개 CGU로 분할**해 LLVM을 병렬 구동한다.

```
릴리스 기본 = 16,  디버그 기본 = 256,  최대 최적화 = 1
┌───┬───┬───┬───┐  병렬 16개               ┌───────────────┐  직렬 1개
│CGU│CGU│CGU│CGU│  → 빌드 빠름              │   단일 CGU     │  → 빌드 느림
└───┴───┴───┴───┘  → CGU 경계 넘는 인라인 X  └───────────────┘  → 크레이트 전체가 한 지평
```
CGU가 많을수록 병렬성↑·빌드↑빠름, 그러나 **CGU 경계를 넘는 인라이닝이 막혀** 최적화 기회를 잃는다. `codegen-units=1`은 그 반대 극단.

**LTO(Link-Time Optimization)** — 크레이트 경계를 링크 시점에 허무는 관문. 핵심은 **ThinLTO의 기전**이다.

```
thin LTO:
  1) 각 모듈이 "함수 요약(summary)"을 방출
       = 콜그래프 · 함수 크기 · 심볼 링크지 등 (본문 IR이 아니라 메타 요약)
  2) thin-link 단계: 요약 인덱스를 보고 "어느 함수를 어디로 import할지" 결정
  3) 백엔드를 병렬로 돌리되, import 결정된 함수만 끌어와 인라인
  → 크레이트 경계 인라인의 대부분 이득을, 병렬성을 크게 잃지 않고 얻음 (실전 권장)

fat LTO (= true):
  전체 크레이트 IR을 하나의 모듈로 합쳐 통째 재최적화
  → 최고 성능 / 최장 빌드 / 최대 메모리 (대형 프로젝트에서 링크가 병목)
```

ThinLTO가 "요약 인덱스로 import를 고른다"는 점이 fat LTO(전부 합침)와의 결정적 차이다 — **전역 최적화의 90%를 병렬성 손실 최소로** 얻는 설계다.

### ④ 회피 / 최적화 전략 — AOT를 "제대로" 쓰는 법

**릴리스 최대 성능 프로파일 (각 노브의 기전 주석):**
```toml
[profile.release]
opt-level = 3          # LLVM -O3 (루프·SLP 벡터화, 공격적 인라인 활성)
lto = "thin"           # 크레이트 경계 인라인 (요약 인덱스 기반). "fat"=최대/최장빌드
codegen-units = 1      # 단일 최적화 지평. 병렬 빌드를 희생하고 인라인 기회 최대화
panic = "abort"        # unwind 테이블(.eh_frame) 제거 → .text↓, landing pad 정리 없어짐
strip = "symbols"      # 심볼 제거 → 바이너리↓ (디버그심볼은 split-debuginfo로 분리 보관)
```

**`target-cpu` — 성능이 아니라 배포 전략의 문제, 그리고 SIGILL의 실체:**
```bash
# x86-64 마이크로아키 레벨(누적):
#   v1 = 베이스라인(SSE2)   ← 기본값. 어디서나 돌지만 SIMD 최소
#   v2 = +SSE4.2 (≈Nehalem)
#   v3 = +AVX2   (≈Haswell)
#   v4 = +AVX-512
# aarch64 기본 = armv8-a
RUSTFLAGS="-C target-cpu=x86-64-v3"     # AVX2 보장되는 최신 x86 풀 대상
RUSTFLAGS="-C target-cpu=znver4"        # AMD Genoa (AVX-512)
RUSTFLAGS="-C target-cpu=neoverse-v2"   # AWS Graviton4 (SVE2)
```
기본이 `v1`/`armv8-a`로 보수적인 이유: 빌드 기계 ≠ 배포 기계일 수 있어서다. `target-cpu=native`로 AVX-512를 켜 빌드한 바이너리를 그 확장이 없는 CPU에서 돌리면, CPU가 미지원 옵코드에서 **#UD(invalid opcode) 예외 → 커널이 SIGILL로 프로세스 종료**시킨다. 즉 이건 "배포 타깃을 얼마나 특정할 수 있나"의 문제다.

**PGO / BOLT — AOT도 실행 프로파일을 먹는다 (★복선, 실측 포함):**
```bash
# cargo-pgo 로 PGO+BOLT 결합 (rustc 배포 빌드가 쓰는 방식과 동일 계열: opt-dist)
cargo pgo instrument   # 계측 빌드
./app < 대표트래픽      # 프로파일(.profraw) 수집
cargo pgo optimize     # 프로파일로 재빌드 (인라인·코드레이아웃·레지스터할당 재조정)
cargo pgo bolt build --with-pgo   # 링크 후 BOLT로 basic-block 재배치
```
- **실측**: PGO 단독으로 대체로 **런타임 10%+**, rustc 자체 사례 **최대 ~15%**. BOLT는 여기에 **핫 basic-block을 물리적으로 모아** i-cache/iTLB 미스를 줄여 추가 이득.
- **핵심 통찰**: **PGO는 JIT의 전유물이 아니다.** Rust·Go(2.2)·GraalVM native-image(2.4) 모두 "실행 프로파일을 빌드 타임에 먹인다." 차이는 딱 하나 — **반영 시점**이다. AOT PGO는 *빌드 시 1회*, JIT은 *실행 내내*. 이 축이 2부 전체의 척추다.

**할당자에 대한 정확한 사실 (3부 복선 정정 씨앗):** `Vec`·`Box`·`String`은 `brk`/`mmap`을 **직접 부르지 않는다.** `#[global_allocator]`(기본 = 시스템 `malloc`)에 위임하고, `malloc`이 그 아래에서 `brk`/`mmap`을 관리한다. `#[global_allocator]`로 jemalloc/mimalloc으로 교체 가능. (3부에서 할당자↔커널 계층을 정확히 해부한다.)

**벡터화와 ARM의 선물 — Graviton 정밀 비교:**
- LLVM 자동 벡터화(루프 벡터라이저 + SLP)를 쓰거나, `std::simd`(포터블 SIMD, nightly)로 명시.
- **x86 고정폭의 함정**: AVX2(256b)로 짠 코드와 AVX-512(512b) 코드는 *서로 다른 기계어*다. AOT로 하나만 고르면 다른 CPU에서 폭을 못 쓴다 → `#[target_feature]` + `is_x86_feature_detected!`로 **여러 버전을 넣고 런타임 디스패치**(function multiversioning)해야 한다.
- **ARM SVE/SVE2의 VLA(벡터-길이-불가지론)**: SVE 코드는 벡터 폭을 **하드코딩하지 않는다**(`vscale` 기반). 같은 바이너리가 하드웨어 폭에 맞춰 실행된다.
  ```
  Graviton3 (Neoverse V1) : SVE  2 × 256-bit 파이프  → 벡터op당 폭 256b
  Graviton4 (Neoverse V2) : SVE2 4 × 128-bit 파이프  → 벡터op당 폭 128b, 파이프 2배
  → VLA 코드는 재컴파일 없이 양쪽에 적응 (폭이 코드에 박히지 않음)
  ```
  이 성질이 AOT의 최대 약점("빌드 시 타깃 못 정하면 손해")을 **아키텍처 차원에서** 완화한다. (반대로 벤치할 땐 "코드 개선 vs 하드웨어 폭 차이"를 분리하기 까다로워진다 — ⑦.)

**빠른 개발 루프:** `cranelift` 백엔드(디버그용, LLVM보다 훨씬 빠른 코드 생성/낮은 최적화)로 반복 빌드를 단축. 최적화는 릴리스에서 LLVM으로.

### ⑤ 세 언어 수렴 — 주제: "기계어의 예측 가능성과 하드웨어 직거래"

```
                Rust (2.1)          Go (2.2)             Java (2.3)
─────────────  ──────────────────  ──────────────────  ──────────────────────
기계어 생성 시점  빌드 타임(AOT)       빌드 타임(AOT)        런타임(JIT), L0→L3→L4
백엔드          LLVM(깊은 최적화)     자체 SSA(얕고 빠름)    C1/C2(프로파일 기반)
추상화 소거      단형화 → cmov/csel   단형화 제한 / 인터페이스 런타임 역가상화 + 인라인
                (제로코스트)          는 itab 간접호출        (실측 타입프로파일)
경계 인라인       #[inline]/제네릭 =    패키지 경계 보수적       인라인 자체가 런타임 결정
                MIR 직렬화; LTO로 확장
아키 특화        target-cpu 빌드시결정  GOARCH 빌드시고정      런타임 실행CPU 감지→그 아키 ★
프로파일 반영     PGO/BOLT(빌드 1회)   PGO(빌드 1회,1.21+)   실행 내내 연속
실행 예측성      ★최고(불변 .text)    높음                 낮음(웜업·deopt 스파이크)
tail latency    ★p99≈p50            안정                 불안정(웜업·deopt·세이프포인트)
```

**핵심 대비**: Rust·Go는 둘 다 AOT지만 **철학이 정반대**다 — Rust는 LLVM에 기대 "느리게·깊게"(코드 팽창·긴 컴파일 감수), Go는 "빠르게·얕게"(2.2). Java는 축이 아예 달라, 빌드 타임엔 아키-불가지론 바이트코드만 만들고 **진짜 기계어 생성을 런타임 JIT에 미룬다**(2.3). 그래서 Java만이 "실행 CPU를 실제로 보고 그 아키(AVX-512냐 SVE2냐) 기계어를 뽑는" 사치를 부린다 — Rust는 그걸 빌드 시 `target-cpu`로 *미리* 결정해야 한다.

### ⑥ 트레이드오프 소결

> **컴파일러에게 최대 정보(타입·소유권·수명)를 주고, 빌드 타임 비용(느린 컴파일·코드 팽창)을 선불로 치르는 대신, 런타임엔 얇고 예측 가능한 기계어를 받는다.**

- **강점의 물리**: `.text`가 불변이므로 ⓐ 분기 예측기가 같은 패턴으로 학습되고, ⓑ JIT 재컴파일로 인한 실행 중단(코드 캐시 갱신)이 없고, ⓒ GC 스톱이 없다 → **p99가 p50에 붙는다**(결정론적 tail latency). 이것이 1부 "제로코스트"의 성능적 실체다.
- **약점의 물리**: 정적이라 **실행 실측 기반 재최적화를 스스로 못 한다.** 어떤 분기가 실제 뜨거운지, 어떤 `dyn` 호출이 사실상 단형인지를 런타임에 배워 반영하지 못한다. PGO/BOLT는 *부분* 보완이지만 "빌드 시 1회 스냅샷"일 뿐, JIT처럼 상황 변화에 적응하지 않는다.

즉 Rust는 **예측 가능성(tail latency)** 을 **적응성(peak throughput 상한)** 과 맞바꾼다. 이 맞바꿈이 왜 롱런 서버에서 역전될 수 있는지는 2.3~2.4에서 회수한다.

### ⑦ DevOps 관점 심화 질문

1. **멀티 마이크로아키 노드풀**: `target-cpu=neoverse-v2`로 빌드한 바이너리를 Graviton3(Neoverse V1) 노드에 스케줄하면 미지원 SVE2 옵코드에서 SIGILL(#UD)이 난다. 노드풀을 아키별로 분리(`nodeSelector`/taint)할지, 런타임 CPU 디스패치(multiversioning)로 단일 이미지를 유지할지 — 이미지 수·바이너리 크기·운영 복잡도의 트레이드오프는?

3. **프로덕션 PGO 파이프라인**: 대표성 있는 `.profraw`를 뽑으려면 실트래픽 분포를 캡처해야 한다. 카나리에서 `cargo pgo instrument` 빌드를 소수 비율로 굴려 프로파일을 수집하는 파이프라인 설계는? 트래픽 패턴이 바뀌어 **프로파일이 낡으면 오히려 느려지는** 위험을 어떻게 감시?

5. **정적 바이너리 관측성**: `strip`한 릴리스 바이너리에 eBPF `uprobe`·`perf`가 심볼을 붙이려면? `split-debuginfo`로 디버그심볼을 별도 아티팩트로 관리하고 프로덕션엔 스트립 바이너리를 배포하는 워크플로(심볼 서버) 구성은?

6. **SVE 벤치 정규화**: VLA 바이너리를 Graviton3(2×256b)·Graviton4(4×128b)에서 벤치할 때 벡터 폭·파이프 수 차이를 어떻게 정규화해 "코드 개선"과 "하드웨어 차이"를 분리하나? `vscale`별 롤라인(roofline) 분석 설계는?

### ⑧ 다음 절 연결

Rust는 **LLVM에 깊이 기대는 "느리지만 정밀한" AOT**다 — 단형화를 `cmov` 한 줄로 녹이고, ThinLTO로 경계를 허물고, PGO/BOLT로 프로파일까지 먹는다. 스펙트럼의 정반대편에 Go가 있다: **자체 백엔드로 "빠르지만 얕은" AOT**. Go는 LLVM을 일부러 쓰지 않고 공격적 최적화를 일부러 포기했다 — 인라인 예산은 **cost 80**에 묶여 있고, 자동 벡터화는 아예 없다. 왜 언어 설계자가 "덜 최적화하는" 결정을 내렸으며 그 천장은 어디인가? 그리고 널리 퍼진 오해("Go엔 PGO가 없다")를 바로잡으며 **정적 컴파일의 진짜 한계**를 짚는다. 2.2로 간다.
