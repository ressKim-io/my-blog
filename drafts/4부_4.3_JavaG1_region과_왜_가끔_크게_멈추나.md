# 4부 4.3 — Java G1: region과 "왜 대체로 안정적이나 가끔 크게 멈추나"

(이 절의 G1 기본값은 샌드박스의 OpenJDK 11로 실측했으며, 여기 인용한 핵심 기본값은 JDK 25까지 동일합니다.)

## ① 도달 지점 — 이 절에서 얻어가는 것

4.2에서 Go의 GC를 봤습니다. 비세대·비이동이라 매 사이클 살아있는 힙 전체를 다시 훑었고, 그래서 디스코드처럼 거대한 캐시를 이고 있으면 2분마다 지연이 튀었습니다. 자연스러운 다음 질문은 이것입니다. **세대를 나누고(오래된 건 덜 훑고) 객체를 옮기면(단편화를 없애면) 이 문제가 풀릴까?** Java의 G1이 정확히 그 길을 택한 GC입니다.

이 절을 끝내면 다음을 스스로 설명할 수 있습니다.

- G1의 **region 모델** — 힙을 똑같은 크기의 조각으로 쪼개고 각 조각에 역할(eden·survivor·old·humongous)을 동적으로 붙이는 구조, 그리고 "Garbage-First"라는 이름의 뜻
- **young GC / mixed GC / Full GC**가 각각 무엇이고 언제 도는가
- **동시 마킹**이 언제 시작되는가 — IHOP 기본 45%
- **evacuation pause**(이사 멈춤)의 실체 — 살아있는 객체를 복사하며 압축하는 STW 비용, 그리고 `MaxGCPauseMillis`(기본 200) 목표
- **humongous 객체**(3.4에서 예고)가 왜 특별 취급되고, 왜 동시 마킹을 촉발하는가
- 그리고 4.3의 제목, **"왜 G1은 대부분 매끄럽다가 가끔 절벽처럼 크게 멈추는가"** — Full GC가 터지는 순간의 물리

3.4에서 "humongous가 50% region을 넘으면 동시 마킹을 유발한다"고 예고한 대목을 여기서 회수합니다.

## ② G1의 실체 — region 모델, 세대별이되 물리적으로 흩어진

G1은 **Garbage-First Garbage Collector**의 줄임말입니다. 이름에 설계 철학이 담겨 있는데, "쓰레기가 가장 많은 영역부터(garbage first) 회수한다"는 뜻입니다. 왜 그런 이름이 붙었는지 이해하려면 먼저 힙을 어떻게 쪼개는지부터 봐야 합니다.

**region이란, 힙을 똑같은 크기로 잘게 나눈 조각입니다.** 전통적인 세대별 GC는 힙을 young(eden+survivor)과 old라는 큰 연속 영역으로 나눴습니다. G1은 다릅니다. 힙 전체를 동일 크기의 region 수천 개로 쪼갠 뒤, 각 region에 "너는 eden, 너는 old, 너는 humongous"라는 **역할을 동적으로** 붙입니다. 그래서 G1에서 "young 세대"란 연속된 물리 공간이 아니라, 지금 eden·survivor 역할을 맡은 region들의 논리적 집합입니다.

```
 전통적 세대별(연속):            G1(region: 크기 같고, 역할은 흩어져 배정):
 ┌───────────┬───────────┐      ┌──┬──┬──┬──┬──┬──┬──┬──┐
 │   young   │    old    │      │E │O │O │F │S │O │E │H │
 │(eden+surv)│           │      ├──┼──┼──┼──┼──┼──┼──┼──┤
 └───────────┴───────────┘      │O │F │E │O │H │H │F │O │ ← H 연속 2칸 = humongous
   큰 영역 2개가 물리적으로 연속    ├──┼──┼──┼──┼──┼──┼──┼──┤
                                 │F │E │O │S │O │F │O │E │
                                 └──┴──┴──┴──┴──┴──┴──┴──┘
                                 E=eden  S=survivor  O=old  H=humongous  F=free
                                 모든 region 크기 동일(예: 1MB), 역할은 매 순간 바뀜
```

왜 이렇게 쪼갤까요. **부분 회수(incremental collection)와 멈춤 시간 예측**을 위해서입니다. 힙을 통째로 훑는 대신 "이번 멈춤에는 이 region들만 회수하자"라고 부분집합을 고를 수 있고, region 하나를 비우는 데 드는 시간을 예측해 **목표 멈춤 시간(`MaxGCPauseMillis`) 안에 들어갈 만큼만** 골라 담을 수 있습니다. 이때 고르는 기준이 바로 "쓰레기가 가장 많은 region부터"입니다. 같은 시간을 쓴다면 쓰레기가 꽉 찬 region을 비워야 회수 효율이 최대가 되기 때문입니다. 이것이 Garbage-First라는 이름의 정체입니다.

그리고 G1은 **이동식(moving)**입니다. region을 회수할 때 그 안의 살아있는 객체를 **새 region으로 복사(evacuation, 이사)**하고 원래 region은 통째로 비웁니다. 복사하며 객체들이 새 region에 촘촘히 다시 담기니 단편화가 사라집니다. 4.1 삼각형에서 Go가 택한 왼쪽(단편화를 감수하는 비이동)과 정반대인 오른쪽(이동 비용을 내고 단편화를 없애는) 꼭짓점입니다.

**실측한 G1 기본값(OpenJDK 11, JDK 25도 동일).** 아래는 샌드박스에서 `java -XX:+UseG1GC -XX:+PrintFlagsFinal`로 직접 뽑은 값입니다.

| 플래그 | 실측 기본값 | 뜻 |
|---|---|---|
| `UseG1GC` | `true` | JDK 9부터 기본 GC |
| `G1HeapRegionSize` | 힙 2GB→1MB (=정확히 2048개) | region 크기. **에르고노믹 자동**은 힙/2048을 2의 거듭제곱으로 **1–32MB** 선택. **수동 지정 시 최대 512MB**(JDK 18+, 표 아래 ★) |
| `MaxGCPauseMillis` | 200 | 목표 멈춤 시간(보장 아님). 웹에 도는 "250"은 오류 |
| `InitiatingHeapOccupancyPercent` | 45 | old 점유율이 이 %를 넘으면 동시 마킹 시작(IHOP) |
| `G1UseAdaptiveIHOP` | `true` | IHOP를 실행 중 적응적으로 조정 |
| `G1MixedGCLiveThresholdPercent` | 85 | old region이 이보다 더 살아있으면 mixed 대상에서 제외(복사 비용 아까움) |
| `G1MixedGCCountTarget` | 8 | 한 마킹 사이클 뒤 mixed GC를 최대 몇 번에 나눠 돌지 |
| `G1HeapWastePercent` | 5 | 회수 가능한 쓰레기가 힙의 이 % 밑으로 떨어지면 mixed 종료 |
| `G1ReservePercent` | 10 | evacuation 실패 대비로 비워 두는 예비 힙 비율 |
| `ParallelGCThreads` / `ConcGCThreads` | 4 / 1 | STW 병렬 워커 / 동시 마킹 워커(코어 4개 기준, 동시는 ¼) |

`ConcGCThreads`가 `ParallelGCThreads`의 약 ¼인 점을 눈여겨보세요. 4.2에서 Go가 마킹에 CPU의 25%를 쓰던 것과 같은 감각입니다. 동시 마킹은 뮤테이터에게 대부분의 코어를 남겨 두고 배경에서 진행합니다.

**★ region 크기 상한 (JDK 18+).** 예전에는 region이 최대 32MB였습니다. 리멤버드 셋이 카드(512바이트) 단위로 region을 매핑했는데 카드 인덱스가 16비트라, 2^16 × 512B = 32MB가 물리적 상한이었습니다. JDK 18(JDK-8276929)에서 리멤버드 셋 구현이 이 1:1 매핑을 없애면서 상한이 **512MB로 확장**됐습니다. 다만 **에르고노믹 자동 선택은 여전히 32MB까지만** 고릅니다. 512MB급 region은 대형 배열이 humongous로 분류돼 단편화를 일으키는 워크로드에서 `-XX:G1HeapRegionSize`로 직접 지정할 때만 쓰입니다. region을 키우면 humongous 임계(50%)도 함께 커져 큰 객체가 일반 객체로 처리됩니다.

## ③ 비용·메커니즘 해부 — young / 동시 마킹 / mixed, 그리고 humongous

G1이 도는 방식은 세 종류의 수집과 그 사이에 끼는 한 번의 동시 마킹 사이클로 이루어집니다. 흐름부터 그린 뒤 하나씩 풉니다.

```
 G1 사이클 흐름:

  [Young GC]─[Young GC]─[Young GC]─ ...   (eden 다 차면 반복, STW 짧음·pause 목표 안)
       │
       │ old 점유율이 IHOP(45%)를 넘으면 다음 young GC에 얹어 시작
       ▼
  ┌─ 동시 마킹 사이클(대부분 뮤테이터와 나란히) ──────────┐
  │  Concurrent Start (STW, young GC에 편승해 루트 스캔)   │
  │   → Root Region Scan (동시)                           │
  │   → Concurrent Mark  (동시, SATB 쓰기 배리어: 4.1)     │
  │   → Remark           (STW, SATB 버퍼 마무리)          │
  │   → Cleanup          (STW 짧음, 텅 빈 region 즉시 반납) │
  └────────────────────────────────────────────────────────┘
       │ 이제 "어느 old region에 쓰레기가 많은지" 파악됨
       ▼
  [Mixed GC]─[Mixed GC]─ ...   (young 전부 + old '일부'를 함께 evacuate, 최대 8회)
       │
       ▼
  회수 가능한 쓰레기가 5%(G1HeapWastePercent) 밑이면 mixed 종료 → 다시 [Young GC]
```

**Young GC(STW evacuation).** eden이 꽉 차면 발동합니다. eden과 survivor의 살아있는 객체를 새 survivor 또는 old region으로 evacuate하고, 비워진 region을 반납합니다. 항상 STW지만 young 영역만 대상이라 짧고, 멈춤 시간이 `MaxGCPauseMillis` 목표 안에 들도록 eden 크기가 조절됩니다. 대부분의 GC 활동이 여기서 끝납니다. **디스코드형 문제에서 G1이 Go보다 강한 이유가 바로 이것입니다.** 오래 사는 거대 캐시는 old region에 앉아 있고, young GC는 그걸 건드리지 않습니다. Go가 매번 전체를 훑던 것과 달리, G1은 평소엔 young만 훑고 지나갑니다.

**동시 마킹 사이클.** old가 계속 차오르면 언젠가 old도 회수해야 합니다. 그 준비로 "어느 old region에 쓰레기가 많은가"를 알아내는 것이 동시 마킹입니다. 방아쇠는 **IHOP**입니다. old 점유율이 전체 힙의 45%(기본, adaptive)를 넘으면, 다음 young GC에 얹어(Concurrent Start) 마킹을 시작합니다. 마킹 자체는 4.1에서 정의한 **SATB(스냅샷) 쓰기 배리어**로 뮤테이터와 동시에 진행되고, 시작과 끝(Remark)에만 짧은 STW가 붙습니다. 이 배리어는 G1이 평상시에도 켜 두는 세금인데, 4.4의 ZGC 로드 배리어와 비교할 지점입니다.

한 가지 정확히 짚습니다. Remark의 STW에서는 SATB 버퍼 마무리와 함께 **약한 참조 처리**(Weak·Soft·Final·Phantom reference)도 일어납니다. G1에서 이 참조 처리는 **여전히 STW**이며, 병렬화(`ParallelRefProcEnabled`, 실측 true)로 단축할 수 있을 뿐 뮤테이터와 겹치지는 않습니다. 뮤테이터와 나란히 도는 진짜 **동시(concurrent) 참조 처리**는 G1이 아니라 **Shenandoah(JDK 16)와 ZGC**의 특성입니다(참고로 JEP 428은 GC가 아니라 Structured Concurrency라 이 주제와 무관합니다). G1의 Remark를 더 줄이려는 이 흐름은 4.4에서 ZGC를 다룰 때 이어집니다.

**Mixed GC(STW evacuation).** 마킹이 끝나면 쓰레기가 많은 old region의 순위표가 생깁니다. 이제 young GC를 돌리되 거기에 **쓰레기 많은 old region 몇 개를 얹어(mixed)** 함께 evacuate합니다. "일부만" 얹는 게 핵심입니다. 한 번에 다 하면 멈춤이 길어지니, 목표 멈춤 시간 안에 들 만큼만 골라 여러 번(`G1MixedGCCountTarget=8`)에 나눠 처리합니다. 이때 너무 꽉 찬 region(85% 넘게 살아있는, `G1MixedGCLiveThresholdPercent`)은 복사 비용이 아까워 제외합니다.

**evacuation pause의 실체.** young이든 mixed든, 실제 멈춤은 "살아있는 객체를 새 region으로 복사하는" 구간입니다. 복사는 두 가지 일을 합니다. 객체 자체를 옮기고(메모리 복사), 그 객체를 가리키던 **모든 포인터를 새 주소로 교정**합니다(4.1에서 이동식 GC가 정밀 스캔을 필요로 한 이유). G1은 region마다 붙은 리멤버드 셋(4.1: 카드 테이블 + RSet)으로 "바깥에서 이 region을 가리키는 포인터가 어디 있는지"를 알기에, 힙 전체를 뒤지지 않고 그 포인터들만 고칩니다.

```
 Evacuation(이사) — 수집 대상 region의 '산 객체만' 새 region으로 복사:

  전(前):  ┌── old region A ──┐ ┌── old region B ──┐   ┌── free ──┐
           │ ●░░●░●░░░●░░░●░░ │ │ ░░●░░●░░░░●░░░░● │   │          │
           └──────────────────┘ └──────────────────┘   └──────────┘
             (● 산 객체, ░ 쓰레기)

  후(後):  ┌── A (빈 region) ─┐ ┌── B (빈 region) ─┐   ┌── new region ──┐
           │      (반납)       │ │      (반납)       │   │ ●●●●●●●●●●      │ ← 산 것만 밀착
           └──────────────────┘ └──────────────────┘   └────────────────┘
             A·B 통째로 free로 반납               단편화 0 (압축 완료)

   대가: (1) 산 객체 복사 + (2) 그것을 가리키던 포인터를 새 주소로 전부 교정 = 이 시간이 STW pause
```

**Humongous 객체 — 3.4 복선 회수.** 3.4에서 "region의 50%를 넘는 객체는 humongous로 특별 취급된다"고 예고했습니다. 회수합니다. **크기가 region의 절반을 넘는 객체를 humongous라 하고, eden을 거치지 않고 old에 humongous region으로 직접 할당됩니다.** 큰 객체를 eden에 넣었다 옮겼다 하는 비용이 아까워서입니다. 여기엔 두 가지 비용이 따라옵니다.

```
 Humongous (크기 > region의 50%):

  region=1MB일 때 700KB 배열 하나:
     ┌──── H region ────┐
     │ [ 700KB 객체 ]    │  ← region 절반 초과 → humongous 분류
     │ (나머지 300KB 낭비)│     region 꼬리를 다른 객체와 못 나눔 = 내부 단편화
     └──────────────────┘

  1.5MB 객체(region 1개 초과) → 연속된 region 2개를 통째로 점유:
     ┌── H ──┬── H ──┐
     │███████│███░░░░│  ← '연속된' 빈 region이 필요. 못 찾으면 할당 실패 → GC 유발
     └───────┴───────┘

  ★ humongous 할당은 old 점유율을 급격히 밀어올려, IHOP를 건드려 '동시 마킹'을 촉발할 수 있음
     (JDK 개선으로 humongous 할당이 직접 concurrent start를 트리거) — 여기가 3.4가 말한 그 지점
```

즉 큰 객체를 자주 만들면(대형 배열·버퍼·직렬화 페이로드) humongous region이 쌓이고, old 점유가 빨리 차 동시 마킹이 자주 돌며, 연속 region을 못 찾으면 할당 실패로 GC가 유발됩니다. 대형 객체 하나가 GC 리듬 전체를 흔드는 셈입니다.

다만 humongous가 **반드시 동시 마킹과 mixed GC를 거쳐야만 지워지는 것은 아닙니다.** JDK 8u40부터 G1은 **eager reclaim(조기 회수)**을 합니다(`G1EagerReclaimHumongousObjects`, 기본 on, 실측 확인). 매 young GC마다 humongous region을 가리키는 참조가 리멤버드 셋 기준으로 하나도 없으면 그 자리에서 통째로 반납합니다. 특히 char·int·long·double 같은 **primitive 배열**은 바깥을 가리키는 포인터가 없어(스캔이 단순) 조기 회수 대상이 되기 쉽습니다. 그래서 '잠깐 쓰고 버리는 큰 버퍼'는 old를 오래 물고 있지 않고 young GC 주기에 빠르게 회수됩니다. 정리하면 humongous의 진짜 비용은 '내부 단편화 + 연속 region 요구 + 동시 마킹 촉발'이지, '회수가 느리다'가 아닙니다.

## ④ 회피·최적화 전략 — Full GC를 어떻게 피하나 (그리고 "왜 가끔 크게 멈추나")

지금까지의 young·mixed·동시 마킹은 전부 **부분적이고 멈춤 시간이 예측·제한된** 활동입니다. 그래서 G1은 대부분 매끄럽습니다. 그런데 제목의 나머지 절반, "가끔 절벽처럼 크게 멈춘다"는 어디서 올까요. **Full GC**입니다.

**Full GC란, G1이 부분 회수로는 도저히 공간을 못 만들 때 마지막 수단으로 힙 전체를 STW로 압축하는 것입니다.** 정상적인 G1에는 원래 없어야 하는, 실패 시 fallback입니다. 방아쇠는 대부분 **evacuation 실패(to-space exhausted)**입니다. evacuation은 산 객체를 "새 region으로 복사"하는 것인데, **복사해 넣을 빈 region이 하나도 안 남으면** 복사를 할 수 없습니다. 힙은 이미 최대 크기라 더 늘릴 수도 없습니다. 이때 G1은 부분 회수를 포기하고 힙 전체를 대상으로 STW 압축을 벌입니다.

```
 "왜 대체로 안정적이나 가끔 크게 멈추나":

  지연 ▲
  200ms┤· · · · · · · · · · · · · · · · · · · · · · ·  ← MaxGCPauseMillis 목표선
       │▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁          █
       │ young/mixed: 짧고 예측된 멈춤             █  ← Full GC (to-space exhausted)
       │ (부분 회수, pause 목표 안에서)            █     힙 전체 압축, pause 목표 무시
   수초┤                                          █     = 수백 ms ~ 수 초 STW
       └───────────────────────────────────────────────▶ 시간
    대부분 매끄럽다가, evacuation 실패의 순간 절벽처럼 큰 멈춤 하나
```

Full GC는 JDK 10부터 병렬(멀티스레드)로 개선됐지만(JEP 307), 그래도 **힙 전체가 대상이고 멈춤 시간 목표를 지키지 않으므로** 힙이 크면 수 초까지 갑니다. 저지연 서비스에서는 이 한 번의 절벽이 SLO를 통째로 깨뜨립니다. 왜 evacuation이 실패할까요. 대개 **회수가 할당을 못 따라간** 결과입니다. 할당이 폭주하거나, 동시 마킹이 너무 늦게 시작돼(IHOP가 너무 높게 잡혀) old를 제때 못 비웠거나, humongous 단편화로 연속 region이 부족하거나, 살아있는 객체가 너무 많아(live-set 과다) 복사할 곳이 부족한 경우입니다.

그래서 회피 전략은 전부 "Full GC 전에 부분 회수가 이기게 만들기"로 수렴합니다.

- **힙과 여유를 늘린다.** `-Xmx`를 키우거나 `G1ReservePercent`(기본 10%, evacuation 대비 예비)를 올려 복사할 빈 공간을 확보합니다.
- **마킹을 더 일찍 시작한다.** adaptive IHOP를 신뢰하되, 급증 워크로드에서는 `-XX:-G1UseAdaptiveIHOP`로 끄고 `InitiatingHeapOccupancyPercent`를 낮게 고정해 old를 미리미리 비웁니다.
- **humongous를 줄인다.** 대형 배열·버퍼를 잘게 나누거나, `-XX:G1HeapRegionSize`를 키워 같은 객체가 humongous로 분류되지 않게 합니다(region이 커지면 50% 임계도 커짐).
- **탐지.** GC 로그(`-Xlog:gc*`)에서 `to-space exhausted`, `Pause Full (Allocation Failure)`가 보이면 이미 절벽을 밟은 것입니다. 이 문자열을 알람으로 거는 게 실무의 기본입니다.

한 가지 역사적 각주를 답니다. G1은 이 절벽을 런타임 스스로 피하려는 **Preventive GC(예방적 GC)**를 JDK 17에 실었습니다(마이크로소프트 기여, JDK-8257774). evacuation 실패가 임박했다고 판단되면 IHOP 도달 전이라도 선제적으로 young 수집을 당겨 Full GC를 피하려는 시도였습니다. 그런데 이 예측이 자주 빗나가 오히려 손해인 경우가 많았고, 그사이 evacuation 실패 자체를 값싸게 처리하도록 런타임이 개선되면서 존재 이유가 옅어졌습니다. 결국 JDK 20에서 기본 비활성화된 뒤 **완전히 제거**됐습니다(JDK-8297639). 그래서 **JDK 25에는 Preventive GC가 없습니다.** 이 짧은 흥망 자체가 4부의 주제를 다시 확인해 줍니다. 런타임이 절벽을 자동으로 없애 주는 '공짜 방어막'은 결국 남지 않았고, 대신 '실패를 값싸게 처리한다'는 정공법이 자리를 지켰습니다.

## ⑤ 3개 언어 수렴 — 같은 "회수"를 세 방식으로

**Rust**는 회수를 컴파일 타임 `drop`으로 끝내니 마킹도, evacuation도, Full GC도 없습니다. 거대 캐시를 이고 있어도 축출 시 즉시 해제됩니다(디스코드가 이걸로 이겼습니다). 대가는 개발자의 수명 관리, 그리고 `Arc` 순환 누수입니다.

**Go**는 비세대·비이동 마크-스윕입니다. 이동이 없으니 evacuation pause도 Full GC 절벽도 없지만, 비세대라 매 사이클 살아있는 힙 전체를 훑고 Mark Assist로 뮤테이터를 징집해 꼬리 지연이 꾸준히 셉니다. "작지만 상시적인 세금" 쪽입니다.

**Java G1**은 세대별·이동식입니다. 세대별이라 평소엔 young만 훑어 디스코드형(거대 old 캐시) 문제에 Go보다 강하고, 이동식이라 단편화가 없습니다. 대신 evacuation이라는 복사 비용을 내고, 그것이 실패하면 Full GC라는 드물지만 큰 절벽을 냅니다. "평소 조용하다가 이따금 큰 청구서" 쪽입니다.

| | 세대? | 이동? | 평소 비용 | 최악의 순간 | 거대 old 캐시 |
|---|---|---|---|---|---|
| **Rust** | — | — | 없음(drop) | 없음 | 강함(즉시 해제) |
| **Go** | 비세대 | 비이동 | 상시 Mark Assist 꼬리 지연 | 전체 스캔 지속 | **약함**(매번 전체) |
| **Java G1** | 세대별 | 이동 | evacuation pause(짧음) + SATB 배리어 | **Full GC 수 초 STW** | 강함(young만 자주) |
| **Java ZGC**(4.4) | 세대별 | 이동(동시) | 로드 배리어 처리량 세금 | allocation stall | 강함 |

## ⑥ 트레이드오프 소결 — 삼각형 위의 G1

```
                     회계의 위치
                  (전부 런타임)
                         ▲
                        ╱ ╲
                       ╱   ╲
       단편화 ◀───────╱─────╲───────▶ 이동 비용
    (Go가 여기)                (G1이 여기)
                                · evacuation으로 압축 → 단편화 0
                                · 대가: 복사 + 포인터 교정 pause
                                · 극단: to-space 고갈 시 Full GC 절벽

   ── 4.3이 드러낸 G1의 성격 ──
     · 세대별 → 평소엔 young만. 디스코드형 문제에 Go보다 구조적으로 강함
     · 이동식 → 단편화 없음. 대가는 evacuation pause와 Full GC 위험
     · 멈춤을 '예측·분할'로 길들였지만(대부분 매끄러움), 부분 회수가 할당에
       추월당하는 순간 그 길들임이 풀리며 Full GC로 무너짐
```

G1은 4.1 삼각형의 오른쪽(이동 비용) 꼭짓점을 택했습니다. 그 선택으로 Go가 안고 살던 두 약점, 즉 단편화와 "거대 old를 매번 전체 스캔"을 해결했습니다. 세대별이라 평소엔 young만 훑고, 이동식이라 압축이 공짜로 따라옵니다. 그 대신 새 청구서 두 장을 받습니다. 하나는 evacuation이라는 복사 pause(짧지만 상시), 다른 하나는 to-space 고갈 시의 Full GC라는 드문 절벽입니다. Go의 지연이 "잔물결처럼 늘 조금씩 튀는" 모양이라면, G1의 지연은 "대체로 잔잔하다가 이따금 큰 파도가 한 번 치는" 모양입니다.

그래서 "GC는 은탄환"이라는 명제는 여기서도 검증적으로 부정됩니다. G1은 Go의 상시 꼬리 지연 세금을 없앤 대신 **드물지만 치명적인 Full GC 위험으로 바꿔치기**했을 뿐입니다. 위험의 총량이 사라진 게 아니라 분포가 바뀌었습니다. 다음 절의 ZGC는 바로 이 "Full GC 절벽"마저 없애려는 시도이고, 그 대가로 또 다른 청구서(배리어 처리량 세금·allocation stall)를 받습니다.

## ⑦ DevOps 관점 심화 질문

1. **GC 로그에서 Full GC의 전조를 어떻게 조기 경보로 만들 것인가?** `-Xlog:gc*`에서 `to-space exhausted`·`Pause Full (Allocation Failure)`·evacuation failure 빈도를 어떤 지표로 뽑아, "절벽을 밟기 전"에 알람을 울릴 것인가. mixed GC 후에도 old 점유가 안 내려가는 추세를 어떻게 잡는가.

2. **컨테이너에서 `-Xmx`와 cgroup 리밋, compressed oops 32GB 절벽(3.4)을 어떻게 함께 잡는가?** `MaxRAMPercentage`로 힙을 컨테이너 리밋에 맞출 때, 힙을 32GB 밑으로 눌러 compressed oops를 유지하는 것과 힙을 키워 Full GC 여유를 주는 것 사이의 균형을 어떻게 계산하는가. 힙 밖(Metaspace·코드캐시·다이렉트)까지 합산하는 cgroup 회계는 어떻게 반영하는가.

3. **`MaxGCPauseMillis`를 낮추면 정말 지연이 좋아지는가?** 목표를 200에서 50으로 낮추면 eden이 작아져 young GC가 잦아지고 처리량이 떨어지며, 역설적으로 old가 빨리 차 Full GC 위험이 커질 수 있습니다. 이 트레이드오프를 부하 테스트로 어떻게 계측·판단하는가.

4. **humongous 할당을 어떻게 탐지하고 없애는가?** GC 로그의 humongous allocation 이벤트와 `-Xlog:gc+heap`으로 humongous region 수를 추적해, 어떤 코드 경로가 대형 객체를 만드는지 JFR allocation profiling으로 짚고, region 크기 조정과 객체 분할 중 무엇을 택할지 어떻게 결정하는가.

5. **adaptive IHOP를 믿을 것인가, 고정할 것인가?** 할당률이 급변하는(트래픽 스파이크) 서비스에서 adaptive IHOP가 마킹을 늦게 시작해 Full GC를 부르는 경우가 있습니다. `-XX:-G1UseAdaptiveIHOP`로 끄고 낮은 값으로 고정하는 결정을, 어떤 관측치(old 증가 속도 대비 마킹 소요 시간)로 내리는가.

6. **`ParallelGCThreads`·`ConcGCThreads`가 cgroup CPU 쿼터와 어긋나면?** GC STW 병렬 워커가 컨테이너에 배정된 코어보다 많게 잡히면 STW가 오히려 길어질 수 있습니다. 컨테이너 CPU 리밋에 맞춰 이 스레드 수를 어떻게 세팅하고, `cpu.stat`의 스로틀링과 GC pause 로그를 어떻게 교차 검증하는가.

7. **`to-space exhausted`가 없는데도 Full GC가 난다면?** 힙 여유가 충분해도, 리플렉션·동적 프록시로 클래스 로딩이 잦은 애플리케이션은 **Metaspace**가 꽉 차 Full GC(Metadata GC Threshold)를 일으킵니다(3.4의 힙 밖 지형). `-Xlog:gc+metaspace`로 Metaspace 확장에 의한 Full GC를 어떻게 식별하고, `MaxMetaspaceSize`와 컨테이너 메모리 쿼터(cgroup은 힙 + Metaspace + 코드캐시를 합산)의 균형을 어떻게 잡는가.

## ⑧ 다음 절 연결

G1은 세대별·이동식으로 Go의 두 약점을 풀었지만, 그 대가로 evacuation pause와 Full GC 절벽을 얻었습니다. 특히 Full GC는 힙이 커질수록 위험해집니다. 힙이 수십~수백 GB인 시대에 "가끔 수 초 멈춤"은 감당하기 어렵습니다. 그래서 마지막 질문이 남습니다. **이사(evacuation)마저 뮤테이터를 멈추지 않고 동시에 할 수 있다면, Full GC 절벽 자체를 없앨 수 있지 않을까?**

**4.4 Generational ZGC**로 갑니다. ZGC는 4.1에서 예고한 **colored pointer(주소 비트에 상태를 새김)와 load 배리어**로, 객체를 옮기는 동안에도 뮤테이터를 세우지 않습니다. 그래서 힙이 수백 GB여도 STW가 sub-ms입니다. G1의 Full GC 절벽이 사라지는 것입니다. 하지만 4.1에서 미리 계산했듯, 그 대가는 **모든 포인터 로드에 붙는 배리어 세금(쓰기 배리어보다 약 5배)**과, 회수가 할당을 못 따라갈 때 뮤테이터가 잠깐 막히는 **allocation stall**입니다. 3.4에서 예고한 그 청구서를 다음 절에서 정확히 해부합니다. G1이 "멈춤을 짧게 나눴다"면 ZGC는 "멈춤을 없애고 처리량으로 냈다"는, 4.1 명제의 가장 극단적인 실증입니다.
