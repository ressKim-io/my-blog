> **이 문서의 성격** — 최종 블로그 원고가 아니라, 최종 집필 에이전트에게 넘길 1차 심화
> 정리(reference-level draft)입니다. `개념 → 하드웨어 동작 → 비용 해부 → 회피 전략 →
> 3개 언어 연결 → 트레이드오프 → 심화 질문`의 순서로 아래에서 위로 쌓아 올립니다. 그림은
> 모두 ASCII로 그렸고, 본문의 수치는 JDK 25(2026-07)·HotSpot 기준이며 버전과 설정에 따라
> 달라질 수 있음을 전제로 읽어 주세요.

# 3.4 JVM의 TLAB: 포인터만 밀면 되는 할당, 그 대가는 GC와 객체 헤더

## ① 이 절에서 도달할 지점

3.2의 Rust는 malloc의 bin에서 맞는 조각을 찾아 왔고, 3.3의 Go는 span의 빈 슬롯을 비트맵으로
짚었습니다. JVM은 여기서 한 걸음 더 나아가, 할당을 **"포인터를 한 번 미는 것"**으로 줄여
버립니다. 이 절은 그게 어떻게 가능하고 무엇을 대가로 치르는지를 봅니다. 다 읽고 나면 다음을
스스로 설명할 수 있게 됩니다.

- **TLAB(Thread-Local Allocation Buffer)**가 무엇이고, 왜 자바의 객체 생성이 사실상 명령어
  몇 개로 끝나는지 — **bump-the-pointer** 할당의 실체.
- 그 극단적으로 싼 할당이 왜 **moving GC(객체를 옮기는 세대별 GC)의 선물**인지. malloc과 Go는
  왜 이걸 못 하는지.
- TLAB가 가득 찼을 때의 **느린 경로**와, TLAB 크기가 스스로 조절되는 방식
  (`TLABWasteTargetPercent`).
- 객체마다 붙는 **헤더(12~16바이트)**라는 세금, 그리고 JDK 25의 **compact object headers**가
  그걸 어떻게 줄이는지.
- **힙 밖의 지형** — Metaspace·코드캐시·다이렉트 메모리·스레드 스택(2.3에서 만난 그 층들)이
  `-Xmx` 바깥에 살고, 그래서 왜 **`-Xmx`로는 OOMKilled를 못 막는지**(2.4·5부의 물리 토대).
- **스칼라 치환(2.3)**이 어떻게 할당 자체를 아예 없애는지, 그리고 그게 Go·Rust의 이스케이프
  처리와 어떻게 같고 다른지.

---

## ② 개념과 물리적 실체

### 세대별 힙: 자바 객체가 태어나는 땅

자바 객체는 **GC 힙** 안에서 태어납니다. 이 힙은 "대부분의 객체는 금방 죽는다"는 경험칙
(세대 가설, generational hypothesis)에 맞춰 **세대별**로 나뉩니다.

- **Young(젊은 세대)** — 갓 만든 객체가 사는 곳. 다시 **Eden**과 두 개의 **Survivor(S0/S1)**로
  나뉩니다. 새 객체는 거의 다 Eden에서 태어납니다.
- **Old(늙은 세대)** — Young에서 여러 번 살아남아 **승격(promotion)**된 장수 객체가 사는 곳.

동작의 핵심은 이겁니다. Eden이 꽉 차면 **Minor GC**가 돌아, Eden에서 아직 살아 있는 객체만
Survivor(또는 Old)로 **복사**하고, **Eden을 통째로 비웁니다.** 죽은 객체는 하나하나 회수하는
게 아니라, 산 것만 옮기고 나머지는 그냥 버리는 것입니다. 이 "통째로 비운다"가 뒤에 나올
bump-the-pointer를 가능하게 하는 열쇠입니다.

(JDK 25 기본 GC인 **G1**은 힙을 균일한 **region**들로 잘라 Eden/Survivor/Old/Humongous를
동적으로 배정하지만, "새 객체는 Eden 역할 region에, 할당은 포인터를 밀어서"라는 큰 그림은
같습니다.)

### TLAB란 무엇인가

Eden은 모든 스레드가 공유하는 땅입니다. 수백 개 스레드가 동시에 "여기서 객체 하나 만들게"
하며 Eden의 같은 위치를 다투면, 할당마다 락(또는 CAS)이 필요해 병목이 됩니다. 그래서 HotSpot은
**각 스레드에게 Eden의 한 조각을 미리 떼어 줍니다.** 이 스레드 전용 조각이 **TLAB(Thread-Local
Allocation Buffer)**입니다.

```text
 JVM 힙
 ┌────────────────── Young ──────────────────┬──────── Old ────────┐
 │  Eden                          S0    S1    │  (승격된 장수 객체)   │
 │  ┌──────┬──────┬──────┬── 공용 ──┐         │                     │
 │  │TLAB  │TLAB  │TLAB  │ 영역     │         │                     │
 │  │ T1   │ T2   │ T3   │(큰 객체) │         │                     │
 │  └──────┴──────┴──────┴──────────┘         │                     │
 └─────────────────────────────────────────────┴─────────────────────┘
   각 스레드는 자기 TLAB 안에서만 할당 → 다른 스레드와 안 부딪힘 → 락 없음
   Minor GC: Eden의 산 객체만 복사 → Eden 통째로 비움 → 할당 포인터 리셋
```

TLAB가 자기 스레드만의 것이라, 그 안에서의 할당은 **누구와도 동기화할 필요가 없습니다.**
바로 이게 자바 할당이 빠른 첫 번째 이유입니다.

---

## ③ 비용과 메커니즘 해부

### bump-the-pointer: 할당이 명령어 몇 개로 끝난다

TLAB는 그냥 연속된 빈 공간이고, "다음에 쓸 위치"를 가리키는 포인터(`top`)와 끝(`end`)만
있으면 됩니다. 객체 하나를 만드는 일은 **`top`을 객체 크기만큼 밀고, 옛 `top`을 돌려주는**
것이 전부입니다.

```text
 TLAB (스레드 전용 연속 구간)
   start          top(다음 빈 자리)                 end
     │               │                                │
     ▼               ▼                                ▼
     ┌───────────────┬────────────────────────────────┐
     │  이미 쓴 객체  │           아직 빈 공간           │
     └───────────────┴────────────────────────────────┘

 new T (크기 s) 의 빠른 경로 =
     if (top + s <= end) {         // 이 TLAB에 자리가 있나?
         p    = top;               // 돌려줄 주소
         top += s;                 // 포인터를 민다
         return p;                 //  ← 락 없음, CAS 없음, 명령어 몇 개
     } else {
         … TLAB가 꽉 참 → 느린 경로(아래)
     }
```

3.2의 malloc이 bin 연결 리스트를 뒤지고, 3.3의 Go가 span 비트맵을 짚던 것과 비교하면,
이건 거의 **덧셈 한 번**입니다. 자바가 "객체를 헤프게 만들어도 빠르다"는 평판을 얻은 바탕이
이 bump-the-pointer입니다.

### 왜 이게 가능한가 — moving GC의 선물

여기서 꼭 짚어야 할 질문이 있습니다. malloc과 Go는 왜 이렇게 못 할까요? 답은 **"객체를
옮길 수 있느냐"**에 있습니다.

포인터를 밀기만 하는 할당은, 공간을 절대 되돌려 쓰지 않고 앞으로만 나아갑니다. 그러면 금방
공간이 바닥납니다. 이게 지속되려면 누군가 주기적으로 **그 땅을 통째로 비워 포인터를 처음으로
되돌려** 줘야 합니다. 자바에서는 Minor GC가 그 일을 합니다 — Eden의 산 객체를 다른 곳으로
**복사(evacuate)**하고 Eden을 리셋합니다. 객체가 물리적으로 이사를 다니므로, 그 객체를
가리키던 모든 포인터도 GC가 고쳐 줍니다.

```text
 Minor GC 전후 — moving GC는 "산 것만 옮기고 Eden을 통째로 비운다"

  [GC 전]  Eden : [살][죽][죽][살][죽][살]   (꽉 참)
           S0   : [살][살]
                    │  산 객체만 다른 곳으로 복사(evacuate)
                    ▼
  [GC 후]  Eden : [                    ]   ← 통째로 리셋, bump 포인터가 맨 앞으로
           S1   : [살][살][살][살]          ← Eden·S0의 생존자가 이사옴
           Old  : [ … 여러 번 살아남은 것은 여기로 승격 … ]

  죽은 객체는 하나씩 회수하지 않는다 — 그냥 버리고 Eden을 리셋할 뿐. 그래서 bump이 가능하다.
```

(GC가 이 복사·승격을 할 때 쓰는 스레드 로컬 버퍼가 **PLAB(Promotion LAB)**로, 말하자면
TLAB의 'GC측 짝'입니다. 할당 쪽에 TLAB가 있듯 회수·승격 쪽에 PLAB가 있는 셈이죠.)

반대로 malloc과 (대체로) Go는 **객체를 옮기지 않습니다(non-moving).** 한번 준 주소는 그
자리에 고정입니다. 그래서 중간에 구멍이 나면 그 구멍을 재활용해야 하고, 그러려면 free list나
비트맵으로 "빈 자리"를 관리할 수밖에 없습니다. 즉 **bump-the-pointer는 공짜가 아니라, "GC가
나중에 객체를 옮겨 주겠다"는 약속을 담보로 미리 당겨 쓴 편의**입니다. 이 빚은 4부(GC)에서
갚습니다.

한 가지 최신 흐름을 미리 짚어 두겠습니다. 위에서 "GC가 Eden을 멈추고(STW) 통째로 비운다"고
한 것은 G1·Parallel 같은 **고전적 모델**입니다. JDK 21의 **Generational ZGC**(JDK 23부터
ZGC의 기본 모드)는 같은 bump-the-pointer를 쓰면서도, 객체 이사(evacuation)를 **애플리케이션을
멈추지 않고 동시에(concurrent)** 해냅니다 — 포인터에 상태 비트를 새긴 **colored pointer**와,
객체를 읽을 때 참조를 고쳐 주는 **load/store barrier**로 이동을 백그라운드에서 처리하죠. 즉
'할당의 싸구려를 STW로 갚던' 시대에서 **'그 빚마저 백그라운드 스레드의 CPU로 갚는'** 시대로
진화한 셈입니다. 이 이야기의 본론(sub-ms 수준의 STW·allocation stall·처리량 세금)은 4부에서
다룹니다.

### TLAB가 가득 차면: 느린 경로와 적응형 크기

`top + s > end`가 되는 순간, 빠른 경로가 막힙니다. 이때 JVM의 판단은 크기에 따라 갈립니다.

```text
 객체(크기 s)를 할당할 때 JVM의 판단 순서

   s가 지금 TLAB에 들어가나? ────────── 예 ──▶ bump-the-pointer (빠른 경로, 락 없음)
        │ 아니오
        ▼
   s가 TLAB를 통째로 낭비할 만큼 작나? ── 예 ──▶ 옛 TLAB 은퇴 + 새 TLAB에서 할당
        │ 아니오 (큼)
        ▼
   s가 G1 region의 50% 이상인가? ─────── 아니오 ──▶ Eden 공용 영역에 직접 (CAS)
        │ 예
        ▼
   Humongous → Old의 Humongous Region에 직접 (+ 동시 마킹 트리거 검사)
```

- **객체가 크면** — TLAB 하나를 통째로 낭비하기 아까우니, **Eden의 공용 영역에 직접** 할당
  합니다(여러 스레드가 공유하므로 여기서만 CAS/락). 
- **객체가 작으면** — 현재 TLAB를 **은퇴(retire)**시키고 새 TLAB를 Eden에서 받아 그 안에
  넣습니다. 이때 옛 TLAB의 남은 자투리는 버려지는데, GC가 힙을 처음부터 끝까지 선형으로 훑을
  수 있어야 하므로(heap parsability) 그 자투리 자리에 **더미 객체(filler)**를 채워 둡니다.
- **객체가 아주 크면 — Humongous Object(G1 기준).** G1 region 크기의 **50% 이상**을 차지하는
  객체는 Eden을 아예 거치지 않고 **Old의 Humongous Region에 직접** 할당됩니다. region은 힙
  크기에 따라 1~32MB로 정해집니다(이 샌드박스 JDK 11 기본은 1MB). 운영에서 이게 특히 중요한
  이유가 있습니다: humongous 객체는 evacuation(이사) 대상에서 빠져 **옮겨지지 않고**, 할당
  직전 G1이 IHOP(Initiating Heap Occupancy Percent, 기본 힙의 45%) 초과 여부를 검사해 **동시
  마킹(Concurrent Marking) 사이클을 앞당겨 트리거**합니다. 그래서 큰 배열·버퍼를 자주 만들면
  TLAB 우회를 넘어 **GC 사이클 자체를 흔들어** G1 성능을 떨어뜨립니다.

여기서 낭비(자투리)와 경합(새 TLAB를 받는 빈도)이 맞물립니다. HotSpot은 이를 두 손잡이로
조율합니다.

- **`-XX:TLABWasteTargetPercent`(기본 1)** — TLAB 은퇴로 버려도 좋은 낭비의 목표치, TLAB
  크기의 약 1%입니다. "이보다 크게 낭비될 것 같으면 은퇴 대신 공용 영역에 직접 할당하라"는
  기준이 됩니다.
- **`-XX:+ResizeTLAB`(기본 켜짐)** — 각 스레드가 GC 주기 사이의 refill(새 TLAB 받은) 횟수를
  세어 TLAB 크기를 스스로 키우거나 줄입니다. 목표는 GC 한 주기당 약 **50회 refill**(=
  `50 ÷ TLABWasteTargetPercent`)입니다. 할당을 많이 하는 스레드는 큰 TLAB를, 적게 하는
  스레드는 작은 TLAB를 갖게 되어, 낭비와 경합이 동시에 눌립니다. (기본 최소 크기
  `MinTLABSize`는 2KB로 확인됩니다.)

정리하면 비용의 계단은 이렇습니다: **거의 항상(TLAB bump, 덧셈) ≫ 가끔(새 TLAB, Eden CAS)
≫ 드물게(Minor GC).** 3.3의 Go가 mcache→mcentral→mheap로 내려가던 그 구조와 정확히 같은
모양입니다 — 이름만 TLAB→Eden공용→GC로 바뀌었을 뿐입니다.

### 객체 헤더: 자바가 객체마다 무는 세금

여기서 자바의 대가 하나가 드러납니다. **모든 자바 객체에는 헤더가 붙습니다.** 할당은 싸도,
객체 하나하나가 자기 몸값 위에 관리 비용을 얹고 다닙니다.

```text
 객체 하나의 실제 크기 (compressed oops, 표준 헤더)
   ┌──────────── mark word 8B ───────────┬─ klass 4B ─┬─ 필드… ─┐
   │ 식별 해시 · GC age · 락 상태          │ (압축 클래스│         │
   │                                      │  포인터)   │         │
   └──────────────────────────────────────┴────────────┴─────────┘
   헤더 12B → 8바이트 정렬 패딩 → 필드 하나 없는 객체도 최소 16B

   JDK 25 compact object headers (JEP 519, 아직 opt-in):
     mark word가 클래스 참조까지 흡수 → 헤더 8B로 축소
```

- **mark word(8바이트)** — 객체의 신분증입니다. `hashCode`, GC가 세는 나이(age), 락 상태
  (동기화) 같은 정보가 여기 삽니다.
- **klass 포인터(압축 시 4바이트)** — 이 객체가 어느 클래스인지 가리킵니다.
- 그래서 표준 헤더는 12바이트이고, 8바이트 정렬 패딩까지 하면 **필드가 하나도 없는 객체도
  16바이트**를 차지합니다. `int` 하나짜리 객체 역시 12B 헤더 + 4B = 16B입니다.

그런데 이 '압축(compressed)'에는 벼랑이 하나 숨어 있습니다. 압축 oops는 참조를 8바이트 정렬
기준의 32비트 오프셋으로 저장해 **최대 32GB 힙**까지 가리킬 수 있습니다. 힙을 **32GB 너머로
키우면 압축이 꺼지고**, 모든 객체 참조와 klass 포인터가 **4바이트 → 8바이트**로 부풀어 힙이
갑자기 무거워지고 GC가 잦아집니다. 그래서 **힙 32~48GB 구간은 오히려 손해**인 '죽음의 구간'
으로 악명 높습니다 — 30GB로 줄이는 편이 40GB보다 빠른 일이 흔하죠. (`-XX:ObjectAlignmentInBytes=16`
으로 정렬을 키우면 64GB까지 압축을 유지할 수 있습니다.)

이 헤더가 왜 중요한가 하면, 3.2의 Rust 구조체나 3.3의 Go 객체는 **이런 객체별 헤더가 없기**
때문입니다(Rust 구조체는 필드 그 자체이고, Go의 GC 메타데이터는 객체가 아니라 span 쪽에
있습니다). 작은 객체 수억 개를 다루면 이 12~16바이트 세금이 힙 크기와 캐시 효율(2.3에서 본
데이터 지역성)을 크게 좌우합니다. 그래서 JDK 25가 **compact object headers(JEP 519)**로 헤더를
8바이트로 줄인 것이 의미가 큽니다 — 벤치마크(SPECjbb2015)에서 힙 22% 감소, CPU 8% 감소가
보고됐습니다. (JDK 25에서는 정식 기능이 됐지만 아직 기본이 아니라 `-XX:+UseCompactObjectHeaders`로
켜야 합니다.)

그리고 그다음 단계가 이미 오고 있습니다. **Project Valhalla(JEP 401, Value Classes, 프리뷰 ·
JDK 28 목표)**는 헤더를 8바이트로 줄이는 걸 넘어, 아이덴티티가 없는 **값 클래스**에서 아예
**헤더를 없애고** 값을 배열·필드에 직접 박아 넣습니다(flattening). 포인터 간접참조도, 원소마다
붙던 헤더도 사라지죠. 이것은 Go·Rust가 처음부터 구조체를 스택 값으로 다루던 방식에 자바가
수렴하려는 시도이자, **자바가 객체 헤더라는 세금을 근본적으로 면제받으려는** 궤도입니다.

### 힙 밖의 지형 — 그리고 OOMKilled의 물리

지금까지는 전부 GC 힙 이야기였습니다. 그런데 자바 프로세스가 쓰는 메모리는 힙이 전부가
아닙니다. `-Xmx`가 감싸는 것은 **GC 힙뿐**이고, 그 바깥에 여러 층이 따로 삽니다.

```text
 -Xmx 가 제한하는 것 = GC 힙(Young+Old) 뿐
 컨테이너(cgroup)가 재는 것 = 아래 전부의 RSS 합

 ┌──────────── 프로세스 RSS (cgroup memory.max 가 재는 값) ────────────┐
 │  ┌─ GC 힙 (-Xmx 는 여기까지만) ─┐                                    │
 │  │  Young(Eden/Survivor) + Old   │                                    │
 │  └───────────────────────────────┘                                   │
 │  Metaspace     클래스 메타데이터 · 메서드 바이트코드 · MDO (native)   │
 │  Code Cache    JIT가 만든 기계어 (기본 240MB, 2.3의 그 층)            │
 │  Direct Memory DirectByteBuffer · Netty off-heap (기본 ≈ -Xmx)       │
 │  Thread Stacks 스레드마다 ~512KB~1MB (-Xss)                          │
 └──────────────────────────────────────────────────────────────────────┘
   → -Xmx=2G 인데 컨테이너 limit=2G 라면? 힙 밖 층이 넘쳐 OOMKilled
```

- **Metaspace** — 클래스의 메타데이터(Klass 구조체, 메서드 바이트코드, 상수 풀), 그리고
  2.3에서 만난 **MDO(MethodData)**가 사는 곳입니다. GC 힙이 아니라 **네이티브 메모리**이며,
  기본적으로 상한이 사실상 무제한이라(`MaxMetaspaceSize` 기본값이 거대) 클래스로더 누수가
  나면 여기서 네이티브 메모리가 샙니다. (초기 GC 임계값 `MetaspaceSize`는 약 21MB로 확인됩니다.)
- **Code Cache** — JIT(C1/C2)이 컴파일한 기계어가 사는 곳으로 기본 **240MB**입니다(2.3에서
  본 세그먼트 코드캐시). 꽉 차면 JIT이 멈추고 인터프리터로 떨어지는 성능 절벽이 생깁니다.
- **Direct Memory** — `DirectByteBuffer`나 Netty가 쓰는 off-heap 버퍼로, 내부적으로는 결국
  네이티브 `malloc`(3.2)을 탑니다. 기본 상한 `MaxDirectMemorySize`는 미설정(0)이면 `-Xmx`를
  따라갑니다.
- **Thread Stacks** — 스레드마다 하나씩, 보통 512KB~1MB(`-Xss`)입니다.

여기서 2.4와 5부의 복선이 물리적으로 닫힙니다. 컨테이너의 cgroup은 이 **모든 층의 RSS 합**을
셉니다(3.1). 그런데 `-Xmx`는 그중 GC 힙만 제한합니다. 그래서 `-Xmx=2G`로 힙을 묶어도,
Metaspace·코드캐시·다이렉트 메모리가 부풀면 프로세스 RSS가 컨테이너 한도를 넘어 **OOMKilled**
됩니다. "힙은 멀쩡한데 컨테이너가 죽는다"는 자바의 대표적 미스터리가, 바로 이 힙 밖 지형
때문입니다.

그래서 컨테이너에서는 `-Xmx`를 손으로 고정하기보다 컨테이너 메모리를 인식하는
**`-XX:MaxRAMPercentage`**로 힙 상한을 자동 계산하게 하되, **힙 밖 몫(Metaspace·코드캐시·
다이렉트·스택)을 반드시 남겨** 둬야 합니다. 이는 3.3에서 본 Go의 `GOMEMLIMIT`을 컨테이너
한도의 90~95%만 힙에 주라던 것과 정확히 같은 고민입니다 — 언어는 달라도, 힙 밖을 남겨
두지 않으면 OOMKilled 된다는 물리는 똑같습니다.

## ④ 회피·최적화 전략

- **스칼라 치환(scalar replacement) — 할당을 아예 없애기.** C2 JIT이 이스케이프 분석으로
  "이 객체는 이 메서드 밖으로 새지 않는다"를 증명하면, 객체를 **만들지 않고** 그 필드들을
  레지스터·스택 값으로 흩어 버립니다. 할당 횟수가 0이 됩니다. 2.3에서 본 그 최적화입니다.
  다만 이건 **JIT이 그 메서드를 뜨겁게 컴파일하고, 인라이닝이 성공하고, 객체가 정말 안 새는**
  경우에만 일어납니다 — 즉 실행 시점의 조건부입니다. 게다가 stock C2의 이스케이프 분석은
  **all-or-nothing**이라, 객체가 **어느 한 분기에서라도** 새면 치환을 통째로 포기합니다. 예컨대
  `if/else`가 서로 다른 객체를 반환해 참조가 합류하거나(control-flow merge), 객체가 인라이닝되지
  않은 메서드로 넘어가면 그대로 힙에 할당됩니다. (GraalVM은 이보다 영리한 **부분 이스케이프
  분석(partial escape analysis)**이 있어, 안 새는 hot path에서만 치환하기도 합니다.) '치환될
  것 같던' 객체가 실제 프로파일에선 힙에 잡히는 괴리가 여기서 생깁니다. (이 조건부성이 Go·Rust
  와의 차이인데, ⑤에서 정리합니다.)
- **이스케이프를 줄이는 코드.** 스칼라 치환이 잘 걸리도록, 짧게 살고 밖으로 새지 않는 객체는
  메서드 안에 가두고 인라이닝을 방해하지 않게 짭니다. `-XX:+PrintEscapeAnalysis`(fastdebug
  빌드)나 JITWatch로 확인합니다.
- **compact object headers로 헤더 세금 줄이기.** 작은 객체가 많은 워크로드라면 JDK 25에서
  `-XX:+UseCompactObjectHeaders`로 헤더를 8바이트로 줄여 힙과 캐시 효율을 개선합니다.
- **TLAB 관측·튜닝.** `-XX:+PrintTLAB`나 JFR(TLAB allocation 이벤트)로 refill·낭비를 보고,
  필요하면 `TLABSize`·`TLABWasteTargetPercent`를 만집니다. 다만 대개는 적응형(ResizeTLAB)이
  알아서 잘합니다.
- **NUMA-aware 할당(`-XX:+UseNUMA`, 기본 꺼짐).** TLAB는 락만 없애는 게 아닙니다. NUMA
  (Non-Uniform Memory Access) 장비에서 이 옵션을 켜면, JVM(Parallel·G1·ZGC)이 스레드가 도는
  CPU와 **물리적으로 가까운 메모리 노드**에 TLAB/young을 배치합니다. 3.1에서 본 "접근 지연은
  결국 물리적 거리"라는 사실을 할당 단계에서 미리 줄이는 것으로, 멀티소켓 서버나 대형
  인스턴스에서 효과가 큽니다(G1의 NUMA 인식은 JDK 14부터). 기본이 꺼져 있으니 NUMA 장비라면
  켜는 걸 검토할 값입니다.
- **힙 밖을 반드시 함께 계측.** `-XX:NativeMemoryTracking=summary` + `jcmd VM.native_memory`로
  Metaspace·코드캐시·다이렉트·스택을 각각 봅니다. 컨테이너 한도는 `-Xmx`가 아니라 이 **총합**에
  맞춰야 합니다.

## ⑤ 세 언어의 수렴 비교 — 할당 삼부작의 결산

3.2~3.4를 한 표로 접으면, 세 언어가 "커널의 큰 덩어리를 잘게 나눠 상각한다"는 같은 목표를
어떻게 서로 다르게 풀었는지가 한눈에 보입니다.

| 축 | Rust | Go | Java(HotSpot) |
| --- | --- | --- | --- |
| 소형 할당 빠른 경로 | malloc: tcache 슬롯(스레드 로컬) | mcache의 span 슬롯(P별, 비트맵) | **TLAB bump-the-pointer** |
| 그게 가능한 근거 | free list 재사용 | 크기 클래스 규격 + P별 캐시 | **Eden을 통째로 비우는 moving GC** |
| 객체마다 붙는 헤더 | 없음 (구조체 = 필드뿐) | 없음 (메타는 span에) | 12~16B (compact 8B, JEP 519) |
| 이스케이프 판정 시점 | 컴파일(AOT) · 결정적 | 컴파일(AOT) · 결정적 | 실행(JIT) · 조건부(deopt 가능) |
| 안 새는 객체의 처리 | 스택에 둠(비용 0) | 고루틴 스택에 둠(3.3) | 스칼라 치환으로 아예 제거 |
| 회수 | 프로그래머 / `Drop` | GC(런타임) | 세대별 GC(런타임) |

두 가지가 두드러집니다.

첫째, **bump-the-pointer가 셋 중 가장 싼 할당**이지만, 그건 Java가 **가장 무거운 GC(객체를
옮기는 세대별 GC)**를 짊어졌기 때문에 얻은 것입니다. 할당의 싸구려와 GC의 무거움은 **한
거래의 양면**입니다. Rust는 정반대편에서 "할당·회수 모두 프로그래머와 malloc이 관리한다"를
택했고, Go는 그 사이 어딘가에 섰습니다.

둘째, **이스케이프 처리의 '시점'이 갈립니다.** Go와 Rust는 컴파일 시점에 "스택이냐 힙이냐"를
**결정적으로** 정합니다(AOT라 늘 같은 답). Java는 실행 시점에 JIT이 **조건부로** 스칼라
치환을 시도하고, 그마저 deopt(2.3)로 뒤집힐 수 있습니다. 그래서 Java의 "할당 없음"은 강력
하지만 **불확실**하고, Go·Rust의 "스택 할당"은 소박하지만 **예측 가능**합니다. 2.4에서 말한
"peak throughput은 Java, tail·예측가능성은 Go/Rust"라는 결론이 여기서도 똑같이 메아리칩니다.

## ⑥ 트레이드오프 소결

JVM은 할당을 **덧셈 한 번**까지 깎아 냈습니다. 스레드마다 Eden 한 조각(TLAB)을 떼어 줘 경합을
없앴고, moving GC가 Eden을 통째로 비워 주기에 포인터를 앞으로 밀기만 하면 됐습니다. 여기까지는
순수한 이득입니다.

그 대가는 세 곳에서 청구됩니다. 첫째, **객체마다 12~16바이트 헤더**(compact로 8B까지) — 작은
객체가 많을수록 힙과 캐시를 갉아먹습니다. 둘째, **힙 밖 지형**(Metaspace·코드캐시·다이렉트·
스택)이 `-Xmx` 바깥에서 부풀어 **OOMKilled**를 일으킵니다. 셋째, 그리고 가장 근본적으로,
bump-the-pointer라는 편의 전체가 **moving GC라는 빚**을 담보로 합니다 — 그 빚의 상환 명세서가
4부입니다.

한 문장으로: **Java는 "할당을 세상에서 제일 싸게, 대신 그 편의를 GC와 객체 헤더로 갚는다"**를
택했습니다. 3.3의 Go(할당 싸게 + GC 있음)와 방향은 같되 더 극단으로 밀어붙였고, 3.2의
Rust(할당·회수 모두 수동, 헤더 없음)와는 정반대 끝입니다.

## ⑦ DevOps 관점 심화 질문

1. 컨테이너 메모리 한도를 정할 때 `-Xmx`만 보면 왜 OOMKilled가 나나요? `jcmd <pid>
   VM.native_memory summary`로 Metaspace·코드캐시·다이렉트·스택을 각각 얼마로 보고, 한도를
   총합의 몇 %로 잡아야 안전할까요?
2. `-Xmx`를 컨테이너 한도의 몇 %로 두는 게 좋을까요? `-XX:MaxRAMPercentage`(컨테이너 인식)와
   고정 `-Xmx` 중 무엇을, 어떤 근거로 택하나요? (2.4의 cgroup 인식 논의와 연결)
3. Minor GC가 잦다면 Eden이 작거나 할당률이 높다는 뜻일 수 있습니다. 할당률(allocation rate)을
   어떻게 계측하고(JFR, `gc` 로그의 Eden 회수량 ÷ 시간), TLAB 낭비·리필과 어떻게 엮어 보나요?
4. 스칼라 치환이 걸릴 거라 기대한 객체가 실제로는 힙에 할당되고 있습니다. JIT 로그·JFR·
   `-XX:+PrintInlining`로 이스케이프 분석·인라이닝 실패를 어떻게 확인하고, deopt storm(2.4)이
   개입했는지 어떻게 가려내나요?
5. 코드캐시(240MB)가 가득 차 JIT이 인터프리터로 떨어지는 절벽을, 어떤 지표
   (`CodeCache` 사용률, `-XX:+PrintCodeCache`)로 조기에 잡고, `ReservedCodeCacheSize`를 어떻게
   조정하나요?
6. compact object headers(JEP 519)를 켜면 힙이 줄지만 아직 기본이 아닙니다. 우리 워크로드에서
   실제 이득(힙·CPU·GC 횟수)을 어떻게 A/B로 측정하고, 켜기 전 호환성(에이전트·Unsafe·직렬화)에서
   무엇을 확인해야 하나요?

## ⑧ 다음 절 연결

이로써 세 언어가 커널의 큰 덩어리를 잘게 나누는 세 가지 방식 — malloc의 arena/bin(3.2),
Go의 mcache/span(3.3), JVM의 TLAB/Eden(3.4) — 을 모두 봤습니다. 3.5에서는 이 셋을 한자리에
모아 **직접 malloc / TCMalloc mcache / TLAB bump**를 수렴 비교하고, 마지막으로 이 절들이 계속
미뤄 온 질문 하나에 다리를 놓습니다: **"그래서 왜 GC가 필연이 되는가."** Go와 Java가 할당을
싸게 만들수록 회수의 짐이 어떻게 런타임으로 넘어오는지를 정리하며, 4부(GC의 한계와 환상)로
넘어갑니다.
