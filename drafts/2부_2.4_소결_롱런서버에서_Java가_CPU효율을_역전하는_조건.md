# 2부. CPU를 쥐어짜는 전략: 기계어 생성과 런타임 최적화
## 2.4 [소결] 롱런 서버에서 Java가 정적 컴파일 언어의 CPU 효율을 "역전"하는 조건

> 2.1~2.3을 관통한 단 하나의 축: **누가·언제·어떤 정보로 기계어를 만드는가.**
> Rust/Go는 *빌드 타임에 확정*(예측 가능하나 못 배움), Java는 *실행 중에 계속 재생성*(무겁고 불안정하나 배움).
> 이 절은 흔히 회자되는 "롱런에선 Java가 C/Rust보다 빠르다"는 주장을 **정직하게 조건부로** 규정하고,
> JIT과 AOT가 더는 이분법이 아니라 **연속 스펙트럼**임을 지형도로 그린다. 2부의 결산이다.

---

### ① 도달 지점

- "Java 역전"이 성립하는 **정확한 4조건**과, 그럼에도 **여전히 열세인 3축**을, 각각 *왜* 그런지까지
- 기계어 생성 방식(codegen) 말고도 CPU 효율을 함께 가르는 **숨은 축들**(메모리 레이아웃·GC의 CPU 점유·호출 규약)
- JIT ↔ AOT가 이분법이 아니라 **연속체**임을 보이는 지형도 (Rust · Go · GraalVM · Leyden · HotSpot)
- 시나리오별 선택을 한 장으로 압축한 **2부 결산 매트릭스**
- 2부 전체를 "프로파일을 언제 반영하고, 재적응이 필요한가"라는 한 축으로 통합, 그리고 3부(메모리 밑바닥)로 넘기는 다리

### ② "역전"의 해부 — 무조건이 아니라 조건부다

먼저 결론 문장을 정밀하게 못박는다.

> **충분히 오래 사는 CPU 바운드 서버의 *정상상태(steady-state) peak throughput*에 한해, C2가 런타임 프로파일(MDO)로 생성한 기계어가 정적 AOT 기계어의 효율을 따라잡거나 앞설 수 있다. 그러나 tail latency·메모리·시동에서는 여전히 열세다.**

"Java가 빠르다"가 아니라 **"어떤 지표에서, 어떤 조건일 때"** 가 전부다. 하나씩 *왜* 그런지 풀어보자.

#### 역전이 성립하는 4조건 (각각의 이유)

```
① 장수명(long-running)     2.3의 웜업 계단(L0→L3→L4)을 다 오를 시간이 필요하다.
                          C2가 MDO를 충분히 모아 역가상화·벡터화를 적용하기 전엔
                          Java는 오히려 느리다. 계단을 오르는 데 분~시간 단위가 든다.

② peak throughput이 지표    목표가 "정상상태 처리량"이어야 한다. 콜드 구간의 나쁜
                          응답을 평균에서 희석할 수 있어야 역전이 "보인다."
                          지표가 p99라면 웜업·deopt 스파이크가 그대로 드러나 역전이 안 된다.

③ CPU 바운드 + 다형성 많음   C2의 무기(MDO 기반 역가상화, 공격적 인라이닝, SuperWord)가
                          실제로 먹히려면, 다형 호출이 잦고 수치/루프 밀도가 높아야 한다.
                          I/O 바운드라면 애초에 CPU 기계어 품질이 병목이 아니다.

④ 안정 워크로드 + 메모리 예산  타입이 안정적이어야(deopt storm 없음) C2의 낙관적 최적화가
                          유지된다. 그리고 코드 캐시(240MB)·MDO·힙을 넉넉히 줄 수 있어야
                          "무거운 대신 영리한" 거래가 성립한다.
```

이 4조건이 모두 맞을 때, 2.3에서 본 C2의 돌연변이 최적화 — *정적 언어가 원리적으로 갖지 못한 정보(MDO)* 를 기계어에 녹인 것 — 가 AOT의 "빌드 시 스냅샷"을 실측으로 앞지른다.

> **④의 함정 — 가장 CPU가 절실한 순간에 JIT의 '배움'이 독이 된다.** 네 조건 중 '안정 워크로드'가 실무에서 가장 자주, 그것도 가장 하필인 때에 깨진다. 핫딜·선착순 이벤트로 **동시 접속이 폭증하는 바로 그 순간**, 평소 정상상태에선 밟지 않던 예외 처리 경로나 처음 보는 다형성 타입이 쏟아진다. 그러면 C2가 정상상태에서 심어둔 낙관적 가정(2.3-6의 타입 가드·CHA)이 무더기로 깨지며 **대규모 역최적화(deopt storm)** 가 터진다 — 핫 메서드들이 일제히 인터프리터로 추락하고, 재프로파일·재컴파일 큐가 밀리며, **가장 CPU가 절실한 피크 타임에 오히려 레이턴시가 수십 배로 튄다.** 정적 언어에는 이런 '피크에서의 자기 붕괴'가 구조적으로 없다(기계어가 불변이므로). JIT의 적응성은 잔잔한 물 위에선 강점이지만, 급변하는 물 위에선 그 자체가 리스크다 — ④ 조건의 무게가 여기에 있다.

#### 그럼에도 여전히 열세인 3축 (각각의 이유)

```
① tail latency (p99/p999)  역최적화 롤백(2.3-6), 세이프포인트 도달 지연 TTSP(2.3-7),
                           GC STW(4부)가 스파이크를 만든다. Rust/Go의 "p99≈p50"
                           결정론(2.1-6)을 원리적으로 못 이긴다.
② 메모리 발자국            코드 캐시 + MDO + JIT 자료구조 + 힙 여유.
                           항상 무겁다 → 노드 빈패킹 불리(5부).
③ 콜드 스타트 / 단수명      서버리스·scale-to-zero·CLI에서 웜업은 순손해.
                           계단을 오르기 전에 프로세스가 죽는다.
```

**즉 "역전"은 peak throughput이라는 한 축에서만, 그것도 4조건이 맞을 때만 일어난다.** 목차 초안의 "Java 역전"을 이렇게 조건부로 다듬는 것이 이 절의 정직함이다.

> **컨테이너의 냉혹한 현실 — OOMKilled (5부 복선).** 열세축 ②(메모리)를 클라우드 네이티브 맥락으로 내리면 더 아프다. 쿠버네티스 파드에서 Java가 겪는 가장 흔한 비극은 **힙은 남아도는데 죽는 것**이다. 컨테이너의 메모리 cgroup 리밋은 힙만이 아니라 **네이티브 메모리(JIT 코드 캐시·메타스페이스·스레드 스택·다이렉트 버퍼)까지 전부 합산**해서 잰다. 그래서 `-Xmx`로 힙을 아무리 묶어도, 코드 캐시·메타스페이스가 팽창하면 총 RSS가 리밋을 쳐 **OOMKilled** 당한다 — "힙은 60%인데 파드가 죽었다"는 그 현상이다. 정적 컴파일 언어는 발자국이 작을 뿐 아니라 그 **궤적이 예측 가능**해서 이 함정에서 훨씬 자유롭다. (off-heap OOMKilled의 정밀 해부는 5부.)

#### 숨은 축 — codegen만으로 CPU 효율이 결정되지 않는다

2.1~2.3은 "기계어를 어떻게 만드나"에 집중했지만, 실제 CPU 효율은 그 밖의 축들에도 좌우된다. 이것들이 위 "역전"을 더 조건부로 만든다.

- **메모리 레이아웃 / 데이터 지역성** (2.2 ③-5, 2.3 ⑤): Go의 `[]struct`는 연속 배치되어 하드웨어 프리패처에 유리한 반면, Java의 `Object[]`는 참조 배열이라 pointer chasing으로 캐시 미스가 잦다. **그래서 CPU 바운드라도 순회가 메모리 바운드라면**, C2의 SuperWord 벡터화 우위가 Go의 캐시 적중률에 상쇄될 수 있다("SIMD로 8개를 계산해도 데이터가 캐시에 없으면 멈춘다"). Valhalla(JEP 401)가 이를 좁히려 하나 JDK 25 기준 프리뷰다.
- **GC의 CPU 점유** (2.2 ③-2, 4부): Java든 Go든 GC는 tail만이 아니라 *처리량*으로도 CPU를 갉는다. Go는 마크 페이즈에 GOMAXPROCS의 ~25%를 백그라운드 워커에 내주고, 할당이 빠르면 Mark Assist가 고루틴을 마킹에 차출한다. Java의 GC도 세대·알고리즘에 따라 처리량 세금을 문다. Rust는 이 세금이 없다(제로코스트).
- **호출 규약** (2.2 ③-1): Go는 1.17부터 레지스터 기반 ABI로 인자를 넘겨 CALL 페널티를 줄였다. 이런 저수준 개선이 "정적이라 얕다"는 인상을 부분적으로 완화한다.

즉 "역전"을 논할 때 codegen만 보면 안 된다 — 데이터 지역성이 벡터화를, GC CPU가 순수 처리량을, 할당 규약이 호출 비용을 각각 흔든다.

### ③ 경계는 무너지고 있다 — JIT ↔ AOT 연속 스펙트럼

2.1~2.3을 "AOT vs JIT" 이분법으로 읽으면 2026년의 현실을 놓친다. 실제 지형은 **연속체**이고, 양끝이 서로를 향해 다가서고 있다. 축의 정체는 하나다 — **프로파일을 어디에 저장하고 언제 반영하는가.** Rust/Go는 프로파일을 *파일*(`.profdata`/`default.pgo`)에 담아 *빌드 시* 소비하고, Java는 *MDO*에 담아 *실행 내내* 소비한다.

```
 순수 AOT ◄──────────────────────────────────────────────► 순수 JIT
 (프로파일=파일, 빌드 시 1회 소비, 재적응 X)      (프로파일=MDO, 실행 내내 소비, 계속 적응)

 Rust          Go            GraalVM           Leyden          HotSpot
 (LLVM)        (자체 SSA)     native-image      AOT캐시          C1/C2
   │             │              │                 │               │
   │ +PGO/BOLT   │ +PGO(1.21)   │ +PGO            │ JEP483/514/515 │ (기준선: MDO)
   │  (~10-15%)  │  (2~7%)      │  (+GraalNN      │ AOT 클래스로딩   │
   │             │              │   정적 프로파일러) │ +AOT 메서드     │
   │             │              │                 │  프로파일        │
   ▼             ▼              ▼                 ▼               ▼
 프로파일을 "빌드 시" 흡수 ────────────►    ◄──── 프로파일/컴파일을 "미리" 흡수
```

**AOT가 프로파일(JIT의 무기)을 흡수하는 방향**
- Rust: `-Cprofile-generate/use`(PGO, 대체로 ~10%+, rustc 자체 ~15%) + BOLT(포스트-링크 basic-block 재배치). (2.1 ④)
- Go: `default.pgo` 자동 적용(1.21 GA), 핫 함수 인라인 예산을 80→2000으로 올리고 역가상화, 2~7%. (2.2 ④)
- GraalVM native-image: `--pgo-instrument → .iprof → --pgo`. JDK 24부터 **GraalNN(신경망 정적 프로파일러)** 로 계측 없는 프로파일 추론까지. → "AOT 컴파일러가 JIT 유사 최적화를 수행." (단 native-image PGO는 **Oracle GraalVM** 기능 — Community Edition엔 없음.)

**JIT이 AOT(정적의 무기)를 흡수하는 방향 — Project Leyden** (2.3 ④-b)
- JEP 483(JDK 24): AOT 클래스 로딩·링킹.
- JEP 514(JDK 25): AOT 캐시 원커맨드(`-XX:AOTCacheOutput`).
- JEP 515(JDK 25): **AOT 메서드 프로파일** — MDO 프로파일을 미리 캐시에 구워, JIT이 *시작하자마자* 기계어 생성. 시동 15~25% 단축(한 사례 73ms/19%, 캐시 +250KB).

**결론**: 진짜 축은 "언어/컴파일 방식"이 아니라 두 질문이다 — **(1) 프로파일을 언제 반영하는가(빌드 1회 ↔ 실행 연속), (2) 배포 후 재적응이 필요한가.** Rust·Go·GraalVM은 (1)을 빌드 쪽으로 당기고 (2)를 포기해 예측성을 산다. HotSpot은 (1)을 실행 내내 열어두고 (2)를 얻는 대신 무게를 진다. Leyden은 그 사이에 새 좌표를 찍는 중이다.

### ④ 2부 결산 매트릭스 — 시나리오별 선택

```
축 \ 런타임        Rust(AOT)     Go(AOT)       Java HotSpot    Java native-image
                                              (JIT)           (GraalVM AOT)
────────────────  ───────────  ───────────   ─────────────   ─────────────────
시동 / 웜업        ★즉시         ★즉시          ✗✗ 느림(계단)    ★즉시(웜업0)
정상상태 peak      높음          중간           ★★최고(4조건시)  중간(PGO로↑)
tail latency      ★★결정론적    안정           ✗ 스파이크        ○ 안정
메모리 발자국      ★★최소        작음           ✗✗ 큼            ★작음
자동 벡터화        △ LLVM        ✗ 수동asm       ○ C2 SuperWord   △(빌드시)
데이터 지역성       ★값 연속       ★[]struct 연속  ✗ Object[] 참조   ✗ Object[] 참조
                                              (Valhalla 대기)  (Valhalla 대기)
배포후 재적응       ✗            ✗              ★자동             ✗
빌드 속도          ✗ 느림        ★★빠름         중간(바이트코드)   ✗✗ 매우느림
멀티아키 대응       빌드매트릭스    빌드매트릭스     ★런타임감지       빌드매트릭스
배포 단순성        정적바이너리    ★정적바이너리   ✗ JVM동반         정적바이너리
```

**시나리오 → 추천 (2부 관점, CPU/기계어·메모리 지역성 축에 한정. 최종 종합은 7부)**
```
저지연·tail 민감(트레이딩·실시간 입찰)    → Rust      (p99 결정론이 지배 지표)
I/O 바운드 MSA·빠른 빌드/배포 리드타임     → Go        (기계어 여지 작고 운영 단순, 지역성 우수)
장수명·CPU 바운드·처리량(대형 백엔드·ETL)  → Java HotSpot (4조건 충족 시 peak 역전)
Java 생태계인데 서버리스/scale-to-zero    → GraalVM native-image (+PGO) (웜업 제거)
```

> **가드레일 — GraalVM native-image의 PGO 장벽.** 위 마지막 시나리오(Java 생태계의 서버리스)를 고를 때 함정이 있다. native-image PGO는 **Oracle GraalVM 전용**(③ 참고)이라, **PGO 없는 Community Edition의 AOT 기계어는 처리량이 C2 JIT은 물론 최신 Go 컴파일러보다도 떨어질 수 있다**(closed-world·프로파일 부재·기본 Serial GC). 즉 "웜업 제거"를 얻는 대신 "정상상태 처리량"을 크게 내줄 수 있으니, 이 선택은 단수명·저처리량 워크로드에만 안전하다. 처리량이 중요하면 Oracle GraalVM(PGO 포함) 또는 Leyden(JIT 유지 + 시동 단축, 2.3-④)을 저울에 함께 올려야 한다.

> 주의: 이 표는 **2부의 축(기계어 생성·CPU 효율·메모리 지역성)** 만 반영한다. 메모리 할당(3부)·GC(4부)·클라우드 경제학(5부)을 얹으면 그림이 달라진다. 최종 의사결정 매트릭스는 7부에서 완성한다.

### ⑤ 2부 트레이드오프 대통합

2부를 한 문장으로 압축하면:

> **"누가·언제·어떤 정보로 기계어를 만드는가"가 성능 프로파일 전체(peak·tail·메모리·시동)를 결정한다. 그리고 공짜 점심은 없다.**

```
빌드 타임에 결정  →  예측성 · 저메모리 · 즉시 시동  │ 그러나 재적응 불가 (스냅샷에 갇힘)
런타임에 결정     →  적응 · peak · 벡터화 자동      │ 그러나 웜업 · 스파이크 · 고메모리
```

Leyden(JIT→AOT)과 PGO(AOT→JIT)는 이 두 세계의 장점을 **부분 병합**하려는 시도다. 하지만 어느 쪽도 대가를 완전히 지우지 못한다 — Leyden도 대표성 있는 훈련 실행이 필요하고, PGO도 프로파일이 낡으면(트래픽 변화) 오히려 해가 된다. **트레이드오프는 이동할 뿐 사라지지 않는다.** 여기에 2부 후반에서 본 숨은 축들 — 데이터 지역성, GC의 CPU 점유, 호출 규약 — 이 겹치면, "어느 언어가 CPU 효율이 높은가"라는 질문 자체가 **워크로드의 형태(메모리 바운드냐 계산 바운드냐, 장수명이냐 단수명이냐, 다형성이 심하냐)에 따라 답이 갈리는** 다차원 문제임이 드러난다. 이것이 2부가 남기는 사고 도구다.

### ⑦ 다음 부 연결 — 3부로

2부는 **기계어**를 다뤘다. 그러나 그 기계어가 끊임없이 만지는 대상은 **메모리**다. 2.3에서 스쳐 간 "이스케이프 분석·스칼라 치환·TLAB·힙 할당", 그리고 방금 본 "MDO·코드 캐시가 사는 그 힙"은 사실 훨씬 깊은 지층 위에 서 있다 — 커널의 `brk`/`mmap`, libc `malloc`, JVM의 TLAB, Go의 mcache/TCMalloc.

3부는 그 밑바닥으로 내려간다. 그리고 1부 목차의 부정확한 복선 하나를 정정한다(2.1 ④에서 이미 씨앗을 심었다): **"Rust가 `brk`/`mmap`을 직접 호출한다"는 부정확하다** — Rust는 대개 시스템 allocator(libc `malloc`, 또는 `#[global_allocator]`로 교체한 jemalloc/mimalloc)에 위임하고, `malloc`이 그 아래에서 `brk`/`mmap`을 관리한다. 3부에서 이 계층(할당자 → 커널)을 정확히 해부하며, 2.2에서 예고한 "왜 힙 할당이 GC의 CPU 세금이 되는가"의 무대를 완성한다.
