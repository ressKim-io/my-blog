# 4부 4.5 소결: 은탄환은 없다 — 비용 보존 법칙과 세 GC의 삼각형

> **이 문서의 성격** — 최종 블로그 원고가 아니라, 최종 집필 에이전트에게 넘길 1차 심화 정리(reference-level draft)입니다. 이 절은 4.1~4.4를 종합하는 소결이므로 새 사실보다 **연결과 검증**에 무게를 둡니다. 그림은 모두 ASCII, 표 형태 비교는 마크다운 표, 수치는 앞 절들의 기준(JDK 25 · Go 1.26 · Linux 7.x)을 따릅니다.

---

## ① 이 절에서 도달할 지점

3.5에서 트레이드오프 삼각형(단편화 ↔ 이동 비용 ↔ 회계의 위치)을 세우고, "GC의 청구서가 실제로 얼마나 무거운지는 4부에서 검증한다"고 약속했습니다. 4.1에서 공용 어휘를, 4.2~4.4에서 Go·G1·ZGC를 하나씩 뜯었습니다. 이제 그 조각들을 한자리에 모아 약속을 이행합니다. 다 읽고 나면 다음을 스스로 말할 수 있게 됩니다.

- 4.1~4.4에서 본 세 GC(그리고 Rust)를 3.5의 트레이드오프 삼각형 위에 **정확히 어디에** 놓는지, 그리고 각자가 그 대가를 **어떤 물리적 청구서로** 내는지.
- **"GC는 은탄환"이라는 프레이밍이 왜 틀렸는지** — 감이 아니라 3.5에서 세운 **비용 보존 법칙**을, 이번에는 실제로 뜯어본 청구서로 검증하는 방식.
- 세 GC의 차이가 사실은 **"할당 단계의 선택(bump냐 슬롯이냐)이 회수 단계를 이미 결정했다"**는 3.5 인과의 사슬의 귀결이라는 것.
- 그리고 이 GC의 청구서(멈춤·처리량 세금·메모리 여유 요구)가 런타임 안에 머물지 않고 **클라우드 경제학으로 번진다**는 것 — 5부로 넘길 씨앗.

한 문장으로 앞당겨 말하면, 4부의 결론은 3.5가 예언한 그 문장의 GC판입니다. **"메모리 회수 비용은 사라지지 않는다. 청구서의 수신처와 형태만 바뀔 뿐이다."**

## ② 여정 회고 — 네 절이 각각 세운 것

종합에 들어가기 전에, 4.1~4.4가 각각 무엇을 확정했는지 한 문단씩 되짚습니다.

**4.1(기본 원리)**은 공용 어휘를 세웠습니다. GC는 "다시 쓰이나?"라는 못 푸는 질문을 "닿을 수 있나(도달성)?"로 근사하는 보수적 장치이고, 삼색 마킹으로 살아있음을 추적하며, 동시 마킹의 안전을 위해 **쓰기/읽기 배리어**를 답니다. 그리고 그 배리어가 곧 "STW를 없앤 값을 처리량으로 치르는" 환전의 실체임을 예고했습니다.

**4.2(Go)**는 비세대·비이동 마크-스윕의 초상입니다. 이동이 없어 pause가 sub-ms지만, 비세대라 매 사이클 살아있는 힙 전체를 훑고, 못 따라가면 **Mark Assist**로 할당하는 고루틴을 GC 노동에 징집해 꼬리 지연을 튀깁니다. 디스코드가 2분마다 10~40ms씩 튄 게 그 증거였습니다(`forcegcperiod` + 비세대 + Mark Assist의 합작).

**4.3(G1)**은 세대별·이동식입니다. region으로 힙을 쪼개 평소엔 young만 훑어 디스코드형 문제에 강하고, evacuation으로 단편화를 없앱니다. 대신 이사를 STW로 하고(evacuation pause), 부분 회수가 할당을 못 따라가면 **Full GC 절벽**(to-space exhausted)으로 힙 전체를 수 초간 멈춥니다.

**4.4(ZGC)**는 그 이사마저 동시로 합니다. colored pointer + load 배리어로 뮤테이터를 안 멈추고 옮겨, pause를 힙 크기와 무관하게 O(1) sub-ms(0.1~0.5ms)로 눌렀습니다. Full GC 절벽이 사라진 대신, **load 배리어 처리량 세금**(쓰기 배리어의 ~5배)과 **allocation stall**(회수가 할당을 못 따라가면 뮤테이터의 할당을 막음, 그런데 pause 지표엔 안 잡힘)이라는 두 청구서를 받습니다.

네 절을 관통하는 한 줄은 4.1의 명제였습니다. **동시성은 지연을 처리량으로 환전한다.** Go는 살짝(Mark Assist), G1은 pause로 정직하게, ZGC는 가장 극단적으로(배리어 세금 + stall) 환전했습니다.

## ③ 트레이드오프 삼각형에 네 방식을 얹다

3.5의 삼각형을 다시 가져와, 이번에는 4부에서 검증한 **각자의 청구서**를 꼭짓점 옆에 실제로 적어 넣습니다. 이것이 4부의 그림 한 장 요약입니다.

```text
                              회계의 위치
                       (사람 · 컴파일러 · 런타임)
                                  ▲
                                 ╱ ╲
                        Rust ──▶ ╱   ╲   회계를 컴파일러·사람에게 넘김
                       (GC 없음) ╱     ╲   → 런타임 회수 0
                               ╱       ╲    청구서: 소유권 인지 비용,
                              ╱         ╲            Arc 순환 누수·원자적 카운터
                             ╱           ╲
             단편화 ◀───────╱─────────────╲───────▶ 이동 비용
          (비이동의 대가)                        (이동의 대가)
                │                                       │
          ┌─────┴─────┐                     ┌───────────┴───────────┐
          │    Go     │                     │   G1              ZGC  │
          │  비이동    │                     │ 이동(STW)      이동(동시) │
          │  마크-스윕  │                     │                        │
          └───────────┘                     └────────────────────────┘
     청구서:                            청구서:
      · Mark Assist 꼬리 지연             · G1  : evacuation pause + Full GC 절벽
      · 비세대 전체 스캔(디스코드)          · ZGC : load 배리어 처리량 세금(~5×)
      · 단편화(size class로 완화)                 + allocation stall(pause에 안 잡힘)

   ── 한 꼭짓점을 아끼면 반드시 다른 꼭짓점에서 낸다 ──
     Rust  : 회계를 컴파일 타임으로 → 런타임 비용 0, 대신 사람이 냄
     Go    : 이동을 포기        → 이동 비용 0, 대신 단편화 + Mark Assist
     G1    : 이동을 채택        → 단편화 0, 대신 pause + Full GC 위험
     ZGC   : 이동을 '동시'로     → pause O(1), 대신 처리량 세금 + allocation stall
```

Rust는 삼각형 꼭대기(회계를 사람·컴파일러에게)에 있어 런타임 회수 자체가 없습니다. Go는 왼쪽(비이동)에서 이동 비용을 0으로 만든 대신 단편화와 Mark Assist를 감수합니다. G1과 ZGC는 둘 다 오른쪽(이동)에 있지만 **지불 방식이 다릅니다.** G1은 이사를 STW pause로, ZGC는 같은 이사를 동시로 치러 pause를 없애고 그 값을 처리량과 stall로 옮겼습니다. 삼각형의 어느 자리에 서든, 비용의 총량은 보존되고 **모양과 청구 시점만** 바뀝니다.

## ④ 최종 비교 매트릭스

삼각형을 축별 수치로 펼치면 다음 매트릭스가 됩니다. 4부 전체를 한 표로 접은 것이자, 7부 의사결정 매트릭스의 골격입니다.

| 축 | Rust | Go | Java G1 | Java ZGC |
|---|---|---|---|---|
| **회계 위치** | 컴파일러 + 사람 | 런타임 | 런타임 | 런타임 |
| **회수 방식** | `drop` / `Rc`·`Arc` | 동시 마크-스윕 | 세대별 이동(evacuation) | 세대별 **동시** 이동 |
| **이동?** | — | 비이동 | 이동(STW) | 이동(동시) |
| **세대?** | — | 비세대 | 세대별 | 세대별 |
| **배리어** | 없음 | 쓰기(하이브리드) | 쓰기(SATB + 카드) | **읽기(LVB)** + 세대 store |
| **멈춤 모델** | 없음 | STW sub-ms(<100µs) + 상시 꼬리 | evacuation pause(CSet에 비례, 목표 200ms) | **O(1) sub-ms(0.1~0.5ms), 힙 무관** |
| **tail latency** | 예측 가능 | Mark Assist로 튐 | 대체로 안정, Full GC 시 폭발(수 초) | 최상(단 allocation stall 예외) |
| **throughput** | 최상 | 양호 | 최상급 | G1 대비 5~10% ↓ |
| **메모리 오버헤드** | 최소(헤더 없음) | 낮음(`GOGC`=100 → ~2× live) | 헤더 + 세대 복사 + region 예비 | 큼(headroom 요구, 작은 힙 불리) |
| **회수가 할당을 못 따라갈 때** | 해당 없음 | **Mark Assist**(뮤테이터 징집) | **Full GC 절벽**(수 초 STW) | **allocation stall**(뮤테이터 블록) |
| **힙 스위트스팟** | 무관 | 중소 | 중대 | 대(수십 GB ~ TB) |
| **compressed oops 32GB 절벽(3.4)** | — | — | 있음 | **없음**(64비트 colored pointer) |

이 표에서 가장 중요한 행은 **"회수가 할당을 못 따라갈 때"**입니다. 평상시엔 셋 다 조용합니다. 진짜 성격은 GC가 몰릴 때 드러나고, 그때 각자 다른 곳이 아픕니다. Go는 요청 스레드가 붙들리고(Mark Assist), G1은 힙 전체가 멈추며(Full GC), ZGC는 할당 자체가 막힙니다(stall). **어느 것도 "안 아픈" 선택지는 없습니다.** 아픈 부위가 다를 뿐입니다.

## ⑤ "은탄환은 없다"의 검증 — 비용 보존 법칙, GC판

3.5에서 세운 보존 법칙은 이랬습니다. **"이 메모리를 아직 누가 쓰는가라는 회계는 사라지지 않는다. 없앨 수 없고 어디에 둘지만 고른다."** 3.5에서는 이것을 논리로 세웠고, 4부에서는 각 GC를 실제로 뜯어 **청구서 실물**을 확인했습니다. 이제 그 청구서들을 한 표에 모으면, "은탄환"이라는 환상이 왜 틀렸는지가 실증됩니다.

핵심은 각 GC가 자랑하는 "공짜처럼 보이는" 지점 뒤에, 4부에서 우리가 실제로 찾아낸 청구서가 반드시 하나씩 붙어 있다는 것입니다.

| GC | 자랑(공짜처럼 보이는 것) | 4부가 찾아낸 숨은 청구서 |
|---|---|---|
| **Go** | "튜닝 없이 sub-ms STW, 싼 할당" | 비세대라 매 사이클 전체 스캔 + Mark Assist가 요청 스레드를 징집(디스코드 2분 스파이크) |
| **G1** | "대체로 매끄럽고 안정적" | to-space 고갈 시 **Full GC 절벽**(수 초, pause 목표 무시) |
| **ZGC** | "힙이 TB여도 pause sub-ms" | load 배리어 **상시 처리량 세금**(~5×) + **allocation stall**(pause 지표에 안 잡히는 지연) |
| **Rust** | "GC 없이 메모리 안전, 런타임 비용 0" | 소유권·수명의 **인지 비용** + `Arc` 순환 누수·원자적 카운터 경합 |

네 행 모두 오른쪽 칸이 비어 있지 않습니다. 이것이 검증의 결론입니다. **비용은 어느 GC에서도 사라지지 않았습니다.** Go는 지연을 "요청 스레드의 즉흥 노동"에, G1은 "드문 큰 멈춤"에, ZGC는 "상시 처리량 세금 + 보이지 않는 stall"에, Rust는 "개발자의 머리"에 청구했을 뿐입니다.

그래서 "GC는 은탄환"이라는 말은 정확히 이렇게 교정됩니다. **GC는 은탄환이 아니라, 청구서를 어디에 숨길지 고르는 서로 다른 거래입니다.** 특히 ZGC를 조심해야 합니다. pause 대시보드가 sub-ms로 아름답게 찍히므로 "드디어 공짜"라는 착시가 가장 강한데, 그 대금은 처리량 그래프와 응답 시간 분포라는 **다른 화면**에 찍혀 있습니다. 청구서를 안 보이게 만든 것이지 없앤 게 아닙니다.

## ⑥ 할당이 GC를 결정했다 — 3.5 인과의 사슬, 완결

4부를 다 보고 나면 3.5에서 세운 인과의 사슬이 완성됩니다. 3.5는 "할당 방식이 GC 성격을 미리 결정한다"고 예고했고, 4부는 그 예언이 정확히 맞았음을 보여 줍니다.

```text
   할당 단계의 선택               →      회수 단계의 운명(4부에서 검증)

   bump-the-pointer (JVM, 3.4)   →   이동식 GC 필수(영역 리셋 전 survivor 이사)
                                     ├─ STW로 이사 → G1: evacuation pause + Full GC (4.3)
                                     └─ 동시로 이사 → ZGC: 배리어 세금 + allocation stall (4.4)

   슬롯 비트맵 (Go, 3.3)          →   비이동 마크-스윕(제자리 재사용)
                                     └─ 이동 비용 0, 대신 비세대 전체 스캔 + Mark Assist (4.2)

   free list (Rust·C, 3.2)       →   GC 없음(수동/RAII 회수)
                                     └─ 런타임 회수 0, 대신 인지 비용 + malloc 단편화
```

"객체를 옮길 수 있느냐"라는 할당 단계의 한 가지 결정이, 회수 단계의 멈춤 모델·배리어 종류·실패 양상까지 전부 도미노처럼 정합니다. bump 할당(3.4)을 택한 JVM은 이동식 GC를 피할 수 없었고, 이동을 STW로 하느냐 동시로 하느냐가 G1과 ZGC를 갈랐습니다. 슬롯 할당(3.3)을 택한 Go는 비이동을 얻은 대신 세대별의 이점을 포기해 디스코드형 취약점을 안았습니다. **4부의 세 GC는 서로 무관한 세 설계가 아니라, 3부에서 내린 할당 결정의 세 가지 논리적 귀결**이었습니다.

## ⑦ DevOps 관점 심화 질문

1. **네 개의 "실패 양상" 중 우리 SLO에 가장 치명적인 것은?** Mark Assist 꼬리 지연(Go), Full GC 절벽(G1), allocation stall(ZGC), 인지 비용·개발 속도(Rust) 중 무엇이 우리 서비스의 SLO를 먼저 깨는지를 기준으로 언어·GC를 좁힌다면, 그 판단을 어떤 지표(p99.9 분포·GC 로그·처리량·리드타임)로 내리겠는가.

2. **"pause만 보는 대시보드"의 맹점을 어떻게 메우는가?** ZGC의 sub-ms pause에 속지 않으려면 allocation stall·처리량·응답 시간 분포를 pause와 **같은 화면**에 올려야 합니다. G1은 반대로 pause 분포의 꼬리(Full GC)를 봐야 합니다. GC별로 "진짜 봐야 할 지표"가 다른 대시보드를 어떻게 설계하는가.

3. **같은 워크로드를 G1과 ZGC로 A/B 하면 무엇이 갈리는가?** 힙 크기·꼬리 지연·처리량·CPU·메모리 headroom의 다섯 축에서, 어느 힙 크기 지점부터 ZGC가 G1을 역전하는지(작은 힙에선 ZGC 불리), 그 손익분기를 부하 테스트로 어떻게 찾는가.

4. **GC 세금을 컨테이너 CPU·메모리 예산과 어떻게 연결하는가?** 백그라운드 마킹(Go 25%·G1 ConcGCThreads·ZGC ConcGCThreads)과 배리어 세금이 cgroup CPU 쿼터를 갉아먹고, GC headroom이 메모리 리밋을 잡아먹습니다. GC가 요구하는 CPU·메모리 여유를 파드 requests/limits에 어떻게 반영하는가.

5. **"GC 없는 Rust"의 값을 어디서 정당화하는가?** 런타임 세금 0과 예측 가능한 지연이 주는 이득이, 소유권·수명의 인지 비용과 개발 속도 저하를 넘어서는 컴포넌트(예: 지연 극단 민감·거대 상주 캐시·사이드카)는 무엇이고, 그 경계를 팀 생산성 지표로 어떻게 긋는가.

6. **세대 가설이 깨지는 워크로드를 어떻게 식별하는가?** 오래 사는 거대 캐시가 힙의 대부분인 서비스(디스코드형)에서, 비세대 Go는 매번 전체를 스캔하고 세대별 G1/ZGC는 유리합니다. pprof·JFR·`/gc/heap/live`로 "라이브 힙이 크고 오래 산다"를 배포 전에 어떻게 가려내는가.

## ⑧ 4부를 닫으며 — 5부로

4부는 "GC의 청구서가 얼마나 무거운가"를 삼각형 위에서 하나씩 검증했습니다. 도착한 결론은 3.5의 예언 그대로입니다. **GC는 은탄환이 아니라, 청구서를 어디에 숨길지 고르는 거래이며, 그 청구서는 보존되고 형태만 바뀐다.** Go는 요청 스레드에, G1은 드문 큰 멈춤에, ZGC는 처리량과 보이지 않는 stall에 숨겼습니다.

그런데 이 청구서들은 런타임 안에 얌전히 머물지 않습니다. **클러스터로 새어 나갑니다.** GC가 요구하는 메모리 headroom(GOGC의 2배 힙, ZGC의 여유 page)은 **노드에 파드를 몇 개 얹을 수 있는가(빈패킹)**를 직접 좌우합니다. GC와 JIT 워밍업이 겹치는 시작 구간은 **콜드 스타트** 지연이 되어 오토스케일링과 서버리스의 경제성을 갉아먹습니다. 그리고 cgroup은 힙과 네이티브 메모리를 합산하므로(1.3·2.4·3.4), 힙 밖 메모리(다이렉트 버퍼·Metaspace·스레드 스택)가 조용히 부풀면 **OOMKilled**로 파드가 죽습니다 — GC 로그에는 아무 이상이 없는데도.

**5부(MSA·클라우드 경제학)**는 바로 이 지점에서 시작합니다. 4부에서 뜯은 GC의 지연·처리량·메모리 세금이, 5부에서 빈패킹·콜드 스타트·off-heap OOMKilled라는 **돈과 SLO의 언어**로 어떻게 번역되는지를 추적합니다. 런타임 안의 삼각형이 클러스터 위의 청구서로 확장되는 여정입니다. (1.3에서 예고한 off-heap OOMKilled의 물리, 2.4의 cgroup 합산 회계, 3.4의 힙 밖 지형이 여기서 하나로 모입니다.)
