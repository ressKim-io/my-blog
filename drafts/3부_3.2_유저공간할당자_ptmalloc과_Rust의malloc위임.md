> **이 문서의 성격** — 최종 블로그 원고가 아니라, 최종 집필 에이전트에게 넘길 1차 심화
> 정리(reference-level draft)입니다. `개념 → 하드웨어 동작 → 비용 해부 → 회피 전략 →
> 3개 언어 연결 → 트레이드오프 → 심화 질문`의 순서로 아래에서 위로 쌓아 올립니다. 그림은
> 모두 ASCII로 그렸고, 본문의 수치는 특정 CPU 세대·커널 설정에서의 측정 예시이므로 환경에
> 따라 달라질 수 있음을 전제로 읽어 주세요.

# 3.2 유저 공간 할당자: malloc은 커널이 준 큰 덩어리를 어떻게 잘게 나누나

## ① 이 절에서 도달할 지점

3.1에서 우리는 "커널은 메모리를 크게, 그리고 게으르게 내어 준다"는 사실을 확인했습니다.
문제는, 프로그램이 실제로 다루는 객체는 대부분 작다는 것입니다. 문자열 하나, 노드 하나,
버퍼 하나. 이 작은 것들을 매번 커널에 직접 요청할 수는 없습니다. 그 사이를 메꾸는 계층이
바로 이 절의 주인공, **유저 공간 할당자(malloc)**입니다.

이 절을 끝까지 읽고 나면 다음을 스스로 설명할 수 있게 됩니다.

- **malloc과 free는 왜 존재하는가.** 3.1에서 본 커널 비용(시스템콜 약 1,400사이클 + 페이지
  폴트)을 객체 하나하나마다 치르지 않으려는 **상각(amortization) 계층**이라는 점.
- **glibc `ptmalloc`의 세 기둥.** 메모리 풀인 **arena**, 실제로 나눠주는 조각인 **chunk**,
  반납된 조각을 크기별로 보관하는 서랍장 **bin** — 이들이 각각 어떤 자료구조인지.
- **`brk`와 `mmap`의 갈림길.** malloc이 요청 크기에 따라 둘 중 무엇으로 커널을 부르는지를
  결정하는 **`M_MMAP_THRESHOLD`**(기본 128KB, 동적 조정, 상한 32MB)의 정확한 규칙.
- **단편화와 RSS의 관계.** 내부·외부 단편화가 왜 생기는지, 그리고 "분명히 free를 했는데도
  프로세스의 메모리 사용량(RSS)이 줄지 않는" 현상의 물리적 원인.
- **2부 복선의 회수.** "Rust가 `brk`/`mmap`을 직접 호출한다"던 서술이 왜 부정확했는지의
  완전한 정정. (Rust는 malloc에 위임하며, 1.32부터 jemalloc 대신 시스템 할당자를 기본으로
  씁니다.)
- **언어별 설계 철학.** 같은 목적(커널 비용 상각)을 두고 Go와 JVM은 왜 malloc을 통째로
  우회하고 자기만의 할당 계층을 짓는지 (3.3·3.4의 예고).

## ② 개념과 물리적 실체

### malloc과 free는 무엇을 약속하는가

`malloc(n)`이 하는 약속은 의외로 단순합니다. "**최소 `n`바이트짜리 연속된 메모리 블록**의
시작 주소를 돌려주겠다. 그리고 그 주소는 일정한 경계에 **정렬(align)**되어 있을 것이다."
이게 전부입니다. glibc 64비트 환경의 기본 정렬은 **16바이트**입니다(`MALLOC_ALIGNMENT =
2 × sizeof(size_t)`).

여기서 "정렬"을 잠깐 풀어 두겠습니다. CPU는 8바이트짜리 값을 8의 배수 주소에서 읽을 때
가장 빠르고, 어떤 SIMD 명령은 아예 16바이트·32바이트 정렬을 강제합니다. 정렬되지 않은
주소에서 읽으면 느려지거나 예외가 납니다. 그래서 할당자는 사용자가 그 안에 무슨 타입을
담든 안전하도록, 넉넉한 경계(16바이트)에 맞춰 주소를 돌려줍니다. 반대로 말하면, 우리가
"17바이트만 주세요"라고 해도 할당자는 내부적으로 16의 배수로 반올림합니다. 이 반올림이
뒤에 나올 **내부 단편화**의 씨앗입니다.

`free(p)`는 그 블록을 반납합니다. 어디서 가져와 어떻게 재사용할지는 전적으로 할당자의
재량입니다. 사용자는 "달라"와 "돌려준다"만 말할 뿐, 그 뒤의 살림살이는 모릅니다.

### 그런데, 이 계층이 왜 필요할까

3.1에서 봤듯 커널에서 메모리를 받아 오는 일은 공짜가 아닙니다. `brk`나 `mmap` 시스템콜
자체가 약 1,400사이클, 게다가 받아 온 메모리를 처음 만질 때마다 페이지 폴트(minor 폴트
수백~수천 사이클)가 따라옵니다. 문제는 서버 프로그램이 이 요청을 **초당 수백만 번** 한다는
점입니다.

구체적으로 그려 보겠습니다. 웹 서버가 요청 하나를 처리하면서 작은 객체 수천 개를 만들었다
버립니다. 만약 이 객체들을 전부 커널에 직접 요청한다면, 요청 하나마다 시스템콜 한 번 +
페이지 폴트가 붙습니다. 초당 수만 건의 HTTP 요청이 들어오면, CPU는 정작 비즈니스 로직이
아니라 커널을 들락거리는 "메모리 주문서 처리"에만 매달리게 됩니다.

그래서 malloc은 **커널과 프로그램 사이의 도매상** 노릇을 합니다. 하는 일은 세 가지입니다.

1. 커널에서 **큰 덩어리를 한 번에** 떼어 옵니다(예: 132KB를 `brk` 한 번으로). — 시스템콜
   비용을 여러 할당에 나눠 상각합니다.
2. 그 덩어리를 프로그램이 요청한 크기대로 **잘게 잘라** 건네줍니다.
3. 프로그램이 `free`한 조각을 버리지 않고 **재고로 쌓아 두었다가**(free list) 다음
   `malloc` 요청에 시스템콜 없이 되팝니다.

정리하면, 3.1이 "도매(커널)"였다면 3.2는 "소매(malloc)"입니다. 이 관점을 쥐고 있으면
아래의 복잡한 자료구조들이 전부 "재고를 어떻게 효율적으로 관리할 것인가"라는 한 가지
질문의 변주로 읽힙니다.

### ptmalloc의 세 겹: arena → heap → chunk

glibc가 쓰는 malloc은 **ptmalloc2**입니다. Doug Lea가 만든 dlmalloc을 Wolfram Gloger가
멀티스레드 환경으로 확장한 물건입니다. 구조는 크게 세 겹으로 포개져 있습니다.

```text
 [ arena : 락(mutex) 하나가 지키는 메모리 왕국 ]
     └── [ heap : 커널에서 받아 온 연속된 땅 ]
             └── [ chunk ][ chunk ][ chunk ] … [ top chunk : 아직 안 나눈 벌판 ]
```

세 겹이 각각 어떤 문제를 푸는지 풀어 보겠습니다.

- **arena — 락 경합을 나누기 위한 단위.** arena는 "메모리 풀 하나 + 그 풀을 지키는 락
  하나"입니다. 만약 모든 스레드가 단 하나의 arena를 공유하면, 여러 스레드가 동시에
  `malloc`을 부를 때 그 락 하나를 두고 줄을 서야 합니다. 이 병목을 풀려고 ptmalloc은
  스레드가 몰릴 때 새로운 arena를 만들어 부담을 분산합니다. 프로그램에 처음부터 있는
  것이 **main arena**이고, 이것만 3.1의 `brk`로 자랍니다. 경합 때문에 추가로 생기는
  **thread arena**는 `mmap`으로 새 땅을 받습니다. 스레드는 TLS(스레드 로컬 저장소) 변수
  `thread_arena`를 통해 자기 arena에 묶입니다. arena 개수의 기본 상한은 64비트에서
  **코어 수 × 8**(`M_ARENA_MAX`), 32비트에서는 코어 수 × 2입니다.
- **heap — arena가 실제로 딛고 선 땅.** arena가 관리하는 연속된 메모리 구획입니다. main
  arena는 프로세스의 program break를 `brk`로 밀어 올리며 자라고, thread arena는 `mmap`으로
  받은 구획을 필요한 만큼만 `mprotect`로 접근 가능하게 켜 가며 씁니다 — 3.1에서 본
  reserve(예약)→commit(사용) 패턴 그대로입니다.
- **chunk — 실제로 손님에게 파는 최소 단위.** malloc이 사용자에게 돌려주는 조각 하나가
  chunk입니다. 헤더와 페이로드로 이루어져 있고, `free`되면 크기별 서랍인 **bin**에 꽂혀
  다음 손님을 기다립니다. 바로 다음 절에서 이 chunk를 뜯어봅니다.

## ③ 비용과 메커니즘 해부

### chunk 하나를 뜯어보기

malloc이 나눠주는 조각 하나가 chunk입니다. glibc 64비트 기준 구조체는 이렇게 생겼습니다.

```c
struct malloc_chunk {
  size_t  mchunk_prev_size;  /* 바로 앞 chunk가 free일 때만 그 크기. 앞이 사용 중이면 앞이 빌려 씀 */
  size_t  mchunk_size;       /* 이 chunk의 크기 + 하위 3비트 플래그 */

  /* ↓ 아래 네 필드는 이 chunk가 free일 때만 의미가 있고, 사용 중일 땐 페이로드가 이 자리를 씀 */
  struct malloc_chunk* fd;          /* free 이중 연결 리스트: 다음 chunk */
  struct malloc_chunk* bk;          /* free 이중 연결 리스트: 이전 chunk */
  struct malloc_chunk* fd_nextsize; /* large bin 전용: 다음 크기 그룹 */
  struct malloc_chunk* bk_nextsize; /* large bin 전용: 이전 크기 그룹 */
};
```

여기서 두 가지 절약 기법이 숨어 있는데, 둘 다 "쓰지 않는 공간을 재활용한다"는 발상입니다.

**첫째, `size` 필드의 하위 3비트를 플래그로 훔쳐 씁니다.** chunk 크기는 항상 16의 배수라
하위 4비트가 늘 0입니다. 이 놀고 있는 비트에 메타데이터를 얹습니다.

- **`P` (PREV_INUSE, bit 0)** — 바로 앞 chunk가 사용 중인가? 0이면 앞이 free 상태라는 뜻이고,
  이 정보로 `free` 시 앞 chunk와 병합할지 판단합니다.
- **`M` (IS_MMAPPED, bit 1)** — 이 chunk가 arena 힙이 아니라 독립적인 `mmap`으로 받은
  것인가? (뒤의 `M_MMAP_THRESHOLD`와 연결됩니다.)
- **`A` (NON_MAIN_ARENA, bit 2)** — main arena가 아니라 thread arena 소속인가?

**둘째, 사용 중일 땐 이웃의 `prev_size` 칸까지 빌려 씁니다.** 이게 **경계 태그(boundary
tag)** 기법입니다. 말로는 헷갈리니 이웃한 두 chunk를 바이트 단위로 늘어놓고 보겠습니다.

```text
 낮은 주소 ────────────────────────────────────────────────────▶ 높은 주소

  ┌────────┬────────────────────────────┬───────────┬────────┬──────────
  │ size_A │  A의 페이로드                │ prev_size │ size_B │ B의 페이로드 …
  │ (헤더) │  ← malloc이 A로 돌려준 포인터 │   of B    │ (헤더) │
  └────────┴────────────────────────────┴───────────┴────────┴──────────
     8B                                   └─ 이 8B가 "B의 prev_size" 칸입니다.
                                             · A가 사용 중이면 → A의 페이로드가 여기까지 잡아먹습니다.
                                             · A가 free되면    → 여기에 "A의 크기"가 기록됩니다.
```

`prev_size` 필드는 원래 "바로 앞 chunk가 free일 때 그 크기를 적어 두는" 칸입니다. 그런데
앞 chunk(A)가 **사용 중**이라면 이 칸은 쓸모가 없습니다. 그래서 그 8바이트를 앞 chunk의
페이로드가 빌려 씁니다. 덕분에 명목상 헤더는 16바이트(prev_size + size)지만, 실질적인
오버헤드는 **8바이트**로 줄어듭니다.

이 규칙을 알고 나면 크기 계산이 자연스럽게 따라옵니다. `n`바이트를 요청하면 실제 chunk
크기는 `align16(n + 8)`이 되고, 최소 chunk 크기는 **32바이트**입니다(`MINSIZE`, 64비트).
예를 들어 `malloc(17)`을 부르면, 17 + 8 = 25를 16의 배수로 올려 **32바이트 chunk**가
잡히고, 실제로 안전하게 쓸 수 있는 공간은 24바이트가 됩니다. 요청한 17바이트와 실제로
차지한 32바이트의 차이 — 이 자투리가 **내부 단편화**입니다.

### free된 chunk를 어디에 보관하나 — bin

`free`된 chunk를 아무렇게나 쌓아 두면, 다음 `malloc`이 맞는 크기를 찾느라 전부 뒤져야
합니다. 그래서 ptmalloc은 반납된 chunk를 크기별로 분류해 **bin**이라는 서랍에 꽂아 둡니다.
설계 철학은 한 문장으로 요약됩니다: **자주 쓰는 작은 것은 락도 없이 즉시, 큰 것은 정확히
맞춰서, 최후엔 커널로.** `malloc(n)`이 빈 자리를 찾는 순서를 먼저 그림으로 훑고, 아래에서
하나씩 풀어 보겠습니다.

```text
 malloc(n) 요청
     │   (위에서부터 차례로 확인 → 찾는 즉시 반환)
     ▼
 ┌──① tcache ─────── 크기 딱 맞는 빈 · ≤1032B · 락 없음
 │      └─ 있음 ▶ 반환
 ├──② fastbin ────── 작은 재고 · ≲128B · 병합 유예
 │      └─ 있음 ▶ 반환
 ├──③ unsorted ───── 방금 free된 대기열을 훑기
 │      ├─ 맞음 ▶ 반환
 │      └─ 안 맞음 ▶ small / large 서랍으로 정리
 ├──④ small / large ─ 크기별 서랍 · exact / best-fit
 │      └─ 있음 ▶ 반환
 ├──⑤ top chunk ──── arena 끝 미분할 벌판에서 잘라냄
 │      └─ 충분 ▶ 반환
 └──⑥ 커널 ───────── brk(힙 확장) · 큰 것은 mmap ────────▶ 반환
```

1. **tcache (thread cache).** 스레드마다 하나씩 갖는 **락 없는** 개인 캐시입니다. 24바이트
   부터 1,032바이트까지의 크기를 16바이트 간격으로 나눈 **64개의 빈**을 두고, 빈 하나에
   최대 **7개**의 chunk를 LIFO로 쌓아 둡니다. 락을 잡지 않으므로 대부분의 소형 할당·해제가
   여기서 곧장 끝납니다. 이게 **glibc 2.26(2017)** 이후 malloc이 빨라진 결정적 이유입니다.
2. **fastbin.** tcache가 꽉 찼거나 담지 못하는 작은 chunk(기본적으로 약 128바이트 이하)를
   받습니다. 속도를 위해 인접 chunk와의 **병합을 일부러 미루고**, 단일 연결 리스트에 LIFO로
   쌓습니다. 빠른 대신 조각을 남기는 전략입니다.
3. **unsorted bin.** 방금 free된 chunk가 잠시 대기하는 이중 연결 리스트 한 줄입니다.
   ptmalloc은 정리를 게을리하는 대신, 다음 `malloc`이 이 대기열을 훑고 지나가며 "마침 크기가
   맞으면 바로 쓰고, 아니면 small/large bin으로 제자리 정리"를 합니다. 일종의 지연 분류입니다.
4. **small bin.** 빈 하나가 **정확히 한 크기**만 담습니다(16바이트 간격, 약 1KB 미만).
   같은 크기 요청이 오면 검색 없이 바로 꺼내 줄 수 있어 빠릅니다. FIFO로 동작합니다.
5. **large bin.** 여기서부터는 빈 하나가 크기의 **범위**를 담습니다. 그래서 그 안에서
   요청에 가장 가까운 것을 골라야 하는데(best-fit), 이를 빠르게 하려고 크기순으로 정렬해
   두고 `fd_nextsize`/`bk_nextsize`라는 스킵 포인터로 크기 그룹을 건너뜁니다.
6. **top chunk (wilderness).** 위 서랍이 전부 비었으면, arena 맨 끝에 남아 있는 커다란
   미분할 덩어리에서 필요한 만큼 잘라 줍니다. arena의 "개척되지 않은 벌판"인 셈입니다.
7. **커널 요청.** top chunk마저 모자라면, 그제서야 `brk`로 힙을 늘리거나 `mmap`을 부릅니다.
   둘 중 무엇을 쓰는지가 바로 다음 이야기입니다.

서랍마다 생김새가 다릅니다. 어떤 것은 단일 연결(한쪽 끝에서만 넣고 빼기), 어떤 것은
이중 연결(중간에서 빼내고 이웃과 병합하기)인데, 그 모양을 한자리에 모으면 이렇습니다.

```text
 단일 연결 · LIFO   (head 한쪽에서만 넣고 뺌 → 락 최소, 이웃 병합 안 함)
   tcache  [24B] ─▶ [C] ─▶ [C] ─▶ [C]      (빈 하나에 최대 7개)
           [40B] ─▶ [C] ─▶ [C]
           [56B] ─▶ (빔)
             ⋮      16B 간격으로 1032B까지 · 64칸
   fastbin [32B] ─▶ [C] ─▶ [C]

 이중 연결 · fd/bk 양방향   (중간에서 빼내기·이웃과 병합하기에 유리)
   unsorted    ⇄ [C] ⇄ [C] ⇄ [C] ⇄     (정리 전 chunk가 잠시 머무는 한 줄, 순환)
   small[k]    ⇄ [C] ⇄ [C]             (칸마다 정확히 한 크기 · FIFO)
   large[k]    ⇄ [C] ⇄ [C] ⇄ [C]       (한 칸이 크기 '범위' · 크기순 정렬)
                    └─▶ fd_nextsize 로 다른 크기 그룹으로 건너뛰기
```

여기서 왜 두 종류인지가 드러납니다. tcache·fastbin은 "제일 자주, 제일 빠르게"가 목표라
한쪽 끝만 만지는 **단일 연결**로 충분합니다. 반대로 unsorted·small·large는 chunk를 중간에서
빼내거나(정렬·재분류) 이웃과 합쳐야(coalescing) 하므로, 양방향으로 오갈 수 있는 **이중
연결**이어야 합니다.

`free`할 때는 반대로, 방금 반납한 chunk의 양옆이 이미 free 상태인지 확인합니다(앞은
`prev_size`, 뒤는 그다음 chunk의 `P` 플래그로 판별). 이웃이 비어 있으면 **병합(coalescing)**
해서 더 큰 free chunk로 되돌립니다. 잘게 쪼개진 구멍이 다시 뭉쳐야 큰 요청을 받을 수 있기
때문입니다.

### brk냐 mmap이냐 — M_MMAP_THRESHOLD

top chunk로도 부족해 결국 커널을 불러야 할 때, malloc은 **요청 chunk의 크기**를 임계값과
비교해 호출 방식을 가릅니다.

> - 요청 크기 **< `M_MMAP_THRESHOLD`** → arena 힙에서 해결 (main arena면 `brk`, thread
>   arena면 힙 구획 확장)
> - 요청 크기 **≥ `M_MMAP_THRESHOLD`** → 이 요청만을 위한 **독립적인 `mmap`** (`IS_MMAPPED=1`)

정확한 수치는 이렇습니다. 임계값의 초기값은 **128KB**입니다. 그런데 이 값은 고정이 아니라
프로그램의 행동을 보고 **동적으로 상향**됩니다. 임계값보다 크면서 상한
(`DEFAULT_MMAP_THRESHOLD_MAX`) 이하인 `mmap` chunk가 `free`될 때마다, 임계값을 그 크기까지
끌어올립니다. 상한은 64비트에서 `4 × 1024 × 1024 × sizeof(long)` = **32MB**, 32비트에서는
512KB입니다.

왜 이렇게 나누는지를 풀어 보겠습니다. 두 가지 상반된 실패를 동시에 피하려는 설계입니다.

- **큰 블록을 `brk` 힙에 두면 반환이 막힙니다.** `brk` 힙은 아래에서 위로 자라는 담과
  같아서, 커널에 돌려주려면 담의 **꼭대기(top)**부터 헐어야 합니다. 힙 한가운데의 큰
  블록이 free돼도, 그 위에 살아 있는 chunk가 하나라도 있으면 커널에 돌려줄 수 없습니다.
  그래서 큰 것은 아예 **독립 `mmap`**으로 따로 떼어 둡니다. 그러면 `free`되는 순간
  `munmap`으로 통째 반납되어 깔끔합니다.
- **반대로, 큰 할당이 반복되면 mmap이 오히려 손해입니다.** 어떤 프로그램이 200KB짜리를
  계속 할당했다 해제한다고 합시다. 매번 `mmap` + `munmap` 시스템콜이 나가면 그게 다시
  비용입니다. 그래서 "이 정도 크기는 자주 재사용되는구나" 싶으면 임계값을 그만큼 올려서,
  다음부터는 mmap 대신 힙에서 재사용하도록 유도합니다. (단, 사용자가 `M_MMAP_THRESHOLD`를
  직접 지정하면 이 동적 조정은 꺼집니다.)

곁들여 알아 둘 손잡이 두 개: **`M_MMAP_MAX`**는 동시에 유지할 수 있는 mmap chunk의 최대
개수로 기본 **65,536**이고, **`M_TRIM_THRESHOLD`**는 top chunk가 이 크기를 넘으면 `brk`를
내려 커널에 반환하는 기준으로 기본 **128KB**입니다.

> **여기서 2부의 복선이 물리적으로 닫힙니다.** "Rust가 `brk`/`mmap`을 직접 호출한다"는
> 서술이 왜 부정확한지가 이제 완전히 보입니다. `brk`냐 `mmap`이냐는 **malloc이** 요청
> 크기와 이 임계값 상태를 보고 내리는 판단이지, 언어가 정하는 것이 아닙니다. Rust의
> `Vec`은 그저 "이만큼 주세요"라고 `malloc`을 부를 뿐입니다.

### 단편화, 그리고 "free 했는데 RSS가 안 줄어드는" 미스터리

단편화는 두 종류입니다.

- **내부 단편화(internal).** 요청 크기를 16바이트나 크기 클래스 경계로 반올림하면서 생기는
  자투리입니다. 앞서 본 `malloc(17)`이 32바이트를 차지하던 그 낭비입니다. 개별로는 작아도
  객체 수억 개가 쌓이면 무시할 수 없습니다.
- **외부 단편화(external).** free된 구멍의 **총량은 충분한데** 잘게 흩어져 있어서, 큰
  연속 블록 요청을 받아 주지 못하는 상태입니다. 병합(coalescing)이 이를 늦추지만 완전히
  없애지는 못합니다.

여기서 실무자가 가장 자주 부딪히는 미스터리 하나를 짚겠습니다. **"트래픽이 빠졌는데도
프로세스 RSS가 피크에 고정되어 내려오지 않는다."** 원인은 앞에서 이미 나왔습니다. glibc의
main arena는 힙의 **꼭대기(top)가 비어 있을 때만** `brk`를 내려 커널에 메모리를 돌려줍니다.
힙 한가운데에 아직 살아 있는 chunk가 하나라도 있으면, 그 위쪽 공간이 아무리 비어도 담을
헐 수 없습니다. 오래 사는 서버에서는 이런 "박힌 못" 하나 때문에 반환이 막히는 일이 흔합니다.

대응책은 두 갈래입니다. glibc를 그대로 쓴다면 `malloc_trim()`을 명시적으로 호출하거나
`M_TRIM_THRESHOLD`를 조정해 반환을 유도합니다. 아니면 아예 **jemalloc**처럼 사용하지 않는
페이지를 시간에 따라 서서히 커널에 돌려주는(decay) 할당자로 갈아탑니다.

### 멀티스레드 경합과 ARM64의 약한 메모리 모델

ptmalloc은 락 경합을 줄이려고 arena를 코어 수 × 8까지 늘린다고 했습니다. 하지만 arena마다
top·bin 같은 관리 구조가 통째로 중복되기 때문에, arena가 늘수록 **메모리 오버헤드(arena
폭발)**가 따라옵니다. 속도(경합 감소)와 메모리(중복)를 맞바꾸는 셈입니다. tcache는 이
문제를 우회합니다 — 스레드 로컬이라 아예 락을 잡지 않으니, 소형 할당이 대부분인 워크로드는
arena 락을 거의 건드리지 않습니다.

**여기서 ARM을 함께 봅니다.** tcache·fastbin의 연결 리스트나, jemalloc·mimalloc의 lock-free
free list는 모두 **원자적 연산(CAS)**에 기대어 여러 스레드가 락 없이 같은 자료구조를
만집니다. 그런데 이게 아키텍처마다 비용이 다릅니다.

여기서 "약한 메모리 모델"이라는 말을 정의부터 풀겠습니다. 현대 CPU는 성능을 위해 메모리
읽기/쓰기의 순서를 **프로그램에 적힌 순서와 다르게 재배열**할 수 있습니다. 이 재배열을
얼마나 허용하느냐가 아키텍처마다 다릅니다. **x86-64는 강한 모델(TSO, Total Store Order)**
이라 재배열이 거의 없어서, 평범한 저장·적재만으로도 대체로 순서가 지켜집니다. 반면
**AArch64(ARM64)는 약한 모델**이라 재배열이 훨씬 자유롭습니다. 그래서 "이 쓰기가 저 쓰기보다
반드시 먼저 보여야 한다"를 보장하려면 **명시적 메모리 배리어**를 프로그래머(또는 컴파일러)가
직접 넣어야 합니다 — ARM64에서는 `ldar`(load-acquire), `stlr`(store-release) 같은 명령이
그 역할을 합니다.

정리하면, 같은 malloc 구현이라도 ARM64에서는 배리어가 더 촘촘히 들어가고, 이 배리어와
캐시라인 경합(false sharing)의 비용이 성능에 드러납니다. Graviton·Ampere 같은 ARM 서버에서
malloc 집약 워크로드를 돌릴 때 x86과 스케일링 특성이 갈리는 지점이 바로 여기입니다.

## ④ 회피·최적화 전략

- **`MALLOC_ARENA_MAX`로 arena 수 제한.** 코어가 많은 컨테이너에서는 arena가 코어 수 × 8까지
  불어나 RSS가 예상보다 크게 튑니다. 환경 변수 `MALLOC_ARENA_MAX=2`(또는 1)로 묶으면 RSS는
  내려가지만 락 경합은 올라갑니다. JVM처럼 내부적으로 native malloc을 쓰는 런타임의 컨테이너
  에서 특히 자주 만지는 손잡이입니다.
- **`malloc_trim()` / `M_TRIM_THRESHOLD`.** 힙 꼬리를 강제로 커널에 돌려주어 RSS를 회수합니다.
  위에서 본 "반환이 안 되는" 문제에 대한 직접적인 대응입니다.
- **할당자 자체를 교체.** 링크 시점에 할당자를 바꾸면 malloc의 계약(인터페이스)은 그대로 둔
  채 내부 전략만 갈아끼울 수 있습니다.
  - **jemalloc** (FreeBSD·Redis·과거의 Rust). 크기 클래스로 요청을 분류하고, arena를 코어에
    고정 배정해 경합을 줄입니다. 무엇보다 사용하지 않는 메모리를 시간에 따라 서서히 반환
    (decay)하는 정책이 있어, 단편화와 tail latency 관리에 강합니다.
  - **mimalloc** (Microsoft). 페이지마다 free list를 따로 두고, 자기 스레드가 반납한 것과
    다른 스레드가 반납한 것을 분리 관리해 경합과 false sharing을 줄입니다. 구현이 작고 빠르며
    캐시 지역성이 좋습니다.
- **크기 클래스를 의식한 자료구조 설계.** 요청 크기가 크기 클래스 경계를 살짝 넘으면 다음
  클래스로 반올림되어 낭비가 커집니다. 예컨대 구조체를 65바이트로 설계하면 80바이트 클래스로
  올림되어 15바이트가 매번 버려집니다. 경계에 맞춰 다듬으면 내부 단편화를 줄일 수 있습니다.

## ⑤ 세 언어의 수렴 비교 — malloc 위임 완전 정정

3.1의 핵심이 "커널은 크게, 게으르게 준다"였다면, 3.2의 질문은 **"그 위의 도매 계층(malloc)을
그대로 빌려 쓸 것인가, 아니면 직접 지을 것인가"**입니다.

| 언어 | `malloc`(ptmalloc) 계층 | 커널을 부르는 주체 | 비고 |
| --- | --- | --- | --- |
| **C / C++** | 그대로 사용 (직접) | `malloc`이 `brk`/`mmap` 호출 | 교과서적 경로 |
| **Rust** | 그대로 사용 (`System` 할당자) | `malloc`이 `brk`/`mmap` 호출 | **★ 위임 — 2부 복선 완전 정정** |
| **Go** | 우회 (자체 할당자) | 런타임이 `mmap` 직접 호출 | 정적 바이너리, 3.3에서 해부 |
| **Java (JVM)** | 객체는 우회 (GC 힙/TLAB) | JVM이 `mmap` 직접 호출 | 단, JNI·Direct 버퍼는 `malloc`, 3.4에서 |

### Rust — 복선의 완전한 회수

Rust의 기본 전역 할당자는 **`System`**, 즉 플랫폼 libc의 `malloc`/`free`입니다(리눅스라면
방금까지 뜯어본 glibc ptmalloc). `Box::new`, `Vec`, `String` — 힙을 쓰는 모든 것이 이 경로를
탑니다. 내부적으로는 `alloc::alloc` → `__rust_alloc` → `System.alloc` → `malloc`으로
내려갑니다.

역사를 알면 정정이 더 분명해집니다. Rust는 초창기(1.0 전후)에 **jemalloc을 기본으로
내장**했습니다. 브라우저 엔진 Servo처럼 할당이 잦은 워크로드에서 성능이 좋았기 때문입니다.
하지만 jemalloc을 끼고 다니면 실행 파일이 커지고, 일부 아키텍처에서는 지원이 끊겼으며,
유지보수 부담도 있었습니다. 게다가 "모든 Rust 프로그램이 할당에 병목이 걸리는 것도 아니다"
라는 판단이 더해져, **RFC 1183**을 거쳐 **Rust 1.32(2019)**에서 기본을 다시 **시스템
할당자**로 되돌렸습니다. 지금은 jemalloc이 필요하면 `#[global_allocator]`로 명시해서 선택
합니다(`tikv-jemallocator` 등). 참고로 컴파일러 `rustc` 자신은 빌드 속도 때문에 여전히
내부적으로 jemalloc을 씁니다.

그래서 2부의 문장은 **이중으로** 부정확했습니다. ① Rust는 커널이 아니라 `malloc`을 부르고,
② `brk`/`mmap`을 고르는 결정은 Rust가 아니라 malloc 구현체가 내립니다. 실무적으로 이게
왜 중요하냐면, **Rust 서비스의 메모리 성격(단편화, tail latency, RSS 반환)이 상당 부분 그
아래 깔린 malloc 구현체에 좌우된다**는 뜻이기 때문입니다. "제로코스트 추상화"로 언어
오버헤드를 없앴어도, 힙의 실제 거동은 malloc이 쥐고 있습니다. 그래서 Rust에서
`#[global_allocator]`를 jemalloc이나 mimalloc으로 바꾸는 코드 한 줄이 곧바로 성능을
바꾸는 것입니다.

### Go와 Java — 도매 계층을 직접 짓다

- **Go.** libc를 링크하지 않는 정적 바이너리라, malloc 계층을 통째로 우회하고 런타임이 직접
  `mmap`으로 arena를 받아 자체 할당자로 나눕니다(3.3). `cgo`로 C 코드를 부를 때에만 그
  부분이 libc malloc을 탑니다.
- **Java(JVM).** 자바 객체는 GC 힙 안의 **TLAB**(3.4)에서 나오므로 malloc을 우회합니다.
  하지만 JVM 자신(C++로 짜여 있죠), **JNI**, **DirectByteBuffer**, Netty 같은 off-heap
  버퍼는 native malloc을 씁니다. 그래서 자바 컨테이너에서 `-Xmx`로 힙을 제한해도 막지 못하는
  RSS가 존재하고, 이것이 **OOMKilled**의 단골 원인이 됩니다(2.4·5부와 연결).

### 수렴점

네 진영 모두 결국 3.1의 커널 계약(`brk`/`mmap`, demand paging, overcommit) 위에 서 있습니다.
갈리는 지점은 단 하나 — 그 위에 **공용 도매상(malloc)을 함께 쓰느냐(C/C++/Rust), 전용
도매상을 자체 건설하느냐(Go/JVM)**입니다. 다음 두 절이 바로 그 "전용 도매상"의 내부
도면입니다.

## ⑥ 트레이드오프 소결

malloc은 커널 왕복 비용을 상각하려고 arena·chunk·bin·tcache라는 정교한 자료구조 탑을
쌓았습니다. 공짜는 아니어서, 그 대가로 세 가지 세금을 냅니다.

1. **단편화** — 내부(반올림 자투리)와 외부(흩어진 구멍).
2. **arena 중복** — 멀티스레드 확장을 위해 관리 구조를 복제하며 생기는 메모리 오버헤드.
3. **반환 지연** — `brk` 힙의 꼬리 제약 탓에 free해도 RSS가 바로 내려오지 않는 문제.

tcache가 속도를, 동적 mmap 임계값이 깔끔한 반환을, arena 분산이 확장성을 벌어다 주지만,
그 각각이 위 세 세금 중 하나로 청구됩니다.

Rust가 이 계층을 **위임**한다는 사실의 무게가 여기서 드러납니다. Rust는 언어 차원의
오버헤드를 "제로코스트"로 없앴지만, 힙 메모리의 실제 성격은 바깥(malloc)에 맡겨 두었습니다.
반대로 Go와 JVM은 이 도매 계층을 스스로 소유해 언어 고유의 의미(가비지 컬렉션, 고루틴
스택)와 완전히 통합하는 대신, 런타임의 무게를 짊어졌습니다(3.3·3.4). **"언어가 가볍다"와
"메모리가 예측 가능하다"는 서로 다른 축의 이야기**라는 점 — 이게 이 절이 남기는 한 문장입니다.

## ⑦ DevOps 관점 심화 질문

1. 코어 32개짜리 노드의 컨테이너에서 RSS가 예상보다 높습니다. 원인이 glibc arena 폭발
   (코어 수 × 8)인지 어떻게 확인하나요? (`/proc/PID/smaps`의 heap 세그먼트 개수, `MALLOC_
   ARENA_MAX`를 바꿔 가며 비교) 그리고 컨테이너의 CPU 쿼터(cgroup)와 glibc가 인식하는
   "온라인 코어 수"는 서로 일치하나요?
2. 서비스 트래픽이 줄어도 RSS가 내려오지 않을 때, 원인이 glibc의 top-chunk 반환 제약인지
   어떻게 검증하고, `malloc_trim()`과 jemalloc의 `dirty_decay_ms` 중 무엇으로 대응하는 게
   맞을까요?
3. glibc·jemalloc·mimalloc 중 하나를 고를 때 p99 tail latency, 단편화율(RSS ÷ 실사용량),
   처리량 중 어디에 가중치를 둬야 하며, 그 판단을 어떤 프로파일러(`jeprof`, `malloc_stats`
   등)로 증명하나요?
4. Rust 바이너리에서 `#[global_allocator]`를 jemalloc으로 바꾼 뒤 "좋아졌다"를 어떻게 수치로
   보이나요? 할당이 병목이 아닌 서비스라면 왜 차이가 거의 없을 수 있나요?
5. JVM 컨테이너의 OOMKilled를 조사할 때, `-Xmx`로 잡은 자바 힙과 JNI·Direct 버퍼가 쓰는
   native malloc RSS를 어떻게 분리해서 계측하나요?(NMT: Native Memory Tracking, `pmap`,
   `jcmd`) 여기에 `MALLOC_ARENA_MAX`가 왜 끼어드나요?
6. AArch64(Graviton/Ampere)에서 malloc 집약 워크로드의 스케일링이 x86과 다르게 나타난다면,
   약한 메모리 모델에 따른 배리어 비용과 캐시라인(64B vs Apple 128B) false sharing 중 무엇을
   먼저 의심하고, 어떻게 관측하겠습니까?

## ⑧ 다음 절 연결

지금까지 C와 Rust가 **공용 malloc**을 어떻게 나눠 쓰는지 봤습니다. 이제 이 계층을 통째로
**우회한** 진영으로 내려갑니다. 3.3에서는 **Go의 자체 할당자**를 엽니다 — 구글 **TCMalloc**
계보를 이은 **`mcache`(각 P마다 하나, 락 없음) → `mcentral` → `mheap`**의 3층 구조, 크기를
표준 규격으로 묶는 **size class**와 **span**, 아주 작은 객체들을 한데 뭉치는 **tiny
allocator**, 그리고 고루틴 **스택이 자라는 방식(연속 스택·copystack)**까지. 특히 2.2에서
이스케이프 분석이 "이건 힙으로 보낸다"고 판정했던 그 객체가 **실제로 `mcache`의 어느 span에
떨어지는지**, 그 착지 지점을 물리적으로 추적하겠습니다.
