### 분석

1. **sungjeon Redis 효율 최고 (1/3 부하)**: polling 5초 + heartbeat 10초 분리로 Redis req/s가 ~300. status 응답 38ms로 가장 빠름. VU 증가 시 Redis 포화 가능성 가장 낮음
2. **suyeon 중간 포지션**: polling 2초로 Redis 부하 ~500. status 57ms로 양호. enter 336ms가 가장 느린 건 JWE 토큰 발급 오버헤드
3. **junsang 처리량 최고 (2배)**: polling 1초로 순번 감지 빠름 → 단위 시간 내 가장 많은 사용자 처리 (1629명). 하지만 Redis 비용 ~1000 req/s로 가장 높음
4. **junsang TTL 만료 리스크**: `waiting-ttl: 30초`로 67건 만료. suyeon도 154건 만료 (Entry TTL 10분이지만 seat-enter에서 ALREADY_ADMITTED 판정). sungjeon은 heartbeat가 TTL 갱신해서 **0건**
5. **대기 시간 p95는 junsang 유리**: polling 1초라 순번 도달 감지가 빠름 (p95 2m13s). sungjeon은 5초 간격이라 3m40s. 사용자 체감 대기 시간은 junsang이 가장 짧음

### 한줄 요약

| 구현체 | 강점 | 약점 |
|--------|------|------|
| **sungjeon** | Redis 효율 최고 (1/3), 안정성 최고 (timeout 0) | 처리량 낮음, 대기 p95 길음 |

| **junsang** | 처리량 최고 (2배), 대기 p95 짧음 | Redis 부하 최대, TTL 만료 리스크 |

### 1000 VU → 3000 VU 스케일 비교

| 지표 | junsang 1K→3K | suyeon 1K→3K | sungjeon 1K→3K |
|------|--------------|--------------|----------------|
| queue_status avg | 110ms → **2.66s** (24배↑) | 57ms → **926ms** (16배↑) | 38ms → **158ms** (4배↑) |
| queue_enter avg | 148ms → **9.43s** (64배↑) | 336ms → **1.9s** (6배↑) | 77ms → **844ms** (11배↑) |
| queue_pass_rate | 96% → **100%** | 88% → **87%** | 100% → **100%** |
| queue_timeouts | 67 → **0** | 154 → **157** | 0 → **0** |

### 분석

1. **sungjeon Redis 효율 압도적**: status avg 158ms로 junsang(2.66s) 대비 **17배**, suyeon(926ms) 대비 **6배** 빠름. 5초 polling + heartbeat 분리가 대규모에서 결정적 차이
2. **sungjeon 선형 스케일**: status 38ms→158ms (4배)로 VU 3배 증가에 비례. 추가 확장 여력 충분
3. **junsang 비선형 포화**: status 110ms→2.66s (24배)로 VU 3배 증가에 8배 악화. 1초 polling으로 Redis 3000 req/s가 bottleneck
4. **suyeon 중간 포지션**: status 57ms→926ms (16배). CDN 캐싱 없이 2초 polling 1500 req/s가 부담. CDN 적용 시 크게 개선 예상
5. **suyeon timeout 지속 (157건)**: 1000VU(154건)와 비슷. JWE 토큰 + 분산 Lock 경합으로 enter 1.9s → seat-enter까지 도달 지연
6. **suyeon 처리량은 최고**: iterations 1180, http_reqs 655/s로 가장 높음. polling 2초가 1초(junsang)보다 효율적이면서 5초(sungjeon)보다 반응 빠름

## 공통 이슈 (3개 구현체)

| 이슈 | 원인 | 해결 |
|------|------|------|
| ApiSuccessResponse 역직렬화 | @JsonCreator 없음 | 3개 브랜치 전부 수정 완료 |
| Istio payment retry 중복 | retryOn: 5xx + POST | 5xx 제거 (reset,connect-failure만) |
| Cloudflare Workers HTML | go-ti.shop/prefix → SPA | api.go-ti.shop/prefix로 우회 |
