# 2부. CPU를 쥐어짜는 전략: 기계어 생성과 런타임 최적화
## 2.3 Java (JIT): 느린 웜업의 반전, C2의 런타임 돌연변이 최적화

> 2.1·2.2에서 본 AOT 언어들은 같은 벽에 부딪혔다 — **정적 스냅샷은 배포 이후의 세상을 모른다.**
> Java는 정반대 도박을 한다. 빌드 시점엔 기계어를 **아예 만들지 않고**(바이트코드만), 실행 중에 프로파일을 보면서 기계어를 **계속 다시 만든다.**
> 그 대가로 Java는 가장 느리게 시작하지만(웜업), 충분히 오래 달리면 정적 언어가 원리적으로 놓치는 최적화를 잡아낸다.
> 1부 1.3에서 심은 복선 — *"가장 무거운 JVM이 왜 롱런 정점 처리량에서 앞서는가"* — 의 답이 이 절에 있다.

---

### ① 도달 지점

- **티어드 컴파일**의 5레벨과 **정확한 임계값**(Tier3=200, Tier4=5000)이 무엇을 세는지
- 프로파일이 실제로 저장되는 물리적 장소 — **MDO(MethodData Object)** 라는 런타임 자료구조
- C2의 "돌연변이 최적화" 5종을 기전 수준으로: 공격적 인라이닝, **인라인 캐시 상태기계**(mono→mega)로 본 역가상화, **SuperWord** 벡터화, 스칼라 치환, 트랩 배치
- **역최적화(deoptimization)** 가 물리적으로 어떻게 일어나나 — 컴파일된 프레임을 인터프리터 프레임으로 되짜는 **scope descriptor**의 역할
- **세이프포인트(safepoint)와 TTSP** — GC가 아닌데도 생기는 멈춤, tail latency의 숨은 근원
- 웜업을 다스리는 세 갈래(C1 고정 / **Leyden AOT 캐시** / native-image)와 **세그먼트 코드 캐시**(JEP 197)
- ARM에서 JIT이 실행 CPU를 감지해 그 아키 기계어를 런타임 생성하는, AOT엔 없는 사치

---

### ② 개념 / 물리적 실체 — 기계어가 "실행 중에" 태어난다

`javac`가 만드는 것은 기계어가 아니라 **바이트코드(.class)** 다. 바이트코드는 특정 CPU의 명령이 아니라, JVM이라는 가상 기계를 위한 중간 명령이다. 그래서 x86이냐 ARM이냐는 이 시점에 **결정되지 않는다.** 진짜 기계어는 **실행 중, JVM(HotSpot) 안에서** 태어난다.

```
[.java] --javac--> [바이트코드 .class]   ← 아키 불가지론. 아직 "기계어"가 아님.
                          │
                          ▼   실행 시작 (java ...)
   ┌──────────────────────────────────────────────────────┐
   │  HotSpot JVM                                          │
   │  1) 인터프리터가 바이트코드를 한 줄씩 실행 (느림)             │
   │       + 실행하면서 관찰 내용을 [MDO]에 기록               │
   │         (호출 횟수, 분기 빈도, 이 자리에 온 실제 타입 …)     │
   │  2) 임계치를 넘은 "핫" 메서드를 JIT이 컴파일                 │
   │       → 기계어를 [코드 캐시]에 적재                        │
   │  3) 이후 그 메서드는 기계어로 실행 (빠름)                    │
   └──────────────────────────────────────────────────────┘
```

여기서 두 개의 물리적 실체를 기억하자.

- **코드 캐시(code cache)** = JIT이 만든 기계어가 쌓이는, `mmap`으로 잡힌 실행 가능(RX) 메모리 영역. *실행 중에 채워진다.* 그래서 **같은 프로그램이라도 실행할 때마다 코드 캐시의 내용이 달라질 수 있다** — 어떤 입력이 들어왔고 어떤 분기가 뜨거웠는지에 따라. 2.1의 Rust 바이너리가 `objdump`로 늘 같은 바이트를 보였던 것과 정확히 반대다. 이 가변성이 "예측 불가능(웜업·스파이크)"의 근원이자 "적응(peak)"의 원천이다.
- **MDO(MethodData Object)** = 각 메서드에 붙는 "관찰 노트". 인터프리터와 C1이 코드를 돌리면서 여기에 기록을 남긴다. 뒤에서 C2가 이 노트를 읽고 최적화를 결정한다. 즉 MDO는 "정적 언어가 갖지 못한 정보(실행 실측)"가 물리적으로 담기는 장소다. 이 절의 거의 모든 마법이 MDO에서 나온다.

### ③ 비용·메커니즘 해부

#### (1) 티어드 컴파일 — 컴파일러가 둘이고, 임계값이 있다

HotSpot은 JIT 컴파일러를 **둘** 쓴다. **C1(client compiler)** 은 빠르게 컴파일하되 최적화는 얕고, 대신 코드에 프로파일 계측을 심는다. **C2(server compiler)** 는 느리게 컴파일하되 최적화가 깊다. 이 둘을 5개 레벨로 엮는다.

```
Level 0 : 인터프리터              (느림, MDO에 프로파일 수집)
Level 1 : C1, 최적화 O · 프로파일 X  (트리비얼 메서드용 — 더 관찰할 게 없음)
Level 2 : C1, 제한적 프로파일        (호출·백엣지 카운터만)
Level 3 : C1, 완전 프로파일          (+ MDO 타입 프로파일까지 — C2에게 줄 데이터 축적)
Level 4 : C2, 최대 최적화            (핫 코드의 종착지)

표준 경로:   0 ──(호출 200회)──▶ 3 ──(호출 5000회)──▶ 4
             인터프리터              C1(프로파일)          C2(고성능)
             Tier3InvocationThreshold=200   Tier4InvocationThreshold=5000
```

임계값의 의미를 정확히: 각 메서드에는 **호출 카운터(invocation counter)** 와 **백엣지 카운터(back-edge counter, 루프가 도는 횟수)** 가 있고, 이 값들이 MDO에 쌓인다. 호출 카운터가 **200**을 넘으면 L3(C1+완전 프로파일)로 컴파일되고, L3에서 프로파일을 충분히 모은 뒤 카운터가 **5000**을 넘으면 L4(C2)로 승격된다. 3→4 전이의 카운터는 MDO의 "프로파일된 호출 수"에서 온다.

왜 굳이 C1을 거치나? C2 컴파일은 비싸다(느리다). 시작하자마자 모든 메서드를 C2로 보내면 컴파일 스레드가 마비되고 시동이 기어간다. 그래서 **C1으로 일단 빠르게 기계어를 주면서 동시에 MDO에 프로파일을 모으고**, 정말 뜨거운 소수만 C2로 올린다. C2 컴파일 큐가 밀리면 L2를 경유하는 등 경로가 유연하게 바뀐다.

#### (2) MDO는 무엇을 담나 — C2의 원재료

MDO를 조금 더 풀면, C2가 왜 정적 컴파일러보다 유리한지가 보인다. MDO에는 대략 이런 것들이 바이트코드 지점별로 기록된다.

```
메서드 foo()의 MDO
├─ 호출 카운터 / 백엣지 카운터                (얼마나 뜨거운가 → 티어 승격 판단)
├─ 각 호출 지점의 "타입 프로파일"             (이 call site에 실제로 온 리시버 타입들)
│    예: line 42의 x.process() → 99.4% Cat, 0.6% Dog
├─ 각 분기의 taken / not-taken 빈도          (분기 예측 배치, 죽은 분기 판단)
└─ null 관찰 여부, 배열 길이 분포 등
```

정적 컴파일러(Rust/Go)는 이 정보를 **빌드 시점에 알 수 없다.** 트래픽이 실제로 어떤 타입을, 어떤 분기를, 얼마나 자주 때리는지는 프로그램을 돌려봐야 안다. C2는 MDO라는 형태로 그걸 손에 쥐고 컴파일한다. 아래 최적화들은 전부 "MDO를 읽었기에 가능한" 것들이다.

#### (3) 웜업 — 처리량이 계단으로 오른다

```
처리량
  ▲
높│                                 ┌────────────  ← L4 (C2) 정상상태
  │                          ┌──────┘  역가상화·벡터화·스칼라치환 적용
  │                   ┌──────┘             (여기서 AOT를 따라잡거나 앞섬)
  │            ┌──────┘  ← L3 (C1+프로파일)
낮│  ┌─────────┘  ← L0 인터프리터
  └──┴─────────────────────────────────────────▶ 시간
     ↑웜업 구간: 느리고, JIT 컴파일 스레드가 CPU를 추가 소모
     ↑ 이 구간의 요청은 tail latency가 나쁨 (콜드)

정적 언어(Rust/Go): t=0부터 ────────────────── 평평하게 높음 (웜업 없음)
```

웜업 비용은 세 겹이다. ⓐ 초기에 인터프리터/L3로 도는 구간이 느리고, ⓑ JIT 컴파일 스레드가 CPU를 훔쳐가며, ⓒ 코드 캐시가 채워지는 동안 성능이 계단식으로만 오른다. 서버리스·CLI처럼 **짧게 사는 프로세스는 이 계단을 다 오르기 전에 죽어** 손해만 본다.

#### (4) C2의 "돌연변이 최적화" — MDO가 있어야 가능한 것들

**① 공격적 인라이닝** — MDO의 호출 빈도가 "이 호출이 뜨겁다"고 알려주므로, C2는 확신을 갖고 깊게 인라인한다. 2.2에서 본 Go의 정적 예산(cost 80)이나 Rust의 정적 휴리스틱과 달리, C2는 **실측 빈도**로 결정한다.

**② 역가상화 + 인라인 캐시 상태기계** — Java에서 메서드는 기본이 `virtual`(오버라이드 가능)이라, 원래는 매 호출이 간접 호출이어야 한다. 이걸 걷어내는 장치가 **인라인 캐시(inline cache)** 다. 인라인 캐시는 "이 호출 지점이 지난번에 어떤 타입을 불렀는지"를 호출 자리에 캐싱하는 기법인데, 상태가 진화한다.

```
monomorphic(단형)  : 한 타입만 관찰됨 → 그 타입의 메서드로 "직접 호출" 캐싱. 가장 빠름.
        │  새로운 타입이 등장
        ▼
bimorphic(2형)     : 두 타입 → C2가 둘 다 인라인 + 타입 가드로 분기 (MDO 프로파일 기반)
        │  타입이 계속 늘어남
        ▼
megamorphic(다형)  : 너무 많은 타입 → 포기하고 vtable/itable 조회로 폴백 (간접 호출)
                     → 이 지점은 인라인 불가, BTB 예측도 흔들려 느림. "deopt storm"의 온상.
```

핵심은 C2가 MDO의 타입 프로파일을 읽어, "이 자리는 99% Cat"이면 `Cat.process()`를 **직접 호출로 치환하고 인라인**하며, 만약을 위한 **타입 가드**(아니면 uncommon trap으로 이탈)를 심는다는 것이다. 정적 언어에선 상상하기 어려운 최적화다 — 다형적으로 *선언된* 호출을 실행 실측으로 단형 호출로 바꾼다. (2.2 Go의 `itab` 간접 호출, 2.1 Rust의 `dyn` vtable과 같은 비용을, Java는 런타임에 깎는다.)

**③ 자동 벡터화 (SuperWord)** — C2에는 **SuperWord**라는 자동 벡터화 패스가 있다(`-XX:+UseSuperWord`, 기본 켜짐). 동작 원리는 이렇다: 루프를 몇 배로 언롤(풀어서 복제)한 뒤, 데이터 의존성을 분석해 **서로 독립인 동형(isomorphic) 스칼라 연산들을 하나의 SIMD 명령으로 묶는다.** 예컨대 `a[i]+b[i]`를 8번 반복하는 스칼라 루프를, `a[i..i+8] + b[i..i+8]`를 한 번에 처리하는 AVX 명령으로 바꾼다. 2.2에서 봤듯 **Go 컴파일러엔 없는 능력**이라, 수치 루프에서 Java는 Go보다 구조적으로 유리할 수 있다(웜업만 넘기면).

**④ 이스케이프 분석 + 스칼라 치환(scalar replacement)** — 2.2 Go의 이스케이프 분석과 같은 개념이지만, Java는 한 발 더 나간다. 탈출하지 않는 객체를 **아예 힙에 할당하지 않고, 그 객체의 필드들을 레지스터/스택 슬롯으로 분해**해 버린다(스칼라 치환). 객체가 통째로 증발한다.

```
before:  Point p = new Point(x, y);  int s = p.x + p.y;   // 힙 할당 1회
after:   int s = x + y;                                    // 할당 0, 객체 자체가 사라짐
```

덤으로 **락 생략(lock elision)**: 탈출하지 않는 객체에 걸린 `synchronized`는 다른 스레드가 볼 수 없으니 무의미하다 → 제거한다.

**⑤ 트랩 배치** — 차가운 분기는 `uncommon trap`(그 경로로 가면 역최적화로 이탈)으로 빼내, 핫 경로의 코드 밀도와 분기 예측을 개선한다.

#### (5) OSR (On-Stack Replacement) — 달리는 중에 엔진 교체

메서드 호출 카운터로는 못 잡는 경우가 있다 — **호출은 딱 한 번인데 그 안의 루프가 수억 번 도는** 코드(예: `main`에서 거대한 for 루프). 이건 호출 카운터가 안 늘어 티어 승격이 안 된다. 그래서 C2는 **백엣지 카운터**(루프가 한 바퀴 돌 때마다 증가)로 이를 감지하고, **실행 중인 스택 프레임을 컴파일된 코드로 교체**해 루프 한가운데로 진입시킨다. 말 그대로 "달리는 자동차의 엔진을 바꿔 끼우는" 최적화다.

#### (6) 역최적화(deoptimization) — 도박의 청구서, 그리고 어떻게 되짜나

C2의 최적화는 전부 **낙관적 도박**이다. "이 호출은 항상 Cat", "이 메서드는 오버라이드 없음"을 가정하고 인라인·역가상화한다. 그 가정이 깨지면(예: 새 서브클래스가 클래스로더에 올라와 가정 무효화, 또는 타입 가드가 빗나감) 그 기계어는 무효가 된다. 여기서 **물리적으로 어떻게 인터프리터로 되돌아가는가**가 핵심이다.

```
C2가 기계어를 만들 때, 함께 [scope descriptor(디버그 정보)]를 심어둔다:
   "이 기계어 지점 = 바이트코드 몇 번째 줄이고,
    지역변수 x는 레지스터 R8에, y는 스택 [SP+16]에 있다" 라는 지도.

역최적화 발생 시:
   낙관적 C2 프레임 ──[가정 위반]──▶ scope descriptor를 읽어
                                    → 그 지도대로 "인터프리터 프레임"을 재구성
                                    → 인터프리터가 그 지점부터 이어서 실행
                                    → 무효 기계어는 폐기, MDO 재수집 → 재컴파일
                                    ↑ 이 순간 그 요청은 tail latency 스파이크
```

즉 컴파일된 코드가 "자기가 원래 바이트코드 어디에서 왔는지"를 지도로 들고 다니기 때문에, 언제든 안전하게 인터프리터 상태로 되짤 수 있다. 역최적화는 C2가 공격적으로 도박할 수 있게 해주는 **안전망**인 동시에, **tail latency 불안정의 주된 원인**이다. 특히 megamorphic 호출 지점에서 반복 역최적화가 일어나는 **"deopt storm"** 은 Java 성능 사고의 단골이다.

#### (7) 세이프포인트(safepoint) — GC가 아닌데도 생기는 멈춤 ★숨은 tail 근원

여기까지 "역최적화 스파이크"를 말했는데, 그게 실제로 어떻게 스레드를 멈추는지가 **세이프포인트**다. 이건 Java tail latency를 이해하는 데 빠지면 안 되는 조각인데 자주 생략된다.

**세이프포인트란**, JVM이 *모든 스레드의 정확한 상태*(어떤 객체가 살아있고, 각 스택 프레임이 어디 있는지)를 아는 순간이다. GC, 역최적화, 스레드 덤프, 클래스 재정의 같은 작업은 **모든 스레드가 세이프포인트에 모여 멈춰 있어야만** 안전하게 할 수 있다. 힙을 옮기는 도중에 어떤 스레드가 옛 주소를 들고 뛰어다니면 안 되니까.

```
스레드들을 세이프포인트로 모으는 법:
  JIT이 메서드 반환 지점과 루프 백엣지마다 "세이프포인트 폴(poll)"을 심어둔다.
  각 스레드는 그 폴에서 "지금 멈춰야 하나?" 플래그를 확인 → 세워지면 정지.

     T1 ──poll──poll──[정지]
     T2 ──poll──────────[정지]
     T3 ──(긴 카운티드 루프, 폴이 최적화로 제거됨)──────── 아직 안 멈춤! ← 병목
         └────────── TTSP(Time-To-SafePoint) ──────────┘
                     이 시간만큼 "다 모일 때까지" 전원이 기다림
```

**TTSP(Time-To-Safepoint)** = 마지막 스레드까지 다 모이는 데 걸리는 시간인데, 이건 GC 일시정지 시간과 **별개**다. JIT이 성능을 위해 긴 카운티드 루프(메서드 호출 없이 도는 정수 루프)에서 세이프포인트 폴을 지워버리면, 그 스레드가 루프를 다 돌 때까지 다른 모든 스레드가 이미 멈춘 채 **수백 ms를 기다릴 수 있다.** `System.arraycopy`로 거대한 버퍼를 복사하는 중이거나, 코어 수보다 스레드가 훨씬 많을 때(oversubscription)도 TTSP가 길어진다.

핵심 함의: **Java의 tail latency 스파이크는 GC만의 문제가 아니다.** 역최적화도, 세이프포인트 도달 지연(TTSP)도 전부 멈춤을 만든다. 이것이 2.1 Rust의 "p99≈p50 결정론"과 대비되는, Java의 구조적 약점이다.

### ④ 회피 / 최적화 전략

#### 웜업을 다스리는 세 갈래

**(a) 웜업을 짧게 — C1에서 멈추기**
```bash
java -XX:TieredStopAtLevel=1 ...   # C2 포기. 웜업 빠름 / peak 낮음.
```
어차피 C2의 긴 계단을 다 못 오를 짧은 프로세스(CLI, 짧은 배치, 일부 서버리스)엔 이게 이득이다.

**(b) 웜업을 앞당기기 — Project Leyden (★JIT/AOT 경계 흐리기, 1부 복선 회수)**
Leyden의 발상은 "매번 실행할 때마다 처음부터 배우지 말고, **훈련 실행(training run)에서 배운 걸 미리 구워두자**"는 것이다.
```
JEP 483 (JDK 24) : AOT 클래스 로딩·링킹   — 클래스 로드/링크를 미리 완료해 캐시
JEP 514 (JDK 25) : AOT 캐시 원커맨드       — 예전엔 record+assemble 2단계였던 걸 하나로
                    -XX:AOTCacheOutput=app.aot
JEP 515 (JDK 25) : AOT 메서드 프로파일 ★   — MDO 프로파일을 미리 AOT 캐시에 저장
                    → JVM이 "시작하자마자" JIT이 기계어 생성을 시작 (프로파일을 이미 손에 쥐고)
                    → 예제 프로그램 시동 15~25% 단축 (JDK24 대비)
                    → 한 사례: 실행 73ms(19% 개선), 캐시는 +250KB(2.5%)만 증가
```
핵심 의미: **Java가 AOT 쪽으로 한 발 내디뎠다.** ③에서 본 "MDO를 실행하며 모으는" 과정을, JEP 515는 미리 구워 처음부터 갖고 시작하게 한다. JIT과 AOT는 이제 이분법이 아니라 **연속 스펙트럼**이다(2.4에서 정리).

**(c) 웜업을 없애기 — GraalVM native-image**
아예 바이트코드를 빌드 타임에 기계어로 뽑는다(closed-world AOT). 웜업 0, 시동 수 ms, RSS 대폭 감소. 대가로 **C2의 런타임 적응(MDO 기반 역가상화·프로파일 인라이닝)을 통째 포기**한다. 이걸 **PGO(빌드 타임 프로파일)** 로 부분 보상한다(2.1의 Rust PGO, 2.2의 Go PGO와 같은 계열) — 역시 2.4에서 수렴.

#### 코드 캐시 — 세그먼트 구조(JEP 197)를 알아야 고갈을 진단한다

코드 캐시는 하나의 덩어리가 아니라 **JDK 9(JEP 197)부터 3개 구역(code heap)** 으로 나뉜다.
```
┌ non-method   : 고정 3MB. 인터프리터·컴파일러 버퍼 등 VM 내부 코드.
├ profiled     : L2/L3 프로파일된 코드. 수명이 짧음(나중에 L4로 대체될 것).
└ non-profiled : L1/L4 코드. 잠재적으로 영원히 남음.
```
나눈 이유: 예전엔 sweeper(안 쓰는 코드 청소기)가 전체를 훑어야 했는데, 구역을 나누면 "절대 안 비우는 non-method 코드"를 건너뛰어 **스윕 시간이 짧아진다.** `-XX:ReservedCodeCacheSize`(기본 240MB)가 큰 것도 이 세그먼트 구조 + 티어드 컴파일을 수용하기 위함이다. **고갈되면 JIT이 멈추고 인터프리터로 추락**(peak 급락)하는 "조용한 성능 절벽"이 생기므로, 대형 앱·람다/제네릭 많은 코드에서 감시가 필수다.

#### 그 밖의 레버
- **인라이닝 튜닝**: `-XX:MaxInlineSize`, `-XX:FreqInlineSize`.
- **명시적 SIMD — Vector API**: JEP 508(JDK 25 기준 **10번째 인큐베이터**, 아직 preview 아님 — Project Valhalla의 값 클래스 대기 중). C2 자동 SuperWord 위에, 개발자가 벡터 폭을 명시 제어. 내부적으로 FFM API로 네이티브 수학 라이브러리에 연결.
- **관측**: JFR(JDK Flight Recorder)로 컴파일·역최적화·세이프포인트 이벤트, `-XX:+PrintCompilation`으로 티어 승격 로그, JITWatch로 인라인/벡터화 결정 시각화, 세이프포인트 로그(`-Xlog:safepoint`)로 TTSP 추적.

### ⑤ 세 언어 수렴 — 주제: "런타임 적응 최적화 vs 정적 스냅샷"

```
                     Java (2.3)               Rust (2.1)          Go (2.2)
──────────────────  ──────────────────────  ─────────────────  ─────────────────
기계어 생성 시점       실행 중 (JIT)            빌드 타임          빌드 타임
프로파일의 물리 장소    MDO (실행 중 기록)        PGO 프로파일 파일    default.pgo 파일
                                            (빌드 시 소비)      (빌드 시 소비)
역가상화             인라인캐시 mono→인라인+가드  단형화(정적)        PGO devirt(빌드1회)
자동 벡터화          ○ C2 SuperWord           △ LLVM 자동         ✗ (수동 asm)
탈출 객체            스칼라 치환(런타임 소거)     스택/힙 정적 결정    이스케이프분석(정적)
워크로드 변화 적응     ○ 자동 (MDO 재수집)       ✗ 재빌드           ✗ 재빌드
시동/웜업            ✗✗ 느림 (계단)           ○ 즉시             ○ 즉시
tail latency        ✗ 스파이크(deopt·세이프포인트) ★안정          안정
메모리(코드캐시+MDO)   ✗✗ 큼                    ★작음              작음
데이터 레이아웃        ✗ Object[]=참조 배치       ★ 연속(값 타입)      ★ []struct 연속
                    (Valhalla 프리뷰 대기)
아키 특화            런타임 실행CPU 감지→그 아키   빌드시 target-cpu   빌드시 GOARCH
```

**관통하는 통찰**: 축은 "백엔드 품질"이 아니라 **"프로파일을 언제 반영하는가"** 다. Rust/Go는 프로파일을 *빌드 시 한 번*(PGO 파일) 소비한다 — 그래서 **예측 가능**하지만 배포 후엔 **못 배운다.** Java는 프로파일을 *실행 내내* MDO에 모아 계속 반영한다 — 그래서 **적응**하지만 **예측 불가능**하고 무겁다. 그리고 이 이분법은 **양쪽에서 무너지고 있다.** Java는 Leyden으로 AOT 쪽에, Rust/Go/GraalVM은 PGO로 프로파일 쪽에 다가선다. 2.4가 이 수렴의 지형도를 그린다.

> **덧 — 메모리 레이아웃이라는 숨은 축 (2.2 ③-5의 뒷면).** Java 성능의 또 다른 구조적 특징은 객체 배열이 **참조의 배열**이라는 점이다. `Point[]`은 각 `Point`가 헤더(마크워드 + 클래스 포인터, 대략 12~16B)를 달고 힙 여기저기 흩어진 것을 가리키는 포인터의 배열이다. 순회하면 포인터를 따라다니며(pointer chasing) 캐시 미스가 잦아 하드웨어 프리패처가 무력화된다 — Go의 `[]struct` 연속 배치(2.2)와 정반대다. 이는 Java가 "모든 객체에 정체성(identity)·헤더·간접 참조를 부여한" 설계의 대가다. (단 `int[]`·`long[]` 등 원시 배열은 Java도 연속이다 — 문제는 *객체*의 배열이다.) **Project Valhalla(JEP 401 값 클래스)** 가 배열 평탄화(heap flattening)와 JIT 스칼라화로 이 격차를 좁히려 하지만, **JDK 25 기준 아직 프리뷰**이며 메인라인 통합은 JDK 28 목표다 — 즉 2부의 기준 시점에서 이 격차는 여전히 실재한다. 이 점은 2.3에서 본 "웜업만 넘기면 C2 SuperWord로 Go를 앞선다"를 **다시 절반쯤 되돌려 놓는다**: 수치 커널이 메모리 바운드라면, 벡터화 우위가 캐시 미스로 상쇄될 수 있다.

### ⑥ 트레이드오프 소결

Java JIT의 계약:

> **가장 느린 시동(웜업)과 가장 큰 메모리(코드 캐시 + MDO + JIT 자료구조)를 지불하는 대신, 실행 실측(MDO)에 근거한 최적화로 정적 컴파일이 원리적으로 놓치는 것을 잡는다.**

- **얻는 것**: 충분히 오래 달리는 CPU 바운드 서버에서 **peak throughput**이 정적 언어를 따라잡거나 앞선다. 다형 호출을 실측으로 단형화하고, 스칼라 루프를 벡터화하며, 탈출 객체를 통째 소거하고, 트래픽 변화에 재적응한다.
- **잃는 것**: **웜업**(콜드 스타트 손해), **tail latency**(역최적화·세이프포인트/TTSP 스파이크), **메모리**(코드 캐시·MDO). 이 셋은 여전히 정적 언어에 열세다.

그래서 1부 1.3의 반전 — "가장 무거운 JVM이 왜 롱런 peak에서 앞서나" — 의 답은 **"MDO라는 실행 실측을 C2가 기계어에 녹여 넣기 때문"** 이다. 단, 이 "역전"은 **무조건이 아니라 조건부**다. 그 조건을 정밀하게 규정하는 것이 2.4다.

### ⑦ DevOps 관점 심화 질문

1. **웜업과 배포 안전성**: 롤링 배포 중 갓 뜬 콜드 JVM에 트래픽이 쏠리면 웜업 구간의 나쁜 tail이 사용자에게 노출된다. K8s `readinessProbe`를 "웜업 완료"에 맞추려면 무엇을 신호로 쓸까(요청 N건 처리? `PrintCompilation`으로 C2 수렴 확인?)? 배포 전 트래픽 미러링으로 코드 캐시·MDO를 데우는 파이프라인 설계는?

2. **역최적화·세이프포인트 폭풍 탐지**: JFR의 `Deoptimization` 이벤트와 `-Xlog:safepoint`의 TTSP를 상시 관측하려면? 어떤 코드 패턴(megamorphic 호출, 긴 카운티드 루프, 대형 `arraycopy`)이 유발하며, 그것을 코드 리뷰/정적 분석에서 선제 차단하는 방법은?

3. **코드 캐시 세그먼트 모니터링**: `ReservedCodeCacheSize` 고갈은 "조용한 절벽"이다. JMX `MemoryPoolMXBean`으로 non-profiled/profiled 각 세그먼트를 따로 보고, 고갈 징후(컴파일 중단 로그, throughput 급락)를 대시보드에 엮는 설계는?

4. **Leyden AOT 캐시의 CI 파이프라인**: JEP 515 AOT 프로파일을 구우려면 **대표성 있는 훈련 실행**이 필요하다. 프로덕션 트래픽을 재현하는 training run을 CI에서 어떻게 구성하고, 코드가 바뀔 때 캐시를 무효화·재생성하나? 훈련 프로파일이 실트래픽과 어긋날 때의 위험은?

5. **HPA 왜곡**: JIT 컴파일 스레드는 웜업 동안 CPU를 추가 소모한다. CPU 기반 HorizontalPodAutoscaler가 이 일시적 스파이크를 "부하 증가"로 오인해 불필요한 스케일아웃을 하지 않게 하려면? 워밍업 CPU를 어떻게 배제/평활하나?

6. **티어 정책과 수명 매칭**: 배치잡(짧음)과 상시 API(긺)는 최적 티어 정책이 다르다. `TieredStopAtLevel=1`(빠른 웜업) vs 풀 티어(높은 peak) vs Leyden vs native-image(웜업 0)를 **프로세스 수명·트래픽 패턴**으로 어떻게 결정하는 의사결정 트리를 세울까?

### ⑧ 다음 절 연결

2.3은 "Java가 롱런 peak에서 앞설 수 있는 **메커니즘**"을 MDO부터 세이프포인트까지 해부했다. 하지만 "앞선다"를 무조건으로 받아들이면 틀린다. 2.4(소결)는 이 **역전이 실제로 성립하는 조건**(장수명·CPU 바운드·다형성 많은 코드·peak throughput 지표)과, **여전히 성립하지 않는 축**(tail latency·메모리·콜드 스타트)을 정직하게 갈라친다. 그리고 Leyden·GraalVM PGO가 이 JIT vs AOT 구도를 어떻게 **연속 스펙트럼**으로 재편하는지 — 2부를 닫고 3부(메모리 할당 밑바닥, MDO와 코드 캐시가 사는 그 힙의 아래층)로 넘기는 다리를 놓는다.
