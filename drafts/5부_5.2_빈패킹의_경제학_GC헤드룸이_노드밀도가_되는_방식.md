## ① 이 절에서 도달할 지점

5.1에서 "런타임이 요구하는 메모리 여유(headroom)"라는 말을 여러 번 썼습니다. Go는 GOGC=100이
만드는 라이브의 2배, ZGC는 동시 수집을 위한 빈 page, G1은 이사(evacuation)용 예비 region.
5.2의 목표는 이 여유가 **어떻게 돈이 되는지**를 끝까지 따라가는 것입니다.

> **"GC가 편하자고 남겨 둔 메모리 여유는, 파드의 `requests`/`limits`가 되고, 그 숫자가 노드에
> 파드가 몇 개 들어가는지를 정하고, 그게 노드 수를, 노드 수가 청구서를 정한다."**

이 절을 다 읽으면 다음을 설명할 수 있습니다.

- 쿠버네티스 스케줄러가 노드에 파드를 채우는 기준이 **실제 사용량이 아니라 `requests`**라는 것,
  그리고 왜 이게 빈패킹의 전부인지.
- 노드의 물리 메모리 중 파드가 실제로 나눠 갖는 몫(Allocatable)이 왜 물리값보다 작은지.
- 같은 워크로드(같은 라이브 집합)를 Rust·Go·Java(G1/ZGC)로 담았을 때, **컨테이너 상한이
  왜 서로 다른 크기의 상자**가 되는지, 그 상자 크기가 노드당 밀도를 어떻게 가르는지.
- GC 헤드룸을 줄여 상자를 압축하는 대가는 무엇인지(공짜가 아니다).

4부의 트레이드오프 삼각형(단편화 ↔ 이동 비용 ↔ 회계의 위치)이 여기서 **밀도 ↔ 안전마진 ↔
지연**이라는 클라우드 경제학의 삼각형으로 번역됩니다.

## ② 빈패킹의 물리 — 스케줄러는 '사용량'이 아니라 '요청'으로 상자를 채운다

### 빈패킹이란 무엇인가 (정의부터)

**빈패킹(bin packing)**은 크기가 정해진 상자(bin, 여기선 노드)에 물건(파드)을 되도록 빈틈없이
채워 상자 수를 최소화하는 문제입니다. 트럭(노드)에 짐(파드)을 실을 때, 짐을 적게 실으면 트럭이
많이 필요하고(비용↑), 빽빽이 실으면 트럭이 적게 듭니다(비용↓). 클라우드에서 트럭 한 대 = 노드
한 대 = 시간당 요금이므로, **빈패킹 밀도는 그대로 비용**입니다.

### 스케줄러가 보는 건 `requests`다 — 이것이 핵심

쿠버네티스 스케줄러가 "이 파드를 어느 노드에 놓을까"를 정할 때 참조하는 짐의 크기는 **파드가
선언한 `requests`이지, 실제 사용량이 아닙니다.** 2 CPU를 요청하고 실제로는 200m만 쓰는 파드도,
스케줄러 눈에는 그냥 "2 CPU짜리 짐"입니다. 메모리도 똑같습니다.

```
노드 (트럭)                             파드 (짐 — 크기는 requests로 '선언'된 값)
┌──────────────────────────────┐        ┌────┐ requests.mem = 1Gi (실사용 300Mi여도
│  Allocatable = 30 Gi          │        └────┘  스케줄러엔 1Gi짜리 짐)
│  ┌──────┐┌──────┐┌──────┐     │
│  │ 1Gi  ││ 1Gi  ││ 1Gi  │ ... │  ◀── 스케줄러는 Σrequests ≤ Allocatable 인 동안 채움
│  └──────┘└──────┘└──────┘     │
│  남는 공간이 다음 짐보다 작으면│
│  → 이 노드엔 더 못 실음 → 새 노드 필요
└──────────────────────────────┘
```

그래서 `requests`를 크게 잡으면 실제로는 놀고 있는 메모리라도 스케줄러는 "찼다"고 보고 다음
파드를 다른 노드로 보냅니다. **요청과 실사용의 간극이 곧 낭비된 노드**입니다. (출처: 쿠버네티스
공식 문서 "Resource Bin Packing", 그리고 스케줄러가 requests로만 판단한다는 점은 여러 비용
최적화 사례의 공통 출발점.)

### 노드의 몫: Allocatable은 물리값이 아니다

한 가지 더. 파드가 나눠 갖는 건 노드의 물리 메모리 전부가 아니라 **Allocatable**입니다.

```
노드 물리 메모리 (Capacity)
 ├─ system-reserved     OS·커널·sshd 등 호스트 몫
 ├─ kube-reserved       kubelet·컨테이너 런타임(containerd) 몫
 ├─ eviction-threshold  축출 방어용 여유 (기본 memory.available<100Mi 등)
 └─ Allocatable    ◀── 파드들이 requests로 나눠 갖는 실제 파이
```

즉 32GiB 노드라고 32GiB를 파드에 못 줍니다. 통상 몇 백 MiB~1GiB가 먼저 빠지고, 남은
Allocatable에 파드를 채웁니다. 노드가 작을수록 이 고정 상납분의 비율이 커져(작은 노드 여러 대
vs 큰 노드 몇 대) 밀도가 나빠집니다 — 노드 크기 선택 자체도 경제학입니다.

### 압축 불가 자원의 재확인

5.1에서 본 비대칭이 빈패킹에서 이렇게 작동합니다. CPU는 `requests`로 스케줄하되 실사용이
넘치면 스로틀로 버티므로 **오버커밋(요청보다 많이 얹기)**이 관대합니다. 메모리는 넘치면
OOMKilled이므로 오버커밋이 위험합니다. 그래서 "메모리 requests를 얼마나 촘촘히 잡느냐"가
빈패킹 밀도와 안정성을 동시에 쥔 손잡이가 됩니다.

## ③ 비용 해부 — GC 헤드룸이 '상자 크기'를 부풀린다

같은 라이브 집합(실제로 살아 있는 객체의 총량)을 담더라도, 어떤 런타임·GC를 쓰느냐에 따라
필요한 컨테이너 상한이 달라집니다. 그 차액이 전부 **GC/런타임 오버헤드**입니다.

### Go: GOGC=100이 만드는 '라이브의 2배'

4부에서 봤듯 Go의 기본 GC는 "직전 수집 후 힙이 라이브의 2배로 자라면 다음 수집"입니다
(`GOGC=100`). 즉 라이브가 300MiB면 힙은 최대 ~600MiB까지 자랐다가 수집됩니다. `GOMEMLIMIT`을
안 주면(5.1) 이 2배가 상자의 기본 크기입니다. 여기에 힙 밖(고루틴 스택, `mmap`, 런타임 자체)이
얹힙니다. 상자를 줄이려면 `GOMEMLIMIT`을 라이브 근처로 당기면 되지만, 그만큼 GC가 자주 돌아
CPU를 더 씁니다 — 4부의 "메모리 여유 ↔ GC 빈도" 교환이 그대로 상자 크기 ↔ CPU로 나타납니다.

### Java G1: 힙 안 예비 + 회계 구조 + 힙 밖

G1의 헤드룸은 여러 겹입니다(측정: OpenJDK 11.0.31, 4 CPU 기준 기본값).

- **`G1ReservePercent = 10`** — 힙의 10%를 "예비"로 비워 둡니다. 이사(evacuation) 도중 대상
  공간이 모자라 실패(promotion/evacuation failure)하는 걸 막기 위한 완충입니다. 즉 힙의 90%만
  실사용에 쓴다고 봐야 합니다.
- **카드 테이블(card table)** — 힙 512바이트당 1바이트, 곧 힙의 약 **0.2%**를 세대 간 참조
  추적용으로 상시 점유합니다(고정·정확값).
- **Remembered Set(RSet)** — region 간 참조를 기록하는 자료구조로, 포인터가 많은 워크로드에선
  상당히 커질 수 있습니다(워크로드 의존이라 고정 수치로 못 박지 않습니다).
- **힙 밖(off-heap)** — Metaspace, 코드 캐시, 스레드 스택, 다이렉트 버퍼, malloc arena. 이건
  `-Xmx`로 조절되지 않으며 5.3의 주제입니다.

또 하나, region 크기는 **상수가 아니라 힙 크기에서 파생**됩니다(측정: JDK 11.0.31).

| `-Xmx` | 측정된 `G1HeapRegionSize` |
|---|---|
| 256m ~ 4g | 1 MB |
| 16g | 4 MB |
| 32g | 8 MB |
| 64g | 16 MB |

설계 의도는 "region 수를 대략 수천 개 수준으로 유지"하는 것이고, 값은 1MB~32MB로 클램프됩니다.
**핵심은 이게 코어 수·힙 크기에서 파생되는 값이지 고정 상수가 아니라는 점**입니다(버전에 따라
경계도 달라질 수 있으니 기준 JDK에서 `-XX:+PrintFlagsFinal`로 확인 전제).

### Java ZGC: 가장 큰 상자, 대신 가장 낮은 지연

Generational ZGC는 셋 중 상자가 가장 큽니다. 이유가 뚜렷합니다(출처: Oracle JDK 25 GC 튜닝
가이드, ZGC 항목).

- **고정 오버헤드 ~3%** — ZGC는 힙 크기의 약 3%를 자체 메타데이터로 상시 점유하며, G1보다
  네이티브 메모리를 더 씁니다.
- **compressed oops 미지원** — ZGC는 colored pointer로 주소 비트를 쓰기 때문에 압축 참조를
  못 씁니다. 3.4에서 세운 "32GB 절벽"이 여기서 갈립니다: **32GiB 이하 힙에서 G1은 객체 참조를
  4바이트로 압축하지만, ZGC는 8바이트를 씁니다.** 포인터가 많은 객체 그래프일수록 이 차이가
  힙 실사용을 키웁니다.
- **동시 수집 헤드룸** — ZGC는 뮤테이터를 안 멈추고 이사하므로, GC가 도는 동안에도 들어오는
  할당을 받아 줄 **빈 공간**이 힙 안에 있어야 합니다. 얼마나 필요한지는 "할당률과 라이브
  집합에 따라 크게 다르다"가 공식 입장입니다. 이 여유가 부족하면 4부에서 예고한 그
  allocation stall이 납니다.

정리하면 ZGC의 상자 = 라이브 + 3% 고정 + (압축 없는) 참조 오버헤드 + 동시 수집 여유. 지연을
sub-ms로 사는 대가를 **메모리 footprint와 처리량**으로 치릅니다(4부 삼각형 그대로).

### 같은 라이브 집합, 서로 다른 상자 (개념도)

```
같은 라이브 집합 300MB를 담는 데 필요한 컨테이너 상한 (개념 설명용 예시 — 측정값 아님)

Rust        │████████░│            라이브 + 할당자 단편화 여유(작음), GC 헤드룸 0
Go(MEMLIMIT)│██████████░░│         GOMEMLIMIT로 헤드룸 압축 (대신 GC 빈도↑)
Java G1     │████████████░░░│      힙(라이브+예비10%+RSet) + 힙밖(Metaspace·코드캐시…)
Go(GOGC100) │████████████████░░│   라이브 2× (GOMEMLIMIT 미설정 시 기본)
Java ZGC    │██████████████░░░░│   +3% 고정 + 압축참조 없음 + 동시수집 여유
            └─ 라이브 ─┘└ 헤드룸/오버헤드 ┘
  * 막대 길이는 개념 설명용 예시이며 실제 측정값이 아닙니다. 상대적 '경향'만 읽어 주세요.
    (Go 2×는 GOGC=100, G1 예비 10%, ZGC 3%+압축참조 없음은 출처 있는 사실이나,
     절대 MB는 워크로드·버전·튜닝에 따라 크게 달라집니다.)
```

## ④ 회피/최적화 — 상자를 줄이는 법과 그 대가

### 상자 크기(런타임 쪽)를 줄이기

- **Go**: `GOMEMLIMIT`을 라이브+여유로 당겨 2배 상자를 압축. 대가는 GC CPU 증가(과하게 당기면
  4부의 GC 데스 스파이럴). `GOGC`와 함께 써서 평상시 여유·한도 근처 방어를 병행.
- **Java**: `MaxRAMPercentage`를 워크로드에 맞게(기본 25%는 대개 과소). 힙 32GiB 절벽을
  의식해, 압축 참조가 유리한 워크로드는 힙을 **32GiB 밑**에 두는 것만으로 실사용이 줄어듭니다
  (G1 한정 — ZGC엔 이 이득이 없음).
- **GC 선택 자체가 상자 결정**: 지연이 절박하지 않다면 G1/Parallel이 ZGC보다 상자가 작습니다.
  밀도(비용)를 사려면 지연을 조금 내주는 식.

### 상자(requests/limits) 잡는 법과 밀도의 교환

- **`requests` = `limits` (Guaranteed)**: 스케줄러가 본 크기 = 커널이 죽이는 선. 밀도는
  보수적이지만 노드 레벨 경합이 없어 안전. 재현성이 중요한 서비스에 적합.
- **`requests` < `limits` (Burstable, 오버커밋)**: 스케줄러엔 작게 보여 **더 촘촘히** 채워지지만,
  여러 파드가 동시에 limits로 치달으면 노드 메모리가 터져 **누군가 OOMKilled**. 이때 축출·킬
  우선순위는 QoS class(Guaranteed > Burstable > BestEffort)를 따릅니다. 밀도를 사는 대신
  안정성을 저당 잡는 셈.

### 스케줄러에게 "빽빽이 채워"라고 말하기

기본 스케줄러 점수 전략은 **`LeastAllocated`**로, 파드를 **분산**시킵니다(가용 자원이 많은
노드를 선호). 빈패킹을 원하면 `NodeResourcesFit` 플러그인의 **`MostAllocated`** 또는
**`RequestedToCapacityRatio`**를 스케줄러 프로파일에 명시해야 합니다. 즉 "촘촘히"는 기본이
아니라 선택입니다(출처: 쿠버네티스 "Resource Bin Packing" 문서). 클러스터 오토스케일러/
Karpenter로 노드를 사후 통합(consolidation)하는 접근도 같은 목표(상자 수 최소화)입니다. 실제로
파드 빈패킹으로 상당한 비용을 절감했다는 사례들이 보고돼 있습니다(예: ClickHouse의 EKS 빈패킹
사례 — 구체 수치는 원문 참조).

## ⑤ 세 언어 수렴 — 같은 워크로드, 다른 밀도

라이브 집합이 같은 서비스를 세 언어로 담았을 때, 상자 크기의 경향과 그 결과는 이렇습니다.

| | 상자를 키우는 요인 | 상자 경향 | 노드당 밀도 | 무엇을 사고 무엇을 파나 |
|---|---|---|---|---|
| **Rust** | GC 없음. 할당자 단편화 여유 정도 | 가장 작음 | 가장 촘촘 | 밀도·예측가능성을 얻고, 개발 비용(소유권)을 냄 |
| **Go** | GOGC=100의 2×(또는 GOMEMLIMIT 여유) + 스택 | 중간 | 중간 | 단순함을 얻고, 헤드룸(또는 GC CPU)을 냄 |
| **Java G1** | 힙 예비 10% + RSet + 힙밖(Metaspace 등) | 중간~큼 | 중간~성김 | 성숙한 처리량을 얻고, 힙밖 여백을 냄 |
| **Java ZGC** | 3% 고정 + 압축참조 없음 + 동시수집 여유 | 가장 큼 | 가장 성김 | sub-ms 지연을 얻고, 밀도·처리량을 냄 |

세로로 읽으면 **"밀도는 곧 무엇을 포기했느냐의 기록"**입니다. Rust는 런타임 여유가 없어 가장
빽빽하지만 그 대가를 컴파일 타임과 개발자 시간에 미리 냈습니다. ZGC는 지연을 극한까지 낮춘
대신 상자가 커져 노드를 더 씁니다. Go는 그 사이에서 손잡이(`GOMEMLIMIT`·`GOGC`) 하나로 밀도와
CPU를 맞바꿉니다.

중요한 건 **"어느 언어가 싸다"가 아니라 "무엇을 최적화하는 클러스터냐"**입니다. 지연 SLO가
빡빡한 결제 API라면 ZGC의 성긴 밀도가 정당한 비용이고, 배치 처리라면 그 여유가 순수 낭비입니다.
5.5에서 이걸 의사결정 표로 정리합니다.

## ⑥ 트레이드오프 소결

한 문장: **GC가 요구하는 메모리 여유는 파드 상자의 크기가 되고, 상자 크기가 노드 밀도를, 밀도가
청구서를 정한다.** 4부의 삼각형이 클라우드에서 이렇게 번역됩니다.

```
        지연 (tail latency)
             ▲
            ╱ ╲     ZGC: 지연↓ 사려고 밀도·처리량을 냄
           ╱   ╲
   밀도 ◀──╱─────╲──▶ 안전마진
 (노드 비용)          (오버커밋 안 할수록 안전, 대신 성김)
   Rust: 밀도 최상      Guaranteed(req=lim): 안전 최상, 밀도 보수
```

밀도를 높이는 세 손잡이 — (1) GC 헤드룸 압축(`GOMEMLIMIT`·GC 선택), (2) 오버커밋(req<lim),
(3) 빈패킹 스케줄러 — 은 각각 CPU·안정성·복잡도를 대가로 요구합니다. 공짜 밀도는 없습니다.

그리고 이 모든 계산의 밑에는 5.1의 전제가 깔려 있습니다. **상자 크기를 아무리 잘 잡아도, 커널이
세는 건 힙만이 아니므로 힙 밖 여백이 부족하면 상자 안에서 파드가 죽습니다.** 그 순간을 5.3에서
해부합니다.

## ⑦ DevOps 관점 심화 질문

1. **requests 산정의 근거** — 메모리 `requests`를 P50 사용량으로 잡을지, P99로 잡을지, GC
   헤드룸을 포함한 최대 힙으로 잡을지에 따라 밀도와 OOM 위험이 어떻게 갈립니까? Go에서
   `GOMEMLIMIT`을 정한 뒤 `requests`를 그 값 기준으로 잡는 순서가 왜 자연스럽습니까?

2. **오버커밋 비율 결정** — 메모리 `requests:limits`를 1:1(Guaranteed)로 둘 서비스와 1:1.5로
   둘 서비스를 무엇으로 나누겠습니까? 노드에 Burstable 파드가 몰렸을 때 동시에 limits로 튀는
   "상관된 스파이크"를 어떤 신호(HPA 이벤트·배치 시각)로 예측합니까?

3. **노드 크기 선택** — 같은 총 vCPU/메모리를 작은 노드 여러 대 vs 큰 노드 몇 대로 살 때,
   `system-reserved`/`kube-reserved` 고정 상납분과 빈패킹 파편(마지막 짐이 안 들어가는 낭비)이
   각각 어떻게 달라집니까? 큰 노드의 장단점을 Allocatable 관점에서 설명해 보십시오.

4. **32GiB 절벽과 힙 사이징** — G1을 쓰는 서비스의 힙을 30GiB에서 40GiB로 올렸더니 실사용이
   비선형으로 늘었습니다. compressed oops 관점에서 무슨 일이 일어난 겁니까? 같은 상황에서
   ZGC라면 왜 이 절벽이 없습니까?

5. **스케줄러 프로파일 vs 오토스케일러** — 밀도를 높이는 데 `MostAllocated` 스케줄링과
   Karpenter/Cluster Autoscaler의 노드 통합(consolidation) 중 무엇을 먼저 손대겠습니까? 둘을
   같이 쓸 때 파드가 자주 재배치되며 생기는 부작용(캐시 콜드·연결 리셋)은 어떻게 완화합니까?

6. **밀도 회귀 감시** — 배포 후 "노드당 파드 수가 줄었다(밀도 회귀)"를 어떤 지표로 잡겠습니까?
   `kube_pod_resource_requests` 합계 대비 Allocatable, 노드별 파편률, GC 헤드룸 증가(힙
   설정 변경) 중 무엇을 대시보드에 올리겠습니까?
