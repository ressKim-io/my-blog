## ① 이 절에서 도달할 지점

5부의 네 절은 각각 하나의 축을 세웠습니다. 5.5의 목표는 이 넷을 4부의 GC 트레이드오프 삼각형과
포개, **한 장짜리 의사결정 그림**으로 만드는 것입니다.

> **"메모리 안전과 성능의 비용은 사라지지 않는다. 다만 누가(사람/컴파일러/런타임) 회계를
> 맡느냐에 따라, 그 비용이 클라우드에서 밀도·안전마진·콜드스타트·처리량 중 어느 청구서로
> 날아올지가 달라질 뿐이다."**

이 절을 다 읽으면 다음을 할 수 있습니다.

- 5.1~5.4의 네 축(회계·밀도·안전마진·시작 구간)을 하나의 프레임으로 설명하기.
- 3.5·4.5에서 세운 **비용 보존 법칙**을 클라우드 축까지 확장한 표로 읽기.
- "어떤 워크로드에 무엇을(Rust/Go/Java-G1/ZGC/native-image)"을 **시나리오별로** 고르고 그
  근거를 대기.
- 왜 "은탄환은 없다"가 GC뿐 아니라 클라우드 전체에서 참인지 말하기.

## ② 5부의 네 축 재조립

```
                        [ 회계의 위치 ]  ← 4부의 세 번째 꼭짓점
                     사람 · 컴파일러 · 런타임
                              │
        이 선택이 아래 네 개의 클라우드 청구서를 동시에 결정한다
                              │
   ┌──────────────┬──────────────┬──────────────┬──────────────┐
   ▼              ▼              ▼              ▼
5.1 회계        5.2 밀도        5.3 안전마진     5.4 시작 구간
커널은 전부     GC 헤드룸이     힙 밖 여백이     웜업 세금이
합산한다        노드 비용       없으면 죽음      오토스케일·서버리스
   │              │              │              │
memory.max    requests/limits  off-heap        콜드스타트/
힙+네이티브    → 노드당 밀도     OOMKilled       JIT 웜업
```

네 축을 관통하는 하나의 문장은 **"런타임에 회계를 위임할수록 개발은 편하지만, 런타임이 세지
않는 곳(힙 밖·시작 구간)이 사각지대가 되고, 런타임이 요구하는 여유가 곧 밀도·비용이 된다"**
입니다. Rust는 회계를 사람/컴파일러에 두어 사각지대가 좁은 대신 개발 비용을 앞당겨 냈고, Java는
런타임에 크게 위임해 편한 대신 힙 밖·웜업이라는 청구서를 받습니다. Go는 그 중간에서 손잡이
몇 개(`GOMEMLIMIT`·`GOGC`)로 위치를 조절합니다.

## ③ 비용 보존 법칙의 클라우드 확장

3.5·4.5의 비용 보존 표를 클라우드 축까지 늘리면 이렇게 됩니다. **각 칸은 "그 언어/GC가 이
비용을 어디로 옮겨 냈는가"**입니다(값은 5.1~5.4·4부에서 출처와 함께 확인한 것).

| 축 | Rust | Go (마크-스윕) | Java G1 (이동·세대) | Java ZGC (동시 이동) | Java + native-image |
|---|---|---|---|---|---|
| **회계 위치** | 사람/컴파일러 | 런타임(비이동) | 런타임(이동) | 런타임(동시 이동) | 컴파일러(빌드타임) |
| **메모리 헤드룸(밀도)** | 없음 — 가장 촘촘 | 라이브 ~2×(GOGC=100) | 힙+예비10%+RSet | +3% 고정·압축참조 없음 → 가장 큼 | 작음(네이티브) |
| **off-heap 위험(5.3)** | 낮음(명시적) | 중간(cgo·스택) | 높음(Metaspace 등 넓음) | 높음 | 낮음 |
| **콜드스타트/웜업(5.4)** | 없음(태생) | 없음(태생) | 있음(C2까지 ~5000회) | 있음 | 없음(닫힌 세계) |
| **피크 처리량** | 최상 | 좋음 | 최상(롱런) | 높음(배리어 세금) | 8~25%↓(보고) |
| **지연(tail)** | 예측가능·낮음 | Mark Assist로 튐 | 대체로 안정·가끔 Full GC | sub-ms | 낮음(웜업 없음) |
| **컨테이너 자동 인식** | 앱/라이브러리 | CPU만(1.25+) | CPU·메모리 둘 다 | CPU·메모리 둘 다 | 해당 없음(네이티브) |
| **주된 클라우드 청구서** | 개발 시간 | GC CPU 또는 헤드룸 | 힙 밖 여백·웜업 | 메모리 footprint·처리량 | 피크 손해·빌드/리플렉션 |

표를 세로로 읽으면 **"공짜 칸이 없다"**가 보입니다. 어느 열도 모든 행에서 이기지 못합니다.
Rust는 밀도·시작·안전마진을 다 가져가지만 개발 시간을 앞당겨 냈고, ZGC는 지연을 극한으로
낮추는 대신 밀도·처리량을 냈으며, native-image는 시작을 사는 대신 피크와 동적성을 냈습니다.
**비용은 이동할 뿐 사라지지 않습니다.**

## ④ 의사결정 매트릭스 — 언제 무엇을

"무엇이 제일 좋냐"는 질문은 틀렸습니다. 옳은 질문은 **"이 클러스터가 무엇을 최적화하는가"**
입니다. 워크로드 성격별로 정리하면(권고이며 워크로드에 따라 다름, 절대 규칙 아님):

| 워크로드 성격 | 지배적 비용 | 우선 고려 | 근거(5부 축) |
|---|---|---|---|
| scale-to-zero 서버리스·이벤트 함수 | 콜드스타트 | Rust · Go · Java(native-image/SnapStart) | 5.4 — 자주 차게 시작. 기본 JVM은 불리 |
| 며칠씩 도는 고TPS API | 피크 처리량 | Java(JIT, G1/ZGC) · Go · Rust | 5.4/4부 — 롱런에서 JIT 피크가 유리 |
| sub-ms tail SLO(결제·광고입찰) | 지연 | Java ZGC · Rust · (Go는 Mark Assist 주의) | 4부/5.2 — 지연 위해 밀도·처리량 지불 |
| 메모리 밀도·비용 극한(멀티테넌시) | 노드 비용 | Rust · Go(`GOMEMLIMIT`) · Java(튜닝) | 5.2 — 상자가 작을수록 촘촘 |
| 빠른 개발·넓은 생태계 | 인력·시간 | Java · Go · (Rust는 학습곡선) | 회계를 런타임에 위임 = 개발 편의 |
| 운영 단순성(컨테이너 자동인식) | 오설정 사고 | Java(양쪽 자동) · Go(CPU만, 메모리 수동) | 5.1 — `GOMEMLIMIT` 수동이 흔한 함정 |

여기엔 4부·5부를 관통하는 리듬이 있습니다. **콜드 스타트가 잦으면 AOT(Rust/Go/native-image)로
기울고, 오래 돌수록 JIT(기본 JVM)로 기웁니다.** 그리고 **지연이 절박하면 밀도·처리량을 내주고
(ZGC), 밀도가 절박하면 지연·개발시간을 내줍니다(Rust/Go).** 어느 쪽도 전부를 가질 수 없습니다.

```
                     피크 처리량 / 롱런
                          ▲
              기본 JVM ───┤─── Rust
              (JIT 웜업 대가) │   (밀도·시작·안전 다 얻고 개발시간 냄)
                          │
   콜드스타트 빠름 ◀───────┼───────▶ 메모리 밀도 높음
   native-image           │           Go(GOMEMLIMIT)
   (피크·동적성 냄)         │
                          ▼
                      낮은 tail 지연
                      Java ZGC (밀도·처리량 냄)
```

## ⑤ "은탄환은 없다"의 클라우드 판

4.5가 "GC는 은탄환이 아니다"를 GC 삼각형으로 증명했다면, 5부는 그 명제가 **클라우드에서도
참**임을 보였습니다. 관리형 런타임(GC·JIT)은 개발 편의를 주지만 그 편의는 **힙 밖 사각지대·GC
헤드룸·웜업 세금**이라는 형태로 컨테이너 청구서에 되돌아옵니다. 반대로 그 청구서를 지우려는
처방(수동 튜닝, native-image, Rust)은 **개발 비용·피크 손해·동적성 제약**으로 대가를 옮길 뿐입니다.

그래서 실무의 성숙도는 "어떤 언어가 최고냐"가 아니라 **"우리 워크로드의 지배적 비용이
무엇이고, 그 비용을 감당 가능한 다른 비용으로 옮겼는가"**를 아는 데 있습니다. 5부는 그 판단에
필요한 계량 도구(cgroup `memory.stat`·NMT·`gctrace`·티어드 임계값·requests/limits)를 손에
쥐여 주는 것으로 제 몫을 다합니다.

## ⑥ DevOps 관점 심화 질문

1. **지배적 비용 식별** — 담당 서비스의 "지배적 비용"(콜드스타트/피크/지연/밀도/인력) 하나를
   고른다면 무엇이며, 그걸 어떤 지표로 증명하겠습니까? 그 선택이 언어·GC·튜닝을 어떻게
   좁힙니까?

2. **비용 이동의 가시화** — GC 헤드룸을 줄여 밀도를 높이면 GC CPU가 오릅니다(비용 이동). 이
   맞교환을 한 대시보드(밀도 지표 ↔ GC CPU/지연 지표)에서 동시에 보이게 하려면 무엇을
   그리겠습니까?

3. **혼합 클러스터의 원칙** — 한 클러스터에 서버리스형·롱런형·지연민감형 워크로드가 섞여
   있습니다. 노드 풀·스케줄러 프로파일·런타임 선택을 어떤 원칙으로 나누겠습니까?

4. **회귀 감시** — 배포로 상자 크기·웜업 시간·힙 밖 사용이 조용히 늘어나는 "경제성 회귀"를
   CI/운영에서 잡으려면 어떤 게이트(예: NMT diff, 콜드스타트 벤치, 밀도 지표)를 두겠습니까?

5. **의사결정의 재검토 주기** — 런타임/GC 기본값은 버전마다 바뀝니다(Green Tea GC, Leyden,
   ZGC 진화). 언어·GC 선택을 언제·무엇을 근거로 재검토하겠습니까? 벤더 벤치를 어디까지
   믿겠습니까?

6. **자동화 대 수동** — Java는 컨테이너 인식이 자동, Go는 메모리 수동입니다. 조직 표준으로
   `GOMEMLIMIT`·`MaxRAMPercentage`·`MALLOC_ARENA_MAX`를 어떻게 강제(어드미션 웹훅·베이스
   이미지·헬름 차트)하겠습니까?

## ⑦ 5부를 닫으며 → 6부로

5부는 런타임의 성능·비용 트레이드오프가 **컨테이너와 노드라는 무대에서 어떻게 돈으로
번역되는지**를 아래에서 위로 쌓았습니다. 회계(5.1) → 밀도(5.2) → 안전마진(5.3) → 시작 구간
(5.4) → 종합(5.5).

여기서 한 가지 역설이 다음 부의 씨앗이 됩니다. 5부 내내 Go의 한계를 짚었습니다 — `GOMEMLIMIT`
수동(5.1), 라이브 2배 헤드룸(5.2), Mark Assist tail(4부). **그런데 정작 쿠버네티스 그 자신이
Go로 쓰였고, 세계에서 가장 큰 규모로 돕니다.** 어떻게? 6부는 그 답을 봅니다 — K8s 엔지니어들이
Go의 한계를 우회한 패턴들(할당을 줄여 GC 압력을 낮추는 `sync.Pool`, 재동기화 부담을 줄이는
Informer 캐시, Kubelet의 동시성 설계, 그리고 5.1에서 본 컨테이너 인식 `GOMAXPROCS`). 1.2(스레드
모델)·2.4(소결)·3.3(Go 할당자)에서 심은 것들이 거기서 회수됩니다.
