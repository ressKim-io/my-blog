## ① 도달 지점 — 이 절에서 얻어가는 것

4.1에서 GC를 읽을 공용 어휘를 세웠습니다. 도달성, 삼색 마킹, 하이브리드 쓰기 배리어, STW와 동시, 이동/비이동, 세대별. 이제 그 어휘를 Go의 GC에 얹습니다. 그리고 4.1의 마지막 명제, **"동시성은 지연을 처리량으로 환전한다"**의 첫 번째 실증을 봅니다.

이 절을 끝내면 다음을 스스로 설명할 수 있습니다.

- Go의 GC가 **언제** 시작되는가 — `GOGC`(기본 100)와 `GOMEMLIMIT`, 그리고 그 뒤의 **페이서(pacer)**
- GC가 도는 동안 CPU의 **25%**가 어디로 사라지는가(`gcBackgroundUtilization`)
- **Mark Assist**가 무엇이고, 왜 이것이 예측 불가능한 꼬리 지연(tail latency)을 만드는가 — 할당하는 고루틴 본인을 GC 노동에 징집하는 메커니즘
- 디스코드(Discord)의 Read States 서비스가 **왜 2분마다 10~40ms씩 튀었는가** — `forcegcperiod`(2분) + 비세대 + 거대 LRU 캐시의 합작
- Go 1.26의 **Green Tea GC**가 이 그림에서 무엇을 바꿨는가

그리고 3.3에서 심어 둔 복선, **"할당이 싸다(P별 mcache·락 없음·bump 포인터)의 청구서가 어디로 오는가"**를 여기서 회수합니다. 답은 Mark Assist입니다. 싸게 많이 할당할수록 그 대금이 GC 지연으로 청구됩니다.

## ② Go GC의 실체 — 동시, 비이동, 비세대

Go의 GC를 한 줄로 정의하면 **동시 삼색 마크-스윕 수집기(concurrent tri-color mark-sweep collector)**이고, **객체를 옮기지 않으며(비이동, non-moving)**, **세대를 나누지 않습니다(비세대, non-generational)**. 이 세 성질이 뒤에 나오는 모든 현상의 뿌리라, 각각 왜 그런지부터 풉니다.

**왜 비이동인가.** 3.3에서 봤듯 Go는 객체 크기를 약 67개의 size class로 반올림해 미리 잘라 둔 슬롯에 할당합니다. 슬롯은 재사용되므로, 죽은 객체가 있던 자리를 제자리에서 비움(스윕)만 해도 다음 할당이 그 슬롯을 다시 씁니다. 굳이 옮길 이유가 없습니다. 그래서 **이동 비용이 0**입니다(4.1 삼각형의 오른쪽 꼭짓점 비용을 안 냄). 대가는 단편화인데, size class로 크기를 표준화해 두어 같은 클래스 안에서는 빈 슬롯이 곧바로 재사용되니 실무 단편화가 크게 억제됩니다.

**왜 비세대인가.** 세대별 GC의 핵심 가치는 두 가지입니다. 하나는 "대부분의 객체는 젊어서 죽는다"는 가설에 기대 영세대만 자주 훑어 작업량을 줄이는 것이고, 다른 하나는 Full GC의 긴 STW를 피하는 것입니다. 그런데 Go는 두 이득이 모두 작습니다. 첫째, Go 컴파일러의 이스케이프 분석(2.2)이 뛰어나서 단명 객체 상당수가 애초에 힙이 아니라 **스택에 할당**됩니다. 스택에 있으면 함수가 끝날 때 자동으로 사라지니 GC 대상이 아닙니다. 즉 "젊어서 죽는" 객체의 상당량을 GC가 아니라 스택 정리가 처리해 버려, 영세대를 따로 둘 이득이 JVM만큼 크지 않습니다. 둘째, Go는 이미 동시 GC로 STW를 sub-ms까지 눌러서(4.1) Full GC의 긴 멈춤 문제 자체가 없습니다.

여기에 결정적인 비용 문제가 하나 더 있습니다. 세대별로 가려면 노년세대가 영세대를 가리키는 포인터를 추적해야 하고(4.1의 리멤버드 셋), 그러려면 쓰기 배리어를 **항상** 켜 둬야 합니다. 그런데 Go의 하이브리드 쓰기 배리어는 GC가 도는 동안에만 켜지고 평소에는 꺼져 있습니다. 세대별을 도입하는 순간 이 배리어가 상시 켜져 평상시 처리량을 갉아먹습니다. Go 팀은 실제로 세대별을 실험했지만 "얻는 것보다 잃는 것이 크다"고 판단해 접었습니다. 정리하면 Go의 비세대는 게으름이 아니라 **이스케이프 분석 + 동시 GC + 배리어 상시화 회피**라는 세 근거 위의 선택입니다. (다만 이 선택의 청구서가 ③의 디스코드 사례에서 정확히 드러납니다.)

**GC 사이클의 실제 단계.** 한 번의 GC는 네 국면으로 돌고, 그중 뮤테이터를 실제로 멈추는 STW는 딱 두 번, 그것도 아주 짧습니다.

```
 한 번의 GC 사이클 (뮤테이터 관점의 멈춤은 [STW] 두 곳뿐):

  [STW]          [────────── 동시 마킹(concurrent mark) ──────────]   [STW]         [── 동시 스윕 ──]
  sweep                                                              mark
  termination    · 뮤테이터는 계속 실행                                termination   · 다음 할당이
  (배리어 ON,    · GOMAXPROCS의 25% = 백그라운드 마킹 워커             (배리어 OFF)    그 크기 클래스를
   루트 스캔 준비) · 부족하면 → Mark Assist로 뮤테이터 징집                            요구할 때 lazy 회수
   < 100µs                                                           < 100µs
       └─ 짧음 ─┘     └───────────── 여기가 길다(힙에 비례) ─────────┘   └─ 짧음 ─┘
```

두 STW(sweep termination, mark termination)가 100µs 미만인 것은 4.1에서 본 하이브리드 배리어가 마킹 종료 시점의 스택 재스캔을 없앴기 때문입니다. 정작 시간이 걸리는 구간은 가운데의 **동시 마킹**인데, 이때는 뮤테이터가 멈추지 않습니다. 그래서 Go는 "STW는 짧다"고 자랑할 수 있습니다. 문제는 그 짧은 STW 뒤에 숨은 두 개의 청구서, **25% CPU 점유**와 **Mark Assist**입니다.

## ③ 비용·메커니즘 해부 — 페이서, 백그라운드 25%, Mark Assist

### GC 페이서 — "언제 시작할지"를 정하는 제어기

**페이서(pacer)란, 프로그램이 힙을 채우는 속도에 맞춰 다음 GC를 언제 시작할지 계속 계산하는 제어기입니다.** 너무 일찍 시작하면 CPU를 낭비하고, 너무 늦게 시작하면 힙이 부풀거나 마킹이 할당을 못 따라갑니다. 그 균형점을 잡는 것이 페이서의 일입니다.

시작 시점의 기준은 **힙 목표(heap goal)**입니다. 공식은 다음과 같습니다.

```
  힙 목표 = 살아있는 힙(live) + (살아있는 힙 + GC 루트) × GOGC / 100
```

`GOGC`의 기본값은 100입니다. 이때 힙 목표는 대략 **라이브의 2배**가 됩니다. 예를 들어 직전 GC 후 살아남은 라이브가 10MB라면, 새로 10MB를 더 할당해 힙이 약 20MB에 닿는 순간 다음 GC가 시작됩니다. `GOGC`는 이 여유분을 조절하는 손잡이입니다.

```
 힙 크기
  ▲
  │ 힙 목표(20MB) ┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈
  │        ╱│           ╱│           ╱│
  │      ╱  │         ╱  │         ╱  │     ← 할당이 힙을 채워 올림
  │    ╱    │       ╱    │       ╱    │
  │  ╱      ▼     ╱      ▼     ╱      ▼
  │ live(10MB) ┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈  ← GC가 쓰레기를 걷어내고 라이브로 복귀
  └────────────────────────────────────────────────▶ 시간
         GC①          GC②          GC③
   힙이 목표에 닿을 때마다 GC 시작 → 톱니(sawtooth) 모양으로 오르내림
```

`GOGC`를 올리면 톱니의 봉우리가 높아집니다. GC가 덜 자주 돌아 CPU는 아끼지만 메모리를 더 씁니다. 내리면 반대입니다.

| `GOGC` | 힙 목표(라이브 10MB 기준) | GC 빈도 | 메모리 | GC CPU |
|---|---|---|---|---|
| 50 | ~15MB (라이브 +50%) | 잦음 | 낮음 | 높음 |
| 100 (기본) | ~20MB (라이브 +100%) | 기본 | 기본 | 기본 |
| 200 | ~30MB (라이브 +200%) | 뜸함 | 높음 | 낮음 |
| `off` | 무한 (자동 GC 없음) | 없음 | 무한정↑ | 0 |

`GOMEMLIMIT`은 여기에 얹는 두 번째 손잡이입니다. Go 1.19부터 들어온 **소프트 메모리 상한**으로, 런타임이 관리하는 메모리(힙 + 스택 + 런타임 자료구조)가 이 값에 다가가면 페이서가 `GOGC`와 무관하게 GC를 더 공격적으로 당겨 옵니다. "소프트"인 이유는 이걸 넘지 않겠다는 보장이 아니라 목표로만 삼기 때문입니다. `GOGC`가 "평소 여유"를, `GOMEMLIMIT`이 "메모리가 빠듯할 때의 안전판"을 담당해 둘이 함께 작동합니다. 다만 3.3에서 짚었듯 `GOMEMLIMIT`은 **아직 컨테이너 cgroup 한도를 자동으로 읽지 않습니다.** 운영자가 파드 메모리 리밋에 맞춰 직접 넣어 줘야 하고, 안 넣으면 5부에서 다룰 OOMKilled로 이어집니다.

### 백그라운드 마킹 25% — CPU는 어디로 갔나

동시 마킹이 도는 동안, Go 런타임은 `GOMAXPROCS`의 **25%**를 마킹 전용 워커(dedicated mark worker)에 고정합니다. 이 값이 런타임 상수 `gcBackgroundUtilization = 0.25`입니다. 왜 하필 25%일까요. 나머지 75%를 뮤테이터에 남겨 처리량을 지키면서, 마킹은 배경에서 꾸준히 진행시키기 위한 타협점입니다.

```
 GOMAXPROCS = 4, GC 마킹 진행 중:

  P0  ████████  뮤테이터(애플리케이션 고루틴 실행)
  P1  ████████  뮤테이터
  P2  ████████  뮤테이터
  P3  ▓▓▓▓▓▓▓▓  dedicated mark worker  ← 4개 중 1개(25%)를 마킹에 고정

  → 애플리케이션은 코어의 75%로 계속 일하고, GC는 25%로 배경 마킹
```

여기서 컨테이너의 함정이 하나 보입니다. `GOMAXPROCS`가 실제 할당된 CPU보다 크게 잡히면(예: 노드는 64코어인데 파드엔 2코어만 배정), 25% 계산의 분모가 틀어져 GC가 실제 몫보다 많은 CPU를 요구하거나 스로틀링에 걸립니다. Go 1.25부터 컨테이너 인식 `GOMAXPROCS`가 들어온 배경이 이것이고, 6부에서 자세히 다룹니다.

### Mark Assist — 할당하는 고루틴을 GC 노동에 징집하다

25% 백그라운드만으로 충분할까요. 뮤테이터가 힙을 아주 빠르게 채우면, 25%짜리 마킹이 그 속도를 못 따라갑니다. 마킹이 끝나기 전에 힙 목표를 넘어 버리면 GC의 존재 이유가 무너집니다. 그래서 Go는 **강수를 둡니다. 할당을 많이 하는 고루틴 본인을 붙잡아 마킹 노동을 시킵니다.** 이것이 **Mark Assist(마크 어시스트)**입니다.

메커니즘은 "할당 크레딧(빚)"으로 돌아갑니다. 각 고루틴은 `gcAssistBytes`라는 크레딧을 들고 있습니다. 고루틴이 메모리를 할당할 때마다 그 크기만큼 크레딧이 깎이고, 크레딧이 **음수(빚, debt)**가 되면 그 고루틴은 **더 할당하기 전에 먼저 GC 마킹을 일정량 수행해 빚을 갚아야** 합니다. 얼마나 갚아야 하는지는 페이서가 정한 **assist ratio**(할당 바이트당 스캔해야 할 작업량의 비례계수)로 결정됩니다. 마킹이 뒤처질수록 이 비율이 커져서, 할당이 많은 고루틴일수록 더 많은 GC 노동에 징집됩니다.

```
 할당 크레딧(gcAssistBytes)으로 본 두 고루틴의 운명:

  정상 고루틴:  할당 요청 → 크레딧 충분(≥ 0) → 그냥 할당하고 자기 일 계속
                ●───────────────────────────────────▶  (추가 지연 없음)

  빚진 고루틴:  할당 요청 → 크레딧 음수(debt) → "할당 전에 GC 스캔부터 하라"
                ●───[ Mark Assist: 남의 GC 일을 대신함 ]───┬─▶ 할당 재개
                    └──── 자기 요청 처리가 이만큼 멈춤 ────┘
                                                          ▲
                                          여기가 tail latency로 튄다
```

이것이 **왜 Go의 꼬리 지연이 튀는가**의 핵심 물리입니다. Mark Assist는 스케줄러가 미리 잡아 둔 일이 아니라 **할당이라는 사건에 반응해** 즉흥적으로 발동합니다. 그래서 어느 요청이 언제 얼마나 붙들릴지 예측하기 어렵습니다. 평소엔 1ms에 끝나던 요청이, 하필 GC 마킹 중에 할당을 많이 하는 바람에 자기 처리 대신 남의 GC 일을 하다가 지연이 튀는 것입니다.

**여기서 3.3의 복선이 회수됩니다.** 3.3에서 Go의 할당은 P별 mcache에서 락 없이 포인터만 밀면 되는, 세상에서 가장 싼 축의 할당이라고 했습니다. 그 싸구려 할당의 청구서가 바로 이 Mark Assist입니다. 할당이 싸니 프로그램은 거리낌 없이 많이 할당하고, 많이 할당할수록 페이서의 25% 백그라운드가 추월당해 Mark Assist가 발동하고, 결국 그 대금을 꼬리 지연으로 냅니다. **할당의 싸구려 가격표는 선불이 아니라 후불이며, 청구서는 GC 창구에서 날아옵니다.**

### 디스코드 사례 — "왜 tail이 튀는가"의 완성형 (2.2 복선 회수)

2.2에서 예고한 디스코드 사례를 이제 회수합니다. 디스코드의 Read States 서비스(사용자가 어떤 채널의 어떤 메시지까지 읽었는지 추적)는 대부분의 시간에 빨랐지만, **정확히 2분마다 지연이 10~40ms로 튀었습니다.** 세 가지가 겹친 결과입니다.

첫째, Go 런타임에는 `forcegcperiod = 2 * 60 * 1e9` 나노초, 즉 **2분**이라는 상수가 있습니다. 시스템 모니터 스레드 `sysmon`이 마지막 GC 이후 2분이 지나면 `forcegchelper`라는 고루틴을 깨워 **할당이 전혀 없어도 GC를 강제로 한 번 돌립니다.** 힙이 목표에 안 닿아도 2분이 최대 간격입니다.

둘째, Read States는 거대한 **LRU 캐시**를 메모리에 이고 있었습니다. 대부분 오래 살아남는 라이브 객체입니다.

셋째, Go는 **비세대**입니다(②). 오래된 객체를 "이건 지난번에 확인했으니 건너뛰자"라고 넘길 방법이 없습니다. 매 GC마다 **LRU 캐시 전체를 처음부터 다시 마킹**해야 합니다.

```
 디스코드 Read States (Go, 비세대):

  힙:  [ 거대한 LRU 캐시 = 대부분 오래 사는 live 객체 ]  + 소량의 쓰레기

  시간 ─┬───────────────┬───────────────┬───────────────┬──▶
        0분             2분             4분             6분
        │               │               │               │
     forcegc         forcegc         forcegc         forcegc   ← sysmon이 2분마다 강제
        │               │               │               │
     LRU 전체 마킹    LRU 전체 마킹    LRU 전체 마킹    LRU 전체 마킹
     (비세대라 오래된 객체를 건너뛸 수 없음)
        ▼               ▼               ▼               ▼
     p99: 10~40ms     10~40ms         10~40ms         10~40ms  ← 규칙적인 스파이크
```

핵심은 **디스코드가 쓰레기를 거의 안 만들었다는 점**입니다. 서비스를 극한까지 튜닝해 할당을 최소화했는데도 스파이크가 사라지지 않았습니다. 쓰레기 양과 무관하게, **스캔 비용은 살아있는 힙의 크기에 비례**하기 때문입니다. 비세대 마크-스윕에서는 "살아있는 것이 많으면 매번 그걸 전부 다시 확인"해야 하고, 마킹이 도는 동안 요청을 처리하려던 고루틴들은 Mark Assist에 징집됩니다. 결국 디스코드는 이 서비스를 Rust로 다시 썼습니다. Rust에는 GC가 없어 LRU에서 축출된 항목이 즉시 해제되고(4.1의 결정론적 drop), 주기적으로 전체를 훑는 주체가 아예 없으니 2분 스파이크가 원천적으로 사라졌습니다. 큰 튜닝 없이도 Rust판이 지연·CPU·메모리 모든 지표에서 잘 튜닝된 Go판을 앞섰습니다.

이 사례가 4.2 전체의 축소판입니다. `forcegcperiod`(2분) + 비세대(전체 스캔) + Mark Assist(뮤테이터 징집)가 합쳐져, "대체로 빠르지만 규칙적으로 튀는" Go GC의 초상을 만듭니다.

### `gctrace`로 이 모든 걸 눈으로 보기

`GODEBUG=gctrace=1`을 켜고 프로그램을 돌리면 GC 사이클마다 한 줄씩 찍힙니다. 형식은 다음과 같습니다(필드 주해).

```
 gc 14 @2.086s 3%: 0.021+4.8+0.012 ms clock, 0.17+1.9/4.5/0+0.098 ms cpu, 24->27->18 MB, 26 MB goal, 8 P
    │   │      │   └──────┬───────┘        └──────────┬──────────┘   └────┬────┘  └──┬──┘  └┬┘
    │   │      │          │                           │                   │          │     └ 사용 P 개수(GOMAXPROCS)
    │   │      │          │                           │                   │          └ 페이서의 힙 목표(26MB)
    │   │      │          │                           │                   └ 힙: GC시작→GC끝→라이브 (24→27→18MB)
    │   │      │          │                           │
    │   │      │          │                           └ CPU 시간: [STW쓸기종료] + [assist / 백그라운드 / idle] + [STW마킹종료]
    │   │      │          │                             ★ 여기 'assist=1.9'가 바로 Mark Assist에 쓴 CPU. 이게 크면 뮤테이터가 징집당하는 중
    │   │      │          └ 벽시계 시간: [STW쓸기종료 0.021] + [동시마킹 4.8] + [STW마킹종료 0.012] ms
    │   │      │            ★ 양끝(STW) 두 개가 <100µs, 가운데(동시마킹)가 김
    │   │      └ 프로그램 시작 후 GC에 쓴 누적 CPU 비율(3%)
    │   └ 프로그램 시작 이후 경과 시간(2.086초)
    └ GC 사이클 번호(14번째)
```

운영에서 볼 곳은 명확합니다. CPU 시간 필드의 **`assist` 값**이 Mark Assist입니다. 이 값이 백그라운드 대비 커지면, 페이서가 할당을 못 따라가 뮤테이터를 징집하고 있다는 신호이고 곧 꼬리 지연으로 나타납니다. 벽시계 필드의 양끝 두 STW가 짧게 유지되는지, 가운데 동시 마킹이 힙 증가에 따라 길어지는지도 여기서 읽힙니다.

## ④ 회피·최적화 전략

### 사용자가 돌리는 손잡이

**`GOGC`를 올린다.** 메모리에 여유가 있다면 `GOGC=200`처럼 올려 GC 빈도를 낮추면 CPU와 Mark Assist 압박이 함께 줄어듭니다. 처리량 지향 배치 작업에 잘 맞습니다.

**`GOMEMLIMIT`으로 상한을 건다.** 컨테이너에서는 파드 메모리 리밋보다 약간 낮게(예: 리밋의 90%) `GOMEMLIMIT`을 걸어 두면, 힙이 거기 다가갈 때 GC가 공격적으로 당겨져 OOMKilled를 예방합니다. 다만 자동이 아니라 수동 설정이라는 점을 잊으면 안 됩니다(5부에서 실무 값 계산).

**할당 자체를 줄인다.** 가장 근본적입니다. 이스케이프 분석이 스택에 남기도록 포인터를 불필요하게 흘리지 않고, 슬라이스·맵은 `make([]T, 0, n)`으로 미리 크기를 잡아 재할당을 막고, 재사용 가능한 큰 버퍼는 `sync.Pool`로 돌려 씁니다. `sync.Pool`은 GC가 만든 압박을 GC 밖에서 완화하는 대표 도구라, 6부에서 쿠버네티스 컴포넌트들이 이걸 어떻게 쓰는지 다시 봅니다. (역사적으로는 큰 더미 배열을 미리 잡아 힙 목표를 부풀리는 "ballast" 기법을 썼지만, `GOMEMLIMIT`이 그 자리를 대체했습니다.)

### 런타임의 답 — Green Tea GC (1.26): 마킹을 싸게 만들다 (3.3 복선 회수)

3.3에서 "Green Tea GC의 정체는 4부에서"라고 미뤄 뒀습니다. 이제 회수합니다. **Green Tea GC는 Go 1.26부터 기본으로 켜진 새 마킹 알고리즘으로, 객체를 하나씩 훑던 방식을 8KiB 단위의 큰 연속 블록(span) 단위로 바꿔 캐시 지역성과 CPU 확장성을 끌어올립니다.** 여전히 비이동·비세대이며, 회수 정책이 아니라 "마킹을 어떻게 스캔하느냐"를 바꾼 것입니다.

문제의식은 이렇습니다. 기존 마킹은 작업 대기열(work queue)에 **개별 객체**를 넣고 하나씩 꺼내 스캔했습니다. 작은 객체들이 힙 여기저기 흩어져 있으면, 큐에서 객체를 하나 꺼낼 때마다 그 객체가 있는 캐시 라인을 새로 읽어야 해서 캐시 미스가 잦았습니다(3.1~3.3에서 강조한 데이터 지역성의 반대). Green Tea는 대기열에 **span(8KiB의 연속 메모리 블록)**을 넣고, 한 span 안에서 스캔할 객체가 몇 개 모이면 그 span을 통째로 훑습니다. 연속 메모리를 몰아서 읽으니 지역성이 좋아집니다.

```
 마킹 work queue: 대기열에 '무엇'을 넣는가

  기존(객체 단위):
     큐 → [obj][obj][obj][obj][obj] ...   작은 객체가 힙에 흩어져 있음
          하나 꺼내 스캔 → 캐시 미스 → 또 하나 꺼내 스캔 → 캐시 미스 ...  (지역성 나쁨)

  Green Tea(span 단위, 8KiB):
     큐 → [ 8KiB span A ][ 8KiB span B ] ...
            │                │
            ▼                ▼
     ┌── span A(8KiB) ──┐   한 span 안에 스캔 대상이 여럿 모이면
     │ ●○●●○●○●●○●○●●○● │   그 연속 블록을 통째로 훑음 → 캐시 지역성↑
     └──────────────────┘   (Ice Lake / Zen 4+ 에서는 SIMD로 span 내부를 병렬 스캔 → 추가 ~10%)
```

효과는 GC 오버헤드 **10~40% 감소**로 보고됐고(마킹에 크게 의존하는 실제 프로그램 기준), 지원 CPU에서 SIMD 병렬 스캔이 약 10%를 더합니다. 중요한 것은 방향입니다. Green Tea는 마킹 자체를 싸게 만들어, ③에서 본 **Mark Assist가 발동할 확률과 강도를 낮춥니다.** 마킹이 빠르면 25% 백그라운드가 할당을 따라잡기 쉬워지고, 그만큼 뮤테이터를 징집할 일이 줄어 꼬리 지연이 완화됩니다. 다만 비세대라는 근본 성질은 그대로라, 디스코드형(거대 live heap 전체 스캔) 문제를 없애지는 못합니다. 줄여 줄 뿐입니다. 필요하면 `GOEXPERIMENT=nogreenteagc`로 옛 방식을 쓸 수 있지만 이 옵션은 1.27에서 제거될 예정입니다.

## ⑤ 3개 언어 수렴 — "언제, 무엇을, 어떻게 회수하나"

같은 "메모리 회수"를 세 언어가 어떻게 다르게 처리하는지, 4.2의 렌즈(트리거·이동·세대·뮤테이터 영향)로 정렬합니다.

**Rust**에는 페이서도, "언제 GC?"라는 질문도 없습니다. 회수 시점은 컴파일 타임에 이미 정해집니다. 소유권이 스코프를 벗어나는 그 지점에서 `drop`이 실행돼 즉시 해제됩니다. 디스코드가 Rust로 옮겨 스파이크를 없앤 게 정확히 이 성질 덕입니다. 대가는 개발자가 소유권·수명을 직접 관리해야 하고, `Arc` 순환은 여전히 누수라는 점입니다(4.1).

**Go**는 페이서가 할당 속도에 맞춰 GC를 트리거합니다. 비이동이라 이동 비용은 0이지만, 비세대라 매 사이클 살아있는 힙 전체를 다시 마킹합니다. 부족하면 Mark Assist로 뮤테이터를 징집해 꼬리 지연이 튑니다. Green Tea(1.26)가 마킹을 싸게 만들어 이를 완화합니다. 강점은 "튜닝 없이도 대체로 좋은 지연", 약점은 "거대 live heap + 엄격한 저지연"에서의 취약함입니다.

**Java**는 세대별이라 대부분의 GC가 영세대(young)만 훑고 끝납니다. 디스코드형 문제, 즉 오래 사는 거대 캐시가 있어도 그걸 매번 전부 스캔하지 않습니다. 노년세대는 가끔만 훑기 때문입니다. 이 지점이 Go 대비 Java의 구조적 강점입니다. 대신 세대별·이동식을 굴리는 대가로 G1은 가끔 Full GC로 크게 멈추고(4.3), ZGC는 배리어 처리량 세금을 냅니다(4.4).

| | 트리거(언제) | 이동? | 세대? | 마킹 중 뮤테이터 | 거대 live heap에서 |
|---|---|---|---|---|---|
| **Rust** | 없음(컴파일 타임 `drop`) | — | — | 영향 없음(GC 없음) | 강함(즉시 해제) |
| **Go** | 페이서(`GOGC`/`GOMEMLIMIT`) + 2분 강제 | 비이동 | **비세대** | **Mark Assist 징집** | **약함(매번 전체 스캔)** |
| **Java G1** | 힙 점유율(IHOP 기본 45%) | 이동 | 세대별 | evacuation pause만 | 강함(young만 자주) |
| **Java ZGC** | 할당률·힙 점유 | 이동(동시) | 세대별 | 로드 배리어 세금 | 강함 |

## ⑥ 트레이드오프 소결 — 삼각형 위의 Go

Go를 4.1의 트레이드오프 삼각형에 얹으면 위치가 뚜렷합니다.

```
                     회계의 위치
                         ▲
                        ╱ ╲
                       ╱   ╲
       단편화 ◀───────╱─────╲───────▶ 이동 비용
      (Go가 여기)              (Go는 여기 비용 0)
       · 비이동 마크-스윕 → 이동 비용 0, 그 대신 단편화 감수(size class로 억제)
       · 회계는 100% 런타임(사람도 컴파일러도 아님)

   ── 4.2가 드러낸 '동시성의 청구서' (STW를 sub-ms로 누른 값) ──
       (a) 백그라운드 25% CPU 상시 점유(마킹 중)
       (b) Mark Assist — 예측 불가한 뮤테이터 징집 = tail latency
       (c) 비세대 → 살아있는 힙 크기에 비례하는 스캔 비용(디스코드)
       (d) forcegcperiod 2분 → 할당이 없어도 규칙적으로 도는 GC
   Green Tea(1.26)는 (a)(b)를 줄이지만 (c)의 근본은 못 없앤다.
```

정리하면 Go GC의 설계는 "STW를 없애고, 이동 비용을 없애고, 튜닝 부담을 없앤다"는 세 가지를 취한 대가로, **동시성의 세금(25% CPU + Mark Assist)**과 **비세대의 세금(전체 스캔)**을 냅니다. 4.1의 명제 그대로, Go는 지연(STW pause)을 처리량과 꼬리 지연으로 환전했습니다. 대부분의 서비스에서는 이 환율이 유리합니다. 튜닝 없이도 pause가 sub-ms이고 처리량도 준수합니다. 하지만 디스코드처럼 **거대한 live heap을 이고 엄격한 꼬리 지연을 요구하는** 워크로드에서는 환율이 불리해져, 같은 삼각형의 다른 꼭짓점(세대별·즉시 해제)을 택한 Java나 Rust가 이깁니다. **"GC는 은탄환"이 아니라 "GC는 특정 환율의 거래"**라는 것, 이것이 Go 편의 결론입니다.

## ⑦ DevOps 관점 심화 질문

1. **`gctrace`의 `assist` 값과 꼬리 지연을 어떻게 연결해 대시보드를 만들 것인가?** CPU 시간 필드의 `assist`가 백그라운드 대비 커지는 순간과 p99 지연 스파이크가 상관되는지, `runtime/metrics`의 `/gc/pauses`·`/gc/cycles`와 함께 어떤 패널로 엮으면 "Mark Assist가 지연을 만드는 중"을 실시간으로 잡을 수 있는가.

2. **컨테이너에서 `GOMEMLIMIT`을 파드 메모리 리밋의 몇 %로 잡을 것인가?** cgroup 리밋을 자동으로 안 읽는 현재, 리밋의 90%가 안전한지, off-heap(cgo·mmap·다이렉트)까지 합산되는 cgroup 회계(2.4)를 감안하면 그 여유가 충분한지. `GOMEMLIMIT`을 너무 낮게 잡으면 GC가 끝없이 도는 "GC 죽음의 소용돌이(death spiral)"는 어떻게 감지하는가.

3. **`GOGC`와 `GOMEMLIMIT`을 함께 쓸 때의 상호작용을 어떻게 검증하는가?** 처리량 지향으로 `GOGC`를 올리면서 `GOMEMLIMIT`으로 상한을 거는 조합에서, 평소엔 여유롭게 돌다 메모리 압박 시 GC가 급증하는 전환점을 부하 테스트로 어떻게 재현·계측하는가.

4. **디스코드형 워크로드를 배포 전에 어떻게 식별하는가?** "라이브 힙이 크고 오래 살며, 쓰레기는 적은데 저지연이 필요한" 서비스를 pprof heap 프로파일·`/gc/heap/live`로 사전에 가려내, Go가 부적합한 후보인지 판단하는 기준은 무엇인가.

5. **Green Tea GC(1.26)로 올린 뒤 무엇을 비교 계측할 것인가?** 업그레이드 전후로 GC 오버헤드(누적 GC CPU %), `assist` 시간, p99 지연을 A/B로 비교할 때, SIMD 이득이 나는 CPU(Ice Lake/Zen 4+)와 그렇지 않은 CPU(예: 일부 Graviton 세대)에서 결과가 어떻게 갈리는지를 어떻게 통제하는가.

6. **`GOMAXPROCS`와 GC의 25% 백그라운드가 cgroup CPU 쿼터와 어긋날 때 무엇을 보는가?** 컨테이너 CPU 스로틀링(`cpu.stat`의 `nr_throttled`)과 GC 마킹 지연이 겹치는 구간을 eBPF나 `cpu.pressure`(PSI)로 어떻게 관측하고, 컨테이너 인식 `GOMAXPROCS`(1.25+) 적용 전후를 어떻게 비교하는가.

## ⑧ 다음 절 연결

Go를 통해 "비세대·비이동 마크-스윕"의 초상을 봤습니다. 이동 비용을 0으로 만든 대신 매 사이클 전체 라이브를 스캔하고, STW를 없앤 대신 Mark Assist라는 예측 불가한 지연을 얻었습니다. 자연스러운 다음 질문은 이것입니다. **세대별로 나누고 객체를 이동시키면 이 문제들이 풀릴까?**

**4.3 Java G1**으로 갑니다. G1은 힙을 수천 개의 region으로 쪼개 세대별로 운용하고, 살아있는 객체를 새 region으로 evacuate(이사)해 단편화를 없앱니다. 디스코드형 문제, 즉 거대 live heap을 매번 전체 스캔하는 문제에 세대별이 어떻게 답하는지(대부분 young만 훑음), 동시 마킹은 언제 트리거되는지(IHOP 기본 45%, 3.4의 humongous 연결), 그리고 "대체로 안정적인 G1이 왜 가끔 Full GC로 크게 멈추는가"를 해부합니다. Go가 지연을 꾸준한 세금으로 낸다면, G1은 대개 조용하다가 이따금 큰 청구서를 내미는 GC입니다. 4.1 삼각형의 왼쪽(단편화)을 포기하고 오른쪽(이동 비용)을 택하면 무엇이 달라지는지, 그 대가를 다음 절에서 봅니다.
