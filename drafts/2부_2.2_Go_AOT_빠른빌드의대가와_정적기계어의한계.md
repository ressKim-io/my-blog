# 2부. CPU를 쥐어짜는 전략: 기계어 생성과 런타임 최적화
## 2.2 Go (AOT): 빠른 빌드의 대가, 그리고 "정적"이라는 진짜 한계

> 2.1의 Rust는 "느리게 빌드하고 깊게 최적화"하는 AOT였다. Go는 같은 AOT지만 정확히 반대 방향으로 튜닝된 언어다.
> Go 설계자들은 **컴파일 속도**를 1급 목표로 못박았고, 그 대가로 기계어 최적화의 상당 부분을 *의도적으로* 포기했다.
> 이 절의 또 다른 임무는 널리 퍼진 오해 하나를 바로잡는 것이다 — **"Go엔 PGO가 없다"는 틀렸다.**
> PGO는 Go 1.21부터 정식(GA)이다. 진짜 한계는 PGO의 유무가 아니라 **"정적"이라는 본질**에 있다.

---

### ① 도달 지점

- Go가 **왜 LLVM을 버리고** 자체 컴파일러(gc)와 자체 SSA 백엔드를 택했는지, 그 설계 상충 — 그리고 SSA가 애초에 무엇인지
- Go 성능의 3대 컴파일러 레버를 **기전 수준으로**: 인라이닝(예산 80이 어떻게 계산되나), 이스케이프 분석(무엇이 "탈출"이고 어셈블리로 어떻게 갈리나), 경계 검사 제거(prove 패스의 원리)
- **인터페이스 호출이 기계어로 어떻게 풀리나** — `itab` 구조체를 필드별로, 간접 호출 어셈블리까지
- Go에 **자동 벡터화가 없다**는 사실의 정확한 의미와 그 천장 (1.25 기준, 1.26 실험 `simd` 포함)
- **PGO(1.21 GA)** 가 실제로 하는 일 — 핫 함수 인라인 예산을 80→2000으로 올리고 역가상화 — 그럼에도 **"빌드 스냅샷"이라는 본질적 한계**
- ARM(GOARCH=arm64)에서의 코드 생성 특성

---

### ② 개념 / 물리적 실체 — "빠른 빌드"라는 제1원리, 그리고 SSA가 뭔지부터

먼저 용어 하나를 풀고 가자. **SSA(Static Single Assignment, 정적 단일 대입)** 는 컴파일러 내부의 중간 표현(IR) 형식인데, 규칙이 단 하나다 — *모든 변수는 딱 한 번만 대입된다.* 원래 코드에서 `x`에 값을 두 번 넣으면, SSA에서는 `x1`, `x2`처럼 매번 새 이름을 준다. 왜 이렇게 하냐면, "이 값이 어디서 와서 어디로 흘러가는가"를 추적하기가 압도적으로 쉬워지기 때문이다. 값의 출처가 유일하니 상수 전파·죽은 코드 제거·경계 검사 제거 같은 최적화를 단순하고 정확하게 돌릴 수 있다. 현대 컴파일러(LLVM 포함)가 거의 다 SSA를 쓰는 이유다.

Go는 이 SSA 백엔드를 **직접** 만들어 쓴다. LLVM을 쓰지 않는다. 컴파일러 이름은 관례적으로 **gc**(Go compiler, Plan 9 계보의 툴체인에서 출발)다. 파이프라인은 이렇다.

```
[소스 .go]
    │  스캔 · 파싱 · 타입 체크
    ▼
[AST]  추상 구문 트리
    │  ★ 이 단계에서 이스케이프 분석 · 인라이닝 결정 (스택/힙, 어떤 호출을 인라인할지)
    ▼
[IR → SSA]  Go 자체 SSA (Go 1.7, 2016 도입)
    │   ┌─────────────────────────────────────────────────┐
    │   │  최적화 패스: LLVM에 비해 "가짓수가 적고 얕다"          │
    │   │  const 전파 · 죽은코드 제거 · nil 검사 제거             │
    │   │  prove 패스(경계 검사 제거) · 약간의 재작성 규칙          │
    │   │  ✗ 자동 벡터화 없음   ✗ 공격적 루프 변환(언롤·퓨전) 없음  │
    │   └─────────────────────────────────────────────────┘
    │  아키별 rewrite rules (amd64 / arm64 / …)로 SSA를 기계 명령으로 낮춤
    ▼
[타깃 기계어] → 정적 링크 → 단일 바이너리 (1부 1.3의 "자족성" 회수)
```

여기서 왜 Go가 이런 선택을 했는지가 핵심이다. Google 규모의 거대한 단일 저장소(monorepo)에서는 "빌드가 몇 초 만에 끝나는 것" 자체가 언어의 존재 이유였다. LLVM은 강력하지만 느리다 — 2.1에서 봤듯 단형화로 부풀어 오른 방대한 IR을 `-O3`로 갈아 넣는 데 시간이 걸린다. Go는 아예 그 비용을 지지 않기로 했다. 그 결과가 위 파이프라인의 두 특징이다. 첫째, **최적화 패스가 적고 얕다.** 둘째, 그래서 **컴파일이 압도적으로 빠르고 결과가 예측 가능**하다. 실제로 릴리스 바이너리를 `objdump`로 까 보면 인라인이 덜 되어 있고, SIMD 명령이 거의 없고, 스칼라 루프가 그대로 남아 있는 걸 흔히 본다.

> 이건 결함이 아니라 **트레이드오프의 방향 선택**이다. "느린 빌드로 3% 빠른 바이너리"보다 "빠른 빌드로 넉넉히 빠른 바이너리"가 조직 전체의 생산성에는 낫다는 판단. 이 판단이 옳은지 그른지는 워크로드가 결정한다(⑥).

### ③ 비용·메커니즘 해부 — Go 성능을 좌우하는 세 레버

#### (1) 인라이닝 — "예산 80"이 실제로 어떻게 계산되나

먼저 인라이닝이 왜 중요한지부터. **인라이닝(inlining)** 은 함수 호출을, 그 함수의 본문으로 통째 갈아 끼우는 최적화다. 얻는 건 두 가지다. 눈에 보이는 이득은 `CALL` 명령과 스택 프레임을 세우고 허무는 비용이 사라지는 것. 하지만 **진짜 이득은 그다음**이다 — 호출이라는 "벽"이 없어지면, 그 너머로 상수 전파·죽은 코드 제거·경계 검사 제거 같은 최적화가 **흘러 들어갈 수 있다.** 인라인이 막히면 이 연쇄 이득이 통째로 증발한다.

Go의 인라이너는 함수를 인라인할지 말지를 **비용(cost) 점수**로 결정한다. 이 점수는 함수 본문의 복잡도를 근사한 값인데, 구체적으로는 **AST(구문 트리) 노드 수에 대략 비례**한다. 문장 하나, 연산 하나가 노드 몇 개씩 비용을 더한다. 그리고 정해진 **예산이 80**이다(`inlineMaxBudget = 80`). 함수의 비용이 80을 넘으면 "너무 복잡함"으로 판정돼 인라인 후보에서 빠진다.

```
함수 add:  cost 8   ≤ 80  → 인라인 O
함수 parse: cost 120 > 80  → 인라인 X → 호출부에 CALL 남음
                              → 그 안의 상수전파·BCE·DCE 기회도 함께 소멸

$ go build -gcflags="-m" .
./main.go:5:6:  can inline add
./main.go:9:10: inlining call to add
$ go build -gcflags="-m -m" .           # -m 두 번 = 이유까지
./main.go:20:6: cannot inline parse: function too complex: cost 120 exceeds budget 80
```

두 가지 세부를 알아두면 그림이 완성된다.

- **중간 함수 인라이닝(mid-stack inlining, Go 1.9+)**: 아주 초기의 Go는 *leaf 함수*(내부에서 다른 함수를 호출하지 않는 말단 함수)만 인라인했다. 1.9부터는 다른 함수를 호출하는 중간 함수도 인라인할 수 있게 됐다. 다만 그런 함수는 비용이 크기 마련이라, 여전히 예산 80에 자주 걸린다. `for`·`range`·`defer`·`recover`·재귀 같은 구조는 비용을 크게 올려 인라인을 막는 단골이다.
- **패키지 경계를 넘는 인라인**: Go 컴파일러는 인라인 가능한(=비용 80 이하) 함수의 본문을 **패키지 export 데이터**에 실어 내보낸다. 그래서 다른 패키지가 그 함수를 호출할 때도 인라인이 가능하다. 이건 2.1 Rust와 흥미로운 대조다 — Rust는 `#[inline]`이나 제네릭이어야 rlib에 MIR이 실려 경계를 넘었는데, Go는 **애너테이션 없이 자동**이되 **예산 80이라는 훨씬 낮은 천장**에 묶인다. 즉 Go는 "경계는 잘 넘지만 애초에 인라인 자체를 보수적으로 한다."

LLVM(Rust)이 훨씬 공격적으로 인라인하는 것과의 격차가 바로 여기서 벌어진다. Go는 "작은 함수는 잘 인라인하지만, 조금만 커지면 CALL로 남긴다."

> **곁가지 — 레지스터 호출 규약(Go 1.17+)이 CALL의 아픔을 줄였다.** 인라인에 실패해 `CALL`이 남더라도, 그 페널티는 과거보다 작다. Go 1.17 이전(ABI0)은 함수 인자를 **100% 스택으로** 넘겼다(Plan 9 유산). 1.17부터 **ABIInternal**이 도입돼, amd64 기준 정수 인자 **9개까지 레지스터**(AX, BX, CX, DI, SI, R8~R11)로 직접 전달한다 — Google 팀 보고로 처리량 **5~10% 개선**. 즉 "인라인 실패 = 치명적 메모리 I/O"라는 옛 공식은 완화됐다. 프레임 셋업 비용은 남아도, 인자는 메모리를 거치지 않고 레지스터로 흐른다. (앞의 이스케이프 분석이 "값을 스택에 유지"하는 것과 방향이 같다 — Go가 메모리 왕복을 줄여온 궤적이다.)

#### (2) 이스케이프 분석 — "탈출"이 정확히 무엇이고, 어셈블리로 어떻게 갈리나

**이스케이프 분석(escape analysis)** 은 "이 변수를 스택에 둘까, 힙에 둘까"를 컴파일러가 정적으로 결정하는 분석이다. 판단 기준은 딱 하나 — **그 변수(정확히는 그 변수를 가리키는 포인터)가 함수의 생존 범위를 넘어 살아남아야 하는가.** 넘어 살아남아야 하면 "탈출한다(escape)"고 말한다.

왜 이게 중요하냐면, 스택과 힙의 비용이 하늘과 땅 차이이기 때문이다.

- **스택에 두면**: 할당 비용이 사실상 0이다. 함수 진입 시 스택 포인터를 몇 바이트 내리는(`SUBQ`) 것으로 공간이 잡히고, 함수가 반환하면 그 프레임이 통째 사라지며 자동 회수된다. GC는 이걸 신경 쓸 필요조차 없다.
- **힙에 두면**: 런타임 할당자(`runtime.newobject`)를 호출해야 하고(그 아래에 mcache·mspan 관리 비용 — 3부에서 깊게), 무엇보다 **나중에 GC가 이 객체를 찾아내 마크하고 회수**해야 한다. 힙 할당 하나하나가 미래의 GC 부담으로 예약되는 셈이다.

컴파일러는 SSA 위에서 포인터가 어디로 흘러가는지를 추적해 탈출 여부를 판정한다. 반환되는가? 전역 변수나 힙 자료구조에 저장되는가? 인터페이스로 감싸지는가(박싱)? 클로저에 캡처되는가? 이 중 하나라도 해당하면 탈출로 본다.

```go
func noEscape() int {
    x := 42
    return x          // 값 자체를 복사해 반환 → 포인터가 안 나감 → 스택
}
func escapes() *int {
    x := 42
    return &x         // x의 주소가 함수 밖으로 나감 → x는 반드시 힙에 있어야 함
}
```
```
$ go build -gcflags="-m" .
./main.go:7:2: moved to heap: x         ← escapes()의 x가 힙으로 이동

; 기계어 수준의 차이
noEscape:  (힙 할당 없음)              escapes:  (힙 할당)
    MOVL $42, AX                          LEAQ type:int(SB), AX
    RET                                   CALL runtime.newobject(SB)   ; ← 런타임 할당 호출
                                          MOVQ $42, (AX)
                                          RET
```

`-gcflags="-m"`이 알려주는 대표 문구 세 가지를 해석할 줄 알면 진단이 쉬워진다.
- `moved to heap: x` — 지역 변수 x가 탈출해서 힙으로 옮겨졌다.
- `x escapes to heap` — 인자/값 x가 힙으로 탈출한다.
- `leaking param: p` — 인자 p의 포인터가 함수 밖으로 새 나간다(반환되거나 저장됨).

이스케이프 분석은 **Go 성능 튜닝의 심장**이다. 실전에서 힙 할당을 줄이는 기법(값으로 전달, 슬라이스 미리 할당, `interface{}`로의 불필요한 박싱 회피, `sync.Pool` 재사용)은 전부 "컴파일러가 이 변수를 스택에 유지하도록 유도"하는 행위다. 그리고 그 효과는 4부에서 다룰 GC 부담과 직결된다.

> **왜 이게 2부(CPU)의 문제인가 — GC가 훔쳐가는 CPU 사이클 (4부 복선).** 힙 할당은 단지 "나중에 치울 쓰레기"가 아니라 *지금 이 순간의 CPU 대역폭*을 갉아먹는다. Go의 GC는 지연 시간(latency)을 극단적으로 낮춘 대신 그 값을 **처리량(throughput)** 으로 치른다. 마크 페이즈 동안 백그라운드 마크 워커가 **GOMAXPROCS의 약 25%**(`gcBackgroundUtilization = 0.25`)를 점유한다 — 그만큼의 코어가 비즈니스 로직에서 빠진다. 게다가 할당 속도가 GC 수거 속도를 앞지르면 **Mark Assist(뮤테이터 어시스트)** 가 발동해, **할당하려는 고루틴을 강제로 마크 작업에 차출**한다(메모리를 받기 전에, 할당량에 비례한 마킹을 먼저 시킨다). 즉 이스케이프 하나를 스택에 붙잡는 것은 미래의 청소비뿐 아니라 **지금 계산에 쓸 CPU 사이클을 지키는** 일이다. 상세 메커니즘·디스코드 사례는 4부에서.

#### (3) 경계 검사 제거(BCE) — prove 패스가 "안전"을 증명하는 원리

Go는 메모리 안전 언어다. 그래서 슬라이스나 배열에 `b[i]`로 접근할 때마다, `i`가 유효 범위 `[0, len(b))` 안에 있는지 **런타임 검사**를 넣는다. 벗어나면 즉시 panic을 던져 버퍼 오버런을 원천 차단한다. 문제는 이 검사가 공짜가 아니라는 것 — 접근마다 비교(`CMP`)와 조건 분기(`JAE`)가 붙어, 핫 루프에서는 명령 수와 분기 예측 압력을 키운다.

Go의 SSA에는 **prove 패스**라는 최적화가 있다. 이름 그대로, 값들 사이의 관계에서 "사실(fact)"을 전파해 **"이 인덱스는 항상 유효하다"를 증명**하면 그 검사를 제거한다. 예컨대 `for i := 0; i < len(b); i++` 루프 안의 `b[i]`는, 루프 조건 `i < len(b)`와 `i >= 0`이 이미 보장되므로 검사가 불필요하다는 걸 증명할 수 있다.

```
; 증명 실패 → 접근마다 경계 검사가 남음
    CMPQ  CX, BX          ; i(CX) vs len(BX)
    JAE   panicIndex      ; unsigned 비교로 i>=len 이면 panic으로 점프
    MOVQ  (AX)(CX*8), DX   ; b[i] 로드

; 증명 성공(BCE) → 위 CMPQ/JAE 두 줄이 통째 사라지고 로드만 남음
    MOVQ  (AX)(CX*8), DX
```

증명이 어려운 패턴에서는 **힌트**를 줘서 도울 수 있다.
```go
_ = b[len(b)-1]              // ① 여기서 "len(b) >= 1" 사실이 확립됨
for i := 0; i < len(b); i++ {
    sum += b[i]              // ② 컴파일러가 ①의 사실을 이어받아 이후 검사를 제거
}
```
첫 줄에서 `b[len(b)-1]`을 미리 만지면 컴파일러가 `len`과의 관계를 파악하게 되어, 뒤따르는 접근들의 검사를 제거한다. 이건 "컴파일러의 증명기를 손으로 거들어주는" 실전 기법이다.

#### (4) 자동 벡터화의 부재 — 가장 낮은 천장

**벡터화(vectorization)** 란 한 명령으로 여러 데이터를 동시에 처리하는 것이다. 예를 들어 `float32` 8개를 한 번의 AVX 명령으로 더하는 식(SIMD, Single Instruction Multiple Data). 스칼라 루프(한 번에 하나씩)를 컴파일러가 **자동으로** 이런 SIMD 명령으로 바꿔주는 최적화 패스를 auto-vectorization이라 한다.

**Go 컴파일러에는 이 패스가 없다.** 스칼라 루프는 스칼라 기계어로 그대로 남는다. 그래서 SIMD 성능이 필요한 곳(암호·해시·수치연산)은 표준 라이브러리조차 **사람이 직접 어셈블리(`.s` 파일)로** 작성한다 — 종종 `Avo`라는 어셈블리 생성기를 써서. `crypto/sha256`, `math` 등이 그렇게 되어 있다.

- **Go 1.25(기준 버전)**: 자동 벡터화 없음, 1급 SIMD 패키지 없음 → SIMD = 어셈블리.
- **Go 1.26(실험)**: `GOEXPERIMENT=simd` 하에 실험적 `simd`/`archsimd` 패키지 등장. `Int8x16`, `Float64x8` 같은 벡터 타입이 하드웨어 벡터 레지스터에 대응하고, 연산은 컴파일러 인트린식으로 하드웨어 명령에 직결된다. 단 **AMD64 전용, 옵트인, 여전히 "개발자가 명시적으로 인트린식을 부르는" 방식이지 자동 벡터화가 아니다.** 개발은 1.27로 이어진다.

즉 방향 자체가 "컴파일러가 알아서 벡터화"가 아니라 "개발자가 손으로 인트린식을 부른다"이다. 이것이 CPU 바운드 수치연산에서 Go의 천장이 Rust(LLVM 자동 벡터화)·Java(C2 SuperWord, 2.3)보다 구조적으로 낮은 이유다.

#### (5) 값 의미론과 데이터 지역성 — 벡터화 부재를 상쇄하는 숨은 무기

Go가 자동 벡터화가 없는데도 CPU 바운드 순회에서 의외로 잘 버티는 이유가 있다. **값 의미론(value semantics)에 기반한 메모리 연속성**이다. 이건 앞의 "천장" 이야기를 절반쯤 되돌려 놓는 반전이라 짚고 간다.

Go에서 `[]Point`(Point는 struct)를 만들면, 각 Point가 **배열 안에 통째로, 연속으로** 박힌다. 별도 힙 객체도, 요소마다의 헤더도 없다. 반면 Java에서 `Point[]`(Point는 일반 클래스)는 **참조(포인터)의 배열**이다 — 실제 Point 객체들은 각자 헤더(마크워드 + 클래스 포인터, 대략 12~16B)를 달고 힙 여기저기에 흩어져 있고, 배열은 그것들을 가리키는 포인터만 담는다.

```
Go  []Point (값이 연속 배치):              Java Point[] (참조의 배열):
┌───────┬───────┬───────┬───────┐          arr → [ptr0][ptr1][ptr2][ptr3]
│ P0    │ P1    │ P2    │ P3    │                 │     │     │     │
│ x,y   │ x,y   │ x,y   │ x,y   │                 ▼     ▼     ▼     ▼
└───────┴───────┴───────┴───────┘              [P0]  [P3]  [P1]  [P2]  ← 힙 곳곳에 흩어짐
 → 한 캐시라인에 여러 요소가 실림              → 요소마다 포인터를 따라 점프(pointer chasing)
 → 순차 접근이 곧 순차 메모리 접근             → 접근 패턴이 힙 주소로 튐
```

이 차이가 CPU 관점에서 결정적이다. CPU에는 **하드웨어 프리패처(hardware prefetcher)** 가 있어서, 메모리를 순차적으로 훑는 패턴을 감지하면 다음 캐시 라인을 L1/L2로 **미리 당겨온다.** Go의 연속 배치는 이 프리패처가 완벽하게 먹혀, 순회하는 내내 데이터가 캐시에 준비돼 있다(높은 캐시 적중률). Java의 참조 배열은 포인터를 따라 힙 곳곳으로 점프하니 프리패처가 무력화되고 **캐시 미스**가 잦다 — DRAM까지 다녀오는 한 번의 미스가 수백 사이클을 날린다.

여기서 2.2의 이야기가 완성된다. Go는 ③-(4)에서 봤듯 자동 벡터화(SIMD)가 없다. 하지만 **SIMD로 한 번에 8개를 계산해봤자, 그 데이터가 캐시에 없어 메모리를 기다리면 소용이 없다.** 순회가 메모리 바운드인 워크로드에서는, Go의 데이터 지역성이 벡터화 부재를 상당 부분 **상쇄**한다. (물론 데이터가 이미 캐시에 다 들어와 있는 순수 계산 커널에서는 SIMD가 여전히 앞선다 — 상쇄는 "메모리 바운드일 때"라는 조건부다.)

참고로 Java도 `int[]`·`long[]`·`double[]` 같은 **원시 타입 배열은 연속**이다 — 문제는 *객체*의 배열이다. 이 격차를 좁히려는 것이 **Project Valhalla(JEP 401 값 클래스)** 의 배열 평탄화(heap flattening)인데, JDK 25 기준 아직 **프리뷰**다(2.3에서 재론).

### ④ 회피 / 최적화 전략 — PGO 오해 정정과 인터페이스의 실체

#### PGO(Profile-Guided Optimization) — 있다, 1.21부터 정식(GA), 그리고 무엇을 하나

**오해 정정**: "Go엔 PGO가 없다"는 낡은 정보다. Go 1.20 프리뷰, **Go 1.21에서 GA(정식)**. 사용법이 놀랍도록 간단하다.

```
1) 프로덕션/대표 워크로드에서 CPU 프로파일을 수집한다 (net/http/pprof 등)
      → 그 프로파일을 default.pgo 라는 이름으로 main 패키지 디렉터리에 둔다
2) 그냥 빌드한다:  go build
      → -pgo=auto 가 1.21부터 기본값이라, default.pgo가 있으면 자동으로 PGO 적용
```

PGO가 실제로 하는 일은 두 가지이고, 둘 다 앞에서 본 레버를 프로파일로 되살리는 것이다.

```
(a) 핫 함수 인라인 예산 확대  ★수치로 확인
    프로파일상 "뜨거운" 함수는 인라인 예산을 80 → 2000 으로 올린다 (inlineHotMaxBudget, 25배).
    → ③-(1)에서 "cost 120이라 인라인 못 했던" 함수도, 그게 핫하면 인라인된다.
    → 즉 호출 경계 최적화가 핫 경로에서 되살아난다.

(b) 역가상화(devirtualization)  ★인터페이스 간접 호출을 직접 호출로
    프로파일상 어떤 인터페이스 호출이 사실상 한 구체 타입만 부른다면,
        w.Write(...)  →  (*os.File).Write(...) 직접 호출로 치환 (+ 인라인 후보로)
```

전형적인 이득은 **2~7%**다. 참고로 Go 컴파일러 자신도 PGO로 빌드되어, 프로그램을 2~4% 더 빠르게 빌드한다.

#### 인터페이스 호출이 기계어로 풀리는 과정 — `itab`을 필드별로

역가상화가 "무엇을 걷어내는지"를 이해하려면, 인터페이스 호출이 원래 기계어에서 어떻게 생겼는지를 봐야 한다. Go의 인터페이스 값은 **두 개의 포인터**로 이루어진 구조다.

```
인터페이스 값  =  ( itab 포인터 , data 포인터 )    ← 흔히 "fat pointer"

data 포인터   → 실제 값(리시버)이 있는 곳
itab 포인터   → 아래 itab 구조체를 가리킴 (인터페이스+구체타입 조합마다 1개, 런타임이 생성·캐시)

type itab struct {
    inter *interfacetype  // 어떤 인터페이스인가        (예: io.Writer)
    _type *_type          // 어떤 구체 타입인가          (예: *os.File)
    hash  uint32          // _type.hash 사본 (타입 스위치 가속용)
    _     [4]byte
    fun   [1]uintptr      // ★ 메서드 함수 포인터 테이블 (가변 길이)
}                         //   fun[0]==0 이면 그 타입이 인터페이스 미구현이라는 표시
```

핵심은 `fun` 배열이다. 이건 "이 구체 타입이 이 인터페이스의 각 메서드를 어떤 함수로 구현하는가"를 담은 함수 포인터 테이블이다. 그래서 `w.Write(p)` 호출은 기계어에서 이렇게 풀린다.

```
    MOVQ  itab(SP), DX          ; itab 포인터를 DX로
    MOVQ  fun_offset(DX), CX    ; itab.fun[k] = Write 구현 함수 포인터를 CX로
    MOVQ  data(SP), AX          ; 리시버(data 포인터)를 첫 인자 레지스터로
    ...                         ; p 등 나머지 인자를 레지스터에
    CALL  CX                    ; ★ 간접 호출 (함수 주소가 런타임에 정해짐)
```

이 `CALL CX`가 **간접 호출(indirect call)** 이다. 두 가지 비용이 붙는다. 첫째, CPU의 분기 타깃 예측기(BTB)가 이 호출의 목적지를 예측하는데, 목적지가 여러 타입으로 흔들리면 예측이 자주 빗나가 파이프라인이 손해를 본다. 둘째, **인라인이 불가능**하다 — 컴파일 시점에 어떤 함수가 불릴지 모르니 본문을 갈아 끼울 수가 없다.

PGO 역가상화는 바로 이걸 노린다. "프로파일상 이 지점은 거의 항상 `*os.File`"이라면, 간접 호출을 `*os.File`의 `Write`를 **직접 호출**하는 코드로 바꾸고(만약을 위한 타입 가드는 남기고), 나아가 그 직접 호출을 인라인 후보로 만든다. **그러나** — 이 판단은 어디까지나 *빌드 시점의 프로파일*에 근거한다. 배포된 뒤 트래픽 특성이 바뀌어 다른 구체 타입이 뜨거워지면, **재빌드하기 전까지는 반영되지 않는다.** 바로 이 지점이 이 절의 결론으로 이어진다.

#### 그 밖의 실전 레버

- **힙 할당 억제**: `sync.Pool`(객체 재사용), 값 전달, 슬라이스 프리얼로케이션(`make([]T, 0, n)`) → 이스케이프 분석이 스택 유지하도록 유도. `-gcflags=-m`으로 회귀를 감시.
- **BCE 힌트**: 위 ③-(3)의 `_ = b[len(b)-1]` 패턴.
- **SIMD가 꼭 필요하면**: 어셈블리 루틴(또는 1.26+ 실험 `simd`).
- **바이너리 축소**: `go build -ldflags="-s -w"`(심볼·디버그 정보 제거).
- 빌드 자체는 손댈 게 거의 없다 — **빠른 빌드가 이미 Go의 기본값**이고, 그게 이 언어의 철학이다.

### ⑤ 세 언어 수렴 — 주제: "정적 컴파일의 최적화 천장과 재최적화 불가"

```
                     Go (2.2)              Rust (2.1)           Java (2.3)
──────────────────  ───────────────────  ──────────────────  ─────────────────
백엔드 / 최적화 깊이   자체 SSA / 얕음        LLVM / 깊음          C1·C2 / 런타임 적응
경계 넘는 인라인       export data(자동,       #[inline]/제네릭     런타임에 결정
                     예산 80 천장)          =MIR직렬화; LTO
자동 벡터화          ✗ (수동 asm/실험 simd)  △ (LLVM 자동)        ○ (C2 SuperWord)
메모리 지역성         ★ []struct 연속배치      ★ 연속(값 타입)       ✗ Object[]=참조배치
                     (프리패처 유리)                              (Valhalla 프리뷰 대기)
다형성 호출 비용       itab 간접호출          단형화=제로코스트      런타임 역가상화로 제거
                     (PGO devirt로 완화)    (dyn이면 vtable)
프로파일 반영 시점     빌드 1회 (PGO 1.21+)   빌드 1회 (PGO/BOLT)   실행 내내 연속 ★
워크로드 변화 적응     ✗ 재빌드 필요          ✗ 재빌드 필요         ○ 자동 재프로파일
빌드 속도            ★★★ 매우 빠름          ★ 느림 (LLVM+단형화)   중간 (바이트코드까지만)
```

**세 언어를 관통하는 통찰**: Go와 Rust는 백엔드의 깊이(얕음 vs 깊음)가 정반대지만, **"정적"이라는 본질을 공유**한다. 둘 다 최선의 무기가 "빌드 시점에 찍은 프로파일 스냅샷(PGO)"이다. 배포된 뒤 세상이 바뀌면 스스로 못 따라간다 — 사람이 프로파일을 다시 떠서 재빌드해야 한다.

Java만 이 틀을 깬다. Java의 JIT은 프로파일을 *빌드 시*가 아니라 *실행 내내* 수집해서 기계어를 계속 갈아엎는다(2.3). 그래서 트래픽 특성이 변하면 Java는 (웜업을 다시 치르지만) **자동으로 재적응**한다. 위 표에서 `itab 간접호출`(Go)과 `vtable`(Rust `dyn`)과 `런타임 역가상화`(Java)를 나란히 보면, 셋 다 "다형성의 가격표"인데 **누가 언제 그걸 깎아주느냐**가 갈린다 — Go/Rust는 빌드 시 프로파일이 있어야, Java는 실행하다 저절로.

### ⑥ 트레이드오프 소결

Go의 계약:

> **컴파일 속도·언어 단순성·운영 예측성을 얻는 대신, 기계어 최적화의 상한(특히 자동 벡터화와 공격적 인라이닝)을 포기한다.**

- **강점이 빛나는 곳**: I/O 바운드 서비스(API 게이트웨이, 프록시, 컨트롤러). 병목이 네트워크·디스크라 CPU 기계어를 몇 % 더 짜낼 여지 자체가 작다. 여기선 "빠른 빌드 + 넉넉한 성능 + 정적 바이너리(1부 1.3)"의 조합이 압도적으로 실용적이다. 게다가 I/O 대기 시 Go 런타임은 OS 스레드는 둔 채 **유저 공간에서 고루틴만 교체**해, 커널 컨텍스트 스위치(Ring 0 진입 = 수천 사이클)를 피하고 남는 CPU 사이클을 다른 연산에 온전히 쓴다(메커니즘은 1.2). 쿠버네티스가 Go로 쓰인 이유의 절반이 이것이다(6부).
- **천장이 드러나는 곳**: CPU 바운드 수치연산(코덱, ML 전처리, 대량 파싱). 자동 벡터화 부재가 그대로 처리량 격차로 나타난다. **단**, 순회가 메모리 바운드라면 값 의미론의 데이터 지역성(③-5)이 그 격차를 부분 상쇄한다 — SIMD로 더 많이 계산해봤자 캐시 미스로 멈추면 소용없기 때문이다. "천장"은 *데이터가 캐시에 있는 순수 계산 커널*에서 가장 뚜렷하다.
- **오해 정정 요약**: PGO는 **있다(1.21 GA)**. 하지만 PGO가 있어도 Go는 여전히 **정적**이다. "PGO 유무"가 아니라 "빌드 스냅샷 vs 런타임 적응"이 진짜 분기점이다.

### ⑦ DevOps 관점 심화 질문

1. **default.pgo의 수명주기**: 프로덕션 pprof 프로파일을 `default.pgo`로 커밋하면, 트래픽 패턴이 바뀔 때 프로파일이 낡는다. 프로파일을 언제·어떻게 갱신하고(주기적 캡처 → PR 자동화?), 낡은 프로파일이 오히려 성능 회귀(잘못된 함수에 예산 2000을 몰아줌)를 유발하지 않는지 어떻게 CI에서 검증할 것인가?

2. **이스케이프 회귀 감시**: 리팩터링이 조용히 힙 할당을 늘려(스택→힙 이동) GC 부담을 키우는 회귀를 어떻게 잡나? `go build -gcflags=-m` 출력을 baseline과 diff하는 게이트, 또는 `benchstat`로 벤치의 `allocs/op` 회귀를 막는 파이프라인 설계는?

3. **벡터화 천장 판단 기준**: pprof CPU 프로파일에서 핫스팟이 "벡터화 불가한 스칼라 루프"로 드러났을 때, 어셈블리(또는 1.26+ 실험 `simd`)로 내려갈지 결정하는 기준은? 아키별 asm 중복이라는 유지보수 비용 대비 이득을 어떻게 정량화하나?

4. **PGO/트래픽 정합성 모니터링**: 배포된 바이너리의 PGO 프로파일과 *현재* 실트래픽 분포가 얼마나 어긋났는지를 지속 관측하려면 어떤 지표를 볼까? 역가상화가 "빗나간"(프로파일과 다른 타입이 뜨거워진) 호출을 pprof로 탐지하는 법은?

5. **itab 비용 계측**: 인터페이스 다형성이 심한 코드베이스에서 간접 호출 비용을 `perf stat`(`branch-misses`, `iTLB-load-misses`)와 pprof로 정량화하고, PGO 역가상화 전후를 A/B로 비교하는 실험 설계는?

6. **빠른 빌드의 가치 정량화**: Go의 "빠른 빌드"를 CI/CD 리드타임(커밋→배포)·개발자 피드백 루프로 환산하면 얼마인가? 이 조직 차원의 이득을, Rust로 얻을 수 있는 런타임 몇 %와 **어떤 단위로 같은 저울에 올릴 것인가**?

### ⑧ 다음 절 연결

2.1(Rust)과 2.2(Go)에서 우리는 AOT의 두 극단을 봤지만, 둘 다 같은 벽에 부딪혔다 — **정적 스냅샷은 배포 이후의 세상을 모른다.** 인라인 예산 80이든, PGO로 2000까지 올리든, 그건 전부 "빌드하는 순간의 판단"이다. 그렇다면 "실행되는 동안 계속 프로파일을 보고, 기계어를 계속 다시 만드는" 쪽은 어떨까? 그 대가로 **가장 느리게 시작(웜업)** 하지만, 충분히 오래 달리면 정적 컴파일 언어의 **CPU 효율을 특정 조건에서 역전**하는 반전이 기다린다. 1부 1.3에서 심어둔 복선 — "가장 무거운 JVM이 왜 롱런 정점 처리량에서 앞서는가" — 을 2.3에서 회수한다. Java JIT로 간다.
